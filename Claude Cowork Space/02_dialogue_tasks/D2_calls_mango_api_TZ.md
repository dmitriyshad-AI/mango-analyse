# D2 — Звонки + Mango API: ТЗ (поддержка, НЕ крупная стройка)

Зона: `services/`, `quality/`, `maintenance/`, pipeline-скрипты, `stable_runtime` (read). Не трогать `channels/`.

## Контекст из аудита
Пайплайн звонков «доведён до сильного состояния» подтверждается на ~80%: инфраструктура реальная
(ASR/Resolve/Analyze, очереди, ретраи, dead-letter, canonical DB). **Вывод: не вкладываться в крупную
переработку — «достаточно».** Задача диалога — рутина дозагрузки + лёгкая гигиена + честность метрик.

## Задачи
1. **Рутинная дозагрузка/обработка** новых аудио по Mango API — как обычно, по утверждённым правилам
   (без авто-ASR/R+A массово без подтверждения; read-only shadow poll → controlled download).
2. **Честность метрики «0 missing»:** периодически аудитить `--exclude-csv` списки и долю `resolve_status=manual`,
   чтобы «0 missing ASR/R+A» не прятал растущие исключения и ручные флаги. Отчёт: реальное покрытие vs
   условное (сколько исключено, сколько manual).
3. **Гигиена кода (с подтверждением):** заархивировать ~15 мёртвых dry-run модулей `transcript_quality_*`
   (одноразовые миграционные скаффолды) — только через manifest и перенос, не удаление.
4. **Риск-заметка для бота/KB:** вся защита PII/денег в Stage15/sanitizer — regex с захардкоженными списками
   имён (потолок recall). Если `bot_export_allowlist.csv` когда-либо подключают к RAG бота — непокрытое имя/
   искажённая ASR-цифра/новый вариант бренда пройдут. Не подключать к RAG без семантического слоя поверх regex.

## Чего НЕ делать
Не запускать массовые batch/ASR/R+A без подтверждения; не менять stable_runtime; не рефакторить монстр-файлы
(transcribe 3117 / analyze 2340 / resolve 2062) ради рефакторинга — только точечно при реальной необходимости.

## Acceptance
Новые звонки обработаны и в canonical DB; отчёт реального vs условного покрытия; мёртвые модули помечены к
архиву (через manifest); зафиксирован риск regex-recall для bot_export_allowlist.
