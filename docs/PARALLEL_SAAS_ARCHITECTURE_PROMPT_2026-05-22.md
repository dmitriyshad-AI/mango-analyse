## 1. Главный контекст

Сейчас главный приоритет проекта - довести Telegram-ботов Фотона и УНПК МФТИ до состояния внутреннего ИИ-сотрудника.

SaaS сейчас НЕ главный продуктовый фокус. Сначала система должна стабильно приносить пользу внутри компании:

- 1-3 менеджера реально пользуются ботом каждый день;
- бот помогает новым лидам;
- бот снижает нагрузку на менеджеров;
- бот не смешивает бренды;
- бот не выдумывает факты;
- есть метрики пользы;
- есть управляемый процесс обновления базы знаний, правил и тестов.

Твоя задача - не строить SaaS прямо сейчас, а спроектировать правильную архитектурную траекторию: как текущего Telegram-бота превратить в переносимого ИИ-сотрудника, а потом в SaaS для других организаций.

## 4. Главные вопросы анализа

### 4.1. Целевая архитектура ИИ-сотрудника

Опиши целевую архитектуру системы:

`канал общения -> adapter -> session/context -> база знаний -> правила безопасности -> генерация ответа -> проверка -> отправка/черновик -> лог/feedback`

Нужно явно разделить:

- Telegram Bot API pilot;
- будущую работу через реальный Telegram-аккаунт/номер;
- сайт / web-chat;
- email;
- CRM/AMO;
- Tallanto;
- customer timeline;
- база знаний;
- правила автономности;
- safety/semantic gates;
- human feedback loop.

### 4.2. Как отделить Telegram-бота от общего ядра

Сейчас Telegram-бот - первый канал. Но ядро ИИ-сотрудника должно быть независимым от Telegram.

Спроектируй слои:

- `ChannelAdapter` - Telegram, сайт, email, будущий личный Telegram-аккаунт;
- `ConversationSession` - состояние диалога;
- `ContextBuilder` - клиент, история, CRM/Tallanto, brand, facts;
- `KnowledgeAccess` - факты и gold-ответы;
- `PolicyEngine` - что можно/нельзя отвечать;
- `AnswerEngine` - черновик/автоответ;
- `PostFilter` - запреты, бренд-разделение, P0;
- `HumanHandoff` - передача менеджеру;
- `FeedbackStore` - оценка сотрудника и ошибки;
- `AuditLog` - воспроизводимость.

Укажи, какие слои уже есть в проекте, какие частично есть, каких не хватает.

### 4.3. Как хранить диалоги, решения, флаги и отзывы

Спроектируй постоянное хранилище для пилота и будущего продукта.

Нужно хранить:

- входящее сообщение;
- ответ бота;
- бренд;
- канал;
- route;
- topic;
- safety flags;
- semantic flags;
- context flags;
- какие факты использовались;
- какие источники были доступны;
- latency;
- был ли найден клиент;
- был ли менеджер подключен;
- отзыв сотрудника;
- исправленный ответ сотрудника;
- итог: помогло / не помогло / опасно / надо добавить факт / надо добавить тест.

Дай схему таблиц/объектов без реализации.

Отдельно опиши:

- что можно хранить в локальной пилотной БД;
- что нельзя хранить в открытых audit packs;
- как маскировать персональные данные;
- как потом переносить это в SaaS.

### 4.4. Multi-brand и будущий tenant isolation

Фотон и УНПК МФТИ сейчас - два бренда внутри одного проекта. В будущем это похоже на SaaS-модель с разными организациями.

Спроектируй контракт изоляции:

- `brand_id`;
- `tenant_id`;
- отдельные токены;
- отдельная база знаний;
- отдельные правила;
- отдельные логи;
- отдельные метрики;
- отдельные сотрудники;
- отдельные CRM/Tallanto credentials;
- запрет cross-brand facts в клиентском ответе.

Нужно объяснить:

- что делать сейчас для Фотон/УНПК;
- что пригодится в будущем SaaS;
- какие ошибки нельзя допустить;
- как тестировать изоляцию.

Спроектируй практичный runbook для запуска на отдельном MacBook M1 Pro:

- где хранить env;

Не пиши команды, которые удаляют файлы или меняют runtime. Только безопасные операции.

### 4.6. Критерии перехода от внутреннего проекта к SaaS

Составь `SAAS_READINESS_GATES`:

Когда можно думать о SaaS:

- сколько дней система работает без P0;
- сколько сотрудников пользуются ежедневно;
- какая доля ответов готова к отправке;
- сколько новых лидов бот довел до следующего шага;
- есть ли стабильная база знаний;
- есть ли feedback loop;
- есть ли раздельные бренды;
- есть ли логи, аудит, rollback;
- есть ли безопасная работа с CRM;
- есть ли понятная установка на новый контур.

Раздели:

- `not_ready`;
- `internal_pilot_ready`;
- `internal_production_ready`;
- `external_pilot_ready`;
- `saas_ready`.

1. Стабилизировать текущего внутреннего Telegram-бота.
2. Отделить ядро ИИ-сотрудника от Telegram-обвязки.
3. Наладить хранение диалогов и feedback.
4. Зафиксировать brand isolation.
5. Перенести запуск на M1 Pro.
6. Только потом думать о внешнем пилоте.
7. SaaS - только после доказанной внутренней пользы.

## 7. Если доступен Claude CLI

Если в среде доступен Claude CLI, используй его как независимого архитектурного ревьюера, но не соглашайся автоматически.

Процесс:

1. Сначала сам подготовь архитектурный черновик.
2. Передай Claude только результат и критерии проверки.
3. Попроси найти:
   - преждевременное усложнение;
   - риски безопасности;
   - слабые места в tenant/brand isolation;
   - где архитектура не поможет текущему боту;
   - где план слишком теоретический.
4. Сравни позиции.
5. В финале зафиксируй:
   - где согласны;
   - где спорите;
   - какое решение принято и почему.

Если Claude CLI недоступен, создай `CLAUDE_ARCHITECTURE_REVIEW_REQUEST.md`, чтобы Дмитрий мог отдать его Claude Cowork вручную.
