# ТЗ-20: черновики на ВСЕ входящие — авто-резолвер в бою (TG Фотон/УНПК + Max)

Дата: 2026-06-13. Постановщик: главный архитектор. Решение владельца (Дмитрий, 13.06): бот готовит черновик-примечание на КАЖДОЕ входящее сообщение, где удаётся однозначно опознать сделку, по каналам Telegram Фотон, Telegram УНПК и Max — сразу. Отмена прежнего ограничения «только утверждённый список 20 пар».

## Обоснование владельца (зафиксировано)

Черновик видят ТОЛЬКО сотрудники, клиенту не уходит → риск минимален даже для VIP. Поэтому переходим от ручного списка пар к авто-резолверу в проде. Per-pair сверка архитектором больше НЕ требуется как блокер включения (76 уже сверены — фон уверенности).

## Что включаем

`DRAFT_LOOP_AUTO_RESOLVER` в проде, режим НЕ `approved_list_only`, а «резолвить все чаты» по каналам telegram (оба бренда) + max. Ручные пары остаются как были.

## Предохранители (НЕ ослаблять — это вся защита)

Черновик пишется ТОЛЬКО при однозначном опознании, иначе чат пропускается (никакой записи):
1. Telegram: точное совпадение числового Telegram ID = chat_id; ровно один контакт AMO; ровно одна ОТКРЫТАЯ сделка. Только username → пропуск.
2. Max: телефон из полей Wappi (не из текста); ровно один контакт; ровно одна открытая сделка; телефона нет в стоп-листе общих семейных номеров. Числовой chat_id для Max как ключ ЗАПРЕЩЁН.
3. Бренд-чек: поле «Организация» сделки не противоречит бренду канала (если заполнено) → иначе `brand_mismatch`, пропуск. Бренд канала — из profile_id.
4. `not_before_ts`: черновики только на сообщения ПОСЛЕ момента включения (старые не трогаем — иначе зальём карточки задним числом).
5. Карантин по 403/allowlist-desync на одну сделку не валит весь цикл.
6. В каждом авто-примечании пометка «привязка автоматическая, проверьте клиента».

## Серверный список сделок

«Писать всем» ломает прежнюю модель жёсткого серверного allowlist по конкретным сделкам. Решение: серверный список переключить в режим, где разрешены ВСЕ сделки опознанных пар, НО:
- запись остаётся только через авторизованную точку AI Office (ключ);
- сохранить аварийный стоп (если надо мгновенно перекрыть — стоп-файл + серверный флаг);
- по-прежнему 0 записей вне «черновик-примечание» (никаких других типов записи в AMO).
Это правка на сервере — точечной дельтой с бэкапом по инструкции, согласовать формат с Дмитрием перед применением.

## Нагрузка

5012 чатов — резолв каждого через AMO дорогой (прошлый полный прогон ~16 мин). Нужен устойчивый режим: кэш «chat_id → lead_id» с TTL, резолв только для чатов с НОВЫМ входящим, троттлинг AMO и Tallanto (Tallanto уже даёт 429 на частоте). Без этого цикл будет тормозить и упираться в лимиты.

## Отрицательные тесты

Все из ТЗ-15 Шаг 5 + новые: чат без опознанной сделки → нет черновика (не ошибка); поток из N чатов с новыми сообщениями → черновики только по опознанным; кэш не выдаёт устаревший lead_id после закрытия сделки.

## Поэтапность включения (безопасность важнее скорости на 1 день)

1. Сначала TG Фотон + TG УНПК (ключ — Telegram ID, надёжнее).
2. Max — следом, в том же ТЗ, но отдельным флагом канала, после подтверждения, что телефон в полях Wappi стабильно есть и стоп-лист подключён.
3. Первые сутки — повышенное внимание: я (архитектор) утром делаю разбор всех черновиков по сырью; жёсткие нули в отправленном (выдумка/бренд/P0/ПДн) = немедленный стоп расширения.

## Приёмка

Отчёт в `_done`: что включено, счётчики журнала за первый прогон (опознано/пропущено по причинам/черновиков создано), подтверждение предохранителей тестами, состояние кэша и троттлинга, полный pytest.
