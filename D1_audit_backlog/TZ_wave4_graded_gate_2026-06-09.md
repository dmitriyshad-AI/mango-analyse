# ТЗ — Волна 4: ГРАДУИРОВАННЫЙ ГЕЙТ (полезный ответ вместо ухода в низкорисковой зоне)

**Статус:** ФИНАЛ (аудит второго архитектора — 4 раунда внесены).
**Дата:** 2026-06-09. **Дерево:** ветка `wave4-graded-gate` от текущего main (Волна 1 в pilot_gold). В main не мержить до замера+приёмки.

## Зачем (корень over-handoff)
По сырью: бот уходит к менеджеру в ~60% ходов, в 167/167 ложных уходов факт БЫЛ найден. Диагноз (подтверждён Волнами 1-3): мы отдали модели ТЕКСТ, но решение «отвечать / уйти» осталось у детерминированного гейта, который **бинарно** даунгрейдит ответ в уход при ЛЮБОЙ находке. Волна 1 сделала гейт ещё строже (+26 `wrong_scope` → больше уходов). Волна 3 (partial-floor) меняла ТЕКСТ ухода (вставляла факт), но РЕШЕНИЕ уйти не трогала → over-handoff не упал, факт вставлялся нерелевантный.

**Цель Волны 4:** бот максимально самостоятелен и полезен — в НИЗКОРИСКОВОЙ зоне он ОТВЕЧАЕТ (с честным маркером неуверенности), а не уходит. БЕЗ потери железной безопасности (0 смешений брендов / 0 P0-провалов / 0 выдумок чисел).

## Несущий принцип (граница, нарушать нельзя)
- **Детерминизм КЛАССИФИЦИРУЕТ находку** (жёсткая / низкорисковая) — по ТИПУ finding + `relation_to_base`. Это НЕ LLM (иначе теряем 0-гарантию).
- **Жёсткая находка → ВЕТО** (уход/блок), как сейчас.
- **Низкорисковая → модель отвечает полезно** из имеющихся grounded-фактов с маркером неуверенности; уход НЕ навязывается.
- **Результат модели снова проходит детерминированное вето** (страховка: если в полезном ответе появилось новое число / чужой бренд / P0 → вето).
- Граница ума: модель работает ТОЛЬКО в зоне, которую детерминизм пометил безопасной. LLM НЕ решает «опасно ли».

## КЛЮЧЕВОЕ отличие от Волны 3 (чтобы не повторить провал)
- Волна 3 меняла ТЕКСТ (вставляла факт перед уходом) → уход оставался, over-handoff не падал.
- **Волна 4 меняет РЕШЕНИЕ маршрута** (route не даунгрейдится в уход на низкорисковом) → бот реально ОТВЕЧАЕТ, а не уходит. over-handoff должен УПАСТЬ.
- Реализация строго на ПРЯМОМ пути (direct), НЕ в pipeline (урок Волны 3: pipeline-правки на пилоте не работают).

## Детерминированная классификация findings (таблица, НЕ LLM)
**ЖЁСТКИЙ класс → вето (уход/блок), как сейчас:**
- смешение брендов (`brand_leak`), P0-коды (возврат/жалоба/юр/спор — preblock раньше модели), `pii_echo`/`internal_leak`/`identity_disclosure`, юр-данные/реквизиты,
- выдумка КОНКРЕТНОГО продуктового числа (`unsupported_product_number`, число НЕ в фактах, `relation_to_base=absent`),
- личная гарантия результата (`made_a_promise` типа «гарантирую/точно поступит»),
- `unsupported_entity` (выдуманная сущность/продукт), `invented_generalization` БЕЗ grounded-факта.

**НИЗКОРИСКОВЫЙ класс → полезный ответ с маркером (НЕ уход):**
- `wrong_scope` (число ЕСТЬ в фактах, но скоуп не дословно совпал — `relation_to_base=adjacent`),
- `general_number_without_marker` (число структурное/нечисло-цена),
- `derived_product_claim` с `relation_to_base=adjacent` (близкий grounded-факт),
- пересказ grounded-факта в будущем времени (не личная гарантия),
- adjacent-факт того же бренда.

**Правило при сомнении:** если класс находки неоднозначен → считать ЖЁСТКИМ (безопасность вперёд). `relation_to_base=absent` → всегда жёсткий.

## Правки (за флагом `TELEGRAM_GRADED_GATE`, default OFF)

**П1. Детерминированная классификация в authoritative gate.** Якорь: `subscription_llm.py` `apply_authoritative_output_gate` (~4727), `GATE_BLOCKING_CODES` (~553), действия finding (`unsupported_product_number:block`, `wrong_scope:downgrade` ~564/575), `relation_to_base`. Ввести функцию `_finding_severity(code, relation_to_base) -> hard|low_risk` по таблице выше. Жёсткий → текущее поведение. Низкорисковый → НЕ даунгрейдить route в уход (см. П3).

**П2. Инструкция модели в первичном direct-промте (0 новых вызовов).** Якорь: сборка direct-промта (`subscription_llm.py` ~2703, где уже просят число со скоупом). Добавить: «Если точного факта под вопрос нет, но есть близкий/смежный ПОДТВЕРЖДЁННЫЙ факт — дай полезный ответ из имеющихся фактов с короткой пометкой неуверенности (например „точную цифру/деталь уточнит менеджер“), НЕ уходи к менеджеру. Уходи только на P0 (возврат/жалоба/юр/спор), смешении брендов или когда факта реально нет.» Цель — модель сразу пишет полезно, гейт перестаёт её за это выгонять.

**П3. Градация route в гейте.** Якорь: точка, где gate даунгрейдит route по findings (`apply_authoritative_output_gate`, direct-ветка ~5869). Если ВСЕ находки хода — низкорисковые (по П1): НЕ даунгрейдить route в `manager_only`/`draft`; сохранить ответ модели (route self/draft с контентом) + проставить маркер неуверенности (детерминированно, если модель не проставила). Если есть хоть одна ЖЁСТКАЯ находка → вето как сейчас.

**П4. Фолбэк-регенерация (1 раз).** Если при низкорисковой находке + наличии grounded-факта модель ВСЁ ЖЕ выдала чистый handoff-текст → одна регенерация с явной инструкцией «дай полезный ответ из этих фактов с маркером». Не зацикливать (макс 1; затем — как есть). Переиспользовать существующий regen-механизм verifier, не плодить.

**П5. Дозировка маркера.** Маркер неуверенности ставится ТОЛЬКО при реальной низкорисковой находке, не как штамп на каждый ответ. Если ход уже грунтован точным фактом (нет находки) — маркера нет. Калибровать на сырье (tone_metric не должен просесть от «мутности»).

**П6. Вето на результат (страховка).** Ответ, прошедший градацию/регенерацию, СНОВА проходит жёсткий детерминированный гейт (бренд/P0/новое число/ПДн). Если в полезном ответе появилось жёсткое → вето (уход). Безопасность не ослабляется ни на шаг.

## Граница безопасности (сквозная, НЕ трогать)
P0-латч/преблок (до модели), бренд-токены (active_brand из канала), выдумка конкретного числа (absent), ПДн-сканы (вкл. эхо из памяти), identity/мета — остаются детерминированным ВЕТО, 0 LLM в классификации безопасности. Классификация severity — детерминированная таблица, не модель.

## Флаг и паритет
`TELEGRAM_GRADED_GATE` default OFF (+ под-флаги П2/П3/П4 для изоляции при отладке, общий включает все). OFF-паритет: со всеми флагами OFF поведение direct = main байт-в-байт (полный pytest зелёный).

## NEG (обязательны, адверсариальные)
1. OFF-паритет: флаг OFF → direct не тронут.
2. ЖЁСТКИЕ границы держатся при ON: смешение брендов → уход; P0 (возврат/жалоба) → менеджер; новое число НЕ из фактов (`absent`) → уход (НЕ маркер); ПДн-эхо → блок; личная гарантия результата → уход.
3. Низкорисковое + grounded: `wrong_scope` (число из факта смежного скоупа) → полезный ответ с маркером, НЕ уход; adjacent-факт → ответ с маркером.
4. Дозировка: ход с точным фактом (нет находки) → маркера НЕТ.
5. Фолбэк: модель ушла при низкорисковой+grounded → 1 регенерация даёт ответ; не зацикливается.
6. Результат-вето: модель в «полезном ответе» вставила новое число/чужой бренд → поймано вето.

## Замер (М1, против A)
Базлайн = прогон A (Волна 1, `20260608_wave_A_w1only`). Один прогон GRADED (Волна 1 + `TELEGRAM_GRADED_GATE`) на канонном наборе, полный env pilot_gold, parallel 4, судья v9.
**Критерии успеха:**
- **over_handoff_turn_rate СУЩЕСТВЕННО вниз** (с ~0.60) — это главное, Волна 4 меняет решение, а не текст.
- `manager_only` route вниз, self/draft с контентом вверх.
- unedited-rate не падает; тон (tone_metric) не просел от маркеров.
- **ЖЁСТКО 0:** новые смешения брендов / P0-провалы / выдумки конкретных чисел / ПДн-эхо — регрейд по сырью + NEG.
Регрейд по сырью обязателен (особенно: реально ли низкорисковые ответы полезны и уместны, не нерелевантны как partial-floor).

## Вне scope
Не трогать жёсткое вето; не отдавать классификацию безопасности модели; per-clause (Волна 3 В6) отдельно; автоном (после пилота). partial-floor (Волна 3 В1) — откатить/отключить (вреден); В3/В4 (promise/redact) — проверить изоляцией отдельно, не в этой волне.

## Аудит второго архитектора (внесён)
- **Р1 «классификацию findings тоже модели — умнее?»:** ОТКЛОНЕНО. Классификация безопасности = детерминизм, иначе теряем 0-гарантию (обсуждено с Дмитрием). Модель только генерит в открытой зоне. Принято.
- **Р1 «неоднозначный finding»:** при сомнении → жёсткий; `relation_to_base=absent` → всегда жёсткий. Принято.
- **Р2 «как graded-gate УБИРАЕТ уход, если partial-floor не убрал?»:** ключевое — graded-gate меняет РЕШЕНИЕ (route не даунгрейдится), а не текст; бот отвечает, а не уходит. Зафиксировано как несущее отличие. Принято.
- **Р2 «маркер-штамп → тон»:** дозировка (П5), маркер только при реальной находке, мерить tone_metric. Принято.
- **Р3 «relation_to_base уже есть?»:** переиспользовать существующую разметку (Волна 1/verifier), не плодить. Принято.
- **Р3 «реализация — точно direct, не pipeline»:** да, урок Волны 3; все правки в direct-ветке (gate 5869, промт 2703, model call 3105). Принято.
- **Р4 «риск: низкорисковый ответ нерелевантен (как partial-floor цену на жалобу)»:** отличие — здесь отвечает МОДЕЛЬ из контекста (а не механическая вставка любого exact-факта), плюс P0 жёсткий (жалоба → уход, не низкорисковая). Регрейд по сырью обязателен на релевантность. Принято.
