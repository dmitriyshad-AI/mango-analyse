# ТЗ-100 (D8): LLM-резолвер детей семьи — «сколько детей и кто есть кто» (v2, после 2 раундов аудита)

Дата: 2026-06-14. Постановка: Claude (архитектор) + решение Дмитрия. Исполнитель: D8 (worktree `codex/tz24-dubli-deti` от main). Заменяет детерминированный черновик и идею ТЗ-101. v2 = учтены правки второго архитектора (раунд 2). Опора: Блок А (перепроверен), Блок Б, оба аудита.

## Решение Дмитрия (14.06): понимание — через LLM, не регексп
Дубли детей из ДВУХ источников: безымянные слоты дробятся (хеш то класса, то предмета) и опечатки имён (Степан/Стёпа). Обе — задачи ПОНИМАНИЯ. Один LLM-шаг (gpt-5.4-mini); детерминизм собирает вход, применяет ответ и ВЕРИФИЦИРУЕТ выход. Гибрид как ADR-001.

## Объём (посчитано по данным, не «несколько тысяч»)
Скоуп = семьи с 2+ детскими слотами ИЛИ ≥1 безымянным слотом = **7 512 семей = 7 512 LLM-вызовов** (профилей с детьми 10 368; с 2+ слотами 4 410; с ≥1 безымянным 7 097; чистых одиночных именованных — пропуск 2 856). Многоимённых (кандидаты-опечатки) — **2 319**. Это и есть ожидаемый `llm_calls_total`; от него считать микропробу и бюджет.

## Переиспользование (НЕ плодить своё)
- Кэш ответов — существующий `src/mango_mvp/services/llm_response_cache.py` (`LLMResponseCache`: sha256 от namespace+provider+model+prompt_version+prompt, атомарная запись). Namespace `child_resolver_v1`. Им уже пользуются `deal_llm.py`, `theme_assigner_llm.py`.
- Конкуренция/ретраи/таймаут — паттерн `src/mango_mvp/question_catalog/theme_assigner_llm.py` (ThreadPoolExecutor, max_concurrency). НЕ звать в голом цикле (7 512 последовательных = часы).
- **ПДн:** child_name — это ПДн ребёнка, уходит во внешнюю модель и в кэш открытым текстом. Кэш-namespace резолвера держать локально в ignored-пути; в отчётах/микропробе имена показывать обезличенно.

## 1. Кого спрашиваем
Только семьи из скоупа выше. Одиночный чистый именованный ребёнок — пропуск без вызова.

## 2. Вход на семью
Список упоминаний: `mention_id`, имя (если есть), класс (строкой, как есть — «6», «6 класс», «6 и 9 класс» не парсить на нашей стороне), предмет(ы), event_at, бренд, короткая опора. `format` НЕ подаём — это поле СЕМЬИ (`is_child_field`={child_name,grade,subject}, builder.py:722; format имеет 0 записей с child_key).

## 3. Выход (строгий JSON)
`{"children":[{"child_id","canonical_name"|"","name_variants":[...],"grades":[...],"subjects":[...],"brands":[...],"mention_ids":[...]}]}`
Промпт-правила модели: опечатки/варианты/уменьшительные ОДНОГО имени → один; РЕАЛЬНО разные имена → разные; при СОМНЕНИИ не сливать (ошибка «слить двух разных» дороже); класс растёт со временем (9→11 — один ребёнок); НЕ выдумывать имён вне входа; не отбрасывать упоминания; каждый ребёнок имеет ≥1 mention_id.

## 4. Идемпотентность
Кэш по хешу нормализованного входа (+model+prompt_version) через `LLMResponseCache`. Тот же вход → из кэша, модель не перевызывается.

## 5. Применение к слотам (детерминированно)
Точка — внутри/вместо `apply_child_slot_merge_candidates` (builder.py:48), СТРОГО до `apply_superseded_rules` (builder.py:50). Каждый distinct child = слот; `child_key`: при canonical_name — производный от него, при пустом — стабильный `child_unnamed_<n>`. Поля переносить со сменой только `child_key` (происхождение frozen, как rekey 524-532). Merge-маркеры `child_slot_merge_candidate` (841) — **потребителя НЕТ** (греп пуст в existing_clients/), для разрешённых семей просто НЕ писать (не тратить силы на пересборку).

## 6. Детерминированная ПРОВЕРКА выхода (вето; при нарушении — семья без изменений + флаг)
1. **Нет выдуманных имён:** каждое canonical_name/name_variant встречается во входе.
2. **Имя↔ребёнок (а не «где-то»):** для каждого упоминания с именем — имя ЭТОГО упоминания каноникализуется в того же ребёнка, к кому упоминание отнесено (ловит misattribution «петя в Васю»).
3. **Полнота мэппинга:** каждый mention_id — ровно у одного ребёнка; ребёнка с пустым mention_ids НЕТ.
4. **Не слиты разные дети по классу:** если упоминания одного ребёнка несут НЕСОВМЕСТИМЫЕ классы в один период (разница ≥3 при пересечении дат ~200 дней) — отклонить (модель ошиблась; ≈133 семьи риска — именно их модель должна оставить раздельно). ВНИМАНИЕ: это НЕ «ни один класс не теряется» — взросление 9→11 у одного ребёнка нормально (карточка показывает последний класс, это верно).
5. **Здравость числа:** детей ≤ числа упоминаний и не больше (числа различных имён + 1 безымянный).
6. **Бренд-провенанс на ФАКТ сохраняется** (ProfileFieldCandidate.brand per-row, contracts.py:69) — rekey бренд факта не трогает. Бренд-разделение клиенту — это фильтр на ВЫХОДЕ бота, не на этой менеджерской карточке; хранение общее (политика 10.06).
7. **Fail-soft:** модель недоступна/таймаут/невалидный JSON → семья без изменений, флаг, не падать.

## 7. Особые случаи
- **Общий телефон 2+ семей.** В профилях `primary_phone` УНИКАЛЕН (0 делимых) — сигнала «спутанный» во входе билдера НЕТ. Стоп-лист лежит ВНЕ git: `~/.mango_secrets/shared_phones_stoplist.json` (локально ~112, не 367 — HANDOFF_tz17). Загрузить его, сматчить по нормализованному телефону к profile_id, такие профили в резолвер НЕ подавать (флаг «возможно две семьи»). **Если файла нет — прогон НЕ запускать** (иначе модель сольёт две семьи), сообщить явно.
- **Мусорный класс** — строкой модели (на нашей стороне не парсить как число).

## Флаг и запуск
Флаг `PROFILE_LLM_CHILD_RESOLVER` default OFF. Прогон LLM — по «да» Дмитрия. Сначала МИКРОПРОБА 15-20 семей (опечатки + безымянные + один shared-phone-кандидат для проверки пропуска), вход/выход обезличенно, потом шире.

## NEG-тесты
- OFF → байт-в-байт как сейчас.
- Опечатки одного имени → один ребёнок; имя не выдумано; misattribution «петя→Вася» отклоняется проверкой 2.
- Два РАЗНЫХ имени → два ребёнка.
- «1 именованный + N безымянных» → безымянные привязаны, имя не задвоено.
- Несовместимые концентрические классы → НЕ слиты (проверка 4).
- Взросление 9→11 у одного ребёнка → один ребёнок, карточка кажет 11 (это НЕ потеря).
- Общий телефон (из стоп-листа) → пропущен; стоп-лист отсутствует → прогон не стартует.
- Невалидный JSON → семья без изменений.
- Ребёнок с пустым mention_ids → ответ отклонён.
- Идемпотентность: повтор = тот же результат (кэш).
- Бренд: факт УНПК и факт Фотона у одного ребёнка сохраняют brand per-row.

## Замер до/после (на снимке)
