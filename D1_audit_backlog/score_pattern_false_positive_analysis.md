# Анализ pattern «N балл*» (Волна 3a) на риск false positive

Автор: второй Claude. Дата: 2026-05-29. Read-only. Источник: `subscription_llm.py` (`UNSUPPORTED_PROMISE_PATTERNS:619`, `find_unsupported_numeric_promises:1617`, `_fresh_fact_texts:7287`, `_claim_supported_by_facts:7338`, `_is_verified_safe_numeric_template:1631`), `client_safe_facts_*.jsonl`.

**Краткий вывод:** да, риск false positive реальный и конкретный. Pattern `\b\d{1,3}\s*балл\w*` (строка 622) — это страж на ВЫХОДЕ (текст драфта бота), и он матчит легитимную статистику результатов «+25 баллов», которая ЕСТЬ в client-safe слое ОБОИХ брендов. Блокировка зависит от того, дошёл ли факт до стража с флагом `facts_fresh=True` — а он по умолчанию НЕ доходит. Хуже: блокировать статистику результатов прямо противоречит политике (safe-template разрешает «показать статистику результатов»).

## Вопрос 1 — есть ли в KB факты с «N баллов»?

Прошёл оба client-safe файла + manager_only. Найдено:

- **`results_social_proof.ege_avg_above_country_pts`** — В ОБОИХ брендах, client-safe:
  - Фотон: «Фотон: средний результат ЕГЭ у учеников выше среднего по стране на **25 баллов**.»
  - УНПК: «УНПК: средний результат ЕГЭ у учеников выше среднего по стране на **25 баллов**.»
  - Новый pattern матчит «25 баллов».
- В `manager_only`: `high_risk_keywords.always_manager_only.14/15` — «обещайте баллов / гарантируйте баллов» (внутренние high-risk маркеры, БЕЗ числа, pattern их не трогает — и правильно, они и так manager_only).
- Других client-safe фактов с «N баллов» нет. Поверхность узкая, но НЕнулевая, и это **факт-социальное доказательство, который менеджеру/боту полезно приводить в продаже**.

## Вопрос 2 — может ли pattern заблокировать их как unsupported_promise, даже когда они в rfk?

ДА, при условии. Механизм:
1. `find_unsupported_numeric_promises(draft, context)` (`:1617`) → `_extract_numeric_promise_claims` (`:7276`) извлекает «25 баллов» как claim (pattern сработал).
2. Затем фильтр: claim остаётся unsupported, если `not _claim_supported_by_facts(claim, _fresh_fact_texts(context))`.
3. **`_fresh_fact_texts` (`:7287`) возвращает `()` (пусто), ЕСЛИ в контексте НЕ выставлен `facts_fresh=True`** (или стоит `facts_stale`). То есть страж НЕ видит rfk автоматически — он берёт факты только из `confirmed_facts`/`knowledge_snippets` И только при явном флаге свежести.
4. Если факт «25 баллов» не дошёл до стража с `facts_fresh=True` → `fact_texts=()` → claim НЕ поддержан → **`unsupported_promise_detected` → route `manager_only`**.

То есть pattern сам по себе НЕ обходит факт-сверку, но (а) добавляет новую форму claim, которую теперь обязательно надо подтвердить фактом, и (б) подтверждение завязано на хрупкий `facts_fresh`-флаг + наполнение `confirmed_facts`. Это тот же корневой механизм, что в §11.3 (unsupported_promise при retrieved_match): сверка есть, но зависит от прокидки контекста. Если контекст не помечен fresh — легитимная статистика «+25 баллов» блокируется.

**Дополнительный аргумент против широкого pattern:** политика проекта ЯВНО разрешает показывать статистику результатов — `RESULT_GUARANTEE_SAFE_TEXT` (`:98`): «…может показать, какая у нас статистика результатов». А `\b\d{1,3}\s*балл\w*` блокирует именно статистику «на 25 баллов выше». То есть pattern конфликтует с утверждённой политикой.

## Вопрос 3 — компенсация

Уже есть более узкий и правильный механизм: `RESULT_GUARANTEE_INPUT_RE` (`:558-560`) целит ОБЕЩАНИЕ/гарантию: `гарантир…балл`, `(сдаст|балл|результат)…гарантир`, `(90|100)…балл`. Новый bare-pattern `\b\d{1,3}\s*балл\w*` дублирует и ПЕРЕхватывает его, цепляя статистику.

Варианты компенсации (по приоритету):

1. **РЕКОМЕНДУЮ — сузить score-pattern до promise-контекста.** Заменить bare `\b\d{1,3}\s*балл\w*` на форму с глаголом-обещанием/будущим рядом, по образцу `RESULT_GUARANTEE_INPUT_RE`: `(?:гарантир\w*|обеща\w*|наберёт\w*|получит\w*|сдаст|до)\s*[^.!?\n]{0,40}\d{1,3}\s*балл\w*` ИЛИ `\d{1,3}\s*балл\w*[^.!?\n]{0,40}(?:гаранти|обеща)`. Тогда «средний результат выше на 25 баллов» (статистика, прошедшее/настоящее) НЕ блокируется, а «наберёте 100 баллов»/«гарантируем 90 баллов» — блокируется. Это согласуется с политикой «статистику показывать можно, обещать результат нельзя».
2. **Whitelist в `_is_verified_safe_numeric_template` (`:1631`).** Добавить точные тексты обоих социальных фактов (foton/unpk «…на 25 баллов») в множество `verified_templates` — тогда `find_unsupported_numeric_promises` вернёт `()` на строке 1622. Минус: матч ТОЧНОГО текста; парафраз бота («в среднем на 25 баллов выше») не попадёт. Годится только если бот выдаёт факт дословно (как safe-шаблон). Это «заплатка», не системное решение.
3. **Системно (корень §11.3) — гарантировать, что `confirmed_facts` + `facts_fresh=True` доходят до стража**, когда факт реально в rfk. Тогда `_claim_supported_by_facts("25 баллов", [«…на 25 баллов…»])` вернёт True (нормализованная подстрока совпадает). Это лечит весь класс, не только баллы.

**Лучшее сочетание:** (1) сузить pattern до обещания + (3) починить прокидку fresh-контекста. (2) — необязательная подстраховка. `_is_verified_safe_numeric_template` существует и подходит как механизм, но для ФАКТА (а не фиксированного шаблона) он неудобен — фактам место в fact-сверке, а не в template-whitelist.

## Вопрос 4 — какие ещё формулировки ложно попадут под pattern

Важно: страж — на ВЫХОДЕ (драфт бота), а не на сообщении клиента. Поэтому клиентское «получил 3 балла» само по себе не блокируется; риск — когда БОТ произносит/повторяет число баллов. Ложные срабатывания у бота:

- **Статистика результатов** (реальный кейс): «средний результат выше на 25 баллов» — единственный client-safe факт с числом баллов; блокируется по в.2.
- **Шкала/общие факты:** «ЕГЭ оценивается до 100 баллов», «проходной — около 80 баллов», «100 баллов — максимум» (общеизвестное, не обещание).
- **Эхо клиента:** клиент «я написал пробник на 85 баллов» → бот «вы набрали 85 баллов, это сильно» → pattern на «85 баллов».
- **Не-ЕГЭ «баллов»:** школьные оценки «5 баллов», рейтинги/отзывы «10 баллов из 10», бонусные «начислим 50 баллов» (если появится лояльность).
- Технически pattern `\d{1,3}\s*балл\w*` цепляет 1-3-значное число + «балл*»: «5 баллов», «25 баллов», «100 баллов» — все матчатся; «1000 баллов» (4 знака) — нет (мелочь).

Все они уйдут в `manager_only`, если не подтверждены fresh-фактом. Для статистики это прямой регресс продажного аргумента; для шкалы/эха — лишняя эскалация.
