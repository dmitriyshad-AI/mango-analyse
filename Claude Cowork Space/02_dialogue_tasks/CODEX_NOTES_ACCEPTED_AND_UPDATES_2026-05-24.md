## Конкретные поправки (вписать в работу)

### П1. P0-синхронизация MAIN↔D4 — через АРТЕФАКТ, не словами (замечание №2)
MAIN (D1) производит ЕДИНЫЙ канонический P0-артефакт — источник правды:
- `p0_recall_spec` (один модуль/файл): жёсткий список сигналов (суд/иск/прокуратура/роспотреб/претензи/
  расторг-договор/возврат-формулировки/верн(ите/уть)+деньги-оплата-средства/`мошенн*`/`незаконн*`/`жал(?:об|у|ова)*`)
  + recall-набор реальных P0 + benign-набор (что НЕ P0).
- D4 НЕ дублирует список вручную: KB-триггеры P0 ИМПОРТИРУЮТ/ссылаются на тот же артефакт (или генерируются
  из него). Тест: один и тот же recall-набор проходит и в коде (MAIN), и в KB-триггерах (D4).
Так список не разъедется между диалогами.

### П2. SaaS-заморозка — сначала карта зависимостей (замечание №3)
Перед тем как пометить productization `parked`/снять с CI (D3): СНАЧАЛА построить карту зависимостей —
какие ТЗ-тесты и какие Mango/processing-скрипты импортируют `productization`. Заморозить ТОЛЬКО реально
изолированную SaaS-поверхность (API/HTTP/scheduler/supervisor/appliance/tenant/ASR-sandbox/mail_archive);
ядро захвата (`mango_office_client`/`product_db`/`repository`/`contracts`/`capture`) и всё, что использует
звонковый контур/D2, оставить рабочим и в CI. Не сломать звонки. Ничего не удалять.

### П3. D4 — preflight границ из-за грязного дерева (замечание №5)
В дереве уже висят удаления старых KB-релизов из параллельных потоков. Перед стартом D4: зафиксировать
ИЗМЕНЯЕМЫЕ файлы блока KB-гигиены; НЕ смешивать с висящими удалениями релизов; работать отдельным коммитом;
если есть конфликт с чужими удалениями — остановиться и согласовать, не откатывать чужое.

### П4. D5 — машинно-читаемый корпус, не только markdown (замечание №6)
Выход D5 — YAML/JSON, который MAIN сможет подключать к генерации. Схема каждого элемента:
```yaml
- brand: foton|unpk|any
  topic: <тема/класс ситуации>
  situation: <когда применять>
  example: <живой пример ответа/реплики, из корпуса/звонков, не выдумка>
  forbidden: [<что нельзя на этой теме>]
  safe_next_step: <один безопасный следующий шаг>
  requires_fact_ids: [<факты, без которых пример не инъектировать>]
  source: <откуда взято>           # звонки/корпус/воронка
  dedup_ref: <ссылка на существующий элемент, если повтор>
```
Markdown-карта пути и принципы — отдельно (для людей), но БОТ потребляет машинный корпус. Каждый элемент с
`source`/`dedup_ref`, чтобы не плодить третий источник правды (есть корпус Claude `03_corpus/` + звонки + воронка).
