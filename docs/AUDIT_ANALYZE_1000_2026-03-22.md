# Audit Analyze 1000

Дата: 2026-03-22

## Объем

- Проверена выборка из первых `1000` звонков со статусом `analysis_status = done`.
- Источник: `mango_mvp.db`
- Витрина: [sales_workbook.xlsx](/Users/dmitrijfabarisov/Projects/Mango%20analyse/sales_workbook.xlsx)

## Методика

1. Автоматические метрики по всем `1000` analyzed calls.
2. Стратифицированная выборка по корзинам:
   - хорошие полные кейсы
   - `non_conversation`
   - `product_missing`
   - `next_step_missing`
   - `subjects_missing`
3. Ручная сверка `analysis_json` с `transcript_text` на конкретных звонках.

## Ручная выборка

### Хорошие кейсы

- `id=405`
  - Корректно выделены: `математика`, `физика`, онлайн-формат, пробная неделя, следующий шаг.
  - Summary по смыслу совпадает с транскриптом.

- `id=406`
  - Корректно выделены: `физика`, `7 класс`, очный групповой формат, оплата, email, следующий шаг.
  - Хороший CRM-конспект, пригодный для таблицы.

- `id=429`
  - Корректно выделен продукт `летний лагерь`, интерес к августовской смене, чувствительность к цене и следующий шаг.
  - `subjects_missing` здесь не ошибка, потому что для лагеря предмет необязателен.

### Пограничные, но приемлемые кейсы

- `id=401`
  - `product_missing` выглядит допустимо.
  - В разговоре есть предмет `информатика`, но продукт действительно не подтвержден.

- `id=407`
  - Это звонок действующему клиенту по прогрессу ученика.
  - `product_missing` допустим, потому что звонок не продажный.

- `id=421`
  - Сервисный звонок по завершению семестра и пропущенным занятиям.
  - `next_step_missing` допустим, потому что продажного шага нет.

- `id=438`
  - Административный вопрос по налоговой справке.
  - Смысл распознан правильно, но summary стилистически слабее: дублируется дата и фраза построена тяжеловесно.

- `id=487`
  - Продукт `индивидуальные занятия` определен корректно.
  - `subjects_missing` допустим, потому что предмет явно не назван.

### Реальные проблемные кейсы

- `id=493`
  - Технический, но содержательный звонок по вступительному онлайн-тесту.
  - Сейчас помечен как `non_conversation`.
  - Это слишком грубая классификация. Корректнее: `technical_call` или `service_call`, но не `non_conversation`.

- `id=528`
  - Тег `non_conversation` конфликтует с остальными полями.
  - В звонке одновременно:
    - стоит `product = летний лагерь`
    - есть `next_step = Отправить материалы`
    - но при этом стоит `non_conversation`
  - Такая комбинация разрушает фильтрацию и сегментацию.

- `id=545`
  - Содержательный сервисный звонок по оплате, расписанию и старту обучения.
  - Сейчас помечен `non_conversation`, что некорректно.
  - Это не продажный звонок, но и не пустой/технический дозвон.

## Выводы

### Что работает хорошо

- Пустых summary нет.
- Диалог больше не сливается в сырой `MANAGER/CLIENT`-дамп.
- Для явных продажных кейсов summary и поля уже пригодны для ручной работы в Excel.
- Консервативное поведение по `product_missing` чаще полезно, чем вредно: система не выдумывает продукт там, где его нет.

### Что работает средне

- `product_missing`, `subjects_missing` и `next_step_missing` как сырые метрики завышают ощущение проблем.
- Во многих случаях это не баг, а нормальный результат для:
  - сервисных звонков
  - административных вопросов
  - звонков действующим клиентам
  - лагерей и смен без явного предмета

### Главная проблема

Текущая ошибка не в summary как таковом, а в таксономии звонков:

- `non_conversation` используется слишком широко.
- В него попадают:
  - реальные технические звонки
  - сервисные обращения
  - административные кейсы
  - короткие, но содержательные разговоры

Из-за этого:

- искажаются фильтры в Excel/CRM;
- часть полезных звонков выглядит как мусор;
- появляются логические противоречия: `non_conversation` вместе с `product` и `next_step`.

## Рекомендации перед следующей волной Analyze

1. Разделить типы звонков:
   - `non_conversation`
   - `service_call`
   - `technical_call`
   - `existing_client_progress`
   - `sales_call`

2. Ужесточить правило для `non_conversation`:
   - не ставить этот тег, если есть подтвержденный смысловой блок:
     - сервисный вопрос
     - техническая проблема
     - обсуждение расписания/оплаты
     - осмысленный следующий шаг

3. Запретить несовместимые комбинации:
   - `non_conversation` + `target_product`
   - `non_conversation` + `next_step`
   - `non_conversation` + длинный содержательный summary

4. Стилистически почистить summary:
   - убрать повтор даты/времени в тексте
   - упростить формулировки административных звонков

5. Не считать `subjects_missing` дефектом для:
   - лагерей
   - смен
   - административных звонков
   - действующих клиентов без обсуждения предмета

## Практическая оценка

- Для ручной проверки и рабочей Excel-витрины текущие `1000` строк уже пригодны.
- Для полностью автоматической сегментации лидов по тегу `non_conversation` или по наличию/отсутствию продукта качество пока недостаточно надежно.
- Следующий шаг перед новой массовой волной: поправить таксономию `Analyze`, а не саму ASR-часть.
