# ТЗ-3: AMO+Wappi, фаза 1 — читающее подключение + черновик в тестовую сделку (приоритет 3, старт после ТЗ-2)

Контекст: доставка спроектирована (Фаза 0: Wappi=транспорт, бренд из profile_id, НЕ из воронки), доступы администратора у Дмитрия ЕСТЬ. Это критический путь пилота: без доставки черновиков менеджеру пилот невозможен.

Правила безопасности (жёсткие):
- Секреты НЕ в репо, НЕ в чат, НЕ в task.yaml: Дмитрий вводит логины/пароли/токены сам, в env-файл вне репозитория (путь согласовать, например `~/.mango_secrets/amo_wappi.env`, права 600). Код читает только из env.
- Фаза 1 — ЧТЕНИЕ. Единственная разрешённая запись: примечание-черновик в ОДНУ тестовую сделку, согласованную с Дмитрием. Allowlist по id сделки в конфиге; попытка записи вне allowlist — исключение + лог (это NEG-тест).
- Боевые клиенты не затрагиваются; авто-отправка клиенту — не существует как код.

Шаги:
1. Схема env-секретов + README для Дмитрия (3 строки: куда положить, как проверить).
2. Клиент AMO: чтение воронок/сделок/контактов; клиент Wappi: список profile_id. Маппинг profile_id→бренд — конфиг-файл, без хардкода в коде.
3. Запись примечания-черновика в тестовую сделку: текст черновика + бренд + время + маркер «ЧЕРНОВИК БОТА, не отправлено».
4. Каркас лога правок менеджера с первого дня: поле «что отправил менеджер» рядом с «что предложил бот» (для unedited_rate и кодов причин) — пока просто структура хранения.

Критерий приёмки: на тестовой сделке появляется примечание с верным брендом из profile_id; чтение воронок работает на обоих брендах; NEG «запись вне allowlist блокируется»; секретов нет ни в одном коммите (проверить грепом по диффу).
