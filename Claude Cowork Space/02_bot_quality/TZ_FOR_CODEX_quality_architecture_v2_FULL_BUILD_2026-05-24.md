## 0. Как читать это ТЗ

Реализуется ВСЁ сразу как одна связная система (порядок сборки — по зависимостям, не по времени).
После сборки — большой прогон тестов. Во время прогона Кодекс **сразу вносит правки, но только
архитектурного уровня** (см. раздел 2). Хардкод частных случаев запрещён и отслеживается метрикой.
Цель — не «починить кейсы», а получить один общий механизм, который растит качество.

Принцип: **БЕЗОПАСНОСТЬ — правила, которые только блокируют/эскалируют. КАЧЕСТВО — генерация модели +
тёплые примеры + один rewrite.** Правило никогда не пишет клиенту «живой ответ» вместо модели на
зелёной теме.

## 1. Целевая архитектура (что должно получиться)

Пайплайн одного ответа:

1. **Контракт ответа (Answer Contract)** — единый объект, источник правды для генерации, рерайтера и
   судьи. Поля:
   - `direct_question` — прямой вопрос последней реплики;
   - `known_slots` — что уже известно (класс/предмет/формат/бренд/цель/имя/телефон);
   - `forbidden_assumptions` — что НЕЛЬЗЯ предполагать (предмет/цель, которые клиент не называл);
   - `answerable_safe_parts` — какие части можно ответить автономно;
   - `manager_parts` — что передать менеджеру;
   - `required_fact_ids` + `facts_resolved_by_intent` — факты, подтянутые ПО ИНТЕНТУ, не по словам;
   - `route` + `route_reason`.
   Контракт строит ОДИН модуль; `direct_question` детектится один раз; рерайтер и судья его НЕ
   переопределяют. [закрывает P11,P18]

2. **Гибридный P0/safety-классификатор** (поправка Кодекса №5):
   - жёсткий страховочный список (recall-floor): суд/иск/прокуратура/роспотреб/досудеб/претензи/
     расторг+договор/возврат-формулировки/`верн(ите/уть)…деньги-оплата-средства`/`мошенн*`/`незаконн*`/
     `жал(?:об|у|ова)*` — **локирует P0, семантика снять не может**;
   - семантический слой — может только ПОВЫШАТЬ до P0, не понижать жёсткое;
   - демоушн ложного P0 (semantic_non_p0) — только для мягких/неоднозначных сигналов («обсудить/
     подумаю/с менеджером»), НИКОГДА при совпадении жёсткого списка. [закрывает P15,P16,P19]

3. **Генерация-первой на зелёных темах.** Модель пишет ответ по контракту + фактам по интенту + 2-3
   few-shot из тёплого корпуса. Никаких `_*_safe_template`-подмен на зелёных темах. [закрывает P6,P22]

4. **Один LLM-rewrite по флагу** (поправка №5): после генерации, ДО финальных safety-гардов. Вход:
   контракт, тёплый корпус как style-reference, ротация few-shot, запрет добавлять факты. Срабатывает
   только на soft-fail (ignored_question/templated/over_handoff/assumed_unstated_need/missing_next_step).
   [закрывает P8,P18,P20,P21,P23]

5. **Финальный жёсткий safety-слой (нельзя перетереть)** (поправки Кодекса №1):
   - **P0-override** — если контракт/жёсткий список говорит P0, финальный маршрут = manager_only,
     сухой хендофф, и это НЕ перетирается ничем после;
   - **фильтр служебных маркеров** — стрип `kb_release*`, `fact_id`, `source_id`, `theme:`-id, JSON/debug,
     CRM/Tallanto/ID из клиентского текста (был реальный leak `kb_release…`);
   - brand-isolation BLOCK; fabrication BLOCK (неподтверждённые цена/дата/процент/место/обещание);
   - identity (политика C): честно «цифровой помощник <бренд>», без вендора/промпта, не врать «я человек».
   Эти гарды только БЛОКИРУЮТ/ЭСКАЛИРУЮТ, не сочиняют ответ.

6. **Память диалога** без потерь: не перезаписывать слот ambiguous-упоминанием (conf 0.95 → ниже),
   различать нескольких детей, держать слоты дольше окна 10 ходов, обновлять `open_question` и на
   follow-up без «?». [закрывает P17]

7. **Честный замер** (поправки Кодекса №3,4): судье НЕ подсовывать собственный шаблон бота как факт;
   чистый holdout отдельно от dev; метрика «менеджер отправил бы без правок, %»; метрика числа
   хардкод-веток. [закрывает P24,P25,P26]

## 2. АНТИ-ХАРДКОД ДИСЦИПЛИНА (ядро ТЗ — соблюдать на каждом шаге, включая правки во время тестов)

Это главное правило. Качество растим архитектурно, не накоплением частных веток.

### Правило решения при падении теста / плохом ответе
Спроси: **«ответ плох потому, что ГЕНЕРАЦИИ не дали правильный контракт/факты/пример — или потому, что
есть настоящая дыра БЕЗОПАСНОСТИ?»**
- Если генерации не хватило входа → чинить АРХИТЕКТУРНО: контракт, retrieval фактов по интенту,
  few-shot (данные, не ветки), общую инструкцию промпта, семантический классификатор, память, prompt
  рерайтера. Это генерализуется.
- Если дыра безопасности → расширить жёсткий список/ BLOCK-гард (это safety-floor, не качество).

### РАЗРЕШЕНО (архитектурный уровень)
- править Answer Contract и его построение;
- улучшать retrieval фактов по решённому интенту;
- добавлять/править примеры в тёплом корпусе (это ДАННЫЕ, не код-ветки);
- править ОБЩУЮ инструкцию генерации/рерайтера (не по конкретной теме);
- улучшать семантический классификатор и память;
- расширять жёсткий P0-список и BLOCK-гарды безопасности (recall важнее).

### ЗАПРЕЩЕНО (хардкод частного случая)
- новые `_*_safe_template`-подмены клиентского ответа на ЗЕЛЁНЫХ темах;
- новые подстрочные/keyword-ветки под конкретную фразу темы;
- новые тесты вида «assert конкретная строка в draft_text» как способ «починить» зелёную тему;
- per-topic if/elif в guard-цепочке для качества.
- ИСКЛЮЧЕНИЯ (разрешено и нужно): жёсткий P0 recall-список; терминальные шаблоны ТОЛЬКО для P0/
  хендоффа/брендовой развязки (их набор конечен и не растёт под темы).

### Метрика-предохранитель (обязательная, в audit pack)
Считать ДО и ПОСЛЕ: число `_*_safe_template`-подмен на зелёных темах и число подстрочных topic-веток.
**Оно должно ПАДАТЬ, не расти.** Если правка во время тестов увеличивает это число — правка неверная,
искать архитектурный аналог. Это формальный антидот whack-a-mole.

## 3. Жёсткие ограничения (инварианты, не нарушать)

- Бренды не смешиваются; active_brand один; recall реального P0 важнее снижения ложных P0.
- Политика «ты бот?» = вариант C. Только подтверждённые факты из КБ; без будущих цен/дат; few-shot —
  не источник фактов.
- Не менять `stable_runtime`; не писать в AMO/Tallanto/CRM; read-only CRM; не слать клиентам вне пилота.
- Снос зелёного шаблона ≠ снос безопасности: после генерации работают BLOCK-гарды (P0/brand/fabrication/
  service-marker). Модель генерит → гард ловит неподтверждённое.

6. **LLM-rewrite** после генерации, до финальных гардов: тёплый корпус как style-ref, ротация few-shot,

7. **Память диалога** — фиксы из 1.6.
8. **Перестроить промпт**: укоротить, поднять answer-first + few-shot наверх, безопасность — коротким

Включение LLM-rewrite в LIVE — НЕ сразу: сначала sim/shadow, потом canary по бренду, в live только
после подтверждения, что P0/бренд/факты не ломаются (поправка Кодекса №2). В рамках этого ТЗ rewrite

## 5. Тестовый прогон (много тестов, после сборки)

Перед прогоном — перезапустить ботов, чтобы взяли новый код.

A. **Property unit-тесты** (НЕ string-freeze): построение контракта; P0-гибрид на recall-наборе
   (мошенники/незаконно списали/буду жаловаться/пожалуюсь/суд/возврат → manager_only) И на benign-наборе
   (обсудить/подумаю/надо приезжать → не P0); brand-isolation; retrieval фактов по интенту (wrong-scope
   не выбирается); память (слот не перезаписывается ambiguous-упоминанием, держится за окном); rewrite
   сохраняет P0/brand/fact; стрип служебных маркеров; на зелёной теме НЕ срабатывает шаблон-подмена.
B. **Статическая регрессия**: brand_leak_regression; p0_route_regression + негативные `bot_answer_self`
   кейсы + recall-набор P0; MEGA v6/v5.
C. **Динамика**: targeted16 (dev-сигнал) → tuning set → **чистый holdout (eval-only, не тюнить)**.
D. **Полный v8 (300+)** — в конце, отдельным длинным прогоном, с resume и полными транскриптами.
E. **Сравнение «до/после»** на holdout: записать baseline старого билда (если нет — зафиксировать сейчас),
   сравнить с новым.

## 6. Цикл архитектурных правок ВО ВРЕМЯ тестов

Гонять → классифицировать каждый провал по правилу решения (раздел 2) → внести АРХИТЕКТУРНУЮ правку →
перегнать → повторять, пока не выполнен acceptance ИЛИ пока не упрёшься в решение Дмитрия/РОП (например,
нужен новый ФАКТ или бизнес-правило — тогда стоп и эскалация, не выдумывать).

Запрещено в этом цикле: «починить» провал шаблоном-подменой, подстрочной веткой или строковым assert.
Если кажется, что иначе никак — это и есть сигнал, что нужен общий механизм (контракт/факты/few-shot/
классификатор), а не ветка.

Каждую итерацию фиксировать: что было, какой КЛАСС причины, какая АРХИТЕКТУРНАЯ правка, дельта на holdout,
изменение числа хардкод-веток.

## 7. Метрики и acceptance

Обязательные нули/100%:
- hard-gate (brand/fabrication/p0_mishandled/revealed_ai) = 0;
- recall реального P0 на recall-наборе = 100%;
- ложный P0 на benign-наборе = 0;
- утечка служебных маркеров (`kb_release` и пр.) = 0.

Качество (на ЧИСТОМ holdout, не на targeted16):
- `ignored_question` и `templated_opening` — значимое снижение vs baseline;
- `over_handoff` — только реальные случаи (live-status/менеджер);
- тон — рост vs baseline;
- «менеджер отправил бы без правок, %» — рост vs baseline (главный сигнал человечности);
- доля чистых `PASS` (не только PASS_WITH_NOTES) — растёт.

Анти-whack-a-mole:
- число `_*_safe_template`-подмен на зелёных темах и подстрочных topic-веток — СНИЗИЛОСЬ;
- новые тесты — property-типа (string-freeze только для безопасности/recall).

Если тон/человечность не выросли при выполнении остального — это сигнал, что шаблон-подмена ещё где-то
перебивает генерацию (искать в пайплайне), а не повод вернуть шаблоны.

## 8. Риски и компенсации

- Снос шаблонов ломает факты/бренд → BLOCK-гарды fabrication/brand/P0 остаются; recall-набор рядом;
  сносить тему за темой под флагом, мерить на holdout, откат при регрессе.
- LLM-rewrite/классификатор медленнее → кешировать, мерить p50/p95, держать одним шагом, fail-closed.
- Подгонка под holdout → holdout eval-only; при «выучивании» пересобрать у Claude.
- Контракт-объект усложнит код → он ЗАМЕНЯЕТ десятки веток, а не добавляется к ним; число веток должно падать.

## Резюме
Это переход от бота-шаблонизатора к боту-консультанту с safety-фильтром: один общий механизм (контракт
ответа → генерация по фактам и тёплым примерам → один rewrite → жёсткий финальный safety-слой) вместо
десятков веток. Главный предохранитель против скатывания в частные фиксы — раздел 2 и метрика числа
веток: если она растёт, правка неверная. Честный вывод о росте качества — только по чистому holdout и
доле «отправил бы без правок».
