# ТЗ — Роадмап качества бота: МОДУЛЬНЫЙ, КОНВЕЙЕРНЫЙ. Для Кодекса (ревью целиком → реализация блок за блоком). 2026-06-04.

Автор: Клод #1. Цель Дмитрия: бот аккуратно продаёт, проактивен (держит инициативу), максимально полезен (реально
помогает по любому вопросу). Опора: разбор 412 диалогов — бот уклоняется к менеджеру в ~33% ходов, и в 87% из них ПРИ
доступных фактах. Главный рычаг качества — перестать уходить, когда можно ответить, не теряя безопасность.

Формат особый (по решению Дмитрия): это НЕ монолит «всё сразу», а НЕЗАВИСИМЫЕ блоки под флагами, реализуемые
КОНВЕЙЕРОМ. Каждый блок — свой флаг (default OFF), свой коммит, своя приёмка. Кодекс ревьюит весь роадмап, потом
делает блок за блоком, НЕ дожидаясь ревью предыдущего (флаги OFF изолируют риск).

## ПРОТОКОЛ РАЗРАБОТКИ (конвейер — читать первым)
1. **Кодекс ревьюит весь роадмап** (Фаза 0): согласен с направлением/блоками/флагами? где иначе и ПОЧЕМУ; порядок;
   риски; вердикт. Это design-first по всему документу разом.
2. После согласования — реализация ПО БЛОКАМ. Каждый блок = отдельный коммит, флаг default OFF.
3. **Конвейер:** закончил блок N → отдал результат → СРАЗУ идёт в блок N+1 (Дмитрий пишет «следующий блок»). НЕ ждёт
   моего ревью блока N.
4. **Моё ревью асинхронно:** я регрейжу блок N по сырью (песочница + M1, когда блок готов к замеру). Правки по блоку N
   собираю в короткую заметку.
5. **Правки батчатся:** фиксы блока N Кодекс внедряет ПОСЛЕ блока N+1 (не прерывая конвейер), потом дальше.
6. **Безопасность конвейера:** НИЧЕГО не включается (флаг ON + приёмка) пока блок не отревьюен по сырью и не принят.
   Непринятый блок лежит в дереве с флагом OFF = ничего не делает. Поэтому стримить вперёд безопасно.
7. **Два стана параллельно:** пока M1 гоняет приёмку готового блока, главный Кодекс реализует следующий и гоняет свои
   прогоны у себя. Грузим оба.
8. **СТОП-правило (правка Кодекса):** конвейер вперёд можно только если флаг блока реально OFF и дефолт не изменился.
   Если моё ревью блока N находит SAFETY-регрессию (бренд, P0, выдумка, leak, faithfulness, не-downgrade-only гейт) —
   конвейер СТОП, чинить СРАЗУ, НЕ «после N+1». Батчатся только КАЧЕСТВЕННЫЕ правки, не безопасностные.
9. **Общий invariant-suite:** каждый новый флаг ПО ОТДЕЛЬНОСТИ + ВСЕ новые флаги ON вместе обязаны держать
   бренд/P0/выдумки/leak/identity/pressure. Гонять на каждом коммите блока.
10. **Merge-база (правка Кодекса):** `d2556018` (A2.1) и `aae2e3b0` (Level A линия) расходятся от `12d4e7ef`. Чисто:
   создать roadmap-ветку от `aae2e3b0`; Блоки 1-4 делать там. ПЕРЕД Блоком 5 — отдельным коммитом cherry-pick/merge
   `d2556018` (A2.1), полный pytest, потом A2.2/A2.3.

## ГЛОБАЛЬНЫЕ ИНВАРИАНТЫ БЕЗОПАСНОСТИ (держатся при ЛЮБОЙ комбинации флагов — не ослаблять)
- **Бренд-разделение:** в одном ответе никогда два бренда; cross-brand → каноничная фраза. active_brand из канала.
- **P0 → менеджер:** возврат-претензия, жалоба, спор оплаты, юр.угроза, гарантия результата. P0 перебивает всё.
- **Нет выдумок:** продуктовые числа только из факта (число-гейт v5); НЕчисловые выдумки («места есть», «рассрочка
  доступна») ловит semantic faithfulness — он ВСЕГДА ВКЛ.
- **Нет утечек** внутренних/мета/лесов; identity policy C; нет давления/обещания результата.
- **Каждый флаг default OFF; флаг OFF = текущее поведение** (контроль). Каждый блок несёт NEG с отрицательным
  контролем: блок НЕ ослабляет защиту от выдумок/бренд-смесей/P0-пропусков.
- Выходной гейт авторитетен и только ПОНИЖАЕТ маршрут (downgrade-only) — как сейчас.

## РЕВЬЮ КОДЕКСА (PASS_WITH_NOTES) — ПРИНЯТЫЕ ПРАВКИ ДИЗАЙНА (обязательны, переопределяют блоки ниже)
1. **Блок 1 разбить на 1a + 1b.** `1a partial facts` — реалистичен (есть `retrieval.facts/.missing`, `subquestions`,

2. **Partial-yield — ОБЩИЙ helper, не одно место.** Уход рождается в многих точках: P0, refund, estimate-fail,
   `empty_facts_no_fabrication`, `contract_manager_only`, `no_draft_fn`, `draft_error`, `semantic_check_unavailable`,
   `hard_verification_failed`, coverage-fail. Helper вставляется в несколько точек, НЕ трогает P0/refund/high-risk.
3. **Блок 2 — калибровка, не новая память.** Память уже в `build_understanding_prompt` +
   `_augment_contract_with_memory_topic`. Риск = слабая/устаревшая/не-enforced. NEG: явная смена темы, camp ≠ обычный
   курс, active_brand только из канала, P0 не маскируется эллипсисом.
4. **Блок 3 — P0 жёстче:** P0/high-risk в ЛЮБОЙ части составного вопроса → ВЕСЬ turn `manager_only` (НЕ «по части»).
5. **Блок 4 — ОДИН владелец next-step policy,** без дубля с A2/X2.
6. **Блок 6 — через существующий `TELEGRAM_DRAFT_X2_REWRITE` (X2),** расширить playbook/режим; НЕ новый флаг. X2 уже
   downgrade-safe (не трогает manager_only/P0/identity), после него `apply_authoritative_output_gate`.

## БЛОКИ (порядок = по эффекту; каждый независим)

### Блок 1 — Частичная польза вместо полного ухода `TELEGRAM_Q_PARTIAL_YIELD`  [цель: ПОЛЕЗНОСТЬ — главный рычаг]
Проблема: бот уходит целиком, когда покрыта НЕ вся потребность (дословное покрытие `_claim_supported_by_facts`),
хотя часть фактов есть (87% уклонений — при фактах).
Подход: разложить нужные факты вопроса на ГРУНТОВАННЫЕ vs НЕДОСТАЮЩИЕ. Если грунтованное непусто → ответить из него +
честно пометить недостающее («по остальному уточнит менеджер») + один следующий шаг. Полный хендофф — только если НЕ
грунтовано НИЧЕГО или P0.
Полуфабрикат (в месте рождения хендоффа):
```python
def partial_yield(required_keys, resolved_facts, has_p0):
    if has_p0: return "manager_full"
    grounded = [k for k in required_keys if k in resolved_facts]
    missing  = [k for k in required_keys if k not in resolved_facts]
    if not grounded: return "manager_full"            # нечего сказать → хендофф (контроль)
    return ("answer_grounded_plus_defer", grounded, missing)  # ответить часть + честно про missing + шаг
```
NEG: P0 → полный менеджер; для `missing` бот НЕ выдумывает (число-гейт/faithfulness держат); бренд цел; ничего не
грунтовано → хендофф. Приёмка: over-handoff на «при фактах» падает; безопасность 0.
**Под-кейс травел/оценка (важно):** три отдельных захода на «оценку дороги» провалились ровно потому, что бот уходит к
менеджеру РАНЬШЕ, чем что-либо сочиняет (route=manager_only до композиции). Это тот же корень. Здесь, в Блоке 1, при
фиксе over-handoff проверить, что бот СОЧИНЯЕТ ответ (в т.ч. бытовую оценку с маркером «ориентировочно…») вместо
преждевременного ухода. Тогда дремлющий число-гейт v5 (`TELEGRAM_A_FREE_NUMBER_GATE`, оставлен OFF) корректно пропустит
общее число и заблокирует продуктовое. Эстимейт-флаги (`TELEGRAM_A_ESTIMATE_MODE`/`FREE_NUMBER_GATE`) держим OFF —
травел решаем структурно тут, не отдельным патчем. Контроль: `unpk_travel_lobnya` t2 («ориентировочно на электричке?»)
даёт оценку с маркером, не уход.

### Блок 2 — Память нити / эллипсис `TELEGRAM_Q_THREAD_MEMORY`  [цель: ИНИЦИАТИВА + полезность]
Проблема: память диалога есть, но слой роняет её на эллипсисе («а онлайн?», «а для 9 класса?») и переспрашивает.
Подход: ПЕРЕД retrieve_facts — если реплика эллиптическая (короткая, без новой темы, отсылает к прошлому) → дотянуть
из dialogue_memory (последняя тема/предмет/класс/формат) → потом ретрив. Не переспрашивать известные слоты
(`do_not_reask_slots` уже в контракте — включить enforce).
Полуфабрикат:
```python
def fill_ellipsis(message, dialogue_memory):
    if _is_followup_ellipsis(message):                # короткая, опирается на контекст, нет новой темы
        return merge_known_slots(message, dialogue_memory)  # подставить topic/subject/grade/format из памяти
    return message
```
NEG: не тянуть устаревший контекст через смену темы; не подставить чужой бренд; P0 не маскируется. Приёмка: меньше
переспросов, нить держится (регрейд по сырью на многоходовых).

### Блок 3 — Составные вопросы `TELEGRAM_Q_COMPOSITE`  [цель: ПОЛЕЗНОСТЬ]
Проблема: клиент спрашивает 2-3 за раз, бот отвечает на одно, хвост роняет.
Подход: разбить на подвопросы (понимание уже размечает intent) → по каждому: грунтовано → ответить, негрунтовано →
честно отложить. Собрать многочастный ответ.
NEG: каждая часть проходит свои гейты (число/бренд/P0); **P0/high-risk в ЛЮБОЙ части → ВЕСЬ turn `manager_only`**
(правка Кодекса — не отвечать на соседние части, иначе бот будто игнорит претензию и продолжает продавать); не
выдумывать ради полноты. Приёмка: многочастные вопросы закрыты целиком, но P0-часть забирает весь ход.

### Блок 4 — «Ответ + следующий шаг» `TELEGRAM_Q_NEXT_STEP`  [цель: ИНИЦИАТИВА]
Проблема: бот выдаёт сухой факт без движения дальше.
Подход: композер завершает ответ ОДНИМ конкретным, ненавязчивым шагом (узнать класс / предложить пробное / связать с
менеджером). Координировать с A2 (не дублировать шаг).
NEG: без давления (гейт давления); ровно один шаг; P0/серьёзный контекст → без продающего шага. Приёмка: ответы
двигают диалог вперёд, тон не впаривающий.

### Блок 5 — Проактивная продажа A2.2/A2.3 `TELEGRAM_A_PROACTIVE` (флаг существует, A2.1 готов)  [цель: ПРОДАЖА]
Реализовать по существующему ТЗ `D1_audit_backlog/TZ_levelA2_proactive_selling_2026-06-03.md` (v3, 6 правок учтены):
A2.2 — handle_objection/qualify/fit_check строго через fact-availability; A2.3 — offer_enroll (последним).
NEG: fit_check фактовой рамкой (не диагноз); возражение без факта = обычный ответ без довеска (не форс-менеджер);
fake-enrollment/PII/анти-барраж/general_advice-страж. Приёмка: набор A2.1 (готов) + A2.2-сценарии.

### Блок 6 — Тон / playbook менеджеров `TELEGRAM_Q_TONE_PLAYBOOK` (или существующий humanity-флаг — Кодекс уточнит)  [цель: ПРОДАЖА/живость]
Проблема: черновики суховаты/шаблонны.
Подход: слой согрева (X2) донастроить на playbook лучших менеджеров (снятие риска, гашение тревоги, честная срочность,
чёткий шаг). СТРОГО downgrade-only: НИКОГДА не меняет числа/факты/маршрут/бренд; P0/серьёзное → без согрева, сухой
хендофф.
NEG: rewrite не трогает числа/факты/route/бренд; P0 остаётся сухим; meta-leak 0. Приёмка: тон теплее по сырью,
безопасность не изменилась.

## ПРОТОКОЛ ЗАМЕРА (честно про сигнал/шум)
- Прогоны ЧИСТО показывают СТРУКТУРНОЕ: «партиал-ответ появился? нить держится? да/нет» — видно сразу.
- Маргинальные дельты (тон, −5% уходов) НИЖЕ шума 1 прогона на 21 сценарии (±11% межпрогонной вариативности). Для них:
  менять ОДИН флаг за прогон (изолировать эффект) + регрейд по сырью; на решающих — больший набор.
- НЕ гонять много флагов в комбинациях ради «what works» — поймаем шум. Комбинации включаем ОСознанно, по одному
  принятому блоку.
- Два стана (M1 + мак Кодекса) — параллельные приёмки, `--parallel 4` каждый.
- Я регрейжу КАЖДЫЙ блок по сырью (не по агрегату судьи) перед включением флага.

## ОЧЕРЁДНОСТЬ (рекомендуемая; Кодекс может оспорить в Фазе 0)
1 (partial-yield) → 2 (память нити) → 3 (составные) → 4 (ответ+шаг) → 5 (A2.2/3) → 6 (тон). 1-3 = самый большой прирост
полезности; 4-5 = инициатива/продажа; 6 = живость. v5 (число-гейт) уже на M1 — не входит в этот роадмап.

## ЧТО НЕ ДЕЛАТЬ
Не ослаблять глобальные инварианты ни одним блоком; не включать флаг без приёмки по сырью; не вливать блоки одним
big-bang-коммитом (каждый — свой); faithfulness всегда ВКЛ; downgrade-only гейт; не git reset; P0-слои не трогать.
Каждая защитная правка — с контрольным отрицательным тестом.
