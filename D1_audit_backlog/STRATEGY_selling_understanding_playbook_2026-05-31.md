# Стратегия: обучение бота приёмам лучших менеджеров (продающее понимание). 2026-05-31.

Автор: Клод 1. Это стратегия СЛЕДУЮЩЕГО этапа — после стабилизации базы (выдумки 0, over-handoff низкий).
Цель: бот не просто отвечает верно, а ведёт к продаже как хороший менеджер — слышит намерение за
словами и применяет приём. Поднимает и пользу (конверсия), и тон (живость).

## 1. Принцип (на чём строим)

Формула, которую мы выстрадали за неделю: **модель ПОНИМАЕТ, детерминированный слой ВЫБИРАЕТ действие,
draft ИСПОЛНЯЕТ, безопасность поверх всего.** Продажа не отдаётся модели целиком (она бы давила и
выдумывала) и не кодируется regex (он не слышит подтекст). Модель распознаёт намерение → детерминированная
таблица «намерение → приём» выбирает ход → draft пишет живо → P0/бренд/факт-гарды важнее любого приёма.

## 2. Источник приёмов — наш playbook (уже есть)

Прочитаны 703 расшифровки звонков лучших менеджеров (Тропина, Козлова) — [[sales_playbook_from_calls]].
Их сильные приёмы уже выделены: снятие риска, гашение тревоги, честная срочность (без давления), чёткий
следующий шаг. Это НЕ выдуманные техники, а то, что реально работает на нашей аудитории. Стратегия —
перенести эти приёмы в бота, не изобретая новых.

### Этап 0 (подготовка): каталог приёмов из playbook
Из расшифровок собрать структурированный каталог: для каждого типа ситуации клиента — конкретный приём +
1-2 реальные формулировки менеджера (как образец тона, не дословно). Это делаю я (read-only анализ
звонков + WhatsApp/Telegram для реальных формулировок клиентов). Результат — таблица «ситуация → приём →
пример фразы». Дмитрий подтверждает, что приёмы соответствуют политике (не давление, не обещания).

## 3. Категории намерения (что бот должен слышать)

Пять сигналов «между строк», которые модель-понимание должна извлекать (расширение существующего
understanding-вызова, без нового вызова):

| Сигнал | Как звучит у клиента | Что за ним |
|---|---|---|
| Возражение по цене | «дорого», «подешевле?», «а скидки?» | цена пугает, нужна ценность/рассрочка |
| Сигнал ухода | «поищу у других», «спасибо, подумаю» | уходит, момент удержания |
| Тревога/недоверие | «вы точно нормальные?», «не обманете?» | страх риска, нужно снять |
| Невысказанная потребность | «у ребёнка тройка», «он стеснительный» | сигнал, не вопрос; предложить |
| Стадия готовности | тон, прямота вопроса | изучает / сравнивает / готов |

## 4. Маппинг «намерение → приём» (детерминированный, из playbook)

- Возражение по цене → не повторять цену сухо: дать ценность (результат/формат) + безопасный способ
  оплаты (рассрочка/Долями по бренду) + мягкий шаг. (приём «снятие финансового барьера»)
- Сигнал ухода → удержать одним конкретным шагом или снять барьер; не отпускать сухим хендоффом.
  (приём «честная срочность» — без давления: места/сроки как факт)
- Тревога → снять риск: пробное занятие, честно про лицензию/гарантии безопасности (НЕ обещать
  результат). (приём «гашение тревоги»)
- Невысказанная потребность → предложить релевантное, даже если прямо не спросили (в рамках фактов).
- Готовность=ready → чёткий шаг к записи/оплате. (приём «чёткий следующий шаг»)

## 5. Волны реализации (очерёдность по бизнес-ценности)

Каждая волна — отдельная правка с проверкой, один сигнал за раз (не смешивать — мерить вклад).

- **Волна 1: возражение по цене + сигнал ухода.** Самое денежное — прямо влияет на конверсию. Бот
  перестаёт «сливать» клиента на цене и уходе.
- **Волна 2: тревога/недоверие.** Снятие риска — поднимает доверие и тон.
- **Волна 3: невысказанная потребность + стадия готовности.** Тонкая настройка под клиента.

## 6. Метрики (как поймём, что работает)

Текущий судья меряет факты/безопасность/тон, но НЕ «продающесть». Нужны метрики:
- доля диалогов, где бот отработал возражение (дал ценность+оплату), а не повторил цену;
- доля удержаний при сигнале ухода (предложил шаг vs отпустил);
- рост тона (живость) — должен пойти к 75-80+;
- БЕЗ роста выдумок и без давления (NEG: возражение про возврат → менеджер; приём не обещает результат).
Понадобится набор сценариев с возражениями/уходом/тревогой (соберу из реальных WhatsApp/Telegram —
там это есть в живом виде).

## 7. Риски и границы (обязательно)

- Приём НЕ нарушает P0/бренд/факт: возражение про возврат/спор → P0 менеджер, не «отработка».
- «Снять риск» ≠ обещать результат/поступление/гарантию (запрещено политикой).
- «Честная срочность» = факт (места/сроки), НЕ искусственное давление.
- Не давить: цель — помочь решить, не впарить. Тон playbook — мягкий, это сохранить.
- Каждый приём — с негативным контролем, чтобы продажа не пробила безопасность.

## 8. Процесс (как весь цикл)

Этап 0 (каталог приёмов) → волна 1 (понимание возражения/ухода + маппинг + draft) → проверка в
песочнике → прогон на сценариях с возражениями → регрейд по сырью → волна 2 → и т.д. Я разбираю и пишу
ТЗ с полуфабрикатами, Кодекс реализует, я проверяю, прогон, регрейд. Дмитрий подтверждает приёмы и
политику на этапе 0.

## 9. Зависимости (сделать ДО старта)

- База стабильна: выдумки ≈0, over-handoff низкий (идёт сейчас — финальный замер 212).
- Реальные формулировки клиентов извлечены (WhatsApp/Telegram) — для каталога и сценариев.
- understanding-калибровка от over-handoff и продающие поля — оба расширяют ОДИН understanding-вызов,
  делать согласованно.
- НЕ начинать продающее, пока база пляшет — иначе не отделим эффект приёма от нестабильности (урок недели).
