## ФАЗА 2 — три правки

### G1 — гард оценки пропускает ОБЩУЮ оценку с маркером
Привести `estimate_guard` к логике free-number-gate: ОБЩАЯ оценка (минуты/км/«примерно столько ехать») + маркер «ориентировочно» → ПРОПУСК (не ронять в хендофф). Грунтовка для общей оценки НЕ требуется.
- **NEG-контроль (обязателен):** ПРОДУКТОВОЕ число без грунтовки (₽ / % / дата / расписание / длительность урока) гард по-прежнему БЛОКИРУЕТ даже с маркером. Тест: «урок ~90 минут» (длительность урока = продукт, не грунтовано) → блок; «дорога ~40 минут» (общее) → пропуск с маркером.

### G2 — partial-yield выдаёт КАЖДУЮ грунтованную под-часть составного
При разборе составного: выдать все грунтованные под-части (курс есть / форматы / цены), даже если ДРУГАЯ часть camp/негрунт; отложить только негрунтованную. Не уходить целиком из-за одной несработавшей части.
- **NEG-контроли (обязательны):**
  1. P0/возврат/high-risk в ЛЮБОЙ части → ВЕСЬ ход менеджеру (не выдавать вокруг P0). Это контроль Блока 3, не ослаблять.
  2. cross-brand-часть → каноничная фраза «Это отдельные организации…», без сравнения условий.
  3. Одиночный вопрос → partial-yield НЕ мисфайрит (отвечает как обычно).

### G3 — срез сырого подвопроса в тексте хендоффа
Не подставлять сырой подвопрос в текст хендоффа. Заменить на нейтральную формулировку («менеджер уточнит детали и вернётся») или ЧИСТУЮ метку темы, не сырой текст вопроса. Расширить linter meta_leak, чтобы ловил этот паттерн.
- **NEG-контроль:** осмысленный хендофф не превращать в бесполезно-общий там, где тема и так чистая; контрольный тест: паттерн «уточнит именно про <сырой вопрос>» срезается, обычный хендофф остаётся.
