# Техническое задание ТЗ-110 (v4) — ADR-002, выбор фактов: ЖИВОЙ режим «модель ведёт» под флагом + тень-диагностика + контракт верификации

- **Дата:** 2026-06-14. Заказчик: Дмитрий. Постановщик: Claude #1. Картирование по сырью + круг аудитора + сверка с Кодексом. Стратегия: разработка под флагами → один большой ночной замер на M1 с качественным логированием для трассировки.
- **Принцип (главное):** понимание «какие факты нужны клиенту» переносим на МОДЕЛЬ; детерминизм остаётся ТОЛЬКО на ВЕРИФИКАЦИЮ её вывода (факт существует, бренд, скоуп, числа из фактов, P0-пол). Это «умно + безопасно», НЕ «безопасно, но тупо». Проверка ≠ понимание.

## Картина по сырью
LLM-ретривер в `pilot_gold_v1` уже выбирает факты из кандидатов и восстанавливает эллипсис (`src/mango_mvp/channels/subscription_llm_parts/direct_path.py:711`); кандидаты уже фильтрованы по бренду+свежести+client_safe (`direct_path.py:491`). Кейвордом осталась деривация «какие факты нужны» (`conversation_intent_plan.py:571`, `telegram_pilot_context_builder.py:171`), которая подсказкой подаётся ретриверу (`direct_path.py:694`) и на перефразах тянет не туда. При этом `required_fact_keys` — сквозной ключ (scope-guard `policy_routing.py:4133`, autonomy, missing_facts, память).

## Режимы (два флага, оба default OFF; в проде оба OFF = паритет)

### Флаг A — ТЕНЬ-диагностика `TELEGRAM_RETRIEVER_NEED_SHADOW`
В том же вызове ретривера (`direct_path.py:736/:681`, без 3-го LLM-вызова) модель ДОПОЛНИТЕЛЬНО возвращает структурированную декларацию нужных фактов (тема / тип факта: цена/расписание/документы/формат/адрес/… / бренд / класс / предмет / формат / продукт / почему нужен / обязателен-или-полезен). При только-A: ответ клиенту НЕ меняется, декларация лишь логируется и сравнивается с кейворд-`required_fact_keys` и с реально выбранными id.

### Флаг B — ЖИВОЙ режим «модель ведёт выбор» `TELEGRAM_RETRIEVER_MODEL_DRIVEN`
При ON: декларация модели (из A) становится ДРАЙВЕРОМ выбора фактов в ретривере — кейворд-`required_fact_keys` НЕ подаётся в `plan_json` (`:694`) и не даёт категорийный байас; модель ведёт «что нужно ответить» сама (с эллипсисом). Выбранные/нужные факты затем проходят детерминированный КОНТРАКТ верификации (ниже). Это и есть перенос понимания на модель в живом пути.
- **Строго ограничить шов:** флаг B меняет ТОЛЬКО драйвер выбора в direct-path-ретривере. Сквозной `required_fact_keys` для scope-guard / autonomy / missing_facts / памяти ОСТАЁТСЯ кейвордным (не трогаем) — иначе ломаем гейты; их перевод на модель — отдельный последующий срез. Так живой эффект на ОТВЕТ меряется, а гейты под защитой.
- Кейворд-фолбэк (`direct_path.py:603`) при таймауте/невалидном JSON/пустом выборе — сохранить (устойчивость).
- При B ON декларация модели логируется так же (тень — диагностическое поле внутри живого режима).

## Контракт верификации выхода (детерминированный, ОБЯЗАТЕЛЕН при B; не ослаблять)
Каждый выбранный/нужный моделью факт перед попаданием в черновик:
- существует (валидация id `:788`); бренд совпал и client_safe и не просрочен (уже candidate-time `:491` — не дублировать); скоуп не конфликтует (`_direct_path_fact_conflicts_slots:391`, оставить МЯГКИМ: конфликт → adjacent, не reject, но логировать `scope_demoted_ids`).
- Числа в ответе — из выбранных фактов: это несёт ВЫХОДНОЙ ГЕЙТ (`post_layers.py:3194/3486`), не дублировать, не ослаблять. P0-пол не обходить.
Контракт = сборка этих проверок в один трассируемый шаг с логом, НЕ замена выходного гейта.

## Качественное логирование (под ночной замер — обязательно)
На каждый ход в `dynamic_summary.json` машиночитаемо (json-поля для grep/diff), чтобы я мог трассировать ошибку до места возникновения:
- кейворд-`required_fact_keys`; модель-декларация нужд (A); счётчик кандидатов; выбранные `exact_ids/adjacent_ids`; `scope_demoted_ids`; отброшенные id; `llm_retrieve.used` / `fallback_reason`; какой режим был активен (A/B/оба/выкл);
- маршрут, P0-сигнал (модель/пол), бренд/скоуп-вердикты гейта.
Стандарт логирования распространить и на остальные функции ночного прогона (model-P0 reasons, action_decision, child-identity resolution).

## Замер (в общем ночном прогоне M1)
- На наборе перефраз (кейсы, где кейворд-деривация промахивается) + золотом наборе.
- Сравнить: OFF (база) vs B ON (модель ведёт). Метрики: доля верных фактов в ответе (мой регрейд по сырью), over-handoff, hard-gate fabrication (ДОЛЖЕН остаться 0), бренд-чистота; диагностика расхождений (A) — где модель просила факт, которого нет в кандидатах (сигнал, что детерминизм сузил неверно).
- M1, `--parallel 4`, 1 прогон, снимок v6.5; мой регрейд + трассировка → архитектурный вывод.

## Дисциплина / приёмка
- Оба флага default OFF, не в `pilot_gold_v1` до замера.
- **OFF = расширенный паритет:** выбор фактов И scope-guard/autonomy/missing/память байт-в-байт как сейчас (NEG-тест). При только-A ON: `draft_text` идентичен OFF (тень не меняет ответ) — тест.
- При B ON: scope-guard/autonomy/missing/память НЕ изменились (они на кейворд-ключах) — тест; меняется только выбор фактов ретривером.
- Ветка от `main`; полный `pytest` зелёный; отчёт в `_done`; мой регрейд по сырью перед включением в боевой профиль.

## Чего НЕ делаем (границы)
- НЕ трогаем сквозной `required_fact_keys` вне ретривера (scope-guard/autonomy/missing/память) — это отдельный последующий срез.
- НЕ ослабляем выходной гейт и P0-пол; НЕ сужаем кандидатов (follow-up при росте базы); НЕ ужесточаем scope в reject; 3-й LLM-вызов не вводим (декларация — в существующем вызове ретривера).

## Риски (из сверки)
1. «Модель видит только то, что код уже отфильтровал» — если детерминизм сузил кандидатов неверно ДО модели, декларация (A) это покажет (модель просит факт не из списка). Ради этого сигнала и тень.
2. Кейворд-фолбэк может вернуть «слова пересилили смысл» — логируем `fallback_reason`, видим частоту.
3. Без строгой проверки модель просит широко → контракт верификации (детерминированный) обязателен при B.
