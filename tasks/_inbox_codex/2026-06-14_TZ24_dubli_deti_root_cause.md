# ТЗ-24: корень дубль-детей в профилях (расследование + фикс за флагом)

Дата: 2026-06-14. Постановка: Claude (архитектор). Повод: C1 из ТЗ-22 реализован верно, но на реальных данных сливает 0 слотов — потому что правило требует совпадения И класса, И предмета у безымянных слотов, а фантомные слоты несут лишь один признак. C1 безопасен, но проблему не решает. Нужен другой подход — у корня.

## Диагноз (из аудита трека, проверить заново по коду)

Безымянный детский слот рождается с НОВЫМ `child_key` (хеш от класса/числа, `stable_child_key`/`explicit_child_key` в `builder.py`) на каждом звонке без названного имени. Такие слоты не сливаются: по имени — нечем (имени нет), по признакам (C1) — обычно нет обоих признаков сразу. Итог: у семьи копятся 13–19 «детей», хотя ребёнок один (≈747 профилей с хеш-слотом от одного значения по прежней инвентаризации).

## Блок А. Расследование (read-only, сначала только цифры)

1. По актуальной базе профилей (`tz21_profiles_after_tail_*` или новее) посчитать: сколько безымянных детских слотов; распределение по числу несомых признаков (только класс / только предмет / оба / ничего); сколько профилей имеют 2+ безымянных слота; сколько — смесь «1 именованный + N безымянных».
2. Выяснить, ОТКУДА берётся новый `child_key` для безымянного слота (точка создания в `builder.py`: `stable_child_key`, `explicit_child_key`, вызовы). Зафиксировать файл+строку.
3. Выход: короткая таблица + вывод «где именно плодятся слоты».

## Блок Б. Дизайн фикса (документ, выбор из вариантов)

Предложить и сравнить (с рисками ложного слияния):
- **Вариант 1 — профилактика при создании:** безымянный слот с признаком (класс/предмет) НЕ минтит новый `child_key`, а прикрепляется к существующему совместимому слоту семьи (приоритет — к именованному, если признак не противоречит). Риск: приклеить к не тому ребёнку.
- **Вариант 2 — пост-слияние по ОДНОМУ признаку** для безымянных, с предохранителем (например только если в профиле ≤1 именованного ребёнка, или безымянные не противоречат друг другу). Риск выше, чем у C1.
- **Вариант 3 — не считать безымянные слоты отдельными детьми:** рендерить их как «неатрибутированные касания», не как отдельных детей в карточке. Минимальный риск, но не «склейка», а сокрытие.
Рекомендация архитектора по умолчанию: начать с Варианта 3 (безопасно для карточки) + замер, Вариант 1 как следующий шаг под флагом.

## Блок В. Реализация (после согласования варианта Дмитрием/архитектором)
