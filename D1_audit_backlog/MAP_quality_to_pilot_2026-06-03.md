# Карта качества до пилота. После завершения чистой архитектуры. 2026-06-03.

Автор: Клод #1. Опора на факт: финальный прогон (over-handoff 0, hard_gate 0, бренд/мета 0, identity C, camp
чист). Архитектура взята; это карта того, что отделяет бота от пилота-черновиков, по приоритету.

## Где мы (факт)
Чистая архитектура (планнер→правила→композер→гейт), монолит снесён (после Фазы 2). Безопасность держится на
каждом шаге. over-handoff 0, ответы автономные и из факта. То есть «безопасно и по делу» — есть. Не хватает —
«живо и продающе».

## Рычаги до пилота (по приоритету)

### A. ГЛАВНЫЙ — продающее понимание (модель ведёт как лучший менеджер)
Сейчас бот отвечает фактом корректно, но НЕ продаёт. Из 703 расшифровок Тропиной/Козловой (playbook) топ-приёмы:
- снятие риска («можно сначала пробное / вернём остаток, ничего страшного»);
- гашение тревоги (спокойный тон, нормализация сомнений);
- честная срочность (реальный дефицит мест, без выдумки);
- чёткий следующий шаг (конкретное «что дальше», не «менеджер свяжется»).
**Как вложить:** в композер-промпт правил (модель ПИШЕТ ответ с этими приёмами из данных правила), а выходной
гейт страхует от выдумок/обещаний. Это и есть смысл чистой архитектуры — модель пишет живо, детерминир. слой
верифицирует. Замер: доля ответов с продающим шагом, тон по сырью, не балл судьи.

### B. Точечные остатки качества (мельче, быстро)
- price без факта («сколько онлайн физика 9?») → честный «менеджер уточнит» с route=manager, не дефер-текст при
  answer_self;
- покрытие составных вопросов (правило→None на «два предмета», «примерно?») — расширить подвопросы;
- тон-полировка f-строк там, где ещё суховато.

### C. Покрытие тем/фактов
- ревизия, где правило часто возвращает None (нет факта) → дозаполнить KB или расширить scope;
- расписание: точные дни групп (часть ещё «менеджер сверит») — по мере публикации в Tallanto.

## Измеримый порог готовности к пилоту (чек-лист приёмки)
Пилот-черновики разрешены, когда на БОЛЬШОМ замере (200-300 диалогов, чистая архитектура):
- БЕЗОПАСНОСТЬ = 0: бренд-смешение 0, P0-пропуск 0, ПДн-утечка 0, выдумка числа/факта 0, identity-нарушение 0;
- over-handoff < порога (ориентир ≤5-8% на отвечаемых; сейчас 0 на узком наборе);
- продающий шаг присутствует в ≥X% автономных ответов (порог согласовать);
- покрытие: правило→None < порога на реальных вопросах;
- тон по СЫРЬЮ (выборка читается живой), не по баллу судьи (он ненадёжен).
Всё мерить по сырью + регрейд, «зелёные тесты ≠ готовность».

## Финальная приёмка
Один БОЛЬШОЙ замер (200-300 диалогов, набор с реальными вопросами + адверсариальные + P0/бренд/identity) на
финальной чистой архитектуре. Это «достаточное качество» как факт, а не ощущение. Делать ОДИН раз, когда A+B
сделаны (не гонять впрок).

## Порядок
1. Фаза 2 (снос) — идёт. 2. Точечные остатки B (быстро, дёшево). 3. Продающее понимание A (главный рычаг,
композер-промпт + NEG-гейт). 4. Покрытие C (по мере). 5. Финальная приёмка (большой замер) → решение о пилоте.

## Что НЕ делаем
Не гоняем большие замеры впрок (только финальный). Не возвращаем X2 как основной тон. Не открываем пилот, пока
порог не взят по сырью. Параллельно зреет линия customer history (для фазы текущих клиентов) — отдельно.
