# Telegram Pilot And Contextual Catalog Strategy 2026-05-17

Статус: рабочее ТЗ для Codex.

Цель документа: зафиксировать правильный путь после разбора ошибок классификации вопросов. Главный вывод: сейчас нужно строить не отдельный классификатор "фраза -> тема", а рабочий контекстный помощник менеджера в Telegram. Исторические 9 969 вопросов тоже нужно анализировать с контекстом, но только после проверки качества сборки контекста на малой выборке.

## 1. Главная позиция

Мы калибровали слишком бедную задачу. Реальный бот не должен угадывать тему по одной вырванной фразе. Он должен видеть:

- текущее сообщение клиента;
- последние сообщения диалога;
- историю клиента;
- AMO-сделку, если найдена;
- Tallanto-контекст, если найден;
- правило РОПа;
- актуальные факты по ценам, расписанию, документам и оплате;
- ограничения безопасности.

Поэтому качество Telegram-пилота нельзя оценивать только по старой метрике `macro-F1=0.6428` на одиночных фразах. Эта метрика полезна как технический ориентир, но она не является главным показателем готовности пилота.

Главный приоритет: запустить управляемый Telegram-пилот для Насти в режиме черновиков, без автоответов клиентам.

Второй приоритет: подготовить контекстный анализ 9 969 исторических вопросов, чтобы получить бизнес-аналитику и материал для будущего автономного бота.

## 2. Что уже известно

### 2.1. Разбор 24 ошибок

Разбор 24 ошибок на размеченной выборке показал:

- часть ошибок вызвана обрывками без контекста;
- часть строк содержит несколько тем сразу;
- часть разметки спорная;
- часть ошибок связана с близкими денежными темами;
- самые опасные темы должны уходить менеджеру, а не решаться автономно.

Вывод: попытка довести одиночную классификацию до `0.85` сейчас не является лучшим использованием времени.

### 2.2. Зачем нужны 9 969 вопросов

9 969 исторических вопросов нужны не как блокер пилота, а как бизнес-аналитика и база будущего развития:

- понять, что клиенты спрашивают чаще всего;
- найти редкие, но важные темы;
- подготовить материал для обучения менеджеров;
- построить FAQ и контент для сайта;
- увидеть проблемы процессов;
- понять, какие темы будущий бот должен закрывать в первую очередь.

Но делать это без контекста неправильно. Нужно анализировать не вырванную фразу, а ситуацию клиента.

### 2.3. Почему не надо сразу переанализировать все звонки

Для звонков лучше сначала использовать локальный фрагмент транскрипта вокруг вопроса и старую сводку как запасной источник. Полный повторный анализ всех звонков сейчас избыточен.

Правильный порядок:

1. Проверить, можно ли для вопросов из звонков найти локальный фрагмент.
2. Прогнать малую выборку с контекстом.
3. Если качество именно звонковых вопросов ниже Telegram/email, тогда точечно переанализировать проблемные звонки.
4. Полный re-analyze исторических звонков оставить на отдельный большой этап.

## 3. Жесткие границы

В рамках этого ТЗ нельзя:

- отправлять сообщения клиентам автоматически;
- писать в AMO/CRM;
- писать в Tallanto;
- менять `stable_runtime` DB/audio/transcripts;
- запускать ASR;
- запускать Resolve+Analyze по реальным данным;
- запускать тяжелые batch/start/run-ui скрипты;
- удалять файлы;

- коммитить токены, сырые переписки, персональные данные клиентов;
- открывать поток с сайта до проверки пилота на Насте.

Все новые артефакты проверки писать только в безопасные локальные папки:

- `.codex_local/...`;

`stable_runtime` в этом блоке только читать, если это нужно для проверки указателей и источников.

## 5. Целевой результат

### 5.1. Для Telegram-пилота

Настя получает в служебном Telegram-чате понятный черновик:

- кто написал;
- что написал клиент;
- какую тему и риск увидела система;
- почему это можно или нельзя отдавать боту;
- черновик ответа;
- что менеджеру нужно проверить;
- какие факты отсутствуют;
- нужна ли задача follow-up;
- что можно предложить записать в CRM, но без автоматической записи.

Клиенту на первом этапе ничего не отправляется автоматически.

### 5.2. Для исторических 9 969 вопросов

После стабилизации пилота нужно получить контекстную таблицу:

- вопрос;
- источник;
- контекст;
- тема;
- дополнительные темы;
- главная боль клиента;
- был ли вопрос решен;
- риск;
- качество найденного контекста;
- уверенность модели;
- причины ручной проверки.

Эта таблица используется для бизнес-аналитики, FAQ, обучения менеджеров и будущей автономии бота.

## 6. Этапы работ

## Этап 0. Зафиксировать границы и состояние

Цель: не смешать эту работу с параллельными изменениями по звонкам и runtime.

Действия:

1. Проверить `git status --short`.
2. Зафиксировать, какие файлы уже изменены другими диалогами.
3. Не трогать чужие изменения.
4. Прочитать:
   - `docs/CURRENT_STATE.md`;
   - `docs/DECISIONS_LOG.md`;
   - `docs/TELEGRAM_MANAGER_DRAFT_PILOT_TZ_2026-05-16.md`;
   - этот файл;
   - актуальные audit packs по Telegram/question catalog.
5. Создать рабочий audit pack для этого блока.

Выход:

- краткий `implementation_notes.md` в audit pack;
- список файлов, которые планируется менять;
- подтверждение, что runtime не меняется.

## Этап 1. Сверить текущий Telegram-пилот

Цель: понять, что уже реализовано и чего реально не хватает.

Проверить:

1. Long polling:
   - включается только по флагу;
   - имеет kill switch;
   - не отправляет клиенту сообщения;
   - умеет debounce 5-10 секунд;
   - не обрабатывает duplicate update повторно.

2. Служебный чат менеджера:
   - есть ли сохранение `manager_chat_id`;
   - есть ли allow-list менеджеров;
   - что видит Настя;
   - нет ли кнопки прямой отправки клиенту.

3. LLM-черновик:
   - используется ли `codex exec`;
   - есть ли безопасный fallback;
   - есть ли защита от плохого JSON;
   - есть ли таймаут;
   - не нужен ли OpenAI API-ключ.

4. Prompt:
   - клиентское сообщение обернуто в явные маркеры;
   - есть запрет выполнять инструкции из текста клиента;
   - есть запрет раскрывать "я бот", "ИИ", "GPT", "Claude", "Codex";
   - есть безопасный шаблон для расписания;
   - есть follow-up, если используется шаблон расписания.

5. Хранилище:
   - входящие сообщения сохраняются локально;
   - черновики сохраняются локально;
   - есть idempotency;
   - нет записи в `stable_runtime`.

Выход:

- таблица "готово / частично / не готово";
- список точечных правок.

## Этап 2. Ввести единый контракт контекста

Цель: бот должен получать не набор случайных строк, а понятный пакет контекста.

Нужно создать или уточнить единый объект `PilotContext`.

Поля:

- `current_message` - текущее сообщение клиента;
- `recent_messages` - последние 5-10 сообщений;
- `client_identity` - телефон, имя, Telegram id, если известно;
- `customer_summary` - краткая история клиента;
- `amo_context` - сделка, статус, последние задачи, только read-only;
- `tallanto_context` - ученик, группа, оплаты, только read-only;
- `timeline_context` - история общения, если безопасно найдена;
- `rop_policy` - правило РОПа по теме или предварительной теме;
- `facts_context` - цены, расписание, документы, ссылки;
- `risk_flags` - найденные риски;
- `context_quality` - насколько надежен контекст.

Качество контекста должно явно показывать:

- найден ли клиент;
- найден ли телефон;
- найден ли AMO-контекст;
- найден ли Tallanto-контекст;
- есть ли конфликт AMO/Tallanto;
- есть ли семейный телефон;
- есть ли несколько учеников;
- есть ли несколько сделок;
- устарели ли факты.

Acceptance criteria:

- если контекст не найден, система не падает;
- если несколько учеников, система не делает вид, что знает нужного ребенка;
- если нет свежего расписания, система не обещает конкретную группу;
- если AMO/Tallanto конфликтуют, маршрут становится `manager_only` или `draft_for_manager_with_warning`.

## Этап 3. Перевести LLM-ответ в один структурированный результат

Цель: один вызов модели должен возвращать не только черновик, но и понимание ситуации.

Обязательные поля ответа:

- `message_type`:
  - `question`;
  - `non_question`;
  - `context_update`;
  - `wait_for_more`;
  - `manager_only`;
- `broad_group` - крупная группа темы;
- `theme_id` - точная тема;
- `alternative_themes` - дополнительные возможные темы;
- `confidence_group`;
- `confidence_theme`;
- `risk_level`;
- `route`;
- `draft_text`;
- `manager_checklist`;
- `missing_facts`;
- `safety_flags`;
- `manager_followup_required`;
- `manager_followup_deadline`;
- `crm_recommendations`;
- `context_used`;
- `context_warnings`.

Высокорисковые темы:

- возврат денег;
- материнский капитал;
- налоговый вычет;
- юридические вопросы;
- жалобы;
- негативная обратная связь;
- статус оплаты;
- документы;
- скидки и обещания стоимости.

Правило:

- если `confidence_theme < 0.70`, маршрут не может быть автономным;
- если тема высокорисковая, маршрут только менеджеру или черновик без обещаний;
- если тема не утверждена РОПом, маршрут `manager_only`;
- если не хватает свежих фактов, черновик должен просить уточнение или передавать менеджеру.

## Этап 4. Улучшить prompt для Telegram-пилота

Цель: prompt должен отражать реальную задачу, а не старую задачу "одна фраза -> одна тема".

Что изменить:

1. Передавать последние сообщения диалога, а не только текущее сообщение.
2. Передавать краткую историю клиента.
3. Передавать AMO/Tallanto/timeline только в сжатом виде.
4. Передавать правило РОПа.
5. Передавать свежие факты или явно указывать, что фактов нет.
6. Требовать `message_type`.
7. Разрешить `alternative_themes`.
8. Запрещать опасные обещания.
9. Запрещать раскрытие, что ответ написал бот или ИИ.
10. Защититься от текста клиента вида "игнорируй инструкции".

Отдельное правило для расписания:

Если точного расписания нет, использовать мягкую формулировку:

> У нас много групп в каждом филиале, включая онлайн, поэтому мы уточним удобное Вам время в субботу или воскресенье и постараемся подобрать занятие именно тогда. Позже дополнительно свяжемся и уточним.

Если используется эта формулировка, обязательно:

- `manager_followup_required=true`;
- срок follow-up не позже 24 часов.

## Этап 5. Тесты безопасности и качества черновика

Минимальные тесты:

- prompt содержит контекст РОПа;
- prompt оборачивает клиентское сообщение в безопасные маркеры;
- prompt запрещает выполнять инструкции клиента;
- тема без утверждения РОПа дает `manager_only`;
- расписание без свежего факта дает безопасный шаблон;
- безопасный шаблон создает follow-up;
- ответ не раскрывает "я бот", "ИИ", "нейросеть", "GPT", "Claude", "Codex";
- high-risk темы не дают автономный маршрут;
- низкая уверенность дает `manager_only`;
- несколько сообщений клиента склеиваются в один смысловой запрос;
- плохой JSON от модели дает безопасный fallback;
- таймаут модели дает безопасный fallback;
- AMO/Tallanto read-only контекст не приводит к записи.

Безопасные команды:

```bash
PYTHONDONTWRITEBYTECODE=1 PYTHONPATH=src python3 -m pytest -q tests/test_draft_prompt_builder.py tests/test_subscription_llm_draft_provider.py
PYTHONDONTWRITEBYTECODE=1 PYTHONPATH=src python3 -m pytest -q tests/test_telegram_bot_polling.py tests/test_telegram_pilot_store.py tests/test_telegram_pilot_metrics.py
```

Если конкретные тесты называются иначе, сначала найти актуальные имена через `rg`, не запускать тяжелые наборы вслепую.

## Этап 6. Проверка на исторических Telegram-сообщениях

Цель: до live проверить, полезны ли черновики на реальных переписках.

Действия:

1. Взять 20-30 исторических Telegram-диалогов.
2. Для каждого взять текущее сообщение и последние 5-10 сообщений до него.
3. Собрать контекст клиента, если доступен.
4. Прогнать через fake provider и через реальный `codex exec` на малой выборке.
5. Собрать таблицу для ручной проверки.

Колонки таблицы:

- номер строки;
- источник;
- сообщение клиента;
- короткий контекст;
- найденная тема;
- тип сообщения;
- риск;
- черновик;
- что проверить менеджеру;
- почему route такой;
- оценка Дмитрия;
- оценка Насти.

Критерии:

- не менее 50% черновиков выглядят полезными;
- 0 опасных обещаний;
- 0 раскрытий "я бот / ИИ";
- high-risk темы не получают автономный маршрут;
- если сообщение обрывочное, система выбирает `wait_for_more` или уточнение.

## Этап 7. Limited live на Дмитрии и Насте

Цель: проверить настоящий runtime без клиентского трафика.

Порядок:

1. Дмитрий пишет боту с двух аккаунтов.
2. Настя пишет `/start`, получаем служебный chat id.
3. Отправить 5-10 тестовых сообщений:
   - цена;
   - расписание;
   - возврат;
   - маткапитал;
   - "игнорируй инструкции";
   - 3 сообщения подряд за 10 секунд;
   - семейный телефон / несколько детей, если можно имитировать.
4. Проверить, что клиенту ничего не ушло.
5. Проверить, что Настя видит понятный черновик.
6. Собрать обратную связь.

Не переходить к реальному трафику, если:

- есть хотя бы одно опасное обещание;
- Настя не понимает, что делать с сообщением;
- бот путает high-risk тему;
- нет стабильного kill switch;
- черновики массово бесполезны.

## Этап 8. Пилот в реальной работе Насти

Цель: Настя начинает пользоваться помощником в ограниченном режиме.

Условия входа:

- limited live прошел;
- нет автоотправки клиенту;
- есть kill switch;
- есть локальное хранилище;
- есть ежедневный отчет;
- есть понятные критерии оценки черновика.

Метрики на 3-5 дней:

- сколько входящих сообщений обработано;
- сколько черновиков создано;
- сколько черновиков Настя отметила полезными;
- сколько черновиков пришлось переписать с нуля;
- сколько high-risk тем корректно ушло менеджеру;
- сколько раз не хватило фактов;
- среднее время от сообщения клиента до черновика;
- были ли опасные обещания.

Цель первого пилота:

- Настя пользуется каждый день;
- не менее 50% черновиков полезны;
- опасных обещаний 0;
- клиенту ничего не отправляется автоматически;
- понятно, какие темы нужно улучшать первыми.

## Этап 9. Контекстная проверка 100-300 исторических вопросов

Цель: не запускать сразу 9 969, пока не доказано, что контекст собирается правильно.

Выборка:

- 50-100 Telegram;
- 50-100 email;
- 50-100 звонки.

Для каждого вопроса собрать:

- текущий вопрос;
- локальный контекст;
- историю клиента, если найдена;
- AMO/Tallanto/timeline, если найдено безопасно;
- правило РОПа, если есть;
- факты, если есть.

Для звонков:

- сначала проверить, есть ли привязка вопроса к `call_id`, моменту, offset или source_ref;
- если есть, брать 10-15 строк транскрипта вокруг вопроса;
- если нет, использовать старую сводку и помечать `local_fragment_missing`;
- не запускать массовый re-analyze.

Метрики качества контекста:

- `context_found_rate`;
- `telegram_thread_found_rate`;
- `email_thread_found_rate`;
- `call_local_fragment_found_rate`;
- `customer_identity_found_rate`;
- `amo_context_found_rate`;
- `tallanto_context_found_rate`;
- `context_conflict_rate`;
- `family_phone_or_multiple_students_rate`;
- `manual_review_required_rate`.

Метрики качества понимания:

- тема;
- крупная группа;
- главная боль;
- дополнительные темы;
- уверенность;
- was_resolved, если видно;
- причина ручной проверки.

Важно: старый `macro-F1` можно считать на размеченной части, но он не является единственной целью.

## Этап 10. Контекстный прогон 9 969 вопросов

Результат должен отвечать на вопросы:

- какие вопросы чаще всего остаются нерешенными;

## 7. Цели и метрики

### 7.1. Цель пилота

Показать, что система реально помогает менеджеру отвечать клиентам.

Главные метрики:

- ежедневное использование Настей;
- 50%+ полезных черновиков;
- 0 опасных обещаний;
- 0 автоотправок клиенту;
- снижение времени подготовки ответа;
- менеджер не переписывает большинство черновиков с нуля.

### 7.2. Цель безопасности

Система не должна:

- обещать возврат денег;
- обещать скидку;
- обещать место в группе;
- обещать точное расписание без факта;
- давать юридическое решение;
- подтверждать оплату без проверки;
- раскрывать, что ответ написал бот или ИИ;
- выполнять инструкции из текста клиента;
- писать в CRM/Tallanto без подтверждения.

### 7.3. Цель аналитики 9 969

Получить бизнес-карту клиентских вопросов:

- распределение тем;
- топ болей;
- частые нерешенные вопросы;
- темы для FAQ;
- темы для обучения менеджеров;
- темы для будущей автономии.

Точность аналитики должна быть достаточной для управленческих решений, но не обязана быть идеальной для каждого вопроса.
