# Контракт правила (Шаг 2a): «выбор формата (очно/онлайн)». 2026-06-02.

Автор: Клод 1. Источник — `_format_choice_safe_template` (98 LOC, УНПК) + `_format_choice_is_disjunctive_question`
+ `_recent_format_choice_was_ambiguous`. Прочитано по сырью.

## Как сейчас (лазанья, 98 строк)
Мешает: распознавание «онлайн или очно / какой формат» + дизъюнктивность вопроса («или») + дни/выходные +
наличие форматов в фактах (has_offline/has_online) + слоты класс/предмет/формат + память о неоднозначности +
композиция «В УНПК есть очные группы и онлайн-формат».

## Целевой контракт

```yaml
rule_id: format_choice
title: Выбор формата (очно / онлайн) — не решать ЗА клиента
intent: format_choice_inquiry
intent_subvariants: [which_format, available_formats, days_weekend]
brand: unpk                            # текущая функция — УНПК (для Фотона аналог по формату)
required_fact_keys:
  - formats.unpk.offline               # есть очные группы
  - formats.unpk.online                # есть онлайн-формат
  - schedule.weekend_slots             # дни/выходные
data_rules:
  never_choose_for_client: true        # «онлайн ИЛИ очно?» — НЕ выбирать за клиента, дать ОБА варианта
  disjunctive_question: present_both    # дизъюнктивный вопрос («или») → перечислить оба, не один
  ambiguity_memory: true               # если недавно формат был неоднозначен и клиент не уточнил — не фиксировать формат
text_effect: composer_generates_from_data   # «В УНПК есть очные группы и онлайн-формат» из ДАННЫХ наличия
route_effect: bot_answer_self
preserve_exceptions:
  - дизъюнктивный «онлайн или очно?» → дать ОБА, НЕ решать за клиента (это наш known урок про формат)
  - если в фактах только один формат (has_offline && !has_online) → «точно есть очный», без выдумки онлайн
  - неоднозначность из памяти: клиент не уточнил формат → не фиксировать (known_format="")
  - дни/выходные → soft (см. правило schedule, weekend=soft_guidance)
```

## Слои
- **Planner:** intent=format_choice + disjunctive? + класс/предмет + uncertainty. Заменяет regex
  «онлайн|очно ... или|формат» и детектор дизъюнктивности.
- **Слой 2:** правила (never_choose_for_client, present_both, ambiguity_memory) + ссылки на наличие форматов.
- **Composer:** «есть очные группы и онлайн-формат» (или только имеющийся) из данных наличия.
- **Гейт:** выдумка формата, которого нет в фактах → downgrade; формат-цена несовпадение → пересекается с
  price.format_match.

## Связь с правилом price
`format_choice` (какие форматы есть) и `price.format_match` (цена нужного формата) — соседние правила одной
оси «формат». В чистой архитектуре обе опираются на planner-поле `format` и факты наличия/цены. Это
устраняет дубль (сейчас формат-логика размазана и по `_format_choice_safe_template`, и по `_price_for_composition`).

## Негатив-тесты
1. «онлайн или очно?» → дать ОБА варианта, НЕ выбирать за клиента.
2. В фактах только очный → «точно есть очный», НЕ выдумывать онлайн.
3. Клиент не уточнил формат (неоднозначность) → не фиксировать формат молча.
4. «по выходным бывает?» → soft-ориентир (schedule weekend).
5. ПОЗИТИВ: «какие форматы есть в УНПК?» → очные + онлайн из факта, route=answer_self.
