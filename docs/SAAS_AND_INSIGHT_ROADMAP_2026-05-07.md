# План развития Mango Analyse: SaaS, Mango API, база знаний и insight-layer

Дата: 2026-05-07

Цель: определить реалистичный порядок развития после завершения исторической обработки и наведения порядка в проекте.

## Главный вывод

Дальше не нужно выбирать только один путь: `Mango API` или `база знаний из разговоров`. Это две параллельные ветки с разным риском и разной пользой.

Рекомендуемый порядок:

1. Сначала закрыть хвосты обработки и навести порядок в проекте.
2. Затем параллельно вести две ветки:
   - Production ingestion: Mango API/capture, AMO/Tallanto sync, supervisor, reliability.
   - Revenue insight: база знаний, вопросы клиентов, ответы менеджеров, outcome linking, гипотезы продаж.
3. UI redesign делать параллельно, но не вместо стабилизации data contracts.
4. PostgreSQL вводить перед настоящим server/SaaS режимом, не ради красоты.

## Есть ли смысл идти в Mango API

Да. У MANGO OFFICE есть API-направление для Виртуальной АТС; в официальной справке перечислены функции API, включая получение истории звонков и получение записей разговоров. Также в справке по интеграциям указано, что для доступности записей должна быть подключена запись разговоров и включена настройка предоставления/генерации ссылок на записи.

Официальные источники для проверки:

- https://www.mango-office.ru/support/api/
- https://www.mango-office.ru/support/integratsiya-api/restapivatshelp/obshchie_voprosy_po_api_vats_mango_office/
- https://www.mango-office.ru/support/integratsiya-api/chasto_zadavaemye_voprosy/amocrm/net_zapisi_razgovora/

Практический вывод: первый live-capture делать через polling истории звонков и скачивание записей, webhook оставить вторым этапом. Polling проще отлаживать, легче делать идемпотентность и повторную загрузку после падения.

## Почему база знаний из разговоров - сильная ветка

У нас уже есть большой массив реальных разговоров за длительный период. Это актив, который можно использовать не только для саммари в CRM, но и для поиска факторов продажи.

Что можно построить:

1. Извлечь вопросы клиентов из каждого разговора.
2. Извлечь ответы менеджеров на эти вопросы.
3. Привязать разговоры к исходу: купил, не купил, продолжает обучение, в работе, повторная сделка.
4. Оценить качество ответа LLM-оценкой по единой рубрике.
5. Посчитать связь между типами ответов и реальной конверсией.
6. Выделить лучшие формулировки на основе данных, а не интуиции.
7. Собрать validated knowledge base для менеджеров и будущего Telegram-бота.

Важная оговорка: корреляция с продажей не равна причинности. Нужно контролировать хотя бы источник/UTM, менеджера, тип курса, давность заявки, статус клиента, сезонность, цену/рассрочку и количество касаний.

## Гипотезы, которые стоит проверить на исторических данных

1. Скорость первого ответа после заявки влияет на конверсию.
2. Наличие четкого следующего шага в конце звонка повышает вероятность покупки.
3. Ответ на цену через ценность/результат работает лучше, чем скидка без диагностики.
4. Упоминание рассрочки повышает конверсию для части сегментов, но может снижать urgency у других.
5. Длинный разговор не всегда лучше: важно количество закрытых вопросов и next action.
6. Повторный контакт в течение 1-3 дней после возражения "подумаю" лучше, чем поздний follow-up.
7. Разные менеджеры по-разному обрабатывают одно и то же возражение.
8. Клиенты с несколькими сделками/несколькими курсами требуют другой follow-up, чем холодные новые лиды.
9. Причина закрытия "действующий клиент" не должна попадать в reopen как потерянная продажа.
10. Источник/UTM может влиять сильнее, чем качество звонка, поэтому его нужно учитывать в моделях.
11. Наличие вопроса родителя о расписании/нагрузке может быть positive buying intent.
12. Если менеджер не проговорил конкретную программу/следующий шаг, шанс продажи падает.
13. Эмоционально теплый разговор без следующего действия часто переоценивается менеджером.
14. Повторное касание после закрытия сделки может выявлять допродажи/возврат клиента.
15. Отдельные возражения могут быть не отказами, а сигналами к правильному предложению: цена, расписание, формат, страх не потянуть.

## SQLite или PostgreSQL

Текущий вывод:

- SQLite оставить для локального batch-режима, ASR worker packs и переносимых M1/M4 задач.
- Для серверного SaaS/multi-process режима перейти на PostgreSQL.
- Для аналитики по большому корпусу добавить DuckDB или аналитические materialized tables поверх Postgres.

Почему SQLite пока нормален:

- Историческая batch-обработка уже работает.
- Переносимые external-worker пакеты проще делать через SQLite.
- Для одиночного Mac и controlled workers это прагматично.

Почему PostgreSQL нужен до SaaS:

- Одновременный capture, ASR, Resolve, Analyze, sync, UI и notifications будут писать/читать параллельно.
- Нужны транзакционные очереди, блокировки, audit log, retries, multi-user access.
- Для демонстраций клиентам нужна предсказуемая эксплуатация, backup/restore и мониторинг.

Решение: не мигрировать прямо сейчас всю историю ради миграции. Сначала зафиксировать data model и pipeline statuses, затем сделать Postgres-compatible repository layer и миграцию активного operational слоя.

## UI: что дорабатывать

UI должен стать не "панелью запуска скриптов", а рабочим приложением.

Минимальные экраны:

1. Dashboard: что сейчас обрабатывается, сколько готово, где хвосты, где ошибки.
2. Coverage: ASR/R+A/Sync по месяцам и источникам.
3. Batch Builder: выбрать период/источник/стадию/лимит и собрать batch без терминала.
4. Worker Control: запуск/стоп потоков, профили M4/M1/server.
5. ROP Queue: Reopen/Follow-up/Manual Review/Top priorities.
6. AMO Writeback: dry-run diff, staged rollout, ошибки.
7. Knowledge Lab: вопросы клиентов, ответы менеджеров, outcome, лучшие ответы.
8. Settings: Mango API, AMO, Tallanto, models, paths, retention.

UI redesign стоит начинать после фиксации data contracts, иначе красивый интерфейс будет ломаться из-за меняющихся статусов и файлов.

## Что можно делать параллельно в разных диалогах

Да, можно вести параллельно, но с жестким разделением write scope.

Параллельный поток A: `Mango API/capture`.

- Пишет в новые модули клиента и capture.
- Не трогает текущий ASR/R+A алгоритм.
- Работает в shadow-mode.

Параллельный поток B: `Knowledge/insight-layer`.

- Читает готовые transcript/analysis/AMO/Tallanto данные.
- Пишет новые скрипты/модели аналитики.
- Не пишет в AMO и не меняет production DB без dry-run.

Параллельный поток C: `UI productization`.

- Сначала описывает data contracts и mock API.
- Потом меняет UI.
- Не должен менять бизнес-логику pipeline без тестов.

## Что делать первым после наведения порядка

Мой выбор: запускать две ветки параллельно.

Первый инженерный блок: Mango API polling POC.

Причина: без автоматического получения звонков проект останется сильным batch-аналитиком, а не автономным сотрудником/сервисом.

Первый revenue/block: Knowledge extraction MVP.

Причина: у нас уже есть данные, это может дать быстрый коммерческий инсайт для отдела продаж без ожидания серверной инфраструктуры.

Не начинать первым:

- Полноценный Telegram sales bot без KB и validated answer base.
- Массовую миграцию в Postgres до фиксации data contracts.
- Красивый SaaS UI до отделения runtime/data от кода и стабилизации статусов.

## Коммерческая логика продукта

Этап 1: автономно использовать внутри своей компании.

- Автоматически подтягивать звонки.
- Обрабатывать и писать саммари/приоритеты в AMO.
- Давать РОПу daily digest.
- Строить базу знаний и улучшать продажи.

Этап 2: демонстрировать потенциальным клиентам.

- Показывать dashboard, ROP queue, AMO writeback, пример базы знаний.
- Работать на обезличенных/демо-данных.
- Уметь подключаться к Mango + AMO как reference integration.

Этап 3: выборочно продавать нескольким компаниям.

- Нужны tenant isolation, secrets management, deployment runbook, support process.
- Не обязательно сразу поддерживать все телефонии и CRM.
- Архитектура должна иметь adapter layer: Mango/другая телефония и AMO/другая CRM меняются адаптерами, core analysis остается тем же.

## Риски

1. Переоценка LLM-score: модель может красиво оценивать, но не отражать реальную продажу.
2. Outcome leakage: если анализ знает итог сделки, он может задним числом рационализировать качество ответа.
3. Confounding: менеджер/source/course/сезон могут объяснять конверсию сильнее, чем формулировка ответа.
4. Data privacy: для SaaS нужны правила хранения аудио, расшифровок и персональных данных.
5. API reliability: Mango/AMO limits и OAuth могут ломать live pipeline.
6. UI без data contracts приведет к красивой, но хрупкой системе.

Можно стартовать SaaS/product development, когда выполнены:

- Coverage v4 закрывает текущие хвосты или явно объясняет исключения.
- Project cleanup выполнен, runtime/data отделены от кода.
- README/runbook/scripts catalog/field mapping готовы.
- Тесты проходят.
- AMO writeback имеет staged dry-run gate.
- Есть решение по DB: что остается SQLite, что переезжает в Postgres.
- Есть отдельные ветки задач для Mango API и Knowledge Layer без пересечения файлов.
