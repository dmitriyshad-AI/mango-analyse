# Контракт правила (Шаг 2a): «цены». 2026-06-02.

Автор: Клод 1. По эталону. Источник — `_soften_current_price_deadline_text` (127 LOC), `_is_future_price_case`
+ `FUTURE_PRICE_INPUT_RE` (608), `_price_for_composition`/`_compose_price_plus_format` (pipeline, формат-скоринг),
~12 PRICE-констант. Прочитано по сырью. ВАЖНО: цены подтверждают правило «значение → в KB-факт, не в код».

## Ключевое отличие от рассрочки/скидок
Числовые ЗНАЧЕНИЯ цен живут в KB-фактах (`prices_regular_2026_27.*`), НЕ в хардкод-константах. Большинство
PRICE-констант в коде — это ДЕФЕРРАЛЫ («менеджер подскажет актуальную стоимость»), потому что цены меняются.
Поэтому в коде остаются не числа, а ПРАВИЛА: совпадение формата (очно/онлайн), будущая цена, раннее
бронирование, бренд-scope. Это прямой пример согласованного решения: значение → KB-факт; правило → слой 2.

## Целевой контракт (слой 2 = ПРАВИЛА + ссылки на факты, значения в KB)

```yaml
rule_id: price
title: Цены (регулярные, интенсив, лагерь/ЛВШ)
intent: price_inquiry
intent_subvariants: [regular, intensive_ege, camp_lvsh, format_choice, price_fix_process]
# Значения — в KB, правило только ссылается на ключи и выбирает по формату/периоду
required_fact_keys:
  - prices_regular_2026_27.*            # ВСЕ значения тут (очно/онлайн × класс × семестр/год)
  - prices.intensive_ege               # интенсив: 1 предмет / 2 предмета
  - prices.camp_lvsh                    # лагерь/ЛВШ
  - early_booking_2026_27               # сроки/выгода раннего бронирования
data_rules:                            # НЕ числа — ПРАВИЛА
  format_match: required               # ответ-цена ДОЛЖНА быть формата вопроса (очно↔онлайн)
  format_no_substitute: true           # нет факта нужного формата → НЕ подставлять другой → уточнить/менеджер
  future_price: do_not_promise         # «цена после 1 июля/августа?» → не обещать будущую, менеджер
  early_booking_deadline: preserve     # срок раннего бронирования НЕ обрезать из текста (Block1 урок)
  brand_scope:
    unpk: {ochno: "5-11 классы", online_olympiad: "только 9 и 11"}
    foton: {online: "семестр/год"}
pure_functions:
  pick_price: "match(asked_format, asked_period) over fact_keys"   # выбор факта по формату+периоду
brand_split: true
blocking_conditions:
  - future_price_after_deadline        # будущая цена → менеджер (не обещать)
  - exact_price_absent_in_fact         # точной цены под класс/формат нет → менеджер/уточнить
  - format_mismatch                    # спрошен очно, факт только онлайн → НЕ подставлять (формат-гард)
route_effect: bot_answer_self          # если факт нужного формата есть; иначе manager/уточнение
text_effect: composer_generates_from_data
preserve_exceptions:
  - формат: цена в ответе = формат вопроса (очно/онлайн); чужой формат НЕ подставлять (это наш формат-гард)
  - будущая цена «после 1 июля» (изменение) ≠ «сколько после 1 июля» (текущая на дату) — не путать
  - раннее бронирование: срок/дедлайн сохранять (не обрезать → клиент не подумает «скидка вечная», Block1)
  - price_fix_process: «как зафиксировать цену» → процесс оформления, не само число
```

## Распределение по слоям
- **Planner:** `intent=price_inquiry` + subvariant + формат(очно/онлайн) + период(семестр/год) + класс +
  uncertainty. Заменяет format-скоринг regex (`preferred_format`, `_price_for_composition` 127 LOC).
- **KB-факты:** все ЧИСЛА цен (не в коде).
- **Слой 2:** ПРАВИЛА (format_match, future_price, early_booking, brand_scope) + `pick_price`.
- **Composer:** пишет ответ из выбранного факта; нет факта формата → честное уточнение/дефер.
- **Гейт:** future_price_after_deadline → manager; format_mismatch → downgrade (это наш формат-гард, теперь
  декларативное правило, а не точечная заплатка); число вне факта → downgrade.

## Связь с прошлой работой
Наш формат-гард цены (`TZ_format_guard_price`) и Block1.1-урок про раннее бронирование — это были ЗАПЛАТКИ
на лазанью. В чистой архитектуре они становятся ДЕКЛАРАТИВНЫМИ правилами (format_match=required,
early_booking_deadline=preserve). То есть инверсия превращает наши точечные правки в строки контракта.

## Обязательные НЕГАТИВНЫЕ тесты
1. Формат: спрошено «очно 6 класс цена», факт только онлайн → НЕ подставлять онлайн (формат-гард).
2. Будущая цена: «сколько будет после 1 июля?» (изменение) → не обещать, менеджер.
3. Точной цены под класс/формат нет → уточнить/менеджер, не выдумать число.
4. Бренд: УНПК очно (5-11), онлайн-олимпиада только 9/11 — не перепутать.
5. Раннее бронирование: ответ сохраняет дедлайн (не «скидка вечная»).
6. price_fix: «как зафиксировать цену» → процесс, не число.
7. ПОЗИТИВ: «онлайн 9 класс цена?» + факт → цена нужного формата, route=answer_self.

## Вывод
Цены = KB-факты (значения) + правила формата/будущей цены/раннего бронирования (слой 2) + composer. Это
показывает третий тип переноса: где значение динамично — оно в KB, а в коде только правило. Наши прошлые
заплатки (формат-гард) становятся декларативными правилами. Семейства: рассрочка/скидки (данные-в-коде),
цены (правила+KB-факты). Следующее по эталону: лагерь/ЛВШ или расписание.
