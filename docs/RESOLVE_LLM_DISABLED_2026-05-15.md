# Resolve LLM выключен по умолчанию

Дата: 2026-05-15

## Что изменилось

`RESOLVE_LLM_PROVIDER` теперь по умолчанию равен `off`.

Если провайдер выключен или задан неизвестным значением, `ResolveService._resolve_with_llm()` сразу возвращает `None` и не создает кандидата с именем `llm`.

## Почему так

Раньше при `RESOLVE_LLM_PROVIDER=off` код мог перейти на правило с провайдером `rule`, но кандидат все равно назывался `llm`. Из-за этого счетчик `llm_used` показывал использование языковой модели там, где реального вызова модели не было.

Теперь `llm_used` означает реальную попытку LLM-resolve через `ollama`, `openai` или `codex_cli`.

## Как включить обратно

В `.env` или окружении можно явно указать один из рабочих провайдеров:

```text
RESOLVE_LLM_PROVIDER=codex_cli
```

Другие допустимые значения:

```text
RESOLVE_LLM_PROVIDER=ollama
RESOLVE_LLM_PROVIDER=openai
```

Для `openai` нужен `OPENAI_API_KEY`. Для `codex_cli` нужен доступный CLI Codex. Для `ollama` нужен локальный Ollama-сервер.

## Что не меняется

- Обычный resolve по правилам остается рабочим.
- Rescue ASR не включается этой настройкой.
- Старые варианты `stereo` и `mono_or_fallback` остаются совместимыми.
- При явно включенном LLM-провайдере поведение resolve сохраняет прежнюю схему кандидата `llm`.
