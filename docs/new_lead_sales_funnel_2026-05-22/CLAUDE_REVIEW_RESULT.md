# Независимая проверка Claude CLI

Дата: 2026-05-22

Статус: read-only review. Claude CLI был доступен и запущен без права редактирования файлов.

## Вердикт Claude

`PASS_WITH_NOTES` как проектный candidate-пакет.

`BLOCKED` для клиентского внедрения без утверждения спорных фактов, автотестов и решения по политике "кто отвечает".

## Что Claude отметил как сильное

- Правило "прямой ответ -> польза -> один следующий шаг" снижает риск давления и формальности.
- Воронка доводит клиента до действия через подбор, пробное, заявку или менеджера.
- Бренды разделены, есть стоп-слова для Фотона и УНПК.
- P0/high-risk темы корректно идут менеджеру.
- 28 исходных сценариев с полем "провал" уже подходят как база для автотестов.

## Блокирующие замечания Claude

1. В пакете перечислены, но не исправлены конфликтные факты в базе знаний:
   - Фотон: "рассрочка 2026-05-22 месяцев";
   - Фотон: "городской лагерь полностью бесплатно";
   - УНПК: "ЛВШ Менделеево, Москва — бесплатно";
   - УНПК: "пробная неделя бесплатно";
   - УНПК: "при помесячной оплате скидка 10%" против "10% за семестр";
   - устаревшие онлайн-цены УНПК.
2. Нет автотестов по новым сценариям.
3. Не утверждена политика ответа на вопрос клиента "кто отвечает?".
4. Нужен отдельный список смысловых gate-правил.
5. В README был указан файл `CLAUDE_REVIEW_RESULT.md`, которого еще не было на момент проверки.

## Неблокирующие риски Claude

- Фраза "помощник менеджера" не закреплена как политика роли.
- По Фотону Т-Банк/Долями лучше оставить как черновик менеджеру, пока Дмитрий/РОП не утвердят автономность.
- Для первого пилота метрик слишком много; лучше выбрать 8-10 ключевых событий.
- Нужен явный запрет обещать результат при вопросах про ЕГЭ/олимпиады.
- Полная сводка менеджеру может быть слишком длинной.
- Не хватало тестов на запрос ФИО преподавателя, лицензии и реквизитов.
- Для "не спрашивать известное" нужно позже определить надежный источник истины.

## Что я изменил после Claude

- Добавил этот файл `CLAUDE_REVIEW_RESULT.md`.
- Добавил `SEMANTIC_GATE_CANDIDATES.md`.
- Добавил короткую 4-полевую сводку менеджеру.
- Зафиксировал, что политика ответа на "кто отвечает?" пока не утверждена и требует отдельного решения.
- Добавил минимальный набор из 10 событий для первого пилота.
- Расширил тестовые сценарии до 30, добавив запрос ФИО преподавателя и запрос лицензии/реквизитов.

## Где согласен с Claude

- Прямое внедрение блокировано до очистки спорных фактов и автотестов.
- Смысловые gate-правила нужны до любого расширения автономности.
- Метрики первого пилота нужно сократить до минимального набора.
- Вопрос "кто отвечает?" нельзя оставлять на импровизацию модели.

## Где финальное решение отличается

Claude предложил "зафиксировать ответ на вы бот?". Я не фиксирую клиентскую формулировку в этом пакете, потому что в текущей памятке сотруднику эта тема прямо вынесена на будущее решение. В пакете оставлена безопасная позиция: до утверждения не раскрывать служебные детали и отправлять такие случаи в ручную проверку.

## Итог

Пакет можно использовать как проектную основу для следующего bot-диалога. Нельзя использовать как готовую клиентскую инструкцию без утверждения фактов, реализации тестов и отдельного смыслового gate.
