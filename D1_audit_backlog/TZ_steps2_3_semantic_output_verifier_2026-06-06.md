# ТЗ — Шаги 2-3 ADR-001: смысловой верификатор выхода («заземление-в-духе»)

**Статус:** v3 ФИНАЛ (два аудита второго архитектора; все блокеры закрыты; готово к выдаче Codex)
**Дата:** 2026-06-06
**Дерево:** worktree `consolidation-clean-type`. **Сначала коммитятся 4 патча** (таймаут understand / жаргон-санитайзер / refund-латч / кросс-бренд-T1, они уже в working tree) — ветка верификатора идёт ОТ их коммита; все координаты пере-grep.
**Основание:** `REGRADE_step4_four_runs_2026-06-06.md`, контракт-таблица ADR-001.

## Зачем (по сырью)

Шаг 4 освободил модель: уходы упали (Claude 9→4), детерминизм чист. Но на ~50 диалогах — 4-5 СЕМАНТИЧЕСКИХ производных: «ОГЭ — базовая, для выравнивания» (назначения нет в базе); «оплата по оферте = подтверждение записи» (склейка РЕАЛЬНЫХ фактов в неподтверждённый порядок); «год-два / большинство ребят» (обобщение/соцпруф как опора); «обычная группа = базовый уровень»; «обычно в очном курсе…». Детерминизм это не ловит и не должен. Это работа смыслового слоя.

## Архитектурное решение (уточнено аудитом по коду)

- **Faithfulness-критик НЕ трогаем.** Он — вето внутри цикла отбора кандидатов (`check_claim_faithfulness`, pipeline ~1821; вызовы в repair/keep-answer/cite-only/composition, ~1.8/ход), держит баланс Шага 4. Его таксономия — КОНКРЕТИКА (расписание/цены/сроки); наши классы — дополнение, не дубль. Его поглощение — кандидат Шага 5, только после теневого прогона.
- **Поглощаем диагноз-классификатор** (subscription ~1962-1970: тот же слот, та же сигнатура «текст+вопрос→JSON») — его классы входят в единый проход.
- **Дух правила простоты сохраняется: +1 LLM-проход на ход — потолок** (steady-state, без учёта редких regen; пересмотр потолка — только через ADR, не молча). Новые смысловые проверки дальше — только расширением ЭТОГО прохода или поглощением, не новыми вызовами.
- **Точка вызова: ровно ОДИН раз на ход, на терминальной точке `build_draft` (оба пути), НЕ внутри `apply_authoritative_output_gate`** — она чистая функция и зовётся ~12 раз (на legacy дважды на ход + на каждом safe-fallback). Чистые safe-шаблоны (`_is_pure_handoff_text`-класс) — скип. Терминальных точек ДВЕ: ~1970 (pipeline-путь) и ~2022 (legacy) — обернуть обе (или single-exit), каждая исполняется ≤1 раза за ход. `FakeSubscriptionLlmDraftProvider.build_draft` (~2708) — тестовая копия, верификатор туда не попадает; учитывать в NEG-тестах.

## Часть A — Шаг 2: повторный детерминированный hard-скан финального текста

После ЛЮБОГО генеративного шага (X2, A2, тон, регенерация верификатора), 0 LLM: число-заземление (Part B Шага 4), бренд-токены, мета/внутренний жаргон («client-safe»-паттерн из патча; NEG-прогоны с `TELEGRAM_OUTPUT_SANITIZER=1` либо опора на безусловный `meta_leak`), идентичность, плейсхолдер, промокод.
**Реестр: РАСШИРЯЕМ существующий `GATE_BLOCKING_CODES`** (subscription ~484-519), не строим параллельную таблицу. Новые действия: `annotate` и `downgrade_keep_text`. `block` — ТОЛЬКО у детерминированных кодов. Менять реестр — только явной правкой.

## Часть B — Шаг 3: единый смысловой проход на терминальной точке

Вход: финальный текст + retrieved_facts + вопрос клиента. Выход — структурные findings:

- `derived_product_claim` — продукту/организации приписано то, чего нет в фактах (назначение группы, состав курса, порядок записи, материалы, размер группы, «у нас принято…») → **downgrade_keep_text**;
- `invented_generalization` — обобщение/соцпруф как ОПОРА рекомендации клиенту («большинство», «обычно за год-два») → **annotate**, даже при хедже (tie-break аудита: хедж не освобождает опору рекомендации, но и не пунтит её);
- `individual_diagnosis` — оценка конкретного ребёнка (true/false кейсы существующего классификатора) → по текущей политике диагноз-гарда;
- `ok`.

**Дискриминатор:** о НАШЕМ продукте/процессе → заземлять; общемировой совет с хеджем БЕЗ продуктовой специфики и БЕЗ роли «опоры рекомендации» → не флагать; перефраз/склейка РЕАЛЬНЫХ фактов без новых утверждений → не флагать (QR/бухгалтерия реальны; флагуется только приписка «= подтверждение записи»); каноничная фраза разделения → не флагать.

**Семантика действий:**
- `downgrade_keep_text` — маршрут `draft_for_manager`, **ТЕКСТ СОХРАНЯЕТСЯ** (не подмена на SAFE_FALLBACK — менеджер должен видеть, ЧТО проверять) + evidence-спан + missing_fact в manager_checklist/metadata. Отдельный `fallback_reason=semantic_verifier_downgrade` (семья output_safety) — иначе бюджет не посчитать по сырью. ОБЯЗАТЕЛЬНО те же safety-флаги, что ставит гейт (`manager_approval_required`, `no_auto_send`, gate-флаг) — будущая автономия ключуется на флаги, не только на route. Новые reasons (`semantic_verifier_downgrade`, `semantic_verifier_unavailable`) ДОБАВИТЬ в exact-match классификаторы семей: pipeline ~506-533 и раннер ~2642-2674 — иначе провалятся в `return reason` и бюджет посчитается мимо семьи.
- `annotate` — маршрут НЕ меняется; заметка менеджеру в manager_checklist/metadata; **НИКОГДА не в draft_text** (судья/факт-аудит не должны оценивать служебные строки). **Лимит ≤2 — только на ПРЕДСТАВЛЕНИЕ** (одна агрегированная заметка менеджеру: ≤2 evidence-примеров + «и ещё N»); в metadata/summary хранятся ВСЕ findings, иначе ломаются Exit-счётчики (уточнение аудита №2 к моему капу — принято).
- **Регенерация: одна попытка**, с ограничением («убери/захеджируй утверждение Х»), таймаут regen-вызова те же ~30с. После неё — **ПОЛНЫЙ повторный `apply_authoritative_output_gate` С client_message+context** (verify_output детерминирован, 0 LLM — дух Части A сохранён; БЕЗ контекста ловушка `text_only` ~3366 молча выключает verify_output!) + один re-check верификатора; **X2/A2/тон НЕ повторяются** (паттерн `_humanity_x2_repo_gate` ~5042); счётчик попытки в metadata. **Regen ЗАПРЕЩЁН на автономных маршрутах** (`AUTONOMOUS_ROUTES` ~482) — там сразу downgrade_keep_text. Не сошлось — downgrade_keep_text.
- **Precedence/dedupe:** если детерминированный код уже сработал (estimate_*, диагноз-гард, число/бренд) — verifier-finding того же класса дедупится и НЕ входит в бюджет. Детерминизм выигрывает всегда.

**Fail-soft (уточнено):** отдельный reason `semantic_verifier_unavailable` (семья provider_runtime), действие = annotate-only («смысловая проверка недоступна»), **маршрут НЕ меняется** — advisory-слой не становится замком через собственную недоступность; жёсткие классы держит детерминизм. НЕ входит в downgrade-бюджет. Собственный wall-clock таймаут **~30с** (не наследовать 90с `timeout_sec`). ✅ РЕШЕНО Дмитрием 2026-06-06: (1) **Недоступность верификатора:** ОДИН повтор с тем же таймаутом (~30с); снова недоступен → `semantic_verifier_unavailable`; в черновиковом режиме — annotate-only, маршрут не меняется; в будущем АВТОНОМЕ — автоотправка хода БЛОКИРУЕТСЯ (fail-closed; сейчас не реализовывать, оставить комментарий у флага). (2) **Производные:** СЕЙЧАС всегда `downgrade_keep_text` (менеджер). В будущем автономе — градация: производная, которая НЕ противоречит базе, близка по смыслу к фактам и НЕ из жёстких классов (число/цена/срок/бренд/P0/обещание результата) — может отправляться С явным маркером неуверенности («точно не подтверждаю, но обычно…»); противоречащая или жёсткого класса — только менеджер. Для этого finding несёт поле **`relation_to_base: contradicts | absent | adjacent`** (+ id ближайшего факта при adjacent) — дёшево сейчас, обязательно потом, и уже сегодня делает заметку менеджеру информативнее («противоречит факту Х» vs «в базе нет»).

**Модель верификатора:** отдельный env `TELEGRAM_SEMANTIC_VERIFIER_MODEL` (default: codex/medium — прагматика по цене); «не тот же инстанс» — слабая гарантия для GPT-мозга, поэтому NEG-фикстуры обязаны пройти реплей минимум одной ДРУГОЙ моделью (model-robust, урок диагноз-гарда: без keyword-обходов).

## Граница (не нарушать)

НЕ трогать: faithfulness-критик и его inner-loop вето; детекторы @4293/@4424; `verify_dialogue_contract_output`; deny-флаги; P0-пол; число-Part B. Верификатор не «единственный замок» ни для одного жёсткого класса. Флаг `TELEGRAM_SEMANTIC_OUTPUT_VERIFIER`, default OFF.

## NEG-контроли (обязательны)

1. 5 кейсов регрейда (фикстуры с их РЕАЛЬНЫМИ фактами) → флагуются с верным классом; «флагуется» = annotate и выше (по tie-break: год-два/большинство → annotate; ОГЭ-выравнивание, оплата=запись, группа=уровень, обычно-состав → downgrade_keep_text).
2. Анти-ложные → 0 флагов: дословный пересказ факта; перефраз факта; склейка двух реальных фактов БЕЗ новой приписки; каноничная фраза разделения; хеджированный общемировой совет без продуктовой специфики и без опоры-рекомендации.
3. Анти-разблокировка: верификатор сказал «ok» тексту с бренд-смешением → детерминизм блокирует.
4. Fail-soft: убитый процесс И TimeoutExpired-путь → прогон завершается, `semantic_verifier_unavailable`, маршрут не изменён, ничего не висит (урок understand()).
5. Диагноз-кейсы ловятся при любом роуте; скип — только чистых safe-шаблонов, НЕ любого manager_only-текста; счётчик гарда > 0.
6. Регенерация ровно одна; после неё **ПОЛНЫЙ гейт заново (вкл. verify_output С контекстом)**, не 6-классовый скан; X2 не перегенерил текст; на автономном маршруте regen НЕ запускается.
7. Один вызов на ход: на legacy-пути и на safe-fallback'ах верификатор НЕ вызывается повторно (счётчик).

## Exit (оба мозга, --parallel 4, по сырью)

- Юнит/реплей: 5/5 флагуются (верный класс/действие), анти-ложные 0, реплей другой моделью пройден.
- Живой перепрогон 13+25: findings с evidence в transcripts/summary; детерминизм без изменений (0 число/бренд/P0, инвариант 0); **downgrade-бюджет ≤ ~10% ходов** (annotate и unavailable в бюджет НЕ входят); fail-soft доказан.
- **Измеримость (новые поля обязательны):** роль верификатора в `_llm_call_summary` (~2065, иначе утонет как утонул bot_diagnosis_guard); per-turn `bot_semantic_output_verifier` (по образцу `bot_authoritative_output_gate`); summary-блок: счётчик findings по классам + downgrade-rate со знаменателем «все bot-ходы». `llm_calls net +1/ход` проверять при чистом кэше (claude-прогоны с кэшем дают битый total=25/37 ходов).
- Латентность: бюджет ~+25-30с/ход (факт: ~30с/codex-вызов), regen-ходы +2 вызова; зафиксировать в отчёте.
- Кэшируемость вызова верификатора оговорить явно (влияет на проверку net+1 при чистом кэше; claude-кэш уже давал битый total). Регрейд-инструментарий читает per-turn `bot_semantic_output_verifier`, чтобы пойманные-и-downgraded производные НЕ считались «выдумками» в следующих регрейдах.
- pytest зелёный.
