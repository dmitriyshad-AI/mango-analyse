# ТЗ Phase 7-strict — удержание контекста на длинных диалогах (3 подфазы)

Автор: второй Claude. Дата: 2026-05-29. Read-only ТЗ (реализует Кодекс). Основано на `long_dialog_context_retention_analysis_2026-05-29.md`.

## Контекст и принцип

Phase 7-strict закрывает три структурных дефекта удержания контекста на 15-25 ходах, найденных в архитектурном анализе: (1) P0-латч не распадается в автономном потоке; (2) исправление слота клиентом теряется молча; (3) нет семантического сжатия истории, окна контекста рассинхронизированы. Это НЕ калибровка (Phase 11) — это структурные изменения, поэтому отдельная фаза.

**Безопасный инвариант (не нарушать ни в одной подфазе):** реальный P0 (legal/payment_dispute и подтверждённая претензия) НИКОГДА не ослабляется; бренд-разделение и 0-fabrication сохраняются. Каждая подфаза имеет контрольный отрицательный тест.

Порядок реализации: **7.2 → 7.1 → 7.3** (7.2 дешевле и снимает самый частый тихий дефект; 7.1 — слой безопасности, осторожно; 7.3 — крупнейшая, опирается на стабильные слоты из 7.2).

# §7.1 — Жизненный цикл P0-латча

## Цель

Латч должен УМЕТЬ автономно отпускаться при разрешении ситуации, не дожидаясь менеджера, НЕ ослабляя реальный P0. Сейчас в автономном потоке латч монотонен и глушит остаток длинного диалога после одного срабатывания (в т.ч. ложного).

## Что сейчас в коде (точно)

- Авторитетный латч: `DialogueP0Latch` (`dialogue_memory.py:118`) с полями `active, codes, primary_risk, started_at, trigger_turn_id, release_event_id` — **схема уже предусматривает релиз**.
- `_next_p0_latch` (`dialogue_memory.py:722`): `if previous.active: return previous` (`:734`) — монотонно; релиз только если `_p0_latch_release_event(context)` (`:730`).
- `_p0_latch_release_event` (`:769-776`): срабатывает ТОЛЬКО на ключи `manager_clear_p0_latch / manager_resolved_p0 / manager_took_over / p0_latch_release_event` — **манагер-only**.
- Второй латч: `HeldState.p0_latched` (`held_state.py:29`), `update_held:119` `new_p0 = held.p0_latched or p0_required` — монотонен, релиза НЕТ вовсе; `turns_seen` (`:34/139`) считается, но не используется.
- Чтение латча: `classify_answer_safety._p0_latch_codes` (`answer_safety_classifier.py:181-207`) → `latch.get("active")`.

## Lifecycle states (целевая модель)

```
INACTIVE → ACTIVE → (SOFTENING) → RELEASED → INACTIVE
```

- **ACTIVE**: латч поднят (`codes`, `primary_risk`, `started_at`, `trigger_turn_id`, + НОВОЕ `trigger_turn_index:int`).
- **SOFTENING** (новое промежуточное): кандидат на автономный релиз — условия релиза начали выполняться, но не подтверждены (1 ход «остывания»). Нужно, чтобы не отпускать на одной случайной нейтральной реплике.
- **RELEASED**: латч снят (`release_event_id` = причина: `manager_*` или `auto:<reason>`); следующий ход → INACTIVE если нет нового P0.

## Правила автономного релиза (консервативные)

Авто-релиз РАЗРЕШЁН только для «мягких» категорий и только при явном сигнале разрешения:
- `primary_risk == "refund"` И исходный P0 был benign-presale, разрешённый по факту `refund_presale_policy` (а не активная претензия) → авто-релиз после 1 нейтрального хода.
- `primary_risk == "complaint"` И эмоция была направлена на бота (не претензия к услуге/деньгам) И ≥2 последующих хода без P0-маркеров → авто-релиз.
- **НИКОГДА авто-релиз для** `legal_threat`, `payment_dispute`, и для `refund`/`complaint` с финансовым/юридическим маркером. Эти — только `manager_*` релиз (как сейчас).
- Доп. предохранитель `turns_seen`: если латч `ACTIVE` дольше `P0_LATCH_MAX_TURNS` (предлагаю 8) И категория «мягкая» И нет повторных P0-маркеров — перейти в SOFTENING; это страховка от «вечного» залипания на ложном срабатывании, не отмена реального P0.

## Таблица переходов латча

| Из | Событие | В | Условие |
|---|---|---|---|
| INACTIVE | latchable P0-код на ходе | ACTIVE | `_latchable_p0_codes` непуст (`dialogue_memory.py:750`) |
| ACTIVE | manager-релиз | RELEASED | `_p0_latch_release_event` (`:769`) — как сейчас |
| ACTIVE | benign_origin + сигнал разрешения (мягкая категория) | SOFTENING | refund-presale resolved / эмоция-к-боту улеглась |
| ACTIVE | новый P0-маркер | ACTIVE | повышение/обновление codes |
| ACTIVE | `turns_seen - trigger_turn_index > P0_LATCH_MAX_TURNS` + мягкая категория + нет P0 | SOFTENING | страховка от вечного залипания |
| SOFTENING | ещё 1 ход без P0 | RELEASED | подтверждение остывания |
| SOFTENING | P0-маркер вернулся | ACTIVE | откат к эскалации |
| RELEASED | нет P0 | INACTIVE | следующий ход |
| RELEASED | новый P0 | ACTIVE | новый триггер |

Жёсткие категории (`legal_threat`, `payment_dispute`, refund/complaint с фин-маркером): переходы ACTIVE→SOFTENING/RELEASED по benign/turns ЗАПРЕЩЕНЫ — только manager-релиз.

## Регресс-тесты

- **Авто-релиз (мягкие):** диалог: t3 benign-presale refund (P0), t4-t5 нейтральные follow-up → на t5 латч RELEASED, t6 отвечает автономно. (Тест-харнесс 7.3/Задача 3, сценарий «refund-presale + продолжение».)
- **Эмоция-к-боту:** t4 «вы не отвечаете» (если до 1a давало complaint-латч), t5-t6 нейтральные → RELEASED.
- **КОНТРОЛЬ — не отпускать реальный P0:** `P0a_refund_claim`/`P0b_complaint` с финансовым маркером, legal-угроза, payment_dispute — латч ACTIVE до manager-релиза, авто-релиз НЕ срабатывает (даже через 8 ходов).
- **20+ ходов:** ложный P0 на t3 (мягкая категория) не должен глушить t10-t20; реальный P0 на t3 — должен держать до конца/менеджера.
- **Manager-релиз** (существующий) продолжает работать (`manager_clear_p0_latch`).

## Риски

- **Главный:** авто-релиз ослабит реальный P0. Митигация: белый список «мягких» категорий + benign_origin + контрольные негативные тесты обязательны в тесте выхода.
- Два латча (held vs memory) разойдутся — поэтому рекомендация свести к одному источнику (правка 5).
- `benign_origin` определяется на ходе триггера — если классификация benign/claim ошибочна (см. Волну 1a), авто-релиз может сработать на реальной претензии, ошибочно помеченной benign. Митигация: авто-релиз ТОЛЬКО для refund/complaint, никогда для legal/payment.

# §7.2 — Событие коррекции слота

## Цель

Распознавать, когда клиент ИСПРАВЛЯЕТ ранее названный слот («нет, я про 10», «не 9, а 10», «передумала», «извините — на самом деле 10 класс»), перезаписывать locked-слот, сбрасывать зависимый кэш и подтверждать клиенту. Сейчас коррекция теряется молча, если сформулирована без триггер-слова «класс».

## Что сейчас в коде (точно)

- `extract_grade` (`new_lead_funnel.py:452`): `\b([1-9]|1[01])\s*класс` — требует слова «класс»; «нет, я про 10» → `""`.
- `_extract_slots_from_turns` (`dialogue_memory.py:485`) → `extract_grade` по СКЛЕЙКЕ client-ходов → ПЕРВЫЙ матч (старое значение).
- `_extract_slots_from_text` (`:490`) — по текущему ходу; в `build_dialogue_memory:255` идёт с `override=True, confidence=0.95`.
- `_merge_slots` (`dialogue_memory.py:501`), `conversation_intent_plan._merge_slots` (`:780`), `_extract_slots` (`:793`).
- `held_state._is_direct_correction` (`:143`) — маркеры коррекции есть, но grade-числа не покрывают и слоты не трогают.
- `_do_not_reask_slots` (`dialogue_memory.py:787`) — метит любой непустой слот → переспрос подавлен.
- `client_confirmed_slots` = `_slots_by_source(..., {"dialogue_memory"})` (`:779/284`).

## Что сделать

1. **Детектор коррекции `detect_slot_correction(text, known_slots)`** (новый, в `conversation_intent_plan.py` или `semantic_roles.py`). Возвращает `{slot: new_value}` для исправлений. Маркеры (две группы, нужна КОМБИНАЦИЯ негации/коррекции + новое значение):
   - корректирующие: «нет,», «не …, а …», «не девят…, десят…», «передума…», «на самом деле», «извините… на самом деле», «я про …», «я имел в виду», «ошиблась/ошибся», «точнее».
   - новое значение: число (1-11) даже без «класс» в корректирующем контексте; предмет; формат.
2. **Расширить `extract_grade`** (`new_lead_funnel.py:452`) или ввести `extract_grade_loose` для корректирующего контекста: ловить «(про|в|на|это)\s*(1[01]|[1-9])(-?[ыйео]\w*)?» когда рядом корректирующий маркер. Не трогать общий `extract_grade` для обычного потока (риск over-match), только в ветке коррекции.
3. **`_extract_slots_from_turns`** (`:485`): брать ПОСЛЕДНИЙ матч grade в истории, а не первый (recency и в истории) — страховка, если коррекция всё же дала «10 класс» позднее.
4. **Перезапись + сброс**: при `detect_slot_correction` → перезаписать слот в `slot_map` (override, max confidence), убрать слот из `do_not_reask` на 1 ход (чтобы при неоднозначности переспросить), сбросить зависимый кэш rfk/контракта (и verifier-cache из Волны 6, если введён — добавить `slot_signature` в hash).
5. **Подтверждение клиенту**: при коррекции драфтер обязан коротко подтвердить («Поняла, ориентируюсь на 10 класс»), а не молча сменить. Hint в промпт драфтера (`dialogue_contract_pipeline.build_draft_prompt`).
6. Синхронизировать с `held_state._is_direct_correction` (`:143`) — расширить маркерами из п.1, чтобы held тоже чистил format/scope согласованно.

## Регресс-тесты

- «t1: 9 класс» → «t5: нет, я про 10» → grade=10, бот подтверждает «10 класс». (Сейчас остаётся 9 — целевой фикс.)
- «не 9, а 10», «передумала, 11 класс», «извините — на самом деле очно» (формат), «не математика, физика» (subject — частично уже работает через `_marker_is_explicitly_negated_before_correction`, проверить).
- **КОНТРОЛЬ:** обычный поток без коррекции не должен ложно срабатывать: «я про 10 класс хотел спросить» (первое упоминание, не коррекция) → grade=10 нормально, без «сброса»; «нет» как отказ от предложения (не коррекция слота) → слот не трогаем.
- 20-ходовой: коррекция на t12 после 11 ходов с grade=9 → корректно переключается.

## Риски

- Over-match: «нет» многозначно (отказ vs коррекция). Митигация: требовать комбинацию (корректирующий маркер + новое значение того же типа слота), не голое «нет».
- Расширение `extract_grade` может задеть обычный поток. Митигация: loose-вариант ТОЛЬКО в корректирующей ветке.
- Сброс кэша при коррекции взаимодействует с Волной 6 (verifier-cache) — согласовать сигнатуры.

# §7.3 — Rolling-summary + синхронизация окон

## Цель

Дать боту смысловую память длинного диалога (а не только слоты) и устранить рассинхрон окон контекста между слоями. Сейчас за пределами ~6-8 ходов держатся только слоты+латч; `conversation_summary_short` — слот-снимок, не смысл; drafter видит ~20 ходов, P0 — 3, intent — 6, ctx — 8.

## Что сейчас в коде (точно)

- Окна: `answer_safety_classifier.py:72` (`recent[-3:]`), `conversation_intent_plan.py:447` (`[-6:]`), `telegram_pilot_context_builder.py:297` (`[-8:]`), `dialogue_contract_pipeline.py:516` (drafter — полная склейка `conversation`), `MAX_TURNS=MAX_PROMPT_TURNS=20` (`dialogue_memory.py:25-26`), `DialogueTurn.text[:700]` (`:95`).
- `conversation_summary_short` = `_conversation_summary_short(slot_map, topic_focus, open_question)` (`:310`) — структурный снимок, не LLM-сжатие.
- `open_loop_summary` = `_open_loop_summary(open_question, risk_flags, pending_actions)` (`:311`).

## Архитектура (целевая)

1. **Rolling semantic summary.** Новый узел (LLM-вызов, можно gpt-5.4-mini, дёшево) или детерминированный сборщик: при `len(turns) > SUMMARY_THRESHOLD` (предлагаю 8) сжимать ходы СТАРШЕ последних K (K=6) в `conversation_summary_semantic` — 3-5 предложений: что спрашивал клиент, что подтверждено, какие открытые петли, эмоциональный тон. Хранить в `DialogueMemory` рядом с `conversation_summary_short` (не вместо — слоты остаются).
   - Failover: если LLM недоступен — детерминированный шаблон из слотов+open_loops (текущий `_conversation_summary_short`), не падать.
   - Подавать в `build_draft_prompt` (`dialogue_contract_pipeline.py:516`) ПЕРЕД последними K ходами: «Кратко о диалоге: <summary>. Последние реплики: <K ходов>».
2. **Синхронизация окон.** Ввести единую константу `CONTEXT_WINDOW` и единый помощник `recent_window(messages, n)`:
   - P0 (`answer_safety_classifier.py:72`): расширить `[-3:]` → общий значимый объём, НО P0 и так подстрахован латчем; минимально — оставить 3 для регексов, добавить чтение `conversation_summary_semantic` для контекста претензии across-turns.
   - intent (`:447`) и ctx (`:297`) — привести к одному окну (предлагаю 8) или к summary+последние K.
   - drafter — оставить ≤20, но подавать summary вместо «голых» 20 ходов на длинных диалогах (снижает размытие).
3. **Open-loops по существу.** Расширить `open_loop_summary` (`:311`) до явного списка «бот обещал X / не закрыл Y / клиент ждёт Z» с привязкой к ходам, переносить за пределы rolling-окна `[-5:]`.
