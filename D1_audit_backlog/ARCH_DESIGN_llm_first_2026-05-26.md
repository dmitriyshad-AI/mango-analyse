# Дизайн новой архитектуры: LLM в начале (понимание) → факты → черновик → верификация → X2. 2026-05-26

Claude, по предложению Дмитрия. Это целевая инверсия из ARCHITECTURE_root_flaw: умный слой ПОНИМАЕТ, детерминизм ВЕРИФИЦИРУЕТ выход.
Предложение Дмитрия по сути верное; ниже — точный пайплайн + критичные уточнения по безопасности.

## Пайплайн (на каждый ход; вся история диалога доступна)
1. **[Детерминированный P0-пре-гейт] (дёшево, defense-in-depth).** Быстрый детектор спора/жалобы/юр.угрозы/чарджбека →
   сразу manager_only + сухой P0-шаблон, STOP. Это страховка: P0 — высшая ставка, не полагаемся ТОЛЬКО на LLM, что он его заметит.
2. **LLM-вызов №1 — ПОНЯТЬ и ЗАПРОСИТЬ данные.** Вход: вся история + active_brand (из канала) + СПИСОК доступных fact_keys (каталог, БЕЗ значений).
   Выход (структурно): `{intent, is_p0, вопрос_клиента_пересказан, needed_fact_keys[], заметки}`.
   ЖЁСТКО: модель только НАЗЫВАЕТ нужные fact_keys, НЕ выдаёт значения фактов сама (иначе вернём выдумку на шаге 1). is_p0=true → manager_only.
   Это заменяет keyword-классификатор и машину held_state: понимание темы/контекста/отрицания/мультизначности делает модель.
3. **[Детерминированный ретривал].** Тянем needed_fact_keys из бренд-скоупного склада confirmed_facts → `facts[]` (только проверенные значения).
   Это убирает «вываливает соседний факт» по корню: модель просит что нужно, значения — из нашего верифицированного склада.
4. **LLM-вызов №2 — ЧЕРНОВИК.** Вход: вся история + `facts[]` (только этот бренд) + тон-гайд + правила (не раскрывать ИИ, не мешать бренды,
   использовать ТОЛЬКО facts[]). Выход: draft_text. Здесь живёт живой тон.
5. **[Детерминированный ВЕРИФИКАТОР выхода]** — крисп, бинарно, бесплатно: бренд-утечка (нет чужих токенов); заземление (каждое число/цена/дата ∈ facts[]);
   мета-утечка/fact_id/раскрытие ИИ; P0-в-тексте (нет обещаний возврата). PASS → отправляем. FAIL → собираем КОНКРЕТНЫЕ находки.
6. **LLM-вызов №3 — X2 РЕМОНТ (только при FAIL).** Вход: черновик + КОНКРЕТНАЯ находка верификатора («ты написал цену X, подтверждённая Y — поправь»;
   «убери упоминание другого бренда») + facts[]. Выход → ПЕРЕ-верификация. Ограниченные ретраи (1-2), иначе сухой безопасный fallback «уточнит менеджер».

## Твой вопрос: нужна ли детерминированная проверка без LLM? — ДА, и это правильное место для детерминизма
Детерминизм ПРОВАЛИЛСЯ на ПОНИМАНИИ (классификация темы), потому что понимание — открытое и нечёткое. Но ВЕРИФИКАЦИЯ жёстких
правил — крисп, бинарна и безопасно-критична, и тут детерминизм ЛУЧШЕ LLM:
- бренд-утечка = подстрочный поиск чужих токенов: мгновенно, 100% надёжно, 0 токенов. LLM-проверка была бы медленнее, платно и могла бы ПРОПУСТИТЬ.
- заземление чисел = «каждое число ∈ facts[]»: быстро, ловит выдумку. LLM мог бы не заметить.
- мета/раскрытие ИИ/P0-в-тексте = подстрочно.
Вывод: детерминизм в ПРАВИЛЬНОМ месте (проверка крисп-ограничений) — идеален и бесплатен. В НЕПРАВИЛЬНОМ (понимание) — хрупок.
А вот мягкое качество («ответил ли по сути, живой ли тон») — НЕ бинарно → это к X2/судье, не к детерминированной проверке.

## Что это даёт
- Убирает ВЕСЬ класс whack-a-mole (мискласс, отрицание, мультизначность, сосед-факт) — их решает понимание модели, а не регулярки.
- Та же гарантия безопасности (бренд/факты/P0/ИИ) — но на ВЫХОДЕ, верификатором, а не пред-фильтром, что кастрирует понимание.
- Упрощает код: keyword-классификатор + машина held_state как источник смысла уходят (остаётся тонкий P0-пре-гейт + верификаторы).
- Лечит тон: модель РАЗГОВАРИВАЕТ (с проверкой), а не заполняет слот.

## Честные оговорки и риски
- Стоимость/задержка: 2 вызова в норме, 3 при ремонте (сейчас 1). Для черновикового бота и «токенов не жалко» — приемлемо; замерить латентность на ход.
- Шаг 1 НЕ должен давать значения фактов — только запрашивать ключи. Иначе выдумка вернётся на входе.
- P0 — детерминированный пре-гейт ОБЯЗАТЕЛЕН как бэкстоп, не только LLM-флаг.
- Заземление-верификатор обязателен ОДНОВРЕМЕННО с свободой шага 2, иначе свобода = выдумка.
- Не-числовая выдумка (выдуманная формулировка без цифр) детерминированно ловится хуже — для неё верификатор может звать дешёвый LLM-чек «всё ли в этом ответе ∈ facts[]» как доп. слой (опционально).

## Миграция (не рвать рабочее)
Строить за флагом параллельно текущему пути; старый путь — дефолт до проверки. Мерить на ПОЛНОМ наборе (round-5+round-6), правило fab=0/hard=0.
Можно поэтапно: сначала №1 (понимание+запрос ключей) + ретривал + №2, со СТАРЫМИ выходными гейтами; убедиться, что класс мискласса ушёл;
потом перенести верификацию на новую схему и подключить X2-ремонт; в конце — выпилить keyword-классификатор/held_state как источник смысла.
Безопасную текущую базу при этом везти в пилот, не блокируя её переделкой.
