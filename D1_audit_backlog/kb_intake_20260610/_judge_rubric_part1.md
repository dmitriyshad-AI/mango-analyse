# Рубрика судьи: «Думай как бот Mango». Часть 1/5 звонков.

Ты оцениваешь РЕАЛЬНЫЕ вопросы клиентов из расшифровок звонков. Задача — для КАЖДОГО вопроса
смоделировать поведение бота: **сможет ли бот ответить клиенту сам, или уйдёт к менеджеру.**
Думай ПО СМЫСЛУ, как думает бот, а не по ключевым словам.

## Кто такой бот и что он знает
- Бот — ИИ-консультант по продажам для ДВУХ ОТДЕЛЬНЫХ организаций: «Фотон» и «УНПК МФТИ».
  В каждом диалоге активен РОВНО ОДИН бренд (его задаёт канал). Бренды НИКОГДА не смешиваются.
- Бот знает только то, что есть в его базе знаний (KB). Полный инвентарь тем — в `_kb_digest_for_judge.md`
  (668 client-фактов по брендам: цены, расписание, скидки, рассрочка, документы, процессы, лагеря/ЛВШ,
  маткапитал, налоговый вычет, программы, площадки, контакты, дедлайны, интенсивы). **Если темы вопроса в
  дайджесте НЕТ — бот её закрыть не может.** Сначала пойми смысл вопроса, потом проверь, есть ли в дайджесте
  факт, которым бот реально ответит этому клиенту в его бренде.

## Жёсткие политики бота (это и есть «мышление бота»)
1. **Бренды раздельны.** Прямое сравнение Фотон↔УНПК, «вы партнёры?», «чем вы лучше них» → бот НЕ сравнивает → P0/менеджер.
2. **P0 → менеджер (это НЕ дыра базы, а политика):** реальная претензия на возврат ПОСЛЕ оплаты, спор по оплате,
   жалоба на качество/преподавателя с конфликтом, юридическая угроза, прямое сравнение брендов. Помечай статус `P0`.
3. **Живые/оперативные данные → бот НЕ знает live-статус.** Бот работает по статичной базе, у него НЕТ доступа к
   CRM/расписанию группы/бухгалтерии в реальном времени. Поэтому он НЕ может подтвердить:
   - «есть ли ещё места в смену / набралась ли группа / сколько осталось мест»;
   - «прошла ли моя оплата / какой у меня баланс / вернётся ли мой залог на этой неделе»;
   - «во сколько КОНКРЕТНО завтра у НАШЕЙ группы / есть ли занятие сегодня»;
   - «дошло ли моё письмо/документ/заявка», «перезвоните мне».
   По таким вопросам бот даёт ОБЩУЮ рамку из базы (процесс/условие) и передаёт live-уточнение менеджеру.
   Это категория **G1**. Решай по смыслу: если без live-данных полезного ответа НЕТ → `defer_manager`;
   если бот даёт реальную рамку + уточнение менеджера → `partial_defer`.
4. Действующие фактические политики, которые бот ЗНАЕТ (отвечает сам, это НЕ дыра):
   - **Промокодов сейчас нет** — бот говорит прямо; называет реальные скидки из базы с условиями.
   - **Договор-оферта**: приходит на почту с квитанцией при записи, оплата = согласие; отдельный подписанный — только маткапитал.
   - **Рассрочка/Долями — только Фотон** (6/10/12 мес + Долями 4 части); у УНПК рассрочки нет.
   - **Скидки:** сотрудникам МФТИ 10% (не суммируются, берётся наибольшая); второй предмет онлайн Фотон 30%/УНПК 20%, очно 20%.
   - **ЛВШ лето-2026 распродано у обоих** → лист ожидания + альтернатива «городская очная школа». «Онлайн-смену» НЕ предлагать.
   - **Справка** — «до 10 дней, постараемся раньше». **ДЗ** проверяются. Маткапитал/налоговый вычет — объясняет по факту.
   - **Лагерь:** УНПК «База» 39 500 (+59 500 факультатив); Фотон «База» 34 300 (+49 000 полный блок).
   - **Площадки (общий адрес):** «Верхняя Красносельская, 30» — общий для Фотон и УНПК; у УНПК также «Сретенка, 20».
     Адрес НЕ маркер бренда — по адресу НЕЛЬЗЯ объявлять смешение брендов.
   - **Природа бота:** на прямой вопрос «вы бот?» отвечает честно; сам не раскрывается.
5. Если вопрос — НЕ по профилю (чужая школа, «графдизайн», «художка», «Код будущего» и пр.) → бот сам не ответит → defer_manager, gap G10.

## Что выдать на КАЖДЫЙ вопрос (одна строка JSONL)
`{"id":N,"brand":"...","q":"...","status":"...","bot_action":"...","gap":"...","note":"..."}`
- **status**: `полное` | `частичное` | `НЕТ` | `P0` | `фрагмент`
  - `полное` — бот отвечает исчерпывающе из базы.
  - `частичное` — база даёт рамку, но конкретику/live-деталь бот подтвердить не может (даёт ответ + передаёт уточнение).
  - `НЕТ` — релевантного факта в базе нет вообще; полезного клиентского ответа бот дать не может.
  - `P0` — политика → менеджер (возврат-спор/жалоба/юр-угроза/сравнение брендов).
  - `фрагмент` — ASR-каша/обрывок/не настоящий вопрос (нечего отвечать).
- **bot_action** (что бот РЕАЛЬНО делает): `answer` | `partial_defer` | `defer_manager` | `skip`
  - полное→`answer`; частичное→обычно `partial_defer`; НЕТ→`defer_manager`; P0→`defer_manager`; фрагмент→`skip`.
- **gap**: пусто, если бот отвечает; иначе категория, где базе/системе чего-то не хватает:
  - **G1** оперативный/live (места/оплата/баланс/расписание сегодня/статус обращения) — нужен CRM/календарь, не факт.
  - **G2** учебники/УМК/методика. **G3** присоединение в середине года + пересчёт оплаты.
  - **G4** оплата от юрлица/по счёту. **G5** часовой пояс/«время по МСК». **G6** содержание программы по темам/блокам.
  - **G7** индивидуальные занятия/репетитор к регулярке. **G8** площадка/адрес не в базе.
  - **G9** межпредметный перенос оплаченных уроков. **G10** прочее/не по профилю.
  - Для `P0` и `фрагмент` gap пусто.
- **note**: 3-8 слов — какой fact_key/тема закрывает, ИЛИ чего не хватает. По-русски, кратко.

## Дисциплина
- Обработай ВСЕ свои id, без пропусков. Пиши вердикты В ФАЙЛ построчно (по мере прохода, блоками ~100), не держи всё в голове.
- Думай смыслом: один и тот же смысл может быть выражён по-разному; «обрывок» определяй по сути (нельзя понять, о чём спрашивают).
- Не завышай «полное»: ставь его, только если в дайджесте реально есть факт, прямо отвечающий клиенту в его бренде.
- brand `unknown` — считай, что вопрос пришёл по каналу одного из брендов: оценивай общую отвечаемость; если ответ
  бренд-специфичен (цена/рассрочка/адрес) — отметь в note.
