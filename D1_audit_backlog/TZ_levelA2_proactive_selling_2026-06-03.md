## Железный принцип (не нарушать)
- **Проактивность ≠ давление.** Эталон — playbook лучших менеджеров (Тропина/Козлова): снятие риска, гашение
  тревоги, честная срочность, чёткий следующий шаг. НЕ «купите сейчас», НЕ искусственный дефицит, НЕ обещание
  результата. Существующие гейты давления/гарантии — сверху, не ослаблять.
- **Один шаг за ход.** Максимум ОДИН вопрос/призыв в сообщении. Не заваливать человека пятью вопросами. Не повторять
  один и тот же вопрос/предложение, если уже спросили или человек отказал.
- **P0 перебивает продажу ВСЕГДА.** Возврат-претензия/жалоба/спор оплаты/гарантия/юр.угроза → менеджер, никакой
  проактивности. (Связано: latch P0.)
- **Бренд заперт.** Все вопросы/предложения — в рамках active_brand. Никогда не сравнивать и не звать в другой бренд.
- **Черновиковая стадия: «записать»/«передать менеджеру» = ЗАХВАТ намерения + тёплый хендофф, НЕ автозапись в CRM.**
  Бот НЕ пишет в AMO/Tallanto без отдельного OK Дмитрия. Бот НЕ заявляет «я вас записал» — он предлагает передать
  менеджеру, который оформит.
- **Привязка к пилоту:** на Этапах 0-2 пилота КАЖДОЕ проактивное сообщение (вопрос/предложение/просьба телефона) —
  это ЧЕРНОВИК, который одобряет менеджер (как все ответы). Автономная проактивность — только на поздних этапах 3-4
  после слоя автоотправки. Эта фича добавляет проактивность в черновик, а не разрешает боту слать самому.
- **Только подтверждённые факты.** Возражения закрываются ФАКТАМИ из базы (рассрочка/скидка/расписание — brand-
  correct), не выдумкой. Нет факта по сути → честно к менеджеру.
- **PII (телефон/время):** захватываем для тёплого хендоффа менеджеру, не эхо-повтор в клиентском тексте, не в CRM
  без OK Дмитрия.
- **Всё за флагом `TELEGRAM_A_PROACTIVE` (default OFF)** + форматирование за `TELEGRAM_A_RICH_FORMAT` (default OFF).
  Своя приёмка + регрейд по сырью до включения.

### 6 правок Кодекса (ПРИНЯТЫ — обязательны для Фазы 2; переопределяют полуфабрикаты ниже)
1. **selling_step планнера НЕ авторитетен.** Планнер только ПРЕДЛАГАЕТ; финальный шаг решает детерминированная
   политика из существующих `selling.*` + route + факты + память. (Уже так в Полуфабрикате 2: planner_step не драйвит.)
2. **НЕ заводить `buying_readiness`-enum.** Использовать существующий `readiness ∈ exploring|comparing|ready` +
   `exit_signal`; маппинг держать в policy-слое. В лестнице: `ready`≡hot, `comparing`≡warm, `exploring`≡cold.
3. **`fit_check` — только фактовая рамка, НЕ суждение о ребёнке.** Нельзя «курс ВАМ подходит». Можно: «по фактам это
   курс для X класса/формата; уровень и группу сверит менеджер». (Усиливает страж general_advice/диагноза.)
4. **Возражение без факта ≠ всегда менеджер.** Если основной вопрос УЖЕ отвечён — просто НЕ добавлять продающий
   довесок (обычный ответ, `none`). Хендофф к менеджеру — ТОЛЬКО если без этого факта нельзя ответить на сам вопрос.
   (Прямо против over-handoff.)
5. **`recent_ignored` — из ЯВНОЙ устойчивой памяти диалога** (последние proactive-шаги + ответы клиента), НЕ из
   «модель помнит». Источник детерминированный.
6. **PII: телефон/время не эхоить клиенту, маскировать в логах.** Если телефон уже известен из канала/Wappi — НЕ
   спрашивать номер заново, спросить только удобное ВРЕМЯ.

Порядок реализации (по Кодексу, принят): **A2.1** offer_callback + захват контакта/времени + гейты (fake-enrollment/
PII/emoji/анти-барраж), флаг OFF → **A2.2** handle_objection/qualify/fit_check строго через fact-availability →

## ФАЗА 2 — реализация
Архитектура: планнер(selling_step + objection_type + готовность + чего не хватает) → детерминированная политика
(выбирает ОДИН следующий шаг по лестнице) → композер (тёплый рендер шага + форматирование) → гейт (давление/выдумка/
бренд/один-вопрос/смайлики/«не записал ли уже») → захват контакта → лог.

### Таксономия проактивных шагов (selling_step)
- `qualify` — задать ОДИН уточняющий вопрос (класс / предмет / онлайн-очно / цель), когда не хватает данных, чтобы
  посоветовать. Пример: «Подскажите, в каком классе ребёнок — подберу точнее?»
- `fit_check` — ТОЛЬКО фактовая рамка, НЕ суждение о ребёнке (правка 3): «по фактам это курс для 9 класса/онлайн;
  уровень и группу сверит менеджер». НИКОГДА «курс ВАМ подходит».
- `elicit_objection` — мягко выяснить сомнение: «Что для вас важно при выборе?» / «Что останавливает?» (ОДИН вопрос).
- `handle_objection` — закрыть НАЗВАННОЕ сомнение фактом: цена→рассрочка/скидка (brand-correct); время→расписание;
  подходит ли→совпадение курса; возврат/претензия→менеджер (P0). Только факт, без выдумки.
- `offer_enroll` — «Хотите, передам менеджеру — он оформит запись на этот курс?» (захват намерения + хендофф; НИКОГДА
  «я вас записал»).
- `offer_callback` — «Давайте менеджер свяжется в удобное время — подскажите телефон и когда удобно?» (захват лида).
- `soft_close` — exit_signal / «просто смотрю» → не давить, оставить дверь открытой: «Хорошо, я рядом — пишите, если
  понадобится».
- `none` — P0 / нет уместного шага → обычный путь (факт/менеджер), без проактивного довеска.

### Полуфабрикат 1 — поля планнера в understand() (там же, где selling-поля; 0 новых вызовов)
```
"selling_step":      "qualify"|"fit_check"|"elicit_objection"|"handle_objection"|"offer_enroll"|"offer_callback"|"soft_close"|"none"
"objection_type":    "price"|"time"|"fit"|"trust"|"refund"|"none"
"readiness":         "exploring"|"comparing"|"ready"  # СУЩЕСТВУЮЩИЙ (правка 2): НЕ заводить buying_readiness; в лестнице ready≡hot, comparing≡warm, exploring≡cold
"missing_qualifier": "class"|"subject"|"format"|"goal"|"none"   # чего не хватает для рекомендации
"contact_shared":    true|false               # человек уже дал телефон/согласие на звонок
```
Промпт-добавка (кратко): «Определи следующий продающий шаг как добрый менеджер: если не хватает данных для совета —
qualify (ОДИН вопрос); подтвердить подходит ли — fit_check; человек сомневается — elicit/handle_objection; готов —
offer_enroll/offer_callback; уходит/«просто смотрю» — soft_close; претензия/возврат/жалоба — none (менеджер). objection_type
по сути сомнения. НИКОГДА не дави, не обещай результат, не выдумывай.»
**Кодекс (Фаза 1):** `buying_readiness` и `exit_signal` уже частично есть в коде (readiness/exit из волны A,
коммит eaecefca). НЕ плодить новый enum — переиспользовать существующие сигналы; если они бинарные, смаппить
(ready→hot, exit→browsing/strong) и зафиксировать соответствие в отчёте. `objection_type`/`missing_qualifier`/
`selling_step` — новые поля той же understand().

### Полуфабрикат 2 — детерминированная лестница политики (выбирает ОДИН шаг; не доверяем планнеру слепо)
```python
import re
# прямой запрос двигаться к оформлению/связи — ловим явно, не ждём readiness=hot
_PROCEED_RE = re.compile(
    r"как (записать|записаться|оплат|внести|поступить)|хочу записать|готов записать|запишите|"
    r"как связаться|свяжите|оставлю (телефон|номер)|можно оплатить|где оплатить", re.I)

def _resolve_selling_step(*, objection_type, readiness, missing_qualifier, has_p0, exit_signal,
                          contact_shared, already_asked, explicit_proceed, qualifier_unlocks_fact,
                          objection_fact_available, fit_confirmed, stalled_asked, recent_ignored) -> str:
    # --- РЕСПОНСИВНЫЕ шаги (реакция на человека) — приоритет, анти-усталость их НЕ глушит ---
    if has_p0:
        return "none"                          # P0 перебивает продажу всегда → менеджер
    if exit_signal in ("strong", "browsing"):
        return "soft_close"                    # не давим на уходящего (тёплая дверь открыта)
    if objection_type == "refund":
        return "none"                          # возврат/претензия → менеджер, не «закрываем» сами
    if objection_type in ("price", "time", "fit", "trust"):
        # есть факт active_brand → закрыть сомнение фактом; НЕТ факта → "none" = обычный ответ БЕЗ довеска (правка 4),
        # менеджер ТОЛЬКО если без факта нельзя ответить на сам вопрос (решает обычный путь, не данный слой)
        return "handle_objection" if objection_fact_available else "none"
    if explicit_proceed:                       # человек прямо просит двигаться → уважаем (мимо анти-усталости)
        return "offer_enroll" if contact_shared else "offer_callback"
    # --- ИНИЦИАТИВНЫЕ шаги (мы предлагаем) — глушатся анти-усталостью ---
    if recent_ignored >= 2:
        return "none"                          # 2 наших шага подряд проигнорированы → не давим больше
    if readiness == "hot":
        return "offer_enroll" if contact_shared else "offer_callback"
    if missing_qualifier != "none" and missing_qualifier not in already_asked and qualifier_unlocks_fact:
        return "qualify"                       # спрашиваем ТОЛЬКО то, что разблокирует реальный факт (не тупик)
    if readiness == "warm":
        if not fit_confirmed:
            return "fit_check"                 # сначала мягко подтвердить, что курс подходит
        if not stalled_asked:
            return "elicit_objection"          # подтвердили, но топчемся → ОДИН раз мягко выяснить сомнение
        return "none"
    return "none"                              # иначе обычный ответ без проактивного довеска
```
`explicit_proceed = bool(_PROCEED_RE.search(question_text))`. `qualifier_unlocks_fact` — True только если по
уточнению реально есть факт в KB active_brand (иначе вопрос тупиковый → менеджер/обычный ответ).
`objection_fact_available` — нужный факт active_brand реально извлечён; без факта НЕ выдумываем и НЕ форсим менеджера:
«none» = обычный ответ без продающего довеска (правка 4), хендофф лишь если сам вопрос без факта неотвечаем. `fit_confirmed`/`stalled_asked` — состояние диалога
(уже подтвердили подходит / уже выясняли сомнение — анти-повтор). `recent_ignored` — счётчик подряд
проигнорированных НАШИХ инициативных шагов (анти-усталость). `already_asked` — какие qualify уже задавали.
Контакт НЕ просить на холодном/первом касании. `planner_step` НЕ драйвит лестницу (логируем для сверки, решает
детерминированная политика — defense in depth).

### Полуфабрикат 3 — композер: рендер шага + форматирование
Композер активируется при флаге ON и `selling_step != "none"`: к обычному ответу из факта добавляет ОДИН проактивный
шаг тёплым тоном. Промпт-добавка по шагу (примеры выше). Жёстко: ОДИН вопрос; без давления/обещаний; в рамках бренда;
из факта.
Форматирование (флаг `TELEGRAM_A_RICH_FORMAT`, применяется ко ВСЕМ gen-ответам):
- короткие абзацы, разделённые пустой строкой; мысль — абзац;
- максимум ОДИН смайлик и ТОЛЬКО в тёплом несерьёзном контексте (Дмитрий: «очень небольшое количество»); в серьёзном
  (возврат/жалоба/менеджер/P0) — без смайликов;
- не «простыня»: 2-4 коротких абзаца лучше одного длинного.
Промпт-добавка: «Раздели ответ на короткие абзацы пустой строкой. Один смайлик уместен в тёплом контексте, не больше;
в серьёзных темах смайликов нет. Заверши ОДНИМ понятным следующим шагом.»

### Полуфабрикат 4 — захват контакта (телефон + удобное время) → тёплый хендофф
```python
import re
_PHONE_RE = re.compile(r"(?:\+7|8|7)?[\s\-\(]?\d{3}[\s\-\)]?\d{3}[\s\-]?\d{2}[\s\-]?\d{2}")
def _extract_contact(text: str) -> dict | None:
    m = _PHONE_RE.search(text or "")
    if not m:
        return None
    return {"phone": m.group(0).strip()}        # время добираем из реплики/следующего хода (free text)
```
При offer_callback/offer_enroll, если человек дал телефон → структура `{phone, preferred_time, course_interest,
active_brand}` → тёплый хендофф менеджеру (как существующий warm-handoff, draft-стадия). НЕ писать в CRM без OK
Дмитрия. В клиентском тексте телефон не эхо-повторять (подтвердить ОДИН раз: «Передал менеджеру, он свяжется …»).

### Полуфабрикат 5 — гейт проактивной безопасности (расширить выходную верификацию)
Сохранить ВСЕ существующие проверки (бренд/давление/гарантия/выдумка-числа/leak) — они сверху. Добавить:
```python
_FAKE_DONE_RE = re.compile(r"я (вас )?записал|вы записаны|запись оформлена|оформил запись|записал на курс", re.I)
def _verify_proactive(text, *, selling_step, context_tag) -> tuple[bool, str]:
    # 1) не заявлять оформление как сделанное (бот только предлагает передать менеджеру)
    if _FAKE_DONE_RE.search(text or ""):
        return False, "fake_enrollment_claim"
    # 2) не больше ОДНОГО вопроса в проактивном ходе (анти-барраж)
    if selling_step in ("qualify", "fit_check", "elicit_objection") and (text or "").count("?") > 2:
        return False, "too_many_questions"
    return True, ""
```
Давление/искусственный дефицит — существующий `_SELLING_PRESSURE_RE` + выдумка-гейт: дефицит/«мест почти нет» —
ФАКТИЧЕСКОЕ утверждение, проходит только если грунтовано (ЛВШ-дефицит — реальный факт, его НЕ блокировать; выдуманный
дефицит другого курса — блок как негрунтованное). **Фолбэк при False:** откат на обычный (непроактивный) ответ или
хендофф (НЕ авто-правка текста). Смайлики — Полуфабрикат 6.
`selling_step` и `context_tag` тред'ятся через `AnswerContract` так же, как существующие selling-поля (gen-стек уже
их тред'ит). `context_tag` ставит политика (p0/refund/manager_only/warm/neutral) — он же управляет стражем смайликов.

### Полуфабрикат 6 — форматирование: страж смайликов
```python
_EMOJI_RE = re.compile("[\U0001F300-\U0001FAFF\U00002600-\U000027BF\U0001F900-\U0001F9FF\U0001FA70-\U0001FAFF]")
_SERIOUS_TAGS = {"p0", "refund", "complaint", "manager_only", "legal", "guarantee"}
def _enforce_format(text: str, *, context_tag: str, max_emoji: int = 1) -> str:  # «очень небольшое количество»
    if context_tag in _SERIOUS_TAGS:
        return _EMOJI_RE.sub("", text or "").strip()        # серьёзный контекст → без смайликов
    found = _EMOJI_RE.findall(text or "")
    if len(found) > max_emoji:                              # срезать лишние смайлики сверх лимита
        out, cnt = [], 0
        for ch in (text or ""):
            if _EMOJI_RE.match(ch):
                cnt += 1
                if cnt > max_emoji:
                    continue
            out.append(ch)
        text = "".join(out)
    return text
```

### Полуфабрикат 7 — анти-повтор / анти-барраж / анти-усталость (детерминированно)
Состояние диалога (тред'ится, как память диалога): `already_asked` (темы qualify), `fit_confirmed`, `stalled_asked`,
`recent_ignored` (счётчик подряд проигнорированных НАШИХ инициативных шагов).
- `already_asked` — не задавать тот же уточняющий вопрос дважды.
- Отказ на offer_callback/offer_enroll → не повторять то же предложение следующим ходом (понизить до soft_close/
  обычного ответа); инкремент `recent_ignored`.
- `recent_ignored >= 2` → инициативные шаги глушатся (лестница вернёт none); РЕСПОНСИВНЫЕ (handle_objection на новое
  сомнение, explicit_proceed, soft_close, P0) продолжают работать.
- Один проактивный шаг за ход (лестница выбирает ровно один).

### Полуфабрикат 8 — лог для регрейда (по аналогии с Уровнем А)
Логировать: выбранный selling_step, objection_type, был ли захват контакта, вердикт гейта. → Клод/менеджер регрейдят
проактивность по сырью (не давит ли, в рамках ли факта/бренда, уместен ли шаг).

### Полуфабрикат 9 — NEG-матрица (детерминированно; приёмка)
- **давление (главный NEG):** попытка композера «надо брать сейчас, потом дороже» (негрунтовано) → `_SELLING_PRESSURE_RE`
  → блок/откат. Проактивность ≠ давление.
- **искусственный дефицит:** «мест почти нет» на курсе без факта → выдумка-гейт блок. ЛВШ-дефицит (реальный факт) →
  НЕ блокировать.
- **«я вас записал»:** композер заявил оформление сделанным → `_FAKE_DONE_RE` блок (бот только передаёт менеджеру).
- **возражение без факта (правка 4):** сомнение price/time/fit без факта active_brand → `objection_fact_available=False`
  → selling_step=none = обычный ответ БЕЗ продающего довеска (НЕ форс-менеджер); менеджер ТОЛЬКО если сам вопрос без
  этого факта неотвечаем; бот НЕ придумывает условие.
- **анти-усталость:** человек проигнорировал 2 наших инициативных шага → следующий ход без проактивного довеска
  (none); но если он задал новое возражение/прямой запрос — реагируем (респонсивные шаги не глушатся).
- **барраж:** 5 вопросов в ходе → `too_many_questions` → откат к одному шагу.
- **P0 перебивает:** возврат-претензия в середине продажи → selling_step=none, менеджер; никакой offer_enroll.
- **возражение-возврат:** objection_type=refund → none (менеджер), бот не «закрывает» сам.
- **бренд:** в боте Фотона не предлагать курс/условие УНПК; cross-brand вопрос → каноничная фраза разделения.
- **бренд + возражение-цена (точечно):** Фотон-бот цену-возражение закрывает рассрочкой Фотона (6/10/12 + Долями),
  НИКОГДА скидкой УНПК «14% за год»; УНПК-бот — честно «рассрочки нет» + скидка 14%, без рассрочки Фотона. Факты —
  строго active_brand.
- **диагноз ребёнка (стык с Уровнем А):** «потянет ли мой?» → не fit_check-диагноз, а менеджер (страж general_advice).
- **PII:** дал телефон → захват в хендофф, не эхо-повтор, не в CRM; подтверждение один раз.
- **форматирование:** серьёзный контекст (возврат) → 0 смайликов; тёплый → ≤1; абзацы разделены пустой строкой.
- **отказ → не повтор:** человек сказал «не сейчас» на callback → следующий ход без того же предложения.
- **флаг OFF:** TELEGRAM_A_PROACTIVE OFF → поведение как сейчас (без проактивного довеска); RICH_FORMAT OFF → формат
  как сейчас.

## Фазирование по риску (рекомендация — реализовывать в этом порядке)
- **А2.1 (первым — макс. польза, мин. риск):** offer_callback + захват контакта + форматирование. Это структурный
  тёплый хендофф (телефон+время→менеджер) — почти нулевой риск, прямая ценность (лид не теряется в нерабочее время).
- **А2.2:** qualify + fit_check + elicit/handle_objection. Переиспользует существующую objection-машину; средний риск
  (давление/диагноз) — закрыт гейтом + стражем general_advice.
- **А2.3:** offer_enroll. Только после ясности «захват намерения → менеджер», не автозапись.

## Приёмка Уровня А2
pytest + NEG-матрица зелёные (флаги ON и OFF). Малый прогон (Claude соберёт): продающие диалоги (квалификация →
возражение → предложение записать/связаться) + адверсариальные (выманить давление/дефицит/«запиши меня сам») +
форматирование (серьёзный vs тёплый контекст). Регрейд по сырью: проактивность уместна и НЕ давит, шаги в рамках
факта/бренда, контакт захвачен корректно, P0 цел, смайлики уместны. Только после — катить (флаги default OFF до
приёмки). На пилоте мерить: доля диалогов с захваченным лидом, unedited_rate проактивных черновиков.
**Важно для метрик:** offer_callback/offer_enroll — это ПРОАКТИВНЫЙ хендофф-захват лида (ценность, цель продажи), НЕ
уклончивый over-handoff (уход от ответа при наличии факта). Считать их ОТДЕЛЬНО — иначе полезный захват испортит
показатель over-handoff.
