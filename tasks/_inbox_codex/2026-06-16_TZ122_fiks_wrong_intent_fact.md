# ТЗ-122 (после круга аудитора) — калибровка `wrong_intent_fact` (главный рычаг хвоста over-handoff)

- **Дата:** 2026-06-16. Заказчик: Дмитрий. Постановщик: Claude #1 (+ аудитор по коду). Исполнитель: D3 (живой бот). Основание — разбор C0 по сырью: главный понижающий сигнал = `wrong_intent_fact` (48), демоутит ВЕРНЫЕ ответы.
- **Этап:** строим за флагом default OFF сейчас; ВКЛЮЧАЕМ только после стабилизации пары model_p0+deal_action в бою (не менять две вещи разом).

## Диагноз (уточнён аудитором по коду)
- `wrong_intent_fact` рождается НЕ в выходном гейте (там его нет в `GATE_BLOCKING_CODES`, он лишь `warn`), а в `_wrong_intent_fact_findings` (`dialogue_contract_pipeline.py:7065`); реальный демоут — в драфт-пайплайне через провал cite-only-recover (`:5855`). Критик (`semantic_output_verifier`) НЕ виноват (downgrade ~10, в основном pass) — НЕ трогаем.
- **Это ТРИ разных правила, калибровать раздельно:**
  1. **Адрес** (`:7079`): `not _asks_address(...) and _draft_uses_address_fact(...)`. `_asks_address` ловит узко (`адрес|площадк|где вы|где находит|куда ехать`). Вопрос «очные курсы 7 класс **где**?» → не матчится → ЛОЖНЫЙ wrong_intent_fact, хотя адрес — законный ответ.
  2. **Лагерь/ЛВШ** (`:7086`): демоут, если контракт не пронёс «лагерь/смена». Кейс «сколько стоит смена → 34 300/49 000» демоутится из-за того, что intent_text ПОТЕРЯЛ «лагерь» → это симптом РЕТРИВЕРА/КОНТРАКТА, не гейта.
  3. **Контактные часы как расписание** (`:7072`): семантически ВЕРНО (10:00–18:00 «на связи» ≠ дни занятий) → НЕ ТРОГАТЬ.

## Шаг 0 (обязательно перед фиксом)
Подтвердить, что на ЖИВОМ direct-path (C0) `wrong_intent_fact` рождается именно в `_wrong_intent_fact_findings` (аудитор нашёл это место; убедиться, что direct path его вызывает, а не отдельный верификатор). Если место иное — чинить там.

## Фикс (инверсия с негатива на ПОЗИТИВНУЮ проверку покрытия)
- **Ядро вопроса покрыто** = факт, использованный в черновике, МАТЧИТСЯ на `needed_fact_keys` под-вопроса того же `question_type`/scope (механика `_has_exact_retrieved_answer_part`/`_matched_scope_fact_keys_for_required_key` — уже в коде). Если матч есть → `wrong_intent_fact` НЕ выставляем, даже если узкий regex вопроса не сработал. Это СУЖЕНИЕ, не ослабление.
  1. **Адрес-ветка:** срабатывать только если адрес НЕ матчится ни на один `needed_fact_key` под-вопроса (тип ≠ location/где/куда); расширить `_asks_address` хотя бы на одиночное «где … (курсы|занятия|приходить|пробное)».
  2. **Лагерь-ветка:** демоут только при РАССОГЛАСОВАНИИ scope (`_camp_scope_from_fact` ≠ `_camp_scope_from_contract`) или полном отсутствии лагерного интента. Параллельно — задача замера C2: почему контракт теряет «лагерь» в intent_text (это корень, не маскировать правилом).
  3. **Контактные часы:** оставить как есть (отдельным кодом/подкодом, чтобы калибровка двух других её не задела).
- **Частичный ответ** (адрес есть, программа-по-классу не заземлена): отдавать заземлённое ЯДРО + явный переспрос недостающей детали (переиспользовать паттерн `_class_schedule_publication_answer`), НЕ дофантазируя число/программу. P0/refund/жалоба — ВСЕГДА полный увод (через `_cite_only_recover_blocked`).

## Не ослаблять (это ловит реальные выдумки)
`derived_product_claim`, `fact_grounding`, `unsupported_promise`, `unsupported_entity`, числовой/scope-гард, P0-пол, бренд-разделение — вне изменяемого правила, остаются как есть. Новый «частичный ответ» обязан проходить через тот же `classify_answer_safety`.

## Замер / приёмка
- Флаг default OFF; OFF = байт-в-байт текущий main (NEG-паритет).
- Раздельный счётчик демоутов по трём веткам (адрес/лагерь/контактные-часы) до и после; реальные демоуты «бот хотел→ушёл» (база 32); число wrong_intent_fact (база 48); send-as-is ПО ХОДАМ; llm_calls_total.
- **POS (должны перестать уходить):** «сколько стоит смена в лагере» (с лагерным контрактом И без); «учебный год/даты старта»; «где очные курсы 7 класс» → адрес + переспрос про программу.
- **NEG-паритет (обязан остаться демоут):** спросили цену → дали расписание/адрес; контактные часы как дни занятий; факт другого scope лагеря (residential vs city); выдуманное число/программа без факта (ловит derived_product_claim — проверить, что не ослаб).
- Микро-набор ~10-15 POS + 10-15 NEG для итерации; полный сет только на приёмке; `--parallel 4`; регрейд по СЫРЬЮ; мой регрейд.
- **СТОП/откат:** любой рост `derived_product_claim`/`fact_grounding`/`unsupported_entity` или хоть один NEG-кейс, прошедший автономным ответом.

## Связь с C2
