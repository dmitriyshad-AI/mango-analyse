# Стратегия связки 5 диалогов Кодекса (оркестрация)

Пин: git `48935d60`. Цель — двигаться быстрее параллельно, но без коллизий и без переусложнения.
Главный принцип: **один диалог = одна зона файлов**; качество доказываем на чистом holdout, ширину не строим вперёд.

## Карта диалогов → зона файлов → задача

| # | Диалог | Зона файлов (не пересекать) | Задача сейчас | Статус приоритета |
|---|---|---|---|---|
| 1 | **MAIN** (разработка+тесты) | `src/mango_mvp/channels/`, bot-тесты | Добить 3 пункта качества → честный замер (targeted16→holdout→full v8) → потом подключить timeline | **ГЛАВНЫЙ, блокирует ширину** |
| 2 | **Звонки + Mango API** | `services/`, `quality/`, `maintenance/`, pipeline-скрипты, `stable_runtime` (read) | Рутинная дозагрузка/обработка + лёгкая гигиена. НЕ вкладываться: пайплайн «достаточно зрелый» | низкий, поддержка |
| 3 | **Улучшения + SaaS** | `productization/`, CI-конфиг, общие доки | **ЗАМОРОЗИТЬ** SaaS/agent/mail (parked, снять с обязательного прогона), один roadmap | средний, разовый |
| 4 | **Историч. каналы → KB** | `knowledge_base/`, KB-релизы/данные, candidate-очереди | Гигиена KB (smoke v6.3 + один источник фактов) + обогащение через approval (не авто) | средний |
| 5 | **TG-воронка** | `product_data/` воронка/playbook, доки воронки | Спроектировать путь до продажи как ПРИНЦИПЫ+примеры (для генерации бота), НЕ код-ветки | средний, дизайн |

## Правила против коллизий (грязное дерево + параллельные потоки)

1. **`channels/` трогает только MAIN.** D5 (воронка) проектирует поведение, но НЕ редактирует channels — отдаёт спеки/корпус, которые MAIN потребляет.
2. **Критичные темы (возврат/жалоба/суд) — только в MAIN** (единый классификатор). MAIN ведёт ЕДИНЫЙ артефакт-источник `p0_recall_spec` (жёсткий список + проверочный и безобидный наборы). D4 НЕ заводит свой список — ссылается/импортирует этот артефакт; один и тот же проверочный набор проходит и в коде, и в триггерах базы.
3. **Сборка/данные KB — только D4.** MAIN читает готовый bot-pack, не правит KB.
4. **SaaS/productization — только D3.** Никто другой туда не пишет.
5. Перед коммитом каждый диалог: только файлы своего блока; не откатывать чужое; отдельные коммиты по категориям.
6. Удаления/архив/чистка runtime — только с подтверждением Дмитрия (по AGENTS.md).

## Последовательность (что от чего зависит)

- D1 — критический путь: пока качество не доказано на holdout, ширина не открывается.
- D2/D3/D4/D5 — могут идти параллельно D1 (разные зоны), но их РЕЗУЛЬТАТЫ вливаются в бота только после замера D1:
  - D4 (KB-гигиена) — можно сразу; новый bot-pack MAIN подхватит при следующем прогоне.
  - D5 (воронка/playbook) — спеки готовятся параллельно; few-shot/принципы заходят в MAIN после базового замера, по одной теме, без хардкода.
  - timeline-подключение — внутри MAIN, после holdout.
- Полный v8 и расширение пилота — после прохождения D1 + чистого holdout.

## Что вынести на мнение Кодекса (главный диалог)

1. Согласен ли с зоной-на-диалог и правилами коллизий? Где видит реальный риск пересечения в грязном дереве?
2. Согласен ли «заморозить SaaS» (D3) — или есть незаменимая зависимость бота от productization (по аудиту — нет, 0 импортов)?
3. Реалистична ли последовательность «сначала 3 фикса + замер, потом ширина»?
4. Как лучше синхронизировать P0 между MAIN (код) и D4 (KB-триггеры), чтобы не разъехались?
5. Что из parked-модулей он считает нужным включить раньше/позже, чем предложено.

## Анти-переусложнение (общий предохранитель на все диалоги)

Метрика «строки-без-потребителя не растут»; число хардкод-веток падает; каждый новый тест — property-типа,
строковый freeze — только для safety/recall. Если задача «требует» новую ветку/шаблон/строковый assert на
зелёной теме — остановиться и искать общий механизм.
