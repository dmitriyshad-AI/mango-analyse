Источник: 6 субагентов (по 10 идей в зоне) + 10 стратегических от Claude = 70 идей.
Привязаны к реальному коду v2. Усилие: S/M/L. Риск: низ/сред/выс.
Ранжирование (топ) — в конце файла, после §8.

## §1. ТОН И ЧЕЛОВЕЧНОСТЬ (T)
- **T1. Узкополосное тепло (только оболочка).** Тёплый промпт копирует факт-предложения дословно, меняет лишь зачин/связки/финал. → меньше отклонений по fact_drift, тепло чаще применяется. S/низ.
- **T2. Двухпроходное тепло с откатом на «мягкий».** Если тёплый отклонён — второй ужатый проход «смягчи только зачин/финал», потом откат. → меньше холодных откатов. M/низ.
- **T3. Ротация сухой P0-фразы по хешу.** Пул из 4-5 узких формулировок, выбор по хешу диалога+ход. → нет дословного повтора «Приняли обращение». S/низ. *(внедрено частично)*
- **T4. Тёплый хендофф = эмпатия+срок+шаг.** Каркас P0-фразы: признание ситуации → кто/когда вернётся → что сделать сейчас. → хендофф как забота. M/сред.
- **T5. Один хороший вопрос вместо пустого хендоффа.** Не хватает 1 слота и тема не P0 → задать ровно один уточняющий вопрос. → часть over_handoff → живой диалог. M/сред.
- **T6. Регистр ответа по client_state.** Контракт ставит ярлык (тревога/спешка/сомнение/деловой), тёплый слой подстраивает тон. → ответ «в тон». M/низ.
- **T7. Антишаблон: запрет повтора зачина в диалоге.** Передавать прошлые зачины, требовать иначе. → уходит ощущение скрипта. S/низ.
- **T8. Few-shot по состоянию, не только теме.** Ключ примера = тема+возражение/состояние. → черновик живее на входе. M/низ.
- **T9. Правило соразмерности длины.** Короткий вопрос → 1-2 фразы, не три абзаца. → меньше правок «слишком длинно». S/низ.
- **T10. Лексикон формы без фактов.** Разрешённые человеческие связки/микро-подтверждения («смотрите», «да, бывает»). → теплота без соблазна добавить факт. S/низ.

## §2. ПОНИМАНИЕ И МАРШРУТ (U)
- **U1. Детектор мягкого возврата как роль.** refund_frame = dispute|policy_curiosity|hypothetical; ЛЮБОЙ → manager_only. → «возвраты бывают?» не утекает в договор+email. M/низ.
- **U2. Тематический замок возврата против де-эскалации.** Защёлкивать тему возврата, снимать только manager-release. → «это не обращение» не разворачивает маршрут. S/низ.
- **U3. Маршрут уважает intent, а не наличие фактов.** manager_only/is_p0 НИКОГДА не становится автономным из-за случайных фактов. → системный фикс. S/сред. *(внедрено)*
- **U4. Гейт соответствия факта теме.** Мини-проверка «факт X отвечает на под-вопрос Y». → «правила поведения» не подставляются на «возврат». M/низ.
- **U5. Трёхуровневый маршрут: self / partial / manager.** Явный answer_partial: ответить на безопасную часть, спорную — менеджеру. → лечит over-handoff на составных. M/сред.
- **U6. Тёплый частичный из мягких/возраженческих фактов.** Для «выходные/формат» подтягивать objection-факты, частичный ответ без точных дней. → меньше сухих хендоффов. M/сред. *(частично)*
- **U7. Confidence контракта как гейт автономии.** Порог уверенности; ниже → partial/менеджер. → меньше уверенных ошибок понимания. S/низ.
- **U8. Сверка со слотами перед переспросом.** Жёстко прокидывать do_not_reask_slots + пост-проверка. → не переспрашивает названное. S/низ.
- **U9. Де-латч темы при явной коррекции.** «Я не про это, а про X» сбрасывает scope, форсит пере-понимание. → не зацикливается на неверной теме. M/сред.
- **U10. Контракт ведёт open-loop составного вопроса.** Очередь под-вопросов в памяти, помечать отвеченные. → не теряет хвост составного. M/низ.

## §3. БАЗА И ИЗВЛЕЧЕНИЕ (B)
- **B1. Текст факта всегда из structured_value.** Детерминированный рендерер «структура→текст»; число в тексте без источника в структуре = blocking. → «2 026 раз» невозможен. M/сред.
- **B2. Кросс-сверка text↔structured_value в гейте.** Все числа/даты текста сверять с raw_value/amount/percentage/weeks. → ловит порчу, что гейт пропускает. S/низ.
- **B3. Sanity-диапазоны полей частоты/недель.** weeks 1..52, раз/нед 1..7; вне → валить снимок. → «weeks=2026» ловится мгновенно. S/низ.
- **B4. Привязка условий к продукту обязательна.** Частота/формат/расписание требуют product+format. → не путает 1 vs 2 раза/нед. M/сред.
- **B5. Recall по теме, не только по ключу.** _matched_fact_keys + related_theme_ids/fact_type. → мягкие факты находятся по теме. M/сред.
- **B6. Каталог синонимов/возражений → ключи.** Таблица «формулировка клиента → fact_key». → стабильный recall мягких фактов. M/низ.
- **B7. Дедуп конкурирующих фактов по (brand, fact_key).** Один активный по приоритету, остальные superseded. → нет противоречий старый/новый. M/сред.
- **B8. Детерминированное ранжирование выдачи retrieve.** Скоринг по специфичности (точный→product/format→свежесть). → самый точный факт первым. S/низ.
- **B9. Гейт свежести valid_until перед выдачей.** Протухшее не отдавать как точное; отчёт о протухающем за 30 дней. → не называет устаревшее. S/низ.
- **B10. Карточка доказательств по каждому факту.** fact_key/source_id/valid_until/missing в скрытый блок менеджеру. → быстрая проверка источника. S/низ.

## §4. ВЕРИФИКАЦИЯ И БЕЗОПАСНОСТЬ (V)
- **V1. Проверка склейки: утверждение против ОДНОГО факта.** Атомарное утверждение из двух разных фактов = unsupported «склейка». → ловит «кабинет на МТС Линк». M/сред.
- **V2. Структурные claims «утверждение→чем подтверждено».** Семантика возвращает {claim, evidence_fact_key, verdict}; пустой ключ = неподтверждено. → машинно-проверяемо. M/низ.
- **V3. Контекст числа: единица+сущность.** Число грундится не только наличием, но и «соседями» (руб/мес/%/класс). → ловит «число верное, но не в том контексте». M/сред.
- **V4. Калибровка тёплого: добавил факт vs перефразировал.** Различать опасное добавление и безобидную живость; не отклонять второе. → меньше отклонений, тон выше. M/сред.
- **V5. Анти-склейка в тёплый промпт.** Явный запрет соединять два факта; примеры. → реже генерит склейку. S/низ.
- **V6. Карточка доказательств в лог хода.** {claim, fact_key, verdicts} к каждому автономному ответу. → регрейд по grep, не догадки. S/низ.
- **V7. Класс риска «именованная сущность вне фактов».** Платформы/продукты/ФИО не из facts → unsupported_entity (whitelist общих слов). → режет склейку даже без числа. M/сред.
- **V8. Доверие к автономии по уверенности семантики.** Серая зона между PASS/FAIL → менеджеру. → тонкие склейки не уходят автономно. M/сред.
- **V9. Второе мнение судьи только на подозрительном.** ≥2 сущности/числа из разных фактов рядом → второй короткий проход. → отлов склеек без удвоения цены. M/низ.
- **V10. Регресс-набор на класс «выдумка-склейка».** Золотой набор кейсов-склеек как обязательный гейт. → регресс виден сразу. S/низ.

## §5. ИЗМЕРЕНИЕ И СИМУЛЯТОР (M)
- **M1. Честный снимок судьи строго из retrieved_facts.** Убрать limit=12, старый отбор — вспомогательным. → нет ложных fabrication. S/низ. *(внедрено)*
- **M2. Анти-завышение: список known_absent.** Блок «чего в базе нет» + не штрафовать за честный хендофф по отсутствующему. → не занижает правильное поведение. S/низ.
- **M3. Детерминированный пред-проход grep чисел claim↔retrieved.** Машинный кандидат-список выдумок до LLM-судьи. → ручной регрейд почти исчезает. M/низ.
- **M4. CI-гейт целостности KB на порчу текста.** Регулярки поверх client_safe (раз/нед >7, NBSP в цифрах, год-в-частоте). → «2 026 раз» ловится зелёным гейтом. M/низ.
- **M5. Бизнес-метрика: дистанция правки менеджером.** Токен-дифф черновик↔правленый, а не бинарь. → метрика ближе к работе менеджера. M/сред.
- **M6. Метрика доли автономных безопасных ходов.** Безопасная тема + ответил сам + чисто + нет over_handoff. → виден прогресс к «автономному продавцу». S/низ.
- **M7. Доверительный интервал + порог второго прогона.** Интервал Уилсона; пересекает планку → needs_second_run. → честность про шум 8 диалогов. S/низ.
- **M8. Разделение «баг бота» vs «баг измерения».** Авто-причина hard-gate: infra/судья/бот. → сразу видно, что чинить. M/сред.
- **M9. Дешёвая регрессия под тронутые зоны.** Карта «персона/топик→файл»; прогон подмножества. → быстрая проверка соседнего. M/сред.
- **M10. Ротация рабочих наборов без касания холдаута.** Генератор персон с seed, холдаут read-only. → свежие провокации без переобучения. M/сред.

## §6. ПРОДУКТОВАЯ ЦЕННОСТЬ (P)
- **P1. Карта следующего шага без давления.** Один конкретный шаг «можно начать с…» из закрытого списка (next_step_hint). → конверсия без впаривания. S/низ.
- **P2. Блок рисков в черновике для менеджера.** Шапка: что проверить, что не подтверждено, почему шаг. → быстрее одобрение, меньше пропусков. S/низ.
- **P3. Персонализация по истории клиента без раскрытия CRM.** read_only-контекст для уровня ответа, без имён/статусов в текст. → не повторяет вводные повторному клиенту. M/сред.
- **P4. Ночной захват лида: справка + 1 квалифицирующий вопрос.** off_hours_mode: польза + честный дисклеймер по срокам. → ловит ночной трафик. M/сред.
- **P5. Библиотека снятия риска из playbook.** Few-shot подтверждённых «снятий тревоги», только при наличии факта. → ближе к лучшим менеджерам. M/низ.
- **P6. Безопасная работа с 3 возражениями (цена/время/сомнение).** Рассрочка/онлайн+записи/пробное — строго по бренду и факту. → меньше потерь на возражении. M/сред.
- **P7. Метрика бизнес-ценности черновика.** «Есть конкретный шаг» + «менеджер использовал шаг». → фокус на ценности, не зелёных тестах. M/низ.
- **P8. Передача менеджеру с готовым контекстом.** Сводка для менеджера: вопрос/слоты/тема/почему передано. → быстрее реакция на горячее. S/низ.
- **P9. Калибровка over-handoff: факт есть — отвечай.** Правило: есть client-safe факт по прямому вопросу → не прятать за «менеджер уточнит». → рост полезности без потери безопасности. M/сред.
- **P10. Тёплый онбординг-первый ход.** Представиться брендом + один вопрос про задачу, без анкеты из 5 пунктов; active_brand из канала. → меньше отвалов на первом сообщении. S/низ.

## §7. СТРАТЕГИЧЕСКИЕ / КРОСС-СРЕЗОВЫЕ (C, от Claude)
- **C1. Двухпроходный черновик: скелет фактов + гуманизация.** Детерминированный фактический скелет (неизменные токены) → тёплый слой обрамляет вокруг. → структурно убирает выдумку И поднимает тон одновременно. M/сред.
- **C2. Golden-path рецепты для топ-намерений.** Для цены/расписания/формата/скидки — связка нужных fact_keys + выверенный каркас ответа. → крепкий консистентный ответ на 80% частых вопросов, меньше over-handoff. M/сред.
- **C3. Контур обучения на правках менеджера.** Логировать accept/edit/reject + правленый текст; правки → few-shot/gold + тесты на повтор. → система учится на реальном использовании, а не на симуляторе. M/сред.
- **C4. Поэтапная автоотправка по whitelist тем.** Авто-отправка клиенту включать только на доказанно-безопасных темах (цена/расписание по подтверждённому факту), расширять по данным пилота. → путь от черновика к автономии без риска. M/выс.
- **C5. Самосогласованность на высоких ставках.** Для рискованных автономных ответов — 2 черновика; авто только если совпали по фактам, иначе менеджеру. → ловит галлюцинацию на генерации. M/сред.
- **C6. Бюджет латентности/стоимости на ход.** Тривиально-безопасный ход — меньше проходов; спорный — полный набор. → all_eligible+семантика+faithfulness не разоряют на масштабе. M/низ.
- **C7. Двухуровневый роутинг моделей.** Дешёвая модель на простое заземлённое, сильная — на спорное/составное. → дешевле и быстрее без потери качества. M/сред.
- **C8. «Известное отсутствие» как факт первого класса (на стороне бота).** Бот чётко говорит «этого у нас нет» (модульные М9/М11, банковская рассрочка УНПК), а не туманный хендофф. → честно и полезно, меньше over-handoff. S/низ.
- **C9. Перманентный кросс-бренд CI-набор.** Набор провокаций «вытащи другой бренд»; гейт 0 утечек перед каждым релизом. → бренд-разделение под постоянным контролем. S/низ.
- **C10. Честная срочность только из факта.** Кодифицировать: «места ограничены» = подтверждённый факт, иначе никогда не выдумывать срочность. → приём playbook без манипуляции. S/низ.

## §9. РАНЖИРОВАНИЕ (Claude + субагент, сведено)

Критерии (по весу): влияние на полезность/тон/over-handoff ИЛИ закрытие свежего бага → разблокировка
доверия к измерению → низкий риск + вписывается в архитектуру → малое усилие/высокая отдача.

### Тир 1 — делать первым (дёшево, высокая отдача, разблокирует)
1. **B2** — кросс-сверка text↔structured_value в гейте. Ловит именно прорвавшийся баг «2 026 раз». S/низ.
2. **M4** — CI-гейт целостности KB на порчу текста. Класс «порча при пересборке» становится зелёным тестом. M/низ.
3. **B3** — sanity-диапазоны полей (weeks 1..52, раз/нед 1..7). Мгновенно валит «weeks=2026». S/низ.
4. **T1+C1+V5** (кластер) — тепло меняет ТОЛЬКО форму / фактический скелет + анти-склейка в промпт. Двойной удар: тон↑ и выдумка-склейка↓. Прямой фикс замеренной проблемы. S-M/низ-сред.
5. **U1** — детектор мягкого возврата как роль. Закрывает свежую утечку «возвраты бывают?→договор+email». M/низ.
6. **P9** — калибровка over-handoff «факт есть — отвечай». Бьёт в главное слабое место, не трогая P0. M/сред.
7. **C8** — «известное отсутствие» как факт (бот чётко «этого нет»). Лечит over-handoff честно, дёшево. S/низ.
8. **M7** — доверительный интервал + порог второго прогона. Разблокирует доверие к метрикам на 8 диалогах. S/низ.

### Тир 2 — следом
9. **V1** — проверка склейки «утверждение против ОДНОГО факта». Системно режет «кабинет на МТС Линк». M/сред.
10. **M3** — детерминированный grep чисел claim↔retrieved до судьи. Снимает зависимость от врущего судьи. M/низ.
11. **U5** — трёхуровневый маршрут self/partial/manager. Лечит over-handoff на составных. M/сред.
12. **T5** — один хороший вопрос вместо пустого хендоффа. Часть over-handoff → живой диалог. M/сред.
13. **V4** — калибровка тёплого (добавил факт vs перефразировал). Меньше отклонений → тон↑ без потери безопасности. M/сред.
14. **M8** — разделение «баг бота» vs «баг измерения». Ускоряет всю отладку. M/сред.
15. **U6+B5+B6** (кластер) — тёплый частичный из мягких фактов + recall по теме. Меньше сухих хендоффов. M/низ.

- **C6/C7** (бюджет латентности / роутинг моделей) — оптимизация преждевременна до закрытия смысла и измерения.
- **P3** (персонализация из CRM) — близко к утечке внутренних данных; отдача на пилоте мала против риска.
- **C5** (самосогласованность ×2 черновика) — удваивает стоимость; выгоду не докажешь при шумном измерении.
- Уже внедрено (только довести, не в топ): **U3, M1, T3**.

### Ключевая логика приоритета
Сначала целостность базы (B2/M4/B3) и доверие к измерению (M7/M3/M8) — иначе любое улучшение тона
нельзя ни подтвердить, ни защитить от регресса (правило «не путать зелёные тесты с готовностью»).
Параллельно — прямые удары по полезности/over-handoff (T1+C1, P9, C8, U5, T5) и по свежей склейке (V1).
