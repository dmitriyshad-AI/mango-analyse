# Смысловой аудит пакета

Артефакт: `docs/new_lead_sales_funnel_2026-05-22/`

Аудитория: внутренний bot-диалог, разработчик, менеджер, РОП.

Статус: `candidate`.

## Вердикт

`PASS_WITH_NOTES` для проектного candidate-пакета.

`BLOCKED` для прямого клиентского внедрения без дополнительных тестов, утверждения спорных фактов и ручной проверки Дмитрием/РОПом.

## Статусы

- formal_pass: `true` - пакет документов создан по структуре ТЗ.
- semantic_pass_for_design: `pass_with_notes` - логика согласована с целями проекта и текущими ограничениями.
- pilot_ready: `false` - нужен перенос в тесты и утверждение фактов.
- production_ready: `false`.

## Что прошло

- Воронка не делает Telegram отдельным продуктом, а остается частью общего контура продаж.
- Бренды разделены: Фотон и УНПК не смешиваются в клиентских сценариях.
- P0/high-risk темы явно уходят менеджеру.
- Сохранен принцип "сначала прямой ответ, потом один следующий шаг".
- Уточняющие вопросы ограничены 1-2 за сообщение.
- Есть правила против повторного запроса известных данных.
- Есть метрики, которые измеряют не только безопасность, но и пользу для продаж.
- Есть сценарии, которые можно превратить в автотесты.
- После независимого Claude review добавлены короткая сводка менеджеру, смысловые gate-кандидаты и 2 дополнительных сценария на преподавателей/юридические данные.

## Блокирующие вопросы для live-внедрения

- В базе есть мусорные или конфликтные факты: рассрочка "2026-05-22 месяцев", "городской лагерь бесплатно", спорные скидки, УНПК-пробная неделя, УНПК-помесячная скидка.
- Не все цены, даты и наличие мест имеют надежное правило актуальности на дату внедрения.
- Нет утвержденной политики ответа на вопрос "кто вы?".
- Не решено, какие follow-up вопросы бот может задавать автономно в многошаговой воронке, если матрица автономности по умолчанию осторожна к продолжениям диалога.
- Нет автотестов по новым сценариям.

## Неблокирующие риски

- Бот может стать слишком осторожным и передавать менеджеру то, что мог безопасно объяснить.
- Бот может стать слишком продажным, если фразы про рост цены или почти распроданные места не ограничить.
- Менеджеры могут не пользоваться сводкой, если она будет длинной.
- Метрик может стать слишком много; для первого пилота нужно выбрать минимум.

## Обязательные будущие проверки

- Нет запрещенных слов другого бренда.
- Нет "места есть", "место закреплено", "оплата прошла".
- Нет "бесплатная пробная неделя" в УНПК.
- Нет "бесплатное очное пробное" без менеджера.
- Нет Т-Банк/Долями/через банк в УНПК.
- Нет "до 36 месяцев" в Фотоне.
- Нет раскрытия AMO/Tallanto/CRM/source_id/ID.
- P0 всегда `manager_only` или не автономный маршрут.
- Каждый безопасный ответ содержит прямой ответ и один следующий шаг.

## Рекомендуемое следующее действие

Передавать пакет в основной bot-диалог как проектную основу, а не как готовую инструкцию для клиента. Следующий практический шаг - утвердить спорные факты, выбрать 10-15 первых сценариев для автотестов и реализовать минимальный слой метрик пилота.
