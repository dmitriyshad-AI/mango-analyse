# План: чистая архитектура с минимумом прогонов. 2026-06-02.

Автор: Клод 1. Цель Дмитрия: надёжная, прозрачная, качественная архитектура; лазанья ПЕРЕНЕСЕНА (не
потеряна); упор на понимание контекста через LLM, а не ловлю блох regex; МИНИМУМ живых прогонов.

## Целевая архитектура (что в конце) — 5 ясных слоёв вместо 8000-строчной лазаньи
1. **Planner (LLM-понимание)** — ОДИН модельный вызов перед retrieve: тема, продукт, класс, формат,
   подвопросы, P0-frame, нужные факты, уверенность. Заменяет 145 regex + 31 `_asks_/_is_`. ← понимание = LLM.
2. **Слой фактов/правил** — доменные правила (скидки 20/30%, рассрочка 6/10/12, стэкинг, маткапитал,
   лицензии, расписание) как СТРУКТУРИРОВАННЫЕ данные + чистые функции (НЕ хардкод-текст вперемешку с
   логикой). ← сюда переносится «лазанья», она не теряется.
3. **Composer (LLM-черновик)** — модель пишет живой ответ из фактов.
4. **Выходной гейт (детерминированный)** — авторитетная охрана: бренд, P0, утечки, выдумки, обещания,
   промокоды, плейсхолдеры, операционная специфика. Чистые функции, только ПОНИЖАЕТ. Заменяет 407 инлайн.
5. **Тонкий runtime** — оркестрация: planner → retrieve → composer → gate → draft. Прозрачный поток.

## Принцип минимума прогонов
Живой прогон (дорогой) нужен ТОЛЬКО когда меняется ГЕНЕРАЦИЯ (понимание/черновик). Гейт и слой правил —
проверяются ОФЛАЙН на уже собранных транскриптах (детерминированно) + юнитами (бесплатно). Итого **3 живых
прогона на всю переделку**.

## Шаги

### Шаг 0 — карта переноса (Клод 1, БЕСПЛАТНО, 0 прогонов)
Доразметить реестр детекторов + факт-пакет Кодекса: каждая из 407 инлайн-точек и 204 детекторов →
{костыль-понимания: выкинуть | доменное правило: перенести в слой 2 | предохранитель: в гейт слоя 4}. Это
карта, по которой переносим, а не угадываем. 90 «требуют подтверждения» — закрыть чтением.

### Шаг 1 — Гейт (Кодекс код; проверка офлайн + 1 прогон)
Реализовать единый авторитетный выходной гейт (ТЗ `TZ_INVERSION_..._v2`), компонующий существующие
проверенные функции (verify_output + P0 + утечки + промокод + плейсхолдер + операционка). Монолит ОСТАЁТСЯ
(двойная защита).
- Проверка БЕСПЛАТНО: юнит-щит + **теневой офлайн** — Клод 1 применяет гейт к УЖЕ сгенерированным черновикам
  из сохранённых прогонов (BLOCK1_1 и др.), сравнивает: гейт ловит то же, что монолит, не перебарщивает.
- **★ Прогон 1 (живой, контрольный):** гейт на свежих диалогах — безопасность держит, over-handoff/тон ок.

### Шаг 2 — Слой фактов/правил (Кодекс код; 0 прогонов)
Перенести доменную бизнес-логику (скидки/рассрочка/цены/маткапитал/расписание) из хардкод-текстов-в-
монолите в структурированный слой правил (данные + чистые функции). Понимание/черновик пока те же.
- Проверка БЕСПЛАТНО: юниты (правила дают те же тексты, что монолит) + теневой офлайн. Генерация по сути не
  изменилась → прогон НЕ нужен.

### Шаг 3 — Planner + качество (Кодекс код; 1 прогон) — здесь меняется ГЕНЕРАЦИЯ
Заменить regex-детекторы (`_asks_/_is_`, 145 re) на ОДИН модельный planner перед retrieve (понимание =
LLM). Сразу под гейтом поднять КАЧЕСТВО: включить X2-тон, добить хвост, добавить продающий «чёткий шаг» —
безопасно, т.к. гейт ловит на выходе.
- Проверка: юниты + **★ Прогон 2 (живой):** новое понимание + качество vs предыдущий — тон вверх (цель 70+),
  уходы вниз, выдумки 0, HARD-P0/бренд держат (гейт). Это главный замер качества.

### Шаг 4 — Снос монолита (Кодекс код; 0 новых прогонов + 1 финал)
Удалить regex-лазанью понимания + дубли route-override + мёртвый код (`_semantic_recover_or_handoff` и
т.п.). Остаются: planner + слой правил + composer + гейт + runtime.
- Проверка БЕСПЛАТНО: **теневой замер «гейт без монолита»** на транскриптах Прогона 2 — гейт ловит ВСЁ
  (доказательство безопасности сноса БЕЗ нового прогона) + юниты.
- **★ Прогон 3 (финал, живой):** чистая архитектура целиком — итог качества + безопасности.

## Итого прогонов: 3 живых (контрольный гейт / понимание+качество / финал). Всё остальное — юниты + теневые
офлайн-проверки на уже собранных транскриптах. Это минимум, ниже которого — слепой полёт.

## Что получаем
- **Надёжно:** на каждом шаге монолит/гейт страхует; снос — только после доказательства теневым замером.
- **Прозрачно:** 5 слоёв с ясными границами вместо лазаньи; промпты planner/composer версионируются.
- **Качество:** тон/продажа поднимаются в шаге 3 под защитой гейта (свобода больше не течёт).
- **Лазанья перенесена:** доменные правила живут в слое 2 как данные, не выкинуты.
- **LLM-понимание:** regex-угадывание заменено модельным planner'ом.
- **Скрипты/промпты/правила:** гейт = детерминированные скрипты-функции; planner/composer = промпты;
  скидки/рассрочка/расписание = правила-данные.

## Безопасность процесса
Предохранители извлекаются+замораживаются (логика модулей канон), политика гейта настраиваема. Снос монолита
ТОЛЬКО после теневого «гейт без монолита» = зелёный. Доменные правила НЕ удаляем — переносим. На каждый
гейт-чек — негативный контроль. Клод 1 проверяет каждый коммит в песочнике; Кодекс самопроверяется.

## ИТОГ adversarial-спора с аудитором (3 реальных недочёта + решения; 4 претензии сняты по сырью)

СНЯТО (аудитор признал ошибку по коду): «promocode/placeholder/zero_collect не существуют» — существуют
(subscription_llm 455/459/1914/1928/3063); «planner = новый недетерминизм + fine-tuning» — понимание УЖЕ
модельное (understand_fn, pipeline 408), planner консолидирует существующее, fine-tuning не нужен;
«офлайн-тест слеп» и «факты статичны» — смешение ролей теста (см. решения).

РЕАЛЬНЫЕ НЕДОЧЁТЫ (согласованы) + РЕШЕНИЯ:
1. **Карта переноса (Шаг 0) не валидирует саму классификацию.** Решение: карта — ГИПОТЕЗА, не истина.
   Каждый «костыль-на-удаление» доказывается НЕГАТИВНЫМ тестом: после его удаления гейт всё равно ловит
   то, что точка ловила. Открылась дыра → это была защита, возвращаем. 90 неуверенных — не трогаем в
   первых волнах. Классификация валидируется поведенчески-негативно, не чтением.
2. **Слой 2 рискует стать каталогом готовых ТЕКСТОВ, дублируя Composer.** Решение (критерий приёмки слоя
   2): хранить ТОЛЬКО структурированные данные (периоды рассрочки [6,10,12], проценты, условия, argmax
   скидок как чистая функция) + НИ ОДНОГО готового предложения. Composer генерирует текст из данных. Где
   правило невозможно представить данными — оно становится client_safe ФАКТОМ в KB, а не текст-константой
   в коде. Итог: SAFE_TEXT-константы исчезают из кода (→ данные слоя 2 или факты KB).
3. **Снос (фаза 4) фиктивен, если `DIALOGUE_CONTRACT_V2_TEMPLATE_REGISTRY` остаётся.** Решение (жёсткий
   критерий завершения): registry + safe_template-диспетчер УДАЛЕНЫ из кода; их роль выполняют слой
   правил+composer (доменное) и выходной гейт (безопасность). Снос НЕ завершён, пока registry в коде —
   это измеримо, не «ощущение чистоты».

ПЕРЕФОРМУЛИРОВКА: калибруемая политика гейта (`GATE_BLOCKING_CODES`: код→block/downgrade/warn вместо
текущей чёрно-белой findings→manager, subscription_llm 1384-1396) — это НОВЫЙ код, ЗАДАЧА фазы 1, не дыра.

УТОЧНЕНИЯ РОЛЕЙ (из снятых претензий, но важно прописать):
- Офлайн-тест на снимке = РЕГРЕССИЯ логики гейта на ИЗВЕСТНЫХ черновиках; НОВЫЕ сценарии ловят живые
  прогоны 1/2; свежесть KB — зона build+semantic-gate, не гейта (в проде гейт на свежих retrieve-facts).
- Planner: fallback обязателен (invalid/low-confidence → manager_only/переспрос); недетерминизм
  страхуется выходным гейтом.

## ИНЖЕНЕРНОЕ РЕВЬЮ КОДЕКСА (автор кода, без фактических ошибок) — 5 правок ПРИНЯТЫ

Кодекс подтвердил главную хрупкость с точными цифрами: 2 core-файла = 508 функций / 11 720 LOC; грубая
разметка — доменное 116ф/2842, route/recovery 59ф/1475, safety 70ф/2285, понимание/regex 65ф/1086; НО
границы ГРЯЗНЫЕ (`apply_high_risk_content_guards` один 665 LOC мешает P0+route+template+brand+policy;
доменные `_*_safe_template` уже несут маршрут). Вывод: шаги 2-3 недооценены, «3 прогона» переоценены.

ПРИНЯТЫЕ ПРАВКИ:
1. **Шаг 2 разбить: 2a (rule inventory) + 2b (миграция по семействам).** Сначала для КАЖДОГО правила описать
   typed-контракт: rule_id, входные факты, allowed scopes, blocking conditions, route effect, text effect,
   required negative tests. БЕЗ «0 прогонов» на весь шаг: если миграция меняет precedence/flags — итоговый
   route/текст меняются → точечная проверка/прогон. Иначе правила «переедут той же лазаньёй в новый файл».
2. **Шаг 3 разбить: 3a planner parity (понимание = прежнее, замер что не сломалось) + 3b тон/X2/продажа.**
   НЕ совмещать — иначе planner-регрессии и tone-регрессии неразделимы.
3. **Классифицировать regex на 4 класса (в Шаге 0), не «regex→planner» огулом:** `planner-replaceable`
   (intent-hints → в planner) | `domain-scope` (scope-якоря → в слой правил) | `safety-gate` (P0/refund/
   legal/payment/promo/placeholder/identity → остаются в гейте) | `text-cleanup`. Сносить можно только
   первый класс; safety/scope-regex НЕ трогать. (Снимает риск регрессии скрытых отрицательных исключений:
   `_asks_refund_policy` исключает налог/вычет; `_current_refund_dispute_signal` отделяет гипотетику от
   претензии; `_payment_method_target_anchors` различает Долями/банк/счёт — это policy, не понимание.)
4. **Теневой «гейт без монолита» проверяет НЕ только финальный текст**, но metadata/route/topic/fact_refs/
   recovery_candidate — монолит менял их ДО финального текста.
5. **«3 прогона» = минимальный БЮДЖЕТ, не достаточное доказательство.** После planner (3a) — СВЕЖАЯ
   реалистичная выборка, не только replay сохранённых транскриптов.

КЛЮЧЕВЫЕ УТОЧНЕНИЯ ДИЗАЙНА (Кодекс, критично):
- **Гейт видит КОНТЕКСТ диалога (P0-latch, история, память), не только финальный draft.** P0/ПДн
  многоходовые: refund-latch, complaint, payment-dispute, zero-collect зависят от нескольких ходов, а не
  от одного текста. Гейт принимает на вход и latch/историю.
- **Гейт = безопасность, НЕ upstream-корректность.** Он ловит плохой текст, но НЕ гарантирует, что бот
  нашёл нужный факт, понял продукт, сохранил тему, дал полезный шаг. Качество (полезность/тон) — это
  planner+retrieval+composer; гейт их не чинит и не заменяет.
- 5-слойная архитектура реалистична, но слои с ЖЁСТКИМИ КОНТРАКТАМИ (planner→structured intent+uncertainty
  +fact_key candidates; rules→policy-engine с брендом/scope/freshness/negative-facts/product-family;
  composer→только из fact_refs/rule outputs; gate→downgrade-only с severity; runtime→память/latch/бренд-
  канал/ПДн/audit).
