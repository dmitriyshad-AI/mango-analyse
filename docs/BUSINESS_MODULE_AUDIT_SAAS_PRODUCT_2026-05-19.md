## 1. Краткий вывод

Коротко:

- для внутреннего Macbook/client-hosted appliance путь выбран правильно;
- для внешней демонстрации задел есть, но нужно упаковать безопасный demo-контур;

## 3. Что уже сделано для будущего SaaS

### 3.1. Разделение на продуктовые слои

Проект уже не выглядит только как набор локальных скриптов обработки звонков. Есть отдельные слои:

- productization/appliance;
- channel runtime;
- customer timeline;
- question catalog;
- knowledge base;
- deal-aware AMO слой;
- AMO/Tallanto read-only контекст;
- safety gates;
- audit packs.

Это правильная форма будущего продукта: ядро не должно напрямую зависеть от одного конкретного скрипта, одной CRM или одной телефонии.

### 3.2. Product DB и локальная клиентская установка

В `src/mango_mvp/productization/product_db.py` есть отдельная product DB на SQLite. Она отделена от `stable_runtime` и подходит для модели:

```text
наш Macbook -> отдельный Macbook/сервер клиента -> позже серверный продукт
```

Сейчас SQLite остается разумным решением для внутреннего пилота и клиентской установки под управлением клиента. PostgreSQL не нужен “для красоты”. Он понадобится позже, когда появятся несколько пользователей, параллельные процессы, hosted-режим, несколько организаций и реальные требования к администрированию.

### 3.3. Адаптерный подход

В `src/mango_mvp/productization/adapters.py` и связанных модулях заложена правильная идея:

- Mango - первый адаптер телефонии, но не вечный центр архитектуры;
- AMO/Tallanto - первые CRM/LMS-источники, но не должны стать форматом ядра;
- core должен работать с нормализованными событиями, клиентами, действиями и проверками.

Это важно для будущей продажи другим организациям, где может быть другая телефония или CRM.

### 3.4. Read-only Product API и локальный dashboard

Есть read-only Product API и HTTP-слой. Важный плюс: write-методы заблокированы по умолчанию. Это хорошая привычка для будущего SaaS:

- сначала показываем состояние;
- потом даем preview;
- потом approval;
- только затем live-действие.

Текущий Product API пока больше похож на локальный операторский слой, чем на внешний SaaS API, но направление верное.

### 3.5. Safety gates и audit packs

В проекте уже системно используются:

- dry-run;
- preview;
- explicit approval;
- readback;
- rollback-подход;
- audit packs;
- запреты на live-send/live-write;
- отдельное понятие `formal_pass` и `semantic_pass`.

Это один из самых сильных SaaS-заделов проекта. Для внешнего продукта это должно стать обязательной нормой, а не разовой практикой.

### 3.6. Channel runtime вместо “одного Telegram-бота”

`src/mango_mvp/channels/` построен правильно: Telegram, Telegram Business, web chat, CRM chat и будущие каналы идут через единые контракты.

Уже есть:

- единый формат сообщения;
- единый формат сессии;
- черновик ответа;
- рекомендуемые действия;
- сигнализация по рискам;
- feedback loop;
- локальное хранилище;
- manager workspace;
- Telegram adapter;
- web/CRM chat adapter.

Это не “одноразовый бот”, а начало общей системы общения с клиентом.

### 3.7. База знаний и правила РОПа

База знаний v3.3 уже имеет:

- `formal_pass`;
- `semantic_pass`;
- bot pack;
- employee pack;
- brand rules;
- bot policy;
- facts registry;
- post-filter;
- разделение Фотон/УНПК;
- ограничения по ценам, скидкам, документам, оплатам, маткапиталу, налоговому вычету.

Это ценно для будущего SaaS, потому что продукт должен уметь жить с обновляемой базой знаний, а не с зашитыми в код ответами.

### 3.8. Customer timeline

`src/mango_mvp/customer_timeline/` - стратегически важный слой. Он сводит звонки, письма, CRM, Tallanto, Telegram/Web-chat и будущие действия в единую историю клиента.

Это не мусор и не временная заготовка. Но сейчас он должен оставаться read-only/preview, пока не закрыты правила по семьям, дублям, нескольким ученикам, нескольким сделкам и Tallanto-семантике.

### 3.9. Тесты

В проекте много тестов по productization, channels, customer timeline, question catalog, deal-aware. Это важная основа для SaaS, потому что без тестов нельзя безопасно продавать систему другим организациям.

На уровне структуры найдено 139 тестовых файлов по ключевым продуктовым зонам:

- `test_productization_*`;
- `test_channels_*`;
- `test_customer_timeline_*`;
- `test_question_catalog_*`;
- `test_deal_aware_*`.

## 4. Что пока остается локальным Macbook-инструментом

### 4.1. Локальные пути и файловые артефакты

Много рабочих процессов завязано на:

- локальные пути;
- CSV/JSON/JSONL/XLSX;
- SQLite-файлы;
- audit-папки;
- `.codex_local`;
- `product_data`;
- `stable_runtime`;
- локальные env-файлы.

Для внутреннего пилота это нормально. Для SaaS нужно будет отделить:

- данные клиента;
- runtime-артефакты;
- временные файлы;
- логи;
- резервные копии;
- секреты;
- публичный интерфейс.

### 4.2. Нет настоящей многопользовательской модели

Пока нет полноценной системы:

- пользователей;
- ролей;
- прав доступа;
- сессий;
- аудита действий конкретного пользователя;
- ограничений “менеджер видит только своих клиентов”;
- админки организации.

Для внутреннего Macbook-пилота это допустимо. Для внешней продажи - блокер.

### 4.3. Tenant isolation пока больше структурная, чем защитная

В коде есть `tenant_id`, tenant config и tenant isolation reports. Но это пока не настоящая граница безопасности. Нельзя считать, что система уже безопасно изолирует организации друг от друга.

Для будущего SaaS нужно будет усилить:

- отдельные product roots;
- отдельные секреты;
- отдельные snapshots;
- отдельные backup/restore;
- запрет чтения чужого tenant root;
- тесты на “данные одной организации не попадают другой”.

### 4.4. Scheduler пока локальный

Scheduler/control-plane выглядит полезно, но это пока локальный контур, а не надежная серверная очередь. Для внутреннего appliance этого достаточно. Для hosted SaaS понадобятся:

- устойчивый сервис;
- контроль зависших задач;
- повторы;
- блокировки;
- наблюдаемость;
- алерты;
- журнал выполнения.

### 4.5. Product API местами связан с локальным runtime

Для внутренней панели можно читать operator/runtime artifacts. Для внешнего SaaS это опасно: публичный продуктовый API не должен напрямую зависеть от исторического `stable_runtime` и локальной структуры папок.

## 5. Что есть в коде, но пока не используется полноценно

1. **Customer timeline** - уже большой и ценный слой, но пока не основной источник истории клиента.
2. **Product API/dashboard** - есть read-only основа, но ещё не рабочее место менеджера.
3. **Tenant isolation** - есть отчеты и проверки, но нет настоящей security-изоляции.
4. **Scheduler health/control plane** - полезный задел, но не основной production scheduler.
5. **Demo tenant / sanitized real demo** - хороший задел для внешнего показа, но нужно привести к актуальному Telegram/KB состоянию.
6. **CRM writeback preview** - полезный слой, но без live-расширения до snapshot/rollback/readback и микропилота.
7. **Question catalog full-run** - полезен для аналитики и FAQ, но не должен быть production-решателем ответов без контекста.
8. **Email archive** - важный будущий канал, но не должен отнимать фокус до Telegram-пилота.

## 7. Проблемы P0/P1/P2/P3

### P0

1. **Нельзя запускать автоответы клиентам.**
   Текущий безопасный режим: только черновик менеджеру. Live-send в Telegram/email/site/CRM chat пока запрещен.

2. **Нельзя продавать внешний SaaS как готовый продукт.**
   Нет достаточной изоляции организаций, прав доступа, установки, поддержки, журналов действий, юридического контура и стабильного внутреннего доказательства пользы.

3. **Нельзя расширять live-запись в AMO/Tallanto без отдельного подтверждения.**
   Любое live-действие требует snapshot, rollback, dry-run, approval, readback и микропилот.

4. **Нельзя считать `tenant_id` настоящей защитной изоляцией.**
   Сейчас это полезная структура, но не безопасность.

### P1

1. **Ближайший внутренний Telegram-пилот еще не закреплен как операционный процесс.**
   Нужно подтвердить bot token, chat id менеджера, kill switch, env flags, локальную приватную DB, свежий KB snapshot и Codex CLI login.

2. **База знаний и Telegram-черновики требуют постоянного semantic gate.**
   `quality_passed=true` не означает “готово к клиенту”. Для клиентских ответов нужен `semantic_pass`.

3. **Customer timeline не готов как основной источник истины.**
   Нужны правила по семейным телефонам, дублям учеников, нескольким сделкам и Tallanto.

4. **Старые state/roadmap документы отстают от свежих audit packs.**
   Это создает риск, что следующий диалог возьмет старую картину проекта.

### P2

1. **Нужна карта актуальности SaaS/productization документов.**
   Старые документы полезны, но нужно явно пометить: “актуально / исторически полезно / устарело”.

2. **Нужна единая карта product_data/stable_runtime актуальных артефактов.**
   Иначе для demo или пилота легко взять старый слой.

3. **Нужно развивать read-only dashboard под менеджера/РОПа.**
   Но только после того, как Telegram-пилот даст реальные сценарии.

4. **Нужно готовить будущие права доступа и журналы действий.**
   Пока можно проектировать и тестировать, но не делать большой auth-проект.

### P3

1. Обновить каталог скриптов.
2. Пометить опасные команды и live-команды.
3. Убрать `.DS_Store` и `__pycache__` из `src` только после отдельного подтверждения.
4. Сверить старые SaaS completion reports с текущим состоянием.

## 8. Какие тесты есть и каких не хватает

### Есть

Есть тесты по:

- productization;
- channel runtime;
- Telegram adapters/polling/manager inbox;
- customer timeline;
- question catalog;
- knowledge base safety;
- deal-aware;
- AMO rollback/writeback guards;
- tenant isolation;
- product DB;
- scheduler health;
- dashboard API.

### Не хватает перед SaaS/внешним показом

1. **Полный smoke “установка на чистый product root”.**
   Должен проверять: init DB, config wizard, healthcheck, demo seed, dashboard open, backup verify, restore dry-run.

2. **Тесты настоящей изоляции организации.**
   Данные tenant A не должны читаться в tenant B ни через API, ни через файлы, ни через snapshots.

3. **Тесты прав доступа.**
   Сейчас нет полноценной модели “админ / РОП / менеджер / аудитор”.

4. **Тесты журналов действий.**
   Нужно фиксировать: кто посмотрел, кто подтвердил, кто отправил, кто запустил импорт, кто откатил.

5. **Тесты setup/upgrade.**
   Для client-hosted модели важно, чтобы обновление версии не ломало product DB и конфиги.

6. **Тесты внешнего demo-сценария.**
   Должен быть один сценарий на 10-15 минут без приватных данных и без live-write.

7. **E2E тест Telegram pilot draft-only.**
   Сообщение клиента -> контекст -> черновик -> manager inbox -> feedback, без отправки клиенту.

- писать в AMO/CRM;

## 10. Что не делать сейчас: какие SaaS-работы нельзя начинать до внутреннего Telegram-пилота

Нельзя начинать сейчас:

1. Внешний SaaS для нескольких организаций.
2. Биллинг и тарифы.
3. Сложную облачную multi-tenant архитектуру.
4. Большую миграцию SQLite -> PostgreSQL только ради “взрослости”.
5. Красивый внешний кабинет клиента вместо рабочего места менеджера.
6. Подключение новых CRM/телефоний как коммерческой платформы.
7. Автономную отправку сообщений клиентам.
8. Автозапуск цепочки capture -> audio download -> ASR -> R+A -> CRM writeback.
9. Массовый email/OCR batch как новый главный фронт.
10. Продажу продукта внешним клиентам без внутренней метрики пользы.

Причина: эти работы увеличат сложность, но не докажут главную ценность. Сейчас нужно доказать, что система реально помогает менеджеру отвечать клиентам быстрее и качественнее.

## 11. Какие SaaS-работы можно делать уже сейчас

Можно делать без распыления:

1. Поддерживать и тестировать адаптерные границы.
2. Усиливать read-only Product API и dashboard только под нужды внутреннего пилота.
3. Делать healthcheck, backup, restore dry-run, config wizard.
4. Готовить demo tenant на обезличенных или безопасно очищенных реальных данных.
5. Писать тесты запретов: no live send, no CRM write, no stable_runtime write.
6. Вести audit packs и semantic review.
7. Документировать путь установки на отдельный Macbook/сервер.
8. Делать tenant config без секретов.
9. Проектировать права доступа на бумаге и в тестовых контрактах.
10. Обновить актуальность SaaS-документов.

## 12. Что обязательно нужно до продажи другим организациям

Перед продажей внешним организациям нужны:

1. **Доказанная внутренняя польза.**
   1-3 менеджера реально пользуются системой ежедневно.

2. **Повторяемая установка.**
   Понятный install/runbook для отдельного Macbook или сервера клиента.

3. **Изоляция данных организации.**
   Отдельные product roots, секреты, базы, snapshots, backups, логи.

4. **Права доступа.**
   Админ, РОП, менеджер, аудитор, технический оператор.

5. **Журнал действий.**
   Кто что сделал, кто подтвердил, кто отправил, кто изменил правило, кто запустил writeback.

6. **Backup/restore.**
   Не просто копия файла, а проверяемый restore dry-run.

7. **Секреты и доступы.**
   Нельзя хранить токены как обычные конфиги в продукте для клиента.

8. **Обновление базы знаний.**
   Понятный процесс: загрузили новые цены/расписание/правила -> проверили -> semantic_pass -> включили.

9. **Каналовая настройка.**
   Telegram, email, сайт, CRM-chat должны подключаться через единый безопасный канал.

10. **Наблюдаемость.**
    Логи, ошибки, healthcheck, “что не работает”, “что давно не запускалось”.

11. **Юридический и персональный контур.**
    Для внешних клиентов нужно отдельно оформить хранение аудио, транскриптов, писем и персональных данных.

12. **Support runbook.**
    Что делать при падении Mango/AMO/Tallanto, при ошибке бота, при неверном ответе, при необходимости отката.

## 13. Минимальный путь от внутреннего пилота к первому внешнему показу

### Шаг 1. Внутренний Macbook pilot

Цель: менеджер получает полезные черновики в Telegram, клиенту ничего автоматически не отправляется.

Минимум:

- свежая база знаний v3.3;
- MEGA/semantic gate;
- manager inbox;
- feedback по черновикам;
- no live send;
- no CRM write;
- метрики полезности.

Критерий: менеджер реально пользуется каждый день, P0 нет, часть черновиков экономит время.

### Шаг 2. Отдельный Macbook/сервер внутри компании

Цель: убрать зависимость от “рабочей машины разработчика”.

Минимум:

- product root;
- product DB;
- config wizard;
- healthcheck;
- backup/restore dry-run;
- operator dashboard;
- локальные секреты;
- понятный запуск/остановка.

Критерий: систему можно поднять по инструкции без ручного знания внутренностей проекта.

### Шаг 3. Безопасный внешний demo

Цель: показать потенциальному B2B-клиенту не скрипты, а рабочий сценарий.

Минимум:

- demo tenant;
- безопасные реальные или обезличенные данные;
- dashboard;
- пример входящего сообщения;
- черновик ответа;
- очередь РОПа;
- база знаний;
- writeback preview без live-записи.

Критерий: демонстрация за 10-15 минут показывает ценность без доступа к приватному runtime и без риска записи во внешние системы.

### Шаг 4. Первый client-hosted pilot

Цель: поставить продукт одной внешней организации под их контроль.

Минимум:

- отдельный root/DB/secrets;
- read-only интеграции;
- no-auto-send;
- no-live-write по умолчанию;
- backup/restore;
- support runbook;
- ручной onboarding;
- понятные ограничения.

Критерий: второй клиент подключается конфигурацией и адаптерами, а не форком кода.

### Шаг 5. SaaS позже

Настоящий hosted SaaS нужен только после 1-2 успешных client-hosted pilots. Тогда появятся реальные основания для PostgreSQL, ролей, центрального хостинга, биллинга и SLA.

## 14. Архитектурные решения, которые могут помешать будущему расширению

1. **Считать локальные пути частью продукта.**
   Надо отделять внутренний runtime от переносимого product root.

2. **Считать SQLite вечным решением.**
   SQLite нормален сейчас, но для hosted/multi-user режима потребуется другой слой хранения.

3. **Привязать ядро к Mango/AMO/Tallanto.**
   Они должны оставаться адаптерами, а не форматом ядра.

4. **Смешать разные уровни безопасности текста.**
   Bot-safe, CRM-internal, ROP-internal и SaaS-safe тексты должны иметь разные правила.

5. **Обойти gates ради скорости.**
   Любой shortcut для live-send/live-write станет будущим P0.

6. **Переоценить LLM-классификацию.**
   Модель помогает, но правила РОПа, свежие факты и manager approval важнее для первого пилота.

7. **Не вести audit/action log.**
   Для SaaS важно не только “что получилось”, но и “кто, когда, почему разрешил”.

8. **Сделать внешний UI раньше рабочего процесса.**
   Красивый интерфейс без ежедневной внутренней пользы не докажет продукт.

## 15. Какие решения должен принять Дмитрий

1. **Подтвердить текущий путь: сначала внутренний Telegram-пилот, потом SaaS.**

2. **Определить критерий успеха пилота.**
   Рекомендация: 3-5 рабочих дней, 1-3 менеджера, 50%+ черновиков полезны, 0 P0, все ошибки уходят в semantic gate.

3. **Выбрать владельца базы знаний.**
   Нужен ответственный за цены, расписания, документы, скидки, маткапитал, налоговый вычет.

4. **Решить, когда включать customer timeline в preview.**
   Рекомендация: только после разбора сложных Tallanto/семейных кейсов.

5. **Выбрать формат первого внешнего показа.**
   Рекомендация: demo tenant + безопасные реальные примеры + no live write.

6. **Решить, что считать “первой продаваемой версией”.**
   Рекомендация: client-hosted appliance, а не облачный SaaS.

7. **Определить допустимый уровень хранения секретов для внутреннего этапа и для внешнего клиента.**
   Для внутреннего этапа `.env` допустим. Для внешнего продукта нужен отдельный процесс.

8. **Решить, когда возвращаться к PostgreSQL.**
   Рекомендация: после client-hosted пилота или при реальной многопользовательской нагрузке.

## 16. Следующие 10 конкретных шагов

1. Завершить и прогнать MEGA v2 для Telegram/KB.

## 17. Итог

SaaS-направление не потеряно. Наоборот, проект уже имеет хороший фундамент: product DB, контракты, адаптеры, safety gates, audit packs, channel runtime, база знаний, customer timeline и read-only API.

Но сейчас лучший путь к SaaS - не строить SaaS. Лучший путь: довести внутренний Telegram-пилот до реального ежедневного использования, доказать пользу, затем упаковать это в повторяемую client-hosted установку, потом сделать безопасный внешний показ, и только после этого думать о настоящем облачном SaaS.
