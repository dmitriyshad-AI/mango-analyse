# ТЗ — Шаг 5: судьба дословного faithfulness-критика (поглощение)

**Статус:** ФИНАЛ v2 (аудит второго архитектора: 15 правок внесено; несущие координаты пересверены мной по коду)
**Дата:** 2026-06-07
**Дерево:** ветка `main`, главная папка.
**Основание:** ADR-001 Шаг 5; Шаг 4 снял вето критика в keep-точках; верификатор выхода работает; правило «--parallel 4» родилось из таймаутов этого критика.

## Зачем (по коду и сырью)

Дословный критик (`check_claim_faithfulness`, pipeline:1978) — последний крупный остаток «детерминизм не доверяет модели». Цена по сырью пинованных прогонов: **~1.4 (GPT-25) / ~2.0 (Claude-14) LLM-вызова на ход** (метод: bot_critic − semantic_match_attempted), таймаут-хрупкость (parallel 8 → 6 ложных хендоффов), двойная семантика с верификатором (дословный ↔ смысловой вердикт на одном тексте).

Важно (правка аудита): при unavailable путь НЕ чистый fail-close — сначала детерминированный recovery (`faithfulness_fn=None`, :3428-3445), хендофф `semantic_check_unavailable` только если recovery пуст. Деградация без критика УЖЕ обкатана в коде (второй прецедент — :5267: кандидат принимается по якорям при недоступном критике).

## Карта ролей критика (5, не 3 — выверена аудитом)

1. **Вето на готовом тексте**: главный драфт :3416-3454, repair-цикл :3471-3510.
2. **Фильтр кандидата** (~10 точек): `_hard_check` call-sites — :2370 (estimate-recover), :2479 (next_step), :3017 (estimate), :3515 (verified_fact_fallback), :3595 (composition), :3633 (coverage-repair), :3690 (coverage cite-only), :5245 (cite_only_recover); :3753 (warmth — в проде МЁРТВ: warmth_fn=None) + `_partial_yield_full_check` из :4983 (partial_yield) и :4859 (composite).
3. **Ре-проверка text-change**: subscription_llm:2461 (под `if facts:`).
4. **Пре-условие quality-фич**: partial_yield :4960, composite :4848, next_step :2470 отказываются работать при `faithfulness_fn=None` (trace `faithfulness_fn_missing`) — снос «в лоб» молча убьёт фичи и регрессирует уход-при-фактах.
5. **Генератор repair-инструкций**: unsupported критика формирует repair-промпт :3459-3464 — после сноса repair реагирует только на детерминированные findings (репараций меньше; учесть в дельте shadow).

## Дизайн

**D1. Распил.** Честно: «детерминированная часть критика» НЕ автономна — `claim_anchors_supported_by_fact` (:2625) проверяет якоря против факта-evidence, который называет LLM. Автономный детерминированный аналог — `new_concrete_anchors` по объединению фактов (:2651) — слабее: пропускает чужое число/срок из СОСЕДНЕГО факта. Компенсация — эскалационная ветка D5.

**D2. Роль 1 (вето) → снести.** LLM-критик в `_hard_check` не вызывается; рубеж держат: число-гард + `new_concrete_anchors` + верификатор (advisory, но его findings через гейт блокируют автономный маршрут — в 25-прогоне `derived_product_claim`: 5) + гейт. Класс `semantic_check_unavailable`-хендоффов от критика исчезает.

**D3. Роль 2 (фильтр кандидата) → детерминизировать.** Кандидат проверяется `new_concrete_anchors` + число-гардом, без LLM; не прошёл — отброшен (как сейчас). Прецедент :5267 уже работает так при недоступном критике.

**D4. Роль 3 (ре-проверка) → верификатору/детерминизму.** Отдельный вызов критика снимается; блокирующий исход — только от детерминированных findings.
**D4b. Роль 4 (пре-условия) → убрать**: guard'ы `faithfulness_fn_missing` заменить на безусловное продолжение (фильтрация уехала в D3) — иначе partial_yield/composite/next_step умирают. Архитектурное решение, принято мной.
**D4c. Роль 5**: repair-цикл остаётся на детерминированных findings; изменение частоты репараций — наблюдать в дельте.

**D5. Классы «ловит только критик» (из дельты shadow) — с эскалацией:**
- классы с продуктовыми ЧИСЛАМИ/сроками/условиями (cross-fact склейка) → перенос в **детерминированный** слой (co-occurrence «число+тема в одном факте»), НЕ в промпт — верификатору это сознательно запрещено калибровкой («не проверяй цены», «НЕ ФЛАГАЙ склейку двух реальных фактов»);
- нечисловые scope-классы → маппинг в СУЩЕСТВУЮЩИЙ код `derived_product_claim` (верификатор отбрасывает неизвестные коды: frozenset :5247, фильтр :5507; новые коды — только с решением Дмитрия по severity);
- ослабление «НЕ ФЛАГАЙ»-правила = пересмотр калибровки верификатора: на каждый класс позитивный И негативный NEG (false positives душат продающий тон — конфликт уже ловили).

## Теневой режим (обязательный этап до сноса)

Флаг `TELEGRAM_FAITHFULNESS_SHADOW` (default OFF): нейтрализация в 3 функциях (`_hard_check`, `_partial_yield_full_check`, reverify subscription:2461) — возвращают `unsupported=(), available=True`, а реальный вердикт ЛОГИРУЕТСЯ **пер-site**: `{"site": "main_draft|repair|partial_yield|estimate|cite_only_recover|text_change", "verdicts": [...], "available": ...}` в metadata + проброс в turn-record раннера (это транспорт, не «просто лог» — честная часть стоимости). Без пер-site дельта неатрибутируема.
**SHADOW по поведению ≈ превью ABSORB** (блокировка отключена — иначе дельту не измерить); отличия только в ролях 4 (пре-условия ещё живы) и латентности. **Только симулятор/M1, в пилотные каналы — никогда.**
Затем `TELEGRAM_FAITHFULNESS_ABSORB` (default OFF): полное D2-D4c.

## Freeze ревизии (порядок с тон-блоком)

Пара SHADOW → решение → ABSORB выполняется на ОДНОЙ пинованной ревизии; мерж тон-волн между ними запрещён; случился — shadow повторить. «Параллельно, но на разных прогонах» НЕдостаточно: тон меняет тексты → дельта критика плывёт.

## Политические решения (зафиксированы, Кодекс не решает)

1. **Автоном при verifier-unavailable**: решение Дмитрия 2026-06-06 — 1 ретрай; снова нет → в черновиках annotate-only, в автономе БЛОК автоотправки. В коде блок НЕ реализован (subscription:10344 фиксирует как будущее) → ABSORB допустим только в черновиковом режиме; **реализация блока — пререквизит включения автонома, не этого ТЗ** (вписано в Границу).
2. **Severity перенесённых классов**: маппинг в `derived_product_claim` = существующее поведение гейта; новые коды/severity — только решением Дмитрия.
3. Судьба пре-условий (D4b) — моё, принято.

## Граница (не нарушать)

- Число-гард, бренд-гарды, детекторы (`_preemptive_format_choice_finding`:4427, `_unconfirmed_schedule_finding`:4451, free-number gate), `verify_dialogue_contract_output`, authoritative gate — не трогаются.
- Верификатор: не добавлять вызовов сверх существующей схемы (verifier + retry + regen-перепроверка).
- `new_concrete_anchors` остаётся во всех точках.
- Развести reason'ы (микроправка): `semantic_check_unavailable` от runtime_error понимания (:2859-2882) → отдельный `understanding_runtime_error`; `warmth_rejected_reason` не путать (греп зашумлён).
- Подзадача раннера: отдельная роль `bot_faithfulness` (сейчас критик и semantic_match слиты в `bot_critic`, runner:194-200) — иначе Exit неизмерим.

## NEG-контроли

1. Выдумка продуктового числа → ловится число-гардом без критика.
2. Производная («в обоих форматах») → верификатор.
3. Формат-подмена цены → верификатор/гейт.
4. Кандидат восстановления с чужим якорем → отброшен детерминированно.
5. P0 при ABSORB ON → менеджер.
6. Бренд-провокация при ABSORB ON → каноничная фраза.
7. Каждый перенесённый класс: позитивный + негативный кейс.
8. Verifier-unavailable при ABSORB ON (черновиковый режим) → annotate-only, маршрут не меняется, НЕ хендофф.
9. partial_yield и composite РАБОТАЮТ при ABSORB ON (роль 4 не умерла; trace без `faithfulness_fn_missing`).
10. Cross-fact склейка (число из соседнего факта в чужой теме) → детерминированная co-occurrence ловит (если класс подтверждён дельтой).

## Exit (M1, --parallel 4, пинованные наборы 25+14, та же машина)

| Критерий | Цель | Кто меряет |
|---|---|---|
| Роль `bot_faithfulness` в llm_calls | 0 при ABSORB (базлайн ~1.4/2.0 на ход) | авто (после подзадачи раннера) |
| Хендоффы `semantic_check_unavailable` ОТ КРИТИКА (по trace, после развода reason'ов) | 0 | авто |
| Выдумки | 0 по сырью | ручной регрейд (я) |
| Дельта shadow | каждый улов «только критик» перенесён (D5+NEG) или явно принят как риск МНОЙ, пер-site | ручной регрейд (я) |
| Уход-при-фактах | не выше базлайна | ручной регрейд (я) |
| Обратный прокси (не «отвечаем хуже там, где раньше уходили»): downgrade гейта + verifier findings на автономных ответах | не выше базлайна (есть в summary) | авто |
| P0/бренд/мета | без регрессий | регрейд + NEG |

Порядок: SHADOW-прогон → мой регрейд дельты → решение → ABSORB-прогон → полный Exit. Freeze ревизии на всю пару.

## Вне scope

Инверсия гейта (после пилота); тон-волны (порядок через freeze); реализация блока автоотправки (пререквизит автонома, отдельное ТЗ); parallel >4 (отдельное решение Дмитрия даже после ухода таймаут-хрупкости).
