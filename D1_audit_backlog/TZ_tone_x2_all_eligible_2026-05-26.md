# ТЗ для Codex: тон — тёплый слой X2 в режиме all_eligible. 2026-05-26

Автор: Claude (ревьюер). Codex: сначала **аудит ТЗ** (§5), потом внедрение после ОК Дмитрия.
Безопасность/понимание/маршрут НЕ трогаем — это уже принято. Здесь только живость (тон).

## 0. Зачем
В прогоне `…understanding_handoff_tripwire_20260526_142904`: тон 58.9, тёплый слой применился
**4 раза из 32 ходов**. Фикс маршрута сделал бот честнее, но суше. Тон не растёт, потому что тёплый
слой почти не зовётся.

## 1. Корень (код)
`src/mango_mvp/channels/dialogue_contract_pipeline.py`, `run_pipeline`, блок ~759-781:
```python
if toggles.form_warmth:
    form_findings = form_check(draft, previous_bot_texts=...)
    if form_findings and warmth_fn is not None:      # <-- тепло зовётся ТОЛЬКО при штампе
        warm_candidate = warmth_rewrite(...)
        if warm_candidate:
            # ре-верификация _hard_check; берём warm_candidate ТОЛЬКО если прошёл
```
`form_check` (стр. 542) ловит только грубые признаки: штамп-зачин, near-repeat, нет шага, канцелярит.
На «пресных, но формально нормальных» ответах он молчит → `form_findings` пусто → тепло НЕ зовётся →
ответ остаётся сухим. Поэтому 4/32.

Примечание: старый `humanity_x2_rewriter` имел режим `all_eligible` (тепло на КАЖДЫЙ подходящий ход).
В v2 этот режим не управляет вызовом тепла — вызов жёстко привязан к `form_findings`. Берём из старой
реализации именно идею режима `all_eligible`.

## 2. Что сделать (направление; точную реализацию Codex предлагает в аудите)
1. **Режим all_eligible для v2-тепла.** Добавить тумблер (например `Toggles.warmth_all_eligible`,
   проброс из `subscription_llm._build_dialogue_contract_pipeline_draft`; можно завязать на уже
   существующий env `TELEGRAM_DRAFT_X2_REWRITE_MODE = linter|all_eligible`). В режиме `all_eligible`
   тепло зовётся на КАЖДОМ автономном ходу (где `route=bot_answer_self`, черновик прошёл hard-check),
   а не только при `form_findings`. Режим `linter` = текущее поведение (по штампу), оставить как опцию.
2. **Ре-верификация — без изменений и обязательна.** Выход `warmth_rewrite` ВСЕГДА прогоняется через
   тот же `_hard_check` (бренд + числа + P0 + семантика). Берём тёплый вариант ТОЛЬКО если прошёл;
   иначе остаётся исходный проверенный черновик. Тепло меняет ФОРМУ, не содержание (§13.8).
3. **Тепло НЕ зовётся** на P0/`manager_only`/fallback-ходах (там сухой узкий хендофф по дизайну).
   Только автономные ответы. (Структурно это уже так — блок тепла в автономной ветке; подтвердить.)
4. **Варьировать сухую P0-фразу.** `_dry_p0_text()` (стр. 873) сейчас возвращает дословно
   «Приняли обращение. Передам его ответственному сотруднику, он вернётся с ответом» — и повторяется
   3 раза подряд (роботизированно). Сделать 3-4 безопасных варианта с ротацией (по номеру хода/хешу).
   Жёстко: каждый вариант остаётся узким и безопасным — без новых обещаний, без запроса данных, без
   фактов/чисел. (Опционально так же `_safe_fallback_text`, стр. 877 — ниже приоритет.)

## 3. Жёсткие критерии приёмки
- Тепло применяется на БОЛЬШИНСТВЕ автономных ходов (не 4/32), средний тон заметно выше 58.9.
- **Безопасность не падает:** fabrication 0, бренд-утечек 0, P0 удержан, нет раскрытия ИИ. Это
  гарантируется ре-верификацией: любой тёплый выход, добавивший число/смысл/другой бренд/раскрытие →
  отклонён, остаётся исходный. В отчёте прогона — счётчик: сколько тёплых вариантов отклонено и почему.
- **Не делает хуже:** если тёплая перепись ухудшает/удлиняет корректный лаконичный ответ и не проходит
  hard-check → откат на исходный (это допустимо, но логировать частоту отклонений).
- P0-фраза: нет дословного повтора 3 раза подряд; варианты остаются безопасными и узкими.

## 4. Trade-off и порядок
- `all_eligible` = +1 вызов модели на каждый автономный ход (стоимость/латентность выше). Дмитрий не
  против тратить больше токенов на качество. Сделать управляемым флагом.
- **Порядок:** прогон на замер тона запускать ТОЛЬКО ПОСЛЕ перевалидации базы правильным билдером
  `build_kb_release_v6_1_team_answers.py` (`quality_passed=true`), иначе тон меряем в шуме ложного гейта.

## 4bis. Уточнения после аудита Codex (ПРИНЯТЫ — обязательны к реализации)
1. **Тепло только для реально автономного ответа.** В v2 есть `force_draft_for_manager=True`-ходы,
   которые всё равно доходят до генерации черновика, а блок тепла стоит до финального `return`.
   Явно ограничить: `eligible_for_warmth = (not force_draft_for_manager) and route == bot_answer_self`.
   Тепло НЕ должно касаться `draft_for_manager`/`manager_only`/fallback.
2. **Мета-статистика по теплу (обязательно для анализа тона).** Кроме `warmed`, логировать на ход:
   `warmth_attempted`, `warmth_mode`, `warmth_rejected_reason`, `warmth_rejected_findings`,
   `warmth_semantic_available`. Иначе после прогона снова будем гадать: тепло не вызвалось / отклонилось
   по hard-check / упало по faithfulness / не изменило текст.
3. **Режим — через существующий env, без нового флага.** Пробросить `TELEGRAM_DRAFT_X2_REWRITE_MODE`
   (`all_eligible|linter`) в `DialogueContractToggles`, не плодить отдельный тумблер.
4. **P0-ротация детерминированная** (по номеру хода/хешу диалога), варианты сухие и без обещаний/данных.
   Тесты перевести с точной строки на СМЫСЛ: `route=manager_only`, `high_risk_manager_only`, нет
   обещаний, нет запроса данных.

**Мониторинг (не блокер):** следить за ДОЛЕЙ отклонённых тёплых вариантов. Высокая доля = тёплый промпт
слишком агрессивен (добавляет смысл/числа) → отдельно ужать промпт. Сейчас просто фиксируем в логе.

## Что НЕ делаем
- Не трогаем понимание/маршрут/верификатор/бренд-скоуп/P0. Тепло — только форма.
- Не ослабляем `_hard_check`. Не тюним под холдаут.
