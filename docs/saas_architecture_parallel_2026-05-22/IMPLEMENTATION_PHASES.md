# Порядок реализации без преждевременного усложнения

## Главный принцип

Не строить SaaS раньше внутренней пользы. Каждый этап должен улучшать текущего ИИ-сотрудника или готовить переносимость без большого отвлечения.

## Фаза 1. Закрепить внутренний Telegram-пилот

Цель: понять, реально ли бот помогает.

Работы:

1. Включить постоянный feedback report.
2. Сохранять вопрос, ответ, route, flags, latency.
3. Собирать verdict сотрудника.
4. Ошибка -> тест или KB-поправка.
5. Следить за P0 и смешением брендов.

Не делать:

- email;
- сайт;
- SaaS UI;
- CRM write;
- новые внешние клиенты.

Критерий:

- 3-5 дней пилота;
- 0 P0;
- понятная доля полезных ответов.

## Фаза 2. Ввести постоянное хранение диалогов

Цель: уйти от памяти процесса и разрозненных логов.

Работы:

1. SQLite store для sessions/messages/decisions/answers/feedback.
2. Daily report.
3. Redaction для audit packs.
4. Backup pilot DB.
5. Restore dry-run.

Критерий:

- после перезапуска история пилота не теряется;
- можно объяснить любой ответ.

## Фаза 3. Отделить ядро ИИ-сотрудника от Telegram

Цель: подготовить будущие каналы без копирования правил.

Не начинать эту фазу, пока Telegram-пилот не стабилен минимум 5-7 рабочих дней без P0 и без массовых правок вручную. Иначе есть риск дестабилизировать живой пилот ради архитектурной чистоты.

Работы:

1. Описать `PilotContext`.
2. Описать `DecisionEnvelope`.
3. Сделать фасад общего ядра.
4. Подключить Telegram через фасад.
5. Добавить fake web-chat тест.

Критерий:

- Telegram больше не является носителем бизнес-логики.

## Фаза 4. Закрепить brand isolation

Цель: Фотон/УНПК как прототип tenant isolation.

Работы:

1. Явный `brand_id` в каждом решении.
2. Логи по бренду.
3. Отдельные KB/gold scopes.
4. Тесты cross-brand.
5. Startup check: token -> brand -> KB.

Критерий:

- ни один ответ не может взять чужой бренд без блокировки.

## Фаза 5. Перенести на M1 Pro

Цель: отделить продукт от рабочего MacBook разработчика.

Работы:

1. Product root.
2. Secrets outside git.
3. Launch/runbook.
4. Logs.
5. Healthcheck.
6. Backup/restore.
7. KB rollback.

Критерий:

- боты можно поднять по инструкции;
- процесс переживает перезапуск;
- есть понятная остановка.

## Фаза 6. Подключить customer timeline в preview

Цель: улучшить ответы известным клиентам, не сломав безопасность.

Работы:

1. Только read-only.
2. Не считать timeline истиной при ambiguous identity.
3. Показывать context confidence.
4. Не раскрывать клиенту CRM/Tallanto/source ids.
5. Разобрать семейные телефоны и несколько детей.

Критерий:

- бот не спрашивает известное;
- при конфликте уточняет, а не угадывает.

## Фаза 7. Подготовить внешний demo

Цель: показать продукт без риска приватных данных и live-write.

Работы:

1. Demo tenant.
2. Safe sample dialogs.
3. Dashboard/report.
4. KB/gold examples.
5. Manager handoff flow.
6. Writeback preview only.

Критерий:

- demo 10-15 минут;
- нет секретов;
- нет live CRM;
- нет реальных персональных данных в открытом виде.

## Фаза 8. Первый client-hosted pilot

Цель: поставить систему одной внешней организации под контролем.

Работы:

1. Tenant root.
2. Separate DB.
3. Separate secrets.
4. Read-only adapters.
5. Backup/runbook.
6. Support process.
7. No auto-send by default.

Критерий:

- второй клиент подключается конфигом и адаптерами, а не форком.

## Фаза 9. Hosted SaaS

Делать только после успешных client-hosted пилотов.

Работы:

- права доступа;
- журнал действий;
- PostgreSQL;
- централизованные логи;
- tenant isolation;
- billing;
- SLA;
- юридический пакет.

Критерий:

- есть реальная потребность, а не архитектурное желание.

## Ближайшие 5 практических действий

1. Закрепить feedback/metrics по Telegram-пилоту.
2. Создать persistent pilot storage.
3. Зафиксировать `DecisionEnvelope`.
4. Добавить brand isolation startup/report checks.
5. Подготовить M1 Pro runbook как операционный checklist.

Если пилот пока нестабилен, пункт 3 откладывается: сначала feedback, storage и P0-протокол.
