# Финальное ТЗ Волны 1a — калибровка `ConversationIntentPlan` + `classify_answer_safety` без LLM. Claude #1, 2026-05-29.

Базовый план: `TZ_volna_plan_6_wave_release_FINAL_v3_for_codex_2026-05-28.md` + delta. Этот файл —
финал Волны 1a после регрейда второго Claude + ответов Дмитрия по анкете.

## §0. Контекст

Волна 0 + 3a уже закоммичены (commits `8cc679b3`, `a7773991`). Кодекс работает над 3b. После 3c +
полного прогона 212 на M1 — переход к 1a (этот документ).

Главная архитектурная поправка от второго Claude: в коде уже есть **выделенный слой намерения** —
`ConversationIntentPlan` (`conversation_intent_plan.py:100`, правиловый) + `classify_answer_safety`
(`answer_safety_classifier.py:49`, авторитетный P0). Это **две согласованные ветки**, калибровать
нужно **обе** синхронно. Иначе план скажет P0, а classify — нет.

**1a = калибровка правил без LLM.** LLM-классификатор (1b) реализуется ТОЛЬКО если измерение
покажет провалы > 5% после 1a.

## §1. Бизнес-решения Дмитрия (анкета P0)

| Группа | Решение | Реализация в коде |
|---|---|---|
| A — финансовая претензия | P0 только если требование СЕЙЧАС; гипотетика, налоговая, тетради — НЕ P0 | **A-tax — провал, чинить** (правка 2). A-гипотетика, A-тетради — **уже OK** |
| B — спор оплаты | всегда P0 | OK, не трогаем |
| C — юр. угроза | всегда P0 | OK, не трогаем |
| D — недовольство | намерение, не слова; «беспокоит качество?» — вопрос | **Над-триггер** «недовол/возмущ», чинить (правка 1) |
| E — эмоция к боту | бот пытается ответить, мягко предлагает менеджера | В основном OK + добавить мягкий CTA (правка 7, опц.) |
| F — нейтральное окончание | НЕ P0 | OK |
| G — гипотетический возврат | НЕ P0, бот сам успокаивает | OK |
| Спор 1 — оплатил, передумал | **Soft handoff** (не молчание) | Сейчас жёсткий P0, **нужен новый режим** (правка 6) |

| Спор 4 — уже жалобу писал | менеджер | OK (ловится «жалоб») |
| Спор 5 — не нравится преподаватель | менеджер | **Под-триггер**, расширить (правка 4) |

## §2. Список правок (7 шт., все детерминированные)

### Правка 1 — D над-триггер «недовол/возмущ»

**Файлы:** `p0_recall_spec.py:42` (`COMPLAINT_RE`), `p0_recall_spec.py:125`
(`has_complaint_signal`), `answer_safety_classifier.py` (где читается `has_complaint_signal`).

**Что:** вынести «недовол/возмущ» из безусловного `COMPLAINT_RE` в **отдельную проверку
комбинации**:
- «недовол/возмущ» → P0 **только** при наличии финансового маркера (деньги/оплата/договор/возврат/
  средства/тариф) ИЛИ маркера услуги-претензии (преподаватель/качество/уроки/преподавание).
- Иначе → НЕ P0 (бот отвечает с аргументами и соц-доказательствами).

**Регресс:** см. `regression_suite_synthetic_siblings.md` блок 1a-D-1 / 1a-D-2 / 1a-D-3.

### Правка 2 — A-tax («налоговая возвращает деньги»)

**Файлы:** `p0_recall_spec.py:143-161` (`codes_from_text`),
`semantic_roles.py:540-549` (`_is_tax_deduction_return_question`),
`semantic_roles.py:502-527` (`_refund_frame`).

**Что:**
- В `codes_from_text` (строка ~149) — honor `refund_frame=="none"`/`payment_source=="tax_deduction"`
  + subject-проверка (если subject — налоговая/третье лицо, не поднимать `refund` код).
- В `_is_tax_deduction_return_question:544` — убрать самодисквалификацию на bare «деньги назад»
  ИЛИ различать «налоговая вернёт деньги назад» vs «вы верните деньги назад» по subject (через
  `_refund_frame` и `tag_message_roles`).

**ОСТОРОЖНО — критический регресс:** правка НЕ должна ослабить реальный «верните деньги» нашему
центру. Контрольный тест: `eval_foton_P0a_refund_claim_01` сценарий продолжает срабатывать как P0.

**Регресс:** блок 1a-A-1 (контроль) + 1a-A-5 (целевой fix tax) + 1a-A-3 (контроль).

### Правка 3 — A-гипотетика «потребую» — НЕ ДЕЛАЕМ

**Почему:** второй Claude проверил `has_marker` (`text_signals.py:16-23`) — он матчит по границе
слова через negative lookbehind `(?<![word])`. «по**требую**» НЕ матчит «требую» (перед «требую»
символ «о»). Значит `demand_hit=""` → срабатывает presale-frame «если» → не P0. **Уже работает
корректно.**

Остаточный edge — редкая формулировка «гипотетически: требую возврат» — низкий приоритет,
откладываем в backlog.

### Правка 4 — Спор 5: претензия к преподавателю (расширение)

**Файлы:** `p0_recall_spec.py:40-45` (`COMPLAINT_RE`) — добавить маркер.

**Что:** новый маркер-список → P0 (либо soft handoff):
- «не нравится … преподаватель/учитель»
- «как ведёт занятия + не нравится/плохо/непонятно»
- «претензия к преподавателю»
- «качество преподавания + плохое/низкое»

**Регресс:** блок 1a-Sp5 в `regression_suite_synthetic_siblings.md`.

### Правка 5 — Спор 3: сбой процесса справки/документа (расширение)

**Файлы:** `p0_recall_spec.py` — новый маркер-список.

**Что:** маркер задержки в комбинации с темой документа → P0/manager:
- «(месяц/долго/уже X недель/так и не) + (справку/документ/договор/доступ к ЛК)»
- «не могу получить + (справку/документ)»

**ОСТОРОЖНО:** не задеть нейтральное «когда будет справка?», «когда придёт договор?» —
это вопрос-уточнение (не P0). Требуется **маркер задержки/неудачи**, не сам факт справки.

**Регресс:** блок 1a-Sp3 (положительный) + 1a-doc-neutral (контрольный «когда справка?» → не P0).

### Правка 6 — Soft handoff (Спор 1: оплатил, передумал, верните)

**ЭТО САМОЕ КРУПНОЕ ИЗМЕНЕНИЕ.** Сейчас в `AnswerSafetyDecision` есть `p0_required: bool`. Нужен
третий режим.

**Файлы:**
- `answer_safety_classifier.py:22` (`AnswerSafetyDecision`) — добавить поле
  `escalation: Literal["hard_p0", "soft_handoff", "none"]`.
- `answer_safety_classifier.py:49` (`classify_answer_safety`) — детектор soft-handoff:
  - `paid_hit` (`semantic_roles.py:147` `REFUND_POST_PAYMENT`) — клиент уже оплатил.
  - `demand_hit` (REFUND_DEMAND) — есть требование.
  - НЕТ `legal_hit` (LEGAL_RE).
  - НЕТ серьёзного `complaint_hit` (после правки 1 — только реальная претензия).
  - → `escalation="soft_handoff"`, `p0_required=False`.
- `dialogue_contract_pipeline.py` — новая ветка маршрутизации для `escalation="soft_handoff"`:
  - Бот выдаёт **безопасный шаблонный ответ** по `refund` (из факта
    `refund_presale_policy.client_safe_text` или нового факта `refund_post_payment.client_safe_text`,
    если есть в KB).
  - Параллельно **помечает** «передал запрос менеджеру» (в metadata, не в текст клиенту).
  - НЕ молчание, НЕ автоматический шаблон «передам менеджеру».

**Открытый вопрос — формулировка safe-template для post-payment refund.** Это бизнес-решение
Дмитрия. См. §5.1 ниже.

**Регресс:** блок 1a-Sp1 — `escalation="soft_handoff"`, в ответе есть существенная информация
(не просто handoff-фраза).

### Правка 7 — E «бот старается лучше» (опционально, в объёме Волны 1a)

**Файлы:** `dialogue_contract_pipeline.py` (понимание + драфтер промпт).

**Что:** при `primary_intent` соответствующем эмоции к боту:
- НЕ устанавливать `escalation` (бот пытается ответить заново, точнее).
- Драфтер при повторной попытке получает hint «клиент недоволен предыдущим ответом, попробуй
  иначе».
- В конце ответа — мягкий CTA: «хотите, я передам ваш вопрос менеджеру для уточнения?» (не
  автоматическая эскалация, а вопрос-предложение).

**Регресс:** блок 1a-E-1/E-2 — `escalation != "hard_p0"`, ответ содержит ретрай-попытку, опция
«хотите менеджера?» предлагается.

**Если не успеваешь — отложи в Волну 1b.**

## §4. План измерения провалов (на 212 диалогах)

**Цель:** решить, нужна ли Волна 1b (LLM-классификатор) или Волна 1a достаточна.

### Шаги:

1. **После реализации правок 1-6** запустить тестовый прогон на 212 диалогах **на M1**.
2. Для каждого хода зафиксировать:
   - `ConversationIntentPlan.primary_intent`
   - `ConversationIntentPlan.refund_frame`
   - `classify_answer_safety.p0_required`
   - `classify_answer_safety.escalation` (новое поле из правки 6)
3. **Эталонные кейсы** для сравнения:
   - 4 эталонных P0 (`P0a_refund_claim_01/02`, `P0b_complaint_01/02`) — все срабатывают как
     hard_p0.
   - `P0_TRUE_POSITIVE_CASES` из `p0_recall_spec.py:89` — контрольный набор.
   - 78 синтетических siblings из `regression_suite_synthetic_siblings.md`.
4. **Метрика провалов:**
   - **False positive** (ложный P0/handoff) — % ходов, где правила сработали, а по
     анкете/эталонам не должны были.
   - **False negative** (пропущенный P0) — % ходов, где правила НЕ сработали, а должны были.
   - **Серая зона** (несогласие двух веток) — % ходов, где `ConversationIntentPlan` и
     `classify_answer_safety` дали разные вердикты.

### Критерий перехода к 1b:

- **Не нужен**, если: FP < 3% И FN = 0 (на эталонах) И серая зона < 5%.
- **Нужен**, если: FP > 5% ИЛИ FN > 0 ИЛИ серая зона > 10%.
- **Граница** (FP 3-5%, серая 5-10%): отдельное решение Дмитрия.

### 5.3. Маркер задержки для Спора 3 (справка/документ)

Какие точные маркеры считать достаточными для P0?
- «месяц» / «уже X недель» — да.
- «долго» — да.
- «не могу получить» — да.
- «когда будет?» — нет (нейтральный вопрос).

**Подтверди список или дополни.**

### 5.4. Маркер качества преподавания для Спора 5

Какие фразы считать претензией?
- «не нравится преподаватель» — да.
- «плохо ведёт занятия» — да.
- «непонятно объясняет» — да или нет?
- «слишком быстро говорит» — нет (это нейтральная обратная связь).

**Граница тонкая. Подтверди.**

### 5.5. Pattern «N балл*» из Волны 3a — пересмотр

В Волне 3a Кодекс добавил `\b\d{1,3}\s*балл\w*` в `UNSUPPORTED_PROMISE_PATTERNS`. Это полезно для
блокировки «100 баллов на ЕГЭ» как обещания результата без факта.

Но может задеть «5 баллов по итогам теста» если в KB будет такой факт. После Волны 1a + измерения
— проверить, есть ли false positive.
