# Детальный разбор слоя намерения для Волны 1a

Автор: второй Claude. Дата: 2026-05-28. Read-only. Источники: `conversation_intent_plan.py`, `answer_safety_classifier.py`, `p0_recall_spec.py`, `semantic_roles.py`, `phase11_16_subpoints_code_recheck.md`, анкета `АНКЕТА_граница_P0_для_Дмитрия` + ответы Дмитрия.

## 1. Как сейчас работает классификация намерения

В коде **две согласованные P0-ветки**, не одна. Их важно не путать при правках Волны 1a.

**Ветка 1 — `ConversationIntentPlan`** (`conversation_intent_plan.py:100` `build_conversation_intent_plan`). Правиловый слой, строится в `telegram_pilot_context_builder.py:160` и кладётся в `context["conversation_intent_plan"]`. Поля (полный список — `:25-59`): `primary_intent`, `intent_confidence` (`_intent_confidence:763`, эвристика), `refund_frame` (`none/presale_policy/dispute`), `risk_signals` (`_risk_signals:408` — маппит refund/legal/complaint/payment_dispute), `payment_method/source`, `route_bias`, `answer_policy`. P0 в плане решается так: `risk_signals = _risk_signals(normalized)` (`:133`) → `answer_plan = build_answer_plan(roles, external_p0=bool(risk_signals) or _held_p0_latched)` (`:183`) → если `answer_plan.p0_required` → `route_bias="manager_only"` (`:192-193`). То есть план P0 опирается на `risk_signals` + `roles.refund_frame` (из `tag_message_roles`) + межходовой латч.

**Ветка 2 — `classify_answer_safety`** (`answer_safety_classifier.py:49`) — авторитетный P0-классификатор на ходе. Свои regex из `p0_recall_spec.py` (`REFUND_RE:14`, `LEGAL_RE:30`, `COMPLAINT_RE:40`, `REPUTATION_RE:47`, `PAYMENT_DISPUTE_RE:52`) + читает план из контекста (`primary_intent`, `risk_signals`, `refund_frame`, `:99-109/:136/:210`) + `p0_latch` (`:143/:181`) + topic/flags. Ключевые «умные» элементы уже есть: `is_benign_hypothetical_refund` (presale, `:83`), `semantic_non_p0`-репарация (`:136-156/:210-227`), `p0_latch` для межходовой липкости.

**Взаимодействие:** план питает `classify_answer_safety` (его `primary_intent/refund_frame/risk_signals`), но `classify_answer_safety` ещё и независимо матчит свои regex по тексту хода. Поэтому при калибровке Волны 1a править нужно В ОБОИХ местах согласованно: `p0_recall_spec` (regex) + `semantic_roles._refund_frame` (frame) + (опц.) `_risk_signals`. Иначе ветки разойдутся.

**Refund-логика** (`semantic_roles._refund_frame:502`): порядок — benign-объекты/не-про-возврат → `demand_hit` (`REFUND_DEMAND:137`) → `paid_hit` (`REFUND_POST_PAYMENT:147`) → tax (`_is_tax_deduction_return_question:540`) → presale-frame (`REFUND_PRESALE_FRAME:160`, уже содержит «если/вдруг/до начала/заранее») → иначе `dispute`. `demand_hit` И `paid_hit` проверяются РАНЬШЕ presale-frame — это источник части провалов ниже.

## 2. Группы анкеты: текущая классификация vs ответ Дмитрия

| Гр. | Намерение (Дмитрий) | Сейчас в коде | Вердикт |
|---|---|---|---|
| **A** | P0 только если требование СЕЙЧАС; гипотетика, налоговая, «тетради» — не P0 | Провал только в tax-кейсе; гипотетика и «тетради» — OK | **ЧАСТИЧНО ПРОВАЛ (только tax)** |
| **B** | всегда P0 | `PAYMENT_DISPUTE_RE:52` ловит двойное списание/чарджбек/«оплатил…не вижу» | **OK** |
| **C** | всегда P0 | `LEGAL_RE:30` ловит суд/прокуратуру/Роспотреб/претензию/незаконно | **OK** |
| **D** | не слова, намерение; «беспокоит качество?» — вопрос, бот отвечает | `COMPLAINT_RE:42` ловит «недовол/возмущ» БЕЗУСЛОВНО → P0 | **ПРОВАЛ (над-триггер)** |
| **E** | бот пытается ответить, мягко предлагает менеджера | «не ответили/бесполезно/толку нет/тупой» НЕ в `COMPLAINT_RE` → не P0 | **В ОСНОВНОМ OK** (риск «ужасн») |
| **F** | не P0 | «посмотрим/не буду ждать/не надо» не в regex; `SOFT_NEGATIVE_ONLY_RE:67` гасит «подумаю» | **OK** |
| **G** | не P0, бот сам успокаивает | presale-frame «если» → `is_benign_hypothetical_refund` → не P0 | **OK** (есть edge, см. A) |

### Разбор провалов группы A (3 кейса Дмитрия)

**A-провал-1 «налоговая возвращает/вернёт деньги» (третье лицо, не наш центр) → должно быть НЕ P0.**
Механизм провала: `REFUND_RE:16` (`возвращ\w*\s+деньги`) ловит «возвращает деньги». Tax-гард `_is_tax_deduction_return_question:540` ставит `refund_frame="none"`, НО `codes_from_text:149` гасит refund только при `benign==presale_policy` или `negated` — а НЕ при frame `none/tax`. Итого: tax-вопрос с «возвращает деньги» → REFUND_RE → код `refund` → **P0**. Над-триггер. Усугубление: строка `:544` сама дисквалифицирует tax-гард, если встретит «деньги назад/верните» — то есть «налоговая, верните мои деньги назад» гарантированно P0, хотя субъект — налоговая.

**A-провал-2 «если потребую возврат» (гипотетика) → должно быть НЕ P0. ПРОВЕРЕНО: КОРРЕКТНО (не провал).**
`has_marker` (`text_signals.py:16-23`) матчит маркер по границе слова в начале (negative lookbehind `(?<![word])`), допуская суффикс. Значит «по**требую**» НЕ матчит «требую» (перед «требую» стоит «о» — словесный символ). Поэтому `demand_hit=""` → срабатывает presale-frame «если» (`semantic_roles:522-524`) → `presale_policy` → `is_benign_hypothetical_refund` → НЕ P0. Остаточный edge только для present-tense демонда внутри условия («гипотетически: требую возврат») — редкая формулировка, низкий приоритет.

**A-провал-3 «возвращаете тетради» → НЕ P0.** Сейчас КОРРЕКТНО: `REFUND_RE` ветки требуют «деньги/оплата/средства/сумма» после «возвращ», а лукахед `:15` исключает «возврат к тем/урок/материал/заняти». «тетради» не триггерит. **OK.**

### Спорные случаи

| Спор | Решение Дмитрия | Сейчас | Вердикт |
|---|---|---|---|
| 1 «оплатил, передумал, верните» | безопасный ответ + передать менеджеру (**soft**) | `demand_hit="верните"` → `dispute` → жёсткий P0 (бот молчит) | **ПРОВАЛ: над-жёстко, нет soft-режима** |
| 2 «вы навариваетесь» | граница; с фин-маркером — P0 | «наварив» невидим regex; «…верните разницу» → demand → P0; чистая эмоция → не P0 | **В ОСНОВНОМ OK** (грубо, но безопасно) |
| 3 «месяц не получил справку» | точно менеджер (P0) | нет маркера претензии → НЕ ловится → не P0 | **ПРОВАЛ: под-триггер** |
| 4 «уже жалобу писал» | менеджер | «жалоб» в `COMPLAINT_RE:41` → P0 | **OK** |
| 5 «не нравится преподаватель» | обязательно менеджер | «не нравится преподаватель» не в `COMPLAINT_RE` → не P0 | **ПРОВАЛ: под-триггер** |

> Важно: согласно процессу анкеты, Кодекс «только сужает без отдельного вперёд». Споры 3 и 5 и под-триггеры — это РАСШИРЕНИЕ; Дмитрий дал явное «обязательно/точно менеджер», то есть это и есть его «вперёд» на добавление. Отметить явно в ТЗ.

## 3. Список правок (добавить / расширить / калибровать)

**Калибровать (сузить над-триггеры):**
1. **D над-триггер «недовол/возмущ».** `COMPLAINT_RE` (`p0_recall_spec.py:42`) ловит их безусловно. По решению Дмитрия (D = намерение, «беспокоит качество?» — вопрос) — требовать комбинацию: «недовол/возмущ» P0 только с фин-маркером (деньги/оплата/договор/возврат) ИЛИ маркером услуги-претензии. Иначе → не P0 (бот отвечает с аргументами). Реализация: вынести «недовол/возмущ» из безусловного `COMPLAINT_RE` в отдельную проверку-комбинацию в `classify_answer_safety`/`has_complaint_signal`.
2. **A-tax.** Honor `refund_frame=="none"`/`payment_source=tax_deduction` в `codes_from_text` (`p0_recall_spec.py:149`): если subject — налоговая/третье лицо, не поднимать `refund`. И убрать bare «деньги назад» из дисквалификатора `_is_tax_deduction_return_question:544` (или различать «налоговая вернёт деньги назад» vs «вы верните деньги назад» по subject). **Осторожно:** не ослабить реальный «верните деньги» нашему центру.
3. **A-гипотетика — НИЗКИЙ ПРИОРИТЕТ (в основном уже работает).** Проверено: `has_marker` по границе слова, «потребую»≠«требую», «если…возврат» → presale → не P0. Остаётся только редкий edge present-tense демонда в явно условной фразе; при желании — гейтить `demand_hit` условным маркером при отсутствии `paid_hit` (`semantic_roles:505/518`). Не блокер Волны 1a.

**Расширить (под-триггеры, с явного «вперёд» Дмитрия):**
4. **Спор 5 — претензия к преподавателю.** Добавить маркер «не нравится…преподаватель/как ведёт занятия/претензия к преподавателю/качество преподавания» → soft-handoff или P0 (Дмитрий: «обязательно менеджер»). Класс — претензия к услуге.
5. **Спор 3 — сбой процесса (справка/документ).** Добавить «месяц/долго/не могу получить + справку/документ/договор» → P0/manager. **Осторожно:** не задеть нейтральное «когда будет справка?» (это группа-вопрос, не претензия). Требуется маркер задержки/неудачи («уже месяц», «так и не», «не могу получить»).

**Новый режим (нет в коде):**
6. **Soft handoff (Спор 1, оплатил-передумал-верните).** Сейчас → жёсткий P0. Нужен промежуточный режим: безопасный шаблонный ответ по `refund` (или нейтральный) + параллельная пометка «передал запрос менеджеру», без «молчания». Это тот самый `soft_handoff`, который обсуждался в ревью LLM-классификатора; в Волне 1a его можно сделать детерминированно для post-payment-refund-intent (`paid_hit` + `demand_hit` + нет legal/complaint).
7. **E «бот старается лучше»** (опц., если в объёме Волны 1a). При `primary_intent=emotion`/таргет-бот — не P0, повторная попытка ответа + мягкое «хотите менеджера?». Уже почти обеспечено тем, что E-фразы не в regex; нужно лишь не дать им случайно попасть в complaint и добавить мягкий CTA.

## 5. Открытые вопросы

1. ~~`has_marker` — граница слова?~~ ЗАКРЫТО: `text_signals.py:16-23` — матч по границе слова в начале (lookbehind), «потребую»≠«требую». A-гипотетика корректна, не провал.
2. Правка 2 (tax) рискованна: нужно надёжно отличить «налоговая вернёт деньги» (не P0) от «верните деньги, иначе пойду в налоговую» (P0). Предлагаю опираться на `payment_source`/subject из плана, а не на голый текст.
3. Спор 3 (справка) — где граница между нейтральным «когда справка?» и претензией «месяц не получил»? Нужен маркер задержки, не сам факт справки. Подтвердить у Дмитрия формулировку маркеров.
4. Soft-handoff (правка 6) — это новый режим маршрутизации, объём больше regex-калибровки; возможно, отдельный подпункт Волны 1a или связка с режимами из ревью LLM-классификатора.
5. Согласованность двух веток (план vs classify) — любую калибровку проверять в ОБЕИХ; иначе план скажет P0, а classify нет (или наоборот).
