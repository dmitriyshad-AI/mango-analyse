# Метрики продажной воронки Telegram

Статус: `candidate`.

Цель метрик: понять, двигает ли бот клиента к покупке и делает ли это безопасно.

## Ежедневные метрики

| Метрика | Что показывает | Комментарий |
|---|---|---|
| Входящие диалоги по бренду | Нагрузка и спрос | Фотон и УНПК отдельно |
| Новые лиды / известные клиенты | Тип аудитории | Не смешивать с поддержкой действующих клиентов |
| Темы первого вопроса | Главные входные потребности | Цена, расписание, лагерь, пробное, скидка |
| Время первого ответа p50/p90 | Скорость реакции | Важнее среднего значения |
| Маршруты | Автономно / черновик / менеджер / fallback | Контроль безопасности |
| Собранные поля | Класс, предмет, формат, цель, город | Минимальная квалификация |
| Следующий шаг предложен | Есть ли движение | Подбор, группа, пробное, заявка, менеджер |
| P0/high-risk переданы | Безопасность | Возврат, жалоба, юридическое, оплата |
| Повтор известных данных | Качество контекста | Сколько раз бот спросил уже известное |
| Очередь ручной проверки | Нагрузка на контроль | Все сомнительные ответы |

## Недельные метрики

| Метрика | Что показывает |
|---|---|
| Конверсия: сайт -> Telegram -> вопрос -> квалификация -> следующий шаг | Главная воронка |
| Конверсия по брендам и продуктам | Где бот полезнее |
| Топ-10 тем, где бот не смог ответить | Что улучшать в базе знаний |
| Топ-10 недостающих материалов | Что готовить для продаж |
| Доля черновиков, принятых менеджером | Качество генерации |
| Доля сильно отредактированных черновиков | Где бот ошибается |
| Повторяемые ошибки по бренду/цене/пробному/лагерю | Приоритет тестов |
| Оценка сотрудников: полезен / бесполезен / опасен | Реальная бизнес-польза |

## Метрики качества

- Ответил ли бот на прямой вопрос.
- Есть ли один понятный следующий шаг.
- Нет ли давления, искусственной срочности и пустого дожима.
- Не выдуманы цена, дата, скидка, место, наличие, срок, гарантия.
- Не смешаны Фотон и УНПК.
- P0/high-risk корректно передан менеджеру.
- Ответ не звучит шаблонно.
- Бот не спросил уже известное.
- Бот не раскрыл AMO/Tallanto/CRM/служебные правила.
- Ответ использует только клиентский и актуальный факт.

## Метрики продаж

| Метрика | Условие успеха |
|---|---|
| Lead captured | Есть контакт и понятная потребность |
| Qualified lead | Есть класс, предмет, формат/город, цель |
| Next-step rate | Клиент получил следующий шаг |
| Objection handled | Цена/время/формат/доверие обработаны без давления |
| Trial/material intent | Клиент согласился на фрагмент, пробный формат или подбор |
| Payment intent | Клиент попросил ссылку, счет, условия оплаты |
| Manager takeover success | Менеджер получил сводку и продолжил без перечитывания |
| Lost reason captured | Понятно, почему клиент ушел |

## События для логирования

| Событие | Поля |
|---|---|
| `session_started` | бренд, источник, новый/известный клиент |
| `message_received` | время, брендовый канал, тип сообщения, есть ли вложения |
| `brand_detected` | бренд, уверенность, признаки смешения |
| `topic_detected` | тема, подтемы, P0/high-risk |
| `known_context_loaded` | найден ли клиент, какие поля надежны, риски дублей |
| `facts_retrieved` | fact_id, бренд, client-safe, актуальность, маршрут |
| `route_decided` | автономно, черновик, менеджер, fallback, причина |
| `answer_generated` | маршрут, использованные факты, safety flags, gold topic |
| `slots_updated` | класс, предмет, формат, цель, город, продукт |
| `asked_known_data_again` | какое поле спросили повторно |
| `cta_offered` | подбор, группа, пробное, заявка, оплата, менеджер |
| `manager_handoff` | причина, сводка, срочность |
| `manager_review` | принято, отредактировано, отклонено, причина |
| `hard_gate_failed` | бренд, P0, выдумка, обещание, утечка служебного |
| `conversation_closed` | успех, ручная передача, клиент ушел, нужна проверка |

## Минимум для первого пилота

Чтобы не перегрузить разработку и менеджеров, для первого внутреннего пилота достаточно 10 событий:

1. `session_started`
2. `message_received`
3. `brand_detected`
4. `topic_detected`
5. `known_context_loaded`
6. `route_decided`
7. `answer_generated`
8. `slots_updated`
9. `manager_handoff`
10. `manager_review`

Остальные события можно добавить после первых 1-2 недель, если они реально нужны для решений.

## Ежедневный отчет

Минимальный отчет для руководителя:

```text
Дата:
Бренд:
Новых лидов:
Квалифицировано:
Получили следующий шаг:
Передано менеджеру:
P0/high-risk:
Среднее/медианное время ответа:
Топ-5 тем:
Топ-5 ошибок:
Что срочно поправить:
Что можно автоматизировать дальше:
```

## Смысловые пороги

Для пилота считать тревожным:

- больше 1 брендового смешения за день;
- любой автономный ответ по P0;
- любое обещание "место есть", "записали", "оплата прошла" без подтверждения;
- больше 10% ответов, где бот спрашивает уже известные данные;
- больше 20% ответов, которые менеджеры сильно переписывают;
- повторяющиеся ошибки по пробному, оплате, скидкам или лагерям.
