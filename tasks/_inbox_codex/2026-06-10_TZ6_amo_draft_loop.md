Это несущая гипотеза И журнала unedited_rate, И ИСТОРИИ бота (строки «Ответ: …» строятся из тех же исходящих). ## Безопасность (всё — тестами, не словами)

- **Транспорт-обёртка, DEFAULT-DENY для ВСЕХ методов**: любой `(method, path)` вне явного allowlist → исключение, включая GET (у Wappi бывают побочные эффекты за GET). Allowlist: перечисленные читающие GET Wappi; AMO — перечисленные GET + единственный POST `/api/v4/leads/{id}/notes`. NEG обязателен: «GET на путь вне allowlist → исключение». Обёртка покрывает Wappi/AMO-клиентов поллера; LLM-канал провайдера (codex exec, sandbox read-only, egress только к LLM-бэкенду) — вне её по дизайну, к Wappi/AMO не ведёт.
- Примечания только в allowlist сделок (`require_note_allowed`) + бренд-ассерт пары (п.4).
- **Идемпотентность ДВУсторонняя**: ДО POST — запись `{profile_id, chat_id, message_id, status: note_pending}`, после успеха — `note_written`; на старте: (а) обработанные message_id грузятся из журнала; (б) `note_pending` БЕЗ `note_written` → ОДНА повторная попытка тем же текстом (`bot_draft_text` уже в журнале) со статусом `note_retried`, повторный провал → `manual_review` (черновик не теряется молча); (в) открытые 4-часовые окна классификации пересобираются реплеем журнала. State-файл — атомарно (tmp+rename). NEG: «примечание записано, state потерян, рестарт → второго примечания нет» И «крах между note_pending и POST → после рестарта ровно одна повторная попытка».
- STOP-файл `~/.mango_secrets/STOP_DRAFT_LOOP`: полный стоп — БЕЗ вызовов бота и БЕЗ записи в AMO, только fetch + сырой журнал; входящие при STOP НЕ помечаются обработанными (после снятия STOP бэклог сгребается объединением в один вызов на чат).
- ПДн: журнал и state — `~/.mango_local/draft_loop/` (ВНЕ репо; дефолт phase1 `runs/...` не использовать — runs/ не в .gitignore). Пункт приёмки: после live-прогона `git status` чистый.
- Секреты только из env-файла; backoff при 429 обоих API.

## Журнал unedited_rate (один, не два)

Расширить СУЩЕСТВУЮЩУЮ схему phase1 `ManagerEditLogRecord`/`append_manager_edit_log` (bump schema_version) полями: profile_id, chat_id, message_id, matched_message_id, draft_route, match_class, ratio, draft_ts, sent_ts, window_closed.

Правило классификации (одно, без двусмысленностей):
- Каждый черновик живёт СВОЁ окно 4 ч и остаётся сравнимым до конца окна. `superseded` (появился новый черновик того же чата) — ПРЕДВАРИТЕЛЬНАЯ пометка, не финал: менеджер мог отправить текст старого черновика позже.
- Каждое исходящее в окне сравнивается со всеми сравнимыми черновиками чата и засчитывается максимум ОДНОМУ — лучшему по ratio (при равенстве — более позднему черновику). `matched_message_id` фиксирует связь. Двойной зачёт одного исходящего запрещён.
- Финальный класс черновика при закрытии окна — по лучшему засчитанному ему ratio: unedited ≥0.95 / edited ≥0.5; не дотянул → `superseded` (если был заменён) / `replaced` (если исходящие были) / `no_reply` (если не было; по timestamp, лениво при первом цикле после истечения окна).
- `_normalize`: casefold, ё→е, схлопнуть пробелы/пунктуацию, срезать эмодзи. Ratio писать всегда; пороги пересмотрим по данным пилота.
- NEG: «текст superseded-черновика отправлен позже → у него unedited»; «одно исходящее, два похожих открытых черновика → unedited только у одного».
