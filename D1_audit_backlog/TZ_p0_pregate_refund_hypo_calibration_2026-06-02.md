# ТЗ — калибровка p0_pregate на гипотетический предпродажный возврат. Для Кодекса. 2026-06-02.

Автор: Клод #1. Независима от планнера/тона — можно делать ПАРАЛЛЕЛЬНО, пока M1 гонит agreement. Это
требование #1 из регрейда Прогона 1 (refund-hypo латч).

## Проблема (по сырью Прогона 1)
Гипотетический возврат ДО оплаты на ПОВТОРНЫХ ходах защёлкивается в `hard_p0` и уходит в чистый хендофф, теряя
тёплый лид. Бот отвечает верно на 1-м ходу («возвращается остаток неистраченных средств»), но на «если уже
начнём… вернёте?», «в целом, без договора, вернут остаток?» — вход `p0_pregate` помечает hard_p0, хотя клиент
ЯВНО говорит «в целом», «без договора», «до оплаты», «просто спрашиваю». Корень — ВХОДНОЙ классификатор, не гейт.

## Анкеры
- `p0_pre_gate` (dialogue_contract_pipeline) → классификатор риска (answer_safety_classifier).
- `is_benign_hypothetical_refund` (p0_recall_spec) — уже различает доброкачественный гипотетический возврат.
- `_presale_refund_policy_template` (pipeline:2911) — спокойная справка про остаток.

## Что сделать (хирургично, осторожно — это P0-слой)
1. В p0_pregate/классификаторе: при refund-сигнале СНАЧАЛА проверять маркеры гипотетичности
   («в целом», «без договора», «до оплаты», «просто спрашиваю», «если передумаю», «гипотетически») —
   и если они есть И нет реальной претензии → НЕ hard_p0, а refund_presale (разрешить presale-справку).
2. Маркеры должны работать на ПОВТОРНЫХ ходах (латч не должен перебивать их): даже если refund звучал раньше,
   на новом гипотетическом ходу с маркером — снова presale-справка, не hard_p0.
3. Реальная претензия НЕ ослабляется: «оплатил + нет занятий / верните деньги недоволен / двойное списание /
   суд / жалоба» → hard_p0 manager_only как есть.

## ОБЯЗАТЕЛЬНЫЙ негатив-контроль (правило #4 — иначе правка не закрыта)
- **NEG-защита:** реальная претензия возврата → hard_p0 manager_only (НЕ ослаблено). Контрольный кейс:
  «оплатил информатику, занятий нет, верните деньги» → manager_only.
- ПОЗИТИВ: «в целом, до оплаты, если передумаю — вернёте?» (1-й И 3-й ход подряд) → presale-справка про
  остаток, route не hard_p0.
- Граница: «верните деньги» БЕЗ маркеров гипотетичности → hard_p0 (как сейчас).
- двойное списание / «оплатил — нет доступа» / юр.угроза / жалоба на качество → hard_p0.

## Рамки
Хирургично: только распознавание refund-гипотетики во входном слое. НЕ трогать выходной гейт, бренд, другие
P0-категории, планнер. stable_runtime не трогать.

## Тест выхода
1. Офлайн на сохранённых refund-диалогах Прогона 1 (eval_*_refund_hypo_*): гипотетические ходы больше НЕ
   hard_p0 (presale-справка), а refund_claim_* остаются hard_p0.
2. NEG-контроль зелёный (реальная претензия P0 не ослаблена).
3. Полный pytest зелёный (кроме ~9 инфра).
4. Живой прогон НЕ обязателен для приёмки правки (офлайн на сохранённых refund-диалогах достаточно); финальная
   проверка — в общем прогоне позже.

## Отчёт
Изменённые строки; офлайн-результат на refund_hypo (стало presale) + refund_claim (остался P0); NEG-контроль;
ревизия коммита для моей проверки по сырью.
