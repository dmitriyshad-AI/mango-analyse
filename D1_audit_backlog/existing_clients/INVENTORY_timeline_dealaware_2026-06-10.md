Дата: 2026-06-10. Трек «действующие клиенты» (бриф BRIEF_existing_clients_track_2026-06-10.md), задача 1.
Метод: чтение кода двумя независимыми субагентами + моя перепроверка по сырью, прогон тестов read-only в песочнице, git-история, артефакты аудитов. Код не менялся.

## 1. Резюме (полстраницы)

Оба модуля реально существуют, обширны и покрыты тестами: customer_timeline (19 файлов, 11 таблиц SQLite) и deal_aware (6 стадий, 12 AI-полей в сделку). Тесты сегодня: **241 зелёный, 1 красный** (протух от времени, см. §6). Собрана каноническая база v5 от 21.05: **16 239 клиентов, 116 086 событий за 2024-01…2026-05**, включая 64 886 выжимок звонков. В CRM реально записано 20 сделок (13.05, проверено перечитыванием) — но это шло ОТДЕЛЬНЫМ скриптом с env-подтверждением и одобрением оператора; сам конвейер целиком в dry-run (safe_to_write=False на каждой стадии). Пакет на 709 сделок подготовлен для построчного ревью РОП; продолжение живой записи = дополнительная работа, не «включить флаг».

Главные факты для проектирования профиля: (а) **профиля-сущности нет** — есть события, нет агрегата «поля с происхождением»; (б) **бренд — слабое место**: оба бренда живут в одном tenant_id="foton", бренд не колонка, а тег по ключевым словам, у 13 597 из 16 239 клиентов (84%) бренд неизвестен; (в) **связь с ботом сознательно разорвана** в трёх местах (флаги чтения, allowed_for_bot=0 на всех 77 482 фрагментах, crm_context={} в цикле черновиков); (г) обновление только батчевое снимками — событийного нет; (д) обе ветки заморожены с ~23.05, конвейер новых клиентов ушёл вперёд без них.

## 7. Выводы для проектирования профиля клиента (вход в задачу 2)

1. **Фундамент переиспользуем**: хранилище+контракты timeline (происхождение уже есть), нормализация телефона, 12 типов ключей сшивки, модель допуска allowed_for_bot/requires_manager_review, цепочка одобрения для выдачи менеджеру. Новое хранилище профилей строить РЯДОМ с timeline (отдельные таблицы), не внутри базы знаний продукта — совпадает с рамкой брифа.
2. **Профиль = слоты ТЗ-8, агрегированные по клиенту**: одна механика (поле+значение+источник+цитата+время), два горизонта (диалог/клиент). Конфликт значений — «позднее побеждает, старое superseded» — уже постулировано в ТЗ-8, тянуть в профиль ту же семантику.
3. **Бренд — блокер №1**: пока бренд не структурное поле и 84% клиентов без бренда, требование «бренды строго раздельны» в профилях невыполнимо. В схеме профиля бренд должен быть обязательным полем с происхождением; для сшивки каналов решить: один клиент с полями по брендам или два профиля.
4. **Событийной модели нет — проектировать с нуля**: сейчас только батчевые снимки. Триггер «диалог затих» НЕ реализован нигде: journal.jsonl цикла черновиков пишет события свободной формы с created_at UTC по ключу (profile_id, chat_id) (draft_loop.py:127-151) — затишье придётся ВЫЧИСЛЯТЬ по зазору времени, поле/механизм для этого предстоит спроектировать. «Звонок расшифрован» — тоже не поток: выжимки приходят батчами. Журнал — подходящая опора, но это кандидат-источник, не готовый механизм.
5. **Сшивка чат-ид ↔ контакт CRM уже спроектирована и реализована** в цикле черновиков (резолвер с кэшем отрицательных результатов, кандидат только в журнал) — профилю нужен тот же резолвер с тем же правилом «неоднозначность → не сшивать молча».
6. **Запись выжимки в карточку CRM** — механизм записи в КОНТАКТ уже существует (send_contact_custom_field_update, amo_integration.py:873), но три вещи придётся достроить: (а) защита полей контакта сейчас — только {Id Tallanto, Филиал Tallanto} против 5 полей у сделки (Телефон/ФИО/Email не защищены!); (б) гейта качества для контакта нет (deal_quality_gate целит в сделку); (в) истории изменений поля нет ни у сделки, ни у контакта. Переиспользовать паттерны deal_aware (гейт, откат, снимок до записи), не сам код.
7. **Свежесть/даты** — считать от даты события/снимка, не от now() (урок красного теста).

## 8. Проверено лично vs предположение
