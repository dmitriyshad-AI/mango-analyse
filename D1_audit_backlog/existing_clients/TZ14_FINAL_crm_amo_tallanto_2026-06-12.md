## Шаг 0 (серверный Кодекс, БЛОКЕР AMO-контура). Починить AMO-токен коннектора

Коннектору — собственный способ жить с токенами AMO: либо отдельная интеграция (свой client_id/secret/refresh), либо общее хранилище токенов с основным процессом (один источник правды, refresh в одном месте, остальные читают).
NEG: работа основной интеграции (refresh в agent_runtime/боте) НЕ отзывает токен коннектора — проверить параллельным вызовом после refresh. Смоук: amo_api_get users. Заодно: WAF пропускает запросы по токену независимо от User-Agent (сегодня резал Python-urllib — выяснено пробой).

## Шаг 1 (D4). Свежие данные AMO + пересчёт дублей + CSV-пакеты

Выгрузка контактов и сделок через коннектор (постранично, limit 50, пауза ≥0,5 с, ретраи на 429) → локальная база среза → пересчёт: дубль-кандидаты (телефон + совпадение нормализованного имени ученика), общие телефоны (2+ семей), многодетные (не дубли). Выход: (а) CSV «живые дубли» (только с активной сделкой/учёбой) на ручное слияние сотрудниками — реюз контура amo_duplicate_after_staff_done; (б) CSV «общие телефоны» на разметку; (в) счётчик-база метрики «новых дублей/неделю».
NEG: в AMO ничего не пишется; повторный прогон даёт те же CSV (идемпотентность); телефоны нормализуются единым utils/phone.

## Шаг 2 (D4). Скан новых заявок: примечание «семья» + черновик «перезвоните»

Скан новых сделок раз в 10-15 мин (паттерн morning scan, без вебхуков; журнал обработанных lead_id). Телефон найден в профильной базе → авто-примечание в сделку: «Телефон известен: семья [родитель]. Дети: A (учится, предмет класс [бренд]), B (обсуждали лагерь [бренд]). Уточните, о ком разговор» (механизм примечаний обкатан пилотом). Телефон у 2+ семей → нейтральное «Общий телефон, см. карточки» без имён. «Перезвоните мне» из диалогов бота → ЧЕРНОВИК задачи-звонка (поля: тип звонок; привязка к активной сделке/контакту; текст с брендом и цитатой; дедлайн из фразы по таблице DELTA §3; dedupe по (chat_id, дата, intent)) — создание задач ЗА ФЛАГОМ, default OFF до отдельного «да» Дмитрия.
NEG: повторный скан не дублирует примечания; сделка без телефона — пропуск со счётчиком; флаг OFF → задачи не создаются вообще.

## Шаг 3 (D4). Карточка контакта v1 «ИИ: профиль клиента»

Как в DESIGN v2 §4-5 (одобрено): поле-textarea в карточке ученика; строки: Ученик / Семья / Договорённость семьи / Возражения / Tallanto-статус / отметка ИИ; бренд-метки [Фотон]/[УНПК]/[—]; единица записи — семья (пакет карточек, hash семьи); телефон 2+ семей → строка «Семья» не пишется; read-before-write (правка человека → manual_review); защита полей (+Телефон/ФИО/Email, отказ пакета с журналом); гейты (identity уникальный, стейл >6 мес, сырой телефон, длина, бренд-смешение в строке); снимок+журнал (product_data/customer_profiles/contact_card_journal/, расширить pii-gitignore-тест)+откат; dry-run default. Старое поле «Авто история общения» не трогаем (решение Дмитрия).
Этап А: 20 семей пилотной зоны, пакет на ревью Дмитрию, рядом старое поле для сравнения.
NEG: из DESIGN v2 §7 целиком (защищённое поле → отказ; повтор = 0 записей; контакт без профиля → пропуск; identity mismatch → блок).

## Шаг 4 (D4). Live-карточка клиента для бота (live_card v1)

Достройка build_live_tallanto_context до контракта DELTA §2: + баланс (`most_abonements.num_visit_left` по contact_id, активный абонемент), + расписание ближайших занятий групп клиента (`class_relations` → `most_class` active: date_start, время, аудитория), + места/набор (`remaining_seats`, статус). Кэш TTL: платежи/баланс 15 мин, расписание 6-24 ч, места 1-6 ч. Стоп-правила: телефон 2+ семей / бренд канала ≠ филиал-бренд / API недоступен → карточки НЕТ (протухшее без маркера не выдаётся); ФИО преподавателей фильтруются на источнике. Бренд из `filial` по утверждённому маппингу (вопрос Р-5 ниже). Потребление карточки в промпте — НЕ в этом ТЗ (главный архитектор).
NEG: multi-phone → нет карточки; API down → нет карточки и нет ошибки бота; неактивные группы (status=notactive) не попадают; фильтр по active — формат фильтра tallanto_select подтвердить пробой в начале шага.

## Шаг 5 (D4, ПОСЛЕДНИЙ, отдельное «да» Дмитрия). Запись в Tallanto

Как в DESIGN v2 §12: write-метод в клиенте (сейчас его нет) → проба на 1 тестовой карточке (частичная запись не обнуляет поля; поле вне синка Tallanto→AMO; чтение после записи == ожидаемому) → история общения в отдельное НОВОЕ поле; интересы — один маркерный блок «=== Рекомендация ИИ === … === конец ===» после человеческого текста; людское не редактируется никогда.
