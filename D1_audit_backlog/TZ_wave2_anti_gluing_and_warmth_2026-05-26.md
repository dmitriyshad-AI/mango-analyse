## §1. Семантическая проверка: структурные claims + анти-склейка (V1+V2)
- Где: `build_faithfulness_prompt` (≈стр.514) + `check_claim_faithfulness` (≈стр.528) в
  `dialogue_contract_pipeline.py`. Сейчас семантика просит плоский `{"unsupported":[...]}`.
- Что сделать:
  - **V2:** просить СТРУКТУРНЫЙ ответ — список `{claim, evidence_fact_key, verdict}`. Код проверяет
    машинно: если `evidence_fact_key` пуст ИЛИ не из переданных `facts` хода → claim неподтверждён.
    Не верить прозе модели — сверять ключ с реальными фактами хода.
  - **V1:** явно инструктировать: каждое атомарное утверждение должно подтверждаться ОДНИМ
    фактом-источником; если утверждение собрано из ДВУХ разных фактов (свойство одного объекта
    приписано другому) → `unsupported` с пометкой «склейка». Пример в промпт: «кабинет на МТС Линк»
    (кабинет — из факта без названия платформы; МТС Линк — из факта про вебинары; вместе — выдумка).
  - Сохранить **fail-closed** как сейчас: кривой JSON / `available=False` → менеджеру.
- Приёмка: на регресс-наборе (§5) склейка → `unsupported`; на легитимных (факт + общая человеческая
  фраза) шума нет.

## §2. Детерминированный класс «именованная сущность вне фактов» (V7)
- Где: `verify_output` (≈стр.846), рядом с проверками бренд/число/мета/P0.
- Что сделать: новый finding `unsupported_entity` — именованные сущности в КЛИЕНТСКОМ тексте
  (платформы: МТС Линк/Webinar; продукты; ФИО; конкретные адреса), которых НЕТ в переданных `facts`
  хода → finding. Обязателен **whitelist общих слов** (Москва, физика, классы, дни недели, бренд
  активного канала и т.п.), чтобы не флагать нейтральное.
- Зачем: ловит сущность, которой нет в фактах, детерминированно — второй барьер к склейке, когда
  числовой grounding молчит.
- Приёмка: сущность не из фактов → finding; нейтральные слова и сущности из фактов — чисто; **не
  растит over-handoff** на пустом месте (проверить на чистом холдауте — добавочных хендоффов нет).

## §3. Тёплый слой: только форма + анти-склейка (кластер T1/C1 + V5)
- Где: `build_warmth_prompt` (≈стр.577), `warmth_rewrite` (≈стр.601).
- Что сделать:
  - **T1/C1:** в промпт явно — «факт-предложения копируй ДОСЛОВНО, переписывай только зачин/связки/
    финал; не добавляй чисел/имён/сущностей/условий вне фактов». Идея фактического скелета: фактовые
    предложения — неизменное ядро, тепло работает только с обрамлением.
  - **V5:** явный запрет склейки — «не соединяй два факта в новое утверждение, не приписывай свойство
    одного объекта другому»; дать пример запрещённой склейки в промпт.
- Приёмка: доля отклонений тёплого по `unsupported_claims` падает (меньше материала для отклонения);
  склейка в тёплом выходе не появляется на регресс-наборе.

## §4. Калибровка ре-верификации тёплого (V4)
- Где: блок тёплого в `run_pipeline` (≈стр.810-825), где `warm_candidate` проходит `_hard_check`.
- Что сделать: при отклонении различать причину — «добавил факт/число/бренд/сущность/условие» (откат
  ОБЯЗАТЕЛЕН) vs «перефразировал без новой конкретики» (можно принять). Сейчас любой `unsupported` →
  откат; цель — не отклонять безобидную живость. Граница СТРОГАЯ: любое НОВОЕ число/имя/сущность/
  условие → откат; на спорном — fail-closed (откат).
- Приёмка: доля отклонений тёплого ниже прежней (была ~30-40%) БЕЗ роста реальных пропусков склейки/
  выдумки на регресс-наборе.

## §7-bis. Уточнения после аудита Codex (ПРИНЯТЫ — обязательны к реализации)
1. **`evidence_fact_key` сверять ТОЛЬКО с `facts` текущего хода** — не с каталогом и не со всей KB.
   Иначе снова замаскируем склейку соседним фактом.
2. **`supported` от LLM недостаточно — машинная проверка ЯКОРЕЙ.** После того как LLM сопоставил
   claim→`evidence_fact_key`, КОД проверяет, что якоря claim (числа, даты, платформы, продукты, адреса,
   ФИО, условия) реально есть в ЭТОМ ОДНОМ факте. Нет «МТС Линк» в факте про кабинет, на который
   ссылается claim → `unsupported/glued`. (Это навык «верификация вместо доверия» применён к самой
   семантике — не верить вердикту модели, сверять машинно.)
3. **V7 — узкий фильтр, не общее распознавание сущностей.** Ловим только рискованные: платформы,
   адреса, именные продукты, ФИО рядом с ролью. Whitelist обязателен: Москва, предметы, классы, дни
   недели, ЕГЭ/ОГЭ, онлайн/очно, активный бренд, «личный кабинет» без названия платформы.
4. **V7 — ВТОРОЙ барьер, главный фикс склейки — V1/V2.** Если оба факта пришли в `retrieval.facts`, V7
   не поймает (сущность есть в фактах). Главное «одно утверждение = один факт» — это V1/V2.
   (Совпадает с подклассом B моего регресс-набора.)
5. **Тёплый: «перефразировал» vs «добавил смысл» — по НОВЫМ ЯКОРЯМ, не по похожести текста.** Сравнивать
   якоря исходного ответа и тёплой версии: новая цена/дата/платформа/адрес/условие/обещание → откат.
   Спорное → откат (fail-closed). Похожесть текста как критерий НЕ использовать (хрупко).
6. **Регресс-набор склейки — обязательный pytest-fixture** `tests/fixtures/dialogue_contract_gluing_regressions.jsonl`,
   а не `audits/_inbox` (там история). Гонять как обязательный гейт.
7. **Не ломать внешний интерфейс.** `FaithfulnessResult` расширить структурными claims (новое поле), но
   `unsupported_claims` оставить как совместимый итог для текущих метаданных/тестов.
