## Почему эти 2 вместе и почему аккуратно
2b.4 = **price + format_choice** — одна «формат-ось» (реестр помечал дубль: `price.format_match` ↔
`format_choice` на planner-поле `format`). НО price — самое тонкое из всех правил:
- значения цен ДИНАМИЧЕСКИЕ (из KB-фактов, не константы — в отличие от рассрочки/скидок);
- формат-зависимый подбор цены УЖЕ работает в `_price_for_composition` (pipeline:1894);
- критичен формат-гард t1 (онлайн-вопрос ≠ очно-цена).
Поэтому price НЕ переписываем — **переиспользуем существующий подбор**, а не реализуем заново.

## Правила переноса

### price (ДИНАМИЧЕСКИЕ значения — только из фактов)
```yaml
data_rules:
  values_from_facts_only: true     # цену НИКОГДА не хардкодить/выдумывать — только из KB-факта
  format_match: true               # цена ДОЛЖНА соответствовать формату вопроса (очно/онлайн) — гард t1
  grade_scoped: true               # цена под класс
  reuse_existing_selector: _price_for_composition   # формат+класс-подбор брать отсюда
blocking → гейт:
  - price_format_mismatch          # цена не того формата → downgrade (формат-гард остаётся в verify_output)
  - fact_grounding                 # цены нет в фактах → downgrade/уточнение, НЕ выдумать число
  - cross_brand
```
ВАЖНО: правило price должно вернуть None (→ фолбэк/уточнение), если формат вопроса не определён или цены под
этот формат/класс нет в фактах. Никаких «средних» или «примерных» чисел.

### format_choice (UNPK, не решать за клиента)
```yaml
data_rules:
  never_choose_for_client: true    # «онлайн ИЛИ очно?» → дать ОБА варианта
  disjunctive_question: present_both
  ambiguity_memory: true           # клиент не уточнил формат → не фиксировать (known_format="")
  single_format_no_invention: true # в фактах только очный → не выдумывать онлайн
route=bot_answer_self
```

## Общие инварианты
- значения — из фактов; route=bot_answer_self; blocking исполняет ВЫХОДНОЙ ГЕЙТ.
- brand_split: цену/формат отвечать только в active_brand, не сравнивать бренды.
- price и format_choice опираются на ОДНО planner-поле `format` (устранить дубль формат-логики).
- НЕ расширять `_manager_route_migrated_rules_override_allowed` на price/format.
- диспетчер — та же точка; фолбэк 4 оставшихся (trial/schedule/camp/enrollment) цел; константы не удалять.

## Тест выхода
1. **Офлайн-регрессия** на 58 диалогах: 4 немигрированных (trial/schedule/camp/enrollment) — дифф 0; формат-гард

2. **НЕГАТИВ-тесты (обязательны):**
   - **формат-match (t1):** «онлайн, сколько?» → онлайн-цена (41800/69900 УНПК), НЕ очно (49000/82000); и
     наоборот.
   - **grounding:** цены под формат/класс нет в фактах → НЕ выдумать число (уточнение/менеджер).

   - **format_choice дизъюнктив:** «онлайн или очно?» → ОБА варианта, не выбрать за клиента.
   - **single format:** в фактах только очный → не выдумать онлайн.
   - **ambiguity memory:** клиент не уточнил формат → не зафиксировать формат молча.
   - **cross-brand:** цена/формат только active_brand, без сравнения.
   - **P0-override:** price/format НЕ перебивают manager_only/P0.
3. **Полный pytest** зелёный (те же ~9 инфра-фейлов).
4. **Один точечный прогон** по price/format × оба бренда + формат-перепутки (~15-18 диалогов, `--parallel 4`):
   убедиться, что цена соответствует формату, выдуманных чисел нет, на «онлайн или очно?» даются оба варианта.
