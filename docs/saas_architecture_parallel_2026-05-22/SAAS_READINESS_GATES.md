# Критерии перехода от внутреннего проекта к SaaS

## 1. Главный принцип

Проект нельзя называть SaaS-ready. Сейчас это внутренний ИИ-сотрудник в стадии пилота.

Текущий статус на 2026-05-22: `internal_pilot_ready` в процессе закрепления. До `internal_production_ready` не хватает постоянного хранения диалогов, ежедневного feedback-отчёта, устойчивой работы несколько дней без P0 и понятного владельца базы знаний.

Переход к внешнему продукту возможен только через измеримые ступени:

```text
not_ready
-> internal_pilot_ready
-> internal_production_ready
-> external_pilot_ready
-> saas_ready
```

## 2. `not_ready`

Состояние: нельзя расширять использование.

Признаки:

- нет стабильной базы знаний с `semantic_pass`;
- нет разделения брендов;
- нет P0-gates;
- нет логов диалогов;
- нет feedback loop;
- бот выдумывает факты;
- бот смешивает Фотон/УНПК;
- нет kill switch;
- невозможно понять, почему был дан ответ;
- нет владельца базы знаний.

Решение: не запускать автономность, не подключать внешних клиентов.

## 3. `internal_pilot_ready`

Состояние: можно тестировать внутри компании.

Минимальные критерии:

- два бренда разделены;
- active KB имеет `semantic_pass=true`;
- P0 fast tests проходят;
- есть kill switch;
- AMO/Tallanto только read-only;
- live CRM write запрещён;
- bot answers логируются;
- сотрудник может отметить ответ как полезный/плохой/опасный;
- ошибки уходят в регистр;
- P0-темы всегда уходят менеджеру;
- бот не раскрывает, что он ИИ/бот;
- бот не раскрывает AMO/Tallanto/source ids.

Рекомендуемый минимум:

- 1-3 сотрудника;
- 3-5 дней наблюдения;
- 0 P0;
- ежедневный feedback report.

## 4. `internal_production_ready`

Состояние: можно использовать внутри компании как рабочий инструмент.

Критерии:

- 1-3 менеджера реально пользуются ежедневно;
- 7-14 дней без P0;
- не менее 50% ответов полезны без полной переписки менеджером;
- есть постоянное хранилище диалогов;
- есть daily metrics;
- есть регулярный semantic review;
- есть владелец KB;
- есть процесс обновления KB;
- есть rollback KB version;
- есть M1 Pro/server runbook;
- есть backup/restore dry-run;
- есть monitoring процесса;
- есть no-live-write policy для CRM;
- есть понятная инструкция сотрудникам.

Метрики:

- входящие сообщения;
- автономные ответы;
- manager_only;
- fallback;
- CRM-context usage;
- asked-known-data-again;
- useful verdict rate;
- dangerous verdict count;
- latency.

## 5. `external_pilot_ready`

Состояние: можно показать или поставить одному внешнему клиенту как пилот под контролем.

Критерии:

- внутренний контур доказал пользу;
- есть demo tenant;
- нет приватных данных в demo;
- установка повторяема;
- данные клиента изолированы;
- токены вынесены из git;
- backup/restore проверены;
- есть support runbook;
- есть договорённость по данным и хранению;
- live-send по умолчанию выключен;
- live CRM write по умолчанию выключен;
- внешнему клиенту честно описаны ограничения.

Нельзя:

- обещать автономного продавца;
- обещать полный SaaS;
- обещать поддержку всех CRM/телефоний;
- включать запись в CRM без staged gate.

## 6. `saas_ready`

Состояние: можно думать о настоящем SaaS.

Критерии:

- несколько внутренних или client-hosted пилотов успешны;
- есть повторяемая установка;
- есть роли и права;
- есть настоящая tenant isolation;
- есть журнал действий;
- есть централизованные логи и алерты;
- есть upgrade/migration process;
- есть secure secret management;
- есть юридический пакет по данным;
- есть SLA/support model;
- есть billing/product packaging;
- есть решение по PostgreSQL/серверному хранению;
- есть тесты на утечки между tenant-ами;
- есть аварийный stop all automation.

## 7. Готовность по блокам

| Блок | Internal pilot | Internal production | External pilot | SaaS |
|---|---|---|---|---|
| KB semantic_pass | Обязательно | Обязательно | Обязательно | Обязательно |
| P0 gates | Обязательно | Обязательно | Обязательно | Обязательно |
| Feedback loop | Минимальный | Полный | Полный | Полный |
| Brand isolation | Обязательно | Обязательно | Tenant isolation | Tenant isolation |
| Storage | Логи/SQLite | SQLite | Изолированный root | Server DB |
| Rights | Manual | Роли внутри | Роли клиента | Полный RBAC |
| CRM write | Запрещено | Staged only | Staged only | Policy-based |
| Auto-send | Очень ограниченно | Ограниченно | Off by default | Configured |
| Monitoring | Базовый | M1/server | Client runbook | Central |

## 8. Stop conditions

Любой переход останавливается, если:

- найден P0;
- бот смешал бренды;
- бот выдумал цену/дату/скидку;
- бот раскрыл внутренний источник;
- бот повторно запросил известные данные в критичном сценарии;
- сотрудник отметил опасный ответ;
- KB обновилась без semantic review;
- процесс запущен без kill switch;
- live write включён без approval.

## 9. Решение для текущего проекта

Текущий целевой статус: `internal_pilot_ready -> internal_production_ready`.

До `external_pilot_ready` нужно сначала получить доказанную пользу внутреннего Telegram-пилота.
