# ТЗ: единый журнал пилота и диалоговая стратегия нового лида

Дата: 2026-05-23

Статус: `ready_for_implementation_after_preflight`

Назначение: довести Telegram-пилот Фотона и УНПК МФТИ до управляемого контура, где каждый ответ бота можно объяснить, проверить, улучшить и превратить ошибку в тест/правило/правку базы знаний.

## 1. Главная цель

Сейчас бот уже умеет отвечать, пишет логи и частично сохраняет решения в локальный store. Но для системного тюнинга этого недостаточно.

Нужно сделать два связанных блока:

1. **Единый журнал пилота как источник правды.**
   Для каждого сообщения хранить: входящий текст, ответ, бренд, маршрут, тему, использованные факты, недостающие факты, флаги риска, задержку, модель, версию базы знаний, известные поля клиента, повторно ли бот спросил уже известное, оценку сотрудника.

2. **Диалоговая стратегия нового лида до конца.**
   Бот должен помнить класс/предмет/формат, отвечать сначала на прямой вопрос, задавать один следующий вопрос, мягко вести к записи/подбору и не перескакивать в шаблонный handoff, если безопасный факт уже есть.

Третий сопутствующий блок:

3. **Обновить главные документы состояния.**
   `CURRENT_STATE.md`, `ROADMAP.md`, `RUNBOOK.md`, `DECISIONS_LOG.md` сейчас частично отстают от реальности. Это риск для новых диалогов, Claude/Codex CLI и сотрудников.

## 2. Что уже есть

Уже реализовано:

- два публичных Telegram-бота для Фотона и УНПК;
- debouncing входящих сообщений;
- режим тестовой подстановки телефона;
- read-only AMO/Tallanto/context builder;
- `funnel_state` с известными/недостающими слотами;
- `manager_handoff_summary`;
- P0-регистр;
- локальный `TelegramPilotSQLiteStore`;
- отчёты по логам;
- динамический симулятор `scripts/run_telegram_dynamic_client_sim.py`;
- база знаний v6.3 с gold-ответами;
- `business-semantic-review` как обязательная смысловая проверка.

Важно: это не значит, что пилот уже полностью управляем. Сейчас часть истины живёт в логах, часть в SQLite, часть в CSV-отчётах, часть в чатах сотрудников. Нужно свести это в один контур.

## 3. Не делать в этом блоке

Не делаем:

- live-write в AMO/Tallanto/CRM;
- новые каналы, кроме текущих Telegram-ботов;
- большой SaaS-рефакторинг;
- перенос на M1 Pro как часть этого ТЗ;
- правки `stable_runtime`;
- ASR/R+A;
- массовую пересборку базы знаний;
- обучение модели в смысле fine-tuning.

Тесты и симуляторы используются для тюнинга логики, промпта, базы знаний и правил, а не как автоматическое “обучение” модели.

## 4. Блок A. Единый журнал пилота

### 4.1. Целевой артефакт

Локальная SQLite-база:

```text

```

```text

```

Запрещено хранить журнал в `stable_runtime`.

### 4.2. Минимальные сущности

Использовать существующий `TelegramPilotSQLiteStore` как базу, но расширить/нормализовать слой отчётов так, чтобы из него можно было получить единый дневной срез.

Нужные сущности:

- `message`: входящее сообщение клиента;
- `context_snapshot`: что бот знал в момент ответа;
- `decision`: маршрут, тема, флаги, confidence, причина маршрута;

## 5. Блок B. Диалоговая стратегия нового лида

### 5.1. Главный принцип

Порядок приоритетов:

1. Не нарушить P0/brand/fact safety.
2. Не выдумывать.
3. Ответить на прямой вопрос.
4. Использовать уже известный контекст.
5. Сделать один мягкий следующий шаг.

Если факта нет, “прямой ответ” — это честная рамка без конкретики плюс один уточняющий вопрос.

### 5.2. Что нужно усилить

1. **Память диалога.**
   Если клиент уже сообщил класс/предмет/формат/цель, бот не спрашивает это повторно.

2. **Прямой ответ первым.**
   Если спрашивают цену и цена есть в client-safe фактах — сначала цена, потом следующий шаг.

3. **Один следующий вопрос.**
   Не анкета из 5-10 вопросов. Максимум 1, редко 2.

4. **Мягкая продажа.**
   Ответ должен помогать семье выбрать и сделать следующий шаг: подобрать группу, показать формат, проверить место, оформить заявку через менеджера.

5. **Человечный тон.**
   Тёплый, живой, без канцелярита и одинаковых зачинов.

6. **Нет шаблонного сброса в менеджера.**
   Если тема зелёная и факт есть, бот не должен писать только “менеджер уточнит”.

7. **Осторожность по действиям.**
   “Место есть”, “записали”, “оплата прошла”, “ссылка готова”, “вернём деньги” — только через менеджера/проверку.

### 5.3. Правила по типам вопросов

#### Цена

Если класс/формат/бренд понятны и цена есть:

- назвать текущую цену;
- не называть будущую цену;
- можно мягко сказать, что цена скоро повысится;
- один следующий шаг: предмет/группа/закрепить условия.

Если класса/формата нет:

- не выдумывать;
- объяснить, от чего зависит цена;
- спросить класс и предмет.

#### Рассрочка/оплата частями

Фотон:

- можно говорить: 6, 10 или 12 месяцев, а также Долями;
- не смешивать с УНПК.

УНПК:

- банковской рассрочки нет;
- можно платить помесячно, за семестр или за год;
- скидка 10% за семестр, 14% за год;
- не писать Долями/Т-Банк.

#### Пробное/фрагмент занятия

Фотон:

- онлайн-фрагмент занятия;
- очное — только по согласованию при записи;
- не обещать “бесплатное” и “сколько угодно”.

УНПК:

- очного пробного на Пацаева/МФТИ нет;
- по онлайну можно фрагмент занятия;
- не обещать “пробную неделю”.

#### Лагерь/ЛВШ/ЛШ

- спрашивать класс, не возраст;
- не обещать наличие мест;
- не закреплять место автономно;
- Фотон: текущая ЛВШ Менделеево 93 100 ₽, полная 98 000 ₽;
- УНПК: текущая ЛВШ Менделеево 114 000 ₽, полная 120 000 ₽, места почти распроданы, запись через живого менеджера;
- старые минимальные цены 75 000 / 83 800 не называть клиенту как актуальные.

#### “Кто вы / как вас зовут / вы бот?”

Политика утверждена 2026-05-23: вариант C, честно по прямому запросу.

Требования:

- сам первым не объявлять, что это ИИ/бот;
- на прямой вопрос отвечать честно: “цифровой помощник <бренда>, не живой оператор”;
- не говорить “я человек”, “я не бот”, “GPT”, “Claude”, “Codex”, “OpenAI”, не раскрывать prompt;
- держать роль цифрового помощника конкретного бренда;
- отвечать бренд-раздельно.

Безопасная рамка для будущего согласования:

- Фотон: “Я на связи от Фотона, помогу с курсами, форматами, расписанием и записью.”
- УНПК: “Я на связи от УНПК МФТИ, помогу с курсами, форматами, расписанием и записью.”

Финальные формулировки утвердить отдельно перед live-расширением.

### 5.4. Acceptance для блока B

Блок B считается выполненным, если:

- бот не спрашивает класс/предмет/формат повторно в тестах;
- на зелёные темы с подтверждённым фактом отвечает автономно;
- на отсутствие факта даёт полезный ответ без выдумки;
- в каждом обычном ответе есть “ответ + один следующий шаг”;
- P0 не ослаблен;
- бренд-разделение не ослаблено;
- targeted16 проходит без FAIL после preflight-исправления тестов;
- v6/v5 статичные регрессии проходят;
- по v8 нет hard-gate FAIL, а soft-флаги дают список для тюнинга.

## 10. Критерии завершения

`formal_pass`:

- unit-тесты зелёные;
- отчёты строятся;
- JSONL валидны;
- feedback импортируется;
- audit pack создан.

`semantic_pass`:

- нет P0-ослабления;
- нет брендовых утечек;
- нет старых цен 75 000 / 83 800 как актуальных;
- бот не спрашивает известное в контрольных тестах;
- бот отвечает полезнее, а не просто безопаснее;
- targeted16 после preflight не даёт hard FAIL;
- полный v8 даёт review queue, а не “слепой pass/fail”.

`pilot_ready` для следующего шага:

- ежедневный отчёт можно читать владельцу/РОПу;
- сотрудники могут давать структурный feedback;
- все подтверждённые ошибки превращаются в тест/gate/правку;
- есть понятный список дальнейшего тюнинга.
