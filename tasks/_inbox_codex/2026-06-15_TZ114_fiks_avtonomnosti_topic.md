# ТЗ-114 — фикс матрицы автономности в direct-path (рассинхрон topic_id) + захардкоженный requires_manager_approval

- **Дата:** 2026-06-15. Заказчик: Дмитрий. Постановщик: Claude #1. Исполнитель: D1/Кодекс. Основание — регрейд приёмки run1 (локализовано по сырью).

## Зачем
Приёмка показала: при `TELEGRAM_DEAL_ACTION_DECISION=1` бот уходит к менеджеру в ~100% ходов. Причина — матрица автономности в direct-path судит по грубому `topic_id=service:S2_unclear` и демоутит всё. Без фикса нельзя ни включать слой действий, ни честно мерить судью действий (замер будет 100% хендофф).

## Что чинить
1. **Топик для матрицы автономности.** `provider.py:921-922` зовёт `apply_autonomy_matrix_guard` в direct-path; гард (`policy_routing.py:2686-2693`, `_autonomy_topic_allowed` `:4752`) читает `result.topic_id` = `service:S2_unclear`. Чинить: подавать гарду ТОНКИЙ топик из плана намерения (`conversation_intent_plan.topic_id`, тот, что реально в `AUTONOMY_MATRIX_SAFE_TOPIC_IDS`), а не грубый сервисный. Матрица остаётся нижней границей безопасности (что разрешено автономно), но теперь судит по верной теме.
   - Альтернатива (если тонкий топик недоступен в этой точке) — не вызывать autonomy-гард в direct-path вообще: он уже прошёл свой выходной гейт; обосновать в отчёте, что выбрано и почему.
2. **Убрать безусловный `requires_manager_approval=True`** (`post_layers.py:1574` и `_deal_action_unknown` `:1200`): ставить только когда менеджер реально нужен (P0/маршрут manager/опасное действие), а не на каждый ответ, включая одобренный `answer_only`.

## Дисциплина / приёмка
- Это фикс ветки `deal_action`-ON (в проде deal_action OFF → поведение НЕ меняется): **NEG — deal_action OFF = байт-в-байт паритет.**
- Позитив: deal_action ON + ход с тонким топиком из safe-набора (например `theme:001_pricing`, цена/расписание) → бот отвечает САМ (нет `autonomy_default_cautious_topic_not_allowed`); P0/опасное действие по-прежнему → менеджер.
- НЕ ослаблять: P0-пол, бренд-гейт, набор разрешённых тем `AUTONOMY_MATRIX_SAFE_TOPIC_IDS` (его не расширяем — только подаём верный топик).
- Ветка от main; полный pytest зелёный; отчёт в _done; мой регрейд по сырью перед включением deal_action в любой замер с автономией.

## Эффект
После фикса можно честно мерить судью действий (deal_action ON больше не валит всё к менеджеру) и оценивать реальный over-handoff.
