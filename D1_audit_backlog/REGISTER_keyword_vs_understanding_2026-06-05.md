# Реестр: где детерминизм подменяет понимание — 2026-06-05

**Вердикт:** главная ещё-не-чинимая архитектурная проблема = **ВХОДНАЯ сторона того же корня**. ADR/Шаг 4 чинят ВЫХОД (модель отвечает → детерминизм верифицирует выход). Но ПОНИМАНИЕ «что хочет клиент» (intent/topic/scope/risk) на входе всё ещё делает keyword-слой. Это симметричный близнец того, что мы уже лечим, не новый чужой изъян.

## VERIFIED на a0448f33 (worktree consolidation-clean-type)

- **Интент/тема/риск — keyword, без модели.** `build_conversation_intent_plan` (`conversation_intent_plan.py:102`) строит план из `semantic_roles.tag_message_roles` (keyword-теги) + `_risk_signals` (regex, :468). Модельного `understand()` в этой сборке НЕТ. `_primary_intent` (:301) — каскад `if "legal" in risk_signals / "refund" / "complaint"…`. Этот план кормит `answer_plan`, `route_bias`, `next_step` и P0-добавку в `classify_answer_safety`. То есть «о чём вопрос и зачем клиент» решают слова. **Это input-половина корневого изъяна.**
- **Модельный планнер существует, но выключен.** Планнер-на-`understand()` есть (флаг `TELEGRAM_RULES_ENGINE_PLANNER_INTENT`, по памяти проекта default OFF / shadow). Значит ПО УМОЛЧАНИЮ понимание = keyword, модель — в тени.

## Кандидаты (субагент читал СТАРОЕ дерево — координаты пере-grep; часть — гипотезы, не подтверждённый баг)

- **P0-негейшн возврата/жалобы** (`p0_recall_spec`, `semantic_roles.is_negated_refund_topic`): keyword + хрупкая обработка отрицания «не про возврат». НО важная поправка: частично митигируется модельным downgrade — `semantic_non_p0` в `answer_safety_classifier` (143-169, я проверял ранее) может снять СТАРЫЙ не-current P0 при чистом текущем сообщении. То есть НЕ «латч навсегда», как заявил субагент. И P0-флор ОБЯЗАН over-триггерить (безопасность); цена over-триггера — лишние уходы, а не утечка.
- **Разбор составных вопросов** (`rules_engine._looks_like_*_question`): «о чём часть вопроса» по маркерам, без разбора структуры → неполные ответы на многотемные.
- **second-subject-скидка / payment-method / формат онлайн-vs-очно** (`rules_engine`, `semantic_roles`): scope по словам и по «или», не по смыслу (вопрос-сравнение vs выбор vs совет).
- **Наполнение памяти диалога** (`dialogue_memory` QUESTION_KIND_MARKERS): тип последнего сообщения (вопрос/ответ/уточнение) по маркерам, теряет нить на эллипсисе «а очно?». **УЖЕ известно** (memory `dialogue_memory_diagnosis`).

## НЕ анти-паттерн (правильный детерминизм — НЕ трогать)

Наличие числа/цены в выходе; токен чужого бренда в выходе; буквальное «я бот»; утечка плейсхолдера/меты. Это верификация ВЫХОДА — она и должна быть жёсткой.

## Рекомендация (последовательность)

1. Сначала добить ВЫХОД: Шаг 4 (идёт) → Шаги 2-3 (смысловой верификатор выхода) → Шаг 5 (снос старых гардов).
2. Потом симметрично ВХОД: включить планнер-на-`understand()` как источник intent/scope (keyword → в подсказку, не в приговор); конкретику по-прежнему верифицировать детерминированно на выходе.
3. P0-флор оставить детерминированным (безопасность), но сохранить/расширить модельный downgrade ложного P0 (уже есть).

Проверяемость на будущее: тот же приём — grep по `_TOPIC_PATTERNS`/`_has_any`/`risk_signals`/`is_*`/`_looks_like_*`/`*_MARKERS`, и спрашивать «это решает СМЫСЛ или проверяет узкий факт?». Понимание → модель; узкий факт на выходе → детерминизм.
