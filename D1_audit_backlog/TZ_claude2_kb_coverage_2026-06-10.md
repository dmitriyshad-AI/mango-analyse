# ТЗ для Клода-2: полная карта вопросов клиентов + сверка покрытия базой знаний + сбор внешних фактов

Ты — отдельный аналитический контур проекта Mango (Telegram/WhatsApp-бот продаж двух учебных центров). Работаешь в папке `Mango analyse` ТОЛЬКО НА ЧТЕНИЕ; все результаты пишешь в НОВУЮ папку `D1_audit_backlog/kb_intake_20260610/`. Кода не трогаешь, git-команд не выполняешь, в базу знаний напрямую ничего не вносишь — твой выход идёт на ревью архитектору (Claude #1), приёмку Дмитрию и только потом в базу через Кодекса.

## Жёсткие правила (нарушение = брак работы)

1. **Бренды строго раздельны:** Фотон и УНПК МФТИ — разные организации. Каждый факт/вопрос помечай: `foton` / `unpk` / `общий-процесс`. Никогда не смешивай их условия в одном факте.
2. **ПДн:** в выходные файлы НЕ попадают ФИО, телефоны, email, адреса клиентов. Цитаты только обезличенные (<имя>, <тел>, <N>). Маскируй ДО любой обработки моделью, где возможно — скриптами.
3. **Не выдумывай:** каждый внешний факт — с источником (URL) и датой проверки; статус всех новых фактов = «черновик, требует подтверждения Дмитрия». Каждый «вопрос без ответа» — с 1-2 обезличенными примерами формулировок.
4. **Острые темы — не дыры:** реальные претензии на возврат денег, споры по оплате, жалобы, юридические угрозы ВСЕГДА ведёт живой менеджер — отсутствие фактов по ним в базе это политика, а не пробел. Помечай их классом `P0-менеджер`, в дыры не записывай.
5. Субагентов использовать можно и нужно (корпуса большие); их выводы перепроверяй выборочно сам.

## Что сделать (этапы с чек-пойнтами, после каждого — короткая сводка в чат)

**Этап А. Полный каталог реальных вопросов клиентов.** По трём корпусам (WhatsApp, Telegram, звонки) вычлени ВСЕ вопросы клиентов (не только первые), нормализуй и кластеризуй по темам. Выход: `questions_catalog.md` — кластеры с: частота по каждому каналу, 2-3 обезличенные формулировки, бренд (если различим), класс (`факт` / `процесс` / `P0-менеджер` / `совет-подбор`). Метод: скрипты для выделения и маскирования + модель для кластеризации; хвост редких вопросов (<3 вхождений) — отдельным списком, не теряй.

**Этап Б. Сверка покрытия базой.** Для каждого кластера (кроме P0) найди в клиентских фактах базы отвечающие факты ПО СМЫСЛУ (не по совпадению слов): статус `полное` (факт прямо отвечает) / `частичное` (отвечает с оговоркой — напиши какой) / `НЕТ`. Выход: `coverage_report.md` — матрица кластер → статус → fact_key найденных фактов → чего не хватает; сверху таблица топ-дыр, отсортированная по частоте вопроса. Это главный результат всей работы. Помни: «у нас есть ответ» значит «бот может ответить текстом факта», а не «слова похожи».

**Этап В. Внешние факты.** Собери с официальных сайтов и карт проверяемые факты: программы/направления, форматы, адреса и как добраться, часы, контакты, опубликованные цены, акции. Выход: `external_facts_draft.md` — таблица: бренд | тема | формулировка факта | источник URL | дата проверки | расходится ли с базой. **Отдельная секция «РАСХОЖДЕНИЯ сайт ↔ база»** — где сайт говорит одно, а база другое (это золото: либо сайт устарел, либо база) — каждую строку с цитатами обеих сторон.

**Этап Г. Сводка для передачи.** `SUMMARY.md`: объёмы обработанного по каналам; топ-15 дыр покрытия с частотами; топ расхождений сайт↔база; список кандидатов в базу (из В + дыр из Б, для которых ответ известен из корпусов — как менеджеры РЕАЛЬНО отвечают, это лучший источник формулировок: приложи обезличенный ответ менеджера как заготовку client_safe-текста); открытые вопросы Дмитрию списком.

## Критерий готовности

Дмитрий читает SUMMARY за 10 минут и понимает: какая доля реальных вопросов покрыта базой (число), что добавить в первую очередь, где база противоречит сайту. Архитектор по coverage_report может ставить ТЗ Кодексу на пополнение базы без повторного анализа корпусов.
