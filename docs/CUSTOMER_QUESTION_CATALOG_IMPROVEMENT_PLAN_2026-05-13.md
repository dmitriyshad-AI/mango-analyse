# План улучшения единого каталога вопросов клиентов

Дата: 2026-05-13

## Краткий вердикт по качеству предыдущего этапа

Предыдущий этап дал полезную и безопасную основу: вопросы клиентов из звонков, Telegram и почты сведены в единую структуру, появились классы вопросов, черновые шаблоны ответов, требования к актуальным фактам и пакет для проверки РОПом.

Но качество пока не финальное для бота. Это больше рабочий черновой каталог, чем утвержденная база вопрос-ответ. Его можно использовать для анализа и подготовки РОПа, но нельзя напрямую включать в автоматическую отправку клиентам.

## Что получилось хорошо

1. Есть единая модель для трех каналов: звонки, Telegram, почта.
2. Процесс безопасный: ничего не пишет в CRM, Telegram, почту, Tallanto и runtime-базы.
3. Для цен, расписания, адресов, скидок, документов и программ введено требование актуального факта.
4. Появился `rop_question_review_pack.xlsx`, то есть РОП может проверять результат в таблице, а не в JSON.
5. Есть тесты на контракты, извлечение, нормализацию, безопасность и сборку.
6. Дополнительный аудит текстовых выгрузок показал `0` находок по email, телефонам, URL и длинным служебным номерам в публичных полях.

## Главный недочет

`общий вопрос` еще слишком широкий, 876 элементов. Следующий шаг - раздробить его на узкие классы и подготовить файл утвержденных ответов для РОПа.

Это главный блокер, потому что 876 элементов в одном классе нельзя нормально утвердить. Там смешаны разные ситуации: уточнения по звонку, отказы, лагерь, переносы, оплата, жалобы, материалы, логистика, неясные фразы и просто шум из каналов.

## Что показывает быстрая проверка `общего вопроса`

Состав класса:

| Источник | Количество |
|---|---:|
| Telegram | 430 |
| Почта | 402 |
| Звонки | 44 |

Вывод: это в первую очередь проблема детализации текстовых каналов, а не проблема обработки звонков.

Первые группы, которые уже видны по простым признакам:

| Потенциальная подгруппа | Примерный объем |
|---|---:|
| Лагерь, смена, поездка | 95 |
| Уточнение, перезвон, контакт, кому/куда писать | 94 |
| Оплата, возврат, чек, сумма | 46 |
| Партнеры, юридические/B2B-вопросы | 29 |
| Перенос, пропуск, отработка, письмо не пришло | 18 |
| Качество обучения, жалоба, обратная связь | 13 |
| Отказ, неактуально, нет интереса | 10 |
| Программа, материалы, задачи | 10 |
| Транспорт и логистика | 5 |
| Неразобранный остаток | 611 |

Эти цифры приблизительные: один вопрос может попадать в несколько признаков, а часть вопросов требует более точной разметки.

## Риски текущего состояния

1. РОПу сложно проверять слишком широкий класс. Один статус и один шаблон не подходят для 876 разных ситуаций.
2. Часть текстов из почты может быть не вопросом клиента, а служебным письмом, пересылкой, автоуведомлением или фрагментом рассылки.
3. Некоторые вопросы требуют не ответа бота, а передачи менеджеру: жалобы, документы, пропуски, возвраты, сложные личные ситуации.
4. Текущие источники фактов найдены как кандидаты, но еще не утверждены как свежие.
5. Пока нет финального файла `approved_question_answers`, который можно безопасно подключать к предпросмотру ответа бота.

## Цель следующего улучшения

Сделать каталог пригодным для реальной проверки РОПом:

- уменьшить класс `общий вопрос` минимум в 3-5 раз;
- создать новые узкие классы для повторяющихся ситуаций;
- выделить шум и не-клиентские тексты в отдельный статус;
- подготовить файл утверждения ответов РОПом;
- не дать будущему боту отвечать там, где нужен менеджер или актуальный факт.

## Этап 1. Дробление `общего вопроса`

Что сделать:

1. Добавить новые намерения и классы:
   - уточнение причины звонка;
   - просьба перезвонить или написать;
   - отказ или неактуально;
   - лагерь/смена/поездка;
   - перенос, пропуск, отработка;
   - письмо, ссылка или доступ не пришли;
   - жалоба или обратная связь по качеству;
   - материалы урока, домашние задания, задачи;
   - оплата, возврат, чек;
   - юридические и партнерские вопросы;
   - транспорт и логистика;
   - шум/служебное сообщение.
2. Добавить тесты на каждую новую группу.
3. Пересобрать каталог.
4. Проверить, что класс `общий вопрос` сократился минимум до 250-300 элементов.

Критерий готовности:

- `общий вопрос` больше не является крупнейшим классом;
- новые классы имеют понятные названия;
- шум не попадает в рабочие ответы РОПа.

## Этап 2. Черный список шума из почты и Telegram

Что сделать:

1. Расширить фильтры автоуведомлений, рассылок, банковских реквизитов, чеков, технических трекеров и пересылок.
2. Добавить отдельный статус `not_customer_question` или аналогичный внутренний флаг.
3. Не удалять такие строки из исходников, но исключать их из `rop_question_review_pack.xlsx`.
4. Добавить отчет: сколько строк отброшено как шум и по каким причинам.

Критерий готовности:

- в таблице РОПа не должно быть очевидных служебных писем;
- шум должен быть объяснимым и проверяемым через счетчики.

## Этап 3. Файл утвержденных ответов РОПом

Что сделать:

1. Создать отдельный выходной файл `approved_question_answers_draft.xlsx`.
2. В нем должны быть поля:
   - класс вопроса;
   - частотность;
   - примеры вопросов;
   - черновой шаблон ответа;
   - можно отвечать боту / только менеджеру / нужен актуальный факт / запрещено;
   - какой факт нужен;
   - источник факта;
   - комментарий РОПа;
   - финальный утвержденный текст.
3. Добавить машинный CSV/JSON-вариант того же файла для будущего подключения к боту.
4. Никакие ответы не считать утвержденными автоматически.

Критерий готовности:

- РОП получает понятный файл для проверки;
- будущий бот может читать только явно утвержденные строки;
- все строки без утверждения остаются в режиме черновика.

## Этап 4. Связка с актуальными файлами цен и расписания

Что сделать:

1. Описать структуру будущих файлов фактов:
   - цены;
   - расписание;
   - адреса;
   - скидки;
   - рассрочка;
   - документы;
   - программы.
2. Добавить проверку свежести: дата файла, владелец, статус утверждения.
3. Подготовить контракт подстановки: шаблон ответа + актуальный факт.
4. Не подставлять факт, если источник не утвержден.

Критерий готовности:

- вопрос “Стоимость подготовки к ЕГЭ по профильной математике для 11 класса” получает не старую цену из переписки, а шаблон и требование взять актуальную цену из утвержденного файла.

## Этап 5. Приоритеты для РОПа

Что сделать:

1. Отсортировать классы по пользе:
   - частые вопросы;
   - вопросы, где бот может реально разгрузить менеджеров;
   - вопросы с риском ошибки;
   - вопросы только для менеджера.
2. Сделать короткий файл `rop_review_priority_top100.xlsx`.
3. Начать с топ-100 классов, а не пытаться утверждать все 809 сразу.

Критерий готовности:

- РОП видит первые 100 классов, которые дадут максимальную пользу;
- есть понятный порядок проверки.

## Этап 6. Контроль качества ответов

Что сделать:

1. Для каждого утвержденного класса сгенерировать 3-5 тестовых клиентских формулировок.
2. Получить ответ по шаблону.
3. Проверить:
   - не выдумана ли цена;
   - не выдумано ли расписание;
   - не отвечает ли бот там, где нужен менеджер;
   - нет ли персональных данных;
   - звучит ли ответ естественно и по-человечески.
4. Сохранить результаты в аудит-файл.

Критерий готовности:

- перед показом людям есть автоматический прогон, который ловит грубые ошибки.

## Этап 7. Подключение к предпросмотру ответа бота

Что сделать:

1. Подключать только утвержденные классы.
2. Для неутвержденных классов показывать менеджеру черновик или рекомендацию, но не отправлять клиенту.
3. Для классов с актуальными фактами показывать: “нужен факт из такого-то файла”.
4. Логировать, какой класс вопроса был выбран и почему.

Критерий готовности:

- бот не отправляет ответы сам;
- менеджер видит полезный предпросмотр;
- все рискованные ответы заблокированы до утверждения.

## Приоритет выполнения

Ближайший проход должен закрыть три вещи:

1. Дробление `общего вопроса`.
2. Создание `approved_question_answers_draft.xlsx`.
3. Создание `rop_review_priority_top100.xlsx`.

После этого можно переходить к связке с актуальными файлами цен и расписания.

## Definition of done следующего прохода

Следующий проход считается завершенным, если:

1. `общий вопрос` сокращен минимум до 250-300 элементов.
2. Появились новые узкие классы для основных подгрупп.
3. Шум из почты и Telegram не попадает в таблицу РОПа.
4. Создан файл `approved_question_answers_draft.xlsx`.
5. Создан файл `rop_review_priority_top100.xlsx`.
6. Добавлены тесты на новые правила классификации и фильтрации.
7. Каталог пересобран без записи во внешние системы.
8. Проверка публичных текстов снова дает `0` находок по контактам и служебным номерам.
9. Документация и аудит обновлены.
