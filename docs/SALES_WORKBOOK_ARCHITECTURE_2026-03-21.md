# Sales Workbook Architecture

Дата фиксации: 21.03.2026

## 1. Жесткое решение по хранению

Нельзя делать единственное хранилище в виде одной "огромной таблицы" и считать, что этого достаточно.

Правильная архитектура для проекта:

1. `call_records` в БД
   Один звонок = одна запись.
   Это source of truth: файл, метаданные, транскрипт, статусы pipeline, итог анализа.

2. Витрина `Calls`
   Один звонок = одна строка Excel.
   Эта таблица нужна руководителю, РОПу, аналитикам и для контроля качества.

3. Витрина `Contacts`
   Один телефон/контакт = одна строка Excel.
   Это рабочая таблица для продаж и повторных касаний.

Такой подход сохраняет историю всех звонков и одновременно дает менеджерам удобную рабочую поверхность.

## 2. Как должна заполняться таблица

Смысловые поля в таблицах не могут заполняться только транскриптом ASR.
Для их заполнения нужен этап `Analyze`.

Рабочий pipeline:

1. `Ingest`
   Индексация файлов Mango.

2. `Transcribe`
   Распознавание звонка ASR-системами.

3. `Resolve`
   Улучшение и контроль качества финального текста.
   Этот этап нужен для качества транскрипта, но не обязан обязательно вызывать LLM для каждого звонка.

4. `Analyze`
   Именно этот этап формирует бизнес-структуру:
   - `history_summary`
   - `structured_fields`
   - `lead_priority`
   - `follow_up_score`
   - `follow_up_reason`
   - рекомендуемый следующий шаг

5. `Export`
   На этом этапе создается Excel-книга:
   - лист `Calls`
   - лист `Contacts`

### Ключевой вывод

Да, чтобы получить осмысленную рабочую таблицу, каждый **содержательный распознанный звонок** должен пройти этап `Analyze`.

На текущем этапе проекта это лучше всего делать через:

- `Analyze provider = codex_cli`
- `Codex model = gpt-5.4`

LLM здесь нужна не для самого ASR, а для извлечения фактов и составления CRM-конспекта.

## 3. Правила для `history_summary`

`history_summary` - это не расшифровка и не пересказ всего диалога.
Это короткий CRM-конспект, который человек читает за 5-10 секунд.

Обязательные правила:

1. Только факты из звонка и метаданных.
2. Никаких фантазий и додумываний.
3. 3-6 предложений.
4. Первое предложение всегда содержит:
   - дату/время звонка
   - имя менеджера, если известно
5. Далее обязательно отражаются:
   - предмет обращения
   - интерес клиента
   - ограничения / возражения
   - точная договоренность / следующий шаг
6. Нельзя вставлять длинные цитаты и куски диалога.
7. Если содержательного разговора не было, это должно быть явно написано.

### Целевой шаблон

1. Когда и кто общался.
2. Что хотел клиент.
3. Что удалось уточнить: ребенок, класс, формат, предметы, продукт.
4. Какие были ограничения или возражения.
5. О чем договорились и когда возвращаться.

## 5. Логика `recommended_product`

Строгое правило:

`recommended_product` можно считать надежным только если в системе есть фиксированный каталог продуктов.

Без каталога допустим только временный режим:

- брать `target_product` из анализа
- либо первый продукт из `interests.products`

Но это не полноценный подбор "курса из наших".

### Следующее улучшение

Нужно добавить каталог ваших продуктов:

- название продукта
- возраст / класс
- предметы
- формат
- сезонность
- признак лагерь / интенсив / годовая программа

После этого Codex CLI должен выбирать **только из списка каталога**, а не генерировать продукт свободно.

## 6. Логика `sale_probability_pct`

Это не "математически точная вероятность покупки", а управленческая оценка приоритета.

Правило:

- использовать `follow_up_score` из анализа как первичный сигнал
- хранить рядом `sale_probability_reason`

Шкала:

- `80-100` - горячий лид
- `55-79` - теплый лид
- `25-54` - холодный лид
- `0-24` - нецелевой или без содержательного диалога

## 7. Логика `recommended_followup_date`

`recommended_followup_date` нельзя полностью отдавать на свободное усмотрение LLM.
Нужна rule-based логика поверх результатов анализа.

Приоритет правил:

1. Если в звонке есть явная дата следующего контакта и она распознана:
   использовать ее.

2. Если договорились отправить материалы:
   follow-up через 2 дня.

3. Если договорились перезвонить:
   follow-up через 1-3 дня, в зависимости от теплоты лида.

4. Если лид горячий:
   следующий день.

5. Если лид теплый:
   через 3 дня.

6. Если лид холодный:
   через 14 дней.

7. Если звонок нецелевой:
   автоматически follow-up не ставить.

## 8. Как использовать Excel-книгу

### `Calls`

Использовать для:

- контроля качества звонков
- разбора менеджеров
- точечного просмотра истории
- поиска удачных / провальных разговоров

### `Contacts`

Использовать для:

- списка повторных касаний
- сегментации клиентов
- выгрузки в CRM
- формирования персональных офферов

## 9. Что считать правильным следующим шагом разработки

1. Догнать массовый `Analyze` через `codex_cli + gpt-5.4`
2. Строить `xlsx`-витрины `Calls` и `Contacts`
3. В отдельном шаге добавить каталог ваших курсов
4. Затем улучшить `recommended_product` до выбора строго из каталога
5. Потом уже интегрировать это с amoCRM
