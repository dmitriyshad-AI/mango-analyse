# IDEAS_agent_3_amo.md — 10 идей: интеграция бота-продавца Mango в AMO CRM

Дата: 2026-06-03
Угол: техническая интеграция бота в AMO CRM через мессенджеры (Telegram + МАКС)

---

## Идея 1. Вебхук AMO → очередь бота (входящий трафик)

**Суть.** AMO отправляет вебхук на эндпоинт бота при каждом новом входящем сообщении (событие `incoming_lead` / `incoming_message` из интеграции с Telegram или МАКС). Бот вычитывает сообщение, formulates черновик и кладёт его в очередь (Redis/DB). Менеджер видит кнопку «Черновик бота» в виджете AMO.

**Почему полезно.** Вся переписка остаётся в AMO — менеджер не переключается между вкладками, история под рукой, ничего не нужно пересылать.

**Сложность:** средняя — AMO webhooks хорошо задокументированы, но нужен стабильный публичный эндпоинт бота и обработка retry/дублей.

**Риск/зависимость:** AMO не гарантирует порядок доставки вебхуков при высокой нагрузке; нужен идемпотентный прием по `message_id`.

---

## Идея 2. Виджет «Черновик бота» в карточке сделки AMO

**Суть.** Кастомный виджет (iframe/JS через AMO Widgets SDK) встраивается в правую колонку карточки сделки. При новом сообщении виджет показывает черновик бота с кнопками «Отправить», «Редактировать», «Передать менеджеру». При нажатии «Отправить» — POST в AMO API `/api/v4/talks/{id}/messages` от имени учётки компании.

**Почему полезно.** Менеджер одобряет черновик одним кликом прямо в карточке, не копирует текст руками. Черновик визуально отделён от истории переписки.

**Сложность:** высокая — AMO Widgets SDK капризный, требует отдельной публикации виджета и OAuth-авторизации.

**Риск/зависимость:** AMO может поменять SDK без предупреждения; виджет нужно поддерживать отдельно от основного кода бота.

---

## Идея 3. Определение active_brand по воронке/источнику сделки

**Суть.** В AMO каждая сделка лежит в конкретной воронке (pipeline). Воронки «Фотон» и «УНПК» создаются отдельно и привязываются к разным Telegram-ботам/номерам МАКС. Бот читает `pipeline_id` из вебхука и однозначно выставляет `active_brand` — никакого угадывания по тексту.

**Почему полезно.** Устраняет класс ошибок бренд-смешения на корню: source of truth — структура AMO, а не контент сообщения.

**Сложность:** низкая — `pipeline_id` приходит в каждом вебхуке AMO, достаточно завести словарь `{pipeline_id: brand}` в конфиге бота.

**Риск/зависимость:** если менеджер перенесёт сделку в чужую воронку руками — бот получит неверный бренд; нужна валидация + алерт.

## Идея 4. Read-only чтение полей сделки/контакта для контекста

**Суть.** Перед сборкой черновика бот делает GET `/api/v4/contacts/{id}` и `/api/v4/leads/{id}` — достаёт поля: имя, класс ребёнка, предмет интереса, стадия сделки, теги, история касаний. Эти поля подаются в планнер как контекст — бот знает, что клиент уже спрашивал.

**Почему полезно.** Черновик персонализирован («Михаил, мы уже обсуждали физику для 9-го класса…»), менеджеру не нужно повторять контекст.

**Сложность:** низкая — стандартный AMO REST API, read-only токен достаточно.

**Риск/зависимость:** поля в AMO заполняются руками менеджеров — если данных нет, бот должен деградировать gracefully, не выдумывать класс или предмет.

---

## Идея 5. Тег «bot_draft_waiting» как очередь одобрения

**Суть.** Когда бот готовит черновик, он через AMO API добавляет тег `bot_draft_waiting` к сделке. Менеджер видит фильтр «Ждут одобрения» в списке сделок. После отправки или отклонения тег снимается и ставится `bot_sent` / `bot_declined`. Все статусы логируются в примечании к сделке.

**Почему полезно.** Не нужен отдельный интерфейс очереди — AMO сам становится дашбордом одобрения. Руководитель видит статистику тегов в аналитике AMO.

**Сложность:** низкая — тег ставится одним PATCH-запросом к AMO API.

**Риск/зависимость:** теги в AMO не иерархические и не типизированные — нужна договорённость в команде не использовать похожие названия тегов вручную.

## Идея 6. Переключатель «бот включён/выключен» на уровне сделки

**Суть.** Кастомное поле сделки `bot_mode` с вариантами: `draft` / `autonomous` / `off`. Менеджер переключает одним кликом в карточке. Бот при получении вебхука читает поле и выбирает режим: draft (показать черновик), autonomous (отправить самостоятельно по safe-темам), off (не отвечать). P0-события всегда переводят в off + уведомляют менеджера.

**Почему полезно.** Менеджер полностью контролирует автономию бота на уровне конкретного клиента, не глобально — безопаснее при пилоте.

**Сложность:** средняя — кастомные поля AMO через API создаются просто, но нужна синхронизация состояния между AMO и внутренней БД бота.

**Риск/зависимость:** если AMO-вебхук задержался, бот может прочитать устаревший режим; нужен TTL кеша или прямой GET перед каждым ответом.

## Идея 7. Исходящие сообщения от имени номера компании через AMO

**Суть.** AMO API `/api/v4/talks/{talk_id}/messages` позволяет отправить сообщение от имени подключённого мессенджера (Telegram компании или МАКС-номер). Бот вызывает этот эндпоинт после одобрения черновика менеджером — клиент видит ответ с номера/аккаунта учебного центра, не с персонального.

**Почему полезно.** Клиент получает ответ с официального номера бренда. При переключении менеджера история сохраняется в одном диалоге AMO.

**Сложность:** средняя — требует OAuth2 токен с правами на talks; важно не перепутать talk_id разных мессенджеров.

**Риск/зависимость:** AMO ограничивает rate-limit отправки; при исходящей инициативе (не ответ, а первый контакт) нужно убедиться, что Telegram-канал не заблокирует сообщение как спам.

## Идея 8. Salesbot AMO как тонкий прокси к боту Mango

**Суть.** Встроенный Salesbot AMO (визуальный конструктор) используется только как триггер и прокси: при определённом событии (новая сделка, входящее сообщение) он вызывает webhook на бота Mango с нужными параметрами. Salesbot не содержит логики ответов — вся логика в Mango-боте. Это позволяет запустить интеграцию быстро, без написания собственного слоя триггеров.

**Почему полезно.** Salesbot уже знает контекст AMO, события, воронки. Не нужно переписывать систему триггеров — используем готовый инструмент AMO как «клей».

**Сложность:** низкая — настраивается в интерфейсе AMO без кода; передаёт `{{lead.id}}`, `{{contact.id}}`, `{{pipeline.id}}` через URL-параметры.

**Риск/зависимость:** Salesbot AMO ограничен по логике ветвления; если нужна сложная маршрутизация — лучше собственный вебхук-обработчик (Идея 1).

## Идея 9. Автоматическое примечание-лог в сделке AMO

**Суть.** После каждого цикла (входящее → черновик → решение менеджера) бот добавляет примечание к сделке: что пришло, какой черновик был предложен, что отправлено / отклонено / передано менеджеру, какой `intent` определил планнер, какие факты использовались. Примечание добавляется через POST `/api/v4/leads/{id}/notes`.

**Почему полезно.** Полный аудит-трейл прямо в AMO — руководитель видит качество черновиков, не нужен внешний дашборд на первом этапе пилота.

**Сложность:** низкая — notes API простой; лог можно форматировать как свёрнутый JSON или читаемый текст.

**Риск/зависимость:** при высоком трафике примечания засоряют карточку; нужна настройка «только менеджер видит» (private note) или периодическая агрегация.

## Идея 10. Сравнение путей: «через AMO» vs «отдельный ТГ-бот»

**Суть.** Не идея внедрения, а архитектурный выбор, который нужно зафиксировать явно:

| Критерий | Через AMO | Отдельный ТГ-бот |
|---|---|---|
| Отправка с номера компании | Да — через AMO talks API | Нет — только с бот-аккаунта |
| Исходящий первый контакт | Да — если клиент писал ранее | Нет — бот не может писать первым |
| История в AMO | Да — всё в одном месте | Нет — нужна ручная синхронизация |
| Сложность запуска | Высокая — OAuth, webhooks, виджет | Низкая — один Telegram-бот |
| Бренд-изоляция | Через pipeline_id — надёжно | Через отдельные bot token — надёжно |
| Зависимость от AMO | Полная — падение AMO = нет бота | Нет — работает независимо |
| Черновик в интерфейсе AMO | Да — виджет в карточке | Нет — нужен отдельный UI |

**Рекомендация.** Начать пилот с «отдельного ТГ-бота» (быстро, без рисков AMO), параллельно строить AMO-интеграцию как следующий этап. Вебхук-прокси (Идея 8) — минимальный первый шаг к AMO-пути без переписывания.

**Сложность:** зависит от выбранного пути.

**Риск/зависимость:** фиксация выбора нужна до начала разработки интеграционного слоя — потом менять дорого.
