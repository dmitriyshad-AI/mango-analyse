# Промт серверному Кодексу (api.fotonai.online) — Шаг 0 ТЗ-14 (выдаёт Дмитрий)

Ты — серверный Кодекс на api.fotonai.online. Задача одна: Шаг 0 ТЗ-14 — дать MCP-коннектору foton-crm-readonly собственный надёжный способ жить с токенами AMO. Сейчас токен суточный и слетает, когда основная интеграция (agent_runtime/бот) делает refresh, — это блокер всего AMO-контура трека.

## Что сделать

1. Разведка: как сейчас хранятся и обновляются токены AMO у основной интеграции и у коннектора; где именно конфликт (общий refresh-токен инвалидируется при чужом refresh).
2. Выбери и реализуй один из двух вариантов (критерий выбора — ноль влияния на основную интеграцию бота, минимум движущихся частей):
   - отдельная интеграция AMO для коннектора (свой client_id/secret/refresh, независимый цикл обновления), ИЛИ
   - общее хранилище токенов с основным процессом: один источник правды, refresh делает ровно одно место, остальные только читают.
3. Автообновление: коннектор переживает истечение access-токена без ручного вмешательства.
4. Заодно проверь WAF: запросы по токену проходят независимо от User-Agent (сегодня резал Python-urllib — выяснено пробой).

## Проверки (обязательно)

- Смоук: amo_api_get users через коннектор — успешный ответ.
- NEG (главная): выполни refresh основной интеграции → сразу параллельный вызов коннектора → токен коннектора ЖИВ, оба работают. Повторить дважды.
- Основная интеграция после твоих правок работает без изменений (бот/agent_runtime не задет).

## Границы

- Только чтение AMO (коннектор read-only, пишущих методов не добавлять).
- Секреты (client_secret, refresh-токены) — не в git, не в отчёт, не в логи; только в принятое на сервере хранилище секретов.
- Хирургические правки, ничего лишнего не рефакторить.
