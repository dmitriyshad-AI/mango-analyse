# Аудит детерминированных проверок смысла — кандидаты на LLM-слой. 2026-05-31.

Автор: Клод 1 (на основе read-only обзора + субагент-картирование). ВАЖНАЯ ОГОВОРКА: субагент
картировал ~100 проверок, но часть «рисков» — гипотетические («может спутать»), не доказанные на
реальных диалогах. Поштучно верифицированы НЕ все. Документ — карта направлений, не приговор каждой
проверке. Перед правкой конкретной проверки — подтвердить вред на сырых диалогах (как делали с ЕГЭ→11).

## Суть проблемы (подтверждена на практике)

Бот понимает смысл клиента детерминированными regex/keyword-проверками (per-message, без контекста
диалога). Они хрупки: ловят по подстроке/слову, не по смыслу. Доказанные на сырье случаи:
- `extract_grade`: «ЕГЭ» → grade=11, перебивает явное «десятый» (поймал я на прогоне памяти).
- draft-промпт: «Физтех» не распознан как «олимпиада по физике» → over-handoff (поймал на v2-нити).
Это та же корневая болезнь, что в [[bot_architecture_root_flaw]]: понимание делает тупой слой.

## Карта (категории, ~100 проверок — детали в субагент-отчёте)

- **P0/риск-детекторы** (p0_recall_spec.py): REFUND_RE, LEGAL_RE, COMPLAINT_RE, PAYMENT_DISPUTE_RE,
  REPUTATION_RE, has_complaint_signal — сложные regex с отрицаниями. Риск ложных P0 («возврат к теме»
  vs «возврат денег», «суд» в «обсудим»).
- **Извлечение слотов** (new_lead_funnel.py): extract_grade (ЕГЭ→11!), extract_subjects (физик→
  физиотерапия?), extract_format, extract_goal, extract_product, extract_camp_direction — IF-цепи,
  первое совпадение побеждает, отрицания обрабатываются слабо.
- **Детекторы намерения** (conversation_intent_plan.py, fact_claim_audit.py): _asks_money_price,
  _asks_class_schedule_days, _asks_address, _asks_refund_policy, _asks_live_availability,
  _asks_training_format_choice — regex по фразам. Риск: «суббота я свободна» → вопрос о днях занятий.
- **Семантические роли** (semantic_roles.py): _enrollment_vs_recording («записать» омоним),
  _transfer_sense, _refund_frame (presale vs dispute) — соседство слов в окне.
- **Матчинг фактов**: _matched_fact_keys, claim_anchors, _SCOPE_MARKERS, _keys_for_topic — по ключам/
  алиасам.
- **verify_output чеки**: маркеры дней, бренд-токены, наши 3 чека (preemptive_format и т.д.) — regex.

## ТОП реальных кандидатов (где вред правдоподобен)

1. **extract_grade «ЕГЭ»→11** — ДОКАЗАН (перебивает явный класс). Самый чёткий кандидат.
2. **draft-промпт «соседний факт»/«сомневаешься»** — ДОКАЗАН (over-handoff на v2). Лечится калибровкой
   промпта (отдельное ТЗ TZ_draft_prompt_calibration) — это уже в работе.
3. **_refund_frame presale vs dispute** — высокий риск (P0-граница: гипотетический возврат vs реальный
   спор). Ошибка дорого стоит (P0). Стоит проверить на сырье.
4. **_enrollment_vs_recording** — омоним «запись» (записаться vs смотреть запись). Проверить частоту.
5. **P0-regex (REFUND/LEGAL/COMPLAINT)** — ложные P0 уводят к менеджеру зря (бьёт по автономии) ИЛИ
   пропускают мягкие формулировки (бьёт по безопасности). Двусторонний риск.

## Рекомендация (НЕ заменять всё разом)

- Точечно: где вред ДОКАЗАН на сырье (ЕГЭ→11, draft-промпт) — чинить сразу.
- LLM-слой понимания как FALLBACK, не замена: дешёвая модель (haiku-класс, low effort) включается
  только когда детерминированный слой НЕ уверен или сигналы конфликтуют (несколько тем сразу,
  отрицание, омоним). Это дёшево (вызов редкий) и согласуется с формулой «модель ПОНИМАЕТ, слой
  ВЕРИФИЦИРУЕТ». Уже частично есть: понимание (build_understanding_prompt) — модельное; беда в том,
  что МНОГО решений принимается ДО/ПОСЛЕ него детерминированно.
- НЕ трогать P0-regex без сверхосторожности: ослабление = риск безопасности. Для P0 модель может
  только ДОБАВЛЯТЬ обнаружение (поймать мягкую жалобу), но не отменять regex-срабатывание.
- Приоритет верификации: прогнать корпус реальных вопросов (WhatsApp/Telegram-экспорт есть) через
  детерминированные детекторы и сравнить с модельным пониманием — измерить РЕАЛЬНУЮ долю расхождений,
  а не гипотетическую. Только потом решать, какие проверки заменять.
