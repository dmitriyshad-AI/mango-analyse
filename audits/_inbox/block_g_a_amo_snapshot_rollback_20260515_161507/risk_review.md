# Risk Review

## Основные риски

1. Реальная AMO-интеграция не проверялась на sandbox-аккаунте, потому что sandbox нет.
   - Компенсация: fake-тесты, dry-run, snapshot, rollback, readback, первый микропилот только 1-5 сделок.

2. Очистка пустых старых значений не автоматизирована.
   - Причина: существующий AMO helper пропускает пустые строки.
   - Решение: `manual_restore_required`, без риска случайно очистить поле неправильным способом.

3. `scripts/write_deal_aware_amo_fields.py` до этого был untracked в рабочем дереве.
   - Решение: изменения ограничены зоной Блока A; unrelated направления не трогались.

4. В рабочем дереве много параллельной разработки.
   - Решение: создана карта границ, коммит должен брать только файлы G/A.

5. Широкий запуск `tests/test_deal_aware_*.py` нашел один красный тест распределения confidence на локальных Phase1 артефактах.
   - Это вне Блока A.
   - Целевые snapshot/rollback тесты прошли.
   - Не исправлялось, чтобы не расширять scope текущего блока.

## Риск live-записи

Live-запись в AMO не запускалась. Текущий результат только готовит безопасный контур для будущего микропилота.

Перед первым live-микропилотом нужно отдельно подтвердить:

- входной CSV;
- operator approval;
- лимит 1-5 сделок;
- наличие snapshot/rollback/readback плана;
- ручное наблюдение за результатом.
