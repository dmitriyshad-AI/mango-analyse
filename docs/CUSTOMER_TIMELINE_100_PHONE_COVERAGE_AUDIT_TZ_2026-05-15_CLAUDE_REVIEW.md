# Смысловое ревью ТЗ «100 телефонов / customer_timeline»

## Общая оценка
План связный, границы безопасности соблюдены (read-only, без AMO/Tallanto/ASR/R+A/stable_runtime). Стратификация и набор выходных артефактов адекватны. Готов идти в работу после правок ниже.

## Риски выборки

1. **Селекционное смещение базы 709.** Это уже отфильтрованный набор «кандидаты для AMO-проверки», то есть скорее «проблемные» телефоны. Выводы о покрытии нельзя экстраполировать на всю клиентскую базу — это нужно явно зафиксировать в `coverage_summary.md` как ограничение валидности.
2. **Нет квот по стратам.** 10 групп перечислены, но не задан минимум на каждую (например, ≥5). При натуральном распределении 100 точек могут полностью промазать мимо мелких, но важных страт («сервисные/рискованные», «несколько возможных клиентов»). Добавить целевые мин/макс квоты и фиксировать фактическое распределение.
3. **Нормализация телефонов.** Перед сравнением выборки с `customer_timeline` обязательна проверка единого формата (E.164/без +7/8 и т.п.). Иначе «не найдено» будет ложным. Это отдельный шаг до измерения покрытия.
4. **Правило выбора сделки AMO для телефона не задано.** Если на телефон несколько сделок — какая попадает в строку аудита? Без правила метрики будут невоспроизводимы.

## Чего не хватает в метриках

- **Свежесть истории по источникам** — `max(event_ts)` и возраст последнего события на источник (call/AMO/Tallanto). Старая история опасна для подсказок менеджеру.
- **Кросс-источниковая согласованность** — расхождения AMO/Tallanto (имя клиента, статус, ответственный). Без этого «ready_for_preview» нечем подтвердить.
- **Хронологическая корректность** — особенно важна после правки single-call post-backfill chronology (commit e49b81b93). Проверить, что события идут в правильном порядке и нет «дублей с разными ts».
- **Качество, а не только наличие** — флаг «нашлось N событий, но все — пустые/сжатые/без транскрипта» отдельно от «нашлось N событий».
- **Знаменатель покрытия** — фиксировать явно: % от выборки (100), % от 709, % от уникальных телефонов. Сейчас «процент покрытия» неоднозначен.

## Критерии приемки — слабое место

Решение «можно/нельзя включать `timeline_preview_enabled` / `timeline_primary_read_enabled`» завязано на вердикты, но **пороги не заданы числом**. Нужно зафиксировать заранее, например:

- preview ON: ≥80% выборки в состоянии `ready_for_preview`, 0 случаев конфликта AMO/Tallanto без флага;
- primary read ON: ≥95% + покрытие по всем стратам + 0 случаев нарушения хронологии.

Иначе вердикт получится субъективным и его легко оспорить на следующем этапе.

## Сценарий «timeline DB не найдена»

Хорошо, что отдельно прописан — но добавить: в этом случае всё равно собрать «теневой» отчет по источникам без timeline (звонки + AMO/Tallanto-связки), чтобы этап не вернул пустоту и было видно, какие телефоны *стали бы* покрытыми после импорта.

## Связь со Stage15 / over-sanitization

В CLAUDE.md фиксировано: автономный бот заблокирован до ревью over-sanitization queue. Если в строке аудита учитывается «есть ли bot-context», стоит явно отметить, что положительный результат по timeline **не снимает** этот блок — добавить дисклеймер в `next_steps.md`, чтобы не было соблазна интерпретировать «timeline готов» как «бот готов».

## Claude-аудит

Вопросы корректные. Я бы добавил пятый: **«какие false-positive в покрытии возможны»** (телефон найден, но события мусорные/чужие) — иначе Claude по умолчанию проверит только полноту, не качество.

## Резюме

ТЗ принять с правками: (1) явные квоты по стратам, (2) нормализация телефонов как pre-step, (3) правило выбора сделки на телефон, (4) метрики свежести/согласованности/хронологии/качества, (5) числовые пороги для вердиктов, (6) дисклеймер про over-sanitization. После этих правок этап безопасен и его результат будет основанием для решения о переводе чтения на `customer_timeline`.
