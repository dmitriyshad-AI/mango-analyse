# ТЗ блок A-2: поднять качество черновика одним батчем. 2026-05-25

От Claude (read-only ревью). Цель — перестать двигаться по одной правке за прогон и жечь токены.
Делаем ОДИН батч: всё безопасное сразу + РОВНО один рискованный рычаг за отдельным флагом.
База для сравнения уже снята: round-5 медиана — походовая отправляемость 39.7%, дильно 5/24, тон ~66, recall 7/8,
FAIL=0, hard=0 (см. REGRADE_block_a_final_2026-05-25.md). Новый прогон сравнивать прямо с этим.

## Принцип
- Безопасные правки (ниже S1-S3) — аддитивно, можно под общим block-A флагом; они почти не могут ухудшить.
- Рискованный рычаг (R1) — за СВОИМ флагом, чтобы откатить его одного, не теряя S1-S3.
- Второй рискованный рычаг в этот заход НЕ берём (X1 — отложен).

## S1-S3: безопасный батч (низкий риск регресса)
- **S1. Убрать протечку служебных фраз в клиентский текст.** Расширить запрет в промпте/пост-фильтре черновика
  на «без служебных пометок», «автономный ответ не требуется», «безопасный вариант» и подобное. Судья поймал
  «без служебных пометок» в perevod_manager_16. Чисто аддитивно — только вычищает плохие фразы.
- **S2. Усилить инструкцию формы ответа в промпте:** сначала буквальный ответ на заданный вопрос, потом 1-2 пояснения
  и один шаг; не начинать соседние ходы одинаково. Это бьёт ignored_question/templated_opening на уровне инструкции.
- **S3. Backstop фильтра (блок D):** `sanitize_answer`/`has_any_safety_risk` вычищают и флагуют `fact_id:`/`fact:v3:`/
  `source_id=`/`trace_id`/сырой JSON. Отдельная область (insights/sanitizers), на метрику качества не влияет, дыру закрывает.
  Можно в тот же коммит — атрибуцию не портит.

## R1: единственный рискованный рычаг (свой флаг, дефолт ВКЛ на тест)
- **Подключить `gold_answers_v3`/`gold_answer_context` в живой контекст черновика** через telegram_pilot_context_builder
  (сейчас промпт их читает, но НИКТО не заполняет — грузится только облегчённый few-shot). Подбор по теме и бренду,
  как уже сделано для few-shot. Поднять лимит примеров (style 3→6, correction 3→4).
- Источник: product_data/bot_improvement_candidates_20260523/01_gold_and_few_shot/ (GOLD_ANSWERS_v3.yaml / bot_gold_answers.json).
- Жёсткие границы (без них R1 не катить): примеры — только тон/структура, НЕ источник фактов; числа/даты/скидки/адреса —
  лишь если есть в confirmed_facts; при конфликте с active_brand/фактами пример игнорируется; гейты после генерации не меняются.
- Флаг: `TELEGRAM_DRAFT_GOLD_V3_CONTEXT` (или контекст-флаг). Откат = выключить его одного; S1-S3 остаются.

## S4: шаблоны — в запас, НЕ полностью убирать (низкий/средний риск)
- Предпочитать ответ-из-факта вместо шаблонного зачина, когда факт есть и тема безопасна (бьёт templated_opening).
- НО шаблоны ОСТАВИТЬ для high-risk/manager_only и денежных тем — они там гарантируют безопасную формулировку.
  Демонтировать только косметический повтор зачинов, не защитные шаблоны. (Это уточнение к идее Кодекса «шаблон только как fallback».)

## X1, X2: НЕ в этот заход (вторые риски, отложены)
- **X1.** Ослабление автономности, чтобы снять over_handoff при наличии факта. Близко к безопасности — отдельный изолированный тест.
- **X2. LLM-редактор качества вторым проходом** (идея Кодекса: переписывать soft-fail живее). Мощно, но это ВТОРОЙ
  рискованный рычаг И самый большой расход токенов (доп. вызов модели на каждый ход — против цели «не жечь»). Делать
  ТОЛЬКО если R1 (gold в контексте) не дотянул. Сначала дешёвый рычаг, потом дорогой.

## Замечания к плану Кодекса (сверка)
- Метрику «отправил бы менеджер без правок» НЕ строить заново — она уже есть: `send_unedited_proxy.unedited_rate`
  (медиана 39.7%), а свежий holdout — round-6. Поднять её как главную, не плодить новую.
- recall 8/8 НЕ делать отдельным «слоем извлечения» вслепую. matkap_sfr_17: бот дал верные сроки СФР на 3-м ходе,
  выдумки нет — это тот же класс «ответил рядом», а не пропавший факт. Сначала проверить: есть ли «10 рабочих» в снимке
  и не чинится ли это самим S2 («сначала буквальный ответ»). Дёшево, и, возможно, recall закроется заодно.

## Протокол прогона (чтобы не жечь токены)
1. Один батч-коммит (S1+S2+S3+R1), НЕ по одной правке.
2. Офлайн pytest (дёшево) → мини-гейт на 5 диалогах: perevod_04, addr_22, tax_18, refund_presale_02, matkap_sfr_17.
3. Только если мини-гейт чистый (FAIL=0, hard=0, без протечки служебных фраз) → ОДИН медианный замер: 3 полных round-5.
4. Потом один прогон round-6 (28 невиданных персон) — финальная проверка обобщения.
5. НЕ перезапускать полные прогоны вслепую после каждого диалога; чинить на мини-гейте.

## Приёмка (по медиане, главная метрика — ПОХОДОВАЯ)
- `send_unedited_proxy.unedited_rate` (доля черновиков «отправил бы как есть») — РОСТ против базы 39.7%; цель ≥45%.
- FAIL=0, hard-gate=0 во всех трёх; recall не хуже 7/8 (а лучше — 8/8); 0 протечек служебных фраз; бренды чисты.

  дальше тон — это уже модель, а не контекст. Тогда честный вывод — пилот на текущем уровне.
