# ТЗ: следующий этап базы знаний Telegram-бота — полезные P1-ответы без потери безопасности

Дата: 2026-05-20  
Проект: Mango Analyse  
Статус: ТЗ для реализации после согласования Дмитрием  
Главная цель: сохранить пройденный P0-уровень безопасности и поднять полезность обычных справочных ответов Telegram-бота до уровня, достаточного для внутреннего пилота с менеджером.

## 1. Короткая суть

Последний MEGA smoke v3 показал важную развилку:

- критические сценарии безопасности прошли;
- обычные справочные ответы пока слишком неполные.

Это значит, что бот уже научился не делать самые опасные вещи: не собирать лишние данные при возвратах и юридических угрозах, не смешивать бренды, не обещать результат, не раскрывать себя как ИИ, не подчиняться вредным инструкциям клиента.

Но бот ещё не стал достаточно полезным помощником менеджера: по ценам, лагерям, налоговому вычету, маткапиталу, расписанию, форматам, материалам и комбинированным вопросам он часто отвечает слишком общо или пропускает важные утверждённые детали.

Следующий этап — не новый большой рефакторинг, а аккуратное достраивание слоя полезных справочных ответов.

Это не провал безопасности. Это провал справочной полноты.

## 3. Что именно не прошло

Все 107 падений относятся к P1/P2, кроме P0-проблем нет.

Основные группы:

| Группа | Количество падений | Смысл проблемы |
|---|---:|---|
| pricing | 22 | Недостаточно точные цены, периоды оплаты, раннее бронирование, 4 недели, индивидуальные занятия, интенсивы |
| camp | 12 | Неполные ответы по ЛВШ, ЛШ, датам, питанию, безопасности, трансферу, скидкам постоянным участникам |
| adversarial | 10 | Сложные/давящие/многошаговые вопросы обрабатываются безопасно, но слишком неполно |
| combined | 8 | В одном вопросе несколько тем, бот отвечает только на одну |
| tax | 7 | Налоговый вычет: сроки, годы, справки на нескольких детей, расчёт суммы |
| matkap | 6 | Маткапитал: возраст, документы, частичная оплата, федеральный/региональный |
| schedule | 5 | Старт, недели семестра, частота занятий, слоты |
| discount | 4 | Скидки действующим ученикам, лагерные скидки, постоянные участники |
| trial | 3 | Пробное занятие, лагерь/предлёнка |
| остальные | 30 | Формат, материалы, результаты, запись, возраст, доступ, интенсивы, медицина, переносы, контакты, B2B |

Главная причина:

В базе знаний и в слое формирования ответа есть много защитных правил, но нет достаточно полного слоя утверждённых справочных ответов по обычным клиентским вопросам.

## 4. Бизнес-цель этапа

Сделать так, чтобы менеджер получал не просто безопасный черновик, а полезный черновик, который:

1. отвечает на вопрос клиента по существу;
2. использует утверждённые факты из базы знаний;
3. не выдаёт неподтверждённые цены, даты, скидки и обещания;
4. не смешивает Фотон и УНПК;
5. честно уходит к менеджеру, если факта нет или он требует проверки;
6. помогает Насте быстрее отвечать клиентам, а не создаёт ещё один слой ручной переписки.

Целевой режим после этапа:

- клиенту всё ещё ничего не отправляется автоматически;
- Настя получает черновик;
- по обычным темам черновик должен быть достаточно содержательным;
- по опасным темам маршрут остаётся ручным.

## 5. Границы работ

Можно менять:

- слой построения черновика Telegram-пилота;
- слой выбора фактов из базы знаний;
- локальные файлы базы знаний в `product_data/knowledge_base/...`;
- тесты;
- локальные скрипты проверки MEGA smoke;
- документы в `docs/`;
- audit pack в `audits/_inbox/`.

Нельзя без отдельного подтверждения Дмитрия:

- отправлять сообщения клиентам;
- писать в AMO, CRM или Tallanto;
- менять `stable_runtime` DB/audio/transcripts;
- запускать ASR;
- запускать Resolve+Analyze по реальным данным;
- удалять старые базы знаний или runtime-артефакты;
- делать `git reset`, `git checkout`, `git clean`;
- запускать тяжёлые массовые batch-скрипты вне этого ТЗ;
- смешивать условия Фотона и УНПК в одном клиентском ответе.

## 6. Главный принцип реализации

Не нужно хаотично править 107 конкретных падений.

Нужно сделать слой, который устойчиво отвечает на классы вопросов:

- цена;
- скидка;
- рассрочка;
- маткапитал;
- налоговый вычет;
- лагерь;
- расписание;
- формат обучения;
- пробное занятие;
- материалы;
- результаты;
- запись;
- доступ;
- перенос;
- контакты;
- сложный вопрос из нескольких частей.

Каждый класс должен иметь:

1. список фактов, которые можно говорить клиенту;
2. список фактов, которые можно показывать только менеджеру;
3. шаблон ответа;
4. условия, при которых нужен менеджер;
5. запреты;
6. тесты.

## 7. Архитектурное решение

### 7.1. Не раздувать `subscription_llm.py`

В последнем блоке часть safety-логики была добавлена прямо в:

`src/mango_mvp/channels/subscription_llm.py`

Для P0 это допустимо, потому что нужна была быстрая защита.

Для P1-слоя так делать нельзя: если все цены, лагеря, скидки и налоговые правила зашить в один Python-файл, система быстро станет хрупкой.

Правильное направление:

- safety guards остаются в коде;
- справочные ответы и факты выносятся в управляемые данные;
- код выбирает подходящий ответ из реестра, а не хранит всю бизнес-логику внутри себя.

### 7.2. Создать слой `answer registry`

Нужно создать или расширить локальный реестр справочных ответов.

Предлагаемая структура:

`product_data/knowledge_base/kb_release_<date>/answer_registry.yaml`

или аналогичный JSON/YAML внутри текущего релиза базы знаний.

Для каждого ответа:

```yaml
answer_id: foton_pricing_online_9_2026_27
brand: foton
topic: pricing
subtopic: online_regular_9
route: draft_for_manager
client_allowed: true
manager_review_required: false
freshness: actual_until_review_date
source_ids:
  - ...
must_include:
  - онлайн
  - 9 класс
  - семестр
  - год
forbidden:
  - УНПК
  - kmipt
template: >
  ...
manager_note: >
  ...
missing_fact_fallback: >
  ...
```

Это не обязательно должен быть именно такой формат, но итоговый формат должен закрывать эти поля.

### 7.3. Разделить факты и фразы

Факт — это проверяемое утверждение:

- цена;
- дата;
- срок;
- процент;
- адрес;
- условие скидки;
- возрастное ограничение;
- список документов.

Фраза — это то, как бот говорит факт клиенту.

В базе должны быть оба слоя:

1. `facts_registry` — что правда;
2. `answer_registry` — как это безопасно сказать.

### Этап 3. Заполнить недостающие факты

Цель: закрыть фактические пробелы, которые мешают P1.

Источники:

- текущая база знаний;
- подтверждённые YAML/документы Клода;
- файл с правками Дмитрия;
- Google Drive документы, если доступен read-only;
- сайты Фотона и УНПК, если факт опубликован и актуален;
- ответы из опросника.

Правила:

1. Если факт подтверждён документом, бот может использовать его.
2. Если факт относится к Фотону, он не используется в ответе УНПК.
3. Если факт относится к УНПК, он не используется в ответе Фотона.
4. Если факт общий только технически, но брендовый контекст разный, писать два отдельных факта.
5. Если факт устаревающий, у него должен быть срок проверки или владелец обновления.
6. Если точного факта нет, ответ должен честно уходить в менеджера.

Минимальные группы фактов:

- цены Фотон;
- цены УНПК;
- раннее бронирование;
- 4 недели / пробный период;
- скидки;
- рассрочка и Долями;
- маткапитал;
- налоговый вычет;
- ЛВШ и ЛШ;
- расписание;
- пробное занятие;
- доступы и личный кабинет;
- материалы и домашние задания;
- результаты;
- запись и продолжение обучения;
- переносы и смена группы;
- контакты и соцсети.

Acceptance:

- каждая P1-группа имеет либо утверждённый факт, либо safe-handoff;
- у каждого точного числа есть источник;
- нет клиентского факта без `brand`;
- нет точного факта без `source_id` или описанного источника.

### Этап 4. Собрать `answer_registry`

Цель: сделать слой полезных ответов управляемым.

Действия:

1. На основе фактов и решений создать реестр ответов.
2. Для каждого ответа указать:
   - бренд;
   - тему;
   - условия применения;
   - маршрут;
   - текст для клиента/черновика;
   - текст для менеджера;
   - запреты;
   - обязательные маркеры;
   - источник;
   - статус проверки.
3. Поддержать комбинированные вопросы: ответ должен уметь покрывать 2-3 темы в одном сообщении, но без перегруза.

Acceptance:

- не менее 80% P1-тестов имеют связанный `answer_id`;
- для оставшихся есть причина, почему нужен handoff;
- реестр можно проверить скриптом без LLM.

### Этап 5. Подключить слой ответов к Telegram draft provider

Цель: LLM должна получать не сырую кучу фактов, а компактный и релевантный контекст.

Действия:

1. На входе определить:
   - бренд;
   - тему;
   - подтему;
   - риск;
   - есть ли несколько тем в одном сообщении.
2. Выбрать 1-3 подходящих ответа из `answer_registry`.
3. Передать LLM:
   - только текущий бренд;
   - только нужные факты;
   - запреты;
   - маршрут;
   - что обязательно упомянуть;
   - что нельзя говорить.
4. Если подходящего ответа нет:
   - использовать safe-handoff;
   - не выдумывать цены, даты, скидки, сроки.

Acceptance:

- P0-guards остаются выше справочного слоя;
- справочный слой не может ослабить маршрут опасной темы;
- prompt injection не может заставить модель обойти `answer_registry`;
- клиентское сообщение всегда обрамлено как пользовательский ввод, а не как инструкция.

### Этап 6. Добавить semantic gates

Цель: ловить смысловые ошибки автоматически.

Новые проверки:

1. `P0_guard_regression_gate`
   - возврат, юридическая угроза, жалоба, гарантия, prompt injection, кросс-бренд.

2. `P1_answer_completeness_gate`
   - если тест требует точные маркеры, они должны быть в ответе;
   - если факта нет, должен быть явный safe-handoff с причиной.

3. `brand_isolation_gate`
   - Фотон не объясняет УНПК;
   - УНПК не объясняет Фотон;
   - кросс-бренд редирект только мягкий и общий.

4. `unsupported_promise_gate`
   - проценты, суммы, даты и сроки должны быть из фактов;
   - иначе маршрут становится ручным.

5. `debug_leak_gate`
   - в черновике нет `source_id`, JSON, snake_case, внутренних ID, `Claude`, `Codex`, `GPT`.

6. `business_plausibility_gate`
   - цена курса не меньше разумного порога;
   - число занятий не превращается в рубли;
   - срок не превращается в цену.

Acceptance:

- semantic gates запускаются отдельно от обычных unit-тестов;
- итоговый отчёт содержит `formal_pass` и `semantic_pass` отдельно;
- каждая новая смысловая ошибка попадает в тест или чек-лист.

## 9. Критерии готовности этапа

Этап считается выполненным только если:

1. `formal_pass=true`:
   - тесты проходят;
   - файлы валидны;
   - MEGA-прогон завершён.

2. `semantic_pass=true`:
   - P0 остался 100%;
   - P1 не ниже 95%;
   - нет кросс-бренд утечек;
   - нет неподтверждённых чисел;
   - нет debug-текста;
   - нет сбора лишних персональных данных в опасных темах;
   - нет извинений/признания вины в жалобах;
   - справочные ответы реально полезнее “уточним и вернёмся”.

3. Создан audit pack:
   - `implementation_notes.md`
   - `changed_files.txt`
   - `test_output.txt`
   - `semantic_review.md`
   - `risk_review.md`
   - `backward_compatibility.md`
   - `mega_smoke_summary.json`
   - `remaining_failures.csv`

4. Есть пакет для Клода/сотрудников:
   - что изменилось;
   - какие факты добавлены;
   - какие решения нужны от человека;
   - какие тесты прошли.

## 10. Что должно остаться совместимым

Нельзя сломать:

- текущий Telegram draft provider;
- P0 safety guards;
- режим `draft_for_manager`;
- отсутствие автоответов клиенту;
- read-only режим AMO/Tallanto;
- бренд-разделение;
- текущий MEGA smoke runner;
- существующие tests по `subscription_llm`.

Если новый `answer_registry` не найден, система должна:

- не падать;
- использовать прежний безопасный fallback;
- не выдавать точные неподтверждённые факты.

## 11. Использование субагентов

При реализации можно использовать до 6 субагентов с `xhigh`.

Рекомендуемое разделение:

1. Субагент 1: анализ 107 failures и группировка.
2. Субагент 2: pricing/discount факты.
3. Субагент 3: camp/schedule/trial факты.
4. Субагент 4: tax/matkap/documents.
5. Субагент 5: semantic gates и тесты.
6. Субагент 6: независимый смысловой аудит результата как внешний критик.

Важно: субагенты не должны править одни и те же файлы одновременно. Если используются worker-субагенты, им нужно дать разные зоны ответственности.
