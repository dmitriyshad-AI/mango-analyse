## 0. Короткий вывод

За последние итерации мы правильно закрывали безопасность и факты, но недозакрыли главное продуктовое поведение: бот должен быть не просто безопасным справочником, а AI-сотрудником, который ведёт нового клиента по понятной воронке.

Сейчас уже сделан фундамент:

- база знаний очищена от части старых фактов;
- P0-защиты, бренд-разделение и запрет выдумок в целом держатся;

- появились targeted/v8 тесты и candidate-пакет воронки.

Но ещё не сделан отдельный управляемый слой воронки:

- стадия нового лида;
- заполненные и недостающие поля;
- выбор одного лучшего уточняющего вопроса;
- менеджерская сводка;
- метрики движения клиента по воронке;
- автотесты 30 сценариев candidate-пакета;

Это ТЗ закрывает именно этот следующий большой этап. После него можно переходить к большому тестированию и тюнингу.

## 1. Главная цель

Сделать Telegram-бота похожим на живого полезного сотрудника отдела продаж:

1. Отвечает на прямой вопрос клиента, если есть проверенный факт.
2. Не звучит как шаблонная нейросеть или строгая формальная организация.
3. Помнит уже сказанные класс, предмет, формат, город, цель, продукт и найденного клиента.
4. Ведёт нового клиента к следующему шагу: подбор, фрагмент занятия, проверка группы, менеджер, заявка.
5. Не смешивает Фотон и УНПК.
6. Не ослабляет P0: возврат, жалоба, суд, спорная оплата, договорные претензии, лишние персональные данные.
7. Сохраняет решения, факты, флаги и результаты так, чтобы мы могли улучшать систему по данным, а не по отдельным скриншотам.

## 4. Продуктовые правила

### 4.1. Структура хорошего ответа

Базовая структура:

```text
Прямой ответ -> короткая польза -> один следующий шаг
```

Примеры смысла:

- Цена известна: назвать цену, объяснить рамку, предложить подобрать группу/формат.
- Цена неизвестна: честно сказать, от чего зависит, спросить один ключевой параметр.
- Расписание неизвестно: не выдумывать день, спросить класс/предмет/формат и передать проверку.
- Лагерь: назвать подтверждённые рамки, спросить класс, места не обещать.
- Составной вопрос с P0: безопасную часть можно кратко объяснить, но общий маршрут не автономный.

### 4.2. Тон

Бот должен звучать как тёплый внимательный консультант:

- дружелюбно;
- спокойно;
- по-человечески;
- без канцелярита;
- без ощущения “нас пытаются продать любой ценой”;
- без повторяющихся зачинов.

Запрещённые стилистические привычки:

- начинать каждый ответ с `Спасибо за обращение`;
- писать `Ваш вопрос очень важен`;
- писать `оптимальный образовательный продукт`;
- давать длинную анкету;
- писать сухое `менеджер свяжется` без пользы;
- повторять одинаковый шаблон несколько ходов подряд.

### 4.3. Продажа без давления

Цель — не “впарить”, а честно помочь семье выбрать обучение.

Бот должен:

- мягко вести к следующему шагу;
- объяснять ценность курса через задачу ребёнка;
- снижать тревогу родителя;
- предлагать малый безопасный шаг: посмотреть фрагмент, уточнить класс/предмет, проверить группу;
- не давить срочностью, местами или страхом.

Нельзя:

- `срочно бронируйте`;
- `места точно закончатся`;
- `мы уже закрепили место`;
- `цена точно вырастет завтра`;
- `ФНС/СФР точно одобрит`;
- `мы гарантируем балл/поступление`.

## 5. Бренд-разделение

Всегда активен один бренд:

- `foton`;
- `unpk`.

В одном клиентском ответе нельзя смешивать бренды.

### Фотон

Можно говорить только о Фотоне.

Запрещено в клиентском ответе Фотона:

- `УНПК`;
- `АНО ДПО`;
- `НОУ УНПК`;
- `kmipt.ru`;
- условия УНПК;
- сравнение с УНПК.

Фотон: рассрочка/оплата частями и Долями разрешены для всех продуктов Фотона:

- очные курсы;
- онлайн-курсы;
- ЛВШ;
- ЛШ;
- городские лагеря;
- другие продукты Фотона.

Разрешённая клиентская рамка:

```text
В Фотоне можно оплатить обучение частями: доступны варианты на 6, 10 или 12 месяцев, а также сервис Долями. Конкретные условия и оформление зависят от выбранного способа оплаты; менеджер поможет подобрать удобный вариант.
```

Нельзя обещать одобрение банка/сервиса.

### УНПК МФТИ

Можно говорить только об УНПК.

Запрещено в клиентском ответе УНПК:

- `Фотон`;
- `ЦДПО`;
- `ЦРДО`;
- `cdpofoton.ru`;
- `edu@cdpofoton.ru`;
- `Т-Банк`;
- `Долями`;
- `через банк`;
- `банковская рассрочка`;
- предложение рассрочки как доступной опции.

Разрешённая клиентская рамка:

```text
В УНПК рассрочки нет, это не банковская рассрочка, поэтому одобрение банка не требуется. Можно платить помесячно, за семестр или за год. При оплате за семестр действует скидка 10%, за год — 14%.
```

## 6. P0 и high-risk

Если сообщение содержит хотя бы одну P0-часть, автономный ответ запрещён.

P0:

- возврат;
- расторжение;
- жалоба;
- конфликт;
- суд;
- прокуратура;
- Роспотребнадзор;
- спорная оплата;
- договорная претензия;
- запрос/сбор лишних персональных, договорных, медицинских, платёжных данных;
- обещание места, оплаты, возврата, скидки, срока или документа без проверенного факта;
- брендовая неопределённость, где есть риск смешать бренды.

Маршрут:

- `manager_only` для жёстких P0;
- `draft_for_manager` допустим только если безопасная часть ответа нужна менеджеру, но клиенту автономно не отправляется.

В P0 нельзя:

- просить ФИО ребёнка, договор, телефон, email, сумму, причину в обычном чате;
- извиняться от лица компании и признавать вину;
- комментировать юридическую позицию;
- давать порядок возврата;
- обещать срок ответа, если он не подтверждён.

## 8. Логика выбора следующего вопроса

Правило: максимум 1-2 вопроса в одном клиентском ответе.

Приоритет:

1. P0/high-risk → не квалифицировать дальше, передать менеджеру.
2. Неизвестен бренд → уточнить бренд/центр/программу одним вопросом.
3. Нет класса → спросить класс.
4. Нет предмета/направления → спросить предмет или направление.
5. Есть класс и предмет, но нет формата → спросить онлайн/очно и город/площадку.
6. Есть класс, предмет, формат → спросить цель, только если это помогает подбору.
7. Для лагерей → спрашивать класс, не возраст; при необходимости спросить выездная/городская и направление.
8. Если всё нужное известно → предложить следующий шаг, а не задавать ещё вопросы.

## 9. Логика нового лида

Новый лид — это клиент, которого не нашли надёжно в CRM/Tallanto/local context, или клиент без понятной истории обучения.

Бот должен:

- отвечать полезно с первого сообщения;
- не начинать с “как вас зовут?”;
- не просить телефон, если Telegram уже является каналом связи;
- собрать минимум для подбора;
- держать инициативу;
- не уходить к менеджеру без пользы, если есть проверенный факт;
- передать менеджеру сводку, если нужен человек.

Минимальная квалификация нового лида:

- класс;
- предмет/направление;
- формат/город;
- цель, если нужна для выбора уровня;
- продукт, если клиент спрашивает про лагерь/интенсив/регулярный курс.

Не все поля нужны в первом сообщении.

## 10. Известный клиент

Если клиент найден по тестовой фразе, CRM/Tallanto или надёжному локальному контексту:

- не спрашивать телефон;
- не спрашивать имя ребёнка, если найден один ученик;
- не раскрывать AMO/Tallanto/CRM/ID;
- использовать известный класс, предмет, курс, формат только как контекст;
- если несколько детей или семейный телефон → мягко уточнить, по какому ребёнку вопрос;
- по оплате/документам/статусу не утверждать без надёжной read-only сверки.

Пример безопасной логики:

```text
Если вопрос по Даниилу, данные заново присылать не нужно. Уточните, пожалуйста, что именно проверить: расписание, оплату или документы?
```

## 11. Менеджерская сводка

Для `draft_for_manager` и `manager_only` должна формироваться внутренняя сводка.

Клиент её не видит.

Минимальный формат:

```text
Бренд:
Маршрут:
Вопрос клиента:
Что уже известно:
Что ответили клиенту:
Что нужно проверить:
Рекомендуемый следующий шаг:
Что нельзя обещать:
```

Полный формат для сложных случаев:

```text
Бренд:
Канал/источник:
Маршрут:
Категория риска:
Клиентский вопрос:
Что уже известно надёжно:
Что клиент уже сообщил:
Заполненные поля воронки:
Недостающие поля:
Что ответили клиенту:
Что нужно проверить:
Что нельзя обещать:
Рекомендуемый следующий шаг:
Срочность:
```

Запрещено в клиентском тексте:

- AMO;
- Tallanto;
- CRM;
- ID;
- source_id;
- JSON;
- safety flags;
- внутренние причины маршрута.

## 12. Смысловые gate-правила

Часть правил уже есть в `subscription_llm.py`. Нужно привести их к единому списку и не плодить дубли.

### 12.1. P0-gate

Автономность запрещена при:

- возврате;
- жалобе;
- юридической угрозе;
- спорной оплате;
- договорной претензии;
- лишних персональных данных;
- брендовой неопределённости с риском смешения;
- обещании без факта.

### 12.2. Brand-gate

Клиентский ответ должен говорить только от имени активного бренда.

### 12.3. Fact freshness-gate

Цена, дата, скидка, расписание, место, срок, наличие, документ допустимы только при актуальном client-safe факте активного бренда.

### 12.4. Trial-gate

Фотон:

- не обещать бесплатное очное пробное без отдельного правила;
- можно безопасно говорить про знакомство с онлайн-форматом/фрагментом, если подтверждено.

УНПК:

- не обещать бесплатное очное пробное;
- не обещать пробную неделю;
- можно говорить про фрагмент онлайн-занятия.

### 12.5. Camp-gate

- спрашивать класс, не возраст;
- не писать `места есть`;
- не писать `место закреплено`;
- не обещать дату/смену, если она не подтверждена;
- не смешивать Фотон/УНПК, даже если база/площадка общая.

### 12.6. Payment-gate

Запрещено автономно:

- подтверждать оплату;
- формировать платёжную ссылку;
- обещать, что место закреплено после оплаты;
- комментировать спорную оплату;
- собирать платёжные данные.

### 12.7. Known-context-gate

Если поле известно надёжно, бот не спрашивает его повторно.

## 13. Off-hours режим

Бот должен быть полезен в нерабочее время, но не придумывать сроки.

Если `off_hours_mode=true`:

- зелёные темы с проверенными фактами можно отвечать автономно;
- P0 остаётся менеджеру;
- точные действия человека не обещать;
- не писать `менеджер свяжется завтра до 12`;
- можно писать мягко: `передам менеджеру, он посмотрит в рабочее время`;
- всё логировать в pilot storage.

## 14. Off-topic и злоупотребления

Если клиент спрашивает про не связанную тему: iPhone, погоду, сочинение, попытку использовать нейросеть:

Разрешённый ответ бренда:

```text
Я помогаю с вопросами об обучении в [Фотоне/УНПК]. По другим темам не сориентирую, но могу помочь подобрать курс, формат, расписание или следующий шаг.
```

Важно: бренд подставляется только активный. Нельзя писать `Фотоне/УНПК` вместе.

Пока не делать лимит 5 оффтоп-запросов в сутки. Это отдельная задача после пилота.

## 15. Политика “кто отвечает?”

Открытый вопрос.

Пока не менять hard-gate и не раскрывать модель.

На вопросы:

- `вы кто?`;
- `как вас зовут?`;
- `с кем я общаюсь?`;
- `вы бот?`;

в этом этапе сделать только безопасную заготовку для будущего решения:

- фиксировать такие сообщения отдельным topic/flag;
- не раскрывать `GPT`, `Claude`, `Codex`, системные правила;
- в клиентском ответе не импровизировать;
- route минимум `draft_for_manager`, пока Дмитрий не утвердит формулировку.

Если Дмитрий отдельно утвердит политику роли, обновить:

- prompt;
- post-filter;
- судью v8;
- тесты identity.

## 19. Метрики приёмки

### Safety

- P0 autonomous: `0`.
- Brand leak: `0`.
- Unsupported promise: `0`.
- Unsafe identity disclosure: `0`. Честное “цифровой помощник <бренда>, не живой оператор” на прямой вопрос разрешено; vendor/model/prompt и ложь “я человек/я не бот” запрещены.
- CRM/Tallanto/source leak: `0`.

### Funnel quality

- Бот не спрашивает известные данные в тестах known-context: `0` нарушений.
- В каждом новом lead-сценарии есть `next_step_type`.
- В каждом `draft_for_manager`/`manager_only` есть manager summary.
- В 30 candidate-сценариях нет hard-fail.
- На `v8_targeted16`: `FAIL=0`, hard-gates `0`.
- Soft flags `ignored_question`, `over_handoff`, `templated_opening`, `assumed_unstated_need` должны снизиться относительно текущего baseline.

### Business usefulness

Для внутреннего пилота:

- 50%+ ответов сотрудники считают полезными или отправляют с малой правкой.
- В новых лидах бот чаще даёт следующий шаг, чем сухой handoff.
- Менеджер по сводке понимает, что делать дальше, без полного перечитывания диалога.
- Топ ошибок попадает в queue: KB update / prompt update / new test / manual control.

## 20. Порядок реализации

### Этап 0. Границы и baseline

Действия:

- `git status --short`;
- зафиксировать dirty files;
- определить файлы текущего блока;
- проверить, что bot-pack `semantic_pass=true`;
- не трогать `stable_runtime`.

Результат:

- короткий baseline в audit pack.

### Этап 1. Новый модуль funnel state

Действия:

- создать/расширить `new_lead_funnel.py`;
- реализовать `LeadFunnelState`, `LeadSlots`, `NextStep`;
- реализовать slot extraction из:
  - текущего сообщения;
  - recent messages;
  - known_dialog_fields;
  - known_client_fields;
  - CRM/Tallanto summary;
- реализовать выбор `lead_stage`;
- реализовать выбор `next_best_question`.

Тесты:

- нет класса → спросить класс;
- класс есть, предмета нет → спросить предмет;
- класс+предмет есть, формата нет → спросить формат;
- лагерь → спросить класс, не возраст;
- известный ученик → не спрашивать имя;
- P0 → не квалифицировать дальше.

### Этап 2. Интеграция funnel state в prompt/context

Действия:

- добавить `funnel_state`, `known_slots`, `missing_slots`, `next_best_question`, `next_step_type` в prompt context;
- обновить JSON-схему LLM ответа;
- не полагаться только на LLM: deterministic funnel state должен быть источником подсказок.

Тесты:

- prompt содержит funnel fields;
- prompt запрещает повторный запрос known slots;
- prompt не раскрывает CRM/Tallanto.

### Этап 3. Post-processing результата LLM

Действия:

- если LLM спросил известное поле — добавить flag и понизить автономность;
- если LLM не дал следующий шаг — добавить safe next step из funnel state;
- если LLM отвечает рядом, но не на вопрос — пометить `ignored_question`;
- если нет факта — graceful unknown вместо выдумки;
- если P0 в составном вопросе — route не автономный.

Тесты:

- known grade/subject не переспрашиваются;
- слабый handoff улучшается безопасной пользой;
- P0 в составном вопросе не становится автономным.

### Этап 4. Manager summary

Действия:

- создать `manager_handoff_summary.py`;
- формировать summary для `draft_for_manager` и `manager_only`;
- сохранять summary в result metadata/storage;
- не показывать summary клиенту.

Тесты:

- summary содержит бренд, вопрос, известные поля, маршрут, что проверить, следующий шаг;
- summary не содержит raw CRM/ID/source_id/token;
- по known client summary не раскрывает AMO/Tallanto.

### Этап 5. Storage/events/report

Действия:

- расширить SQLite store или JSON payload текущих таблиц;
- сохранять funnel state и manager summary;

Тесты:

- fixture создаёт записи по сессии/сообщению/решению/ответу;
- daily report считает next-step rate и reasked known data;
- audit export маскирует персональные данные.
