## 1. Зачем

Глобальная цель — автономный продавец полного цикла. Сегодня бот пишет только текст-черновик: явного решения «совершить действие по сделке» в пайплайне нет. Это ТЗ добавляет такой слой **под флагом (по умолчанию выключен)**. Пока бот не исполняет действия сам — он **рекомендует следующий шаг менеджеру**, человек в контуре. Это и фундамент автономной продажи, и недостающий сигнал для судьи действий ТЗ-21.

## 2. Логика работы (обсуждаем) — предложение модели → решение детерминированного слоя → рекомендация менеджеру

**Шаг 1. Модель ПРЕДЛАГАЕТ действие (`action_proposal`).** После понимания и фактов модель предлагает одно следующее действие из закрытого списка. Список **выведен из частых вопросов клиентов** (звонки/мессенджеры/почта; темы взяты из gold-базы — pricing, discounts, installment, schedule, trial, camps, addresses, matkap, tax, platform, identity, P0):

| Действие (контракт-имя из ТЗ-21) | Тип вопроса клиента | Оценка в итерации 1 |
|---|---|---|
| `answer_only` | факт-вопрос: цена, скидки, рассрочка, адрес, маткапитал, налог, платформа/записи, «вы бот?» | оценивается |
| `send_schedule` (отправить расписание) | «когда занятия / какое расписание / нагрузка» | оценивается (с заземлением) |
| `send_materials` (материалы / фрагмент пробного) | «можно пробное / посмотреть урок / материалы» | оценивается (только онлайн-фрагмент) |
| `send_crm_data` (баланс / остаток занятий из CRM) | «сколько у меня осталось / мои оплаты / баланс» | оценивается (строгая идентификация) |
| `capture_lead` (зафиксировать заявку) | «оставлю заявку / интересно / запишите» | оценивается |
| `schedule_followup` (перезвон/напоминание) | «перезвоните / напишите позже» | оценивается |
| `send_payment_link` (предложить оплату) | «хочу оформить / куда платить» | оценивается (порог ниже) |
| `send_document` (договор/счёт) | «нужен договор / счёт / документы» | наблюдается, не гейтим |
| `advance_stage` (продвинуть этап AMO) | переход по воронке | наблюдается, не гейтим (нет словаря этапов) |
| `handoff_manager` (менеджеру) | P0 (возврат/жалоба/спор/юр.), «есть ли места», вне зоны | оценивается |

`send_materials` заменил прежний `book_trial`: очного/бесплатного пробного у нас нет, есть только онлайн-фрагмент занятия (gold `trial_class`). Имена-идентификаторы — единый контракт с ТЗ-21 (английские); русское — только в тексте рекомендации менеджеру. Новые значения (`send_schedule`, `send_materials`, `send_crm_data`) добавляются в контракт ТЗ-21 как версия v2 (синхронно, чтобы судья их читал). Это **сырое предложение модели**, не команда.

**Шаг 2. Детерминированный слой ПРИНИМАЕТ решение (`action_decision`).** Ключевая правка после ревью: решение принимает НЕ модель и НЕ keyword-«воронка», а проверка **по фактическому выходу бота и последней реплике клиента**. **Где это живёт (первый шаг Кодекса — короткая разведка, это и есть ответ на Поставку 0 ТЗ-21):** носитель `action_decision` — слой на **боевом** пути (там, где собирается финальный результат и `conversation_intent_plan`). `night_funnel_shadow` — ночной/теневой модуль, боевой путь его не вызывает; его проверки (P0, бренд, заземление чисел) **переиспользуем как библиотеку**, но носитель решения ставим на боевой путь — не вешаем дневную продажу на ночной гейт. Кодекс сперва письменно подтверждает точную точку-носитель, потом реализует.

Предусловия для самого необратимого действия — `send_payment_link` — это **явные, проверяемые по выходу сигналы** (а не слоты модельной воронки):
1. **цену назвал сам бот из подтверждённых фактов** — в истории есть ход, где конкретная сумма прошла проверку «есть в фактах» (механика уже есть: проверка заявленных чисел); не «слот цены заполнен», а «бот реально произнёс сумму, и она из фактов»;
2. **продукт однозначен** — определитель продукта не вернул «непонятно»;
3. **клиент явно подтвердил** — детерминированный маркер согласия на ПОСЛЕДНЕЙ реплике клиента («да, оформляйте / беру / куда платить»), а не модельная оценка «готов».

**Асимметрия (важно):** модельная оценка готовности может только **понизить** действие (не предлагать), но **никогда не разблокировать** оплату. Нет явных сигналов 1–3 → оплата запрещена, чем бы модель ни была уверена. Если предусловия установить нельзя → `action_decision = unknown` (не `answer_only`).

**«Да» клиента — необходимое, но НЕ достаточное.** Оплату предлагаем только если в той же реплике НЕТ возражения/сигнала ухода (берём существующие `objection`/`exit_signal`, не наивный паттерн «да» — иначе словим «да это дорого» → оплата на возражении). Асимметрия «понижать, не разблокировать» распространяется на ВСЕ продающие действия; `capture_lead` низкориск (клиенту ничего не обещает).

**Предусловия и рамки НОВЫХ действий (важно):**
- `send_schedule` — отправлять расписание только когда известны класс + предмет + формат и расписание есть в проверенном источнике (группы Tallanto); точные дни не выдумывать (gold: «точный день без факта» запрещён); бренд верный. Иначе `answer_only` + уточнение.
- `send_materials` — только онлайн-фрагмент; НЕ обещать «бесплатное», «очное пробное», «сколько угодно» (gold `trial_class`); материал существует для класса/предмета/формата.
- `send_crm_data` (баланс/остаток занятий) — **самое чувствительное: это персональные данные конкретного клиента.** Жёсткие предусловия: (1) клиент **уверенно идентифицирован** (совпадение контакта в CRM / механика «пишу с номера X»); (2) бренд карточки = бренд диалога; (3) **НЕ P0** (спор по балансу/оплате = P0 → менеджер). Не тот клиент или неуверенная идентификация → НЕ отправлять, `handoff_manager`. В пилоте — только рекомендация менеджеру с данными из карточки; бот сам данные не отправляет. Никогда не отдавать данные одного клиента другому (проверка соответствия контакта карточке — детерминированная).

**Защёлка P0 — самым последним шагом, из единого источника.** Действие проверяется на P0 **на финальном `SubscriptionDraftResult`, после ВСЕХ стражей маршрута — включая `apply_autonomy_matrix_guard`** (он идёт после текстового P0-override, поэтому формулировка «после override» недостаточна). Дизъюнкцию P0 берём из **единого источника** (`p0_recall_spec` — общий реестр кодов + финальные `route==manager_only`/`p0_required`/`is_p0` результата); **слой действия НЕ заводит собственных регулярок P0** (иначе четвёртый рассинхронный детектор). Любой P0 → `action_decision = handoff_manager`, продающее снято. Обязательный именованный тест мультитемы: «жалоба + вопрос о цене» в одном сообщении → действие = только менеджер, продающего нет.

**Шаг 3. В пилоте `action_decision` — РЕКОМЕНДАЦИЯ менеджеру, не исполнение.** Расширяем уже существующее поле «Рекомендуемый следующий шаг» в сводке для менеджера. Формулировка **не понуждающая и условная**: не «отправь ссылку на оплату», а «клиент назвал готовность; если подтвердите — менеджер сформирует ссылку». **Бот по живой CRM/оплате сам не ходит.** Если последняя реплика клиента — вопрос, `send_payment_link` не рекомендуется даже при заполненных слотах (вопрос ≠ готовность платить). **При `action_decision = unknown`** поле «Рекомендуемый следующий шаг» либо пустое, либо «нет явного сигнала к действию» — **нельзя** подставлять туда эвристику воронки (`next_best_question`), иначе менеджер примет угадку за решение слоя действия.

**Порог шума.** Продающую рекомендацию (`book_trial`/`send_payment_link`) выдаём только при `уверенность ≥ порог` И выполненных явных предусловиях; иначе `answer_only` без рекомендации. `handoff_manager`/`answer_only` от порога не зависят (они безопасны). Порог калибруется на золотом наборе ТЗ-21 §7.6, не выдумываем число.

**Позже** (будущий шаг флага, не сейчас): когда качество `action_decision` доказано симулятором ТЗ-21 + наблюдением — флаг можно поднять до авто-исполнения низкорисковых действий. Привязка к матрице автономности и правилу «автономия только при точном совпадении зоны».

## 3. Жёсткие рамки (нарушать нельзя)

1. **P0 всегда главнее.** Дизъюнкция P0-сигналов, последним шагом; на P0 продающее запрещено.
2. **Бренды раздельны** на уровне действия.
3. **Без выдумок:** сумма/ссылка — только из подтверждённых фактов (проверка по выходу).
4. **Решает детерминированный слой по явным сигналам, не модель и не keyword-воронка.** Модельная готовность только понижает.
5. **Два поля:** `action_proposal` (модель, сырое) и `action_decision` (после проверки, авторитетное). Эмиттер ТЗ-21 сериализует **именно `action_decision`** в свою метку `action_intent` (не третье имя), читает только его, никогда не выводит действие из текста. **Появление `action_decision` = ответ «сигнал есть» на Поставку 0 ТЗ-21 → снимает заморозку Поставки 2 ТЗ-21.**
6. **Согласованность текст↔действие — по ТЗ-21 §7.4 в обе стороны:** `promise_without_action_intent` (текст обещал — решения нет) и `action_not_in_text` (решение есть — текст не реализует). Рассинхрон чинится **понижением до `answer_only` + флаг**, НЕ переписыванием текста (переписывание вернёт выдумку).
7. **Флаг по умолчанию выключен;** включённый = только рекомендация менеджеру, не авто-исполнение.
8. **Грязное дерево git:** ветка от `main` (канон после уборки), не коммитить в области CRM-диалога; у Кодекса только тесты, прогон на M1 `--parallel 4`.

## 4. Что делает Кодекс (пайплайн)

1. Ввести **два поля**: `action_proposal` (модель) и `action_decision` (после проверки). За флагом (по умолчанию off).
2. **Сначала разведка (ответ Поставки 0 ТЗ-21): подтвердить точку-носитель `action_decision` на БОЕВОМ пути** (рядом с финальной сборкой результата / `conversation_intent_plan`). `night_funnel_shadow` боевой путь не вызывает — берём из него проверки как библиотеку, носитель ставим на боевой путь. P0/бренд/заземление переиспользовать, не переписывать.
3. Предусловия `send_payment_link` — по явным сигналам выхода/последней реплики; «да» необходимо-не-достаточно (через `objection`/`exit_signal`); асимметрично (модель только понижает). Предусловия НОВЫХ действий: `send_schedule` (класс+предмет+формат известны, расписание из проверенного источника, дни не выдумывать), `send_materials` (только онлайн-фрагмент, без обещаний «бесплатно/очно»), `send_crm_data` (строгая идентификация клиента + бренд карточки + не P0; не тот клиент → менеджер). Раздел 2.
4. **P0 OR-защёлка действия** — на финальном результате после ВСЕХ стражей (вкл. `apply_autonomy_matrix_guard`); дизъюнкция из единого источника `p0_recall_spec`, без новых регулярок.
5. Порог `уверенность` для продающих рекомендаций (значение — из калибровки ТЗ-21 §7.6).
6. Расширить поле «Рекомендуемый следующий шаг» в сводке менеджеру (не плодить новый формат); формулировка не понуждающая.
7. Контракт имён действий — единый с ТЗ-21 (английские идентификаторы; русский — только в тексте рекомендации). `advance_stage`/`send_document`/`schedule_followup` держать в контракте как **наблюдаемые-не-гейтящиеся** (нет словаря этапов AMO — кандидат снимок `20260513`).
8. Тесты (зелёные в зоне): флаг off = поведение не меняется; P0 OR-защёлка (вкл. мультитему) перебивает продажу; предусловия по явным сигналам; выдуманную сумму не пропускает; бренд; рассинхрон текст/действие → понижение; `unknown` при неустановленных предусловиях.
9. Отчёт в `tasks/_done/`.

## 5. Критерии приёмки (боевые, итерация 1)

- **Флаг off → регресс зелёный** (текущий пайплайн не изменился).
- **Флаг on (детерминированные тесты):** на P0-кейсах (вкл. мультитему) `action_decision=handoff_manager` 100%, продающих на P0 — ноль; межбрендовых действий — ноль; выдуманных сумм — ноль; `send_payment_link` только при явных сигналах 1–3; при вопросе клиента в последней реплике оплата не рекомендуется.
- **Поставлено поле `action_decision`** (готов сигнал для эмиттера ТЗ-21).
- Формат рекомендации понятен менеджеру (примеры).

**Важно (разведение зависимости):** прогон симулятора ТЗ-21 с **включённым судьёй действий** — это **Поставка 2 ТЗ-21**, отдельная веха **после** приёмки этого ТЗ (тогда судья размораживается, читая `action_decision`), с калибровкой §7.6. Не вешаем приёмку ТЗ-23 на ещё не размороженного судью.

## 6. Что НЕ делаем

- ❌ Бот не ходит по живой CRM/оплате сам (флаг в режиме рекомендации).
- ❌ Не ослабляем P0/бренд/анти-выдумку ради «состоявшегося» действия.
- ❌ Решение о действии не выводим из текста и не отдаём одной модели без детерминированной проверки.
- ❌ Предусловие оплаты не строим на слотах keyword-воронки (только явные сигналы выхода).
- ❌ Рассинхрон текст/действие не «чиним» переписыванием текста.
- ❌ `advance_stage` не гейтим в этой итерации (нет словаря этапов).
- ❌ Не коммитим в грязные области; ветка от канона.

## 7. Решения Дмитрия (закрыто) и остаток

**Закрыто:**
1. Порог «можно предлагать оплату» (цена из фактов + продукт однозначен + явное «да» клиента) — **принят**.
2. Список действий **доработан по анализу частых вопросов**: `book_trial` → `send_materials`; добавлены `send_schedule` и `send_crm_data` (§2). Гейтятся: `answer_only`/`send_schedule`/`send_materials`/`send_crm_data`/`capture_lead`/`schedule_followup`/`send_payment_link`/`handoff_manager`; наблюдаемые — `advance_stage`/`send_document`.
3. Маршрут — **главному Кодексу пайплайна** (Кодекс-2 занят Graphify); рекомендации — в общую очередь менеджера.
4. Номер — **ТЗ-26**.

**Остаток на подтверждение:** рамки `send_crm_data` — «уверенная идентификация» = совпадение контакта в CRM (механика «пишу с номера X»), и в пилоте это **только рекомендация менеджеру** (бот сам данные клиента не отправляет). Рекомендую так; если хочешь строже (баланс всегда через менеджера) — скажи.

## Приложение — журнал ревью двух архитекторов

**Раунд 1** (аудитор проверял по коду пайплайна):

- **Б1 (предусловия висели на keyword-воронке — корневой изъян на самом опасном действии)** — принято: предусловия оплаты теперь по явным сигналам выхода (цена из фактов + продукт≠«непонятно» + явное «да» клиента), асимметрия (модель только понижает). Раздел 2, шаг 2.
- **Б2 (дублирование — выходной проверяющий уже есть в `night_funnel_shadow`)** — принято: расширяем существующий гейт, не плодим новый; поле рекомендации менеджеру тоже расширяем существующее. Раздел 4.
- **Б3 (P0 на мультитеме может проскочить — P0 размазан по 4+ точкам)** — принято: P0 OR-защёлка действия последним шагом, после финального override; именованный тест мультитемы. Раздел 2/3.
- **Б4 (расхождение с ТЗ-21: «модель предлагает» vs «эмиттер читает решение»)** — принято: два поля `action_proposal`/`action_decision`; эмиттер ТЗ-21 читает только `action_decision`; `unknown` при неустановленных предусловиях. Раздел 3.5.
- **В1 (шум рекомендаций)** — принято: порог `уверенность` для продающих (калибровка ТЗ-21 §7.6); безопасные действия от порога не зависят.
- **В2 (симметрия текст↔действие)** — принято: переиспользуем ТЗ-21 §7.4; чиним понижением, не переписыванием.
- **В3 (преждевременное давление)** — принято: формулировка не понуждающая; при вопросе клиента оплату не рекомендуем.
- **В4 (`advance_stage` без словаря этапов)** — принято: наблюдаемый-не-гейтящийся, как в ТЗ-21 §7.1.
- **В5 (циклическая зависимость приёмки от замороженного судьи)** — принято: приёмка итерации 1 = регресс + детерминированные тесты + поставка `action_decision`; прогон судьи действий — Поставка 2 ТЗ-21, после.
