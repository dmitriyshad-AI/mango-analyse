### 3 обязательных требования Шага 2b (корни over-handoff из Прогона 1 — НЕ гейт, вышестоящие)

Эти 3 дефекта — целевая работа миграции. Внести как требования + негатив-тесты в правила:

1. **refund (правило enrollment_process / refund-ветка):** гипотетический предоплатный возврат =
   спокойная справка про остаток неистраченных средств НА ВСЕХ ходах, даже при связке «возврат + после
   старта», если есть явные маркеры гипотетики («в целом / без договора / до оплаты / просто спрашиваю»).
   Реальная претензия (оплатил+нет занятий / верните деньги недоволен) → P0 менеджер.
   NEG-тест: `refund_hypo` (3+ хода гипотетики) → НЕ hard_p0, держать справку; `refund_claim` → P0.
   Корень: входной `p0_pregate` не читает маркеры гипотетики — калибровать его, не выходной гейт.

2. **identity (`are_you_bot`):** на прямой вопрос «бот или человек?» политика C («Я помощник менеджера…»)
   срабатывает на ПЕРВОМ ходу, НЕ шаблонный хендофф. Композер генерит ответ C сразу; гейт только блокирует
   утечку природы, не сам ответ.
   NEG-тест: «это бот или живой человек?» первый ход → ответ C, route=answer_self (НЕ draft_for_manager).

3. **address/contact (правило contact_address):** адрес — known client-safe факт, бот отвечает из факта
   СИММЕТРИЧНО по брендам (сейчас УНПК отвечает, Фотон 3 хода уходит в менеджера). Фотон-адрес = Скорняжный.
   NEG-тест: Фотон «где очные занятия / адрес?» → ответ из факта (Скорняжный), route=answer_self.
