# CRM / Calls Quality Defect Class Map

Дата: 2026-05-10
Статус: рабочая карта классов проблем перед v5-планом.

## Зачем этот документ

Цель - остановить цикл "Claude нашел примеры -> Codex закрыл примеры -> Claude нашел новые примеры".
Ниже перечислены не конкретные строки, а классы дефектов, которые могут портить CRM, ROP-пакеты, bot knowledge base и будущий SaaS-продукт.

## Принцип классификации

Каждая новая находка должна попадать в одну из категорий:

- известный класс, уже есть detector/gate;
- известный класс, detector/gate слабый;
- новый класс, нужен threat-model update;
- допустимый бизнес-case, не баг;
- спорный case, нужен manual/LLM review.

## A. Классы дефектов на уровне звонка

### A1. Non-conversation попал в contentful

Примеры классов:

- IVR, автоинформатор, голосовой помощник;
- voicemail / автоответчик;
- менеджер оставил сообщение, но живого клиента не было;
- служебная проверка телефонии;
- короткий шум/обрывок/тишина;
- wrong number / случайный набор.

Риск: в CRM появляется фейковая история и фейковые следующие шаги.

### A2. Out-of-domain контакт попал в sales/service

Примеры классов:

- поставщики, интеграторы, подрядчики;
- HR/вакансии;
- банки/госорганы/фонды;
- доставки, хозтовары, документы, интернет, оборудование;
- чужая компания, другая организация;
- справочники/проверки/службы.

Риск: менеджер видит мусор как потенциального клиента.

### A3. Low-intent / no-request контакт ошибочно считается полезным

Примеры:

- клиент не подтвердил интерес;
- тема не уточнена;
- обучение/класс/предмет/продукт не обсуждались;
- только "перезвоните / кто звонил" без контекста;
- фраза "помощь не требуется";
- не удалось определить источник обращения.

Риск: CRM засоряется слабыми follow-up задачами.

### A4. Service-call и active-client context ошибочно трактуются как новый лид

Примеры:

- текущий клиент уточняет оплату/ссылку/расписание;
- действующий клиент закрыт в AMO как lost по причине "действующий клиент";
- сервисная жалоба ошибочно повышает вероятность продажи.

Риск: неверные reopen/follow-up рекомендации.

### A5. Содержательная часть звонка частично потеряна из-за transcript-quality

Примеры:

- ASR исказил бренд/фамилию/предмет;
- speaker split ошибся: клиентские реплики попали в менеджера;
- одна модель ASR лучше другой, но final transcript выбрал хуже;
- transcript содержит повторы/мусор;
- summary построен по короткому фрагменту, а не по полной сути.

Риск: все downstream-выводы выглядят уверенно, но опираются на плохой текст.

## B. Классы дефектов на уровне клиентской цепочки

### B1. Один человек / семья размазаны по нескольким телефонам

Сейчас цепочка в основном строится по normalized phone. Для SaaS этого мало.

Риск: история неполная, менеджер не видит весь контекст.

### B2. Несколько людей/компаний склеены в один телефон

Примеры:

- общий номер;
- старый владелец номера;
- корпоративная телефония;
- несколько детей/родителей на одном номере.

Риск: история разных сущностей смешивается.

### B3. Несколько сделок на один контакт не разведены по смыслу

Примеры:

- покупка одного курса, интерес к другому;
- сервисная сделка и новая продажа;
- ребенок 1 и ребенок 2.

Риск: writeback в сделку/контакт может давать неверный next action.

### B4. История вне звонков отсутствует

Email, мессенджеры, Telegram и личные сообщения не входят в текущую цепочку.

Риск: мы считаем менеджера неответившим или клиента холодным, хотя общение было в другом канале.

### B5. Старая история переоценивается как актуальная

Примеры:

- звонок был давно;
- клиент уже купил/отказался/ушел;
- следующий шаг устарел;
- вероятность продажи не деградировала по времени.

Риск: отдел продаж получает устаревшие рекомендации.

## C. Классы дефектов CRM writeback

### C1. Noise rows в AMO-ready

Класс F-002/F-006: detector ловит прошлый аудит, но пропускает новые формулировки того же шума.

Нужен не только detector, но и population-level counter / class-marker prevalence.

### C2. Обрезанные CRM-тексты

Поля заканчиваются на `...` или Excel/formatter обрезает смысл.

Риск: менеджер видит неполную историю.

### C3. Дублирование истории

Краткая история и хронология могут повторять одно и то же без добавочной ценности.

Риск: карточка становится длинной и плохо читаемой.

### C4. Неверное поле назначения

Примеры:

- полезная история только в контакте, но менеджер работает из сделки;
- служебная аналитика попала в публичное/неподходящее поле;
- protected fields (`Id Tallanto`, `Филиал Tallanto`) случайно стали write targets.

Риск: менеджеры не получают контекст или портятся системные поля.

### C5. PII / third-party data leakage в CRM-текст

Примеры:

- чужой телефон внутри истории;
- email/telegram handles;
- ФИО третьих лиц;
- служебные адреса/кабинеты.

Для CRM часть PII допустима, но только если это именно контакт клиента и поле предназначено для этого.

### C6. CRM/internal vs bot/public boundary leakage

Цены, бренды, адреса, дедлайны допустимы в CRM-internal, но недопустимы в bot-safe allowlist.

Риск: бот начнет обещать устаревшие цены/адреса/условия.

### C7. Writeback stale-source drift

Пример уже был: `CANONICAL_EXPORT.txt` указывал на старый слой.

Риск: в CRM пишется устаревшая история при зеленом новом gate.

## D. Классы дефектов scoring / ROP recommendations

### D1. Lead score не учитывает всю цепочку

Если score построен по последнему звонку, он может игнорировать старый сильный интерес или наоборот старый отказ.

### D2. Outcome source неполный

2025 больше в Tallanto, 2026 больше в AMO. Если outcome не подтянут из обоих источников, корреляции и рекомендации искажены.

### D3. Действующий клиент ошибочно считается lost/reopen

Закрытая проигранная сделка с причиной "действующий клиент" не должна быть reopen candidate.

### D4. Follow-up предлагается без учета менеджерской реальности

Примеры:

- контакт уже в работе в мессенджере;
- менеджер уже сделал следующий шаг вне звонка;
- клиент просил не звонить.

### D5. Неверная персонализация предложения

Примеры:

- перепутан ребенок/класс/предмет;
- сервисный вопрос превращен в продажное предложение;
- клиенту предлагают уже купленный продукт.

## E. Классы дефектов bot / knowledge base

### E1. "Лучший ответ" выбран без outcome-контекста

Ответ может звучать красиво, но не приводить к продаже.

### E2. LLM quality score не совпадает с реальной конверсией

Нужна отдельная метрика: LLM-score + sales outcome + ROP review.

### E3. Вопрос клиента неверно выделен из цепочки

Один звонок может содержать несколько вопросов, возражений и скрытых сигналов.

### E4. Сигнал клиента и вопрос клиента смешаны

Для бота нужны policy/reaction signals, а не только FAQ-вопросы.

### E5. Tenant policy не отделена от универсального ответа

Для SaaS нельзя зашивать цены, филиалы, бренды и правила Фотона как универсальные.

## F. Классы дефектов архитектуры / продукта

### F1. SQLite / local files не масштабируются до server mode

Пока это нормально для локального batch. Для SaaS нужны Postgres, миграции, очереди и locks.

### F2. Нет tenant config как источника истины

Нужны конфиги:

- отрасль;
- продукты;
- допустимые lead types;
- CRM mapping;
- protected fields;
- privacy policy;
- bot safety policy.

### F3. Нет единого lifecycle / release gate

Нужны статусы:

- raw import;
- ASR;
- R+A;
- quality backfill;
- canonical rebuild;
- CRM-ready;
- dry-run;
- staged writeback;
- live writeback.

### F4. Audit corpus narrow / audit overfitting

Frozen corpus должен включать:

- synthetic adversarial;
- реальные прошлые findings;
- random sample;
- forward-looking broad class markers;
- positive examples.

### F5. Нет population-level recall метрик

Detector может показывать `0 blocking_rows`, потому что сам не видит класс.
Нужны параллельные prevalence counters и random audit.

### F6. Артефакты и папки разрастаются без release manifest

Нужен порядок:

- release manifest;
- latest pointers;
- archive old runs;
- immutable audit results;
- cleanup policy.

### F7. Недостаточная наблюдаемость

Нужны dashboards/metrics:

- сколько звонков на стадиях;
- сколько blocked/manual/ready;
- причины block;
- drift от прошлого релиза;
- качество ASR/R+A;
- writeback errors.

## G. Классы дефектов интеграций

### G1. Mango API ingestion еще не live

Пока загрузка звонков ручная. Для продукта нужен controlled capture через API.

### G2. AMO OAuth/runtime brittle

Нужны:

- refresh-token lifecycle;
- health check;
- safe dry-run;
- retry/backoff;
- protected-field enforcement.

### G3. Tallanto matching неполный

Телефоны могут быть в разных форматах, у одного клиента несколько телефонов, не все outcome попадают в AMO.

### G4. Cross-system conflict resolution

AMO и Tallanto могут расходиться по имени, статусу, оплате, филиалу, владельцу.

## H. Классы безопасности и юридического риска

### H1. PII в артефактах аудита

Audit packs должны быть минимизированы и redacted там, где это возможно.

### H2. Access control в SaaS

Менеджер, РОП, владелец, техподдержка и внешний клиент должны видеть разные уровни данных.

### H3. Retention / deletion policy

Нужно понимать, как долго храним аудио, транскрипты, PII и LLM outputs.

### H4. Secrets leakage

OAuth tokens, AMO/Tallanto/Mango credentials, API keys не должны попадать в пакеты/логи.

## Предварительный вывод

Главный класс текущей проблемы - не конкретные строки AMO-ready, а `F4/F5`: narrow audit corpus + отсутствие population-level recall метрик. Пока detector учится только на прошлом аудите, следующий аудит будет находить новые формулировки того же класса.

Правильный следующий этап:

1. независимый class-discovery от Claude по широкому пакету;
2. объединить карту классов Claude + Codex;
3. создать v5 план не по regex-правкам, а по архитектурным gates;
4. только после этого продолжать реализацию detector/gate/export.
