## Принцип (зачем всё это)
Сейчас: хрупкий regex ПОНИМАЕТ на ВХОДЕ, безопасность решается по угаданному намерению, в монолите
(`subscription_llm.py` ~8000 строк, **407 инлайн brand/P0/manager_only-решений vs 41 вызов модулей**).
Цель: модель ПОНИМАЕТ (падает мягко переспросом), а детерминированная охрана проверяет ГОТОВЫЙ ЧЕРНОВИК
на ВЫХОДЕ — только ПОНИЖАЕТ до хендоффа/блокирует, НИКОГДА не повышает и не выдумывает. Безопасность с
ВХОДА → на ВЫХОД.

## Что УЖЕ есть (заморозить как канон, НЕ переписывать)
Выделенные модули: `answer_safety_classifier.classify_answer_safety` (P0), `p0_recall_spec.hard_codes_from_text`
(HARD-P0), `fact_claim_audit.audit_fact_claims` (выдумки), `fact_scope_spec.scope_family_for` (scope),
`verify_output` (pipeline:1416 — бренд/числа/мета/сущности по черновику), санитайзеры/бренд-токены. Это
КАНОН. Их логику не менять; гейт их ВЫЗЫВАЕТ.

## Архитектура (4 фазы + 1 замер)
- **Фаза 1:** единый авторитетный `safety_gate` ПОВЕРХ всего, на выходе build_draft. Монолит остаётся как
  есть (двойная защита). Гейт — последнее слово.
- **★ ЗАМЕР безопасности (обязателен):** гейт ловит ВСЁ, что ловил монолит (P0 8/8, бренд 0, выдумки 0),
  при ещё живом монолите. Парашют раскрылся — только тогда режем трос.
- **Фаза 2:** отнять у монолита ИНЛАЙН-власть над безопасностью — его route-override'ы становятся
  ПОДСКАЗКАМИ (soft), финальное слово только у гейта. Свести 407 инлайн-точек к сигналам.
- **Фаза 3:** за гейтом заменить regex-понимание (`_asks_/_is_`, 145 re.*) модельным planner'ом волнами,
  удаляя детекторы по группам. Реестр fact_id (черновик сверяется с источником факта).
- **Фаза 4:** снести монолит понимания. Гейт НЕ выкидывать никогда.
- **Финальный замер** после фаз 2-4.

## ФАЗА 1 — точные полуфабрикаты

### 1.1 Единая функция-гейт (downgrade-only, чистая)
Новый модуль `output_safety_gate.py` (или в существующем safety-модуле):
```python
@dataclass(frozen=True)
class GateVerdict:
    route: str            # итоговый route (только тот же или БОЛЕЕ строгий)
    draft_text: str       # тот же текст ИЛИ безопасная замена/хендофф
    blocked_reasons: tuple[str, ...]
    downgraded: bool

def apply_output_safety_gate(
    *, draft_text: str, route: str, contract, facts: Mapping[str,str],
    active_brand: str, client_message: str, context,
) -> GateVerdict:
    """ПОСЛЕДНЕЕ слово по безопасности. Только ПОНИЖАЕТ. Никогда не повышает route и не выдумывает текст."""
    reasons = []
    # 1. P0 по СМЫСЛУ входа (страховка) + по контракту
    safety = classify_answer_safety(client_message=client_message, context=context, route=route)
    if safety.p0_required or hard_codes_from_text(client_message):
        return _force_manager(route, contract, context, reason="p0")      # downgrade → manager_only
    # 2. Проверка ЧЕРНОВИКА (выход): бренд/числа/мета/сущности/обещания
    findings = verify_output(draft_text, facts=facts, active_brand=active_brand,
                             contract=contract, client_message=client_message, context=context)
    # 3. Выдумки/scope по фактам
    findings += _as_findings(audit_fact_claims(draft_text, facts=facts, contract=contract))
    if _has_blocking(findings):                 # бренд-утечка/выдумка/мета/обещание
        return _downgrade_or_block(draft_text, route, findings, contract, context)
    # чисто → пропустить как есть (НЕ повышая route)
    return GateVerdict(route=route, draft_text=draft_text, blocked_reasons=(), downgraded=False)
```
ВАЖНО: гейт НИКОГДА не делает route более автономным и не генерит новый контент — он либо пропускает, либо
понижает/заменяет на проверенный хендофф (cite-only из факта допустим только если уже построен ранее и
проходит verify; гейт сам ничего не сочиняет).

## ★ ЗАМЕР после фазы 1 (критерий перехода к фазе 2)
Прогон на целевом наборе + P0/жалоба/возврат/бренд кейсы. КРИТЕРИЙ (всё обязательно):
- HARD-P0 (претензия/спор/юр/жалоба) → manager_only, как до гейта (гейт НЕ ослабил).
- Бренд/мета violated_gates = 0. Выдумки = 0.
- over-handoff НЕ вырос значимо (гейт downgrade-only, не должен душить валидные ответы — если душит,
  значит verify_output/audit слишком строги, калибровать ДО фазы 2).
- `output_safety_gate_downgraded` срабатывает там, где монолит и так понижал (гейт дублирует защиту, а не
  создаёт новую дыру/строгость).
Только при зелёном — фаза 2. Иначе чинить гейт, монолит ещё страхует.

## ФАЗЫ 2-4 — ДИЗАЙН + ПРИНЦИПЫ (точные полуфабрикаты — после фазы 1)

### Фаза 2 — отнять инлайн-власть монолита (свести к подсказкам)
Принцип: каждый из ~407 инлайн route-override в монолите → либо удаляется (если гейт уже ловит то же), либо
превращается в `hint`/flag для planner'а, НЕ финальный route. Финал — только гейт. Делать ГРУППАМИ
(brand-инлайн, P0-инлайн, refund-инлайн…), после каждой группы — юнит-щит + сверка, что гейт держит. Реестр
детекторов (`REGISTRY_detectors`) — карта, что hint vs что удаляем.

### Фаза 3 — заменить понимание моделью + реестр fact_id
Принцип: за авторитетным гейтом regex-понимание (`_asks_/_is_`, 145 re.*) заменяется модельным planner'ом
(тема/продукт/класс/формат/подвопросы/уверенность). Детекторы удаляются волнами по группам. Реестр fact_id:
каждый факт в черновике несёт id источника, гейт сверяет (усиление audit_fact_claims). Понимание падает
МЯГКО (переспрос), не уходом.

### Фаза 4 — снос монолита понимания
Когда понимание целиком в planner'е и инлайн-власть снята — удалить мёртвую темплейт-машинерию и regex-слой.
Гейт остаётся навсегда. Сюда же — удаление подтверждённо-мёртвого (`_semantic_recover_or_handoff` и т.п.).

## НЕГАТИВНЫЙ КОНТРОЛЬ (на каждый чек гейта, критично)
- HARD-P0 → manager_only; cross-brand → блок; обещание результата/поступления → блок; утечка лицензии/
  source_id/мета → блок; число вне факта → блок/понижение. На КАЖДЫЙ — позитив (ловит) + негатив (не
  перебарщивает на чистом ответе). Гейт downgrade-only: тест, что он НЕ повышает route и НЕ сочиняет текст.
- Гипотетический возврат (без претензии) → отвечает про остаток (не заблокировать зря).
- Замер: бренд/мета 0, выдумки 0, HARD-P0 держат — на каждом переходе фаз.
