# Полный аудит проекта Mango Analyse — что реализовано, что работает, что мёртвое

Пин: git `649c9397` (read-only). Метод: чтение кода + 6 независимых разборов с evidence file:line.
Сопутствующие: `FINDINGS_quality_strategy.md` (бот), `KB_QUESTIONS_AND_EDITS.md` (база), `GLOBAL_STRATEGY.md` (стратегия).

## Карта проекта (~135K строк Python)

Два контура: (1) исторический пайплайн звонков — зрелый; (2) productization/SaaS — преждевременный.
Плюс продуктовый фокус — Telegram-бот, который сейчас НЕ использует большинство построенного.

## Ключевые находки по контурам

### Пайплайн звонков — зрелый, но «сильное состояние» на ~80%
Инфраструктура реальная (ASR/Resolve/Analyze, очереди, ретраи, dead-letter). Оговорки: Resolve по умолчанию
rule-only (LLM off) — структура диалога эвристическая; «0 missing ASR/R+A» считается против allowlist-урезанного
знаменателя + `resolve_status=manual` зачитывается как покрытие → метрика условная; вся PII/деньги-защита —
regex с захардкоженными списками имён (непокрытое имя/искажённая ASR-цифра пройдут). Монстр-файлы
(transcribe 3117, analyze 2340, resolve 2062). **Вывод: не вкладываться больше, «достаточно».** Низкоусилийная
гигиена: аудитить exclude-CSV (чтобы «0 missing» не прятал растущие исключения) + заархивировать ~15 мёртвых
`transcript_quality_*` dry-run модулей.

### Бот (channels) — главный фокус, но качество строится не там
Подробно в `FINDINGS_quality_strategy.md`: качество производят правилами + ~90 канцелярит-шаблонами, рерайтер
нейтрализован, измерение поощряет подмену. Уточнение из KB-аудита: **шаблоны в БАЗЕ — инструкции, проблема в
КОДЕ бота.** Лечится ТЗ v2 (контракт ответа + генерация + один rewrite + safety-block).

### CRM/история — построено, но НЕ включено (крупнейший parked-актив)
`customer_timeline` (14.9K), `deal_aware`, `insights` — **ни один не в живом пути бота.** `telegram_pilot_context_builder`
и `customer_context_for_draft` имеют 0 импортёров (orphan). Живой бот строит CRM-контекст отдельно
(`build_read_only_crm_context`), но и его роняет белый список ключей промпта (`read_only_customer_context` не в
`_ALLOWED_CONTEXT_KEYS`). customer_timeline выключен из-за soft-гейта `p1_missing_recommended_action` (флаг
«нет рекомендованного действия», не «плохие данные») → 82% manual — **самоналоженный потолок, не качество данных.**
Это самый высокоокупаемый parked-актив: 15K строк уже построены, нужно лишь ослабить гейт и подключить.

### Productization/SaaS — преждевременный, заморозить
43.5K строк почти полного SaaS-каркаса (захват, продуктовая БД, read-only API+HTTP, планировщик, supervisor,
appliance-loop, 14 стадий гейтов, tenant-обвязка) при НУЛЕВОМ внешнем клиенте и НУЛЕВОЙ связи с ботом. ASR-sandbox
13 файлов/7.2K + mail_archive 8K = ~15K строк (35%) под запрещённую/чужую функцию. 16K строк тестов на каркас.
**Заморозить (park):** оставить ядро захвата (`mango_office_client`, `product_db`, `repository`, `contracts`,
`capture`), остальное — снять с обязательного зелёного прогона до решения о продуктизации.

### CRM-запись / агент — корректно gated, но ~11K строк непотребляемой поверхности
AMO/deal writeback + agent_runtime выключены и недостижимы из бота. Риска live нет, но это поверхность, которую
надо держать выключенной (никто не должен зарегистрировать live-handlers / щёлкнуть WRITE_AMO_LIVE).

## Сквозные неэффективности
1. **Огромный parked-балласт:** SaaS 43.5K + timeline-история 15K (off) + agent/writeback 11K (dead) + ~15 мёртвых
   quality-модулей + mail_archive 8K — десятки тысяч строк без живого потребителя, требующие держать тесты зелёными.
2. **Десинк фактов** в сборке бот-KB (хардкод цен поверх YAML).
3. **Regex-потолок** везде, где нужна семантика (PII, P0, intent, факты по словам).
4. **Монстр-файлы** (subscription_llm 4279, transcribe 3117, mail_archive 8059, analyze 2340).
5. **Конкурирующие планы/ТЗ** (TZ_FULL_VISION call-first vs аудиты bot-first) → организационный churn.

## Что реально РАБОТАЕТ и приносит пользу сегодня
- Пайплайн звонков → канонический набор данных (~66K звонков).
- Оффлайн-аналитика инсайтов/ROP-паков (как материал, не в реальном времени).
- Живой Telegram-бот (безопасный, но пока «шаблонный»).
- База знаний v6.3 (бренд-чистая, цены актуальные) — как клиентский источник для черновиков.
- Канонический слой звонков + инвентарь (maintenance).

Всё остальное (SaaS, timeline, deal_aware, agent_runtime, mail_archive) — построено, но не в работе.
