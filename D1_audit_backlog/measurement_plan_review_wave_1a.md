Автор: второй Claude. Дата: 2026-05-29. Read-only. Источник: `TZ_wave_1a_FINAL_for_codex_2026-05-29.md` §4 + код (`answer_safety_classifier.py`, `conversation_intent_plan.py`, `p0_recall_spec.py`).

**Вердикт: PASS_WITH_NOTES.** Направление верное (FP/FN/серая зона + ИЛИ-критерий), но есть 5 дыр, которые надо закрыть до прогона, иначе измерение даст ложно-успокаивающую картину.

## Сразу: расхождение в формулировке критерия

В постановке задачи критерий записан «FP > 5% **И** FN > 0 **И** серая > 10%» (AND). В самом ТЗ §4 (`:186`) — через **ИЛИ**. Это критично: при AND 1b почти никогда не запустится (нужны все три сразу), при ИЛИ — любой триггер. **Правильно ИЛИ** (TZ), AND — опечатка постановки. Дальше считаю по ИЛИ.

## Вопрос 1 — корректность метрики FP/FN/серая + упущенные edge-cases

Метрики разумны как направление, но:

1. **Нет размеченного ground-truth на 212.** FP/FN требуют ЗНАТЬ правильную P0-метку каждого хода. У прогона 212 есть judge-результаты, но НЕТ разметки по группам анкеты (A-G). Без размеченного подмножества FP/FN на реальном трафике посчитать нельзя — только распределение и «серую зону» (несогласие веток). FP/FN чисто считаются лишь на размеченных наборах (синтетика-siblings, `P0_TRUE_POSITIVE_CASES`). **Нужен отдельный gold-labeled срез 212** (или выделенный набор) для FP/FN на реальном трафике.
2. **Soft handoff ломает бинарность.** После правки 6 `escalation ∈ {hard_p0, soft_handoff, none}` — это 3 класса, а FP/FN бинарны (P0 / не P0). Ход «должен быть soft, стал hard» — это over-эскалация, но НЕ пропущенный P0. Нужна матрица 3×3 с раздельным учётом: over-escalation (none→soft/hard, soft→hard) vs under-escalation (p0→soft/none). Бинарные FP/FN скрывают soft-измерение.
3. **Знаменатель не задан.** FP «% ходов» — от ВСЕХ ходов или от не-P0 ходов? P0 редки (~единицы % из ~1000 ходов на 212). 3% от всех ≈ 30 ложных эскалаций — много. Считать FP от не-P0 ходов И давать абсолютные числа, не только %.
4. **Латч-каскад не меряется.** Один ложный P0 латчится через `p0_latch` и травит последующие ходы. Мерить и turn-level FP, и «диалоги с ≥1 ложным P0», и распространился ли латч.
5. **Новое поведение не меряется.** Качество soft_handoff (дал ли существенный ответ, а не просто «передам менеджеру») и ретрая (правка 7) FP/FN не ловят — нужны отдельные acceptance-проверки.
6. **FN на эталонах ≠ FN на трафике.** FN=0 на эталонах необходимо, но недостаточно: опасен пропущенный реальный claim в живом трафике, а он виден только на gold-разметке.

## Вопрос 2 — реалистичность критерия перехода к 1b

- **ИЛИ-логика верна** (любой из трёх → 1b). `FN > 0 → 1b` — сильно и правильно (любой пропуск P0 недопустим).
- **Но FN измеряется «на эталонах», а эталоны — это то, на чём правила и калибровались → циркулярно.** FN=0 почти гарантирован by construction и неинформативен. **FN надо мерить на HELD-OUT размеченном наборе**, который НЕ использовался при настройке правок 1-6.
- **Пороги без знаменателя неинтерпретируемы.** «FP < 3%» — от чего? Зафиксировать базу (не-P0 ходы) и абсолют.
- **Смешаны базы:** FN «на эталонах», FP/серая «на 212» — разные наборы в одном критерии. Развести явно: какой набор для какой метрики.
- **Нет хард-гейта на контроль.** 4 эталонных P0 и `P0_TRUE_POSITIVE_CASES` должны быть 100% (не %-критерий, а блокирующий гейт): если хоть один не сработал — откат, независимо от FP/серой.
- Граница 3-5% / 5-10% → Дмитрий — ок.

Итог: критерий реалистичен по структуре, но требует (а) held-out gold для FN, (б) знанателей, (в) хард-гейта на контроль, (г) 3-классового учёта soft.

## Вопрос 3 — откуда брать эталоны по группам A-G + споры 1-5

Что УЖЕ есть в коде (готовые размеченные наборы):
- `P0_TRUE_POSITIVE_CASES` (`p0_recall_spec.py:89`, 18 кейсов) — покрывает A/B/C (+ payment_dispute).
- `P0_BENIGN_CASES` (`p0_recall_spec.py:110`, 11 кейсов) — покрывает F/G и benign-возврат («если не понравится…», «справка для вычета», «возврат к теме»).
- 4 эталонных P0 (`P0a_refund_claim_01/02`, `P0b_complaint_01/02`) — из eval-сценариев.
- ~22 синтетических siblings блока 1a (`regression_suite_synthetic_siblings.md`).

ЧЕГО НЕ хватает (надо завести):
- **D** (недовольство без фин-маркера → не P0), **E** (эмоция к боту → не P0), **Спор 1** (soft_handoff), **Спор 3** (справка → P0), **Спор 5** (преподаватель → P0) — в коде размеченных эталонов мало/нет.
- **Рекомендация:** намайнить реальные формулировки из транскриптов 212 (`dynamic_dialog_transcripts.jsonl`) по каждому паттерну (grep по «не нравится преподаватель», «месяц…справк», эмоции к боту «не ответили/бесполезно», post-payment «оплатил…верн») → разметить по ответам Дмитрия → заморозить как **measurement-gold** (реальные фразы, не только синтетика). 10-15 на группу.
- **Критично:** measurement-gold для FP/FN должен быть ОТДЕЛЬНЫМ от tuning-siblings (иначе FN циркулярен, см. в.2). Два набора: tuning (siblings, можно смотреть при настройке) и gold (held-out, смотреть только на замере).

## Вопрос 4 — кто запускает M1 и минимальный лог

- **Кто:** прогон 212 — это eval-харнесс, которым сделан `runs/20260528_103821_eval` (его запускает Кодекс/Дмитрий, НЕ Claude — по правилам проекта Claude не запускает eval/ASR/Resolve). «M1» — машина Дмитрия с харнессом. Claude может только разметить gold и проверить лог post-hoc.
- **Минимальный лог на КАЖДЫЙ ход** (для диагностики FP, не только факта FP):
  - `dialog_id`, `turn`, `client_message`.
  - Ветка-план: `ConversationIntentPlan.primary_intent`, `refund_frame`, `route_bias`, `intent_confidence` (есть в `to_prompt_view()`).
  - Ветка-classify: `p0_required`, `escalation` (после правки 6), `risk_codes`, **`evidence`** (какой маркер сработал — REFUND_RE/COMPLAINT_RE/… — без него FP не диагностируется), `semantic_non_p0` (всё есть в `AnswerSafetyDecision.to_json_dict()` `:34`).
  - `p0_latch.active` + `p0_latch.codes` (для латч-каскада).
  - Финальный `route` хода.
  - `gold_label` (если ход из размеченного gold).
- Хорошая новость: `to_json_dict()` (classify) + `to_prompt_view()` (plan) УЖЕ сериализуют почти всё — лог дешёвый (дамп двух объектов на ход).
- Для «серой зоны» — логировать вердикты ОБЕИХ веток на один вход, иначе несогласие не посчитать.

## Вопрос 5 — есть ли инфра для замера FP/FN, или с нуля

Не с нуля — есть строительные блоки, но харнесс замера надо дописать:
- ЕСТЬ: `AnswerSafetyDecision.to_json_dict()` (`:34`) и `ConversationIntentPlan.to_prompt_view()` (`:61`) — сериализация. Размеченные фикстуры `P0_TRUE_POSITIVE_CASES`/`P0_BENIGN_CASES` (`:89/:110`).
- FP/FN НА ФИКСТУРАХ — тривиально СЕЙЧАС, без прогона: цикл по `P0_TRUE_POSITIVE_CASES` (recall/FN) + `P0_BENIGN_CASES` (benign-pass/FP) + siblings. ~0.5 дня, можно до 212.
- FP/FN НА РЕАЛЬНОМ ТРАФИКЕ (212) — НЕТ харнесса: нужен (а) gold-разметка среза, (б) скрипт-скан, сравнивающий лог с gold. ~0.5-1 день.
- СЕРАЯ ЗОНА — НЕТ: прогнать `build_conversation_intent_plan` и `classify_answer_safety` на каждом ходе и сравнить вердикты. Переиспользовать сериализацию. ~0.5 дня.
- Итого: ~1-1.5 дня на харнесс поверх существующей сериализации, не с нуля.

## Список правок к §4 (для PASS)

1. Развести наборы: **tuning-siblings** (настройка) и **held-out measurement-gold** (замер FN/FP). FN мерить ТОЛЬКО на gold, иначе циркулярно.
2. Сделать метрику 3-классовой (hard_p0 / soft_handoff / none): матрица 3×3, раздельно over- и under-эскалация.
3. Задать знаменатели и абсолюты: FP от не-P0 ходов + абсолютное число; отдельно «диалоги с ≥1 ложным P0» (латч-каскад).
4. Добавить хард-гейт: 4 эталонных P0 + `P0_TRUE_POSITIVE_CASES` = 100% (блокирующий, не %-критерий).
5. Расширить лог: `evidence`, `risk_codes`, `p0_latch`, обе ветки, gold_label (всё доступно через `to_json_dict`/`to_prompt_view`).
6. Завести measurement-gold по D/E/Споры 1/3/5 из реальных транскриптов 212 (10-15 на группу), разметить по ответам Дмитрия.
7. Acceptance для нового поведения: soft_handoff даёт существенный ответ; ретрай (правка 7) реально происходит — отдельные проверки, не через FP/FN.
