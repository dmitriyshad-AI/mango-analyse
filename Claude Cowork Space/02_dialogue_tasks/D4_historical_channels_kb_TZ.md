# D4 — Исторические каналы → база знаний: задача (гигиена базы + обогащение через одобрение)

Зона файлов: `knowledge_base/`, релизы и данные базы, очереди кандидатов, исторические каналы (почта/телеграм/
звонки — как КАНДИДАТЫ). Не трогать `channels/`.

## Преднастройка (по замечанию Кодекса №5 — грязное дерево)
Перед стартом зафиксировать список изменяемых файлов блока. В дереве уже висят удаления старых релизов базы
из параллельных потоков — НЕ смешивать с ними; работать отдельным коммитом; при конфликте с чужими
удалениями остановиться и согласовать, чужое не откатывать.

## Часть 1. Гигиена базы v6.3 (закрыть до расширения автономности)
1. **Прогнать настоящую проверку (smoke/Stage6) по v6.3.** Сейчас релиз помечен «проверка не запускалась»
   (`_smoke_not_run`): зелёный статус — это только формальная сборка, а не доказательство поведения.
2. **Свести сборку базы к ОДНОМУ источнику правды.** Сейчас сборщик переписывает цены/контакты прямо в коде
   поверх исходного YAML (например 93100; 114000/120000; контакт ВК; трансфер) и меняет глобальные переменные
   на ходу. Убрать эти правки-в-коде: всё берётся из YAML, сборщик факты не переписывает.
3. **Точечные правки фактов** (из `01_strategy/KB_QUESTIONS_AND_EDITS.md`): проверить верх онлайн-года Фотона
   (52 500 вне подтверждённого списка); сделать фильтр «Долями/Т-Банк/рассрочка» зависимым от бренда (чтобы он
   не резал законный ответ Фотона про рассрочку).

## Часть 2. Отлов критичных тем — общий артефакт с главным диалогом (замечание Кодекса №2)
НЕ заводить свой отдельный список критичных тем. Импортировать/ссылаться на `p0_recall_spec`, который ведёт
главный диалог (D1). Один и тот же проверочный набор (мошенники/незаконно/жаловаться → к менеджеру) должен
проходить и в коде бота, и в триггерах базы. Так списки не разъедутся.

## Часть 3. Обогащение из исторических каналов — ТОЛЬКО через одобрение
1. Из почты/телеграма/звонков собирать КАНДИДАТОВ (факты/стиль/тесты/вопросы) в очередь одобрения
   `kb_candidate_approval_queue_<дата>/` со статусами (новый кандидат / нужно подтверждение РОПа / нужен
   первоисточник / только стиль / только тест / отклонён как небезопасный / одобрен в базу / одобрен в примеры).
2. **Запреты:** телеграм/звонок/почта — НЕ источник цены/даты/скидки без подтверждения; таблицы с личными
   данными не класть в подсказку модели; ответы менеджеров не переносить дословно (личные данные/старые цены/
   смешение брендов/обещания). Один бренд в ответе.
3. Стиль-кандидаты и частые вопросы → в живые примеры/тесты ТОЛЬКО после одобрения (в бота их вливает главный диалог отдельно).

## Связка с главным диалогом
- Новый пакет для бота главный диалог подхватит при следующем прогоне; не править этот пакет из главного диалога.
- Критичные темы — единый артефакт (см. Часть 2), не два списка.

## Приёмка
v6.3 (или v6.4) с реально прогнанной проверкой; сборка базы из одного источника (нет правок цен в коде);
точечные правки фактов внесены; обогащение идёт через понятную человеку очередь одобрения со ссылкой на
источник; бренды чистые, устаревших цен нет; критичные темы — общий артефакт с главным диалогом.
