**Фактически каноничная всё ещё v6.3, не v6.4.** Проверка:

- На диске (текущая ветка `codex/d1-humanity-layer-20260525`) самый свежий релиз —
  `kb_release_20260520_v6_3_team_answers*`. **Папки релиза v6.4 нет** ни в рабочем дереве,

- Код бота: `DEFAULT_KB_SNAPSHOT_PATH` (`dialogue_contract_pipeline.py:37-39`) указывает на
  **v6.3-снимок** `kb_release_20260520_v6_3_team_answers/kb_release_v3_snapshot.json`. Бот
  использует v6.3.
- v6.4 существует как **невлитая git-ветка** `codex/kb-v6_4-20260529` — 4 коммита поверх HEAD:
  post-payment refund facts, scoped UNPK online price, unify duplicate keys, expose client-safe
  facts. Это реализация моего пакета v6.4 из прошлой серии.

Два расхождения, которые стоит знать:

1. Эти 4 коммита меняют только исходные YAML + сборщик + тесты; **машинный снимок
   `snapshot.json` в них не пересобран.** Значит даже после влития v6.4 нужно пересобрать
   релиз, иначе бот продолжит читать старый снимок.
2. Ветка v6.4 правит `scripts/build_kb_release_v3_from_claude_handoff.py`, а CLAUDE.md прямо
   пишет: «НЕ использовать старый…; использовать `build_kb_release_v6_1_team_answers.py`».
   Похоже на процессное расхождение — стоит свериться, каким сборщиком собирать v6.4.

Вывод: говорить «у нас v6.4» пока рано. v6.4 — в работе на ветке, не каноничная, не в боте.

## 3. Реальные узкие дыры (кандидаты, НЕ применять)

Каждый кандидат — черновик. Где это фактическое утверждение о механике, ставлю
`status: candidate_needs_owner_confirmation` — бот не должен выдавать это как факт без
подтверждения Дмитрия. Бренды раздельно. Высокий риск (возврат) — только маршрут к менеджеру.

### Дыра A. Результаты вступительного теста и группа (регулярные курсы)

Очень частый узкий вопрос (тема «результат/гарантии», 115): «тест прошли — вы сами видите
результат или прислать?», «когда узнаю, в какую группу?», «надо отправлять результаты?».
В KB есть placement для лагеря и общая логика подбора группы, но нет короткого ответа именно
на «прислать ли результаты теста по регулярному курсу».

Кандидат (оба бренда, безопасная формулировка-маршрут, без выдумки механики):

```yaml
# facts_for_bot_FOTON.yaml → presentation_format_facts_2026_05_21.client_facts
entrance_test_results_followup:
  status: candidate_needs_owner_confirmation
  condition_client_text: клиент спрашивает, прислать ли результаты вступительного теста / когда узнает группу (регулярный курс)
  client_safe_text: Спасибо! Дополнительно присылать результаты не нужно — по итогам тестирования куратор подберёт подходящую по уровню группу и сообщит. Если нужно, группу потом можно поменять.
```

```yaml
# facts_for_bot_UNPK.yaml → tg_unpk_verified_2026_05_21.client_facts
entrance_test_results_followup:
  status: candidate_needs_owner_confirmation
  condition_client_text: клиент спрашивает про результаты вступительного теста / распределение в группу (регулярный курс УНПК)
  client_safe_text: Спасибо! Результаты теста отдельно присылать не нужно — по ним куратор подберёт группу по уровню и сообщит. Перевод между группами возможен позже.
```

Подтвердить у Дмитрия: правда ли центр видит результат на своей стороне (если нет — заменить
на чистый handoff «куратор сориентирует»).

### Дыра B. «Оплатить пробно несколько занятий» (платное пробное)

Повторяется: «могу оплатить сначала пробно 4 занятия?», «пробное можно?». В KB есть только
бесплатное пробное (Фотон — через менеджера; УНПК — фрагмент онлайн). Платного блока «N занятий»
как подтверждённого продукта нет → безопасный маршрут к менеджеру.

```yaml
# оба бренда (FOTON и UNPK), client_facts
paid_trial_block_request:
  status: candidate_needs_owner_confirmation
  condition_client_text: клиент просит оплатить пробно несколько занятий (платное мини-пробное)
  client_safe_text: По пробному формату и оплате за несколько занятий вперёд сориентирует куратор — подберём удобный вариант под ваш курс и предмет.
  bot_route: draft_for_manager
```

### Дыра C. «Есть ли отбор/конкурс детей?»

Вопрос про отсев при зачислении. В KB подразумевается, что тест — для подбора группы, а не
отсева, но прямого client_safe ответа нет.

```yaml
# оба бренда, client_facts
admission_selection_clarify:
  status: candidate_needs_owner_confirmation
  condition_client_text: клиент спрашивает, есть ли отбор/конкурс для зачисления
  client_safe_text: Отбора или конкурса для зачисления нет. Вступительное тестирование нужно не для отсева, а чтобы подобрать подходящую по уровню группу.
```

Подтвердить: верно ли для олимпиадных групп (там может быть уровневое условие).

### Дыра D. Механика заявления на возврат (ВЫСОКИЙ риск)

«Где написать заявление на возврат, можно онлайн?», «как вернуть деньги за оставшиеся занятия?».
По CLAUDE.md реальная претензия на возврат после оплаты = `manager_only`. Ветка v6.4 уже
добавляет post-payment refund facts — **рекомендация:** туда же завести безопасную строку про
подачу заявления, без автономной обработки.

```yaml
# оба бренда — внутри будущих refund-фактов v6.4 (manager_only)
refund_application_intake:
  status: candidate_for_v6_4_refund_block
  bot_route: manager_only
  condition_client_text: клиент спрашивает, где и как подать заявление на возврат (в т.ч. онлайн)
  client_safe_text: Подскажу по заявлению на возврат — передам менеджеру, он пришлёт форму и порядок оформления.
  bot_must_not_say_note_internal: Не обещать сроки/суммы возврата автономно; реальная заявка — только через менеджера.
```

## 4. Как встраивать (для Codex, после «ок» Дмитрия)

1. Править ТОЛЬКО исходные YAML в `kb_release_20260520_v6_3_team_answers_sources/facts/`
   (или их v6.4-версию на ветке) — не трогать собранные пакеты руками.
2. Снять `status: candidate_*` и согласовать формулировки с Дмитрием по каждому факту.
3. Собрать релиз **каноничным** сборщиком `scripts/build_kb_release_v6_1_team_answers.py`
   (НЕ старым `..._v3_from_claude_handoff.py`) — заодно закрыть расхождение из §1.2.
4. Пересобрать снимок и обновить `DEFAULT_KB_SNAPSHOT_PATH`, иначе бот не увидит правок.
5. Прогнать semantic gate: `scripts/run_kb_semantic_review.py`.
6. Каждая правка — с условием применения; бренды не смешивать; высокий риск (возврат) —
   только маршрут к менеджеру.
