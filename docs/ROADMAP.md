# Roadmap

Дата обновления: 2026-05-15

Назначение: короткая рабочая дорожная карта. Подробная историческая версия находится в `docs/PROJECT_HISTORY_AND_PRODUCT_ROADMAP_2026-05-13.md`.

## Цель проекта

Сначала создать внутреннего ИИ-сотрудника для отдела продаж и клиентских коммуникаций. Потом, после подтвержденной пользы внутри компании, двигаться к продукту для других организаций.

## Целевое состояние

Система должна:

- видеть историю клиента по звонкам, письмам, AMO, Tallanto и мессенджерам;
- понимать конкретную сделку AMO;
- показывать менеджеру краткую полезную картину;
- предлагать следующий шаг;
- помогать писать ответы клиентам;
- не писать опасные данные без проверки;
- давать РОПу понятные правила и контроль качества;
- в будущем работать через сайт, Telegram, Max, CRM и рабочее место менеджера.

## Этап 0. Порядок в работе

Статус: выполнено для текущего блока.

Нужно:

- зафиксировать git-границы;
- создать карту текущих направлений разработки;
- не запускать параллельно новые крупные блоки;
- держать актуальную правду в документах.

Ключевой файл:

`docs/CURRENT_DEVELOPMENT_BOUNDARIES_2026-05-15.md`

## Этап 1. Безопасная запись в AMO

Статус: реализовано в коде без live-запуска. Нужны audit pack, коммит и только потом переход к PBF.

Нужно:

- pre-write snapshot;
- rollback;
- retry/backoff;
- resume;
- readback;
- fake-тесты;
- микропилот 1-5 сделок только после готового контура.

Ключевое ТЗ:

`docs/TOP3_PRIORITY_FIXES_TZ_2026-05-15.md`

## Этап 2. Исправить post-backfill историю

Статус: выполнено в коде, нужен отдельный коммит PBF.

Нужно:

- закрыть красный тест `test_contact_summary_does_not_embed_full_latest_summary`;
- не потерять полезную историю по нескольким касаниям;
- не дублировать длинные пересказы в CRM-истории.

## Этап 3. Сделать AMO payload полезнее для менеджера

Статус: следующий блок.

Нужно:

- бюджет диапазоном;
- комментарий к бюджету;
- чувствительность к цене;
- интерес к скидке;
- защита от выдумывания бюджета и обещаний скидки.

## Этап 4. Структурировать возражения

Нужно:

- сохранить человекочитаемое поле для менеджера;
- добавить внутренний структурный список;
- не писать JSON в AMO до появления реального потребителя.

## Этап 5. Связать каталог вопросов и deal-aware

Нужно:

- сохранять явные source-ключи;
- построить индекс `call_id -> тема/правило`;
- блокировать конфликтные случаи: сервисная тема и продажный дожим, manager-only тема и автоматическое действие.

## Этап 6. Встроить customer timeline

Нужно:

- подтвердить покрытие по телефонам и клиентам;
- сделать read-only context provider;
- подключить сначала preview;
- затем перевести основное чтение истории на timeline с fallback.

## Этап 7. Первый внутренний запуск

Нужно:

- выбрать 1-2 менеджеров;
- дать им понятный рабочий процесс;
- не автоматизировать отправку без человека;
- замерять ежедневное использование и качество подсказок.

Главная метрика:

сколько менеджеров реально пользуются системой каждый день.

## Этап 8. Расширение внутри компании

Нужно:

- подключить больше менеджеров;
- добавить форму сайта;
- добавить Telegram/Max;
- подключить почтовые черновики;
- сделать рабочее место менеджера.

## Этап 9. Переход к продукту для других организаций

Условие перехода:

- система стабильно используется внутри компании;
- есть метрики пользы;
- есть понятные процессы внедрения;
- есть защита данных и права доступа;
- есть повторяемая установка для другой организации.
