# Контракт изоляции брендов и будущих организаций

## 1. Зачем это нужно

Сейчас Фотон и УНПК МФТИ - два бренда внутри одного проекта. Архитектурно это уже похоже на будущую SaaS-задачу: разные организации или направления не должны смешивать факты, ответы, логи, токены и правила.

Текущая ошибка смешения Фотон/УНПК должна рассматриваться как прототип будущей ошибки “данные одного клиента SaaS попали другому клиенту”.

## 2. Термины

### `tenant_id`

Организация или контур владения данными.

Примеры:

- `foton_group_internal`
- `external_school_001`

### `brand_id`

Бренд или учебный центр внутри tenant.

Примеры:

- `foton`
- `unpk`

### `channel_id`

Канал общения.

Примеры:

- `telegram_bot_foton`
- `telegram_bot_unpk`
- `site_chat_foton`
- `email_edu_kmipt`

## 3. Базовое правило

Каждое входящее сообщение должно получить:

```text
tenant_id
brand_id
channel_id
knowledge_release_id
policy_version
```

Если `brand_id` не определён надёжно, автономный ответ запрещён.

## 4. Что должно быть раздельным

Для каждого бренда/tenant:

- токены Telegram;
- env-файлы;
- база знаний;
- bot pack;
- gold-ответы;
- правила автономности;
- post-filter phrases;
- логи;
- метрики;
- customer timeline context;
- CRM/Tallanto credentials;
- manager allow-list;
- service runbook;
- backup path.

## 5. Что может быть общим

Общим может быть:

- код ядра;
- формат `ChannelMessage`;
- формат `DecisionEnvelope`;
- общий PolicyEngine;
- общий PostFilter;
- общий storage schema;
- общий dashboard код;
- общая система тестов.

Но общие компоненты всегда должны получать `tenant_id` и `brand_id`.

## 6. Правила для Фотон/УНПК сейчас

### Фотон

Бот Фотона может использовать только:

- факты `brand=foton`;
- общие факты, разрешённые для обоих брендов;
- gold-правила Фотона;
- адреса Фотона;
- цены Фотона;
- скидки Фотона;
- правила рассрочки Фотона.

Не может:

- называть условия УНПК;
- использовать адреса УНПК как свои;
- обещать УНПК-скидки;
- сравнивать себя с УНПК без специального безопасного правила.

### УНПК МФТИ

Бот УНПК может использовать только:

- факты `brand=unpk`;
- общие факты, разрешённые для обоих брендов;
- gold-правила УНПК;
- адреса УНПК;
- цены УНПК;
- скидки УНПК;
- правила оплаты УНПК.

Не может:

- обещать Т-Банк или Долями, если это правило Фотона;
- использовать Верхнюю Красносельскую как адрес регулярных занятий УНПК;
- называть цены Фотона;
- смешивать ЛВШ Фотона и ЛВШ УНПК.

## 7. Контракт данных

Каждый факт базы знаний должен иметь:

- `fact_id`
- `tenant_id`
- `brand_scope`
- `client_safe`
- `freshness_status`
- `source_ref`
- `source_sha256`
- `valid_from`
- `valid_until`
- `allowed_routes`
- `semantic_pass`

Каждый ответ должен сохранять:

- `used_fact_ids`
- `used_gold_rule_ids`
- `active_brand`
- `cross_brand_blocked`
- `brand_policy_version`

## 8. Тесты изоляции

Обязательные тесты:

1. Фотон-вопрос не получает УНПК-факты.
2. УНПК-вопрос не получает Фотон-факты.
3. Общий факт используется только если он явно общедоступный.
4. Смешанный вопрос по двум брендам не получает автономный ответ.
5. Ответ с чужим брендом блокируется post-filter.
6. Gold-ответы не копируются между брендами.
7. Tenant A не может прочитать логи tenant B.
8. Tenant A не может использовать KB tenant B.
9. Tenant config не содержит секретов.
10. Audit pack не содержит токены и персональные данные.

## 9. Будущий SaaS-контракт

Этот раздел не является текущим заданием на реализацию. Его нельзя начинать до Фазы 8 из `IMPLEMENTATION_PHASES.md`, то есть до первого контролируемого client-hosted pilot.

Для внешнего клиента:

```text
tenant_root/
  config/
    tenant.json
    brands/foton_like_brand.json
  secrets/
    не в git, не в audit pack
  db/
    product.sqlite
    channel.sqlite
  knowledge/
    releases/
  logs/
  reports/
  backups/
```

На этапе client-hosted можно использовать отдельный root на клиента. Это проще и безопаснее, чем сразу строить общий облачный multi-tenant.

## 10. Главный риск

Нельзя считать `tenant_id` настоящей защитой, пока:

- нет прав доступа;
- нет тестов на изоляцию файлов;
- нет запрета чтения чужого root;
- нет раздельных секретов;
- нет журналов доступа.

До этого `tenant_id` - это структура данных, а не безопасность.
