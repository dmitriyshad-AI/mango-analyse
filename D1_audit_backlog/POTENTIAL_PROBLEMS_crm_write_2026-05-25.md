## ПОТЕНЦИАЛЬНЫЕ ПРОБЛЕМЫ (фиксируем ВСЕ; severity)

### Привязка / «не тот клиент» (самое бизнес-рисковое)
- [P1] deal_attribution матчит звонок→сделку ТОЛЬКО по нормализованному телефону (deal_attribution.py:53,199);
  ФИО/ребёнок/предмет не участвуют. Общий семейный номер → звонок про одного ребёнка уйдёт в сделку другого.
  «confidence»=сумма наличия источников, не вероятность правильности; при одном кандидате почти всегда >порог 0.69 (:401-422).
- [P1] legacy `find_contact_by_phone` берёт `contacts[0]` без верификации (clients/amocrm.py:120-122) — заметка/PII
  может уйти не тому контакту при неоднозначном поиске. (Актуально только при включённом legacy-sync.)

### Бренд (нарушение правила «не смешивать»)
- [P1] Бренд Фотон/УНПК НЕ различается НИГДЕ в путях записи (sync_amocrm, deal_aware, deals.py): `active_brand`
  не задаётся, соответствие бренда сделки бренду звонка не проверяется. Текст другого бренда из звонка попадёт в карточку дословно.

### Текст в CRM (PII / маркер ИИ / отсутствие чистки)
- [P1] Перед записью НЕТ semantic-фильтра/`sanitize_answer`. В заметку legacy идёт телефон, ФИО менеджера, оффер;
  runtime `deal_summary`/AI-поля — сырой вывод модели (deal_text_builder). PII/служебное не чистится.
- [P2] Маркер «[AI]»/«AI-» попадает в карточку клиента — против правила нераскрытия природы бота.

### Гейты записи / доступ (defense-in-depth)
- [P1] Гейт live-записи стоит только в РОУТЕРЕ; сами `write_analysis_to_lead` (deals.py:1229) и
  `send_lead_custom_field_update` (amo_integration.py:989) внутреннего kill-switch НЕ имеют. Сегодня не-роутерных
  вызовов нет (проверено), но любой будущий скрипт/воркер, импортировав функцию, запишет в живой AMO. Нужен внутренний env-гейт.
- [P1] `require_api_key`: при пустом `AUTH_PRINCIPALS` отдаётся `DEFAULT_DEV_CONTEXT` без ключа (auth.py:184-186) —
  если API задеплоен без настроенных принципалов, writeback-роутер открыт (с подтверждением в payload).
- [P2] quality_gate: блок каталожных конфликтов (manager_only/платёжная/чувствительная тема) ТИХО отключается,
  если не передан `question_index` (deal_quality_gate.py:94,303-304) — «проход» = отсутствие hard-findings, без позитивной проверки смысла.

### Идемпотентность / откат / устойчивость
- [P1] legacy заметка/задача НЕидемпотентны, commit в конце батча: при частичном сбое повтор создаёт ДУБЛИ
  (нет дедупа по source_call_id).
- [P2] amo_rollback: изначально пустое поле штатно НЕ откатывается (manual_restore_required), а первичное заполнение —
  массовый случай; snapshot пишется append каждый прогон → повторный запуск делает old_value «грязным».
- [P2] PROTECTED_LEAD_FIELDS (телефон/ФИО/email) защищаются только в preflight; сам http-слой их не применяет —
  защита держится на том, что payload собирается из белого списка AI-полей (хрупко при расширении полей).
- [P2] Нет явного таймаута у openai/ollama в analyze/runtime (операционный риск зависания воркера).
