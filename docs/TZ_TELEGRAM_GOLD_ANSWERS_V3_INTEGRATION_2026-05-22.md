# ТЗ: интеграция GOLD_ANSWERS_v3 в базу знаний и диалоговое поведение Telegram-ботов

Дата: 2026-05-22

Источник:

`/Users/dmitrijfabarisov/Claude Projects/Foton/gold_answers_2026-05-21/GOLD_ANSWERS_v3_2026-05-21.docx`

Статус источника: Дмитрий передал как хорошие примеры ответов и правила, которые нужно применять.

## 1. Контекст

У нас уже есть два Telegram-бота:

- Фотон;
- УНПК МФТИ.

Текущая система уже неплохо закрывает безопасность:

- P0-темы уходят к менеджеру;
- бренды разделены;
- запрещены неподтверждённые цены, даты, скидки, сроки, места и обещания;
- бот не должен раскрывать, что он бот/ИИ;
- есть защита от prompt injection;
- есть контекст клиента из AMO/Tallanto/local history в read-only режиме;
- есть база знаний v6.3 и текущий сборщик `scripts/build_kb_release_v6_1_team_answers.py`.

Главная нерешённая проблема: бот всё ещё часто звучит безопасно, но недостаточно живо и продажно. Он может отвечать общо, уходить в “менеджер уточнит”, не держать инициативу и не всегда отвечать прямо на вопрос клиента.

`GOLD_ANSWERS_v3` закрывает именно этот пробел: документ задаёт эталон качества ответа по 10 ключевым темам для двух брендов.

## 2. Цель

Сделать так, чтобы Telegram-боты использовали `GOLD_ANSWERS_v3` как слой качества:

- отвечали теплее и человечнее;
- чаще отвечали автономно по зелёным темам;
- отвечали сначала на прямой вопрос;
- не уходили в “менеджер уточнит”, если факт уже есть;
- мягко вели клиента к следующему шагу;
- не смешивали Фотон и УНПК;
- не выдумывали факты ради полезности;
- не превращали gold-ответы в дословные шаблоны.

## 3. Важный принцип

Gold-ответы — это не готовые скрипты для копирования.

Они задают:

- тон;
- структуру ответа;
- границы того, что можно и нельзя говорить;
- безопасный следующий вопрос;
- плохие примеры;
- стандарт “что значит хороший ответ”.

Бот должен писать живо и ситуативно, а не повторять один и тот же gold-текст.

## 4. Что я могу сделать самостоятельно

### 4.1. Извлечь и структурировать gold-ответы

```yaml
status: approved_by_dmitry_for_dialog_quality
use_as: answer_quality_and_tone_reference_not_raw_fact_source

global_tone_rules:
  - ...

topics:
  pricing:
    foton:
      gold_answer_example: ...
      allowed_claims:
        - ...
      forbidden_claims:
        - ...
      if_fact_missing: ...
      next_question: ...
      bad_examples:
        - ...
```

### 4.2. Подключить gold-ответы к сборке KB

Обновить:

- `scripts/build_kb_release_v6_1_team_answers.py`;
- bot pack;
- employee pack;
- handoff pack.

Важно: gold-ответы не должны автоматически становиться клиентскими фактами. Они должны попадать в отдельный слой:

- `bot_gold_answers.json`;
- `gold_answer_rules.yaml`;
- раздел в employee pack “Эталоны хороших ответов”.

### 4.3. Синхронизировать факты из gold-документа с текущей KB

Для каждого утверждения из gold-документа нужно определить статус:

- `already_confirmed_in_kb`;
- `new_dmitry_approved_fact`;
- `conflicts_with_current_kb`;
- `style_only_not_fact`;
- `manager_only_or_needs_check`.

Особое внимание:

- Фотон-рассрочка: Дмитрий подтвердил 2026-05-22, что формулировка gold “6 или 12 месяцев” верна и должна заменить/уточнить текущую базовую формулировку “до 36 месяцев” для клиентского ответа.
- Фотон/УНПК цены: использовать только текущие цены, не будущие.
- Повышение цены: Дмитрий подтвердил 2026-05-22, что можно мягко говорить “цена скоро подрастёт” без точной даты; будущую цену и точную дату повышения не называть.
- Пробное занятие: проверить различие Фотон/УНПК.
- Расписание по выходным: можно говорить только в рамках подтверждённых фактов.
- Адреса: не смешивать регулярные занятия и летние школы.
- Лагеря: не обещать наличие мест и не выдумывать даты смен.
- Запись: Дмитрий подтвердил 2026-05-22, что правило по умолчанию — дистанционная запись. Если клиент очень хочет личную встречу, можно назначить встречу с сотрудником только по согласованию; бот не должен сам приглашать “приехать познакомиться”.

### 4.4. Обновить prompt builder

Файл:

`src/mango_mvp/channels/draft_prompt_builder.py`

Добавить правила:

- Используй gold-ответы как ориентир качества, не как дословный текст.
- В зелёных темах отвечай сам, если есть подтверждённый факт.
- Сначала ответь на прямой вопрос.
- После ответа задай максимум один следующий вопрос.
- Не завершай каждый ответ одинаковой фразой.
- Не злоупотребляй “менеджер уточнит”.
- Тон: тёплый, уверенный, живой, без канцелярита и без ощущения строгой организации.
- Продажа через помощь: не давить, а честно подбирать под ребёнка.

### 4.5. Обновить post-processing и guards

Файл:

`src/mango_mvp/channels/subscription_llm.py`

Добавить/усилить проверки:

- если gold-ответ допускает автономию, но нет client-safe факта — не называть конкретику;
- если ответ содержит будущую цену или дату повышения — блокировать/переводить к менеджеру;
- если ответ приглашает приехать знакомиться — флаг `offline_visit_invitation_blocked`;
- если ответ дословно копирует один и тот же gold-шаблон слишком часто — флаг `template_like_answer`;
- если ответ не отвечает на прямой вопрос — флаг `ignored_question`;
- если ответ содержит брендовые маркеры другого бренда — текущий cross-brand guard остаётся обязательным.

### 4.6. Улучшить составные вопросы

Добавить правило:

Если вопрос состоит из нескольких частей:

- безопасная справочная часть — ответить;
- часть без факта — дать безопасную рамку;
- P0/high-risk часть — передать менеджеру;
- если есть хотя бы одна P0-часть, route не может быть `bot_answer_self_for_pilot`.

Пример:

“Сколько стоит, есть ли скидка, и можно ли вернуть деньги если не подойдёт?”

Ожидаемо:

- цена/скидка — можно раскрыть в черновике, если есть facts;
- возврат — только менеджер;
- итоговый route: `manager_only` или `draft_for_manager`, не автономный ответ.

## 5. Где нужна помощь Дмитрия

На текущем этапе я могу сделать большую часть самостоятельно, но нужны финальные решения по смыслу:

1. Фотон-рассрочка: подтверждено Дмитрием 2026-05-22.
   - В клиентском ответе использовать “6 или 12 месяцев”.
   - Старую формулировку “до 36 месяцев” синхронизировать/убрать из клиентского слоя, если она конфликтует.

2. Повышение цены: подтверждено Дмитрием 2026-05-22.
   - Можно говорить “цена скоро подрастёт” без точной даты.
   - Не называть будущую цену и точную дату повышения.

3. Запись дистанционно: подтверждено Дмитрием 2026-05-22.
   - По умолчанию запись и оформление дистанционно.
   - Если клиент очень хочет встречу, можно назначить встречу с сотрудником по согласованию.
   - Бот сам не приглашает “приехать познакомиться”.

4. Пробное Фотон: подтверждено Дмитрием 2026-05-22.
   - Можно говорить: “Пробное занятие есть, оформляется дистанционно при записи”.
   - Слово “бесплатное” не использовать как обещание.
   - Лимит бесплатных пробных бот не озвучивает.
   - Формулировка: “пробное занятие есть по согласованию с менеджером при записи на очные курсы и по умолчанию на онлайн-курсах”.

5. Пробное УНПК: подтверждено Дмитрием 2026-05-22.
   - Бесплатного очного пробного сейчас нет.
   - Бот рассказывает про формат и преподавателей и помогает понять, подойдёт ли курс.
   - Про онлайн-пробную неделю УНПК говорить можно, но без жёсткого “бесплатно”.

6. Адреса: подтверждено Дмитрием 2026-05-22.
   - Фотон: Верхняя Красносельская, 30 — основной адрес в Москве и для регулярных очных занятий, и для летней школы.
   - УНПК: Верхняя Красносельская, 30 — только в контексте летней школы/городского лагеря.
   - УНПК регулярные курсы в Москве — Сретенка, 20.

7. Лагеря: подтверждено Дмитрием 2026-05-22.
   - Фотон: можно говорить “есть выездная в Менделеево и городская в Москве, действует ранняя цена” без точных сумм и дат, пока клиент не уточнил смену.
   - УНПК ЛВШ Менделеево: можно и нужно назвать полную стоимость 120 000 ₽ и актуальные ранние цены 89 900 ₽ / 83 800 ₽, если вопрос про цену.
   - Бот никогда сам не говорит “места есть”; только “проверим наличие”, “подберём смену”, “закрепим заявку”.
   - По лагерям уточнять класс, а не возраст, потому что группы делятся по классам и уровню знаний.

Если Дмитрий не отвечает сразу, реализация должна:

- внедрить gold как стиль и структуру;
- спорные факты положить в `gold_fact_sync_report.csv`;
- не выпускать спорные факты в client-safe автономный ответ.

## 6. Acceptance criteria

### Formal pass

- Новый источник `gold_answers_v3.yaml` создан.
- Сборщик KB проходит.
- Bot pack содержит отдельный gold-layer.
- Employee pack содержит человекочитаемые эталоны.
- Тесты проходят.

### Semantic pass

- Gold-ответы не скопированы дословно как шаблоны.
- Бот отвечает теплее и конкретнее по 10 темам.
- Бот не называет неподтверждённые будущие цены, даты, наличие мест, сроки и банковские решения.
- Бот не смешивает Фотон и УНПК.
- Бот не приглашает “приехать познакомиться”.
- Бот не раскрывает, что он бот/ИИ.
- По составным вопросам P0-часть блокирует автономность.
- По безопасным частям составного вопроса бот всё равно даёт пользу.
- По известному контексту бот не спрашивает повторно уже известные данные.

### Pilot readiness

После реализации нельзя писать “готово к клиентскому потоку”.

Можно писать:

- “готово к внутреннему пилоту/проверке сотрудниками” — только после targeted smoke и semantic review;
- “готово к расширению автономности” — только после повторного v7 + holdout/живой проверки.

## 7. Что не делать

Без отдельного подтверждения Дмитрия:

- не запускать полный MEGA на 16 часов;
- не отправлять сообщения клиентам;
- не писать в AMO/Tallanto/CRM;
- не запускать ASR/R+A;
- не менять `stable_runtime`;
- не удалять старые артефакты;
- не импортировать спорные факты из gold-документа как client-safe без синхронизации.

## 9. Итоговый ожидаемый результат

После выполнения этого ТЗ бот должен стать заметно ближе к живому менеджеру:

- меньше “нейронных” шаблонов;
- больше прямых ответов;
- больше инициативы;
- мягче продажный тон;
- лучше использование контекста клиента;
- меньше повторных вопросов;
- сохранение P0-безопасности и бренд-разделения.
