# ТЗ: локальное наполнение customer timeline по 100 группам телефонов/сделок

Дата: 2026-05-15

## Контекст

Предыдущий аудит показал: код чтения `customer_timeline` есть, но готовой локальной `customer_timeline.sqlite` нет. Поэтому флаги `timeline_preview_enabled` и `timeline_primary_read_enabled` включать нельзя.

Следующий безопасный шаг: создать локальную базу вне `stable_runtime`, наполнить ее только из уже готовых read-only артефактов и повторить аудит на той же выборке 100 групп телефонов/сделок.

## Что делаем

1. Используем прежнюю выборку:
   - `audits/_inbox/customer_timeline_100_phone_coverage_20260515_175607/selected_phones.csv`.
2. Читаем источники только как файлы:
   - `stable_runtime/deal_aware_stage709_all_batches_20260514_v1/all_709_candidates_for_amo_review.csv`;
   - `stable_runtime/sales_master_export_20260513_human_history_v8_normalized/master_calls_ru.csv`;
   - `stable_runtime/sales_master_export_20260513_human_history_v8_normalized/master_contacts_ru.csv`.
3. Пишем только новую локальную БД:
   - `product_data/customer_timeline/deal_aware_sample_20260515/customer_timeline.sqlite`.
4. Для каждой группы создаем:
   - одну сущность клиента;
   - связи по всем телефонам группы;
   - связь с AMO-сделкой;
   - Tallanto-связи только если они уже есть в готовом master contacts;
   - события звонков Mango;
   - событие AMO-сделки;
   - внутреннее событие deal-aware истории;
   - bot-context только для будущего ручного использования, не для автоответа.
5. Повторяем аудит:
   - покрытие по 100 группам;
   - единая связь клиента;
   - наличие событий;
   - наличие AMO/Mango/Tallanto источников;
   - ручные риски deal-aware;
   - решение по флагам.

## Что нельзя делать

- Не писать в AMO/CRM/Tallanto.
- Не запускать ASR.
- Не запускать Resolve+Analyze.
- Не менять `stable_runtime`.
- Не удалять файлы.
- Не считать зеленый аудит истории готовностью автономного бота.

## Критерии приемки

- Новая DB создана вне `stable_runtime`.
- Источники из `stable_runtime` только читаются.
- Тесты на локальный импорт проходят.
- Повторный аудит создает `coverage_summary.json`, `coverage_summary.md`, `timeline_coverage_report.csv`.
- Отчет явно разделяет:
  - найдено ли в timeline;
  - готово ли к preview;
  - требует ли ручной проверки;
  - можно ли включать флаги.

## Следующий шаг после этого блока

Если покрытие 100 групп стало высоким, но большая доля строк требует ручной проверки, не включать основной режим. Следующий шаг: расширить локальный импорт на все 709 строк и отдельно добавить более чистую контрольную выборку, чтобы проверить не только сложные случаи, но и обычных клиентов.
