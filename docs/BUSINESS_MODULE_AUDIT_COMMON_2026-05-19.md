## 3. Что реально полезно бизнесу уже сейчас

1. **Telegram-пилот в режиме черновиков.** Уже есть контур “сообщение клиента -> контекст -> черновик менеджеру -> безопасные маршруты”. После MEGA v1 два P0 закрыты, Claude подтвердил `PASS_WITH_NOTES` для внутреннего draft-only пилота.

2. **База знаний v3.3.** Она уже полезна как источник контекста для черновиков и сотрудников. Самое важное: появились смысловые проверки, бренд-разделение Фотон/УНПК и защита от неподтверждённых обещаний.

3. **Safety-слой для клиентских ответов.** Закрыты критичные классы: возврат без сбора ФИО/договора/телефона, юридическая угроза без сбора данных, жалоба без извинений от лица компании, УНПК-рассрочка, ЗВШ-лист ожидания, промокоды.

4. **Mango runtime после обновления 2026-05-16.** Звонковый слой актуализирован: 65 100 actionable-звонков, missing ASR = 0, missing Resolve+Analyze = 0, runtime gates зелёные.

5. **Customer Timeline.** Раньше выглядел как изолированный слой, но теперь ясно, что это будущая основа единой истории клиента. Удалять или списывать нельзя.

6. **Deal-aware + AMO safeguards.** Есть сухие проверки, writeback guards, snapshot/rollback концепция, readback, качество CRM-текста. Это фундамент для осторожного обновления CRM.

7. **Question catalog и правила РОПа.** Даже если старый standalone F1 не идеален, сам каталог полезен для аналитики вопросов, правил бота, FAQ и обучения менеджеров.

## 4. Что есть в коде, но не используется полноценно

1. **Customer Timeline.** Большой и ценный слой, но пока не включён как основной источник истории клиента. Нужны правила семейных телефонов, дублей учеников и Tallanto-семантики.

2. **Email-слой.** Почтовые данные и ответы менеджеров могут дать сильный материал для черновиков писем, но email-пилот пока не должен отвлекать от Telegram.

3. **Productization/SaaS-модули.** В `src/mango_mvp/productization/` много заделов под будущий сервис: tenant config, runtime, scheduler, capture, appliance, isolation. Это полезно, но сейчас не должно становиться главным фронтом.

4. **Question catalog -> deal-aware связь.** Есть отдельные блоки, но они ещё не стали единым рабочим контуром “вопрос клиента -> тема -> правило РОПа -> сделка -> безопасный следующий шаг”.

5. **AMO/Tallanto read-only контекст.** Есть сильные модули, но бизнес-ценность появится только при аккуратном выводе в рабочее место менеджера и Telegram/email-черновики.

## 5. Что выглядит мусором или устарело

Не удалять без отдельного подтверждения.

1. **Документы состояния частично устарели.** `CURRENT_STATE.md` и `ROADMAP.md` ещё не полностью отражают KB v3.3, MEGA v1, свежие semantic rules и текущий Telegram-статус.

2. **Старый каталог скриптов отстаёт.** В `scripts/` уже 222 файла, а старый каталог скриптов не покрывает новые Telegram/KB/deal-aware/customer timeline команды.

3. **Много runtime-версий.** В `product_data/knowledge_base` и `stable_runtime` много волн сборок. Это нормально для активной разработки, но нужна карта актуальности, иначе легко взять не тот слой.

4. **Технические артефакты в `src`.** Видны `.DS_Store` и `__pycache__` внутри `src/mango_mvp`. Это не P0, но гигиенически плохо.

5. **MEGA v1 как тестовый контракт устарел сразу после первого прогона.** Он полезен как стресс-тест, но содержит хрупкие/противоречивые ожидания. Нужен MEGA v2.

## 6. Проблемы P0/P1/P2/P3

### P0

1. **Широкий запуск бота без MEGA v2 и ручного внутреннего пилота опасен.** MEGA v1 нашёл реальные P0, они закрыты, но сам набор требует исправления и полного повторного прогона.

2. **Автоответ клиенту пока нельзя включать.** Текущий безопасный режим: только черновики менеджеру, без автоотправки.

3. **Live-запись в AMO/Tallanto нельзя расширять без отдельного подтверждения и проверенного snapshot/rollback/readback.**

### P1

1. **Актуальная правда проекта размазана между свежими audit packs и устаревшими state-документами.** Это прямо мешает управляемой работе.

2. **Telegram-пилот требует MEGA v2.** Нужно привести route names, safety flags, expected/forbidden к реальному контракту и прогнать 251 тест заново.

3. **Нужно закрыть рабочую git-группу Telegram/KB/MEGA.** Сейчас в git есть незакоммиченные изменения по `subscription_llm.py`, `run_telegram_stage6_kb_eval.py`, тестам и новым MEGA/KB-папкам.

4. **Customer Timeline ещё не даёт менеджеру стабильную “единую историю”.** Технически работает, но нужны правила семейных телефонов, дублей и Tallanto-статусов.

### P2

1. **Email-черновики не должны стартовать с нуля.** Их нужно строить на тех же safety rules, базе знаний и истории клиента, что Telegram.

2. **Question catalog нужно переосмыслить как контекстную аналитику, а не только standalone-классификатор.**

3. **SaaS-заделы надо сохранить, но не делать главным фокусом до внутреннего использования.**

4. **Нужна карта актуальных runtime/product_data артефактов.**

### P3

1. Обновить каталог скриптов.
2. Пометить опасные команды.
3. Убрать `.DS_Store`/`__pycache__` только после отдельного подтверждения.
4. Разделить старые документы на “актуально / история / устарело”.

## 7. Какие тесты есть и каких не хватает

Есть:

- 1532 теста собираются через `pytest --collect-only`.
- Есть тесты по AMO writeback guards, readback, rollback, Telegram polling, Telegram manager inbox, Stage6 KB eval, subscription LLM, bot policy, tenant config, customer timeline, transcript quality.
- Есть смысловые regression-тесты по P0 Telegram/KB.
- Есть MEGA v1 на 251 сценарий и внешний Claude review.

Не хватает:

1. **MEGA v2 как чистый контракт.** V1 содержит артефакты и не может быть финальным gate.

2. **End-to-end тест “реальное сообщение -> контекст -> черновик -> manager inbox”.** Сейчас много частей покрыто отдельно, но нужен небольшой полный сценарий.

3. **Email draft safety tests.** До email-пилота нужны такие же P0-guards, как в Telegram.

4. **Customer Timeline semantic tests.** Нужны проверки семейных телефонов, дублей учеников, multiple deals, no Tallanto vs broken matching.

5. **Knowledge base diff tests.** При новой версии базы знаний нужно явно показывать “было -> стало”: факты, разрешения, риски, брендовые изменения.

## 10. Какие вопросы нужно решить Дмитрию

1. **Когда считаем Telegram-пилот достаточно безопасным для Насти?** Моя рекомендация: после MEGA v2 full pass по P0 и `PASS_WITH_NOTES` или лучше по semantic review.

5. **Какой следующий “живой” канал после Telegram?** Моя рекомендация: email-черновики, но только после первого Telegram-пилота.

6. **Когда возвращаемся к Customer Timeline?** Моя рекомендация: сразу после MEGA v2 и первого пилотного запуска, потому что история клиента сильно повысит качество черновиков.

7. **Что считать успехом пилота?** Моя рекомендация: менеджер реально пользуется каждый день; 50%+ черновиков полезны; нет P0; скорость ответа выше.
