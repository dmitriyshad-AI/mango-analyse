## Раздел 1. Карта слоя «что у нас есть»

Слой состоит из четырёх подсистем плюс связка с Tallanto.

**Разбор звонков (analyze).** `src/mango_mvp/services/analyze.py` (~2370 строк) — единственная точка входа `AnalyzeService.run()`: батч по базе звонков, на каждый звонок собирается промпт из транскрипта → модель → нормализация → поле `analysis_json` (класс звонка, продукт, следующий шаг, возражения). Качество — пачка модулей в `src/mango_mvp/quality/`: `non_conversation.py` (сейфгарды «живой длинный разговор ≠ автоответчик»), `transcript_quality_*`, `crm_text_quality_detector.py`. Запуск — только по явному «да» Дмитрия (правило проекта).

**Профили клиентов (семьи).** Сборщик `scripts/build_tz16_profiles_v7.py` и `scripts/refresh_customer_profiles.py` поверх слоя `customer_timeline` (хранилище событий с происхождением). Профиль = агрегат слотов по клиенту, объединение по телефону; дети — отдельными слотами со склейкой по имени. Базы: старая `product_data/customer_profiles/tz12_working_batch3/` (v6) и новая `tz16_profiles_v7_20260612/` (v7).

**Выгрузки в AMO + генератор «Авто история общения».** Контур deal-aware: `src/mango_mvp/deal_aware/` (`deal_text_builder.py`, `deal_writeback.py`, `deal_quality_gate.py`, `stage1_snapshot.py`) собирает 12 AI-полей сделки, проводит через гейт качества, делает снимок/откат. Текст поля контакта «Авто история общения» собирает `scripts/write_amo_ready_contacts.py` (`_compose_auto_history`), запись в AMO — `amo_integration.py` (`send_contact_custom_field_update`, ~стр. 873). Формирование сводки лида — `src/mango_mvp/amocrm_runtime/deals.py`.

**Живая карточка (live_card).** `src/mango_mvp/amocrm_runtime/tallanto_context.py` — `build_live_tallanto_context`: платежи, баланс (остаток занятий), расписание ближайших занятий, места/набор, с TTL и происхождением на блок. Стоп-правила «нет карточки лучше неверной» (multi-телефон, бренд-мисматч, недоступность API).

**Связь с Tallanto.** `tallanto_api.py` (только чтение: get/list/search), `tallanto_matching.py`, `tallanto_deal_ranking.py`, `tallanto_context.py`, `phone_context.py`. Главная сущность Tallanto — ученик; семья выводится склейкой по телефону.

Точки входа в бой: цикл черновиков `scripts/run_amo_wappi_draft_loop.py` (Wappi → бот → примечание в сделку AMO). Большинство тяжёлых скриптов слоя — батчевые, запускаются вручную.

## Раздел 2. Как это работает (потоки данных)

**Цепочка звонка → карточка:**
аудио → (transcribe, две модели) → (resolve, слияние вариантов) → **analyze** (выжимка + класс + продукт) → `analysis_json` → каноническая база звонков `canonical_calls_master.db` → импорт в `customer_timeline` (событие с происхождением) → **сборка профиля** (агрегат по семье, склейка детей) → база `customer_profiles.sqlite`.

**Цепочка профиль → поле AMO:**
профиль/события → deal-aware `stage1_snapshot` (снимок сделок) → `deal_text_builder` (12 AI-полей + история звонков) → `deal_quality_gate` (гейт) → `write_amo_ready_contacts._compose_auto_history` (текст поля «Авто история общения») → `send_contact_custom_field_update` → поле контакта в AMO.

**Цепочка живой карточки (в момент диалога):**
телефон клиента → `phone_context` (матч в timeline) → `build_deal_dossier` (`deal_dossier.py`) → `build_live_tallanto_context` (живой запрос в Tallanto: баланс/платежи/расписание/места) → блок live_card → промпт бота.

**Границы.** Батчевый контур (звонки→профили→поле AMO) и живой контур (live_card в момент диалога) — разные пути; в первом данные «застывшие» (снимки мая), во втором — свежие из Tallanto. Поле «Авто история общения» пишется батчем (в AMO не обновлялось с 12.06 — это норма). Рядом есть ручное поле «История общения» — людское, не трогается.

## Раздел 3. Неэффективность кода (ранжировано по влиянию)

**Высокое влияние:**

1. **[ПРОВЕРЕНО] N+1 запросов к Tallanto в живой карточке.** `tallanto_api.py:529–543` — `classes_by_ids` дёргает отдельный HTTP-GET на каждый class_id. Плюс `build_contact_context` (`tallanto_api.py:554–574`) — 7 последовательных вызовов на контакт; поиск по телефону (`409–427`) перебирает 5 полей × ~8 вариантов = до 40 запросов без ранней остановки. При нескольких контактах легко упереться в лимит коннектора 60/мин и таймаут карточки.
2. **[ГИПОТЕЗА] N+1 в сводке лида.** `deals.py:940` — `fetch_lead` отдельным запросом в цикле по сделкам контакта; AMO поддерживает пакетную выборку, код её не использует.
3. **[ГИПОТЕЗА] Лишняя загрузка неиспользуемых данных Tallanto.** `opportunities`, `requests`, `course_relations` грузятся в полном объёме (до 40 записей), но в `live_card` не попадают — только в `compact_contexts`. Трафик и время впустую.

**Среднее влияние:**

4. **[ГИПОТЕЗА] Четыре реализации `normalize_phone`** с расходящимся поведением (`utils/phone.py`, `telegram_history.py`, `insights/phone_identity.py`, `productization/mail_archive.py`). Разные форматы ключа → тихие промахи матчинга телефона (а телефон — главный ключ сшивки семьи).
5. **[ГИПОТЕЗА] Нет индекса на `primary_phone`** в базе профилей (`crm_summary.py:169`) — поиск семьи по телефону = полный скан ~18К строк в Python.
6. **[ГИПОТЕЗА] Двойной вызов** `detect_non_conversation_signals` внутри `_analyze_text` (`analyze.py:2288–2295`).

**Низкое влияние (технический долг):**

7. **[ГИПОТЕЗА] Размножение мелких утилит**: `safe_text/_safe_text` — 25+ копий по слою; `load_json`/`write_sqlite`/`read_csv` — по 6–19 копий в `deal_aware/` и `quality/`.

### 4.2. Грязные выжимки (analyze)

«Болезнь v6»: длинный живой разговор сворачивался в `non_conversation` (из-за O1), продукт приписывался сервисным звонкам. **[ПРОВЕРЕНО по доку-регрейду + коду сейфгардов]** v7 лечит: переключение на полный промпт + `long_client_live_safeguard`/`edtech_live_safeguard` в `non_conversation.py`; регрейды батча-15 и хвоста-3439 подтвердили — рецидива нет, выдуманных продуктов 0.
**Что ещё НЕ покрыто:** 77 звонков `blacklist_77` остаются на v6 (57 из 77 ещё не перепрогнаны, ждут отмашки); хвост 3439 перепрогнан, но **не влит** в каноническую базу. Профили активных семей, чьи звонки в этих списках, строятся на старых грязных выжимках. **[ГИПОТЕЗА]** по охвату — точный список активных среди blacklist не сверялся.

### 4.3. Дубль-дети и склейка имён

**[ГИПОТЕЗА, дорожка 1]** Склейка детей идёт **только по имени** (`builder.py:593–596`); класс/предмет не учитываются. Следствия: безымянный слот `child_<hash>` (например хеш от «9») встречается в 747 профилях и никогда не сливается; крупные профили накапливают 13–19 детских слотов из-за нового `child_key` на каждый звонок; 3671 запись `child_name` содержит 2–7 слов (полное ФИО) — при нестандартном имени нормализация не срабатывает. То есть нормализация есть, но проходит мимо безымянных и ФИО-вариантов.

### 4.4. Косметика (задвоения слов)

Дорожки прямого шаблона, гарантированно рождающего «класс кл», не нашли. Артефакты задвоения, скорее всего, приходят из самих выжимок модели, а не из склейки строк. **[ГИПОТЕЗА]** — отдельный прицельный греп по живым полям не дал чистого подтверждения; оставляю как открытый пункт.
