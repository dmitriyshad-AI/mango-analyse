## Контекст

Проверялись таблицы:
- `product_data/question_catalog/customer_question_classes.xlsx`

Цель проверки: понять, можно ли отдавать пакет РОПу как первую рабочую версию и какие ошибки классификации нужно закрыть до этого.

## Что подтвердилось

1. Черновики ответов можно и нужно держать прямо в таблицах.
2. Колонка `Черновик шаблона ответа` добавлена в основные таблицы и перенесена ближе к названию класса вопроса.
3. РОПу удобнее проверять таблицу, когда рядом видны:
   - крупный класс вопроса;
   - подкласс;
   - класс вопроса;
   - черновик ответа;
   - реальные примеры;
   - статус ответа;
   - право бота отвечать самостоятельно.

## Что исправлено после аудита

1. Вопрос про стоимость летней выездной школы больше не уходит в документы.
   - Был риск: `письма / справки / подтверждающие документы`.
   - Должно быть: `стоимость / ... / летняя школа`.

2. Вопрос `Если деньги придут к вам позже, вы нам вернете эти пятьдесят семь тысяч?` уходит в возврат денег.
   - Был риск: общий класс оплаты.
   - Теперь: `изменение / отмена услуги / возврат денег и штрафы`.

3. Технический признак из звонка больше не перебивает явный текст вопроса.
   - Сначала анализируется сама фраза клиента.
   - Только если по тексту ничего не понятно, используется fallback-сигнал из звонка.

4. Черновики ответа стали не техническими строками, а более полезными заготовками.
   - Для цены, расписания, адреса, формата, программы, записи, документов, доступа и обратной связи теперь есть отдельные каркасы ответа.
   - Для вопросов, где нужен менеджер, шаблон прямо запрещает боту обещать условия или индивидуальное решение.

5. Добавлен CSV полного пакета для РОПа.
   - `product_data/question_catalog/rop_question_review_pack.csv`
   - Это нужно для Claude-аудита и автоматических проверок.

6. Усилен фильтр служебного шума.
   - Примеры вроде `возражение/ограничение: время`, пустых звонков и маркетинговых/служебных фраз не должны становиться клиентскими вопросами.

7. После дополнительного Claude-аудита добавлен отдельный слой читаемых примеров для РОПа.
   - В таблицах теперь есть `Примеры для РОПа` / `Примеры вопросов для РОПа`.
   - Отдельно сохранены `Безопасные примеры` для автоматических проверок.
   - Это позволяет РОПу читать более живые фразы, а системе - не терять обезличенную безопасную версию.

8. Усилен фильтр почтовых тем и пересылок.
   - Subject-only строки вида `Re: ...` не должны попадать как вопросы клиента.
   - Уведомления о записи, расписании, закрытии группы, переносах и служебные блоки пересылки фильтруются жестче.
   - В текущем top100 не осталось прямых `Re:`, `пересылаемое сообщение`, `отправлено из Mail`, `вы записаны онлайн`, `уведомление о закрытии`.

9. Добавлены дополнительные подклассы.
   - Документы: материнский капитал, документы для лагеря, подтверждение оплаты, копии договоров/реквизиты.
   - Доступ: личный кабинет, ссылка не пришла, чат/группа, ошибка платформы.
   - Обратная связь: просьба перезвонить, удобное время, жалоба, прогресс/результат.
   - Общий вопрос: переезд/Москва, запись, решение семьи, проходная/вход.

## Что пока не идеально

1. Класс `общий вопрос` все еще слишком широкий.
   - В нем остаются разные ситуации: пустые звонки, общие претензии, просьбы созвониться, неполный контекст.
   - Его нельзя отдавать боту как единый сценарий ответа.

2. Есть крупные корзины, которые нужно дробить анализом примеров:
   - `общий вопрос`;
   - `обратная связь`;
   - `качество обучения / обратная связь`;
   - `доступ / технический вопрос`;
   - `программа курса`;
   - `формат обучения`;
   - `письма / справки / подтверждающие документы`;
   - `оплата / возврат / чек`;
   - `изменение / отмена услуги`;
   - `стоимость`;
   - `расписание`.

3. Часть классов содержит смешанные примеры.
   - Один пример может одновременно быть про цену, документы и формат.
   - Для таких строк нужен отдельный признак `много тем в одном вопросе`, а не насильственное попадание в один класс.

4. Часть FAQ-вопросов сейчас слишком строго помечена как `только менеджер`.
   - Например, стандартные вопросы про налоговый вычет или список документов можно будет разрешить боту после утверждения РОПом и проверки актуального файла фактов.

## Следующий проход

1. Сделать второй слой дробления для 10-15 самых больших классов.
2. Добавить признак `много тем в одном вопросе`.
3. Отдельно выделить стандартные FAQ, где бот сможет отвечать сам после проверки актуальных фактов.
4. Пересобрать полный пакет для РОПа так, чтобы широкие классы ушли вниз или были помечены как `нужно дробить`.
