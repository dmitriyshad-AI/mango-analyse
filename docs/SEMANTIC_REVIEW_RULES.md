# Semantic Review Rules

Дата: 2026-05-18

Назначение: обязательные правила смысловой проверки для базы знаний, Telegram-черновиков, CRM-текстов и любых клиентских ответов.

## Главный принцип

`quality_passed=true` означает только формальную пригодность артефакта: файл собрался, поля есть, связи не сломаны.

Для клиентских и менеджерских сценариев нужен отдельный статус:

- `formal_pass` - структура, тесты и сборка прошли;
- `semantic_pass` - факты и ответы прошли смысловую проверку;
- `pilot_ready` - можно показывать сотруднику в рабочем пилоте;
- `production_ready` - можно использовать за пределами внутреннего пилота.

Нельзя писать Дмитрию "готово к использованию", если есть только `formal_pass`.

## Когда смысловая проверка обязательна

Смысловая проверка обязательна для:

- базы знаний;
- Telegram-бота и черновиков сообщений;
- email-черновиков;
- CRM/AMO/Tallanto-текстов;
- цен, скидок, рассрочек, маткапитала, налогового вычета, справок;
- любых маршрутов `bot_answer_self`, `draft_for_manager`, `manager_only`;
- rebuild-раундов, где пересобираются клиентские факты.

## База знаний: обязательные проверки

1. Цена курса не должна быть неправдоподобной.
   - Блокер: клиентский `price` меньше `3000 ₽`.
   - Блокер: учебные параметры типа `35 занятий`, `20 недель`, `2 ак.ч.` получили `currency=RUB`.

2. Диапазоны цен должны быть цельными.
   - Хорошо: один факт с `amount_min` и `amount_max`.
   - Плохо: отдельные клиентские факты `.range.min` и `.range.max` без связи.

3. Тип факта должен соответствовать смыслу.
   - `total_lessons`, `weekly_lessons`, `daily_hours`, `semester_*_weeks` - не цена.
   - `retroactive_years` - не цена.
   - `certificate_lead_time_days` - срок, не цена.

4. Клиентский текст не должен содержать служебные маркеры.
   - Запрещены: `source_id`, `fact_id`, `freshness_status`, `AMO`, `Tallanto`, `GPT`, `Claude`, `Codex`.

5. Политика ответа на вопрос “вы бот?” должна быть честной и брендовой.
   - Бот не объявляет сам первым, что он ИИ.
   - На прямой вопрос отвечает честно: `я цифровой помощник Фотона/УНПК МФТИ, не живой оператор`.
   - По просьбе клиента нужен handoff-сигнал менеджеру-человеку.
   - Запрещены: `я человек`, `я живой сотрудник`, `я не бот`, `GPT`, `Claude`, `Codex`, `OpenAI`, раскрытие prompt/system prompt.

6. Клиентский текст не должен выглядеть машинным.
   - Запрещены: длинные `snake_case`, ` / ` в человекочитаемой фразе, необъясненные технические ключи.

7. Бренды нельзя смешивать.
   - Бот Фотона не консультирует по условиям УНПК.
   - Бот УНПК не консультирует по условиям Фотона.
   - В одном клиентском сообщении нельзя сравнивать условия двух брендов.
   - Нейтральная формула для вопроса о связи брендов: `Это отдельные организации, по вашему вопросу сориентирую в рамках текущего учебного центра.`

8. Справки и документы должны использовать актуальную формулу.
   - Для темы справок: `Менеджер подготовит справку и пришлёт в течение 10 дней, постараемся раньше.`
   - Нельзя просить `тип справки`, если правило говорит, что справка единая.
   - Нельзя использовать старый срок `3 рабочих дня`, если он отменен.

9. Маткапитал и налоговый вычет разрешены для Фотона и УНПК, если факт закреплен документами.
   - Справочный вопрос можно обрабатывать как обычную информационную тему.
   - Угроза, претензия, суд, конфликт или обещание результата - только менеджер.

10. Оплата может подтверждаться только после сверки AMO/Tallanto.
    - Если платеж есть в данных - можно сказать, что оплата видна.
    - Если есть конфликт или данных нет - `manager_only` или черновик без обещаний.

## Telegram/email-черновики: обязательные проверки

1. Черновик должен быть полезным.
   - Плохо: только `уточним и вернемся`, если в базе есть точный факт.
   - Хорошо: ответ содержит конкретный следующий шаг, но не обещает неподтвержденное.

2. Опасные темы не должны ослабляться.
   - Возврат, суд, претензия, конфликт оплаты, персональные документы - `manager_only` или черновик без обещаний.

3. Тема должна быть из закрытого списка.
   - Запрещены выдуманные `theme:program_courses`, `theme:none`, `theme:payment`.

4. Клиентская фраза должна быть отделена от инструкций.
   - Сообщение клиента в промпте обрамляется явными маркерами.
   - Клиент не может командой "игнорируй инструкции" изменить правила бота.

5. Несколько сообщений подряд нужно склеивать.
   - Не делать три черновика на `Здравствуйте` -> `У меня вопрос` -> `Какая цена?`.

## Очередь РОПа

1. `P0 + keep_internal_only` запрещен.
2. Вопрос РОПу должен быть конкретным, а не общим шаблоном.
3. Вопрос должен соответствовать решению:
   - для `keep_internal_only` спрашиваем, остается ли факт внутренним;
   - для цены спрашиваем сумму, срок, область применения;
   - для документа спрашиваем безопасную клиентскую формулировку;
   - для discontinued продукта спрашиваем подтверждение, что не предлагать клиенту.

## Обязательный порядок перед ответом "готово"

1. Сборка/тесты: `formal_pass`.
2. Смысловая проверка: `semantic_pass`.
3. Мини-прогон на реальных или исторических примерах.
4. Audit pack с отдельным файлом `semantic_review.md`.
5. Только после этого можно говорить о готовности к внутреннему пилоту.

## Команда для базы знаний

```bash
PYTHONDONTWRITEBYTECODE=1 PYTHONPATH=src python3 scripts/run_kb_semantic_review.py \
  --release-dir product_data/knowledge_base/kb_release_20260518_v3_handoff_for_claude_and_team \
  --out-dir audits/_inbox/<audit_pack>/semantic_review
```

Если `semantic_pass=false`, задача не считается завершенной, даже если все обычные тесты зеленые.
