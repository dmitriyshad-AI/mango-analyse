## ФАЗА 1 — Поведение бота: few-shot + answer-first + гейты (ядро)

Из ТЗ A Этапы 1-3.

1.1. **Few-shot reference selector** — выбор 2-3 примеров по бренду+теме+route из моего (уже свежего)
корпуса; правила инъекции — мой ТЗ §9 (только при подтверждённых `required_fact_ids`; bad→good — в
пост-фильтр; holdout не трогать).

1.2. **Диалоговая стратегия:** answer-first (первое предложение отвечает на вопрос) + no-reask-known
(не переспрашивать класс/предмет/формат/имя из контекста или CRM) + один мягкий следующий шаг +
no-fact-helpful шаблоны вместо пустого хендоффа.

1.3. **Пост-фильтр / семантические гейты:** P0, бренд-изоляция, неподтверждённая конкретика
(цена/дата/процент/место/гарантия), forbidden_phrases, known-data reask, template-детектор. При
нарушении — усилить route или перегенерировать.

1.4. **Замер «до»:** static P0/brand regression → targeted16 → tuning set. Зафиксировать baseline по
soft-флагам (ignored_question, templated_opening, over_handoff, reasked_known) и тону.

Выход фазы: бот отвечает прямо, помнит контекст, реже зря уходит к менеджеру; есть измеренный baseline.

## Чего НЕ делать

- Не строить playbook до пересинхрона корпуса и политики C.
- Не плодить параллельный источник формулировок (книги — каркас, тексты — из корпуса/звонков).
- Не вводить 8 субъективных метрик «впрок» без проверки стабильности.
- Не грузить PII-CSV в промпты; не тюнить по holdout; не объявлять успех по targeted16 в одиночку.
- Не ослаблять P0/бренд/анти-выдумку ради тона или продажности.
