# Bot Safety Threat Model

Дата: 2026-05-10
Статус: рабочая версия перед frozen adversarial corpus

## Цель

Сделать bot-safe слой пригодным для controlled allowlist и будущего Telegram-бота без бесконечного цикла “нашли литерал -> добавили regex -> нашли новый литерал”.

Критерий качества должен проверять классы риска, а не конкретные найденные слова.

## Scope

В scope:

- `bot_safe_answer`
- `customer_question_example`
- `when_not_to_use`
- bot export allowlist
- ROP bot drafts
- sanitizer для подготовки базы знаний бота

Вне scope текущего шага:

- сырые транскрипты
- manager/internal answers
- CRM writeback для менеджеров
- ROP/internal аналитика
- autonomous bot runtime

## Основные классы угроз

### P0: деньги и коммерческие условия

Что нельзя выпускать в bot-safe export:

- конкретные цены: `7900`, `88000`, `147000`, `50 000 рублей`
- скидки и проценты: `10%`, `пять процентов`
- персональные рассрочки и платежные планы
- обещания по возврату денег
- конкретные условия акции или ранней оплаты
- платежные провайдеры, если это выглядит как конкретная инструкция: Альфа, Сбер, Т-Банк, Яндекс Сплит

Ожидаемая замена:

- `[CURRENT_PRICE]` -> “актуальную стоимость”
- `[PAYMENT_OPTIONS]` -> “актуальные варианты”
- `[REFUND_POLICY]` -> “действующие правила изменения или отмены услуги”

### P0/P1: персональные данные

Что нельзя выпускать:

- ФИО клиента, ребенка, родителя
- телефоны, email, messenger handles
- одиночные русские имена в клиентском контексте
- фамилии и отчества после role/placeholders: `ученик Гамзяков`, `ученик Алексеевичу`
- фамилии педагогов: `преподаватель Лукина`, `будет вести Кондрашова`
- фамилии после слов `фамилия/фамилию/фамилии`

Ожидаемая замена:

- клиентские имена -> “ученик” или безопасная нейтральная формулировка
- телефоны/email -> “удобный контакт”
- педагогические ФИО -> роль без имени: “преподаватель”, “педагог”

### P1: адреса, кабинеты, филиалы, локальные объекты

Что нельзя выпускать в multi-tenant/SaaS-safe bot export:

- улицы, проспекты, переулки, метро
- конкретные кабинеты: `кабинет 324`
- кампусы/локальные сокращения: КПМ, ФТИ, корпус ФТИ
- конкретные филиалы, если они не подтверждены tenant config

Controlled single-tenant режим может быть мягче, но frozen corpus должен считать это риском для SaaS hardening.

Ожидаемая замена:

- `[CURRENT_LOCATION]` -> “адрес, который подтвердит менеджер”

### P1/P2: сроки и дедлайны

Что нельзя выпускать без актуализации:

- `до конца дня`, `до конца недели`, `до каникул`
- `до 15 числа`, `17 числа`
- календарные даты и интервалы
- точное время звонка/занятия
- “бронь держим”, “забронировали”

Ожидаемая замена:

- `[CURRENT_DEADLINE]` -> “актуальное окно записи”

### P2: promises / commitments

Что нельзя выпускать как обещание бота:

- “вернемся сегодня”
- “перезвоним до конца дня”
- “компенсируем занятие”
- “возместим”
- “проверим и напишем как только”

Ожидаемая замена:

- `[SERVICE_PROMISE]` -> “менеджер свяжется с вами после проверки”

### P2: документы и файлы

Что нельзя выпускать:

- названия конкретных файлов/документов
- `Разбивка 1`, `таблица Excel`, `файл Word ...`

Ожидаемая замена:

- `[CURRENT_DOCUMENT]` -> “материал, который пришлет менеджер”

### P3: tenant-specific brand artifacts

Что нужно контролировать для SaaS:

- Фотон/МФТИ/ФТИ/КПМ и ASR-искажения бренда
- локальные названия филиалов
- имена внутренних программ, если tenant-specific

Для текущей компании часть этого допустима в internal/ROP, но bot-safe export должен быть готов к tenant config.

### P3: over-sanitization cluster repeats

Что не является утечкой, но ухудшает качество ответа:

- повтор одной и той же безопасной замены 2-3 раза подряд: `адрес, который подтвердит менеджер, адрес, который подтвердит менеджер`
- соседние generic placeholders в одном предложении без полезного смысла
- ответ, который стал формально безопасным, но слишком общим для бота

Ожидаемая реакция:

- safety gate не должен считать это P0/P1 утечкой
- строка должна попадать в `over_sanitization_candidates` / ROP-review
- post-processing должен схлопывать соседние одинаковые placeholders до одного

## ASR-Tolerance Requirements

Для каждого критичного класса нужны варианты с ошибками распознавания:

- `Гамазяков` -> `Гамзяков`, `Гамозяков`, `Гамасяков`
- `Скорняжный` -> `Корняжный`, `Скоряжный`
- `МФТИ` -> `ФТИ`, `НФК`, `НПК МФТИ`
- `Альфа-банк` -> `Альфа`, `Алфа`, `Альфа банк`

Цель: fixture должен проверять не только правильные написания, но и ASR-corrupted варианты.

## Архитектурные правила

### Sanitizer

- Делает безопасную замену текста.
- Работает до fixed point.
- Если fixed point не достигнут за `MAX_SANITIZER_PASSES`, строка блокируется.
- Должен быть idempotent: `sanitize(sanitize(x)) == sanitize(x)`.

### Detector

- Должен быть независимым от sanitizer.
- Не должен импортировать regex-правила sanitizer.
- Его задача: найти остаточные риски после sanitizer.
- Detector нужен как защита от circular validation.

### Frozen Corpus

Должен состоять из трех слоев:

1. Synthetic adversarial cases: 300-1000 строк по threat classes.
2. Hand-curated cases: реальные находки Claude/GPT-аудитов, включая ASR-искажения.
3. Random real rows: 200 строк из реального allowlist/corpus с фиксированным `random.seed(42)`.

Corpus замораживается для release gate. Его нельзя расширять во время проверки конкретного релиза.

## Exit Criterion

Controlled allowlist можно включать, если выполнены все условия:

- frozen corpus проходит с `0 P0/P1`
- `0 fixpoint_not_reached` на release corpus
- `0 allowlist_idempotence_failures`
- Stage 15 hard gate green
- independent detector показывает `0 P0/P1`
- over-sanitization queue явно отделена от safety queue и не используется для autonomous bot без ROP review

Claude/GPT-аудит на свежем корпусе после этого используется как periodic monitoring, а не как бесконечный release blocker.
