# ТЗ (Codex/D8) — Пересчёт «следующего шага» по всем каналам

Дата: 2026-06-21. Канон `main`. Параллельный трек Этапа 2, независим. Правила: сначала думаем; хирургически; цель → критерий + NEG. **Read-only по данным; в боевую timeline/AMO/Tallanto не писать; работать на read_api / тестовой ленте; YAML не трогать.**

## 0. Зачем (наблюдение Дмитрия)

Сейчас «следующий шаг» в карточке берётся из последнего ЗВОНКА и часто уже выполнен — например, «отправить документы на почту», а по почте видно, что они отправлены. Нужен детерминированный пересчёт: **следующий шаг = из самого свежего события клиента по ВСЕМ каналам** (звонок + почта + мессенджеры + оплата Tallanto + примечание AMO), с закрытием шага, если более позднее событие его выполнило.

## 1. Что есть

- `read_api.customer_timeline()/list_events_by_customer` (read_api.py:222, store.py:1128) — события клиента `WHERE customer_id=?`, отсортированные по `event_at`, по всем `source_system`. Данные для пересчёта есть.
- Риск (из аудита Этапа 2): склейка двух клиентов по общему телефону на ИНГЕСТЕ (`infer_identity_conflicts`, ingestion.py:1129) — тогда чужое событие попадёт в ленту и закроет шаг.

## 2. Что строить

1. **Детерминированный резолвер `next_step(события_клиента_по_дате)`:** по самому свежему РЕЛЕВАНТНОМУ событию определить актуальный шаг. Тип события — из `event_type` (`MANGO_CALL`/`EMAIL_MESSAGE`/… есть в `contracts.py:81-85`) и `source_system`. Правила закрытия явно (не эвристикой): «звонок: отправить документы» + позднее «письмо/оплата: документы/чек/оплата прошли» → шаг закрыт/обновлён; источник+дата шага («от ДД.ММ.ГГГГ, канал»). ⚠️ **Список НЕ-закрывающих типов зафиксировать:** массовые кампании (`allowed_for_bot=False`, outbound_campaign), служебные уведомления, bounce — НЕ закрывают шаг (иначе рассылка ложно «выполнит» шаг).
2. **Строго per-customer:** только события ОДНОГО `customer_id`; не смешивать.
3. 🔴 **Читать открытые конфликты идентичности (аудит):** перед закрытием шага проверить `timeline_conflicts`/`readiness.open_conflicts` (есть в read_api: `customer_profile.readiness.open_conflicts`, `safe_for_automatic_bot`). Если по этому `customer_id` открыт `ambiguous_identity` (склейка по общему телефону) — НЕ закрывать шаг, отдать «уточнить у менеджера». Резолвер склейку не создаёт — она приходит сверху, поэтому единственная защита = уважать конфликт.
4. **Маркировка уверенности:** противоречивые события → «уточнить у менеджера», не угадывать.

## 3. Предохранители
Только события одного клиента; не закрывать шаг по событию, привязанному к этому клиенту ошибочно (см. NEG про общий телефон); read-only; детерминированно (одни и те же события → один и тот же шаг); не писать в боевую базу/AMO.

## 4. Приёмка + NEG

**Приёмка:** на синтетике «звонок: отправить документы» + позднее «письмо: документы отправлены» → шаг помечен закрытым/обновлён, а не висит; источник+дата шага проставлены; детерминизм (повтор → тот же результат). **NEG (обязательно):** ⚠️ **главный — склейка по общему телефону:** два клиента с общим телефоном НЕ должны закрывать шаг друг друга (проверять на склеенном `customer_id`, а не на разных id) — при открытом `ambiguous_identity` шаг НЕ закрывается; **массовая кампания/служебное письмо НЕ закрывает шаг**; противоречивые события → «уточнить у менеджера», не ложное закрытие.

## 5. Отчёт (мне на регрейд)
Логика правил закрытия; примеры на 10–15 клиентах (был шаг → стал, с источником/датой); результат NEG по общему телефону; подтверждение детерминизма.

## 6. Не входит
Запись пересчитанного шага в AMO (D7/боевой шаг); «мост»/ингест (D4); подключение к боту (Этап 3).
