# Mango Analyse Roadmap (2026-03-07)

## 1) Исходное ТЗ (сжатое)

- Массово обработать экспорт Mango (до 100k звонков).
- Получать расшифровки диалогов с ролями и таймингами.
- Делать структурный конспект звонка для CRM:
  - короткая история общения;
  - отдельные поля: ФИО, контакты, класс, интересы/формат/предметы, возражения, next step.
- Синхронизировать результат в amoCRM и использовать его для персональных офферов/приоритизации обзвона.
- Работать локально на MacBook (полуавтономно, с UI и стабильным runtime).

## 2) Что уже сделано

### 2.1 Транскрибация и надежность

- Ingest аудио с парсингом метаданных из имени файла Mango.
- Dual ASR (MLX Whisper + GigaAM), merge и экспорт:
  - `*_text.txt` (диалоговый формат с таймингами),
  - `*_variants.json` (варианты и метаданные merge).
- Stereo/mono fallback, защита от зеркальных каналов, фильтры кросстока.
- Retry/dead-letter/worker режимы.
- Semi-prod запуск в фоне + QC-отчеты по прогону.
- Stable runtime snapshot и отдельный launcher UI/CLI.

### 2.2 Качество диалога

- Добавлены фильтры артефактов/эхо-дублей между ролями.
- Улучшено восстановление `manager/client` из очищенного таймлайна.
- QC-аудит на 400 звонков: есть метрики риска и список проблемных файлов.

### 2.3 Анализ и схема данных

- Введена версия анализа `analysis_schema_version=v2`.
- Добавлены блоки:
  - `history_short`,
  - `crm_blocks` (people/contacts/student/interests/commercial/objections/next_step/lead_priority),
  - `evidence`,
  - `quality_flags`.
- Сохранена обратная совместимость со старыми ключами (`summary`, `interests`, `student_grade`, ...).

### 2.4 Выполнен "шаг 3.1"

- Добавлена миграция схемы анализа без перераспознавания:
  - CLI-команда: `mango-mvp migrate-analysis-schema ...`
  - dry-run, force, limit, target version.
- Поддержка миграции v1 payload -> v2 нормализованный payload.

## 3) Что еще предстоит (по приоритету)

### P0 (обязательно для боевого использования)

1. Финальная карта полей amoCRM:
   - сопоставить все `crm_blocks` с реальными `field_id`/ноутами/тасками.
2. Контур контроля качества анализа:
   - флаги low-confidence и ручная валидация части звонков.
3. Прод-режим sync:
   - dry-run -> staged rollout -> full rollout,
   - отчеты об ошибках по контактам/полям.

### P1 (масштаб 100k и операционка)

1. Планировщик batch-обработки (ночные окна, лимиты нагрузки, пауза/возобновление).
2. Dashboard по воронке:
   - конверсия по менеджерам,
   - топ возражений,
   - распределение интересов.
3. Автоматический reprocessing policy:
   - переобработка только проблемных звонков по quality flags.

### P2 (повышение качества и персонализация)

1. Fusion ASR -> LLM consolidation (2 версии -> 1 финальная) на выбранных кейсах.
2. Генерация персонального оффера на основе истории взаимодействий.
3. Версия `v3` схемы анализа (когда накопим репрезентативный массив).

## 4) Что я могу сделать самостоятельно (без вашего участия)

### Шаг A: Довести amoCRM-интеграцию до production-ready

1. Добавить расширенный mapping `crm_blocks -> custom fields`.
2. Добавить валидацию payload перед sync и понятные error-коды.
3. Добавить dry-run отчеты в JSON/CSV по каждой попытке sync.

### Шаг B: Контур мониторинга качества

1. Автоматически собирать weekly quality report по QC JSON.
2. Маркировать low-confidence звонки и складывать в отдельный recheck queue.
3. Добавить команду экспорта "списка на ручную проверку".

### Шаг C: Полубоевой throughput-контур

1. Добавить профили запуска (light/normal/aggressive) по нагрузке.
2. Ввести предохранители CPU/RAM/температуры (снижение параллелизма).
3. Добавить команду безопасного resume-run после sleep/reboot.

### Шаг D: Подготовка к v3 схемы

1. Вынести правила схемы в отдельный модуль миграций.
2. Добавить smoke-команду "проверить совместимость payload версий".
3. Добавить changelog схемы (`v1 -> v2 -> v3`).

## 5) Предлагаемая следующая итерация (конкретно)

1. Закрыть P0.1 + P0.3 (amoCRM mapping + staged sync) на тестовом сегменте 300-500 звонков.
2. Включить quality-gate: sync только для звонков без критичных quality flags.
3. Сделать управляемый nightly-run:
   - ingest/transcribe/analyze/sync,
   - финальный отчет по качеству и CRM-обновлениям.
