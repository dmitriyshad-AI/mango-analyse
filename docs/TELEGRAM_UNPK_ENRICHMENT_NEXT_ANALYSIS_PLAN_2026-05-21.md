# План дальнейшего анализа Telegram-истории УНПК МФТИ

Дата: 2026-05-21

Статус: план следующего read-only этапа. Базу знаний, AMO, Tallanto, CRM, Telegram и `stable_runtime` не менять без отдельного подтверждения.

## 1. Честный вывод

Папка `product_data/telegram_history_analysis/unpk_full_history_enrichment_20260521/` дала ценный, но не утвержденный материал:

- 2484 клиентских вопроса/кейса;
- 2321 кандидат на факт для базы знаний;
- 180 кандидатных фраз менеджеров;
- 200 новых тестовых сценариев;
- 706 P0/high-risk кейсов;
- 5576 кандидатов событий для customer timeline;
- 78 strong_unique, 50 probable, 19 ambiguous и 546 unmatched identity links.

Главная ценность не в том, чтобы “скормить переписку боту”. Главная ценность в том, чтобы превратить историю в:

- частотную карту реальных вопросов;
- список дыр в базе знаний;
- безопасные правила маршрутизации;
- хорошие проверяемые шаблоны менеджерских ответов;
- регрессионные тесты против опасных ответов;
- customer timeline candidates для будущего персонализированного контекста.

Менеджерские ответы нельзя считать истиной. В истории есть полезные формулировки, но также могут быть устаревшие даты, цены, наличие мест, скидки, индивидуальные обещания и лишний сбор персональных данных.

## 2. Что реально стоит использовать

### Использовать в первую очередь

1. `customer_question_candidates.csv`

Это главный источник реальных клиентских формулировок. Его нужно использовать для:

- частотной карты тем;
- кластеров вопросов;
- выявления пробелов базы знаний;
- создания тестов;
- понимания, где бот должен молчать и отдавать менеджеру.

2. `coverage_against_current_kb.csv`

Это стартовая карта: где база знаний покрывает вопрос, где покрывает частично, где факта нет, где нужен ручной маршрут.

3. `p0_route_cases.csv`

Это база safety-тестов. Возвраты, жалобы, юридические угрозы, оплата, договоры, справки, СФР, документы и персональные статусы должны становиться регрессионными проверками.

4. `manager_draft_phrase_candidates.csv`

Использовать не как факты, а как стиль и паттерны:

- как вежливо начинать ответ;
- как вести к следующему шагу;
- как задавать уточняющий вопрос;
- как не давить в продаже;
- как формулировать отказ или ограничение.

5. `conflicts_with_current_kb.csv`

Это очередь для владельца базы знаний и РОПа: где история противоречит текущей базе, динамическим данным или правилам безопасности.

6. `new_smoke_tests_candidates.jsonl`

Это кандидаты в тесты. Их надо почистить, сгруппировать, усилить expected behavior и добавить в тестовый набор только после ревью.

7. `identity_link_candidates.csv` и `customer_timeline_candidates.jsonl`

Использовать для будущей персонализации, но только read-only. Strong/probable/ambiguous/unmatched должны оставаться видимыми классами уверенности.

### Использовать осторожно

1. `kb_fact_candidates.csv`

Это не “новые факты”, а очередь на проверку. Особенно осторожно:

- цены;
- скидки;
- даты;
- наличие мест;
- документы;
- адреса;
- оплата;
- СФР/маткапитал;
- любые условия договора.

2. `attachment_text_extracts.jsonl` и `voice_transcripts.jsonl`

OCR/ASR полезны для поиска тем и кандидатов, но не для прямого утверждения фактов. Любой факт из вложения требует сверки с оригинальным документом и владельцем процесса.

### Не использовать напрямую

- raw Telegram export;
- персональные данные;
- ответы менеджеров как готовую истину;
- OCR/ASR как единственный источник факта;
- name-only identity matching;
- любые Telegram-derived ответы для автономной отправки клиенту.

## 3. Как извлечь максимум пользы

Нужен не один большой импорт, а воронка принятия решений.

### Этап 1. Тематическая карта и приоритеты

Взять `customer_question_candidates.csv` и построить укрупненные кластеры:

- расписание, даты, места, наличие мест;
- выбор курса и программы;
- цена, скидки, частичная оплата;
- оплата, чек, квитанция;
- договоры, заявления, справки;
- СФР/маткапитал;
- лагерь/выездная школа;
- формат онлайн/очно;
- доступы, ссылки, ЛК, ДЗ, чаты;
- жалобы, возвраты, негатив, юридические угрозы;
- брендовая путаница.

Для каждого кластера посчитать:

- сколько вопросов;
- какой риск;
- есть ли покрытие в текущей базе;
- какую выручку/конверсию тема может затрагивать;
- сколько ручной работы менеджеров может снять;
- можно ли в будущем дать автономный ответ или только черновик.

Результат этапа:

- `telegram_unpk_theme_priority_matrix.csv`;
- `telegram_unpk_top_50_question_clusters.md`.

### Этап 2. Очередь обогащения базы знаний

Из `kb_fact_candidates.csv`, `coverage_against_current_kb.csv` и `conflicts_with_current_kb.csv` собрать review queue:

- candidate_id;
- тема;
- текущий статус в базе;
- что именно не хватает;
- пример реального вопроса;
- кандидатный ответ;
- риск;
- нужен ли документ;
- нужен ли РОП;
- можно ли после утверждения отвечать клиенту;
- какой тест добавить.

Приоритеты:

- P0: безопасность и запреты: возврат, жалоба, суд, договор, оплата, СФР, персональные данные, брендовая путаница;
- P1: частые коммерческие вопросы: даты, места, цена, скидки, программа, формат, запись;
- P2: редкие формулировки, стилистика, улучшения шаблонов.

Результат этапа:

- `telegram_unpk_kb_enrichment_review_queue.csv`;
- `telegram_unpk_p0_policy_additions.md`;
- `telegram_unpk_fact_questions_for_rop.md`.

### Этап 3. Проверка менеджерских ответов

Нельзя автоматически брать “хороший ответ” из истории. Ответ считается эталонным только если проходит рубрику.

Рубрика оценки ответа менеджера:

- отвечает на конкретный вопрос клиента, а не рядом;
- факт подтвержден базой знаний, документом, AMO/Tallanto или РОПом;
- нет устаревшей даты, цены, скидки или наличия мест;
- нет юридического/финансового обещания без основания;
- не смешивает УНПК, Фотон и ЦДПО;
- не просит лишние персональные данные;
- ведет к следующему шагу;
- тон нормальный: вежливо, кратко, без давления;
- клиенту понятно, что делать дальше;
- можно переиспользовать как шаблон без персональных данных.

Классы ответов:

- `gold`: можно использовать как шаблон после проверки фактов;
- `silver`: хорошая логика, но нужно переписать;
- `bronze`: полезная идея, но клиентский текст не брать;
- `reject`: не использовать;
- `manager_only`: оставить только для ручного маршрута.

Результат этапа:

- `telegram_unpk_manager_answer_scorecard.csv`;
- `telegram_unpk_gold_phrase_candidates.md`;
- `telegram_unpk_rejected_answer_patterns.md`.

## 4. Как обогащать текущую базу знаний

Обогащение должно идти не из Telegram напрямую, а через утвержденные пачки.

### Правильная цепочка

1. Реальный вопрос клиента.
2. Проверка текущей базы знаний.
3. Если факта нет, формулируем вопрос РОПу/владельцу процесса.
4. Находим первичный источник: документ, оферта, сайт, CRM/Tallanto read-only, регламент.
5. РОП утверждает или отклоняет.
6. Только после этого создается обновление базы знаний.
7. Для каждого нового факта добавляется тест.
8. После обновления запускается смысловой аудит.

### Что добавлять в базу знаний

1. Факты:

- даты и форматы только с valid_from/valid_until или refresh rule;
- цены и скидки только с условиями и сроком действия;
- порядок оплаты и документов;
- правила справок, договоров, заявлений;
- СФР/маткапитал;
- программа и ограничения лагерей/смен;
- доступы, ЛК, чаты, ДЗ, записи занятий.

2. Правила:

- когда только менеджер;
- когда черновик менеджеру;
- когда после approval можно дать автономный ответ;
- как не смешивать бренды;
- как работать с вложениями;
- как не собирать лишние персональные данные.

3. Шаблоны:

- короткий ответ на частый вопрос;
- уточняющий вопрос;
- мягкий next step;
- безопасный отказ;
- ручная передача менеджеру.

4. Тесты:

- частые вопросы;
- P0/high-risk;
- brand isolation;
- payment/document status;
- missing facts;
- media/voice cases;
- identity ambiguity.

## 5. Как понять, что сотрудники отвечали грамотно

Нужна не субъективная похвала, а проверяемая оценка.

### Сигналы хорошего ответа

- клиент получил ясный ответ;
- есть следующий шаг: записаться, прислать данные, выбрать дату, оплатить, дождаться менеджера;
- нет лишних обещаний;
- нет опасной автономности;
- нет путаницы брендов;
- нет лишнего сбора данных;
- менеджер не спорит и не давит;
- если вопрос рискованный, менеджер не импровизирует, а переводит в ручной разбор.

### Сигналы плохого ответа

- ответил не на тот вопрос;
- дал цену/дату/скидку без источника;
- пообещал наличие мест;
- подтвердил оплату/договор без проверки;
- попросил слишком много персональных данных;
- смешал УНПК и Фотон;
- не предложил следующий шаг;
- ответил слишком поздно или слишком сухо;
- оставил клиента в неопределенности.

### Как делать оценку честно

1. Сначала убрать из оценки персональные данные.
2. Сгруппировать ответы по сценариям.
3. Сравнить каждый ответ с текущей базой и правилами.
4. Если есть подтвержденный исход клиента после ответа, учитывать результат, но не считать его единственным критерием.
5. Дать каждому ответу оценку по 10-балльной рубрике.
6. Эталоном считать только ответы, где:
   - факт подтвержден;
   - текст безопасен;
   - есть следующий шаг;
   - РОП согласен;
   - ответ можно обобщить без персональных данных.

## 8. Критерии готовности следующего этапа

Этап считается полезным, если:

- есть топ-кластеры вопросов, а не просто длинные CSV;
- каждый KB-кандидат имеет понятный следующий шаг: approve/reject/needs_source/manager_only/test_only;
- есть не менее 50 оцененных менеджерских ответов;
- есть не менее 30 кандидатов на факты/правила для РОПа;
- есть не менее 50 очищенных тестов;
- P0-темы не попали в автономность;
- нет прямого переноса фактов из Telegram в базу;
- есть список “что нельзя использовать”;
- есть понятный пакет для принятия решений, а не только данные.

## 9. Мой рекомендуемый порядок

1. Сначала сделать `theme priority matrix`, потому что она покажет, где больше всего пользы.
2. Потом собрать `ROP review queue` только по P0/P1.
3. Затем оценить менеджерские ответы по рубрике и выбрать gold/silver/reject.
4. После этого подготовить KB patch plan и тесты.
5. Только после утверждения Дмитрием/РОПом менять базу знаний.

## 10. Бизнесовая позиция

Не стоит пытаться автоматизировать весь Telegram сразу. Это создаст иллюзию прогресса и риск ошибок.

Стоит быстро забрать три типа пользы:

- безопасность: P0 gates и тесты;
- продажи: лучшие next-step формулировки и частые вопросы;
- база знаний: подтвержденные факты по самым частым темам.

Самый ценный первый результат для бизнеса: не новый “умный бот”, а надежный менеджерский черновик, который знает частые вопросы, не врет по динамическим фактам и вовремя отдает рискованные случаи человеку.
