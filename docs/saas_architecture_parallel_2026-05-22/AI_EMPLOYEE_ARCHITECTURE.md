# Целевая архитектура внутреннего ИИ-сотрудника

## 1. Назначение

ИИ-сотрудник - это не Telegram-бот сам по себе. Это общий контур, который принимает обращение клиента из любого канала, собирает контекст, проверяет правила, готовит ответ или передаёт вопрос менеджеру, сохраняет решение и учится на обратной связи.

Целевая цепочка:

```text
канал общения
-> адаптер канала
-> сессия диалога
-> сбор контекста
-> база знаний и gold-ответы
-> правила автономности
-> генерация ответа
-> смысловая и безопасностная проверка
-> автоответ / черновик / передача менеджеру
-> журнал действий
-> отзыв сотрудника
-> тест или правка базы знаний
```

## 2. Основные слои

### 2.1. Канал общения

Каналы:

- Telegram Bot API - текущий живой пилот.
- Будущий Telegram-аккаунт/номер - возможный канал для более естественного общения, но не текущий фокус.
- Сайт / web-chat - будущий вход для новых лидов.
- Email - будущий канал для писем и длинных переписок.
- CRM/AMO-чат - будущий канал внутри CRM.
- Max/WhatsApp - будущие адаптеры по той же схеме.

Канал не должен решать бизнес-логику. Его задача:

- принять сообщение;
- нормализовать его в общий формат;
- передать в ядро;
- получить решение;
- выполнить разрешённое действие.

### 2.2. Адаптер канала

Адаптер приводит внешнее событие к единому внутреннему объекту:

```text
ChannelMessage
ChannelSession
ChannelAttachment
ChannelRenderedReply
SendResult
```

Что уже есть:

- `src/mango_mvp/channels/telegram_adapter.py`
- `src/mango_mvp/channels/web_chat_adapter.py`
- `src/mango_mvp/channels/telegram_bot_polling.py`
- `src/mango_mvp/channels/telegram_business_runtime.py`
- `src/mango_mvp/channels/telegram_history.py`

Что важно сохранить:

- live-send выключен по умолчанию;
- raw payload не сохраняется в открытые отчёты;
- дубли сообщений отсекаются через устойчивый ключ;
- канал не знает, можно ли отвечать автономно.

### 2.3. Сессия диалога

Сессия объединяет несколько сообщений одного клиента в один контекст.

Нужно хранить:

- канал;
- бренд;
- thread/chat id;
- текущую историю сообщений;
- найденный телефон или другой идентификатор;
- состояние тестового режима;
- route последнего ответа;
- активные ограничения.

Текущее ограничение: по runbook история живёт в памяти процесса и очищается после перезапуска. Для пилота это допустимо, но для внутреннего продукта нужен SQLite-журнал диалогов.

### 2.4. Сбор контекста

Контекст должен собираться в один объект, а не в набор случайных строк.

Состав:

- текущее сообщение;
- последние сообщения;
- активный бренд;
- найденный клиент;
- локальная история звонков;
- AMO read-only;
- Tallanto read-only;
- customer timeline read-only;
- факты базы знаний;
- gold-ответы;
- правила РОПа;
- матрица автономности;
- флаги риска.

Что уже есть:

- `src/mango_mvp/channels/pilot_context.py`
- `src/mango_mvp/channels/telegram_pilot_context_builder.py`
- `src/mango_mvp/customer_timeline/context_provider.py`
- read-only CRM/Tallanto через `api.fotonai.online` в серверном режиме.

Ключевое правило: если клиент найден, бот не должен заново спрашивать имя, телефон, класс, бренд или статус, если эти данные надёжны и не конфликтуют.

### 2.5. Доступ к базе знаний

База знаний должна восприниматься как версионированный источник фактов и правил.

Для каждого ответа нужно знать:

- `knowledge_release_id`;
- путь к snapshot;
- `snapshot_sha256`;
- `semantic_pass`;
- `active_brand`;
- какие факты были использованы;
- какие факты были недоступны;
- какие gold-правила повлияли на тон/структуру.

Gold-ответы используются как эталон качества, а не как дословный скрипт.

### 2.6. Правила автономности

По умолчанию бот не отвечает автономно.

Автономный ответ возможен только если:

- бренд известен;
- тема входит в матрицу автономности;
- вопрос самостоятельный;
- нет P0/high-risk темы;
- есть проверенный клиентский факт;
- факт относится только к активному бренду;
- нет смешанного вопроса с опасной частью;
- post-filter не нашёл блокеров.

P0 всегда выше продажности и человечности:

- возврат денег;
- жалоба;
- конфликт;
- юридическая угроза;
- претензия;
- спорная оплата;
- рискованные персональные данные;
- брендовая неопределённость.

### 2.7. Генерация ответа

Answer Engine должен получать не “сырое сообщение”, а подготовленный безопасный пакет:

- клиентский текст в явных границах;
- бренд;
- разрешённые темы;
- подтверждённые факты;
- недостающие факты;
- запреты;
- стиль gold-ответов;
- последние сообщения;
- что уже известно о клиенте;
- route constraints.

Текущее состояние:

- `src/mango_mvp/channels/draft_prompt_builder.py`
- `src/mango_mvp/channels/subscription_llm.py`
- LLM через `codex exec`, без OpenAI API-key.

Архитектурное правило: Answer Engine предлагает ответ, но не принимает финальное решение об отправке.

### 2.8. Проверка после генерации

PostFilter должен проверять:

- смешение брендов;
- цены/скидки/сроки без факта;
- будущие цены и даты повышения;
- обещания возврата или решения;
- раскрытие AMO/Tallanto/CRM/source_id;
- слова “я бот”, “ИИ”, “нейросеть”, GPT/Claude/Codex;
- приглашение на очную встречу без согласования;
- повторный запрос уже известных данных;
- шаблонность и канцелярит;
- отсутствие прямого ответа на вопрос.

Если post-filter находит P0, route должен быть `manager_only` или безопасный черновик менеджеру, но не автономный ответ.

### 2.9. Отправка, черновик и передача менеджеру

Внутренний пилот допускает два режима:

- `bot_answer_self_for_pilot` - только по зелёным темам и проверенным фактам;
- `manager_only` / `draft_for_manager` - для всего рискованного, спорного и неполного.

Для будущих каналов нужны одинаковые действия:

- отправить клиенту;
- отправить менеджеру черновик;
- создать задачу менеджеру;
- попросить уточнение;
- заблокировать ответ;
- пометить для РОПа.

Ни один канал не должен обходить Policy Engine.

### 2.10. Журнал и feedback

Каждое решение должно сохраняться:

- что пришло;
- какой контекст был доступен;
- какие факты использованы;
- какой ответ сгенерирован;
- какие флаги сработали;
- какой route выбран;
- что отправлено клиенту или менеджеру;
- был ли отзыв сотрудника;
- исправленный вариант сотрудника;
- итог: помогло / не помогло / опасно / добавить факт / добавить тест.

Без этого невозможно перейти от “бот вроде отвечает” к управляемому продукту.

## 3. Что уже есть и чего не хватает

| Слой | Статус | Комментарий |
|---|---|---|
| Telegram Bot API | Есть | Два публичных пилотных бота запущены |
| Channel contracts | Есть | Нормализованные сообщения и сессии уже заложены |
| Web/CRM chat adapter | Частично есть | Есть read-only adapter, live-контур не нужен сейчас |
| Email adapter | Частично/будущее | Email обрабатывается отдельным направлением |
| Context builder | Есть частично | Нужна единая версия для всех каналов |
| Knowledge access | Есть частично | Нужен явный контракт версий KB/gold/facts |
| Policy engine | Есть частично | Матрица автономности есть, но нужна единая точка решения |
| Answer engine | Есть | Через `codex exec`, нужен общий интерфейс |
| PostFilter | Есть частично | Уже много guard-логики, нужен единый слой для всех каналов |
| Human handoff | Частично | Нужно сделать одинаковым для Telegram/email/site/CRM |
| Feedback store | Частично | Есть отчётность, но нужен постоянный журнал |
| Audit log | Частично | Audit packs есть, нужен runtime action log |

## 4. Главный архитектурный принцип

Система должна принимать решение так:

```text
не "Telegram решил ответить",
а "ядро разрешило каналy выполнить действие".
```

Это критично для будущего SaaS. Иначе каждый новый канал начнёт накапливать свои исключения, свои правила и свои риски.
