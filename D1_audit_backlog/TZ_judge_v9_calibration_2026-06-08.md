## Зачем (доказано по сырью)

1. Hard fabrication на уже-обезвреженные ходы ×2 (baseline ov_value_expensive T3, wave2 ov_comp_three T2): verifier finding + downgrade + draft_for_manager → судья всё равно hard.
2. judge_fail_unspecified при ясном rationale (wave2 ov_p0_payment): **первопричина найдена исполнением — `\bp0\b` не матчит «p0_mishandled»** (подчёркивание = словесный символ, граница не срабатывает); ни один паттерн JUDGE_GATE_TEXT_PATTERNS не матчит этот rationale.
3. Исторический «fabrication на реальный факт»: экстракция клеймов в fact_claim_audit — ручной список ~20 паттернов, всё мимо него судится только по фактам хода.
4. **Двойной источник правил**: judge_spec из набора жёстко кодирует «нет в подтверждённых фактах = fabrication» — противоречит новым правилам, модель получает оба текста (вероятный соисточник пере-классов).

## J1. Verifier-aware вердикт — ПОЛНАЯ матрица (решение привязано к ФИНАЛЬНОМУ маршруту)

Судье передаётся блок верификатора целиком: `checked / skipped / skip_reason / unavailable / findings (код+action+evidence)` + bot_route (route уже есть :1563 — судья обязан его ИСПОЛЬЗОВАТЬ).

| Производный клейм (НЕ число/цена/срок) | Автономный маршрут (bot_answer_self*) | Уведён (draft_for_manager / manager_only) |
|---|---|---|
| метка верификатора (finding по ЭТОМУ клейму) | **hard** (annotate/keep на self = текст дошёл) | soft `derived_claim_draft` |
| checked, чисто (верификатор «проспал») | **hard** | soft `derived_claim_draft` |
| skipped / unavailable | **hard** (скип — продакшен-дыра бота, клиент получил непроверенное) | soft `derived_claim_draft` |

Жёсткие инварианты поверх: выдуманное ЧИСЛО/цена/срок/расписание = **hard в любом маршруте, всегда** (менеджер может не заметить в черновике). Метка считается только если относится к ТОМУ ЖЕ клейму (по evidence/span) — одна метка не индульгенция всему ходу.
Динамика (не сюрприз): после фикса skip-гейта (волна 2.1 №6) класс ov_value_unpk станет checked→finding→downgrade→soft в будущих прогонах — это ожидаемо.
**Порог в приёмке смоука:** счётчик `derived_claim_draft` в сводке; бюджет — связать с downgrade-бюджетом верификатора (≤10% по checked); каждый случай — поимённо в очередь разбора.

## J2. Гейт-enum: три линии обороны

1. **Детерминированный фикс паттернов** (первая линия, закрывает наблюдённый кейс бесплатно): канонические имена гейтов литералами — `p0[_\s-]?mishandled`, `made[_\s-]?a[_\s-]?promise`, `brand[_\s-]?leak`, `internal[_\s-]?leak`, `revealed[_\s-]?ai`, `fabrication`.
2. Промпт: «verdict=FAIL без violated_gates — недопустимый ответ».
3. Re-ask (третья линия): только для completed-диалогов; промпт включает исходный JSON судьи и просит ТОЛЬКО заполнить violated_gates; verdict/rationale/soft_flags — из первого ответа; re-ask вернул PASS → игнорировать (остаётся FAIL + фолбэк-код). Infra-коды (timeout/infra_error, :1110) — не parse-issue.
Фолбэк-код ОСТАЁТСЯ в violated_gates (иначе hard_gates_passed развалится, :1747); сводка показывает его отдельной строкой `judge_parse_issues` (цель: 0).
