## Почему фазами (зависимость от Волны 1)

Волна 2 расширяет ПОНИМАНИЕ входа: модель выбирает тему / факты / слоты вместо keyword. Её безопасность опирается на ВЫХОДНОЙ слой (Волна 1: Б1 скоуп-осознанный числовой гейт + Б5 верификатор хэндофф-claims). Логика: чем шире модель выбирает вход, тем надёжнее выход должен ловить её ошибки.

**Разблокировка Фазы B:** замер Волны 1 показывает 0 новых выдумок / бренд-утечек / P0-нарушений при расширенном выходе → выход надёжен → можно шире доверять вход. Если выход течёт — Волна 2 сужается или ждёт. Поэтому финальный дизайн «насколько агрессивно» нельзя писать до замера Волны 1.

## Карта Б2 — тема и факты выбираются keyword ДО модели

- **Продюсер** `conversation_intent_plan.py`: `_primary_intent` (~301, regex/role-каскад) → `_topic_for_intent` (~539) / `_topic_switch_decision` (~515) → `_required_fact_keys` (~571) + `_fact_scope_constraints` (~176) → выдаёт `required_fact_keys` / `fact_scope` / `blocked_neighbor_scopes` (через `fact_scope_spec.blocked_neighbors_for`).
- **Потребитель** `telegram_pilot_context_builder.py`: применяет `fact_scope` / `blocked_neighbor_scopes` к РЕТРИВАЛУ — решает, какие факты модель вообще увидит.
- **Дефект:** промах keyword по теме → модель не получает нужный факт и не может это исправить (видит только то, что отобрал keyword).
- **Связь с P0 (граница!):** `_primary_intent` кормит не только факты, но и маршрут / P0-преблок. Значит Волна 2 трогает ТОЛЬКО выбор фактов/темы для наполнения, НЕ P0-маршрут.

## Карта Б4 — память диалога keyword + вето на модельный слот

- `_extract_slots_from_text` (~869): слоты (класс/предмет/формат/цель) — regex по реплике.
- Модельный апдейт памяти: `update_memory_llm` (~483) → `_apply_memory_llm_update` (~537) → `_memory_llm_can_override_slot` (~608) → `_memory_llm_slot_supported_by_latest_client` (~624).
- **Вето (624–668):** модельный слот принимается ТОЛЬКО если значение подтверждено keyword/алиасом в последней реплике клиента (grade: цифра/порядковое; subject: алиасы `информат/математ/физик/…`; format: маркеры `онлайн/очно`). Иначе апдейт отклоняется.
- **Дефект:** модель не может проставить слот на эллипсисе/перефразе, если regex не нашёл слово, — даже если поняла верно. Скрипт решает «что понять про клиента».

## Граница безопасности (НЕ отдавать модели — остаётся жёстким)

- Бренд-фильтр (`active_brand`) и client_safe-предикат выдачи фактов.
- P0-латч/преблок и маршрут до модели.
- Выходной гейт (Волна 1) и верификатор.

Волна 2 = расширить ПОНИМАНИЕ (тема/факты/слоты); выдача и маршрут остаются детерминированными. Модель ПРЕДЛАГАЕТ, детерминизм РЕШАЕТ «что не войдёт» (паттерн hard-filters из wide_fact_pack).

## Открытые вопросы (закрыть в Фазе B)

1. **Граница выбора фактов:** модель предлагает тему/факты, детерминизм отбрасывает чужой бренд / не-client_safe. Подтвердить инвариант: модель не может протащить факт чужого бренда.
2. **Б4 вето (624):** заменить keyword-вето, а НЕ снять наголо. Снятие = риск галлюцинации слота (модель проставит класс/предмет, который клиент не называл → ложный контекст → выдумка по сценарию). Нужен критерий обоснованности слота; калибровать по замеру.
3. **Существующий планнер** `TELEGRAM_RULES_ENGINE_PLANNER_INTENT` (step3a, на `understand()`, keyword-маршрут, флаг OFF) — Волна 2 строится ПОВЕРХ него или заменяет? Свериться в Фазе B, не задвоить intent на модель.
4. **Ширина расширения** — калибруется замером Волны 1 (надёжность выхода).
