# ТЗ: ночная рекламная воронка (автономная вечерняя отправка). Claude #1, 2026-05-27.

Решение Дмитрия 2026-05-27: строить ночную воронку. Цель — НЕ терять вечерних/ночных лидов с рекламы
(Яндекс Директ/РСЯ → лендинг cdpofoton.ru или kmipt.ru → Telegram-бот), когда менеджера нет онлайн.

## ГЛАВНОЕ: это самый высокий по ставкам режим — безопасность ПЕРВИЧНА
В отличие от черновикового режима, здесь сообщение уходит клиенту БЕЗ человека в петле. Значит любая
ошибка (бренд-микс, выдумка, мисхэндл P0) видит реальный клиент. Поэтому строим в таком порядке:
сначала предохранители, потом автономия; и запускаем стадиями (shadow → узкий live → шире).

## Инварианты (НЕ ослаблять, fail-closed)
- Бренд из КАНАЛА (kmipt.ru-бот = УНПК, cdpofoton.ru-бот = Фотон); НИКОГДА не угадывать. Кросс-бренд
  вопрос → безопасная фраза про отдельные организации.
- P0 (реальная претензия по возврату, жалоба, юр.угроза, гарантия результата, спорная оплата) и любое
  «спорное/чувствительное» → НИКОГДА не авто-отвечать; безопасный холд + очередь менеджеру на утро.
- Не выдумывать (closed-world), не смешивать бренды, не раскрывать числа не из retrieved_match.
- Не писать в AMO/Tallanto/CRM (лид-захват пишем в ОТДЕЛЬНОЕ лид-хранилище, не в CRM, до отдельного
  разрешения Дмитрия). ASR/Resolve/Analyze не запускать.

## Архитектура потока (ночь, лид с рекламы)
1. Лид приходит в бот по диплинку с лендинга → канал задаёт `active_brand` + UTM (источник/кампания/
   предмет-интент, если лендинг его передал).
2. Понимание → retrieve_facts (как в v2) → черновик → ВЕРИФИКАТОР (наш fail-closed) → **GATE автоотправки**.
3. GATE решает один из трёх исходов: AUTO-SEND / SAFE-HOLD / MANAGER-QUEUE (см. ниже).
4. В любом исходе — лид-захват (имя/класс/предмет/контакт по одному вопросу) + назначенный утренний шаг.
5. Всё логируется (см. логирование).

## GATE автоотправки (per-message, детерминированный, fail-closed)
Сообщение уходит клиенту автономно ТОЛЬКО ЕСЛИ выполнено ВСЁ:
- маршрут = answer_self И факт по спрошенному = `retrieved_match` (точный подвопрос+scope);
- бренд чист; числа все retrieved_match (нет no_match/other_brand/wrong_scope);
- нет P0/чувствительного; нет meta-leak (источник/JSON/«я бот» без политики);
- ответ не пустой и не «передам менеджеру».
Иначе:
- если есть безопасная частичная польза → **SAFE-HOLD**: «Сейчас нерабочее время, менеджер свяжется
  утром до [час]. Пока подскажу, что могу сам: …» + лид-захват. Никаких неуверенных утверждений.
- если P0/чувствительное/совсем нет факта → **MANAGER-QUEUE**: спокойный холд «передам менеджеру, он
  свяжется утром», без авто-ответа по существу.
Принцип: при ЛЮБОМ сомнении деградируем к безопасному холду, а не к отправке.

## Лид-захват + утренний хендофф
- Собрать по одному вопросу: класс, предмет, формат, имя, контакт. Не анкета из 5 пунктов сразу.
- Сложить карточку лида в отдельное лид-хранилище (НЕ CRM): бренд, UTM, что спрашивал, что бот ответил/
  удержал, настроение, метка «ночной холодный», предложенный шаг. Менеджер утром видит готовый контекст.
- Дать конкретику: «менеджер свяжется до [час] [дата]» (а не «когда-нибудь»). Отвал №1 — открытое
  «сообщим, когда появится».

## Стоп-кран
- Глобальный ручной kill-switch: команда → бот переходит в режим «только холд/менеджер утром».
- Авто-трип по аномалии за скользящее окно (последние N диалогов): всплеск SAFE-HOLD/MANAGER-QUEUE/
  ошибок, ИЛИ срабатывание детектора бренд-микса/выдумки/wrong_scope → авто-переход в холд-режим + алерт.
- Лимит на ночь: не более N автономных диалогов за окно, дальше — холд (защита от шторма/атаки).
- Анти-провокация/джейлбрейк фильтром ДО модели («ты бот», «притворись менеджером», «другой бренд»,
  инъекции) → фиксированный безопасный ответ.

## Стадии запуска (так «строим воронку» = строим безопасно)
1. **SHADOW (первая):** бот генерит решение GATE и сообщение, но клиенту НЕ отправляет — только логирует
   «что отправил бы». Утром менеджер/я смотрим: правильны ли решения AUTO-SEND vs HOLD на РЕАЛЬНЫХ лидах.
   Это даёт реальный сигнал с НУЛЕВЫМ риском для клиента и валидирует GATE против реальности.
2. **Узкий LIVE:** включаем автоотправку только для (а) одного рекламного источника, (б) cap N/ночь,
   (в) самых безопасных типов (цена/формат/адрес/расписание-ориентир с retrieved_match). P0/спорное —
   по-прежнему холд.
3. **Шире:** расширяем источники/типы по мере чистых метрик.
Критерий перехода shadow→live и расширения: чистые метрики shadow (доля корректных AUTO-SEND решений,
0 бренд-миксов/выдумок/мисхэндла P0) + базовая линия на реалистичном УНПК-наборе и judge_v2. Здесь
измерительная работа и становится гейтом graduation — она не пропадает.

## Логирование (durable-актив)
Каждый ход: вход клиента (+UTM/бренд) → retrieved факты и их уровни → черновик → решение GATE (send/
hold/manager) + причина → что реально произошло. Это и пилотный сигнал, и обучение, и аудит безопасности.

## Что за Дмитрием (бизнес/доступы)
- С какого БРЕНДА начать: УНПК (под него уже есть реалистичный набор/eval — чище мерить) или Фотон (там

- Рабочие часы (граница «ночь»), какой рекламный источник/кампанию пускать первой в LIVE, cap N/ночь.

## Порядок
Кодекс строит каркас SHADOW (decision engine + GATE + стоп-кран + лид-хранилище + логирование + тесты) →
ПАССИВНЫЙ запуск shadow (наблюдает копию входящих, НЕ владеет ответом) → я регрейжу решения GATE на
реальных shadow-логах + базовую линию на реалистичном наборе → при чистых метриках и отдельном решении
включаем узкий LIVE. Автономный LIVE без зелёного GATE на реальных данных НЕ катим.

## REV2 — обязательные уточнения перед реализацией (приняты от Кодекса 2026-05-27)
Все семь — ТВЁРДЫЕ предусловия каркаса SHADOW.

1. **SHADOW пассивен, не «съедает» клиента.** Shadow НИКОГДА не владеет путём ответа: он наблюдает КОПИЮ
   входящих (tee/реплей логов или тестовый канал) и логирует «что отправил бы». Реальный поток клиента
   обрабатывается как сейчас (менеджер/черновик). Запрещено: shadow читает Telegram-updates и оставляет
   клиента без ответа.
2. **LIVE нельзя включить случайно — два независимых предохранителя.** (а) `SHADOW_ONLY=1` по умолчанию;
   (б) отдельный runtime-блокер, запрещающий `send_message` клиенту без явного live-token/подтверждения.
   Тумблер mode в bot_control.json САМ ПО СЕБЕ live не включает — нужны оба снятых предохранителя.
3. **Уровни фактов (retrieved_match/wrong_scope/...) — ОБЩИЙ модуль, не копия.** Вынести единую функцию
   сверки фактов и переиспользовать её И в runtime-GATE, И в sim-судье (judge_v2). Иначе измеритель и
   реальный бот снова разойдутся. Один источник правды для уровней.
4. **SAFE-HOLD — тоже клиентское сообщение, проверять его.** Текст холда шаблонный, проходит бренд/факт-
   проверку; обещать утренний звонок/время МОЖНО только если реально есть процесс утренней обработки
   лидов и точный рабочий час бренда. Нет процесса → мягкий честный текст без обещания звонка.
5. **Лид-хранилище = ПДн.** Путь ВНЕ `stable_runtime`; явная схема; маскирование ПДн в audit pack; срок
   хранения; запрет коммита (gitignore); тест «не пишем в AMO/Tallanto/CRM».
6. **(добавил Claude) Shadow-лог должен быть регрейдабельным.** На каждый ход логировать: вход+UTM+бренд →
   извлечённые факты и их УРОВНИ → would-be текст → решение GATE + причина. Без would-be текста и уровней
   я не смогу независимо оценить, верны ли решения AUTO_SEND.
7. **(добавил Claude) Бизнес-зависимость — процесс утренней обработки.** Обещание «менеджер свяжется
   утром» имеет смысл только если кто-то реально разбирает лид-очередь утром. Это вопрос к Дмитрию ДО
   live; в shadow — просто фиксируем лиды, без обещаний клиенту.
