## 3. ДЕЙСТВИЕ 1 (критично): заменить устаревшие копии в pack

В `bot_improvement_candidates_20260523/01_gold_and_few_shot/` лежат **старые** версии. Замени один-в-один:

```
01_corpus_files/gold_dialogues_multiturn_2026-05-23.yaml   -> pack/01_gold_and_few_shot/gold_dialogues_multiturn_2026-05-23.yaml
01_corpus_files/few_shot_warm_answers_2026-05-23.yaml      -> pack/01_gold_and_few_shot/few_shot_warm_answers_2026-05-23.yaml
01_corpus_files/few_shot_advanced_pack_2026-05-23.yaml     -> pack/01_gold_and_few_shot/few_shot_advanced_pack_2026-05-23.yaml
```

Плюс положи рядом (их не было в pack):
```
01_corpus_files/training_pairs_and_routes_2026-05-23.yaml  -> pack/01_gold_and_few_shot/  (или 04_tests для пар)
01_corpus_files/holdout_eval_2026-05-23.yaml               -> pack/04_tests_and_failure_signals/  (eval-only)
```

**Почему критично:** твоя копия gold = 20 диалогов, где фраза «Скоро повышение, по текущей разумно
успеть» повторяется дословно 5 раз, нет слоя возражений/эмоций и нет честного раскрытия ИИ. Если
few-shot-селектор берёт few-shot из старой копии — он заливает боту ровно тот шаблонный тон, который
мы лечим. После замены обнови строки в `FILE_ROLE_REGISTRY.csv` / `DEDUP_MAP.md` при необходимости.

## 5. ДЕЙСТВИЕ 3 (важно): пересобрать holdout — он загрязнён

В `04_tests_and_failure_signals/`: `v8_dynamic_tuning_set` (223 персоны) и `v8_dynamic_holdout_set`
(83 персоны) вырезаны из общего пула `v8_dynamic_client_sim` (306). Проверка «пересечений persona id
нет» прошла, потому что **id переименовали**, НО по содержанию **47 из 83** holdout-персон — это почти
клоны тюнинговых (тот же типаж + цель + бренд, отличие только класс/предмет/формат, тот же скрипт
поведения и критерии). Тюнясь на одних, бот учится проходить их близнецов в holdout → честного сигнала нет.

Что сделать (любой из путей):
- пересобрать holdout на **действительно новых** персонах (другие типажи/формулировки/связки тем), либо
- выкинуть 47 клонов и измерять только на ~36 реально новых, либо
- взять мой `holdout_eval_2026-05-23.yaml` (18 базовых + 2 идентичности, специально непересекающиеся)
  как чистую затравку и дорастить до нужного объёма.

Дисциплина: holdout — `eval_only`, никогда не в few-shot и не в тюнинг промпта.

## 9. Как подключать мои few-shot (правила инъекции)

- Брать 2–3 примера, совпадающих по **активному бренду И теме**; для нейтральных тем — `general_any_brand`
  (там нет брендовых фактов).
- В каждом примере есть схема: `brand`, `required_fact_ids`, `forbidden_if_fact_missing`, `route`, `teaches`.
  **Подставлять пример только если его `required_fact_ids` реально подтверждены в КБ активного бренда.**
  Если факта нет — брать вариант из `no_fact_examples` (полезно без конкретики).
- Пары «плохой→хороший» (`bad_good_pairs`, `training_pairs`) — это топливо для **пост-фильтра/rewrite**,
  НЕ как прямой клиентский ответ.
- `crm.customer_loaded` — это runtime-контекст пилота (CRM-фид), а НЕ KB-факт. Не путать.
- `holdout_eval` — только финальная оценка. Не в few-shot, не в тюнинг.
- В лог решения писать выбранные few-shot ID и причины пропуска (как в твоём Этапе 1).
