## ЖЕЛЕЗНЫЙ ПРИНЦИП ВОЛНЫ (читать первым)
Автономный ответ разрешён ТОЛЬКО когда факт привязан к ТОЧНОМУ подвопросу и совпадает по scope —
уровень **retrieved_match** (как в M3), а НЕ «нашёлся какой-то client-safe факт бренда». Нет точного
факта → НЕ выдумывать и НЕ подменять соседним. **Глобального override «любой факт → автономный ответ»
НЕ вводить.** Смежный факт — только как явно вторичная справка, не как ответ на вопрос. Реальный P0
перебивает продажу. Анти-повтор — детерминированный, без добавления фактов.

Инварианты (НЕ ослаблять): бренды не смешивать; closed-world (никаких выдуманных «да/нет»); настоящий P0
(реальная претензия/возврат-спор/жалоба/юр.угроза/гарантия результата/спорная оплата) → менеджер;
hard-gate 0; фаза 1+2 возврата (диалоги 01/02) не сломать; анти-склейка Волны 2, M3-числа, целостность KB
не трогать. Холдаут не тюнить. Судью в этой волне НЕ трогать (отдельное ТЗ). Сырого судью не
интерпретировать — регрейд мой.

## Шаг 1 (S2+S5, СНАЧАЛА) — Измерение over_handoff по ходам + кандидаты false_handoff с уровнями

**Зачем первым:** без честной метрики не отличить улучшение от шума (на 8 диалогах всё скачет).

**Что сделать:** (а) в `dynamic_summary` считать over_handoff ПО ХОДАМ (доля реплик бота с уходом в
менеджера/мягким хендоффом), не по диалогам. (б) Для каждого такого хода детерминированно определить
уровень доступности факта по СПРОШЕННОМУ подвопросу, переиспользуя логику M3:
`retrieved_match` / `same_brand_global_match` / `wrong_scope` / `no_match`. (в) `false_handoff` считать
ТОЛЬКО при `retrieved_match` (был точный факт по точному подвопросу, а бот всё равно ушёл). Остальные
уровни — НЕ ложный хендофф. Вывести список с уровнями.

**Приёмка:** в сводке есть `over_handoff_turn_rate` и список `false_handoff` ТОЛЬКО уровня
retrieved_match. Поведение бота не меняется. На текущем коде список покажет реальные ложные хендоффы
(ожидаемо диалог 22 t1 — адрес был точным фактом).
**Инвариант:** метрика считается из артефактов прогона, маршрут не трогает; уровни — как в M3, не «где-то
в базе был факт».

## Шаг 2 (R1, УЖЕСТОЧЁН) — Точный факт по точному подвопросу переопределяет осторожность

**Проблема:** маршрут идёт от `contract.answerability` модели; если модель перестраховалась, бот уходит
в менеджера даже при ТОЧНОМ факте. Диалог 22 t1: адрес (Сретенка, 20) — точный факт, но бот ушёл.

**Что сделать:** разрешить ответ вместо хендоффа ТОЛЬКО если факт по СПРОШЕННОМУ подвопросу имеет уровень
`retrieved_match` (точный ключ + совпадение scope: бренд, и где применимо класс/формат/площадка). Тогда
наличие такого факта переопределяет осторожный `answerability`. **НЕ** разрешать ответ, если факт
`same_brand_global_match`/`wrong_scope`/`no_match` — это не ответ на вопрос (вспомнить баг
`course_rules_safe` вместо refund). Реализация: перед `force_draft_for_manager` (~957-969) проверять
именно `retrieved_match` по прямому вопросу, не «есть ли какой-то факт».

**Приёмка:** диалог 22 t1 → бот сразу даёт адрес (точный факт). Случай, где найден только факт другого
scope/смежный → бот НЕ отвечает им как ответом (идёт в шаг 3). Нет точного факта → не выдумывает.
**Регресс-тест:** (1) retrieved_match + answerability=manager_only + не P0 → route bot_answer_self, текст
= точный факт; (2) только same_brand_global/wrong_scope → НЕ автономный ответ этим фактом; (3) no_match →
мягкий хендофф (шаг 3).

## Шаг 3 (R6, УЖЕСТОЧЁН) — Когда точного факта нет: честно про спрошенное; смежное — только вторично

**Проблема:** диалог 04 — спросили «перевод на счёт без карты?», бот ОТВЕТИЛ про рассрочку/ЛВШ (подмена).

**Что сделать:** при отсутствии `retrieved_match` по спрошенному строить ответ в строгом порядке:
(1) честно и коротко про СПРОШЕННОЕ: «по оплате переводом на расчётный счёт точного подтверждения нет —
именно это уточнит менеджер»; (2) ТОЛЬКО ПОТОМ, по желанию, смежная справка, явно помеченная как
вторичная: «из подтверждённого: доступна оплата частями на 6/10/12 мес»; (3) один шаг. Смежный факт
НИКОГДА не подаётся как ответ на вопрос и не идёт первым. Если смежного нет — просто честный микро-хендофф
по спрошенному + шаг. manager_only (без ответа) — только для P0 или когда сказать совсем нечего.

**Приёмка:** диалог 04 t1 → «по переводу на счёт без карты точного подтверждения нет, менеджер уточнит
именно это; из подтверждённого — оплата частями 6/10/12 мес» — один раз, спрошенное идёт первым, подмены
нет. **Регресс-тест:** вопрос без точного факта, но со смежным → в ответе сперва честно про спрошенное,
смежное помечено вторичным; вопрос без точного и без смежного → честный микро-хендофф, без выдумки.

## Шаг 4 (T6) — Отвечать на конкретику клиента, на ИМЕННО заданный вопрос

**Что сделать:** (а) клиент назвал класс/формат, факт задан на диапазон (retrieved_match по диапазону,
куда класс попадает) → подавать «для 7 класса: …», не показывать служебную вилку (не попадает в
диапазон-факт → не утверждать). (б) Отвечать на именно заданный вопрос; нет точного факта по нему — шаг 3
(не подменять). Чинить в черновике/понимании (тёплый слой факты не меняет).

**Приёмка:** «цена для 7 класса» → «для 7 класса: семестр X, год Y» без «5–11» и без дубль-пересказа.
**Регресс-тест:** конкретный класс из диапазона-факта → в ответе класс клиента, не диапазон.

## Шаг 5 (T3/T7) — Детерминированный анти-повтор (без новых фактов)

**Проблема:** диалоги 04/22 повторяли одну фразу 3-4 раза.

**Что сделать:** ДЕТЕРМИНИРОВАННО: хранить последние 2-3 финальных ответа бота в нити; при повторе по
форме ИЛИ смыслу — не отдавать тот же текст. Если клиент переспрашивает то, на что точного факта нет, —
сменить ТАКТИКУ (а не перефразировать факт): признать «этот пункт я уже отметил для менеджера» +
предложить другой шаг (захват контакта/смежная вторичная справка). **Не свободный переписыватель** — новые
факты не добавлять (иначе выдумка). Распространить на `_safe_fallback_text` и draft-ответы (сейчас ротация
только у `_REFUND_POLICY_TEXTS`/`_DRY_P0_TEXTS`).

**Приёмка:** диалоги 04/22 — ни одной буквально/смыслово повторённой фразы; при повторном вопросе бот
меняет ход, не добавляя фактов. **Регресс-тест:** клиент 3 раза давит вопросом без точного факта → 3
РАЗНЫХ хода, последний — смена тактики, без новых фактов.

## Шаг 6 (M6+R5, ОГРАНИЧЕН) — Уточнение/разбор; но реальный P0 перебивает продажу

**Что сделать:** (а) не хватает одного слота (класс/предмет/формат) → ОДИН уточняющий вопрос, не хендофф,
не угадывать. (б) Неоднозначность → «вы про X или Y?». (в) Составной вопрос разбирать по частям ТОЛЬКО
если в нём НЕТ реального P0: безопасные части ответить (по правилам шагов 2-3), гипотетическую/нет-факта
часть честно отметить. **ОГРАНИЧЕНИЕ:** если в сообщении есть реальная претензия/спор оплаты/жалоба — P0
ПЕРЕБИВАЕТ всё: бот НЕ отвечает параллельно на коммерческие части и НЕ продаёт, уходит в менеджера по P0.

**Приёмка:** «сколько стоит и можно ли вернуть (гипотетически)?» → цена + гипотетический возврат
отвечены. «Занятий нет, верните деньги, и сколько стоит продление?» → P0 (реальная претензия) перебивает:
менеджер, без параллельной продажи. **Регресс-тест:** составной с реальным P0 → НЕ отвечает на
коммерческую часть; составной без P0 → безопасные части отвечены. Не более одного уточняющего вопроса/ход.
