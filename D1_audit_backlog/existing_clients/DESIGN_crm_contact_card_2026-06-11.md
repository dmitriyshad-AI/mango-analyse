## 1. Ключевая вводная о модели данных

Tallanto: главная сущность — УЧЕНИК; ФИО родителя — просто поле. AMO копирует эту логику ради синка (ключи «Id Tallanto»/«Филиал Tallanto» защищены). Следствия: (а) менеджер не видит других детей родителя; (б) заявка с телефона действующего клиента неотличима от заявки про нового ребёнка; (в) дубли карточек.

Вердикт двух архитекторов: «контакт = ученик» — ЗАКОННАЯ модель для школы (ученик носит группы/балансы/посещаемость), перестройка на «контакт = родитель» дорога и проблему не решает. Дефект не в модели, а в отсутствии СЕМЕЙНОГО ВИДА поверх неё и крана дублей. Наш профиль ТЗ-12 (объединение по телефону) — и есть семья: 8 895 семей с AMO-контактами, 601 многодетная (2+ ученика Tallanto), 834 (9,4%) размазаны на 2+ карточки, 367 контактов спутаны между семьями (общие телефоны).

Подробности — v1-разделы, инвентарь кода в §2 прежней редакции (deal_writeback.py:21-28, deal_quality_gate.py:22-57, amo_rollback.py:279-284, render_crm_summary).

## 4. Формат карточки v1 — адаптирован под «контакт = ученик»

Поле: **«ИИ: профиль клиента»** (предложение; textarea). Карточка КОНКРЕТНОГО ученика, семья — строкой:

```
Ученик: Роман, 8 класс, математика [Фотон]; интерес: годовой курс, онлайн.
Семья: родитель Елена; ещё дети: Софья — 5 кл, обсуждали лагерь [УНПК] (карточка 12345).
Договорённость семьи: 02.05.2026 — отправить расписание, перезвон в июле [Фотон].
Возражения/чувствительность: цена (спрашивали скидки).
Tallanto: активная группа, баланс есть; стадия AMO: переговоры.
Обновлено ИИ 11.06.2026 по истории общения. Поле вручную не править.
```

Правила (дополнительно к v1): единица записи — СЕМЬЯ (все карточки учеников семьи обновляются одним пакетом, hash на уровне семьи; частичная запись пакета → откат); телефон привязан к 2+ семьям → строка «Семья» НЕ пишется (skip + журнал, до ручного разбора общих телефонов); дети-merge_candidate (817 групп) — одной строкой на группу, не двумя «разными» детьми; чувствительное (диагнозы и т.п.) не выводится — фильтры render_crm_summary.

v2-расширение (отдельным решением позже): каналы и счётчики касаний — требует новых слотов билдера.

## 5-7. Безопасность, этапность, каркас ТЗ — как в v1, с правками

(§5 v1 целиком в силе.) Этап А: 20 контактов = 20 СЕМЕЙ пилотной зоны, рядом старое поле для сравнения. Identity-гейт ужесточён: телефон контакта обязан УНИКАЛЬНО совпадать с одной семьёй.

## 10. Заявки с известного телефона — главный быстрый рычаг

Регулярный скан новых сделок (раз в 10-15 мин, паттерн morning scan; вебхуки НЕ строим — новая инфраструктура): телефон контакта новой сделки находится в профильной базе → авто-ПРИМЕЧАНИЕ в сделку (механизм примечаний обкатан пилотом): «Телефон известен: семья Елены. Дети: Роман (учится, мат. 8 кл [Фотон]), Софья (обсуждали лагерь [УНПК]). Уточните, о ком разговор». Журнал обработанных lead_id (повторы исключены); телефон 2+ семей → примечание «общий телефон, уточните» без имён; без телефона → пропуск. Ограничение честно: к заявке, схваченной менеджером за 2 минуты, скан опоздает; ночному трафику (цель проекта) успевает всегда.

## 11. Дубли — кран, чистка, метрики

1. **КРАН (пункт №1 по цене/эффекту, конфигурация, не код):** дубли рождают формы/интеграции, создающие контакт вместо поиска по телефону. Изучить и осознанно включить контроль дублей AMO/настройки форм. Нюанс модели: «прикреплять к существующему контакту» при 3 детях = риск приклеить заявку нового ребёнка к карточке старого; целевое поведение — контакт существующий + сделка новая + примечание §10 подсказывает контекст.
2. **Чистка — реюз готового контура** amo_duplicate_after_staff_done.py + recheck (принцип «CSV кандидатов, сотрудники сливают сами, в AMO не пишем»): (а) дубль = 2+ контакта одного телефона с одинаковым нормализованным именем ребёнка; разбирать ТОЛЬКО живых (активная сделка/учёба, ~150-250 шт., 1-2 дня одного человека), архивный хвост закрыть решением «не трогаем»; (б) 367 спутанных контактов — отдельный список, действие «пометить: общий телефон» (исключает их из строки «Семья»), это различение, не слияние.
3. **Метрики успеха:** новых дублей/неделю (тот же детектор) до/после крана; доля новых заявок с известным телефоном, получивших примечание.

## 12. Запись в Tallanto (задача Дмитрия)

Факт по коду: наш Tallanto-клиент сейчас ТОЛЬКО читает (tallanto_api.py — get/list/search; write-методов нет). План: (1) добавить метод записи + проверить права API; (2) ПРОБА на 1 тестовой карточке, с проверками «частичная запись не обнуляет другие поля» и «поле не участвует в синке Tallanto→AMO»; (3) история общения → отдельное НОВОЕ поле (целиком наше, перезапись по hash); (4) интересы → ровно ОДИН маркерный блок «=== Рекомендация ИИ, обновлено ДД.ММ === … === конец ===»: свой нетронутый блок заменяем; блок порван/правлен человеком → manual_review; блока нет → дописать в конец. Текст людей не редактируется НИКОГДА. NEG: чтение после записи == ожидаемому (ловит молчаливую обрезку длинного поля — иначе риск срезать человеческий текст).
