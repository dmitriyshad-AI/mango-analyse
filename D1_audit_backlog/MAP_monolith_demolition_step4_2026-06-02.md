## ГЛАВНАЯ НАХОДКА (по сырью) — приоритет инвертирован
Поток в guard-chain (subscription_llm):
- `apply_high_risk_content_guards` (зелёный блок, строка 4002) вызывается на **1663/1681 — РАНЬШЕ** диспетчера.
- `apply_dialogue_contract_v2_template_dispatcher` (1501) на **1905**, и в начале:
  `if _safe_template_already_applied(result): return result` — **пропускает rules_engine, если монолит уже
  применил шаблон.**

Вывод: для доменного хода МОНОЛИТ срабатывает первым и перебивает rules_engine. Правила выигрывают ТОЛЬКО на
ходах, где зелёный блок отказался. Это ОБРАТНО задуманному (правила первичны, монолит — фолбэк). Зелёный блок
НЕ знает про rules_engine (проверки `_rules_engine_result_applied` в нём нет).

**Следствие для Шага 4:** нельзя просто удалить монолит — на каждом ходу, где сейчас выигрывает зелёный блок,
поведение изменится. Сначала сделать rules_engine авторитетным, потом сносить.

## План Шага 4 (волнами, с негатив-контролем)

### Фаза 0 — измерить теневое перекрытие (офлайн, 0 LLM)
На сохранённых транскриптах (58 Прогона 1 + agreement-прогон) посчитать на КАЖДЫЙ доменный ход: применился
`rules_engine_*_applied` ИЛИ монолитный `*_safe_template_applied`. Это покажет, как часто монолит сейчас
перебивает правила. База для оценки риска инверсии.

### Фаза 1 — сделать rules_engine АВТОРИТЕТНЫМ (инверсия приоритета)
Вариант (выбрать по сырью): (а) перенести вызов диспетчера ПЕРЕД зелёным блоком; ИЛИ (б) в зелёном блоке для
МИГРИРОВАННЫХ доменных интентов отдавать ход rules_engine (yield), оставив зелёный блок только для cross_brand/
terminal. Гейт Шага 1 остаётся финальной страховкой. **Это меняет генерацию → нужен точечный прогон.**

### Фаза 2 — снести доменный монолит (после Фазы 1)
Убрать 17 доменных safe_template (C) + 4 доменные записи реестра (matkap/tax/olympiad/trial) + их `_produce_`
обёртки + их вызовы в зелёном блоке + мёртвые текст-константы (`*_SAFE_TEXT`, используемые ТОЛЬКО снесёнными
функциями — грепнуть каждую перед удалением). Зелёный блок усохнет до cross_brand+terminal; реестр — до 4
безопасностных specs.

### Что ОСТАЁТСЯ после Шага 4
rules_engine (16 правил) — единственный доменный путь; safe_template остаётся ТОЛЬКО для безопасности
(cross_brand, terminal/identity, result_guarantee, admission_guarantee); инфраструктура диспетчера; выходной
гейт. Чистая целевая архитектура.

## Обязательные негатив-контроли (каждая фаза)
- бренд-разделение: cross_brand-фраза по-прежнему выдаётся (УНПК-бот про Фотон) — НЕ сломать;
- identity: «ты бот?» → политика C (terminal) — НЕ сломать;
- P0-обещания: гарантия результата/поступления → менеджер (result/admission_guarantee) — НЕ ослабить;
- HARD-P0 возврат/жалоба → manager_only;
- выдумки/утечки = 0 (гейт ловит);
- офлайн-регрессия: на доменных ходах ответ из rules_engine идентичен или лучше (не хуже монолита).

## Риски и как снять
- **Главный риск:** после инверсии (Фаза 1) на ходах, где раньше выигрывал монолит, теперь ответит правило —
  может отличаться текст/маршрут. Снять: Фаза 0 заранее показывает где расхождения; точечный прогон после
  Фазы 1 их подтверждает; гейт страхует от утечек.
- **Скрытые вызовы:** перед удалением каждой функции — grep по ВСЕМ channels (не только зелёный блок); часть
  зовётся и из `apply_high_risk_content_guards` второй раз (2349/2359 — другой поток), проверить оба.
- **Константы:** удалять `*_SAFE_TEXT` только если греп подтвердил 0 использований вне снесённых функций.
