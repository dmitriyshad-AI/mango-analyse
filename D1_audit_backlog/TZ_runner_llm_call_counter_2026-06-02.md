# Мини-ТЗ — счётчик LLM-вызовов в раннере симулятора. Для Кодекса. 2026-06-02.

Автор: Клод #1. Запускать ПОСЛЕ 2b.6 (не срочно, лёгкая правка). Цель: каждый прогон показывает фактическую
стоимость в LLM-вызовах, чтобы планировать размер набора, а не оценивать «по структуре».

## Что сделать (разово, в раннере)
В `scripts/run_telegram_dynamic_client_sim.py`:
1. Завести счётчик, инкрементирующий на КАЖДЫЙ вызов модели `generate()` — отдельно по ролям:
   `client` (персона), `bot_draft`, `bot_critic` (faithfulness и пр., если есть), `judge`.
   Проще всего — тонкая обёртка вокруг `.generate()` каждой роли (client/bot/judge), которая ++счётчик.
2. В `dynamic_summary.json` (и в `dynamic_summary.md`) добавить блок:
   ```json
   "llm_calls": {"total": N, "client": .., "bot_draft": .., "bot_critic": .., "judge": ..,
                 "dialogs": D, "turns": T, "avg_calls_per_dialog": N/D}
   ```
3. Работать при обоих режимах: `--judge-mode codex` (реальный) и `fake` (счётчик judge=0, т.к. fake — не LLM).
   В `--rejudge`-режиме считать только judge-вызовы.

## Рамки
Хирургично: только раннер + summary-запись. НЕ менять логику бота/гейта/правил, не трогать stable_runtime,
KB. Это измерительная правка, поведение прогона не меняет → живой прогон для проверки НЕ нужен: достаточно
`--judge-mode fake` на 2-3 диалогах (дёшево) + сверка, что числа в summary совпали с числом ходов×ролей.

## Тест выхода
- Прогон `--judge-mode fake` на 2-3 диалогах: `llm_calls.total` ненулевой и сходится с диалоги×ходы×роли
  (client + bot_draft [+ bot_critic]); judge=0 в fake-режиме.
- `dynamic_summary.json` содержит блок `llm_calls`.
- Существующие тесты раннера зелёные.

## Отчёт
Изменённые строки + пример блока `llm_calls` из summary тестового прогона + ревизия коммита.

После этого все будущие прогоны показывают факт вызовов; в ТЗ останется лишь пункт «summary содержит llm_calls».
