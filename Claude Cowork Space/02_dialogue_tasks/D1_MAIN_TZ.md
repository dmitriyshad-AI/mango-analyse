# D1 (ГЛАВНЫЙ) — задача: довести качество бота и честно его замерить

Зона файлов: только `src/mango_mvp/channels/` и тесты бота. Точка отсчёта: коммит `48935d60`.
Опора: «полное ТЗ архитектуры» (`02_bot_quality/TZ_FOR_CODEX_quality_architecture_v2_FULL_BUILD`) +
проверка билда от Claude. Замечания Кодекса учтены. Перед работой пересверить номера строк (дерево «грязное»).

Правило: качество рождается в ответе модели; защитные фильтры только блокируют опасное, а не пишут ответ.

## Часть 1. Закрыть 3 пункта из проверки билда (главное)

### 1.1. Дыра в отлове критичных тем
Сейчас «мошенники / незаконно списали / буду жаловаться» не ловятся нигде в `channels`; единый
классификатор `answer_safety_classifier` зовёт старую функцию `_has_complaint_signal`, которая их пропускает.
- В едином классификаторе: `жалоб\w*` → `жал(?:об|у|ова)\w*`; добавить `мошенн\w*`, `незаконн\w*`.
- Тесты: набор реальных критичных фраз («вы мошенники», «незаконно списали», «буду жаловаться»,
  «пожалуюсь») → всё к менеджеру (100%); рядом безобидный набор («обсудить с менеджером», «подумаю»,
  «надо приезжать») → НЕ критичная тема (0 ложных срабатываний).
- Приоритет: жёсткий список ЛОКИРУЕТ тему (модель/семантика не может его снять); семантика может только
  ПОВЫСИТЬ до критичной; снятие ложной критичности — только для мягких фраз. При сомнении/ошибке/таймауте — к менеджеру.

### 1.2. Довести консолидацию (это и есть замечание Кодекса про «заменить, а не добавить слой»)
Новый классификатор сейчас — обёртка над старыми функциями `_has_*_signal` (в `conversation_intent_plan`).
Нужно сделать один источник: либо вся логика внутри классификатора и старые функции удалены, либо они —
приватные помощники ОДНОГО классификатора без второго вызова из старого пути. Зафиксировать число
детекторов критичных тем до/после — должно стать 1.

**Артефакт-источник правды (по замечанию Кодекса):** оформить `p0_recall_spec` — один модуль/файл, где лежит
жёсткий список сигналов критичных тем + проверочный набор реальных критичных фраз + безобидный набор.
Этот же артефакт диалог D4 (база знаний) ИМПОРТИРУЕТ или ссылается на него, а не переписывает руками.
Один и тот же проверочный набор проходит и в коде (D1), и в триггерах базы (D4).

### 1.3. Реально убрать шаблоны-заглушки на «зелёных» темах (поэтапно)
В `subscription_llm` ~29 вызовов `_safe_template(` (88 упоминаний). Сейчас они только «отключаются при
наличии контракта», но не удалены. Убирать **по одной теме за раз, под флагом** (по замечанию Кодекса —
не рубить всё разом): вместо подмены — контракт ответа + факты по смыслу запроса + живые примеры + ответ
модели; защитный фильтр только блокирует/передаёт менеджеру. Жёсткие шаблоны ОСТАВИТЬ только для критичных
тем, передачи менеджеру и развязки брендов. Зафиксировать число шаблонов-заглушек до/после — оно должно УПАСТЬ
(это формальный признак, что качество переехало в ответ модели).

## Часть 2. Честный замер (после Части 1)

Порядок: модульные тесты (по свойствам, не по конкретным строкам) → статические проверки критичных тем и
брендов (+ проверочный набор) → быстрый прогон `v8_targeted16` (сигнал для отладки) → **разбор замечаний ПО
КЛАССАМ проблем**, не по единичным случаям → правки архитектурного уровня → повтор `targeted16` →
**ОДИН прогон закрытой проверки** (`03_corpus/v8_holdout_clean_2026-05-24.jsonl`, только оценка, НЕ подгонять)
→ полный прогон v8 (300+, с продолжением и полными расшифровками). Перед закрытой проверкой — перезапустить
ботов, чтобы взяли новый код.

Для отладки использовать набор `03_corpus/v8_dev_set_2026-05-24.jsonl` (36 диалогов) и `targeted16`.
Закрытую проверку (`v8_holdout_clean`) НЕ трогать для подгонки — это финальная честная оценка.

## Часть 3. Второй проход модели (рычаг человечности)
Включить переписыватель слабого ответа только по флагу и только на мягкие промахи; в подсказку класть живые
примеры из тёплого набора (`03_corpus/few_shot_warm_answers`); чередовать примеры; после переписывания —
снова защитные фильтры по критичным темам/брендам/фактам. Сначала только в симуляции, в живой бот — после
подтверждения, что критичные темы/бренды/факты не ломаются.

## Часть 4. Подключить историю клиента (после замера)
После закрытой проверки: read-only поставщик истории клиента (модуль `customer_timeline`) → в контракт
ответа как `known_client_history`; ослабить слишком строгий фильтр (`p1_missing_recommended_action`, из-за
него 82% «на ручную проверку» — это самоналожено, не плохие данные); сначала предпросмотр под флагом, потом
основное чтение с запасным путём. Это даёт персонализацию (вся суть проекта); код уже написан (~15 тыс. строк).

## Приёмка
- отлов реальных критичных тем = 100%, ложных = 0, грубых нарушений безопасности = 0, утечек другого бренда = 0,
  утечек служебных пометок (`kb_release` и т.п.) = 0;
- число шаблонов-заглушек и число детекторов критичных тем СНИЗИЛОСЬ;
- на закрытой проверке: рост по «ответил на вопрос», «не шаблонно», «тон», «менеджер отправил бы без правок %» к базовому уровню;
- судье не подсовывают собственный шаблон бота как факт; закрытую проверку не подгоняли;
- собран отчёт с «число веток до/после» и «закрытая проверка до/после».
