# ТЗ-15 v3: строгий авто-резолвер «Wappi-чат → AMO-сделка» для черновикового контура

Дата: 2026-06-12 (v3 — вечер: добавлены Max по телефону и офлайн ретро-сверка по решению Дмитрия). Постановщик: главный архитектор. Аудит вторым архитектором пройден (BLOCKED → правки внесены в v2).
Статус политики: направление одобрено Дмитрием (разбор 12.06). Включение live-write авто-пар и каждое пополнение серверного списка — ТОЛЬКО по отмашке Дмитрия.

## Зачем

Черновики пишутся только по ручному файлу пар. За 48 часов — 36 живых кандидатов, привязан 1. Wappi не отдаёт сделку в `crm_entities` (пусто), но связь восстанавливается через AMO-контакт по полю `Telegram ID`.

## Контрольный инцидент (НЕ повторить)

Сверка архитектора 12.06 по read-only CRM: из 4 связок, заявленных «однозначными», чистых 2.
- `49804475` — сделка ЗАКРЫТА (статус 143, closed_at заполнен) → обязательный отрицательный тест, отдельный код причины `closed_lead`.
- `49845425` — открыта, но требует ручной проверки бренда (одна воронка с фотоновской сделкой; воронка — не бренд-маркер по канону).

## Шаг 1. Резолвер (за флагом `DRAFT_LOOP_AUTO_RESOLVER`, default OFF)

Правила матча — все обязательны, иначе в ручную привязку с кодом причины:

Telegram-каналы:
1. Ключ — числовой `Telegram ID` контакта AMO, равный chat_id. Матч только по `username` → `username_only` (username меняется и переиспользуется). Для Max числовое совпадение chat_id ЗАПРЕЩЕНО всегда (это не Telegram ID; совпадение чисел = запись чужому человеку).
2. Контакт по ключу ровно один, иначе `multi_contact`.
3. У контакта ровно одна АКТИВНАЯ сделка: closed_at пуст, статус не из {142, 143}, не удалена. Коды: `closed_lead` / `no_active_lead` / `multi_active_lead` / `deleted_lead`.
4. Бренд черновика задаёт `profile_id` канала (как сейчас). Воронку как бренд-фильтр НЕ использовать (канон: воронка = процесс). ДЕТЕРМИНИРОВАННЫЙ бренд-чек (из документа CRM-канона 12.06): поле сделки «Организация» (4 юрлица). Если поле заполнено и юрлицо принадлежит ДРУГОМУ бренду → `brand_mismatch`, не пишем. Пустое поле — не блокер (пометка Шага 3 остаётся страховкой).

Max-каналы (решение Дмитрия 12.06 — включаем сразу, ключ — ТЕЛЕФОН):
5. ПЕРВЫЙ проверочный пункт работы: отдаёт ли Wappi для Max-чатов телефон собеседника в полях данных. Если НЕТ — Max остаётся на ручных парах, доложить в отчёте и не изобретать обходов (телефон из ТЕКСТА переписки использовать запрещено).
6. Если телефон в полях есть: нормализация номера → ровно один контакт AMO по этому телефону (`multi_contact` иначе) → ровно одна активная сделка (правило 3) → телефона НЕТ в стоп-листе общих семейных номеров (367 номеров из трека действующих клиентов, identity_links; код `shared_phone`). Любой сбой — ручная привязка.

## Шаг 2. Отметка «не раньше чем» (защита от заспама задним числом)

НЕ через список `processed` в state.json (он режется до 5000 записей и не переживает потерю состояния). Вместо этого:
- в строке каждой пары (ручной и авто) — поле `not_before_ts`;
- сообщения с `timestamp <= not_before_ts` пропускаются с записью в журнал;
- сообщения с timestamp=0 / непарсибельным временем НЕ проходят мимо отметки (пропуск);
- для авто-пары `not_before_ts` = момент создания пары.

## Шаг 3. Хранение авто-пар, пометка, журнал

- Авто-пары — в ОТДЕЛЬНОМ файле (не в ручном `draft_loop_pairs.json`), путь в конфиге. Подхват: перечитка на каждом цикле (без рестарта). Идемпотентность: повторный резолв той же пары не создаёт дубль события и не сдвигает `not_before_ts`.
- Пометка в примечании авто-пары: `Привязка автоматическая (Telegram ID). Канал: <бренд> (<Telegram/Max>). Если сделка по другому учебному центру — черновик НЕ использовать, сообщите архитектору.`
- Журнал `auto_pair_created`: profile_id, chat_id, contact_id, lead_id, ключ матча + снимок сделки на момент резолва (status_id, closed_at, pipeline_id) — для регрейда.
- Отклонённые кандидаты — отчёт ручной привязки с кодами причин. ПДн (chat_id, username, имена) — ТОЛЬКО вне репозитория (`~/.mango_local/draft_loop_inventory/`); в `_done` — счётчики, коды причин и путь к файлу.

## Шаг 4. Серверный список разрешённых сделок

- Сервер = независимый источник правды. Жёсткий allowlist на точке записи AI Office НЕ снимать.
- Серверный список пополняется ТОЛЬКО вручную, батчем, по отмашке Дмитрия после сверки архитектором. Цикл серверный файл не трогает НИКОГДА. Правки на сервере — точечной дельтой с бэкапом (процедура 12.06).
- Авто-пара без серверного разрешения не пишет: 403 от точки записи по конкретной сделке = карантин ЭТОЙ пары + событие `allowlist_desync` в журнал; в глобальный счётчик auth-ошибок НЕ считать (сейчас «403» матчится `_is_auth_error_exception` и после лимита глушит весь контур — это поведение для карантинного случая отключить).

## Шаг 5. Отрицательные тесты (обязательны)

1. Закрытая сделка (кейс `49804475`) → не пишем, код `closed_lead`.
2. Две активные сделки → не пишем. 3. Ноль активных → не пишем. 4. Удалённая → не пишем.
5. Матч только по username → `username_only`. 6. Два контакта на один Telegram ID → `multi_contact`.
7. Max-чат: матч по числовому chat_id → ЗАПРЕЩЁН (не пишем), даже если число совпало с чьим-то Telegram ID.
8. Сообщение старше `not_before_ts` → черновика нет; timestamp=0 → черновика нет.
9. 403 по одной сделке → карантин пары, контур продолжает работать по остальным.
10. Повторный резолв той же пары → нет дубля события, отметка не сдвигается.
11. Флаг OFF → поведение байт-в-байт прежнее.
12. Max: телефон встречается у двух контактов → не пишем; телефон из стоп-листа общих номеров → `shared_phone`, не пишем; телефон, выуженный из текста сообщения (а не из поля Wappi) → не используется вовсе.
13. Поле «Организация» сделки = юрлицо другого бренда → `brand_mismatch`, не пишем; пустое поле «Организация» → пишем (с пометкой Шага 3).

## Шаг 6. Офлайн ретро-сверка «бот против сотрудника» (решение Дмитрия 12.06)

Цель: сверить качество черновиков на РЕАЛЬНЫХ старых вопросах, где уже есть фактический ответ сотрудника.
- Отдельный режим запуска (например `--retro-report`): для пар из списков (ручных и авто) взять входящие за последние 48 часов ДО водяной отметки, построить черновики и положить рядом фактический ответ сотрудника из той же переписки Wappi.
- НИКАКИХ примечаний в AMO из этого режима — только файл-отчёт ВНЕ репозитория (`~/.mango_local/draft_loop_inventory/retro_compare_*.json`), в `_done` — счётчики и путь.
- Ограничение расхода: не более 30 диалогов за запуск, число вызовов модели — в отчёт.
- Этот отчёт — сырьё для регрейда архитектора (сравнение с живым менеджером — северная звезда эпохи).
- Водяная отметка живого контура (Шаг 2) при этом ОСТАЁТСЯ: живые примечания — только на новые сообщения. Ретро — отдельным отчётом, не в карточки.

## Приёмка

1. Сухой прогон `--once` с резолвером в режиме «только отчёт» (без записи): счётчики авто-пар и отклонений по кодам в `_done`, сырьё вне репо.
2. Архитектор сверяет каждую авто-пару по read-only CRM.
3. Пополнение серверного списка и live-write авто-пар — отдельной отмашкой Дмитрия.
