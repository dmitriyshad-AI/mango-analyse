# ТЗ — Волна 2, Фаза B: понимание входа на модель (Б2 выбор фактов/темы + Б4 память слотов)

**Статус:** ФИНАЛ. Аудит второго архитектора — 3 раунда (дизайн) + 3 раунда (ширина), внесён. **Ширина вписана** по регрейду пробы Волны 1 (см. §Ширина) — ТЗ готово к реализации. Якоря сверены по коду на `6351de27`.
**База:** ветка `wave2` поверх `wave1` (Волна 2 использует числовой гейт Волны 1 как страховку выхода). Оба флага default OFF. **Реализацию за флагами OFF можно вести параллельно; ВКЛЮЧАТЬ на замере — только после чистой пробы Волны 1.** Мерж в main — по цепочке, после приёмки.

## Ключевой инсайт (по коду)

Модель УЖЕ понимает вход — Волна 2 расширяет влияние этого понимания, не добавляя вызовов:
- `understand()` (`dialogue_contract_pipeline.py:977`) возвращает `planner_intent` / `planner_slots` / `planner_confidence`; на эллипсисе использует `topic_focus` + `known_slots` (промпт :794). Существующий флаг `TELEGRAM_RULES_ENGINE_PLANNER_INTENT` уже переключает МАРШРУТ на модельный intent (`subscription_llm:1310-1324`), есть shadow-метрики согласия planner↔keyword.
- `update_memory_llm` (`dialogue_memory:483`) уже просит модель: «извлеки только явно сказанное, не выдумывай», «active_brand менять нельзя». Слоты возвращаются моделью.

То есть «отдать вход модели» = расширить уже существующие модельные сигналы на выбор фактов и память, а не строить новый понимающий слой.

## Б2 — выбор фактов и темы на модель (ПОВЕРХ планнера)

**Проблема:** `conversation_intent_plan._required_fact_keys`/`_fact_scope_constraints` выбирают `required_fact_keys`/`fact_scope` для ретривала по keyword. Промах keyword по теме → модель не получит факт.
**Дизайн (флаг `TELEGRAM_PLANNER_FACT_SELECT`, default OFF):**
- Источник тем/слотов для ретривала — `planner_intent`/`planner_slots` из `understand()` (переиспользование, 0 новых вызовов).
- `required_fact_keys`/`fact_scope` = **объединение** keyword ∪ planner (recall-first), НЕ замена. Промах одного покрывается другим (правка аудита A1).
- Fail-soft: при `planner_confidence` < порога — только keyword (как маршрут сейчас).
- Расширенная выдача проходит wide_fact_pack hard-filters и скоуп-осознанный числовой гейт Волны 1 на выходе.

## Б4 — память слотов на модель (РАСШИРИТЬ окно вето, не снять)

**Проблема:** `_memory_llm_slot_supported_by_latest_client` (`dialogue_memory:624`) принимает модельный слот только если значение подтверждено keyword/алиасом в ПОСЛЕДНЕЙ реплике. На эллипсисе/перефразе (клиент сказал раньше) модель держит слот верно, а вето рубит.
**Дизайн (флаг `TELEGRAM_MEMORY_LLM_SLOT_WINDOW`, default OFF):**
- Заменить проверку «только последняя реплика» на проверку по ОКНУ клиентских реплик (последние 8 уже коерсятся в `update_memory_llm`).
- **Анти-галлюцинация (сохраняем!):** модельный слот принимается, только если его значение присутствует где-то в истории клиента (детерминированная страховка поверх модельного «не выдумывай»). Какое значение ТЕКУЩЕЕ — решает модель (она видит контекст и свежесть); страховка лишь против выдумки, поэтому смена слота 9→10 не застревает (правка аудита A2).

## Граница безопасности (жёсткое — НЕ отдаём модели)

- `active_brand` — из канала, не из модели (промпт памяти прямо запрещает менять). После модельного выбора retrieved-факты ВСЕГДА проходят бренд-фильтр hard-filters (инвариант, адверсариальный NEG — правка аудита A3).
- client_safe-предикат выдачи; P0-латч/маршрут до модели; числовой гейт Волны 1; семантический верификатор выхода.
- Модель расширяет ПОНИМАНИЕ (тема/факты/слоты), не ДОПУСК.

## Чем страхуется расширенный вход (ответ аудиту B1)

- Числа из шире выбранных фактов → скоуп-осознанный гейт Волны 1 (Б1).
- Хэндофф-claims → верификатор Волны 1 (Б5).
- Качественные нечисловые обобщения в обычном ответе → существующий семантический верификатор выхода. **Новый верификатор НЕ добавляем** (раздувание + хрупкая калибровка из step5); наблюдаем нагрузку (`checked_turns`/`downgrade`) на замере Волны 2; дыра → отдельное ТЗ (не строим впрок).

## Ширина (вписана по пробе Волны 1 — умеренная)

Калибровка по регрейду пробы Волны 1: выход держит числа (Б1 `unsupported_product_number` 10→4, `wrong_scope` ловится), а качественные обобщения верификатор (Б5) **помечает (annotate), но не вырезает** → расширенный вход держим умеренным, иначе менеджер утонет в пометках.

- **Порог `planner_confidence` для fact-select (объединение keyword ∪ planner):** 0.6 — ниже маршрутного 0.72, т.к. planner здесь ДОБАВЛЯЕТ факты (не заменяет), а лишний факт отфильтруют hard-filters + числовой гейт Волны 1; не ниже — чтобы не плодить обобщения. При `confidence < 0.6` или пустых `planner_slots` → keyword-фолбэк.
- **Окно памяти Б4:** переиспользовать существующее окно последних 8 ходов `update_memory_llm` (НЕ вводить новый параметр); анти-галлюцинация = значение слота встречается в клиентских репликах этого окна; свежесть (какое значение текущее) — на модели.
- **Темы при неуверенности:** 2 категории (recall-first); «тема не распознана» → `fallback_core` (существующий).
- **Потолок (несущее, правка аудита):** ширина НЕ увеличивает лимиты wide_fact_pack (≤60 фактов / ≤10 000 симв). planner влияет на ВЫБОР внутри лимита, не на размер — это держит нагрузку верификатора (проба Волны 1: Б5 уже дал +40% проверок, 53→75).
- Один порог — один замер, без перебора порогов (дисциплина «1 прогон»).

## NEG

Б2: (1) OFF-паритет; (2) промах keyword по теме покрыт planner (объединение); (3) `planner_confidence` низкий → keyword-фолбэк; (4) адверсариально: planner_slots с продуктом чужого бренда → бренд-фильтр не пускает.
Б4: (1) OFF-паритет; (2) эллипсис — слот из прошлой реплики держится при ON; (3) смена 9→10 класс не застревает (свежее побеждает); (4) галлюцинация слота (значение не звучало в истории клиента) → отклонено.

## Вне scope

P0-маршрут; новый верификатор качественных claims; Б7 (автономия — после метрик пилота); финальные числа ширины (до замера Волны 1). При реализации Кодекс делает свою read-only карту `understand()`/планнера/`update_memory_llm` ДО кода.
