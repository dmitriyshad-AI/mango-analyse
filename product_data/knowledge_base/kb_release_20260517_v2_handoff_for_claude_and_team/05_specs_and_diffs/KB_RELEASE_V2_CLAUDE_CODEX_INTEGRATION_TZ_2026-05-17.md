## 2. Решения Дмитрия, которые считаем утверждёнными

Эти решения являются входными условиями ТЗ.

1. Если факт закреплён в документах, бот может его использовать после проверки актуального факта.
2. Маткапитал и налоговый вычет — обычные информационные темы, их не нужно всегда отправлять в `manager_only`. Их можно использовать и для УНПК МФТИ, и для Фотона: образовательная лицензия есть у обоих брендов. При этом клиентский ответ должен оставаться строго в рамках активного бренда.
3. Бот имеет read-only доступ к AMO и Tallanto. Если оплата реально подтверждена в данных, бот может аккуратно сообщить клиенту, что оплата отмечена.
4. По рассрочке текущая позиция Claude считается верной: Т-Банк/Долями для Фотона, УНПК не работает с рассрочкой. Важно: это внутреннее правило для маршрутизации и фильтрации фактов, а не фраза для клиента. Бот УНПК не должен писать клиенту, что “в УНПК рассрочки нет, а в Фотоне есть”. Бот Фотона также не должен консультировать по условиям УНПК.
5. Цены по ЛВШ, Менделеево и регулярным программам считаются взятыми из сайта и документов, но должны попасть в машинный реестр с источниками. ЛВШ Менделеево есть и в УНПК МФТИ, и в Фотоне, но это разные брендовые предложения/смены в разное время. Бот должен использовать только факты активного бренда.
6. Юридическую лексику нужно ослаблять аккуратно: справочный вопрос отдельно, угроза/претензия/суд отдельно.
7. УНПК МФТИ и Фотон для клиента — разные бренды. Это правило Claude верно и должно стать системным правилом бота. Будет отдельный бот/режим для Фотона и отдельный бот/режим для УНПК МФТИ. В одном клиентском сообщении нельзя смешивать бренды, цены, правила, условия оплаты, документы, скидки, рассрочку, контакты или лагерные условия разных брендов.

Дополнительное уточнение Дмитрия от 2026-05-17:

- УНПК МФТИ уже не сотрудничает с Фотоном и официально не знает, что происходит в Фотоне и какие там правила.
- Фотон официально не консультирует по условиям УНПК МФТИ.
- Если клиент бота УНПК МФТИ просит то, чего нет в УНПК, его можно очень аккуратно и завуалированно передать менеджеру/другим организаторам, но нельзя консультировать по Фотону.
- В обратную сторону также можно направлять, но нельзя консультировать по чужому бренду.
- Любая cross-brand информация является внутренней, а не клиентской.
- Внутреннюю историю клиента можно использовать полностью: если в AMO/Tallanto/истории есть данные из обоих брендов, бот может учитывать их для понимания клиента, но в клиентский ответ выводит только факты и формулировки активного бренда.
- Опросник РОПа v5 не импортировать как готовую политику: Дмитрий отдельно исправит его. До исправления использовать только для анализа различий, но не как источник утверждённых фраз.
- `bot_answer_self` можно проектировать уже сейчас, потому что сначала он будет тестироваться на сотрудниках, затем на небольшой лояльной подготовленной группе клиентов. При этом текущий первый пилот всё равно должен поддерживать режим черновиков для менеджера.

- `cross_brand_policy`: `forbidden_for_client`, `manager_handoff_only`, `brand_neutral_allowed`;

- `related_theme_ids`;
- `forbidden_promises`;
- `forbidden_client_mentions`;
- `notes`.

### 6.1. Статусы свежести

Использовать статусы:

- `fresh_verified` — факт можно использовать для точного ответа при соблюдении маршрута;
- `document_verified` — факт есть в документе, но может требовать владельца/период;
- `needs_owner_confirmation` — нужен РОП/бухгалтерия/юрист/маркетинг;
- `dynamic_needs_check` — цена, место, расписание, остаток мест или акция, требующая свежей проверки;
- `internal_only` — только для менеджера/бота, не клиенту;
- `conflicting` — есть противоречие;
- `do_not_use` — не использовать.

### 6.2. Правило допуска к клиентскому ответу

`allowed_for_client_answer=true` допустим только если:

- факт имеет источник;
- факт относится к правильному бренду;
- факт разрешён для текущего `active_brand`;
- факт не содержит условий другого бренда;
- факт не является cross-brand сравнением;
- нет конфликта с другим источником;
- факт не требует live-проверки либо live-проверка выполнена;
- маршрут темы разрешает ответ;
- нет high-risk признака конфликта, угрозы, возврата, жалобы или юридического спора.

Никогда не ставить `allowed_for_client_answer=true`, если факт одновременно объясняет условия УНПК и Фотона в одной клиентской формулировке. Такие факты можно хранить только как `internal_only` или `manager_handoff_only`.

Если один и тот же смысловой факт есть у обоих брендов, например маткапитал, налоговый вычет, справки или похожая скидка, его нужно хранить как два отдельных брендовых факта:

- `brand=foton`;
- `brand=unpk`.

`brand_neutral` использовать только для реально нейтральных фактов без названий брендов, цен, юридических лиц, контактов и условий оплаты конкретного бренда.

## 8. Блок B. Правило брендов УНПК / Фотон

### 8.1. Что сделать

Создать `brand_rules.yaml`:

- УНПК МФТИ и Фотон для клиента разные организации;
- бот должен работать в режиме строго одного активного бренда: `active_brand=foton` или `active_brand=unpk`;
- отдельный бот Фотона не консультирует по УНПК;
- отдельный бот УНПК не консультирует по Фотону;
- в одном клиентском сообщении нельзя упоминать условия, цены, скидки, оплату, документы или правила двух брендов одновременно;
- нельзя писать клиенту фразы вида “в УНПК этого нет, зато в Фотоне есть”;
- нельзя писать клиенту фразы вида “Фотон делает так, а УНПК делает иначе”;
- нельзя предлагать продукт другого бренда как “наш”;
- нельзя объяснять клиенту внутреннюю историю отношений брендов;
- если клиент спрашивает про связь: по умолчанию использовать нейтральную формулировку “Это отдельные организации, по вашему вопросу сориентирую в рамках текущего учебного центра.”; если вопрос чувствительный — передать менеджеру;
- фразу “На данный момент никак, раньше сотрудничали” не делать автоматической до отдельного подтверждения РОПа/юриста;
- если в текущем бренде нет продукта, бот не консультирует по другому бренду, а ставит флаг `brand_switch_manager_review_required` и формирует нейтральный handoff без названия чужого бренда, если это разрешено;
- все фрагменты источников, где УНПК и Фотон сравниваются или описываются вместе, помечать как `cross_brand_mixed`.

### 8.1.1. Разрешённые и запрещённые клиентские формулировки

Для клиента разрешено:

- “Сейчас уточню подходящий вариант у менеджера.”
- “Передам менеджеру, он подскажет, какие есть варианты по вашему запросу.”
- “По этому вопросу лучше сверить актуальные условия, чтобы не ошибиться.”
- “Это отдельные организации, по вашему вопросу сориентирую в рамках текущего учебного центра.” — базовая безопасная формулировка.

Для клиента запрещено:

- “В УНПК рассрочки нет, а в Фотоне есть.”
- “Фотон предлагает X, а УНПК Y.”
- “Мы можем перевести вас в Фотон/УНПК.”
- “У них есть места/цены/условия.”
- “Фотон и УНПК — наши две компании.”
- “На данный момент никак, раньше сотрудничали.” — не использовать автоматически до отдельного подтверждения.
- Любое сравнение условий двух брендов.

Внутренне для менеджера можно показывать:

- `brand_switch_manager_review_required`;
- “запрос не подходит под текущий бренд”;
- “возможен handoff менеджеру без консультации по другому бренду”;
- “не писать клиенту условия другого бренда”.

- Cross-brand фрагменты из текущей базы v1 не попадают в `draft_text`.
- При `active_brand=unpk` слово “Фотон” не появляется в клиентском `draft_text`, кроме разрешённой фразы про прошлое сотрудничество при прямом вопросе клиента.
- При `active_brand=foton` слово “УНПК” не появляется в клиентском `draft_text`, кроме разрешённой фразы про прошлое сотрудничество при прямом вопросе клиента.
- В prompt всегда есть краткое правило бренда, если вопрос касается брендов, цен, оплаты, лагеря или документов.

### 8.4. Известные риски текущей базы v1

Предварительная проверка показала, что в текущих источниках v1 есть фрагменты, где бренды смешаны:

- скрипты, где клиенту предлагают “две компании УНПК МФТИ и Фотон”;
- скрипты “перевода с УНПК на Фотон”;
- таблицы, где в одном тексте идут тарифы Фотона и АНО/УНПК;
- внутренние фразы про то, что УНПК и Фотон раньше сотрудничали;
- смешанные реестры “УНПК Фотон”.

В v2 такие фрагменты нельзя использовать как клиентские ответы. Они должны быть:

- `internal_only`;
- `cross_brand_mixed=true`;
- `forbidden_for_client=true`;
- доступны только как подсказка менеджеру, если нужно.

## 9. Блок C. Маткапитал, налоговый вычет, справки

### 9.1. Новая логика

Маткапитал и налоговый вычет не должны всегда попадать в `manager_only`.

Обе темы доступны для обоих брендов:

- УНПК МФТИ;
- Фотон.

У обоих брендов есть образовательная лицензия. Поэтому нельзя считать, что для Фотона маткапитал или налоговый вычет автоматически требуют handoff к УНПК. Каждый бот отвечает только в рамках своего бренда и своих документов.

Нужно разделить:

1. Обычный информационный вопрос.
2. Вопрос по документам/перечню данных.
3. Нестандартный случай.
4. Юридический спор или гарантия.

### 9.2. Маршруты

Примеры:

- “Можно оплатить маткапиталом?” → `draft_for_manager` или осторожный информационный ответ, если факт утверждён.
- “Какие документы нужны для маткапитала?” → черновик с перечнем данных, если инструкция утверждена.
- “Гарантируете, что СФР одобрит?” → менеджер.
- “Можно получить налоговый вычет?” → информационный черновик.
- “Верните 13% прямо сейчас” → менеджер.
- “Вы обязаны по закону...” → зависит от тона; при угрозе менеджер.

### 9.3. Acceptance criteria

- Информационные вопросы по маткапиталу/налоговому не блокируются грубым keyword guard.
- Информационные вопросы по маткапиталу/налоговому разрешены для обоих брендов при наличии брендового факта.
- Бот Фотона не упоминает УНПК в ответе по маткапиталу/налоговому.
- Бот УНПК не упоминает Фотон в ответе по маткапиталу/налоговому.
- Юридические угрозы по этим темам остаются `manager_only`.
- Бот не обещает одобрение СФР/ФНС.
- Бот не обещает конкретный возврат налога как гарантию.
- Бот явно пишет, что менеджер/бухгалтерия проверит детали, если вопрос требует документов.

## 10. Блок D. Оплата и подтверждение оплаты

### 10.1. Новая логика

Если клиент пишет “я оплатил”, это не всегда high-risk.

Нужно разделить:

1. Клиент сообщил об оплате.
2. Клиент просит подтвердить оплату.
3. В AMO/Tallanto есть совпадающее подтверждение оплаты.
4. В AMO/Tallanto есть расхождение.
5. Данных нет.

### 10.2. Проверка

Использовать только read-only доступ.

Код должен уметь принять в контекст:

- `amo_payment_status`;
- `tallanto_payment_status`;
- `payment_last_seen_at`;
- `payment_amount`, если есть и безопасно показывать;
- `payment_source_confidence`;
- `payment_conflict`.

Если AMO и Tallanto совпадают:

- можно аккуратно сказать: “Вижу, что оплата отмечена.”

Если есть расхождение:

- `manager_only`;
- флаг `payment_source_conflict`;
- чеклист менеджеру: “Сверить AMO и Tallanto”.

Если данных нет:

- бот не подтверждает оплату;
- пишет: “Передам менеджеру на сверку”.

### 10.3. Acceptance criteria

- Бот не подтверждает оплату без данных.
- Бот может подтвердить оплату при совпадении AMO/Tallanto.
- Бот уходит к менеджеру при расхождении.
- Бот не раскрывает внутренние поля AMO/Tallanto клиенту.

## 11. Блок E. Рассрочка

### 11.1. Новая логика

Принять текущую позицию Claude:

- рассрочка через Т-Банк доступна для Фотона;
- `Долями` — для коротких программ;
- УНПК не работает с рассрочкой — только как внутреннее правило, не как клиентская формулировка;
- нельзя обещать одобрение банка;
- условия банка обновлять минимум раз в квартал.

### 11.2. Acceptance criteria

- Бот не предлагает рассрочку УНПК.
- Бот УНПК не пишет клиенту, что в Фотоне есть рассрочка.
- Бот Фотона не пишет клиенту, что в УНПК рассрочки нет.
- Если клиент УНПК спрашивает про рассрочку, бот не консультирует по Фотону, а делает нейтральный handoff менеджеру.
- Бот не обещает одобрение банка.
- Бот не обещает “без переплаты”, если это не подтверждено свежим фактом.
- Бот просит менеджера отправить ссылку.
- Если факт о рассрочке старше допустимого периода, маршрут уходит в `draft_for_manager` с проверкой.

## 12. Блок F. Цены, скидки, ЛВШ, Менделеево

### 12.1. Новая логика

Цены и скидки из Claude считаются актуальной смысловой основой, но должны попасть в реестр фактов с:

- источником;
- брендом;
- продуктом;
- периодом действия;
- владельцем;
- ограничениями.

Особенно:

- регулярные курсы 2026/2027;
- скидки за второй предмет;
- многодетность;
- сотрудники МФТИ;
- “приведи друга”;
- ЛВШ Менделеево;
- городская летняя школа;
- интенсивы ОГЭ/ЕГЭ.

ЛВШ Менделеево:

- есть в обоих брендах;
- факты должны быть разделены по `brand=foton` и `brand=unpk`;
- смены, даты, цены, юридические лица и условия не смешивать;
- если клиент пишет в бот Фотона, бот использует только Фотон-факты по ЛВШ;
- если клиент пишет в бот УНПК, бот использует только УНПК-факты по ЛВШ.

### 12.2. Acceptance criteria

- Бот не называет цену другого бренда.
- Бот не смешивает смены, даты и цены ЛВШ разных брендов.
- Бот не обещает скидку без факта.
- Бот не говорит количество мест в лагере, если факт помечен как динамический и не проверен.
- Бот может дать содержательный ответ по ЛВШ, если факты прошли проверку маршрута.
- Даты, цены и места имеют `dynamic_needs_check`, если быстро меняются.

## 13. Блок G. Юридическая лексика и high-risk

### 13.1. Новая логика

Не каждое слово “закон” или “юридический” должно блокировать.

Разделить:

- справочный вопрос;
- просьба о документе;
- претензия;
- угроза судом;
- возврат;
- жалоба;
- нарушение прав.

### 13.2. Что оставить `manager_only`

Всегда менеджер:

- возврат денег;
- расторжение договора;
- отказ от обучения с требованием денег;
- жалоба;
- претензия;
- Роспотребнадзор;
- прокуратура;
- суд;
- иск;
- угроза негативным отзывом;
- нарушение прав;
- гарантия баллов/поступления;
- требование персональной скидки как условия покупки.

### 13.3. Что не должно автоматически блокироваться

Не блокировать только по ключевому слову:

- “можно ли по закону...”;
- “налоговый вычет”;
- “справка для налоговой”;
- “материнский капитал”;
- “оплатил, проверьте”.

### 13.4. Acceptance criteria

- `detect_high_risk_input_markers` больше не блокирует информационные вопросы.
- Угрозы и жалобы остаются заблокированными.
- Тесты покрывают ложные срабатывания:
  - “возврат к теме”;
  - “жалоба на сайт работает?”;
  - “какая скидка действует?”.
- Тесты покрывают реальные риски:
  - “хочу вернуть деньги”;
  - “подаю в суд”;
  - “жалоба на преподавателя”;
  - “обещайте 90 баллов”.

## 18. Метрики качества

### 18.1. Качество базы

Минимум:

- Claude YAML прочитан без ошибок.
- 100% top-level секций либо импортированы, либо объяснённо отложены.
- Все точные факты имеют бренд или `brand_neutral/internal`.
- Нет клиентских фактов, где одновременно используются УНПК и Фотон.
- Все смешанные брендовые фрагменты помечены `cross_brand_mixed`.
- Все точные факты имеют source reference.
- Нет фактов с `allowed_for_client_answer=true` без маршрута.
- Есть отдельный список конфликтов.

### 18.2. Качество черновиков

На Stage 6:

- 0 выдуманных `theme_id`;
- 0 служебных маркеров в `draft_text`;
- 0 раскрытий “я бот/ИИ/GPT/Claude/Codex”;
- 0 неподтверждённых числовых обещаний;
- 0 ослаблений по возврату/жалобе/угрозе;
- ответы по ценам/лагерю/программе стали содержательнее;
- менеджеру понятно, что проверить.

### 18.3. Качество маршрутов

Ожидаемая логика:

- маткапитал info → не автоматический `manager_only`;
- налоговый info → не автоматический `manager_only`;
- маткапитал/налоговый доступны для обоих брендов при наличии брендового факта;
- ЛВШ Менделеево разделена по брендам;
- оплата подтверждена AMO+Tallanto → можно аккуратно подтвердить;
- оплата конфликтует → `manager_only`;
- возврат → `manager_only`;
- жалоба → `manager_only`;
- суд/иск/претензия → `manager_only`;
- вопрос про бренд → правило УНПК/Фотон.
- при вопросе о связи брендов базовая формулировка: “Это отдельные организации, по вашему вопросу сориентирую в рамках текущего учебного центра.”;
- фраза “раньше сотрудничали” не используется автоматически до отдельного подтверждения.

## 23. Решение наставника

Я бы реализовывал это не как “заменить базу Codex на базу Claude”, а как аккуратное объединение:

- Claude даёт смысл, факты и человекочитаемость.
- Codex даёт проверяемость, маршруты, тесты и защиту.

Самая важная часть — не импорт YAML, а правильный слой допуска:

`факт найден -> факт связан с источником -> факт относится к active_brand -> факт не смешивает бренды -> маршрут темы разрешает -> AMO/Tallanto проверены при необходимости -> только тогда бот использует факт`

При этом “контекст для понимания” шире, чем “текст для клиента”: бот может учитывать всю доступную историю клиента, но перед генерацией клиентского текста обязан отфильтровать факты и формулировки по активному бренду.

Так мы получим сильного помощника для Насти без прежней проблемы “бот отвечает слишком пусто” и без новой проблемы “бот уверенно обещает лишнее”.
