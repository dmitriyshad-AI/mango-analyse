# План фундаментальной пересборки слоя маршрута. 2026-05-30.

Автор: Claude #1. Основание: `TRACE_analiz_over_handoff_2026-05-30.md` + проверка кода HEAD `6f849cff`.
Это план, не реализация. Реализует Кодекс по отдельным ТЗ после решения Дмитрия.

## Принцип (изъян → решение)

Сейчас детерминированный слой плодит параллельные точки ухода к менеджеру (три слоя), а решение
«можно ли ответить» размазано и дублировано. Цель: **модель понимает и черновит; единый тонкий
детерминированный слой ВЕРИФИЦИРУЕТ выход (вето по конкретным опасностям) и принимает маршрут в ОДНОЙ
точке.** Это не переписывание системы, а рефакторинг слоя под существующими ~2000 тестами.

## Что СОХРАНЯЕМ (не трогаем)

- База знаний v6.4 (838 фактов, разделение брендов, фильтры).
- Верификатор фактов как ПРОВЕРЯЮЩИЙ (`verify_output` / `_hard_check` находки) — он нужен; меняем не
  его, а реакцию на находки.
- P0-спеки, разделение брендов.
- Тёплый слой — по коду он безопасен, в уход не роняет (отвергнутый тёплый кандидат оставляет
  исходный черновик, маршрут не меняет, `dialogue_contract_pipeline.py:1383-1395`).
- ~2000 тестов — это регресс-щит пересборки.

## Что ПЕРЕСОБИРАЕМ

1. **Единая точка решения маршрута (Router).** Свести три слоя (ранние гейты `run_pipeline`,
   ветки `_hard_check`, пост-цепь гвардов + `route_permission`) в одну функцию
   `decide_route(verification, p0, brand, autonomy)`. Поздние слои не перебивают ранние — решение одно.
2. **Инвертировать умолчание.** Сейчас: «к менеджеру, если хоть что-то не подтверждено дословно».
   Стало: «автоответ по умолчанию; к менеджеру — только если верификатор нашёл КОНКРЕТНУЮ опасность
   (бренд / P0 / выдумка / чужой scope)». Детерминированный слой = вето, а не маршрутизатор.
3. **Семантическое покрытие вместо дословного.** Заменить `_claim_supported_by_facts` (дословное
   вхождение подстроки, `subscription_llm.py:7740`) на проверку через верификатор/семантику. Это
   убирает массу ложных уходов на живых фразах. Главный единичный рычаг.
4. **Расширить спасение автоответа.** `_can_autonomously_replace_failed_draft` (2714) спасает только
   `fact_grounding`. Часть других находок (например предмет, взятый из ретривнутого факта) безопасно
   лечится автоматически — расширить набор спасаемых, не ослабляя бренд / P0 / мета.
5. **Схлопнуть дубли гвардов.** `unstated_subject` / `unconfirmed_operational_specificity` /
   `unsupported_promise` делят дословную проверку покрытия — после п.3 их можно объединить в один
   семантический верификатор-вето вместо трёх параллельных регэксп-гвардов.

## Этапы (волнами; у каждой тест входа/выхода + ОБЯЗАТЕЛЬНЫЙ негативный контроль)

- **Этап 0 — щит.** Заморозить негативные тесты безопасности: бренд / P0 / выдумка / мета не должны
  ослабнуть ни на одном следующем шаге. Без щита пересборку не начинать (правило #4). Безопасно, до пилота.
- **Этап 1 — Router без смены поведения.** Собрать все точки демоушена в одну `decide_route()`,
  поведение 1:1 со старым, под тестами. Чистый рефакторинг, регрессий быть не должно. Безопасно, до
  внутреннего пилота.
- **Этап 2 — семантическое покрытие** (п.3). Меняет поведение → прогон + регрейд. Цель: over_handoff
  вниз, hard_gate=0 держится. Перед клиентским пилотом.
- **Этап 3 — инверсия умолчания + расширение спасения** (п.2, п.4). Самый рисковый, обязательный
  негативный контроль на каждой правке. Прогон + регрейд.
- **Этап 4 — схлопнуть дубли гвардов** (п.5). Уже на сведённой логике.

## Тайминг относительно пилота

- Внутренний пилот на сотрудниках — может идти ПАРАЛЛЕЛЬНО (терпит шероховатости, даёт продуктовый
  сигнал «нужны ли черновики вообще»).
- Этапы 0-1 — безопасны, можно до / во время внутреннего пилота.
- Этапы 2-3 (меняют поведение) — перед КЛИЕНТСКИМ пилотом, через прогон + регрейд.
- Точечные правки S4 / C2 (отдельное ТЗ) идут раньше и независимо — они закрывают выдумки, не маршрут.

## Оценка сложности (честно)

Не однодневно: маршрут размазан по трём слоям, `_hard_check` вызывается в ~7 точках пайплайна,
дословная проверка покрытия — общий узел нескольких гвардов. Но и НЕ переписывание с нуля: 8000 строк
`subscription_llm` не выкидываем — мы СВОДИМ логику маршрута в одну точку и заменяем один проверочный
узел, под существующими тестами. Риск управляем: щит (Этап 0) + поэтапный прогон.

## Развилка, которую надо закрыть ДО пилота (настройка, не пересборка)

`route_permission` гасит автоответ, если автономия не включена в контексте (`_autonomy_enabled`,
читает `policy.allow_autonomous / enabled / bot_answer_self_enabled / context.autonomy_enabled`,
`subscription_llm.py:6484`). Подтвердить: в пилотном контексте этот флаг выставляется для безопасных
тем? Если нет — даже идеальный бот уйдёт в черновики. Это конфигурация запуска, проверить отдельно.
