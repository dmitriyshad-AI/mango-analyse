Автор: Клод 1. Это образец, КАК доменное правило переезжает из лазаньи в чистый слой. Источник —
`_installment_safe_template` (subscription_llm 67 LOC) + 6 текст-констант (127-187). Прочитано по сырью,
включая скрытые исключения (на них указывал Кодекс). Если формат ок — по нему мигрируем остальные семейства.

## Как сейчас (лазанья) — что смешано в одной функции
`_installment_safe_template` в 67 строках делает СРАЗУ: бренд-правило (только foton) + распознавание
намерения по ключевым словам (рассроч/долями/частями/помесяч/банк/переплат) + проверку intent-плана
(payment_method → skip) + различение со скидкой («не про рассрочку») + scope-роутинг (банк/помесячно/
Долями/лагерь) + выбор одного из 6 захардкоженных ТЕКСТОВ. Это и есть «понимание+scope+текст+безопасность»
в одном куске — то, что разносим по слоям.

## Целевой контракт правила (слой 2 = ДАННЫЕ, не текст)

```yaml
rule_id: installment
title: Оплата частями / рассрочка / Долями
# 1. ВХОД (намерение) — распознаёт PLANNER (модель), НЕ regex в этом правиле.
#    Правило лишь объявляет, на какое намерение оно отвечает.
intent: payment_installment            # planner отдаёт intent + uncertainty
intent_subvariants: [general, bank_decision, monthly, dolyami, no_overpayment, camp]
# 2. НУЖНЫЕ ФАКТЫ (ключи KB; composer пишет ТОЛЬКО из них)
required_fact_keys:
  - installment.foton.periods            # [6,10,12] месяцев
  - installment.foton.dolyami            # сервис Долями
  - installment.foton.scope              # очные/онлайн/ЛВШ/ЛШ/обычные курсы
  - installment.unpk.absent              # у УНПК банковской рассрочки НЕТ
  - installment.unpk.alternatives        # помесячно/семестр/год
  - discount.unpk.semester_year          # семестр 10%, год 14%
# 3. ДАННЫЕ ПРАВИЛА (структура, НЕ готовые предложения)
data:
  foton:
    available: true
    periods_months: [6, 10, 12]
    services: [dolyami]
    applies_to: [ochno, online, lvsh, lsh, regular_courses, camp]
    regular_also: [monthly, semester]
    approval: bank_or_service_decides    # Т-Банк/Долями решает одобрение
    no_overpayment_for_client: true
  unpk:
    bank_installment_available: false    # КЛЮЧЕВОЕ бренд-различие
    alternatives: [monthly, semester, year]
    semester_discount_pct: 10
    year_discount_pct: 14
# 4. SCOPE / БРЕНД-ЭФФЕКТ
brand_split: true                        # ответ РАЗНЫЙ по active_brand (Фотон есть / УНПК нет банковской)
# 5. УСЛОВИЯ БЛОКИРОВКИ → менеджер (не отвечать правилом)
blocking_conditions:
  - payment_dispute            # «оплатил, нет доступа», спор → P0 менеджер (гейт)
  - org_payment                # «оплата от организации» → менеджер (нет факта)
  - asks_exact_percent_or_parts # точное число частей/% → не обещать, менеджер оформит
# 6. ROUTE-ЭФФЕКТ
route_effect: bot_answer_self            # если факт есть и не P0; иначе manager (решает гейт/planner)
# 7. ТЕКСТ-ЭФФЕКТ
text_effect: composer_generates_from_data   # НЕ хардкод; composer пишет живо из data+subvariant
#   подвариант выбирает planner (bank_decision/monthly/dolyami/no_overpayment/camp/general)
# 8. СКРЫТЫЕ ИСКЛЮЧЕНИЯ (СОХРАНИТЬ — иначе регрессия, предупреждал Кодекс)
preserve_exceptions:
  - intent=discount && "не про рассрочку/долями/оплату частями" → НЕ отвечать про рассрочку (это про скидку)
  - intent in {payment_method, invoice_monthly} → НЕ подменять способ рассрочкой
  - банковский перевод/по счёту ≠ рассрочка ≠ Долями — не подменять (PAYMENT_BANK_TRANSFER случай)
  - «банк/одобрение/решение» → подвариант bank_decision (не обещать одобрение заранее)
  - предоплата ≠ рассрочка (разные вопросы)
```

## Распределение по слоям (инверсия)
- **Planner (понимание):** отдаёт `intent=payment_installment` + subvariant + uncertainty. Заменяет
  keyword-детекцию (рассроч/долями/банк…) из 67-строчной функции.
- **Слой 2 (правило-данные):** структура `data` выше (периоды, сервисы, бренд-различие, скидки). НИ ОДНОГО
  готового предложения. Если факт меняется (скидка год 14→15%) — правится ДАННОЕ, не текст.
- **Composer:** пишет живой ответ из `data` под выбранный subvariant. Шесть FOTON_*_SAFE_TEXT констант
  ИСЧЕЗАЮТ из кода (их содержание = композиция из data).
- **Гейт (безопасность):** ловит blocking_conditions (payment_dispute→manager, обещание одобрения→block),
  не подмену способа, бренд-утечку (рассрочка в боте УНПК как банковская → block).

## Обязательные НЕГАТИВНЫЕ тесты (доказывают перенос без регрессии)
1. УНПК: «есть рассрочка?» → «банковской нет; помесячно/семестр/год, скидка 10/14%» — НЕ обещать
   банковскую рассрочку (бренд-различие).
2. Фотон: «оплатил, доступа нет» (спор) → менеджер, НЕ ответ про рассрочку (blocking payment_dispute).
3. «не про рассрочку, я про скидку» → ответ про скидку, НЕ про рассрочку (preserve discount-исключение).
4. «банковский перевод по счёту» → НЕ подменять рассрочкой (способ оплаты).
5. «через банк одобрят?» → bank_decision: не обещать одобрение заранее.
6. «предоплата сколько внести?» → НЕ ответ про рассрочку (предоплата ≠ рассрочка).
7. ПОЗИТИВ: Фотон «можно частями?» → 6/10/12 + Долями, живо, route=answer_self.

## Критерий приёмки эталона
Правило `installment` = ДАННЫЕ (yaml/структура) + ссылки на факты + список исключений + негатив-тесты.
Composer генерирует текст. 6 текст-констант и 67-строчная функция УДАЛЕНЫ (их роль: planner+data+composer+
gate). Если правило нельзя выразить данными без потери — это сигнал, что часть уходит в client_safe факт KB,
а не в код. По этому эталону мигрируем: скидки → цены → лагерь → расписание → пробное → платформа.
