## КОММИТ 1 — достройка продающих сигналов (anxiety + unmet_need + readiness)
Та же машинерия, что Волна 1 (understand()-детекция + gen-композиция + гейт), 0 новых вызовов — расширяем
`selling` ещё тремя полями. **Флаг: `TELEGRAM_A_SELLING_SIGNALS_FULL` (default off)** — гейтит ТОЛЬКО три новых
сигнала. Базовый gen Волны 1 (objection=price/exit_signal) живёт под `TELEGRAM_A_SELLING_MODE=gen` и в
кумулятивных прогонах ВСЕГДА включён (он — проверенный фундамент). То есть конфигурация `C1` =
gen-Волна1 + 3 новых сигнала.

- **anxiety: bool** — тревога/недоверие («вы точно нормальные?», «не обманете?», подтекст «боюсь, что зря
  деньги»). Приём: гашение тревоги — снять риск честно (пробное ПРИ ФАКТЕ; «есть лицензия на образовательную
  деятельность»), спокойный тон. NEG: НЕ обещать результат; номер лицензии/реквизиты НЕ светить; гарантию не давать.
- **unmet_need: str** — невысказанная потребность («у ребёнка тройка по физике», «он стеснительный»). Приём:
  услышать и предложить релевантное ИЗ фактов, даже без прямого вопроса. NEG: НЕ диагностировать ребёнка; НЕ
  обещать оценку («исправим на пятёрку»); предложение строго из фактов бренда. ⚠ **САМЫЙ РИСКОВЫЙ сигнал волны:**
  модель извлекает свободный текст потребности И строит предложение — для него gen-проверка обещаний/давления
  критична, и `unmet_need` как free-text НЕ должен попадать в клиентский текст дословно (только как триггер
  приёма). Читаю по сырью внимательнее всего.
- **readiness: exploring|comparing|ready** — «куда платить / как записаться» = ready → чёткий следующий шаг из
  факта enrollment. NEG: шаг только из факта; нет факта → честный менеджер; не выдумывать порядок оплаты.

Общий NEG коммита: P0/возврат/жалоба перебивает ЛЮБОЙ приём (selling только `if not risk_signals`, как d57bff06);
нейтральная реплика → все сигналы none/false (детекция не гиперактивна); бренд один; числа из факта; гейт
страхует, fallback на det-пол.

## КОММИТ 2 — покрытие C: ТОЛЬКО КОД, составные вопросы из СУЩЕСТВУЮЩИХ фактов (KB НЕ трогать)
Из B2-аудита (`stepB_quality_remnants` audit pack — список rule→None; Клод его НЕ читал, он в непримонтированном
.phase12: Кодекс работает по нему и в отчёте ПЕРЕЧИСЛЯЕТ, что именно закрыл). Расширить обработку СОСТАВНЫХ/
«примерных» вопросов («два предмета», «примерно?», «онлайн и очно вместе») — чтобы правило отвечало из УЖЕ
ИМЕЮЩИХСЯ в снимке фактов по обоим подвопросам, а не возвращало None.
**КРИТИЧНО — это чисто КОД (логика подвопросов). KB-снимок НЕ редактировать, новые факты НЕ дописывать.** Если
факта для подвопроса в снимке нет → честный «второй уточнит менеджер», НЕ выдумывать. Дозаполнение KB фактами —
ОТДЕЛЬНАЯ задача через сборку релиза + semantic review (не этот коммит, не ждём Tallanto-данные здесь). Флаг:
`TELEGRAM_A_COVERAGE`.
NEG: составной с фактами обоих подвопросов → ответ из обоих; составной, где второго факта НЕТ → первый из факта +
честный менеджер по второму (НЕ выдумка); бренд раздельно; числа из факта; не-составной НЕ сломан; P0 цел.
Консервативно: если список B2 мал/пуст — Кодекс отчитывается и коммит минимальный (не выдумывать работу).

## КОММИТ 3 — нить диалога: удержание ТОЛЬКО продающих слотов (узко, НЕ рефактор памяти)
Узкая задача, НЕ общий рефактор `dialogue_memory`: гарантировать перенос между ходами ровно продающих слотов —
**предмет, класс, формат, active_brand** — чтобы возражение по цене на ходе N → готовность на ходе N+3 дали
чёткий шаг БЕЗ переспроса уже названного. Закрыть КОНКРЕТНЫЕ места, где эти слоты теряются на эллипсисе/смене
темы. Память НЕ переписывать — трогать только перенос этих слотов. Флаг: `TELEGRAM_A_THREAD`.
NEG: диалог возражение → пауза/смена темы → готовность держит предмет/класс/формат; P0 в середине перебивает; нет
переспроса названного клиентом; нить НЕ тянет устаревший слот после явной смены (новый класс/предмет вытесняет
старый). Если перенос слотов уже корректен — Кодекс подтверждает прогоном NEG, коммит минимальный (не выдумывать).

## Флаги и кумулятивный прогон M1 (зачем за флагами)
Все три флага (`TELEGRAM_A_SELLING_SIGNALS_FULL`, `TELEGRAM_A_COVERAGE`, `TELEGRAM_A_THREAD`) — **default OFF**.
Значит прод-дефолт (без флагов) = проверенный gen-Волна1 (`TELEGRAM_A_SELLING_MODE=gen`, objection/exit); новые
модули opt-in, пока кумулятивный прогон их не подтвердит. Базовая линия во ВСЕХ конфигурациях — gen-Волна1.
M1 одним ночным сеансом гоняет 3 кумулятивные конфигурации:
- **Конфиг 1** = база + `SELLING_SIGNALS_FULL` (3 новых сигнала);
- **Конфиг 2** = Конфиг 1 + `COVERAGE`;
- **Конфиг 3** = Конфиг 2 + `THREAD` (сложные многомодульные диалоги на нить).
Один бандл, не три. Клод читает по ДЕЛЬТЕ (что добавил каждый слой) + теги по модулю. Атрибуция багов
БЕЗОПАСНОСТИ — уже из per-commit pytest+NEG (прогон для неё не нужен); прогон — для качества/тона/нити по сырью.

## Тест выхода (per commit, КРИТИЧНО)
- Каждый коммит: `pytest` зелёный + NEG модуля (детерминированно, атрибуция на уровне кода).
- Бандл собирается, флаги переключают каждый модуль независимо.
- Полный `pytest` после C3 зелёный.
- Прогон НЕ запускаешь — M1 (сценарии и бандл соберёт Клод).

- Продажу НЕ отдавать модели целиком; числа НЕ в обход гейта; НЕ обещания/давление; НЕ диагноз ребёнка; НЕ
