# D7 Phase 1 Design — Claude как первоклассный мозг

Дата: 2026-06-04
Ветка: codex/d7-claude-first-class-brain
База: 176e6336 Quality block 6 tune X2 tone playbook
Статус: дизайн без правок кода

## Короткий вердикт

План Клода в целом правильный: Claude нужно подключать не только как мозг бота, но и как возможный мозг клиента, судьи, памяти, смысловой проверки и продажного слоя. При этом дефолт должен остаться Codex/GPT, а честный замер лучше делать в смешанной схеме: бот на Claude, судья на GPT/Codex.

Главная поправка к ТЗ: текущий раннер Claude использует `--bare`, а локальная проверка показала, что `claude --bare -p` сейчас не видит подписку и падает с `Not logged in`. Поэтому перед A/B нужно сделать режим запуска Claude, совместимый с подпиской Дмитрия, либо явно настроить API/helper. Иначе инфраструктура будет казаться сломанной, хотя сама модель доступна.

Вторая поправка: проблему "служебных лесов" нельзя считать только промптовой. Если такой текст дошёл до клиента, нужен не только model-robust prompt fix, но и постоянная регрессионная проверка/guard на фразы вроде `уточнение по текущей теме`, `Тема: предмет`, технические detail/handoff-слоты.

## Что подтверждено по реальному коду

Файл раннера: `scripts/run_telegram_dynamic_client_sim.py`.

Текущая архитектура на базе `176e6336`:

- `CodexJsonModel` есть в раннере и используется для большинства ролей.
- `ClaudeJsonModel` и `ClaudeCliRunner` уже есть.
- `--bot-mode` поддерживает `codex`, `claude`, `fake`.
- `--client-mode` поддерживает только `codex`, `fake`.
- `--judge-mode` поддерживает только `codex`, `fake`.
- `--memory-mode` поддерживает только `codex`, `fake`, `off`.
- `--semantic-mode` поддерживает только `codex`, `fake`, `off`.
- `--selling-mode` сейчас означает не модель, а режим поведения: `gen` или `det`; при `gen` используется `CodexJsonModel`.

Итого: Claude сейчас реально подключён только к роли бота. Для полного Claude-прогона нужно расширить все остальные роли.

## Что подтверждено по Claude CLI

Команды проверки:

- `claude --version` -> `2.1.162 (Claude Code)`.
- `claude --help` подтверждает:
  - флаг `--model`;
  - пример полного имени модели `claude-opus-4-8`;
  - флаг `--effort`;
  - уровни `low`, `medium`, `high`, `xhigh`, `max`.

Дополнительные безопасные пробы без инструментов:

- `claude-sonnet-4-6` с `--effort xhigh` успешно отработал.
- `claude-opus-4-8` был распознан CLI и появился в `modelUsage`; tiny-проба остановилась только из-за слишком низкого лимита бюджета `$0.05`, а не из-за неизвестной модели.
- `claude --bare -p ...` вернул `Not logged in`; это критично для текущего `ClaudeCliRunner`, потому что он строит команду с `--bare`.

Вывод: модельные строки `claude-sonnet-4-6` и `claude-opus-4-8` можно считать доступными для дальнейшей проверки, но раннер надо привести к режиму, который видит подписку.

## Где я согласен с планом Клода

1. Claude должен идти через тот же выходной контроль, что и GPT. Это уже частично так для роли бота: `ClaudeCliRunner` подменяет CLI-вызов, но результат всё равно проходит через текущий провайдер и защитные проверки.

2. Дефолт должен остаться Codex/GPT. Claude должен быть включаемым режимом, а не заменой по умолчанию.

3. Смешанная схема нужна для честного замера. Если бот и судья оба Claude, судья может не увидеть типовые ошибки того же семейства моделей.

4. Первый A/B надо делать без правки промптов. Так мы измеряем модель, а не одновременно модель и новые инструкции.

5. Если Claude печатает служебный текст вроде `уточнение по текущей теме. Тема: ...`, править нужно не только под Claude, а так, чтобы стало лучше и для GPT: заменить технический слот на естественную формулировку.

## Где я уточняю или спорю

1. `claude --help` не перечисляет все модели. Он подтверждает механизм `--model` и пример `claude-opus-4-8`, но Sonnet подтверждён отдельной короткой пробой, а не самой справкой.

2. `--bare` в текущем виде рискован. Он полезен для чистого, дешёвого запуска без лишней памяти, но сейчас не видит подписку. На фазе 2 нужно либо убрать `--bare` для subscription-режима, либо добавить явный `--claude-auth-mode`.

3. `--selling-mode` лучше не превращать напрямую в `claude`. Сейчас это переключатель логики `gen/det`, а не выбор модели. Чище добавить отдельный выбор мозга для продажного слоя, например `--selling-brain codex|claude`, чтобы `det` остался детерминированным режимом.

4. Opus `max` может быть слишком дорогим и медленным для массового A/B. Практичнее порядок: Sonnet `xhigh`, Sonnet `max`, Opus `xhigh`, Opus `max` только если первые три не закрыли проблему или нужен верхний предел качества.

5. В артефактах замеров надо фиксировать не только итоговый verdict, но и полную конфигурацию ролей: режим, модель, усилие, timeout, provider, судья, `all_brains`, вариант промпта. Иначе A/B плохо воспроизводим.

## Основные риски

### 1. Самооценка одной и той же моделью

Если бот Claude и судья Claude, оценка может быть слишком доброй к собственному стилю и ошибкам. Такой прогон полезен для автономности, но не для финального качества. Для честного сравнения рекомендую: бот Claude, судья GPT/Codex, затем отдельный сырой аудит Клодом #1.

### 2. Два набора промптов

Если сделать отдельные промпты под Claude и под GPT, мы начнём сравнивать не мозги, а разные системы инструкций. Поэтому промпт-правки только общие: убрать технические плейсхолдеры, сделать слот естественным, усилить запрет на служебный текст.

### 3. Задержка и стоимость

Полный Claude-прогон на всех ролях, особенно Opus `xhigh/max`, будет сильно медленнее. Для ночных замеров допустимо, для интерактивного SaaS-режима нет. В SaaS-контуре нужен выбор мозга по роли и тарифу, но сейчас достаточно CLI-флагов.

### 4. Инфраструктурная ошибка маскируется под ошибку модели

Из-за `--bare` можно получить `Not logged in` и решить, что Claude не работает. Это не проблема модели, а режим запуска.

### 5. Утечка служебного текста

Сейчас в коде есть точка, которая может породить фразу вида `уточнение по текущей теме. Тема: ...`: `src/mango_mvp/channels/dialogue_contract_pipeline.py`, функция `_compose_topic_question`. Это не обязательно единственный источник, но это первый подозреваемый слот для Phase 2B.

### 6. Утечка ключей через ошибки

Код пробрасывает хвост `stderr` в исключения/infra-error. Перед логированием нужно маскировать `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `Bearer ...`, `sk-...` и похожие секреты. Это не связано только с Claude, но при добавлении нового CLI-мозга риск растёт.

## Рекомендуемый план Phase 2

### Шаг 0. Починить режим запуска Claude

До A/B добавить безопасный выбор режима запуска:

- режим для подписки Дмитрия: без `--bare`;
- режим для API/helper: с `--bare`, если он настроен;
- тест, что команда собирается без ключей в аргументах и логах.

### Шаг A. Claude для всех ролей

Добавить Claude-ветки в сборщики:

- клиент;
- судья;
- память;
- смысловая проверка;
- продажный слой.

Добавить `--all-brains codex|claude`, но не делать Claude дефолтом.

Для продажного слоя не ломать `gen/det`: лучше отдельный флаг мозга, потому что `det` означает отсутствие LLM.

### Шаг B. Тесты инфраструктуры

Покрыть:

- все роли умеют строить `ClaudeJsonModel`;
- `--all-brains claude` выставляет все LLM-роли в Claude;
- mixed-конфигурация `bot=Claude`, `judge=Codex` возможна;
- fake/off/det режимы не ломаются;
- Claude-ответ проходит через тот же выходной контроль.
- ошибки CLI маскируются перед попаданием в отчёты;
- служебные фразы вида `уточнение по текущей теме` и `Тема: предмет` не проходят в клиентский текст.

### Шаг C. Дешёвый A/B

Сначала без правки промптов:

- GPT/Codex baseline;
- Claude Sonnet `xhigh`;
- Claude Sonnet `max`;
- Claude Opus `xhigh`;
- Claude Opus `max`, если оправдано.

Оценивать по сырью: ушёл ли служебный текст, нет ли пустых ответов, нет ли выдумок, не сломался ли бренд/P0.

В каждый run artifact добавить конфигурацию ролей: мозг, модель, усилие, timeout, судья, prompt variant, количество LLM-вызовов, задержку и инфраструктурные ошибки.

### Шаг D. Только если проблема осталась — model-robust prompt fix

Если Claude продолжает печатать служебные конструкции, править конкретный слот, а не переписывать весь промпт.

Обязательная проверка после правки: GPT не стал хуже.

## Финальный Phase 1 verdict

К Phase 2 переходить можно, но первым подпунктом должен быть не A/B, а исправление/настройка способа запуска Claude CLI под подписку Дмитрия. Без этого `ClaudeCliRunner` может падать на `--bare`, и мы получим ложную инфраструктурную ошибку.

Дальше реализация должна идти в этой отдельной ветке, без правок конвейера качества, базы знаний и бизнес-гейтов.
