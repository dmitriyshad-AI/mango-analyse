# ТЗ — слой типизированных смысловых ролей (следующий корневой слой после контекст-удержания)

Автор: Claude (Cowork, независимый ревьюер). Дата: 2026-05-25.
Статус: референс-реализация готова и протестирована, передаётся Кодексу на внедрение и замер.

## 0. Откуда это ТЗ

После round-4 (PASS 5 / PWN 16 / FAIL 3, «отправил бы» 20.8%, hard-gate 2) Кодекс
дал верную корневую диагностику: оставшиеся 3 FAIL — НЕ опасные (нет утечки бренда,
нет пропуска реального P0, нет выдуманной цены), а смысловые. Корень — пять вещей:

1. распознавание формата/темы держится на КАСКАДЕ частных `if has_marker(...)` —
   правим один кейс, всплывает соседний (классический «крот»);
2. возврат типизируется растущим списком узких regex (`PRESALE_REFUND_*`) —
   фразы вроде «передумаю до начала, деньги вернут?» в список не попали → ложный P0;
3. способ оплаты (рассрочка/долями) и источник оплаты (маткапитал) лежат на одной
   ветке → баг «рассрочка при маткапитале»;
4. на реплику с несколькими вопросами бот отвечает на один (мультитема не разведена);
5. шаблон/rewriter иногда затирает уже корректный ответ (шаблон не как fallback).

Это ТЗ закрывает п.1–4 кодом (распознаватель ролей + политика) и п.5 правилом для
пост-слоёв. Пятый «генеративный» аспект (как сформулировать многотемный ответ) —
инструкция в промпте генерации, описана в §5.

## 1. Принцип (не нарушать)

- Качество — из ГЕНЕРАЦИИ. Распознаватель ролей даёт СИГНАЛЫ (оси), а не финальный
  ответ. Парафразы, которые ключи не ловят, добирает модель, видя всю историю + факты.
- «ЗАМЕНЯТЬ, а не ДОБАВЛЯТЬ». Новый распознаватель ролей должен ВЫТЕСНИТЬ старый
  каскад в `conversation_intent_plan._fact_scope_constraints` и узкие regex
  `PRESALE_REFUND_*`, а не лечь поверх. Измеримо: число частных `if`-веток
  распознавания и число узких refund-regex в диффе должно УПАСТЬ.
- Слои ВЫЧИТАЮТ. Политика (`decision_policy`) не пишет текст — она задаёт план
  (какие темы закрыть, маршрут, запрещённые связки, можно ли шаблон).
- P0 консервативно. Этот слой НЕ ослабляет внешний P0; он только корректно типизирует
  предпродажный возврат (benign) и добавляет свой dispute-P0.

## 2. Что считается корнем (оси ролей)

Реплика клиента описывается набором ОРТОГОНАЛЬНЫХ осей (а не одним интентом):

| Ось | Значения | Назначение | Какой баг закрывает |
|---|---|---|---|
| `training_format` | online / ochno / vyezd_camp | формат ОБУЧЕНИЯ | «очно» в «тОЧНОй» |
| `enrollment_vs_recording` | enroll / recording | смысл слова «запись» | «запись урока» ≠ «записаться» |
| `transfer_sense` | group / money / manager | смысл слова «перевод» | перевод денег ≠ на менеджера ≠ в группу |
| `payment_method` | dolyami / rassrochka / edinorazovo | СПОСОБ оплаты | — |
| `payment_source` | matkap / tax_deduction / sertifikat | ИСТОЧНИК оплаты | «рассрочка при маткапитале» |
| `asks_place` | bool | вопрос про адрес/город | место ≠ формат |
| `refund_frame` | none / presale_policy / dispute | тип возврата | ложный P0 на предпродажный вопрос |
| `topics` | список | ВСЕ темы реплики | мультитема |

Ключевой сдвиг: способ и источник оплаты — РАЗНЫЕ оси; место, формат и оформление —
РАЗНЫЕ оси. Их разъединение убирает целый класс перекрёстных ошибок без отдельных
патчей под каждое слово.

## 3. Типизация возврата (заменяет PRESALE_REFUND_* regex)

Возврат типизируется по двум общим признакам, а не по списку фраз:

- `dispute` (P0 → менеджер), если есть ТРЕБОВАНИЕ («верните», «требую», «хочу возврат»,
  «расторгнуть договор») ИЛИ признак уже состоявшейся оплаты («оплатил», «списали»,
  «после оплаты», «заключили договор»);
- `presale_policy` (безопасный ответ о политике), если возврат назван в
  ГИПОТЕТИЧЕСКОЙ/УСЛОВНОЙ/ДО-ОПЛАТНОЙ рамке («если», «вдруг», «до начала»,
  «до оплаты», «передумаю», «не понравится», «какие условия/правила») БЕЗ требования и
  без оплаты;
- `none`, если возврат не упомянут.

Так «А если передумаю до начала, деньги вернут?» становится `presale_policy` БЕЗ
внесения фразы в список — потому что нет ни оплаты, ни требования, есть условная рамка.
Это и есть «общим механизмом, а не по кейсам».

## 4. Политика поверх ролей (`decision_policy.build_answer_plan`)

На вход — роли + внешний P0 + флаг «есть ли уже содержательный ответ». На выход — план:

- `answer_topics` — ВСЕ темы, которые ответ обязан закрыть (мультитема);
- `route` — `manager_only` при P0, иначе `bot_answer_self`;
- `p0_required` — внешний P0 ИЛИ refund dispute;
- `forbidden_pairs` — связки, которые в одном ответе называть нельзя
  (`matkap+installment`: маткапитал — источник, рассрочку/долями в тот же ответ не подаём);
- `template_allowed` — True ТОЛЬКО когда нет содержательного ответа и нет тем/P0
  (шаблон строго как fallback; при наличии ответа шаблон может лишь уточнить стиль).

## 5. Генерация (инструкция, не код)

В промпт генерации добавить (поверх уже внедрённого контекст-слоя):
- «Если в реплике несколько вопросов (`answer_topics` длиной > 1) — ответь на КАЖДЫЙ,
  не выбирай один»;
- «`forbidden_pairs` — это связки, которые НЕ предлагать вместе (например, оплату
  маткапиталом и рассрочку в одном ответе)»;
- «`refund_frame=presale_policy` — это вопрос ПОЛИТИКИ возврата до покупки: ответь
  спокойно о порядке, без личных обещаний и без эскалации»;
- числа — только из подтверждённых фактов по теме (правило не меняется).

## 6. Что НЕ делает этот слой (честные пределы)

- Это НЕ семантическая модель. Распознавание осей — по границам слов (ключи), поэтому
  парафраз без ключевого слова (пример: «сколько занятий в неделю» как вопрос
  расписания) ось `schedule` может не поднять. Это допустимо: на такие случаи отвечает
  генерация, видя историю и факты. Слой ролей отвечает за СТРУКТУРНУЮ развязку
  (не перепутать оси, не защёлкнуть ложный P0, показать все темы, запретить кривую
  связку), а не за полноту распознавания каждой формулировки.
- Поэтому метрика приёмки ниже — про СТРУКТУРНЫЕ классы, не про recall парафраз.

## 7. Приёмка на round-5 (свежий слепой holdout, disjoint c dev)

Hard-gate (обязательно):
- 0 `p0_mishandled` (реальные возвраты-споры/legal/жалобы по-прежнему уходят менеджеру);
- 0 `brand_leak`, 0 `fabrication`.

Рост качества относительно round-4 (целевое):
- «отправил бы без правок» ≥ +5 п.п. (round-4 = 20.8%);
- предпродажные возвраты (`refund_frame=presale_policy` в тесте) НЕ эскалируются в P0
  (round-4 кейс `benign_refund_22` должен стать «отправил бы»/PWN, не FAIL);
- многотемные реплики получают ответ на ВСЕ вопросы (падение `ignored_question`);
- 0 случаев «маткапитал + рассрочка в одном ответе»;
- без роста `over_handoff` и без роста ложного P0.

Анти-«крот» (проверю по диффу):
- число частных `if`-веток в `_fact_scope_constraints` УПАЛО (старый каскад заменён
  вызовом распознавателя ролей, а не дополнен);
- узкие refund-regex (`PRESALE_REFUND_POLICY_RE`, `PRESALE_REFUND_CONTEXT_RE`,
  `BENIGN_HYPOTHETICAL_REFUND_RE`) заменены типизацией `refund_frame`, а не оставлены
  «спящими» рядом;
- старых наивных подстрочных проверок не прибавилось.
