# Независимый ревью: LLM-классификатор намерения (Путь B). Второй Claude, 2026-05-28

**Вердикт (1 строка): PASS_WITH_NOTES — направление здравое (гибрид + режим «бот старается лучше»), но как написано сейчас это net-регресс по `false_handoff` и по детерминизму P0; обязательны 5 правок ниже, и главная — не плодить второй слой намерения, а подключить LLM к УЖЕ существующему `ConversationIntentPlan`.**

Ключевой факт, который меняет рамку ревью: в коде УЖЕ есть выделенный слой намерения — `ConversationIntentPlan` (`conversation_intent_plan.py:100` `build_conversation_intent_plan`), правиловый, с полями `primary_intent`, `intent_confidence`, `refund_frame` (`none/presale_policy/dispute`), `risk_signals`, `route_bias`, `payment_method/source`. Он строится в `telegram_pilot_context_builder.py:160` и прокидывается в `context["conversation_intent_plan"]`, откуда его уже читают `classify_answer_safety` (`:177`), драфтер-промпт (`draft_prompt_builder.py:230`), `answer_quality_rewriter`, гайрды. То есть предлагаемый «новый LLM-классификатор» во многом дублирует существующий компонент.

## Ответы на 7 вопросов

**1. Архитектурная корректность.** Сам по себе шаг «семантическая классификация до `classify_answer_safety`» корректен и не ломает межходовые механизмы — НО только если соблюдён принцип монотонности к безопасности: `p0_latch` и regex-база читаются ВНУТРИ `classify_answer_safety` из `dialogue_memory`, и новый слой не должен их обходить. Риск в текущем описании: режимы эскалации Уровня 2 решаются ДО `classify_answer_safety`; если LLM скажет «neutral», а латч активен (реальный P0 с прошлого хода) — нельзя позволить LLM перебить латч. Правило: финальный P0 = `regex_baseline OR p0_latch OR llm_escalation` (объединение, только в сторону безопасности). Как описано — не гарантировано.

**2. Output schema.** Переусложнена. `primary_intent` + три булевых `*_detected` + `requires_p0_escalation` + `soft_handoff_recommended` — это «суп из булевых», который провоцирует противоречивые комбинации (например `legal_threat_detected=true` при `requires_p0_escalation=false`). Свернуть до: `intent` (enum) + `escalation` (`hard_p0|soft_handoff|none`) + `confidence` + `rationale`. `claim_now` vs `hypothetical_refund` — РАЗЛИЧАТЬ (это и есть водораздел Hard P0 vs none), не сливать. Не хватает одного поля: `refund_subject: our_center|third_party|none` — именно оно закрывает кейс «налоговая вернёт деньги» (third_party → не P0). И: `intent`/`confidence` уже существуют в `ConversationIntentPlan` как `primary_intent`/`intent_confidence` — не вводить второй словарь, переиспользовать имена.

**3. Failover.** Таймаут/невалидный JSON → regex-only — нормально. Но **`confidence < 0.5 → soft handoff` — ошибка**: это ровно та over-эскалация, которую мы и чиним (false_handoff). На неуверенности нельзя по умолчанию передавать менеджеру — это теряет клиента на безобидных вопросах. Правильный дефолт: низкая уверенность → откат на правиловый `ConversationIntentPlan` (regex-база), а не handoff. Упущено: (а) детерминизм — LLM недетерминирован, а тут он на пути P0; нужен `temperature=0` + кэш по нормализованному сообщению, иначе получаем §11.4 (флаппинг верификатора) на САМОМ опасном слое; (б) one-way ratchet: LLM может только повышать защиту, не понижать; (в) shadow-режим перед доверием (сравнить LLM vs regex на N днях, прежде чем дать ему решать).

**4. Точка вставки.** «До `classify_answer_safety` отдельным параметром» — субоптимально. Правильное место — **продьюсер `ConversationIntentPlan` в `telegram_pilot_context_builder` (там, где сейчас `build_conversation_intent_plan`)**: LLM-вариант плана, заполняющий тот же `intent_view`, с правиловым `build_conversation_intent_plan` как fallback. Тогда `classify_answer_safety` НЕ трогаем — он уже читает план из контекста. Это убирает новый параметр и второй словарь намерения; всё downstream-потребление (драфтер, quality_rewriter, гайрды) получает улучшение бесплатно.

**5. Latency и параллелизм.** Частично. `retrieve_facts` зависит от `needed_fact_keys` из Понимания (LLM #1), а классификатор — от `client_message` + истории, входы независимы → параллелить МОЖНО по входу. Но структурный блокер: пайплайн синхронный (последовательные вызовы `understand_fn`/`draft_fn`); параллелизм требует asyncio/threads в рантайме, которого сейчас нет в этом пути. И второй нюанс: классификатор на критическом пути к маршруту (P0 решает, идти ли в retrieve/draft), поэтому «спрятать» его за retrieve полностью не выйдет — экономия latency меньше, чем кажется. Реалистично: temp=0 + кэш дают больше, чем параллелизм.

**6. Регрессия на 4 эталона.** Главный риск. Предложенный Уровень-1 regex УЖЕ их валит: список «суд/прокуратура/двойное списание/жалобу писал» НЕ содержит «верните деньги»/«требую возврат» — то есть `P0a_refund_claim` на Уровне 1 НЕ ловится и зависит от LLM (Уровень 2). Если LLM поднят и ошибётся (классифицирует как `hypothetical` с confidence>0.8) — реальный P0 понижается. Страховка обязательна: **сохранять текущую полную regex-базу (`REFUND_RE`+`not benign`) как нижний порог; финальное решение — объединение, LLM может только добавлять P0/handoff, не снимать.** `P0b_complaint` — аналогично. Без этого правила слой = регресс по безопасности.

**7. Альтернатива (Путь A / B / третий).** Не A (расширять LLM #1 Понимания — связывает безопасность с фактологией, обе деградируют вместе) и не B-как-описано (новый вызов + новая схема = второй словарь намерения, дрейфующий от `ConversationIntentPlan`). **Третий вариант (рекомендую): «LLM-backed `ConversationIntentPlan`»** — отдельный вызов (изоляция Пути B сохраняется), но он ЭМИТИТ существующую схему плана (`primary_intent/intent_confidence/refund_frame/risk_signals`), а правиловый `build_conversation_intent_plan` — это failover. Один слой, одна схема, всё текущее потребление работает без правок.

## Что упустил Claude #1

1. **Существование `ConversationIntentPlan`.** Проект уже имеет слой намерения с нужными полями и проводкой — предложение строит параллельный. Это главный риск дублирования и расхождения двух таксономий.
2. **Недетерминизм на P0-слое.** LLM в пути P0 без temp=0/кэша = §11.4 на самом дорогом месте. Решения P0 станут плавать между ходами.
3. **Регрессивный дефолт.** `confidence<0.5 → soft handoff` увеличивает false_handoff — противоречит цели Волны 1.
4. **Эмпирика устарела.** Часть «regex не работает» примеров УЖЕ работает (см. `phase11_16_subpoints_code_recheck.md`): «если не понравится…» → `refund_frame=presale_policy` (11.6); «налоговая вернёт» → tax-ветка в `_refund_frame`; «не ответили нормально» → `semantic_non_p0` и, вероятно, не входит в `COMPLAINT_RE` (11.13). До постройки LLM стоит ИЗМЕРИТЬ, сколько серых кейсов реально падает сегодня — обоснование «нужен LLM» частично неактуально.
5. **Хорошее, что стоит сохранить:** режим «Bot tries harder» при `emotion_target=bot` — это реальный новый класс (школьники тестируют бота), его правилами не закрыть. Это сильнейшая часть предложения.

## Финальный список правок (для PASS)

1. **Не новый параллельный классификатор, а LLM-вариант `ConversationIntentPlan`** с правиловым `build_conversation_intent_plan` как failover; точка — `telegram_pilot_context_builder`, схема — существующая. `classify_answer_safety` не трогать.
2. **Монотонная безопасность:** финальный P0 = `full_regex_baseline OR p0_latch OR llm_escalation`. LLM может только повышать защиту. Сохранить полную `REFUND_RE`-базу, не сужать Уровень 1 до криминальных маркеров.
3. **Failover-дефолт:** `confidence<0.5` → правиловый план (НЕ soft handoff). Soft handoff — только при явном `escalation=soft_handoff` от LLM с confidence порогом.
4. **Детерминизм:** `temperature=0` + кэш по нормализованному `(client_message, last_3_turns)`; shadow-режим (LLM vs regex) N дней до передачи решающей роли.
5. **Схема:** свернуть булев-суп до `intent` + `escalation` + `confidence` + `rationale` + добавить `refund_subject (our_center|third_party|none)`; переиспользовать имена из `ConversationIntentPlan`.

С правками 1-5 — это улучшение, а не регресс; режим «tries harder» оставить. Без правок 2 и 3 — внедрять нельзя (регресс по P0 и по false_handoff).
