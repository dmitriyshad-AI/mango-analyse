## 0. Принцип (одной фразой)
УМНЫЙ слой (LLM) ПОНИМАЕТ диалог и черновит; ДЕТЕРМИНИРОВАННЫЙ слой ВЕРИФИЦИРУЕТ выход на жёсткие правила.
Сейчас наоборот: keyword-классификатор решает смысл, LLM лишь заполняет прозу и не может поправить ошибку выше. Это и есть корень багов.

## 1. Пайплайн (на каждый ход; доступна ВСЯ история диалога + active_brand из канала)
```
[0] P0 пре-гейт (детерминированный, дёшево) ── спор/жалоба/юр/чарджбек → manager_only + сухой P0 → STOP
        │ (не P0)
[1] LLM #1 «ПОНЯТЬ» → Контракт ответа (структура, см. §2). Вход: вся история + active_brand + КАТАЛОГ fact_keys (без значений).
        │  Правило: называет нужные ключи, НЕ выдаёт значения. is_p0/manager_only → STOP в менеджера.
[2] Ретривал (детерминированный) ── needed_fact_keys → значения из бренд-скоупного склада confirmed_facts → facts[]
[3] LLM #2 «ЧЕРНОВИК». Вход: вся история + контракт + facts[] + тон-гайд + правила. → draft_text
[4] Выходной ВЕРИФИКАТОР (детерминированный, см. §4) ── PASS → отдать; FAIL → собрать findings
[5] LLM #3 «X2-РЕМОНТ» (только при FAIL) ── draft + конкретные findings + facts[] → правка → ПЕРЕ-верификация
        │  ≤2 попыток; если всё ещё FAIL → сухой безопасный fallback «уточнит менеджер по нужной детали»
[6] Отдать черновик менеджеру
```
Нормальный путь: 2 вызова модели (#1 понять, #2 черновик). При ремонте: 3+. P0 — 0 вызовов генерации.

## 2. Контракт ответа (выход LLM #1) — единый источник смысла
Строгий JSON. Поля:
- `active_brand`: foton|unpk (берётся из канала, в контракт для самопроверки).
- `current_question`: пересказ ИМЕННО текущего вопроса клиента (что он хочет СЕЙЧАС).
- `composite_subquestions`: [] — если сообщение составное, разложить на под-вопросы.
- `continued_topics`: [] — темы, которые клиент продолжает из прошлых ходов.
- `denied_topics`: [] — темы, которые клиент ЯВНО отрицает («не про возврат» → ["refund"]). Критично для негейшна.
- `switched_topics`: [] — на что клиент переключился.
- `known_slots`: {class, subject, format, brand_context...} — что уже известно из диалога (не переспрашивать).
- `needed_fact_keys`: [] — какие ключи фактов нужны для ответа (ТОЛЬКО ключи из каталога; НЕ значения).
- `forbidden_substitutions`: [] — что нельзя подставлять как ответ (соседние scope: office_hours, schedule, другой бренд, denied_topics).
- `answerability`: "answer_self" | "manager_only" — можно ли ответить автономно (есть факт, не P0) или только менеджер.
- `is_p0`: bool + `p0_reason` — дублирует детерминированный пре-гейт как второй сигнал.
- `confidence`: 0..1 — уверенность понимания (низкая → осторожнее, чаще handoff).

Инвариант: контракт НЕ содержит значений фактов (цен/дат/адресов). Только ключи и смысл. Значения появляются на шаге [2] из склада.

## 3. Стадии — точные спеки
### [0] P0 пре-гейт (детерминированный backstop)
Существующие REFUND_RE/PAYMENT_DISPUTE_RE/legal/complaint. Реальный спор/требование/пост-оплатный возврат/чарджбек/юр.угроза →
manager_only_p0, сухой P0-текст, STOP. Это страховка: не полагаемся ТОЛЬКО на LLM #1, что он заметит P0.

### [1] LLM #1 «Понять»
- Промпт: «Ты разбираешь диалог продаж учебного центра (бренд {active_brand}). Верни СТРОГО JSON-контракт (§2).
  Пойми, что клиент хочет СЕЙЧАС; учти отрицания («не про X» → X в denied_topics, НЕ в текущем вопросе); held-контекст;
  составные вопросы. Назови needed_fact_keys ТОЛЬКО из каталога ниже. Значения фактов НЕ придумывай и НЕ пиши.
  Если это спор/жалоба/возврат-требование/юр.угроза — is_p0=true. Каталог fact_keys: {...}.»
- Парсер: валидирует схему; при кривом JSON / неизвестных ключах — деградация в безопасный handoff (не падать).

### [2] Ретривал по ключам
- Из бренд-скоупного склада (client_safe_facts_<brand>) достаём значения для needed_fact_keys → facts[] (key→value, valid_until).
- Ключи не найдены / факта нет → пометить missing[]. Это сигнал для черновика: честный узкий handoff по этой детали.
- Кросс-бренд ключи игнорируются (бренд-скоуп жёсткий).

### [3] LLM #2 «Черновик»
- Вход: вся история + контракт + facts[] (+ missing[]) + тон-гайд (живой, из плейбука) + правила:
  «Отвечай ТОЛЬКО на current_question/composite_subquestions. Используй ТОЛЬКО facts[] для конкретики (числа/даты/адреса).
  Не подставляй forbidden_substitutions и denied_topics. Нет факта (missing) → честно: эту деталь уточнит менеджер, без подмены соседним.
  Не раскрывай природу ИИ. Не упоминай другой бренд. Живо, по-человечески, один следующий шаг.»

### [4] Выходной верификатор — см. §4.
### [5] X2-ремонт — см. §5.

## 4. Выходной верификатор (детерминированный, крисп, бесплатный)
Проверяет draft_text; любой провал → findings + на ремонт/fallback. Это правильное место детерминизма (бинарные правила):
- **brand_leak**: нет токенов чужого бренда (foton: «унпк/мфти/unpk»; unpk: «фотон/цдпо/црдо/foton/долями»).
- **fact_grounding**: каждое число/цена/дата/адрес в draft ∈ facts[] (нормализация пробелов в числах). Иначе — выдумка.
- **forbidden_scope**: draft не отвечает из denied_topics/forbidden_substitutions (переиспользовать scope-логику).
- **meta_leak / ai_disclosure**: нет служебных фраз (fact_id, «без служебных пометок», «я ИИ/бот» по своей инициативе).
- **p0_in_output**: нет обещаний возврата/гарантий результата/спорной оплаты.
Замечание про не-числовую выдумку: чистый детерминизм её ловит хуже. Опционально — дешёвый LLM-чек «всё ли утверждаемое ∈ facts[]» как доп. слой (за отдельным флагом).

## 5. X2-ремонт (критика-направленный)
- Вход: draft + КОНКРЕТНЫЕ findings верификатора + facts[]. Промпт: «Исправь ровно это: <finding>. Не вводи фактов вне facts[]. Маршрут/смысл не меняй.»
- Выход ПЕРЕ-верифицируется. ≤2 попытки. Если не чисто → сухой безопасный fallback по нужной детали («точную X уточнит менеджер»).
- Гарантия: X2 не может сделать ответ менее безопасным — его выход всегда проходит §4.

## 6. Флаги, что заменяется/упрощается
- Всё за флагом `TELEGRAM_DIALOGUE_CONTRACT_PIPELINE` (default OFF), старый путь — дефолт до проверки.
- В новом пути НЕ источники смысла: keyword `_TOPIC_PATTERNS`, машина held_state (её роль — память контекста — забирает LLM #1, читая всю историю).
  Остаются: P0 пре-гейт (backstop), бренд-скоуп склада, выходные верификаторы. Меньше «мини-мозгов», нет споров классификаторов.

## 7. Инварианты безопасности (НЕ нарушать ни в одной фазе)
1. Контракт называет КЛЮЧИ, значения — только из склада (no fabrication на входе).
2. P0 — детерминированный пре-гейт ОБЯЗАТЕЛЕН (не только LLM-флаг).
3. Заземление-верификатор включён ОДНОВРЕМЕННО со свободой генерации (свобода без заземления = выдумка).
4. Бренд-скоуп жёсткий: склад фактов и проверки — только активный бренд.
5. Нет факта → честный узкий handoff, НЕ подмена соседним.
6. X2-выход всегда ре-верифицируется; при неудаче — безопасный fallback, не «протолкнуть».

## 8. Приёмка и замер

- Цель класса мискласса: на round-5 диалогах, что падали (perevod_04, recording_03, schedule_20, refund_presale_02) — нет мисклассов и нет дампа соседнего факта.
- Качество: походовая send_unedited_rate (+ мягкая категория «мелкая правка», когда внедрим) — рост против базы; тон вверх.
- Сравнение «параллельно»: гнать новый путь (флаг ON) и старый (флаг OFF) на одном наборе → видно, реально ли инверсия убирает класс и поднимает качество.

## 11. ДОПОЛНЕНИЯ ПОСЛЕ СВЕРКИ С КОДЕКСОМ (2026-05-26) — приняты, усиливают дизайн
Claude+Codex сошлись по архитектуре независимо. Пять усилений к §2/§4/§9:

**11.1 Контракт = план ответа ПО ПОДВОПРОСАМ (а не плоские списки).** Для каждого под-вопроса:
`{subquestion, answerable: self|manager, needed_fact_keys[], next_step}`. P0-под-вопросы → manager, остальные могут отвечаться, если факт есть.

**11.2 У каждого слота памяти — ИСТОЧНИК (anti-fabrication контекста).** `known_slots` хранит не значение, а `{value, source}`:
например `{class: {value: "9", source: "client_turn_1"}}`. Правило: бот НЕ утверждает слот (класс/предмет/формат/цель) без источника = слова клиента или факт.
Это закрывает «приписал класс/предмет из воздуха» (recording_03).

**11.3 Верификация СМЫСЛОВЫХ утверждений, не только чисел (ключевое против не-числовой выдумки).** Числа/бренд/P0/мета — детерминированно (крисп).
А утверждения вроде «по будням», «есть пробное», «запись сохраняется», «можно забронировать» — отдельным ДЕШЁВЫМ LLM-проходом «верность фактам»:
«каждое конкретное утверждение в черновике ∈ facts[] ИЛИ из слов клиента — да/нет + что лишнее». Не прошло → ремонт/узкий хендофф.
Честно: открытую проверку утверждений детерминированно не сделать; это второй фокусный LLM-вызов на ПРОВЕРКУ (узкая задача, отдельный от генератора), плюс детерминированный пол.

**11.4 Настоящая история диалога с ролями/ходами (требование интеграции).** LLM #1 получает реальную последовательность «клиент:…/бот:…/клиент: нет, я про другое»,
а не приблизительно восстановленный список. От этого напрямую зависит понимание поправок/отрицаний.

**11.5 Цель — ЗАМЕНИТЬ старый каскад, а не жить поверх него навсегда (уточнение §9).** Сейчас новый путь проходит через ВСЕ старые гейты — это правильная
ВРЕМЕННАЯ страховка (Фаза 1). Но старые гейты надо разделить на два класса:
 - ВЕРИФИКАТОРЫ БЕЗОПАСНОСТИ (бренд-сепарация, P0, выдумка/заземление, sanitize) → ОСТАВИТЬ как выходной слой;
 - ПОНИМАЮЩЕ-ПЕРЕПИСЫВАЮЩИЙ каскад (классификаторы тем, шаблоны, intent-guards, humanity-rewriter-маршрутизаторы) → ПОСТЕПЕННО ВЫПИЛИТЬ в новом пути.
Иначе старый каскад продолжит перетирать чистый ответ контракта (та же болезнь «умный понял — глупый переписал»). Фаза 3 = оставить только верификаторы безопасности.

**Итоговый контур (Claude+Codex):** полная история → смысловой контракт (план по подвопросам + слоты с источником) → факты по ключам из склада →
ответ на все подвопросы → проверка КАЖДОГО утверждения (детерминизм для крисп + LLM-верность для смысла) → финальный ответ. Старый каскад заменяется, а не наслаивается.

## 12. СЛОЙ ЧЕЛОВЕЧНОСТИ (приоритет: СНАЧАЛА по сути и заземлённо, ПОТОМ тепло)
Человечность — часть контракта, не заплатка в конце. Тёплый-но-не-туда ответ хуже корректного-чуть-сухого.
1. **client_state в контракте.** LLM #1 определяет ситуацию/тон клиента: тревожная мама / торопится / сравнивает цену / сомневается / раздражён / просто уточняет.
   Это настраивает РЕГИСТР ответа (и подсказывает приём из плейбука: снятие риска, гашение тревоги, честная срочность, чёткий шаг).
   Оговорка: client_state меняет тон, но бот НЕ называет эмоцию вслух («вижу, вы переживаете»), если это не явно уместно — иначе навязчиво.
2. **Стилевые примеры в черновик (= R1).** LLM #2 получает 2-4 коротких живых примера по ТЕМЕ и БРЕНДУ — как стиль, НЕ источник фактов. Нужно углубить корпус gold/few-shot до 3-5 вариативных примеров на тему (сейчас ~1 → штамп).
3. **Свободная генерация + заземление фактов** — ядро схемы (§3-§4): живо пишем, конкретику заземляем.
4. **МЯГКАЯ проверка ФОРМЫ (≠ жёсткая безопасность).** Отдельно от верификатора безопасности: ответил ли прямо / шаблонный зачин / повтор прошлого хода / есть ли один мягкий следующий шаг / канцелярит.
   Это наш `humanity_linter`. Срабатывание → ТРИГГЕР X2-тепла, НЕ жёсткий блок (в отличие от бренд/факт/P0, которые блокируют).
5. **X2 = слой «сделай теплее».** Только ПОСЛЕ проверки смысла/фактов; меняет форму, не содержание; выход ре-верифицируется (бренд/факты/P0).

Порядок: понять (контракт+client_state) → факты → черновик (свободно, со стилевыми примерами) → ЖЁСТКИЙ верификатор безопасности → МЯГКАЯ проверка формы → X2-тепло (ре-верификация) → ответ.
Это Фаза 2 (после того, как Фаза 1 докажет, что инверсия убирает класс мискласса). Сначала «по сути», потом «тепло».
