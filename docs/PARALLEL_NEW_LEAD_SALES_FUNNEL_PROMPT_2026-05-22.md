## Жесткие границы

Работа только read-only.

Нельзя:

- править код;
- править базу знаний;
- менять prompt/draft provider;
- менять текущие Telegram-боты;
- запускать live-ботов;
- писать в AMO, CRM, Tallanto;
- запускать ASR или Resolve+Analyze;
- менять `stable_runtime`;
- добавлять факты в client-safe слой;
- смешивать Фотон и УНПК в клиентских сценариях.

Все новые идеи, факты, формулировки и материалы помечай как `candidate`, пока они не утверждены Дмитрием/РОПом.

## Главная цель

Спроектировать путь нового клиента:

`сайт -> Telegram -> первый вопрос -> полезный ответ -> уточнение -> подбор -> запись -> оплата`

Нужно понять:

- где бот должен держать инициативу;
- какие 2-4 вопроса он должен задавать;
- где он должен отвечать сам;
- где он должен передавать менеджеру;
- какие материалы помогут продаже;
- какие метрики покажут, что бот реально двигает клиента к покупке.

## Важный стиль

Бот должен звучать не как формальная организация и не как нейросеть.

Нужный стиль:

- теплый;
- дружелюбный;
- живой;
- уверенный;
- без давления;
- без канцелярита;
- с ощущением, что мы правда хотим помочь родителю и ребенку.

Продажа = консультативная помощь. Не "впихнуть", а честно подобрать хороший вариант и мягко вести к следующему шагу.

## Что нужно сделать

### 1. Карта воронки

Создай карту этапов:

1. Новый клиент пишет первое сообщение.
2. Бот определяет бренд.
3. Бот определяет тему.
4. Бот отвечает на прямой вопрос.
5. Бот собирает минимум данных для подбора.
6. Бот предлагает следующий шаг.
7. Бот передает менеджеру, если нужен человек.
8. Менеджер получает краткую сводку.
9. Клиент доходит до записи/пробного/оплаты.

Для каждого этапа укажи:

- цель этапа;
- что делает бот;
- что нельзя делать;
- какой следующий шаг считается успешным;
- какие риски есть.

### 2. Уточняющие вопросы

Составь матрицу 2-4 безопасных вопросов для новых лидов.

Раздели по брендам:

- Фотон;
- УНПК МФТИ.

И по ситуациям:

- клиент спрашивает цену;
- клиент спрашивает расписание;
- клиент спрашивает "что выбрать";
- клиент спрашивает про пробное;
- клиент спрашивает про лагерь;
- клиент сомневается;
- клиент пишет мало информации;
- клиент хочет "просто узнать".

Важно: бот не должен устраивать анкету из 10 вопросов. Максимум 1-2 вопроса за один ответ.

### 3. Где бот держит инициативу

Опиши сценарии, где бот должен сам двигать диалог:

- предложить подобрать вариант;
- уточнить класс и предмет;
- предложить формат;
- предложить запись/пробное/заявку;
- мягко передать менеджеру с уже собранной сводкой.

Отдельно опиши, как делать это без давления.

### 4. Где передавать менеджеру

Составь правила передачи менеджеру:

- P0/high-risk;
- нет проверенного факта;
- нужна проверка мест/расписания/оплаты;
- нужен договор/документ/возврат;
- конфликт или жалоба;
- клиент просит действие, которое бот не должен выполнять сам.

Для каждого правила дай пример короткого клиентского ответа и внутренней сводки для менеджера.

### 5. Метрики

Составь список метрик, которые реально показывают пользу:

- клиент получил прямой ответ;
- клиент получил следующий шаг;
- бот не переспросил уже известное;
- бот собрал класс/предмет/цель/формат;
- клиент оставил данные;
- клиент дошел до менеджера;
- клиент дошел до записи/пробного;
- доля ответов, готовых к отправке без правки;
- доля слишком шаблонных ответов;
- доля опасных тем, верно переданных менеджеру;
- среднее время ответа.

Раздели метрики на:

- ежедневные;
- недельные;
- метрики качества;
- метрики продаж.

### 6. Материалы для поддержки продаж

Предложи список материалов, которые стоит подготовить:

- мини-презентация Фотона;
- мини-презентация УНПК;
- карточки курсов;
- FAQ;
- "как выбрать курс";
- "что будет после заявки";
- скрипты ответов на возражения;
- сравнение форматов внутри одного бренда;
- материалы для лагерей;
- материалы для родителей, которые сомневаются.

Для каждого материала укажи:

- кому нужен;
- где использовать;
- какую проблему закрывает;
- какие факты нужны;
- риск смешения брендов.

### 7. Тестовые диалоги

Подготовь 20-30 сценариев диалогов для проверки воронки:

- 10 Фотон;
- 10 УНПК;
- 5-10 сложных/смешанных.

Каждый сценарий:

- кто клиент;
- первое сообщение;
- что бот должен понять;
- что бот должен спросить;
- какой следующий шаг должен получить клиент;
- когда нужен менеджер;
- что считается провалом.

### 8. Если доступен Claude CLI

Если в среде доступен Claude CLI, используй его как независимого смыслового ревьюера, но не соглашайся автоматически.

Процесс:

1. Сначала сам подготовь черновик воронки.
2. Передай Claude CLI только результат и критерии проверки, без своей защиты.
3. Попроси Claude найти слабые места: где бот будет давить, где будет слишком формальным, где не доведет до продажи, где риск смешения брендов.
4. Сравни свою позицию и позицию Claude.
5. Сохрани:
   - где вы согласны;
   - где спорите;
   - какое финальное решение принял ты и почему.

Если Claude CLI недоступен, создай отдельный файл `CLAUDE_REVIEW_REQUEST.md`, чтобы Дмитрий мог отдать его Claude Cowork вручную.
