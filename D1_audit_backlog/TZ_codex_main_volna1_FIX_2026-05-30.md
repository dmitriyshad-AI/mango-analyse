## Падение 2 — operational расписание (баг в КОДЕ, цель не достигнута)

`test_volna_peresborki_operational_guard_uses_retrieved_fact_metadata_semantically` (стр. 3983):
claim «обычно есть вечерние группы» при факте «…доступны группы в вечернее время» → `manager_only`
вместо автоответа.

Корень (проверил напрямую): `_fact_match_anchors("обычно есть вечерние группы")` = пусто →
`if not claim_anchors: return False`. У текстового расписание-claim нет конкретных якорей
(числа/дата/бренд), а словоформа «вечерн**ие**» не даёт `condition:evening` (факт «вечерн**ее**» даёт).
Передача факта и verified-путь работают — корень только в извлечении якорей.

**Фикс (в коде, `_fact_match_anchors`):** извлекать смысловые якоря расписания по ОСНОВАМ слов, чтобы
claim и факт давали сопоставимые якоря независимо от словоформы:
- `вечерн*` → `condition:evening`; `утренн*` → `condition:morning`; `дневн*` → `condition:day`;
  `выходн*` / `суббот*` / `воскресен*` → `condition:weekend`; `будн*` / `будень` → `condition:weekday`;
  при необходимости иные условия, реально встречающиеся в фактах расписания.

Тогда subset-логика заработает: `{condition:evening}` (claim) ⊆ `{condition:evening, brand:unpk}` (факт)
→ подтверждён → автоответ.

**Обязательный негативный контроль (правило #4):**
- claim про ДРУГОЕ время/день (утренние vs вечерние; будни vs выходные) НЕ подтверждается фактом про
  другое — subset не выполнится → блок;
- существующий `..._blocks_wrong_scope_schedule_claim` остаётся зелёным;
- promise-гвард (числа / проценты / баллы) НЕ ослаблять — конкретные якоря обязательны
  (NEG «25%» / «25 баллов» = не подтверждён сохранить).
