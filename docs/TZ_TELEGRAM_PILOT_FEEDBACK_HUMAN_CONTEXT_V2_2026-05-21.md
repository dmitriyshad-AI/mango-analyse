# ТЗ: Telegram pilot feedback + человечность ответов + умное использование контекста клиента

Дата: 2026-05-21

## 1. Контекст

В проекте уже запущены два отдельных Telegram-бота:

- бот Фотона;
- бот УНПК МФТИ.

Текущий статус:

- база знаний v6.3 подключена;
- бренды разделены;
- P0-защиты включены;
- есть матрица автономности;
- бот умеет читать AMO/Tallanto/local context в read-only режиме по кодовой фразе с телефоном;
- быстрые проверки пройдены:
  - `v6 autonomy fast`: 32/32;
  - `v5 P0 fast`: 10/10;
  - точечные тесты по боту/КБ: 137 passed;
  - общий сбор тестов: 1604 tests collected.

Но после первых тестов появился важный продуктовый сигнал от коллег:

> Бот отвечает нейронно. Хочется, чтобы он отвечал более по-человечески. Если бот уже авторизовал пользователя и нашёл данные о нём в CRM/Tallanto, он не должен заново спрашивать то, что уже знает. Нужны больше сообразительность, живое общение и меньше шаблонности, но без потери безопасности.

Это правильное замечание. Сейчас бот уже стал безопаснее, но следующая задача — сделать его полезнее и естественнее.

## 2. Главная цель

Сделать первый контур пилота, который одновременно:

1. собирает понятные метрики и ошибки по тестированию сотрудников;
2. помогает оценить пользу бота, а не только безопасность;
3. делает ответы менее шаблонными и более человеческими;
4. использует уже известные данные клиента из AMO/Tallanto/local history;
5. не просит повторно данные, которые уже надёжно найдены;
6. не раскрывает служебные данные CRM/Tallanto клиенту;
7. сохраняет P0-защиту: возврат, жалобы, юридические угрозы, спорная оплата, брендовые риски.

## 3. Что значит “по-человечески”

Это не значит “болтливо”, “панибратски” или “как нейросеть”.

Хороший ответ бота должен:

- звучать как живой вежливый менеджер;
- отвечать на конкретный вопрос, а не рядом с ним;
- быть коротким, но не пустым;
- не начинаться каждый раз одинаково;
- не злоупотреблять словами “уточню”, “передам”, “менеджер свяжется”;
- не повторять очевидное;
- не спрашивать то, что уже известно;
- мягко вести клиента к следующему шагу;
- не давить на продажу;
- не обещать неподтверждённые цены, даты, скидки, места, возвраты и решения.

Плохой “нейронный” ответ:

- “Спасибо за обращение! Менеджер свяжется с вами.”
- “Для точного ответа уточните класс, предмет, формат, город, телефон, имя ребёнка.”
- “Я помогу подобрать оптимальный образовательный продукт.”
- “Ваш вопрос очень важен для нас.”

Хороший ответ:

- “Да, по 5 классу в Фотоне есть очные мини-группы. Сейчас могу сориентировать по формату и стоимости, а если подскажете предмет, помогу выбрать ближайший вариант.”
- “Если пишете по Даниилу, я вижу контекст. Повторно данные присылать не нужно — лучше уточните, речь про расписание или оплату?”
- “По возврату сразу передам вопрос ответственному сотруднику. Дополнительно ничего в чат присылать не нужно.”

## 4. Что значит “не спрашивать известное”

Если бот по кодовой фразе или по будущей авторизации нашёл клиента в AMO/Tallanto/local history, он должен различать:

### 4.1. Данные можно использовать для понимания

Например:

- имя родителя;
- имя ученика;
- класс;
- текущий курс;
- бренд;
- формат обучения;
- наличие сделки;
- статус оплаты;
- Tallanto-группа;
- последние обращения;
- история звонков.

Но эти данные нельзя механически раскрывать клиенту.

### 4.2. Данные можно использовать в ответе осторожно

Допустимо:

- “Если вопрос по Даниилу, я вижу контекст.”
- “По вашему текущему обучению лучше передам менеджеру, чтобы он проверил детали.”
- “Похоже, речь про действующее обучение. Тогда не буду спрашивать данные заново.”

Недопустимо:

- “Я вижу в AMO сделку 48645046.”
- “В Tallanto у вас контакт id 12345.”
- “У вас было 85 звонков.”
- “В CRM указано, что вы отказались из-за цены.”
- “Ваша оплата прошла 12 мая на сумму X”, если это не подтверждено надёжным payment context и не разрешено политикой.

### 4.3. Что бот не должен спрашивать повторно

Если значение надёжно найдено:

- не спрашивать имя родителя;
- не спрашивать имя ребёнка;
- не спрашивать телефон;
- не спрашивать класс, если он есть и не конфликтует;
- не спрашивать бренд/центр, если канал уже брендовый;
- не спрашивать “вы уже учитесь у нас?”, если контекст это показывает.

Но если есть риск семьи/нескольких детей/дублей:

- бот не должен угадывать;
- нужно уточнить мягко:
  - “Правильно понимаю, вопрос по Даниилу?”
  - “У вас в истории несколько учеников, уточните, пожалуйста, по кому вопрос?”

## 5. Жёсткие границы безопасности

Нельзя в рамках этого ТЗ:

- писать в AMO, Tallanto, CRM;
- менять `stable_runtime`;
- запускать ASR;
- запускать Resolve+Analyze;
- отправлять сообщения через личный Telegram-аккаунт менеджера;
- подключать широкий клиентский поток;
- раскрывать клиенту внутренние ID, статусы, source_id, JSON, safety_flags, CRM/Tallanto technical fields;
- называть цены, скидки, даты, места, возвраты или решения без проверенного client-safe факта;
- смешивать Фотон и УНПК в одном клиентском ответе;
- автоматически отвечать по P0-темам.

P0-темы всегда выше человечности и продажности:

- возврат денег;
- жалоба;
- конфликт;
- юридическая угроза;
- претензия;
- суд;
- Роспотребнадзор/прокуратура;
- спорная оплата;
- персональные данные в рискованном контексте;
- брендовая неопределённость.

## 6. Что нужно реализовать

### Блок A. Журнал и метрики пилота

Создать скрипт:

`scripts/build_telegram_public_pilot_feedback_report.py`

Вход:

- `.codex_local/telegram_pilot_bots/logs/`;
- дата или диапазон дат;
- бренд: `foton`, `unpk`, `all`;
- выходная папка.

Выход:

- `pilot_messages.csv`;
- `pilot_messages.jsonl`;
- `pilot_summary.json`;
- `pilot_summary.md`;
- `semantic_review_queue.csv`;
- `regression_candidates.csv`;
- `employee_review_sheet.csv`;
- `daily_pilot_report_YYYY-MM-DD.md`.

Метрики:

- входящие сообщения;
- ответы бота;
- среднее время ответа;
- медианное время ответа;
- доля автономных ответов;
- доля `manager_only`;
- доля fallback;
- доля ответов с CRM/Tallanto context;
- доля ответов, где бот не спросил уже известные данные;
- доля ответов, где бот всё же спросил известные данные;
- доля “слишком шаблонных” ответов;
- доля “полезных” ответов по автоматической эвристике;
- очередь ручной проверки.

### Блок B. Очередь смысловой проверки

В `semantic_review_queue.csv` класть:

- все P0/high-risk ответы;
- ответы с ценами, датами, процентами, сроками;
- ответы с CRM/Tallanto context;
- ответы, где бот спросил имя/телефон/класс при наличии найденного контекста;
- ответы с fallback;
- ответы с off-topic;
- ответы с несколькими темами;
- ответы с долгой задержкой;
- ответы, похожие на шаблонные.

Колонки:

- `review_id`;
- `brand`;
- `timestamp`;
- `input_text`;
- `answer_text`;
- `route`;
- `topic_id`;
- `safety_flags`;
- `context_flags`;
- `known_client_fields`;
- `asked_again_fields`;
- `latency_seconds`;
- `why_review`;
- `human_verdict`;
- `human_comment`;
- `next_action`.

### Блок C. Детектор “бот спросил известное”

Добавить внутреннюю проверку:

- извлечь из context поля, которые считаются известными:
  - parent_name;
  - student_name;
  - phone;
  - grade;
  - active_brand;
  - known_course;
  - current_group;
  - payment_status, только если надёжно подтверждён;
- проверить, спрашивает ли ответ снова эти данные.

Примеры повторного ненужного запроса:

- “Напишите, пожалуйста, ФИО ребёнка” при известном student_name;
- “Укажите телефон” при известном phone;
- “Какой класс у ребёнка?” при известном grade;
- “Вы уже учитесь у нас?” при найденном Tallanto/AMO context.

Если найден повторный запрос:

- добавить флаг `asked_known_data_again`;
- положить строку в `semantic_review_queue`;
- в тестах считать это P1, а если контекст содержит P0 — P0/P1 в зависимости от темы.

### Блок D. Context-aware prompt rules

Обновить правила промпта так, чтобы модель явно получала инструкцию:

1. Если данные клиента уже известны из контекста, не спрашивай их заново.
2. Не раскрывай источник данных: AMO, Tallanto, CRM, internal history.
3. Если в контексте несколько учеников или конфликт, уточни, по кому вопрос.
4. Если контекст надёжный, используй его для более умного ответа.
5. Если контекст ненадёжный, не делай вид, что знаешь.
6. Если известный клиент спрашивает безопасную справочную тему, отвечай по теме, а не начинай заново как с нового лида.
7. Если новый клиент, мягко веди к следующему шагу: класс, предмет, цель, формат.

### Блок E. Human tone rules

Добавить правила тона:

- не начинать каждый ответ с “Спасибо за обращение”;
- не писать “Ваш вопрос очень важен”;
- не использовать “оптимальный образовательный продукт”;
- не писать слишком длинные абзацы;
- не использовать канцелярит;
- не раскрывать “я бот / ИИ”;
- не звучать как универсальная нейросеть;
- использовать короткие живые фразы;
- если уместно, обращаться к конкретной ситуации клиента;
- завершать безопасным следующим шагом.

Добавить автофлаги:

- `template_like_answer`;
- `too_generic_answer`;
- `asked_known_data_again`;
- `human_tone_review_required`.

### Блок F. Human tone scorer

Создать лёгкую эвристическую оценку ответа:

Плохие признаки:

- “Спасибо за обращение” в каждом ответе;
- “менеджер свяжется” без полезного содержания;
- “для точного ответа уточните” + длинный список вопросов;
- “оптимальный”, “индивидуальный подход”, “образовательный продукт”;
- больше 4 уточняющих вопросов подряд;
- нет ответа на основной вопрос;
- нет следующего шага.

Хорошие признаки:

- есть прямой ответ;
- есть 1-2 уточняющих вопроса, если они нужны;
- учтён бренд;
- учтён known context;
- есть мягкий следующий шаг;
- нет выдуманных фактов.

Выход:

- `human_tone_score` от 0 до 100;
- `human_tone_flags`;
- `why_low_tone_score`.

Это не должно блокировать ответ автоматически. Это метрика и очередь проверки.

### Блок G. Памятка сотруднику

Создать:

`docs/TELEGRAM_PILOT_EMPLOYEE_TESTING_GUIDE_2026-05-21.md`

Обязательно добавить раздел:

“Как проверять человечность ответа”

Сотрудник должен отмечать:

- ответ звучит живо или шаблонно;
- бот спросил то, что уже должен знать;
- бот помог продвинуть клиента к следующему шагу;
- ответ можно было бы отправить клиенту почти без правки;
- бот звучит как нейросеть;
- бот слишком осторожничает.

## 7. Тесты

Добавить:

`tests/test_telegram_public_pilot_feedback_report.py`

Минимальные тесты:

1. Скрипт создаёт все отчёты из тестовых логов.
2. Токены не попадают в отчёты.
3. Телефон маскируется в публичном отчёте.
4. P0/high-risk попадает в очередь проверки.
5. Ответ с ценой/датой/процентом попадает в очередь проверки.
6. Ответ с CRM/Tallanto context попадает в очередь проверки.
7. Если known student_name есть, а ответ просит ФИО ребёнка, ставится `asked_known_data_again`.
8. Если known phone есть, а ответ просит телефон, ставится `asked_known_data_again`.
9. Если known grade есть, а ответ просит класс, ставится `asked_known_data_again`.
10. Шаблонная фраза “Спасибо за обращение, менеджер свяжется” получает низкий human_tone_score.
11. Живой короткий ответ с учётом контекста получает нормальный human_tone_score.
12. Пустые логи не ломают отчёт.

Тесты provider/prompt:

1. Известный клиент: бот не спрашивает телефон и ФИО заново.
2. Несколько учеников: бот уточняет, по кому вопрос.
3. Новый клиент: бот задаёт 2-3 безопасных вопроса для подбора курса.
4. P0-тема: человечность не ослабляет manager_only.
5. CRM/Tallanto ID не попадают в `draft_text`.
6. Ответ не содержит “бот”, “ИИ”, “GPT”, “Claude”, “Codex”.

ТЗ выполнено, если:

1. Есть скрипт отчёта по логам пилотных ботов.
2. Есть daily report по пилоту.
3. Есть очередь смысловой проверки.
4. Есть реестр ошибок пилота.
5. Есть памятка сотрудникам.
6. В отчётах нет токенов.
7. Известные данные клиента не запрашиваются повторно в тестах.
8. CRM/Tallanto служебные детали не раскрываются.
9. P0-защита не ослаблена.
10. Появилась метрика `human_tone_score`.
11. Появились флаги:

    - `template_like_answer`;

## 10. Что останется за рамками

Не входит в это ТЗ:

- перенос на M1 Pro как службу;
- запись в AMO/Tallanto;
- полноценная авторизация клиента по Telegram-аккаунту;
- подключение клиентского трафика с сайта;
- email-черновики;
- полный MEGA на 16 часов;
- переобработка звонков;
- SaaS-кабинет.
- отдельная политика ответов на мета-вопросы “как вас зовут?”, “вы кто?”, “с кем я общаюсь?” — зафиксировано как будущая задача, чтобы не смешивать её с текущим исправлением повторного запроса уже известных данных клиента.

## 11. Практический результат

После этого блока у Дмитрия будет не просто “боты запущены”, а управляемый пилот:

- видно, как бот отвечает;
- видно, где он звучит плохо;
- видно, где он спрашивает лишнее;
- видно, где не хватает факта;
- видно, какие ошибки нужно чинить;
- каждую подтверждённую ошибку можно превращать в тест или правило.

Это прямой путь к боту, который не только безопасен, но и реально похож на полезного сотрудника продаж.
