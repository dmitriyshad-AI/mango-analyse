# ТЗ для D1 — чинить КОРЕНЬ holdout, не кейсы (упор на реализацию)

Источник: независимый регрейд чистого holdout (Claude) `telegram_d1_holdout_clean_after_punchlist_20260524_164854`
(26 диалогов, FAIL 5, hard-gate 4, тон 60.4, ignored_question 20/26). Дефекты РЕАЛЬНЫЕ (судья прав, не строгость).

## Рамка (главное)

Качество — из ГЕНЕРАЦИИ: контракт ответа выбирает факт под СПРОШЕННУЮ под-тему и отвечает на последний прямой
вопрос. Правила — только для БЕЗОПАСНОСТИ. **Не добавлять 5 веток/шаблонов под 5 кейсов holdout.** Чинить 3 корня,
из которых эти кейсы вытекают. Анти-«крот»: число веток/пост-фильтров/детекторов после правок должно **упасть или
не вырасти** (несколько общих правил заменяют разрозненные). Точные функции/места — на твоё усмотрение (ты знаешь
текущий код лучше; ниже — ориентиры по модулям, не жёсткая привязка к строкам).

## Корневой диагноз

5 классов holdout сводятся к 3 механизмам:
- **Корень A — выбор соседнего факта вместо спрошенного** (3 из 4 hard-gate): маткапитал→налоговый вычет,
  расписание занятий→часы работы офиса, дневной лагерь→выездная ЛВШ.
- **Корень B — P0 ловится по ключевому слову, а не по смыслу, и не на каждом ходу**: «верните деньги / дважды
  списали» бот проигнорировал и продолжил продавать, пока клиент не дописал «жалоба».
- **Корень C — не потребляет прямой вопрос: повтор реплики, сброс слотов, приписывание потребности**:
  ignored_question 20/26, дословные повторы, «приписал программирование», over_handoff при наличии факта.

## Корень A — факт строго по спрошенной под-теме (near-neighbor disambiguation)

Ориентир: `answer_contract.py` (facts_resolved_by_intent / required_fact_ids), `conversation_intent_plan.py`,
первый проход генерации (`draft_prompt_builder.py` / `subscription_llm.py`).

Реализация (механизм, не кейсы):
1. Интент-план относит спрошенную под-тему к КОНКРЕТНОМУ topic_id факта. Факт с другим topic_id в ответ НЕ
   подставляется. То есть `required_fact_ids` резолвятся только из фактов, чей topic совпадает со спрошенным.
2. Это страховочный список поверх ОБЩЕГО правила из п.1 (у каждого факта есть тема, у вопроса — требуемая тема; нет
   совпадения — факт нельзя использовать как ответ). Закрытый набор «легко путаемых соседей», которые НИКОГДА не
   заменяют друг друга (общее правило, а не патч под кейс):
   - `matkap` ≠ `tax_deduction` (и ведомства не путать: маткапитал → СФР, вычет → ФНС);
   - `class_schedule` (расписание группы) ≠ `office_hours`/контактный график;
   - `ls_city` (дневная городская школа) ≠ `lvsh_mendeleevo` (выездная с проживанием);
   - обобщить: если у ближайшего факта topic_id ≠ спрошенному — он неeligible.
3. Если точного факта под под-тему НЕТ — НЕ брать соседний, а честная рамка + один точечный вопрос (или черновик
   менеджеру). Лучше «у меня нет подтверждённого расписания этой группы» , чем «офис работает 10-18».
4. Это в ПЕРВОМ проходе генерации (в промпте/контракте), не в пост-рерайтере.
5. Защитный guard (это уже правило безопасности — допустимо): если черновик ссылается на факт с topic_id ≠ теме
   прямого вопроса → блок «факт не из той темы» → draft_for_manager, по аналогии с brand-leak guard.

## Корень B — P0 по смыслу + латч (безопасность, приоритет №1)

Ориентир: `answer_safety_classifier.py` (единый P0), `conversation_intent_plan.py` (refund/complaint/legal),
`p0_recall_spec.py`.

Реализация:
1. P0 определять прежде всего по СМЫСЛУ намерения, но словари-маркеры НЕ убирать — они остаются страховочной
   сеткой (P0 срабатывает, если ИЛИ семантика, ИЛИ слово-маркер). Для P0 лучше перестраховаться, чем пропустить.
   Расширить recall возврата/спорной оплаты: «верните деньги», «дважды списали», «вернуть оплату», «хочу деньги
   назад», «ошибочно списали» — это P0 БЕЗ слов «жалоба/суд».
   **Баланс с ложным P0:** в holdout есть персона `hold_foton_false_p0_benign_09` — благожелательное/гипотетическое
   упоминание («у знакомых был возврат, а у вас как?») НЕ должно эскалироваться. Высокий recall — на реальное
   НАМЕРЕНИЕ клиента вернуть деньги / сообщить о проблеме; на спорном случае — manager DRAFT, а не полный P0-стоп.
   Так поднимаем recall, не создавая новый класс ложных P0 (это тоже был бы «крот», только наоборот).
2. P0 проверять на КАЖДОМ ходе клиента, не только на первом. P0, появившийся в середине диалога, перебивает текущий
   (продажный) план НА ТОМ ЖЕ ходе, а не на следующем.
3. **P0-латч**: как только в диалоге сработал любой P0 — маршрут остаётся `manager_only` до конца (без возврата к
   продаже), пока не будет события менеджера/человека.
4. На P0 — строгая `bot_phrase_p0`, ZERO collect, без продающего продолжения (это уже в политике; чинить надо
   именно ДЕТЕКЦИЮ: своевременность и recall).
5. Поведенческая приёмка (не кейс): вставить фразу возврата/спорной оплаты в ЛЮБОЙ ход ЛЮБОЙ персоны → маршрут
   этого хода = `manager_only`, в ответе нет ни одной продающей фразы.

## Корень C — потребление прямого вопроса + анти-повтор + слоты

Ориентир: первый проход `subscription_llm.py`, `answer_quality_rewriter.py`, `dialogue_memory`.

Реализация:
1. Генерация сначала отвечает на ПОСЛЕДНИЙ прямой вопрос клиента — фактом, корректным по теме (связано с Корнем A).
2. Удержание слотов: класс, предмет, формат (онлайн/очно), названные клиентом, сохраняются в плане и НЕ
   сбрасываются на другой формат до явной смены клиентом.
3. Анти-повтор на уровне ГЕНЕРАЦИИ: перед отправкой сравнить с предыдущей репликой бота; если совпадение ≥ ~85% —
   это сигнал, что вопрос не услышан → **перепланировать ответ на НОВОЕ в последней реплике клиента (дельту)**, а не
   повторять. Не просто блокировать повтор, а отвечать на новый под-вопрос.
4. Не приписывать неназванный предмет/потребность: в `forbidden_assumptions` контракта внести «предмет/цель, не
   названные клиентом». Если клиент сказал только «точные предметы / тревога за уровень» — не выбирать конкретный
   предмет за него.
5. Если на вопрос ЕСТЬ факт (например, «проверяете ли уровень перед группой?» → есть факт про вступительное
   тестирование) — ответить им, а не уходить в хендофф. Снизить over_handoff там, где факт доступен.

## Что НЕ делать

- Не добавлять отдельные ветки/шаблоны под 5 кейсов holdout (это и есть «игра в крота»).
- Не «лечить» строгость судьи — он прав, дефекты реальные.
- Не возвращать «4 части Долями» (запрещено в bot-pack).
- Не наращивать число пост-фильтров: корни A и C — в генерации; B — в едином P0-классификаторе.

## Порядок реализации (согласован с Codex 2026-05-24)

**B → A → C.** Сначала Корень B (P0 — важнее всего, безопасность), затем A (даёт hard-gate ошибки), затем C
(человечность/польза). Full v8 запускать только после этого и после свежего honest-holdout.

## Dev-цикл и самопроверка (НЕ финал)

- Написать НОВЫЕ dev-сценарии под 3 корня (не копии holdout-персон): P0 в разных позициях диалога; near-neighbor
  темы (маткапитал/вычет, расписание/офис, дневной/ЛВШ); повтор-провокации; приписывание предмета.
- Гонять на dev + targeted16 (отладка). **Чистый holdout НЕ трогать** — он только для финального честного замера,
  под него не подгонять.
- Анти-«крот»: показать, что число решающих веток в intent_plan/subscription_llm и число пост-фильтров не выросло
  (лучше — упало), а общие near-neighbor/P0-правила заменили частные.

## Критерий готовности (повторный честный замер)

На СВЕЖЕМ holdout (новые персоны, не те, что чинил): доля «отправил бы без правок» вверх; ignored_question,
повторы, over_handoff вниз; **hard-gate = 0, в т.ч. 0 p0_mishandled**; 0 утечек брендов. Только после этого — full v8.
