# Скан процессных вопросов по корпусу обращений Mango. Клод #2, 2026-06-09

Цель (ТЗ): подтвердить и дополнить первичный список процессов (`Foton/Процессы_для_базы_Mango.docx`,
9 категорий / 25 процессов) по РЕАЛЬНОМУ корпусу и дать частотность. Только ПРОЦЕССНЫЕ вопросы
(«как/что мне делать»), не фактические («сколько стоит / какое расписание»). Факты для базы не
формулирую — это команда. Read-only, код/прогоны не трогал.

## Метод и честные границы

- **Источник правды — сырьё.** Греп по процессным паттернам → чтение выборки контекстов → группировка
  в 25 процессов + кандидаты на новые → подсчёт. Скрипт: `scripts/process_questions_scan.py`.
- **Счётчики — это паттерн-кандидаты, выверенные на выборке, а не точная истина.** Первый прогон дал
  завышение (например «договор» ловил «расторгнуть договор» = возврат и голые упоминания; «справк» —
  медсправку; «кабинет» — номер физического кабинета; «пропуст» — идиому «пропустила информацию»).
  Я уточнил 7 самых шумных паттернов под процессную интенцию и пересчитал. Остаточный шум возможен —
  абсолютные числа считать как **верхнюю границу**, а не точную частоту.
- **Чат-сигнал чище звонков.** WhatsApp/Telegram — это дискретные клиентские сообщения-вопросы.
  Звонки — длинный агрегированный client-канал, где голое ключевое слово даёт больше ложных
  срабатываний. Поэтому даю ДВЕ колонки (чат / звонки); для приоритета опирался прежде всего на чат,
  звонки — как подтверждение объёма.
- **Бренд свято.** Фотон ≠ УНПК. WhatsApp — бренд по чату; Telegram-экспорт — весь УНПК; звонки —
  бренд по маркерам в транскрипте, без маркера → `unk` (не приписываю бренд силой).
- **ПДн обезличены:** телефоны/email/ФИО → плейсхолдеры; имён/телефонов/адресов клиентов в отчёте нет.
