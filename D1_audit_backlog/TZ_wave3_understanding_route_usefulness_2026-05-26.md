бот честно отвечать на КОНКРЕТНЫЙ вопрос. Безопасность Волн 1-2 НЕ ослаблять (целостность базы, M3,
анти-склейка, тёплый слой остаются).

## Сквозной принцип (важно для всех пунктов)
Бот НЕ выводит «нет» из отсутствия факта (closed-world запрещён). Отвечать «нет» можно ТОЛЬКО при
явном отрицательном факте. И «да», и «нет» без источника — небезопасны. См. KB-фикс.

## §1. P9 — «факт есть → отвечай» (главный рычаг тона)
- Где: route-логика в `run_pipeline` + контракт `understand`/`parse_contract`.
- Проблема: бот уходит к менеджеру там, где по прямому вопросу ЕСТЬ подтверждённый client-safe факт.
- Что сделать: если по отвечаемой части есть подтверждённый факт (retrieval по этому под-вопросу не
  пуст), нет P0 и нет missing — отвечать автономно, не прятать факт за «менеджер уточнит». Менеджер —
  только для P0 / неподтверждённого / спорного.
- Приёмка: на наборе, где факт есть (цена/расписание/формат по подтверждённому), бот отвечает сам;
  P0/неподтверждённое/missing — по-прежнему к менеджеру (НЕ ослабить). over_handoff падает, тон растёт.

## §2. C8 + ответ на КОНКРЕТНЫЙ да/нет (закрывает диалог 23)
- Где: контракт (понять, что задан да/нет про существование X) + черновик/верификатор.
- Что сделать:
  - Если задан да/нет-вопрос «есть ли X?»: бот отвечает ИМЕННО про X. Запрещено аффирмативное «Да»,
    если подтверждён ДРУГОЙ факт (Y), а не X (как «Да» про банк, подтверждая общую оплату).
  - Если есть явный отрицательный факт про X → честно «нет, X нет; но есть Y» (Y только если факт).
  - Если нет ни «да», ни «нет» факта про X → НЕ хватать соседний факт, НЕ говорить «Да»; узкий честный
    ответ/уточнение или менеджер. Никогда не выводить «нет» из пустоты.
- Приёмка: диалог 23 — бот НЕ говорит «Да» про банковскую рассрочку УНПК; при наличии отрицательного
  факта говорит «нет, но есть помесячно»; без факта — не affirmирует и не выдумывает «нет».

## §3. U1 — мягкий возврат (закрывает диалог 02)
- Где: контракт / P0-детект мягких форм возврата.
- Что сделать: любая форма возврата (спор / «до оплаты» / «просто интересно правила») → менеджер, НО
  честной фразой именно про возврат («правила возврата уточнит менеджер»), без сухого шаблона и без
  подмены другим фактом (не «правила поведения», не договор+email как единственный ответ).
- Приёмка: диалог 02 — возврат честно к менеджеру по теме, без подмены; не «expected_route_not_followed».

## §4. U5 — трёхуровневый маршрут self / partial / manager
- Что сделать: для составного вопроса — ответить на безопасную часть автономно, спорную/неподтверждённую
  отдать менеджеру (частичный ответ), вместо полного хендоффа всего сообщения.
- Приёмка: на составных бот даёт частичный ответ + узкий хендофф по спорной части; over_handoff падает.

## §5. T5 — один хороший вопрос вместо пустого хендоффа
- Что сделать: если не хватает ОДНОГО слота (класс/предмет/формат) и тема не P0 — задать ровно один
  конкретный уточняющий вопрос (без фактов), а не уходить к менеджеру.
- Приёмка: часть over_handoff превращается в живой уточняющий вопрос; ни одного выдуманного факта в вопросе.

## §6. Что НЕ делаем / инварианты
- Не ослаблять P0/бренд-скоуп/верификацию Волн 1-2 (целостность, M3, анти-склейка, тёплый слой).
- Closed-world запрещён: «нет» — только из явного отрицательного факта.
- Холдаут не тюнить.

## §7. Критерии приёмки волны (с защитой от перегиба)
- over_handoff заметно ниже И тон выше, ПРИ ЭТОМ: выдумки 0, бренды чисто, P0 держится, склейка Волны 2
  НЕ регрессирует (прогнать регресс-набор склейки), M3 без рискованных чисел.
- Диалог 23 → честный ответ (нет/уточнение, не «Да»); диалог 02 → возврат честно к менеджеру по теме;
  составные → частичный ответ; неполный слот → один вопрос.
- НЕ перегнуть в обратную сторону: P9/U5 не должны выдавать неподтверждённое автономно (проверить, что
  P0/missing/неподтверждённое по-прежнему к менеджеру).
- Независимая проверка Claude: регресс-набор склейки (не регрессировал), M3 vs золотой, P0-набор (возврат
  всё ещё P0), и что C8 не выводит «нет» без факта.

## §7-bis. Уточнения после аудита Codex (ПРИНЯТЫ — обязательны к реализации)
1. **P9 — НЕ глобальный override (главный риск).** «факт есть → отвечай» работает ТОЛЬКО для конкретной
   безопасной ЧАСТИ (под-вопроса) и при совпадении ТЕМЫ факта с под-вопросом. Запрещён общий override
   `manager_only → answer_self` при наличии любого факта — иначе повторим «факт есть, но не по теме»
   (диалоги 02/23). Точка маршрута: `dialogue_contract_pipeline.py` ~стр.903 (`answerability != answer_self
   → draft_for_manager`) — менять аккуратно, по под-вопросам.
2. **C8 — явный тип вопроса `existence_yes_no`** в контракте для «есть ли X?». Запретить НЕ только «Да»,
   но и мягкие аналоги («можно», «доступно», «да, есть вариант»), если подтверждён только соседний Y, а
   не X. «Нет» — только при явном отрицательном факте (closed-world запрещён).
3. **U5 — частичный ответ ПО ПОД-ВОПРОСАМ:** self-часть отвечает, manager-часть узко передаётся; НЕ
   переводить весь ход в автономный.
4. **T5 — уточняющий вопрос только если** недостаёт ровно одного слота И нет P0/денег/возврата/договора.
5. KB-фикс делать раньше Волны 3 (после ОК Дмитрия); формулировку отрицательного факта — строже (см.
   `KBFIX_unpk_no_bank_installment_2026-05-26.md`, ревизия 2).
