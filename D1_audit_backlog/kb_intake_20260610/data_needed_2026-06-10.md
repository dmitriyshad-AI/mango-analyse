# Какие данные нужны, чтобы бот отвечал на (почти) ВСЕ вопросы. Клод #2, 2026-06-10

Всё по этой работе собрано в одной папке `D1_audit_backlog/kb_intake_20260610/`. Отчёты:
`questions_catalog.md` (каталог вопросов), `coverage_report.md` (покрытие — дыры по темам), `external_facts_draft.md`
(сайт↔база, расхождения), `SUMMARY.md` (сводка), `coverage_estimate_3000…md` + `coverage_per_question_audit…md`
(честная оценка покрытия), `_verdicts_all.jsonl` (вердикт по каждому из 3000 вопросов, аудируемо).

Чтобы бот закрывал почти весь спрос, нужны ТРИ разных типа данных — они принципиально разной природы.

## 1. Контентные факты — команда напишет/подтвердит (закрывает ~94% → ~97% по содержанию)
Это конечный, перечислимый список. Источник формулировок — как РЕАЛЬНО отвечают менеджеры (есть в диалогах) + сайт.

**а) Решить расхождения сайт↔база** (из `external_facts_draft.md`) — без этого бот говорит неверно:
- Платформа онлайн-курсов: **SohoLMS (база) или МТС Линк (сайт)?**
- Длительность занятия УНПК: **2 или 3 ак.ч?**
- Предметы УНПК: актуальны ли **обществознание, робототехника, английский** (бот сейчас их отрицает)?
- Онлайн-кэшбэк «приведи друга» УНПК: **5 000 или 10 000 ₽?**
- Адреса УНПК: добавить **Лобню, Жуковский, Первомайскую, Верхнюю Красносельскую** (в базе только 4).
- Часы работы; детализация цен ЕГЭ/ОГЭ по продукту; поправить `total_alumni` («2 026» → «более 2000»); канал **Max** у Фотона.

**б) Контентные дыры** (из пер-вопросного аудита, стабильные ~3%):
- **G6** — содержание программ по темам/блокам (что именно проходят по предмету/уровню).
- **G3** — присоединение в середине года + пересчёт оплаты (частотный сценарий).
- **G7** — есть ли индивидуальные занятия/репетитор к регулярным курсам (нужен факт «да/нет»).
- **G2** — учебники/методика/УМК (по чему занимаются).
- **G5** — часовой пояс: «время указываем по Москве» (дёшево — один факт, а спрашивают регулярно).
- **G4** — оплата от юрлица/по счёту организации.
- **G8** — площадки не в базе (см. адреса выше).
- **G9** — межпредметный перенос оплаченных уроков; **G10** — маршрут по чужим программам («Код будущего» и т.п.).

**в) Бренд-зеркала УНПК** (у Фотона факт есть, у УНПК нет — из `coverage_report.md`): срок/форма результатов теста;
тех.требования + @support; «центр не видит пароли»; бесплатное пробное в другой группе.

**г) Процессы** — подтвердить уже готовые черновики `processes_client_safe_drafts_2026-06-09.md` (по брендам).

## 2. Живые/оперативные данные — это НЕ факты, нужна СИСТЕМА (самый большой рычаг, ~18% спроса)
Эти вопросы бот не закроет из статичной базы НИКОГДА — нужен read-only доступ к CRM/Tallanto/таймлайну клиента:
- «Есть ли места в смену / набралась ли группа?» → живой остаток мест.
- «Прошла ли моя оплата / мой баланс / вернётся ли залог?» → статус из бухгалтерии/CRM.
- «Есть ли занятие сегодня / во сколько завтра у нашей группы?» → живой календарь группы.
- «Дошло ли моё письмо/документ/заявка?» → статус обращения.

Это ровно тот customer-pipeline / customer timeline, по которому я делал отдельную диагностику
(Resolve/Analyze/таймлайн). **Пока этой связки нет — на любой статус/наличие бот будет честно отвечать
«уточнит менеджер».** Это не пробел базы, а граница автономии: либо подключаем живые данные, либо это поле менеджера.

## 3. Остаётся человеку — по политике (не «недостающие данные»)
~2% — реальные претензии на возврат после оплаты, споры оплаты, жалобы на качество, юр-угрозы, и прямые
сравнения брендов. Это сознательно ведёт менеджер; добавлять сюда «данные» не нужно.

## Честный потолок
Даже со всем выше **«100% на все вопросы» недостижимо.** Останется длинный хвост: вопросы не по адресу (чужой
профиль, «художественная школа», «график дизайн»), разовые странности, ASR-каша в расшифровках звонков, и
**непрослушанные голосовые** в WhatsApp/Telegram (их сейчас нет в данных — нужен ASR). Реалистичная цель:
**~95–97% по содержанию (через факты из п.1) + закрытие живых вопросов через CRM (п.2)**; остальное — менеджер.

## Что для этого нужно ОТ Дмитрия (коротко)
1. Решения по расхождениям и открытым вопросам (платформа, длительность, предметы, кэшбэк, адреса) — список в `SUMMARY.md`.
2. Решение по live-данным: подключать ли бота к CRM/таймлайну (read-only) — это закрывает самый большой пласт (~18%).
3. Подтверждение черновиков процессов и новых фактов перед заводом в базу (через Codex, semantic-гейт).
