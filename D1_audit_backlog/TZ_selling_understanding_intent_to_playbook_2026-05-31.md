# ТЗ MAIN — «продающее понимание»: намерение между строк → приём playbook. 2026-05-31.

Автор: Клод 1. Стратегическое ТЗ (НЕ на сейчас — после стабилизации over-handoff). Цель: бот читает
намерение/желание клиента между строк и ОТВЕЧАЕТ приёмом продавца, а не буквально по словам.

## Что есть сейчас (проверено чтением 9e874ccf)

- `client_state` извлекает МОДЕЛЬ в `build_understanding_prompt` («ситуация/тон клиента»). НО используется
  только для ВЫБОРА ТОНА (регистр ответа), не для ДЕЙСТВИЯ. Эмоцию вслух называть нельзя.
- `primary_intent` / `risk_signals` (conversation_intent_plan) — ДЕТЕРМИНИРОВАННО (keyword).
- `urgency`/`hot_lead`/`commercial_risk` (signals.py) — ДЕТЕРМИНИРОВАННО (markers).

Разрыв: бот ПОНИМАЕТ тон (модель), но (а) понятое идёт только в тон, не в действие; (б) намерение/
сигналы — по словам, теряют подтекст. Между «понял возражение» и «применил приём» нет моста.
Playbook у нас ЕСТЬ ([[sales_playbook_from_calls]]: снятие риска, гашение тревоги, честная срочность,
чёткий шаг — Тропина/Козлова).

## Категории непонимания «между строк» (что добавить)

1. Возражение под вопросом: «дорого», «подешевле?», «подумаем» → не факт-запрос, а возражение.
2. Сигнал ухода: «поищу у других», «спасибо, подумаю» → момент удержания.
3. Тревога/недоверие: «вы точно нормальные?», «не обманете?» → нужен приём снятия риска.
4. Невысказанная потребность: «у ребёнка тройка», «он стеснительный» → сигнал, не вопрос.
5. Стадия готовности: «сколько стоит?» от изучающего vs готового → разный следующий шаг.

## ПРИНЦИПИАЛЬНАЯ РЕАЛИЗАЦИЯ (архитектура)

Ключ: понимание — МОДЕЛЬНОЕ (расширить существующий understanding-вызов, БЕЗ нового вызова — дёшево);
действие — детерминированный маппинг «сигнал → приём» в draft-промпте; безопасность — поверх всего.

**Шаг 1. Расширить выход `build_understanding_prompt` (contract) полями продающего понимания:**
```
"selling": {
  "objection": "price|doubt|comparison|none",
  "exit_signal": true|false,        // уходит ("поищу у других", "подумаю")
  "anxiety": true|false,            // тревога/недоверие
  "unmet_need": "<кратко, если есть невысказанная потребность>",
  "readiness": "exploring|comparing|ready"
}
```
Модель это извлекает из текста+истории (она и существует, чтобы читать подтекст). Поля в contract.

**Шаг 2. Маппинг «сигнал → приём» в draft-промпте (детерминированный, из playbook):**
- objection=price → «не повторяй цену сухо: дай ценность + безопасный способ оплаты (рассрочка/Долями
  по active_brand), предложи следующий шаг».
- exit_signal=true → «удержи мягко: один конкретный следующий шаг или сними барьер; не отпускай сухим
  хендоффом».
- anxiety=true → «сними риск приёмом: пробное занятие/честно про лицензию/без давления — НЕ обещай
  результат, НЕ дави».
- unmet_need → «предложи релевантное по потребности, даже если прямо не спрошено (в рамках фактов)».
- readiness=ready → «дай чёткий шаг к записи/оплате».
Приёмы — формулировки из playbook (Тропина/Козлова), не агрессивные.

**Шаг 3. Безопасность поверх (НЕ ослаблять):**
- Возражение про возврат/спор оплаты → P0 (менеджер), НЕ «отработка возражения».
- Приёмы только на безопасных темах; бренд/факт/P0-границы важнее приёма.
- «Снять риск» ≠ обещать результат/гарантию (запрещено).

## Почему так (а не иначе)

- Понимание модельное: подтекст ловит только модель, regex здесь бессилен.
- Без нового вызова: расширяем СУЩЕСТВУЮЩИЙ understanding — дёшево.
- Действие детерминированное: маппинг сигнал→приём предсказуем и тестируем (не отдаём продажу на откуп
  модели целиком — она ПОНИМАЕТ, детерминированный маппинг ВЫБИРАЕТ приём, draft исполняет).
- Это прямое продолжение [[bot_goal_autonomous_sales]] и playbook.

## Тесты

- «дорого» при наличии цены+рассрочки → ответ с ценностью+рассрочкой, не сухая цена.
- «поищу у других» → удержание (шаг/барьер), не сухой хендофф.
- «вы точно нормальные?» → снятие риска (пробное/лицензия честно), без давления и без обещаний.
- «у ребёнка тройка по физике» → предложение помощи по физике, не ожидание прямого вопроса.
- НЕГАТИВ: возражение про возврат → P0 менеджер (не отработка); приём не нарушает бренд/факт/P0.

## Приоритет

ПОСЛЕ over-handoff и стабильной базы. Начать с objection(price) + exit_signal — самые бизнес-критичные
для конверсии. anxiety/unmet_need/readiness — следующей волной. Каждый сигнал — отдельный коммит с NEG.
