# ТЗ — Транспорт пилота: Wappi + amoCRM (черновики в карточке сделки)

**Статус:** ФИНАЛ дизайна (аудит второго архитектора внесён). Live-реализация ждёт доступов от Дмитрия. UX полей — финализировать после уточнения у менеджеров.
**Дата:** 2026-06-08. **Основание:** обсуждение с Дмитрием (Salesbot не используется; черновик в поле сделки; воронки ОБЩИЕ; менеджер копирует черновик вручную; приоритет — аннотации бота, не скорость) + Фаза 0 (amo_wappi_phase0).

## Режим
Черновиковый. Бот клиенту НЕ пишет. Бот кладёт черновик + аннотации в карточку сделки AMO; менеджер копирует, правит, отправляет клиенту через чат AMO / вкладку Wappi. Скорость не критична → можно асинхронно.

## Архитектура (поток одного входящего)
1. Клиент пишет в один из 4 профилей Wappi (Фотон-Telegram, Фотон-MAX, УНПК-Telegram, УНПК-MAX).
2. Wappi шлёт вебхук `incoming_message` на наш сервис (`profile_id`, текст, чат, контакт).
3. Наш сервис: `active_brand` = из `profile_id` ВХОДЯЩЕГО (не из воронки!) → генерит черновик прямым путём (база нужного бренда, гейты безопасности).
4. Сервис пишет в карточку сделки AMO (по контакту/чату) через AMO API: черновик + аннотации + контекст лида.
5. Менеджер видит в карточке, копирует/правит/шлёт.
6. Разница «черновик → отправленное» копится как gold-пак v2.

## Поля в карточке сделки (кастомные, заполняет бот)
- **«Черновик бота»** — последний черновик ответа (перезаписывается новым; всегда актуальный).
- **«Проверь / неуверенность»** — аннотации гейта и верификатора: где бот неуверен, что перепроверить (ПРИОРИТЕТ Дмитрия).
- **«Контекст лида»** — что бот выяснил: бренд, класс/предмет/формат/цель, контакт; для P0 — пометка «P0: в менеджера, бот ответ не предлагает».
- **«Статус черновика»** — сгенерён / устарел (клиент написал новое) / P0 (только алерт).

## Пять несущих механизмов
1. **Канал→бренд (инвариант).** `active_brand` = `profile_id` каждого входящего сообщения, НЕ сделка/воронка (воронки общие!). Черновик строго одного бренда. NEG: входящее из Фотон-профиля → черновик Фотон, даже если в карточке есть УНПК-переписка; бренды в одном черновике не смешиваются.
2. **Аннулирование черновика.** Новое сообщение клиента до отправки → старый черновик помечается «устарел», поле обновляется новым. Менеджер не отправит протухший ответ.
3. **Стоп-кран.** Флаг в нашем сервисе мгновенно отключает генерацию черновиков (менеджер работает вручную, Wappi/AMO не затронуты).
4. **Сохранение правок менеджера (gold v2).** Сравнение «поле Черновик» vs реально отправленное (исходящее Wappi/AMO) → diff в обучающий поток. Если нет исходящего вебхука Wappi — diff по ленте сделки AMO (проверить в Фазе 0).
5. **Аннотации верификатора менеджеру.** Пометки гейта (downgrade/annotate, где бот неуверен) + собранный контекст — в поля «Проверь» и «Контекст лида».

## Безопасность
- Бренд per-message из `profile_id` (общая воронка бренд НЕ определяет) — главный инвариант.
- P0 (возврат/жалоба/юр/спор оплаты) → бот НЕ предлагает черновик-ответ клиенту, а пишет в «Контекст лида» алерт «P0 — менеджеру» + суть. Не маскировать P0 под обычный черновик.
- ПДн: черновик не эхает данные клиента (pii-окно из main); внутренние данные (юрлица/реквизиты/мета) в клиентский черновик не попадают.
- Никаких авто-отправок клиенту в пилоте (только поле; отправляет человек).

## Фазы внедрения (live ждёт доступов)
- **Фаза 0 — live-discovery** на ТЕСТОВОМ профиле Wappi + тестовой сделке: снять реальный формат вебхука Wappi (`incoming_message`, поля), проверить запись кастомных полей AMO, есть ли исходящий вебхук (для gold v2). Ничего боевого.
- **Фаза 1 — dry-run:** на тестовой воронке бот пишет черновик в поля по реальным входящим, клиенту НЕ уходит ничего. Сверяем качество/бренд/аннотации.
- **Фаза 2 — пилот:** менеджеры используют черновики на боевых сделках (первые N дней — мой ежедневный разбор 100% черновиков).

## Что нужно от Дмитрия (доступы/решения)
1. AMO: поддомен + API-токен (OAuth-интеграция).
2. Wappi: 4 `profile_id` + токены (Фотон-TG, Фотон-MAX, УНПК-TG, УНПК-MAX) + куда слать вебхук (публичный URL нашего сервиса).
3. Сервер с публичным адресом для приёма вебхуков 24/7 (где развернём сервис).
4. Тестовый профиль Wappi + тестовая сделка/воронка (отладка не на боевых).
5. Создать кастомные поля сделки (или указать существующие) под черновик/аннотации/контекст.
6. Уточнить у менеджеров: реально ли поле в карточке удобно, или нужна доработка UX.

## Аудит второго архитектора (внесён)
- **Р1 «бренд из воронки нельзя — воронки общие»:** инвариант `active_brand` = `profile_id` per-message, NEG на смешанную карточку. Принято.
- **Р1 «поле перезаписывается → риск отправки устаревшего»:** механизм аннулирования + статус «устарел». Принято.
- **Р2 «gold v2 требует исходящих»:** проверить наличие Wappi outgoing-вебхука в Фазе 0; фолбэк — diff по ленте AMO. Принято.
- **Р2 «P0 не должен выглядеть как готовый черновик»:** P0 → алерт в «Контекст лида», без ответа-предложения клиенту. Принято.
- **Р3 «нет prefill в поле ввода»:** подтверждено ограничение; для пилота менеджер копирует из поля (Дмитрий согласовал). 1-клик-отправку не закладываем.

## Вне scope
Авто-отправка клиенту (ночной автоном — после метрик пилота); 1-клик «отправить как есть» (нет prefill); Salesbot-сценарии (не используется).
