# План итерационного улучшения таблицы вопрос-ответ для РОПа и будущего бота

Дата: 2026-05-14

## Цель

Получить контрольный файл минимум из 100 реальных классов вопросов клиентов и 100 качественных черновиков ответов, который можно дать РОПу на проверку без очевидных технических ошибок, неверных классов и опасных обещаний.

Файл не является разрешением на автоматическую отправку клиентам. Это пакет проверки качества: РОП должен увидеть вопрос, примеры, предлагаемый ответ, ограничения, нужные актуальные факты и решение по самостоятельности бота.

## Почему нужен новый цикл

Предыдущий файл с широкими классами показал, что проблем было много:

- автоподсказка могла ошибаться из-за названия класса;
- исторические ответы менеджеров иногда неверные или устаревшие;
- часть ответов слишком общая и не помогает РОПу;
- вопросы про оплату, документы, договоры и маткапитал нельзя смешивать;
- нельзя оставлять старое `нет`, если вопрос требует проверки документов или региона;
- РОПу нужен файл, где можно глазами проверить не только класс, но и сам ответ.

## Что должно быть в новой таблице

Минимум 100 строк. Одна строка = один класс вопроса, выбранный из приоритетных вопросов.

Колонки:

1. Номер.
2. Приоритет.
3. Крупный класс.
4. Узкий класс.
5. Вопрос клиента простым языком.
6. Реальные примеры вопросов.
7. Источники: звонки, Telegram, почта.
8. Количество вопросов в классе.
9. Предлагаемый ответ.
10. Почему ответ именно такой.
11. Что бот может делать: готовый ответ / ответ после проверки факта / черновик менеджеру / только менеджер.
12. Какие факты нужны: цена, расписание, договор, документы, скидка, оплата, адрес, программа.
13. Какие данные нужно подставить: ФИО ученика, курс, предмет, период, сумма, дата, ссылка, документ.
14. Риск ошибки.
15. Что должен проверить РОП.
16. Решение РОПа.
17. Исправленный ответ РОПа.
18. Комментарий РОПа.

## Правила качества ответа

1. Ответ должен быть полезным и конкретным, а не общим текстом `передам менеджеру` без пояснения.
2. Если вопрос требует актуального факта, ответ обязан использовать placeholder в квадратных скобках: `[актуальная цена]`, `[актуальное расписание]`, `[ФИО ученика]`.
3. Нельзя выдумывать цену, расписание, скидку, статус оплаты, статус договора или юридическое решение.
4. Нельзя обещать возврат, перерасчет, зачисление маткапитала или قبول договора без проверки.
5. Исторический ответ менеджера не считается правильным ответом.
6. Автоподсказка и ответ должны строиться по тексту вопроса и метаданным класса, а не по случайным словам в широком названии.
7. Если вопрос содержит несколько тем, это нужно явно пометить.
8. Шум, пересылки, рекламные письма, короткие `да/угу/ок` не должны попадать в контрольный файл.
9. Материнский капитал: безопасная позиция - в целом оплата обучения возможна при соблюдении условий, но конкретный случай нужно проверять по региону, сертификату, договору и лицензии.
10. Все ответы должны быть на русском языке и понятны не программисту.

## Итерационный процесс с Claude

### Итерация 0. Аудит плана

Codex отдает этот план Claude. Claude должен найти слабые места в методике, критериях качества и составе таблицы.

После аудита Codex либо принимает правки Claude, либо аргументированно отклоняет, но итоговый план фиксирует в коде и файлах.

### Итерации 1-5. Генерация, аудит, правка

1. Codex генерирует контрольный файл минимум на 100 строк.
2. Codex прогоняет локальные тесты и проверки качества.
3. Codex отдает файл Claude.
4. Claude проверяет:
   - неверные классы;
   - слабые или бесполезные ответы;
   - опасные обещания;
   - факты без источника;
   - смешение тем;
   - непонятные placeholders;
   - неудобство для РОПа.
5. Codex внедряет правки.
6. Цикл повторяется до 5 раз или останавливается раньше, если Claude дает `PASS`.

## Критерии остановки

Остановиться можно, если Claude дает `PASS` или `PASS_WITH_LIMITATIONS` без P1/P2, а локальная проверка показывает:

- минимум 100 строк;
- 0 пустых предлагаемых ответов;
- 0 ответов со старым техническим мусором: `актуальное окно записи`, `актуальные варианты`, `{documents}`;
- 0 автоматических разрешений на отправку без проверки РОПа;
- 0 строк, где `возврат` появился в ответе без реального вопроса про возврат;
- 0 строк, где маткапитал получает короткое `нет`;
- все ответы имеют понятный статус самостоятельности бота;
- все фактовые ответы имеют placeholders или список нужных фактов.
