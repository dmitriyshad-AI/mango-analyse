## Направление 1 — маршрут должен уважать manager_only (точный баг)

### Корень (код)
`src/mango_mvp/channels/dialogue_contract_pipeline.py`, `run_pipeline`, строка ~635:
```python
if contract.answerability != "answer_self" and not retrieval.facts:
    return ... route="manager_only" ... fallback_reason="contract_manager_only"
```
`manager_only`-контракт уходит к менеджеру ТОЛЬКО если фактов не достали вообще. Если ретривал
вернул ЛЮБЫЕ факты — этот гард пропускается, и пайплайн рисует автономный ответ (`bot_answer_self`).

### Как проявилось (диалог 02, turn 3)
- Клиент: «хочу заранее понимать правила [возврата]».
- Контракт ПОНЯЛ верно: `current_question` про возврат/отмену, `answerability=manager_only`.
- Но в `needed_fact_keys` попал `…course_rules_safe` (правила ПОВЕДЕНИЯ) — семантический мисматч по
  слову «правила». Ретривал его достал → гард строки 635 пропущен → бот автономно ответил про
  «правила поведения и цифровой этикет» на вопрос про ВОЗВРАТ. Клиент: «вы опять не про то».

### Что сделать (направление; точную реализацию Codex предлагает в аудите)
1. **Маршрут уважает answerability.** Если `contract.answerability != "answer_self"` ИЛИ `is_p0` —
   ход НЕ должен становиться `bot_answer_self` из-за того, что «какие-то факты достались». Такой
   контракт идёт к менеджеру или `draft_for_manager`, но не выдаётся как уверенный автономный ответ.
2. **Возврат/отмена/возврат денег — всегда менеджер** (политика CLAUDE.md). При де-эскалации со стороны
   клиента («это не обращение, просто спрашиваю правила») тема остаётся возвратной → менеджер. Бот даёт
   чистую честную фразу типа «правила возврата уточнит менеджер по телефону», а НЕ сваливается на
   другой «правила»-факт.
3. **Не подменять факт по теме семантически близким с другим смыслом.** Нет client-safe факта про
   правила возврата (в базе только налоговый вычет — это другой «возврат»). Если факта по теме нет →
   это `missing` → честный узкий хендофф, а не ответ соседним фактом.

### Критерии приёмки
- Диалог 02: бот НЕ отвечает про правила поведения на вопрос про возврат; возврат уводится менеджеру
  одной honest-фразой. Никаких автономных ответов при `answerability=manager_only`/`is_p0`.
- Регресс: обычные ценовые/фактовые вопросы (диалог 01, 03) по-прежнему отвечаются автономно — фикс
  не должен «загонять к менеджеру» всё подряд.

## Направление 2 — менее бинарный хендофф (мягкие/objection факты)

### Корень
На вопрос «бывают ли группы по выходным?» (диалоги 20, 22) ретривал достаёт только
`contacts_unpk.schedule` и `…regular_courses_schedule_publication` («расписание в июне»). Мягкий
факт-возражение `objection_responses…«Разные слоты по выходным»` (он ЕСТЬ в снимке v6.3) под этот тип
вопроса НЕ достаётся. У бота нет материала на тёплый частичный ответ → он повторяет сухое «точно не
подтвержу» и уходит. Жёсткого факта «очно на Сретенке по сб/вс» в базе нет — поэтому полностью
утверждать расписание бот не может и не должен.

### Что сделать (направление)
1. Для «мягких» вопросов (будут ли группы по выходным / удобное время / формат) разрешить
   контракту/ретривалу подтягивать objection-факты («разные слоты по выходным») и давать тёплый
   ЧАСТИЧНЫЙ ответ + честный следующий шаг, вместо сухого хендоффа.
2. Формула безопасного частичного ответа: «обычно бывают разные слоты, в том числе по выходным;
   точное расписание появится в июне — тогда сверим под ваш класс». Это не обещание конкретных дней.

### Жёсткий инвариант (НЕ нарушать)
- objection/мягкий факт НЕЛЬЗЯ подавать как подтверждённое ТОЧНОЕ расписание. Конкретные дни/время не
  выдумывать. «Разные слоты по выходным» — это «бывают разные варианты», а не «занятия в сб и вс».
- Возврат/жалоба/P0 этого направления НЕ касаются — там по-прежнему менеджер.

### Критерии приёмки
- Диалог 22: на вопрос про выходные бот даёт тёплый частичный ответ из мягкого факта + «точное
  расписание в июне», а не повторяет сухое «не подтвержу». При этом конкретные дни не утверждает.
- Регресс: бот НЕ начинает выдавать objection-факты как точное расписание (проверить на диалоге 20).
