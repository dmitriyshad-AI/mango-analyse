## Зачем

Ядро этапа. Сейчас решение о маршруте размазано по **~41 точке** (19 в `dialogue_contract_pipeline.py`
+ 22 в `subscription_llm.py`), в трёх слоях, поздний перебивает ранний. Умолчание: «к менеджеру, если
хоть что-то». Цель — свести в одну точку и перевернуть умолчание: «бот отвечает сам, к менеджеру —
только при конкретной опасности».

**Профиль риска максимальный:** бот отвечает клиенту напрямую. Переворот умолчания — самый опасный шаг
всего этапа: если хоть одно вето-условие потеряется при сведении, то, что обязано идти к менеджеру
(возврат/жалоба/бренд/выдумка), станет автоответом живому клиенту. Поэтому щит — первым и полным.

## Часть A — ПОЛНЫЙ ЩИТ (делать ПЕРВОЙ, до переворота)

Правило #1: инвентаризировать ЧТЕНИЕМ ВСЕ точки ухода к менеджеру в обоих файлах (ищи присваивания
`route="manager_only"` / `route="draft_for_manager"`, `force_draft_for_manager`, и пост-цепь
`_apply_dialogue_contract_v2_guard_chain` + `_dialogue_contract_v2_route_permission_guard`).

Свести их в явный список категорий-вето. На КАЖДУЮ — негативный тест, что после правки 4 она всё ещё
уводит к менеджеру. Минимальный обязательный набор (сверить, что ни одна не потеряна):

1. P0 hard → `manager_only`: возврат-претензия, жалоба, юр.угроза, спор оплаты, гарантия результата/поступления.
2. `p0_pre_gate` (pipeline ~943).
3. refund без presale-факта (~974).
4. бренд: `brand_leak` / cross-brand.
5. выдумка: `fact_grounding`; `unsupported_entity` без покрытия retrieved_facts.
6. `forbidden_scope`.
7. `meta_leak`.
8. `ai_disclosure` без прямого вопроса (политика C).
9. `unsupported_promise` (числа/проценты/баллы без факта).
10. `unconfirmed_operational_specificity` без факта.
11. `unstated_subject` (предмет не из сообщения/слотов/retrieved_facts активного бренда).
12. high-risk маркеры (`detect_high_risk_input_markers` / `is_high_risk_result`).
13. `payment_confirmation` без надёжной сверки.
14. бренд `unknown` → `draft_for_manager`.
15. `semantic_check_unavailable` (судья недоступен) → к менеджеру (безопасный fail-safe — сохранить).
16. пустой черновик / нет `draft_fn`.

**Без полного щита Часть B не начинать.** Это предохранитель переворота (правило #4).

## Часть B — единый `decide_route` + переворот умолчания

Свести 41 точку в одну функцию `decide_route(...)` — единственное место, где определяется итоговый
маршрут. На вход — результаты всех вето-проверок (категории A) + наличие покрывающего факта + флаг/тема
автономии. Логика:

- сработало hard-P0 / legal / payment_dispute → `manager_only`;
- иначе сработало «мягкое» вето (выдумка / бренд / scope / promise без факта / operational без факта /
  unstated / meta / ai / unknown-brand / judge-unavailable / пустой черновик) → `draft_for_manager`
  (или `manager_only` по тяжести, как сейчас);
- иначе (НИ ОДНО вето не сработало) И автономия по теме разрешена И есть покрывающий факт →
  **`bot_answer_self`** (это и есть перевёрнутое умолчание);
- поздние слои НЕ перебивают `decide_route` — решение одно.

Поведение легитимных вето не меняется. Меняется ТОЛЬКО умолчание для случая «вето пусто»: раньше
часто `draft_for_manager` «на всякий случай», теперь — автоответ.

## Тесты входа/выхода

- Вход: диалог с покрывающим фактом, который сейчас уходит к менеджеру из-за осторожного умолчания → к менеджеру.
- Выход: тот же → `bot_answer_self`; и ВСЕ 16 категорий-вето (Часть A) по-прежнему уводят к менеджеру
  (полный негативный контроль — это главное в этой правке).
