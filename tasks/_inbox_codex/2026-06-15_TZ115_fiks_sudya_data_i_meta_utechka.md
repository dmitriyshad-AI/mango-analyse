## Часть 1 — судья перестаёт метить выдумкой дату из окна валидности факта
**Проблема (по сырью):** бот пишет дату из окна валидности факта (ключ `..._before_2026_07_01` → «до 01.07.2026»), а число-аудит судьи метит её `no_match` → fabrication. Причина: индекс чисел `snapshot_number_index` (`scripts/run_telegram_dynamic_client_sim.py`) строится только из текстовых полей факта; дата окна туда не попадает. Парсинг ТЕКСТА бота («до 01.07.2026») работает — править regex НЕ нужно.
**Фикс (СУДЬЯ/инструмент, не бот):**
- Завести хелпер `_fact_window_date_keys(fact) -> set[str]`: парсить ТОЛЬКО якорь `before_(\d{4})_(\d{2})_(\d{2})` в `fact_key` → нормализовать в `date:DD.MM.YYYY` (через тот же `_number_claim_index_key`/`normalize_date_claim`).
- `valid_until` брать в индекс ТОЛЬКО если он СОВПАДАЕТ с датой из `before_`-токена того же факта (страховка). **Сырой `valid_until` всех фактов индексировать НЕЛЬЗЯ** — это поле протухания записи (502 факта с `2026-12-31` и т.п.), иначе бот, выдумавший «до 31.12.2026», ложно получит match и судья перестанет ловить настоящую выдумку.
- В `snapshot_number_index` добавить ОДНОЙ строкой: `for window in _fact_window_date_keys(fact): index[brand].setdefault(window, set()).add(key)` (узкое конфликт-окно для мерджа).
**Не ослабляет ловлю выдумок:** индекс пополняется только реальными дедлайнами из `before_`-ключей.
**Тесты:** (1) бот-дата = дате `before_`-ключа → грунтована (не fabrication, `hard_gate_cause` → measurement_suspect, не bot_issue); (2) выдуманная дата вне `before_`-окна (и не совпадающая с ним `valid_until`) → по-прежнему `no_match`/fabrication.

## Часть 2 — выходной гейт ловит служебную фразу «в фактах нет» наружу
**Проблема (по сырью):** `has_meta_leak` (`src/mango_mvp/channels/humanity_guards.py`) проверяет только список `_META_CLIENT_MARKERS` (фраз про факты нет); короткие «в фактах нет», «не указан в фактах» доходят до клиента.
**Фикс (выходной гейт, детерминированно):** в `has_meta_leak` добавить ОТДЕЛЬНЫЙ `_META_FACT_PHRASE_RE` (НЕ в список подстрочных маркеров — иначе ложные срабатывания на «фактах» в обычной речи), с УЗКИМ окном якоря (`{0,20}`) на «факт/фактах/базе/данных»:
`(?:в\s+факт\w+\s+нет|нет\s+в\s+факт\w+|(?:не\s+указан\w*|не\s+уточнен\w*|отсутствует)[^.\n]{0,20}\b(?:факт\w+|баз\w+|данн\w+))` (с `re.I`, нормализацией ё→е).
При `has_meta_leak=True` гейт уже `block`/санитайз → подменит на безопасный fact-ответ/хендофф. В `meta_markers_present` добавить пометку `fact_phrase_leak`.
**Важно (не «дочинивать» зря):** `_sanitize_humanity_meta_text` чистит по фиксированному списку строк и эту фразу НЕ вырежет — это ОК: `has_meta_leak(cleaned)` останется True → ветка падает в fact-ответ/хендофф (желаемое поведение).
**POS-тесты:** «в фактах нет информации», «цена не указана в фактах», «нет в фактах» → meta_leak (не доходит до клиента).
**NEG-тесты (НЕ блокировать):** «точное время мы согласуем», «дату уточнит менеджер», «расписание уточняется», «у нас нет занятий по выходным», «класс не указан в анкете», «класс не указан в договоре», «в группе нет свободных мест».
