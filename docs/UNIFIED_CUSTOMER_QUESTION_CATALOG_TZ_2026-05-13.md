# ТЗ: единый каталог вопросов клиентов

Дата: 2026-05-13

Владелец работ: Codex, SaaS/productization диалог.

Цель: построить единый каталог реальных вопросов клиентов из звонков, Telegram и почты, начиная с 2025-01-01, чтобы затем подготовить проверенные ответы для бота, CRM-диалогов, сайта и менеджеров.

## 1. Почему это текущий самый актуальный этап

Это следующий правильный этап, если не трогать качество распознавания и обработки звонков, которое сейчас ведется в отдельном диалоге.

Причина простая: чтобы бот отвечал хорошо, ему недостаточно общей базы знаний. Ему нужен список реальных узких вопросов, которые клиенты уже задавали, и понятный статус по каждому вопросу:

- можно ли отвечать автоматически;
- нужен ли менеджер;
- есть ли утвержденный ответ;
- где ответ подтвержден: в базе знаний, на сайте, в звонках, в Telegram, в письмах или РОПом;
- какие темы чаще всего повторяются.

Без этого бот будет выглядеть как общий справочник. С этим слоем он станет системой, которая опирается на реальные продажи и реальные вопросы клиентов.

## 3. Что именно нужно построить

Нужно создать продуктовый слой "единый каталог вопросов клиентов".

Он должен собирать реальные вопросы из:

- звонков;
- Telegram;
- почты;
- в будущем из сайта, CRM-чатов и других мессенджеров.

Каждый найденный вопрос должен быть приведен к нормальной форме и привязан к узкому классу вопроса.

Пример:

- исходные формулировки:
  - "Сколько стоит подготовка к ЕГЭ по математике?";
  - "А цена за математику 11 класс какая?";
  - "Сколько в месяц занятия по профилю?";
- единый узкий класс:
  - "Стоимость подготовки к ЕГЭ по профильной математике для 11 класса";
- статус:
  - ответ нужен РОПу на проверку;
  - можно отвечать ботом только после утверждения цены.

## 3.1. Важное правило: ответ состоит из шаблона и актуальных фактов

Вопрос-ответ нельзя проектировать как статичный текст, особенно для цен, расписания, скидок, адресов и набора групп.

Правильная модель:

1. Есть узкий класс вопроса.
2. Есть утвержденный шаблон идеального ответа.
3. Есть отдельный источник актуальных фактов: файл с ценами, файл с расписанием, файл с адресами, файл с условиями рассрочки, файл с правилами пробных занятий.
4. При подготовке ответа система берет шаблон и подставляет в него свежие факты.
5. Если свежего факта нет, факт устарел или источники противоречат друг другу, бот не должен уверенно отвечать. Он должен передать вопрос менеджеру или дать безопасный ответ без конкретной цены/даты.

Пример:

- класс вопроса: "Стоимость подготовки к ЕГЭ по профильной математике для 11 класса";
- шаблон ответа: "Подготовка к ЕГЭ по профильной математике для 11 класса стоит {price} за {period}. Формат: {format}. Можно начать с {trial_option}.";
- источник фактов: актуальный файл цен;
- подставляемые поля: цена, период оплаты, формат, пробное занятие, дата актуальности;
- если цена не найдена или файл старше допустимого срока, бот не отвечает конкретной суммой и передает вопрос менеджеру.

Значит, каталог вопросов должен хранить не только "какой ответ дать", но и какие актуальные факты нужны для ответа.

## 6. Минимальная схема одного вопроса

Каждая строка в `customer_question_items.jsonl` должна содержать:

- `question_item_id` - стабильный идентификатор вопроса;
- `source_channel` - источник: звонок, Telegram, почта;
- `source_ref` - ссылка на исходный артефакт без раскрытия лишних персональных данных;
- `occurred_at` - дата и время, если известны;
- `customer_text_redacted` - очищенный текст вопроса клиента;
- `manager_text_redacted` - очищенный ответ сотрудника, если есть;
- `question_class_id` - ссылка на узкий класс вопроса;
- `intent` - что клиент хотел узнать или решить;
- `product` - продукт или направление, если понятно;
- `grade` - класс ученика, если понятно;
- `subject` - предмет, если понятно;
- `format` - онлайн, очно, группа, индивидуально, если понятно;
- `price_related` - связан ли вопрос с ценой;
- `schedule_related` - связан ли вопрос с расписанием;
- `documents_related` - связан ли вопрос с документами;
- `safety_flags` - риски: персональные данные, деньги, обещания, спорный ответ;
- `answer_evidence_status` - есть ли подтвержденный ответ;
- `answer_source` - откуда взят ответ, если он есть;
- `requires_dynamic_facts` - нужны ли актуальные факты для ответа;
- `dynamic_fact_types` - какие факты нужны: цена, расписание, адрес, скидка, рассрочка, пробное занятие, документы;
- `fact_freshness_required` - насколько свежими должны быть факты.

## 7. Минимальная схема класса вопросов

Каждая строка в `customer_question_classes.csv` должна содержать:

- `question_class_id` - стабильный идентификатор класса;
- `canonical_question` - нормальная формулировка вопроса;
- `narrow_scope` - что именно входит в этот класс;
- `exclusions` - что не входит, чтобы не смешивать разные вопросы;
- `examples_redacted` - 3-5 очищенных примеров;
- `count_total` - сколько раз вопрос встречался;
- `count_calls` - сколько раз в звонках;
- `count_telegram` - сколько раз в Telegram;
- `count_email` - сколько раз в почте;
- `first_seen_at` - первая встреча;
- `last_seen_at` - последняя встреча;
- `products` - связанные продукты;
- `grades` - связанные классы;
- `subjects` - связанные предметы;
- `answer_status` - статус ответа;
- `answer_template_id` - ссылка на шаблон ответа, если он есть;
- `required_fact_keys` - какие актуальные факты нужно подставить в шаблон;
- `fact_source_refs` - из каких файлов или таблиц должны браться факты;
- `fact_freshness_policy` - правило свежести фактов;
- `fallback_when_fact_missing` - что делать, если факта нет или он устарел;
- `bot_permission` - может ли бот отвечать сам;
- `manager_handoff_reason` - почему нужен менеджер, если нужен;
- `rop_review_priority` - приоритет проверки РОПом.

## 8. Статусы ответа

Для каждого класса вопроса должен быть один из статусов:

- `approved_answer_exists` - есть надежный утвержденный ответ;
- `draft_answer_exists_needs_review` - есть черновик, нужна проверка РОПа;
- `needs_rop_answer` - ответа нет, РОП должен дать формулировку;
- `manager_only` - бот не должен отвечать сам;
- `source_conflict` - источники противоречат друг другу;
- `outdated_or_time_sensitive` - ответ зависит от актуальных цен, расписания или набора;
- `template_ready_needs_current_fact` - шаблон ответа готов, но для ответа нужен свежий факт из актуального файла;
- `fact_missing_or_stale` - нужный факт отсутствует или устарел;
- `not_enough_context` - без уточнения нельзя ответить безопасно.

## 8.1. Источники актуальных фактов

Нужно сразу заложить отдельный реестр источников фактов. Это не сами вопросы клиентов, а документы, из которых бот будет брать актуальные значения.

Минимальные типы источников:

- цены и абонементы;
- скидки, рассрочка и акции;
- расписание групп;
- адреса и форматы занятий;
- правила пробных занятий;
- правила записи и документы;
- программы по классам и предметам.

Для каждого источника нужно хранить:

- путь к файлу или таблице;
- владелец файла;
- дата последнего обновления;
- срок, после которого факт считается устаревшим;
- какие поля из него можно подставлять в ответы;
- что делать, если файл не найден, пустой или противоречит другому источнику.

Важное правило: вопросы про цены и расписание не должны получать статус "бот может отвечать сам", если не настроен источник актуальных фактов и проверка свежести.

## 9. Этапы реализации

### Этап 1. Инвентаризация источников

Цель: понять, из каких файлов реально можно извлекать вопросы.

Результаты:

- создан `source_coverage_report.md`;
- перечислены все доступные источники звонков, Telegram и почты;
- для каждого источника указан статус: можно читать, частично можно читать, заблокировано, нет текста;
- отдельно указано, какой период покрыт;
- зафиксированы пробелы, например отсутствие части Telegram-истории или неполная выгрузка почты.

Этап нельзя считать завершенным, пока не проверены:

- последняя база знаний по звонкам;
- последний пакет проверки РОПом;
- локальные Telegram-артефакты;
- локальные почтовые артефакты;
- документы базы знаний КЦ.

### Этап 2. Контракты и тесты

Цель: сначала зафиксировать структуру данных, чтобы дальше не строить одноразовые таблицы.

Результаты:

- создан модуль `src/mango_mvp/question_catalog/`;
- описаны структуры вопроса, класса вопросов и отчета;
- написаны тесты на обязательные поля;
- написаны тесты на стабильные идентификаторы;
- написаны тесты на запрет сырых телефонов, email и полных ФИО в публичных выходных файлах.

### Этап 3. Извлечение вопросов из звонков

Цель: использовать уже готовую обработку звонков без повторного запуска распознавания и анализа.

Результаты:

- извлечены вопросы из `enriched_reviews.csv`;
- извлечены полезные пары "вопрос клиента - ответ менеджера";
- сохранена связь с сигналом, продуктом, исходом и статусом безопасности;
- не изменены runtime-БД, аудио и транскрипты.

### Этап 4. Извлечение вопросов из Telegram

Цель: использовать локальный архив Telegram, не отправляя и не получая live-сообщения.

Результаты:

- прочитаны доступные локальные Telegram-экспорты;
- выделены реальные входящие вопросы клиентов;
- выделены ответы сотрудников, если они находятся рядом в диалоге;
- групповые, канальные и служебные сообщения отделены от клиентских;
- если текст недоступен в текущем артефакте, это явно записано как блокер.

### Этап 5. Извлечение вопросов из почты

Цель: использовать локальные почтовые архивы без новых live-действий.

Результаты:

- прочитаны доступные локальные выгрузки почты;
- выделены вопросы клиентов и ответы сотрудников;
- вложения не публикуются в выходных файлах, но учитываются как источник контекста, если уже извлечен текст;
- письма-рассылки, автоответы и технический мусор отделены от клиентских вопросов;
- если часть почты не выгружена, это записано в отчет.

### Этап 6. Нормализация и группировка

Цель: превратить тысячи сырых формулировок в понятные узкие классы вопросов.

Результаты:

- одинаковые по смыслу вопросы объединены;
- слишком широкие классы разбиты на узкие;
- вопросы про цену, расписание, документы, формат, адрес, программу и пробное занятие не смешиваются;
- для каждого класса есть примеры;
- группировка повторяема: одинаковый вход дает одинаковый результат.

### Этап 7. Связь с базой знаний и статусами бота

Цель: понять, на какие вопросы уже можно отвечать, а какие надо отдать РОПу.

Результаты:

- каждый класс вопроса сопоставлен с базой знаний КЦ, сайтами, звонками или ответами сотрудников;
- каждый класс вопроса сопоставлен с шаблоном ответа, если шаблон уже возможен;
- для вопросов про цены, расписание, скидки, рассрочку, адреса и набор групп указано, какие актуальные факты нужно подставить;
- создан реестр файлов-источников для актуальных фактов;
- отмечены противоречия между источниками;
- вопросы про цены, расписание, скидки и рассрочку помечены как чувствительные к актуальности;
- для чувствительных вопросов задано поведение при отсутствии свежего факта;
- для каждого класса указан статус: бот может отвечать, нужен черновик, нужен РОП, нужен менеджер.

### Этап 8. Пакет для РОПа и сотрудников

Цель: получить не просто технический файл, а рабочий инструмент для утверждения ответов.

Результаты:

- создан `rop_question_review_pack.xlsx`;
- в нем есть приоритеты;
- в нем есть реальные очищенные примеры;
- в нем есть место для идеального ответа;
- в нем есть место для решения: бот может отвечать сам или должен передавать менеджеру;
- отдельно собран список самых частых вопросов без утвержденного ответа.

### Этап 9. Аудит, тесты и итоговый отчет

Цель: убедиться, что слой пригоден для дальнейшей разработки бота.

Результаты:

- прогнаны все тесты нового слоя;
- создан итоговый отчет `docs/CUSTOMER_QUESTION_CATALOG_AUDIT_2026-05-13.md`;

- указано, сколько классов можно отдавать боту;
- указано, сколько классов требует РОПа;
- указано, какие блокеры остались.

## 10. Распределение на 6 субагентов

Codex может привлечь до 6 субагентов. Каждый субагент работает только на чтение и возвращает конкретный результат.

1. Субагент по звонкам:
   - проверяет последнюю базу знаний по звонкам;
   - описывает, какие поля можно использовать;
   - находит ограничения и риски.

2. Субагент по Telegram:
   - проверяет локальные Telegram-артефакты;
   - определяет, где есть тексты сообщений;
   - предлагает правила выделения клиентских вопросов.

3. Субагент по почте:
   - проверяет локальные почтовые архивы;
   - определяет, где есть тексты писем и вложений;
   - предлагает правила отделения реальных клиентских вопросов от рассылок.

4. Субагент по структуре данных:
   - проверяет, достаточно ли полей в вопросе и классе вопроса;
   - ищет, какие поля нужны будущему боту и единой истории клиента.

5. Субагент по безопасности и качеству:
   - проверяет, нет ли сырых персональных данных в выходных файлах;
   - проверяет, что бот не получает право отвечать на спорные вопросы.

6. Субагент по тестам и приемке:
   - проверяет, что есть тесты на все ключевые правила;
   - проверяет, что итоговая команда повторяемая;
   - проверяет, что есть понятный отчет для Дмитрия и РОПа.

Главный Codex отвечает за интеграцию, код, тесты, итоговый отчет и исправление найденных проблем.

## 11. Критерии готовности

Этап считается завершенным только если выполнено все:

1. Есть единый каталог вопросов из доступных источников.
2. Источники явно перечислены, включая непокрытые и частично покрытые.
3. Звонки использованы из готовых артефактов, без нового распознавания и анализа.
4. Telegram использован только из локальных артефактов, без live-отправки и live-чтения.
5. Почта использована только из локальных артефактов, без удаления и без отправки писем.
6. Все выходные файлы не содержат сырые телефоны, email и полные ФИО в публичных примерах.
7. Для каждого класса вопросов есть частотность и источники.
8. Для каждого класса вопросов есть статус ответа.
9. Для вопросов с ценами, расписанием и другими изменяемыми фактами есть шаблон ответа и список нужных фактов.
10. Для актуальных фактов есть реестр источников и правило свежести.
11. Если факт отсутствует или устарел, бот не получает право отвечать конкретикой.
12. Для РОПа создан удобный файл проверки.
13. Есть список вопросов без ответа.
14. Есть список спорных или противоречивых вопросов.
15. Есть тесты на структуры, нормализацию, группировку, безопасность и команду сборки.
16. Команда сборки повторяемая.
17. Итоговый отчет объясняет, что можно использовать для бота уже сейчас, а что нельзя.

Если какой-то источник невозможно обработать, работа не останавливается. Нужно:

- обработать все остальные источники;
- записать точную причину блокера;
- указать, что нужно сделать для разблокировки;
- включить блокер в итоговый отчет.

## 12. Что делать сразу после этого этапа

После появления единого каталога вопросов следующий этап:

1. Создать черновики идеальных ответов по каждому приоритетному классу.
2. Для изменяемых тем создать не статичные ответы, а шаблоны с подстановкой актуальных фактов.
3. Подключить файлы с ценами, расписанием, скидками, адресами и правилами записи как источники фактов.
4. Разделить ответы на:
   - бот может отвечать сам;
   - бот задает уточняющий вопрос;
   - бот передает менеджеру;
   - нельзя отвечать без актуального расписания или цены.
5. Отдать РОПу и сотрудникам файл проверки.
6. После проверки собрать утвержденную базу ответов.
7. Подключить утвержденные шаблоны и источники фактов к сервису предпросмотра ответов бота.

Это уже будет следующий слой: не "что спрашивали", а "как идеально отвечать".
