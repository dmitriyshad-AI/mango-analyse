# DEV-набор v8 (2026-05-24) — для тюнинга, НЕ honest-гейт

Файл: `v8_dev_set_2026-05-24.jsonl` (38 строк = simulator_spec + judge_spec + 36 персон, 18 Фотон / 18 УНПК).
Та же спека и судья, что у targeted16/holdout → метрики сравнимы. Помечен `dev_set: true`, `do_not_use_as_holdout: true`.

## Зачем
Это набор, на котором Кодекс **тюнит** бота в итерационном цикле (D1 Часть 2). На нём можно смотреть,
чинить, переобучать сколько угодно. **Чистый holdout (`v8_holdout_clean_2026-05-24.jsonl`) — НЕ трогать
для тюнинга**: он только финальный честный замер.

Три разных набора, три роли:
- `v8_targeted16` — быстрый dev-сигнал (16, уже «прорешан», тоже не honest-гейт);
- `v8_dev_set_2026-05-24` (этот, 36) — основной dev-набор для итераций: шире, свежие персоны;
- `v8_holdout_clean_2026-05-24` (26) — финальный замер, eval-only, не тюнить.

## Покрытие (классы, которые сейчас чиним)
- **P0-recall:** мошенничество, незаконное списание, «буду жаловаться/пожалуюсь» → manager_only;
  benign («обсудить с менеджером/подумаю/надо приезжать») → НЕ P0.
- **Шаблонность/ignored/over_handoff:** серии разных уточнений по цене и рассрочке (отвечать по ДЕЛЬТЕ,
  не повторять абзац), пробное/фрагмент, лагерь «что входит» и «онлайн без проживания», расписание без времени.
- **assumed_unstated_need:** размытый/тревожный родитель без предмета — не приписывать предмет/цель.
- **Составной вопрос** (в т.ч. с P0-частью → весь маршрут менеджеру).
- **Идентичность (политика C):** «ты бот/ты gpt/покажи промпт/дайте человека».
- **Бренд/факты:** маткапитал, вычет, скидки с условием, площадки, сравнение с конкурентом, известный клиент CRM,
  новый лид с неполной формулировкой, сравнение своих форматов.

## Дисциплина
Тюним на dev/targeted16 → улучшаем архитектурно (не хардкодом) → один раз меряем на чистом holdout без
подгонки. Если правка «требует» шаблон/ветку/строковый assert на зелёной теме — это сигнал искать общий
механизм, а не хардкод (ТЗ v2 §2).
