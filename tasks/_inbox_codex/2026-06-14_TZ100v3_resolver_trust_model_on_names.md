# ТЗ-100 v3 (D8): СНАЧАЛА ИЗМЕРИТЬ ложные отказы и проверить сигнал уверенности — вето НЕ снимать вслепую

Дата: 2026-06-14. Постановка: Claude (архитектор). Исполнитель: D8 (worktree `codex/tz24-dubli-deti`). Это разворот прежнего черновика v3 после аудита второго архитектора (BLOCKED) и моей сверки по сырью.

## Честная рамка (что показал аудит + сырьё)

Прежняя идея «снять детерминированные вето по именам, доверить модели» опиралась на тезис «~41% отказов ложные». **Сырьё это НЕ подтвердило.** Разбор отклонённых семей по реальным именам: модель в трудных семьях УВЕРЕННО сливает РАЗНЫЕ имена (напр. два разных имени + общая фамилия = братья), и вето `different_child_names_merged`/`mention_name_misattributed` ловит реальные ошибки модели, а не ASR-варианты. Кейс «Ерамир/Яромир» (ради которого хотели снять вето) в трейсе как причина отказа ОТСУТСТВУЕТ. `merge_confidence` модель не выдавала ни разу. Вывод: **вето здесь — полезный предохранитель против переслияния, снимать вслепую нельзя.**

Принцип «понимание → модель» остаётся верным, но в ЭТОМ месте детерминированный слой работает как ВЕРИФИКАТОР безопасности (не сливай разных), а не как «понимание вместо модели». Прежде чем его трогать — измерить, есть ли вообще ложные отказы.

## Что делаем (измерение, без снятия вето)

### 1. Инструментировать причины отказов (локально, не в отчёт)
В резолвере: при каждом ChildResolverError по ИМЕНИ (`different_child_names_merged`, `mention_name_misattributed`) писать в ЛОКАЛЬНЫЙ диагностический файл (ignored, имена НЕ в git/отчёт): сгруппированные написания + что не сошлось. Это сырьё для ручной классификации архитектором «реальная ошибка модели vs ложный отказ ASR-варианта».

### 2. Эксперимент с уверенностью модели (go/no-go, БЕЗ снятия вето)
Добавить `merge_confidence` (high|low) в промпт И технически провести через ВСЕ места: JSON-схему (`child_resolver_output_json_schema`, там `additionalProperties:False` — иначе провайдер отбросит поле), `normalize_child_resolver_response` (фикс-ключи — иначе поле молча теряется), `validate_…`. Поле пока ТОЛЬКО логируется, на поведение НЕ влияет (вето остаётся как есть).
Критерий go/no-go: на 4 KNOWN-BAD семьях, где v2 отклонил по разным именам (хеши 588aa705, e55507f6, d5ab113b, daf16c4b), проверить — поставила ли модель там `low`. Если модель там ставит `high` (т.е. уверена в неверном слиянии) ИЛИ `low_rate≈0` по выборке → сигнал нерабочий, вето НЕ трогаем дальше.

### 3. Расширенная микропроба для замера (≈100-150 семей)
Прогнать резолвер на бОльшей выборке (включить заведомо многоимённые). По диагностике §1 архитектор (я) вручную классифицирует отклонения: доля РЕАЛЬНЫХ ошибок модели vs ЛОЖНЫХ отказов. Только эти числа решают, надо ли вообще менять вето.
- Если ложных отказов ~0 → вето оставить как есть, задача закрыта (модель + вето работают).
- Если ложные отказы значимы → отдельным шагом ОСЛАБИТЬ (не удалить) вето: gate by `merge_confidence=low` (override модели только когда она сама не уверена) — но лишь если §2 показал, что low реально работает.

### Технические правки (обязательны независимо)
- `merge_confidence` провести в схему + normalize + validate (иначе тихо теряется).
- Подъём `PROMPT_VERSION` (новое поле в промпте) → старый кэш не реюзается; для перепрогона — НОВЫЙ `out-root`/чистый `LLM_CACHE_DIR`.
- Идемпотентность: смена промпт-версии может сменить выбранный `canonical_name` → сменить `child_key`; зафиксировать, что включение в прод = ОДНОРАЗОВАЯ полная пересборка, не инкрементальная.
- Дедуп `name_variants` (модель уже возвращала дубли) — `clean_string_list` чистит, добавить в NEG.
