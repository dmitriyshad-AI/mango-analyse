# ТЗ — «Миссия и образцы»: дать боту понятия о хорошем (позитивный слой)

**Статус:** ФИНАЛ v2 (аудит второго архитектора: 2 критичные дыры закрыты, факт-чек кодом)
**Дата:** 2026-06-08
**Основание:** вердикт Дмитрия по транскриптам («знает что нельзя, не знает что такое хорошо»); застой метрик 3 волны. Факт-чек аудита: промпт = ~31 200 символов, **130 правил, из них 92 запрета** — диагноз «клетка» подтверждён числом.

## M1. Миссия-преамбула и переструктура

Преамбула (дух; финальный текст целиком — подпись Дмитрия):
«Ты — менеджер-консультант учебного центра [бренд]. Родитель пришёл с задачей про ребёнка. Твоя цель: реально помочь разобраться и довести до записи на подходящий курс. Продажа — это помощь: польза с первого ответа, предугадывай следующий вопрос, веди к понятному шагу. Не дави; честность важнее сделки. **Если правило безопасности или передача менеджеру противоречит цели записи — правило важнее.**»
Структура: миссия → 4 принципа помощи (= ПЕРЕНОС существующего playbook-блока промпта :179-186, не новый контент) → факты/контекст → жёсткие ограничения (короткий блок).

## M2. Золотые образцы

- **Канал требует шлюза (критично):** few_shot_style_examples живой и работает БЕЗ флага (build_few_shot_reference вызывается безусловно, pilot_context:205). Новые образцы — ТОЛЬКО в отдельном паке (свой путь по образцу MANGO_TELEGRAM_FEW_SHOT_WARM_PATH) или с тегом `mission_gold: true`, который фильтруется ТОЛЬКО при `TELEGRAM_BOT_MISSION_GOLD=1`. Иначе флаг — фикция.
- **Стиль масок (критично):** НЕ value-плейсхолдеры («[цена из фактов]» — санитайзер их НЕ ловит, утекут клиенту). Использовать обкатанный ИНСТРУКЦИОННЫЙ стиль `_mask_precise_claims` («[цену не называть без факта]», few_shot_reference:351-359), а лучше — образцы вовсе без скобочных токенов («точную сумму по вашей программе назову после уточнения»). Дополнительно: расширить strip-регекс на скобки со словом «факт».
- **Бюджет:** пул 6-10 образцов на тему/бренд в YAML; пер-ход кап (6 шт. / 900 симв.) НЕ трогаем — канал сам отберёт 2-4 под тему.
- **Источник (решение Дмитрия 2026-06-08: «думающий бот, не шаблоны»):** НЕ рукотворные тексты (ни Кодекса, ни архитектора — оба черновика отклонены как «написанные»). Образцы = **РЕАЛЬНЫЕ обмены «вопрос клиента → ответ живого менеджера» из Telegram-экспорта** (3000+ исходящих сообщений менеджеров). Процесс: архитектор отбирает лучшие обмены (критерии: польза с первого ответа, тепло, чёткий шаг, бренд-чистота — менеджеры иногда мешали бренды, фильтровать!) → анонимизация + инструкционные маски чисел → линтер → утверждение Дмитрия. Плейбук звонков — только как критерий отбора, не как текст. Retention-скрипты — **M2 ПОГЛОЩАЕТ мини-ТЗ B4**.
- **Линтер пака (детерминированный тест):** каждый образец кончается шагом; нет цифр вне инструкционных масок; бренд-тег обязателен; рассрочка/Долями/пробное — только foton-записи.
- **Состав образцов утверждает Дмитрий до коммита.**

## M3. Анти-выдумочный якорь

Якорь «числа/даты/условия только из фактов; не знаешь — честно скажи и предложи шаг» — это ПЕРЕНОС существующего барьера (:173-174) на видное место, не дубль: диета M4 не должна удалить оригинал как «продублированный». Страховки не ослабляются (число-гард, верификатор со скип-гейтом, детекторы, судья v9 на замере).

## M4. Промпт-диета (130 правил → цель ≤70; отрицаний 92 → ≤45)

Классификация удаляемого — чекпойнт-список с колонкой «что покрывает» (детектор/образец/NEG/судья) и греп-паттерном провала для контрольного прогона:
- **НЕ удалять** (факт-политика, не стиль): «не приглашай приехать», «по лагерям класс не возраст», «не называй сроки связи менеджера», правило `recent_ignored>=2` (:688 — неудаляемо, держит анти-наседалу) → перенести в жёсткий блок.
- **Удалять безопасно** (есть детерминированный бэкстоп): догадки по расписанию (детектор :4451), повторы числовых запретов (число-гард+верификатор).
- **Удалять только с заменой образцом**: «не анкетируй» (образец-замена обязан показываться и в no-fact ветке — иначе дыра ровно там, где анкета и возникала), «не начинай одинаково».
Список утверждают: архитектор (структура) + Дмитрий (клиентские формулировки).

## M5. Граница

P0/safety-шаблоны и маршрутизация нетронуты; бренд-фильтрация образцов; новых LLM-вызовов нет; **MISSION_GOLD в пилотные каналы — НИКОГДА до приёмки Дмитрия** (как shadow Шага 5).

## Флаги и откат (раздельные — для ответа на вопрос «нужны ли образцы вообще»)

ДВА флага (default OFF): `TELEGRAM_BOT_MISSION` (M1 преамбула + M4 диета) и `TELEGRAM_BOT_GOLD_REAL` (M2 реальные образцы; работает только поверх MISSION). Это позволяет измерить «миссия-только» vs «миссия+реальные образцы»: если думающему боту хватает миссии — образцы не нужны вовсе (гипотеза Дмитрия), если нет — увидим вклад. Откат: флаги OFF, пак остаётся тегированным, диета ревертится (отдельный коммит).

## Порядок с Шагом 5 (подтверждён аудитом)

1. Волна 3.1 — в замере (проверка сантехники, без полного регрейда).
2. Это ТЗ: реализация за флагом → согласование образцов Дмитрием → коммит (диета — отдельным коммитом).
3. **Freeze** → бандл → пара Шага 5: SHADOW-прогон с MISSION_GOLD=1 (shadow = превью финальной конфигурации: вето нейтрализовано, gold-стиль критик НЕ режет, только логирует — absorb-first был бы хуже: окно без критика до замера его дельты на gold-текстах) → мой регрейд дельты (теневая дельта + миссия-эффект в одном прогоне — мой регрейд тяжелее обычного, план учитывает) → ABSORB → контрольный замер.
4. **Диагностические клетки** (та же замороженная ревизия): SHADOW-прогон пары идёт с MISSION=1+GOLD_REAL=1; при провале exit или росте выдумок — клетка MISSION=1/GOLD_REAL=0 (проверка гипотезы Дмитрия «думающему боту хватит миссии») и/или MISSION=0/SHADOW=1 — по сигналу, не все сразу.
5. Пак Дмитрию — после пары.

## NEG

1. Выдумка числа/даты при свободе → ловится; контрольный прогон: 0 по сырью.
2. Образец Фотона не утекает в УНПК-диалог (линтер + прогон).
3. P0 при MISSION_GOLD=1 → маршрут/тексты не меняются.
4. Явный отказ → без дожима.
5а. Числа из образцов не материализуются (греп).
5б. **Скобочные токены не утекают**: греп `\[[^\]]*\]` по клиентским текстам = 0.
6. Промпт-диета не удалила жёсткие правила (чекпойнт-список с покрытием).
7. **Вялый клиент**: 2-3 хода без интереса → ≤1 CTA суммарно, телефон не переспрашивается (анти-наседала).
