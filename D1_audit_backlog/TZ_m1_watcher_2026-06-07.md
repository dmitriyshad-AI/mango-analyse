# ТЗ — Вотчер М1: автоматическая шина задач (снять Дмитрия с роли курьера)

**Статус:** ФИНАЛ v2 (аудит второго архитектора: 4 критичные дыры закрыты, NEG 6→11; 2 политических вопроса — Дмитрию)
**Дата:** 2026-06-07
**Цель:** задачи на прогонную станцию М1 идут файлами через Яндекс без ручной пересылки; Дмитрий уходит из контура исполнения, остаётся в контуре решений.

## Поток

1. Файл задачи кладётся в Яндекс `OpenClaw/Actual Mango Tests/tasks/_inbox_m1/`.
2. Вотчер на М1 (поллинг 2-5 мин) захватывает → исполняет → результаты в `runs/` → отчёт в `_done/` (или `_failed/`).
3. Статус виден по папкам Яндекса. **Правило «один писатель»:** главная машина пишет ТОЛЬКО в `_inbox_m1/` и `STOP`; М1 — ТОЛЬКО в `_running/`, `_done/`, `_failed/`, `runs/`, `watcher_status.txt`; чужие файлы никто не редактирует (главная защита от конфликт-копий Яндекса).

## Формат задачи

`YYYY-MM-DD_<тег>.task.yaml` + обязательный ready-маркер `<имя>.ready` (внутри sha256 yaml-файла; пишется ПОСЛЕ yaml):
```yaml
id: 20260607_wave2_tone_25_codex      # имя выходной папки выводится из id — отдельного output НЕТ
requires_bundle: mango_clean_7d69eefc  # строго ^mango_clean_[0-9a-f]{8}$; overlay-бандлы вне scope v1
set: <относительный путь внутри Яндекс-папки тестов>   # realpath внутрь папки, .jsonl/.yaml, без .., симлинки запрещены
set_sha256: <sha набора>               # защита от полу-синхронизированного набора
brain: codex                           # codex | claude (модель/auth-mode для claude зашиты в вотчере)
parallel: 4                            # максимум 4, больше — отказ
max_hours: 8                           # потолок; по истечении — terminate process group → _failed: timeout
env:                                   # ТОЛЬКО имена TELEGRAM_* (+DIALOGUE_CONTRACT_DEBUG_TRACE);
  TELEGRAM_TONE_WARM_FRAME: "1"        # значения ^[A-Za-z0-9_.:/\-]{0,64}$; всё прочее — отказ
```
Поля `script`/`output` отсутствуют намеренно: скрипт — константа (`run_telegram_dynamic_client_sim.py`), имя папки = id. `--snapshot` берётся из `BUNDLE_INFO.txt` бандла, НЕ из задачи. PYTHONPATH и прочую обвязку вотчер ставит сам и из задачи не принимает. Остальные CLI-дефолты зашиты константами и целиком печатаются в отчёт.

## Защита от не-атомарности Яндекса

1. **Задача**: берётся только пара yaml+ready со сходящимся sha; размер+mtime стабильны два опроса подряд; ошибка парсинга свежего файла → 3 цикла ожидания, только потом `_failed`. Конфликт-копии (`имя (1).task.yaml`) — игнор с предупреждением. `id`, уже встречавшийся в `_done`/`_failed`, → `duplicate_id` (защита от переигрывания при ресинке).
2. **Бандл**: главный Кодекс при сборке пишет `manifest.json` (путь+sha256+размер каждого файла, итоговое число) и кладёт его ПОСЛЕДНИМ. Вотчер разворачивает только при 100% сверке манифеста; несовпадение = «ещё льётся» → ждать до 2 часов → `_failed: bundle_incomplete`. Readiness-маркеры (rules_registry и т.п.) — второй рубеж. **Встречное требование сборщику бандлов — часть этого ТЗ.**
3. **Результаты**: сначала копия `runs/<id>` в Яндекс, отчёт в `_done/` — ПОСЛЕДНИМ: наличие отчёта = маркер полноты результатов для главной машины.

## Исполнитель — детерминированный скрипт, НЕ агент

Питон-скрипт + launchd. Никакого LLM в контуре исполнения: задача либо матчится на схему, либо `_failed: task_schema_mismatch`. Нестандартное — на ручной канал (осознанная граница). Исполняемая копия вотчера живёт ВНЕ рабочих копий (фиксированный путь), обновляется ТОЛЬКО руками оператора вне окна прогонов; версия вотчера — в каждом отчёте и в watcher_status.

## Безопасность

1. Allowlist операций: сверить+развернуть clean-бандл (каждый в СВОЮ папку `mango_m1_work/<bundle_id>/`, маркер `.deployed_ok`; ничего не удаляется и не перезаписывается — правило «удаления только с Дмитрием» соблюдено конструкцией); readiness; прогон симулятора по схеме; копирование результатов. Всё прочее — отказ.
2. Вотчер сам НЕ делает сетевых вызовов вообще; сеть разрешена только дочернему прогону (OpenAI/Anthropic).
3. Без pip install: общий venv; readiness-проверка `python3 -c "import mango_mvp"` + свободный диск + для brain: claude — валидность логина.
4. **Single-flight**: пока в `_running/` есть `*.task.yaml` — новые не берутся; терминальный переход уносит ВСЕ артефакты задачи из `_running/` разом.
5. **Сироты**: в `claimed`-файл пишутся pid вотчера и process group прогона; на старте вотчер проверяет живость сироты (kill -0 + ps) — жив → задач не брать (`orphan_run_alive`), ждать; `interrupted` ставится только по мёртвому процессу; перезапуск прерванной задачи — всегда НОВАЯ задача `<id>_v2` решением человека.
6. **Стоп-кран**: файл `tasks/STOP` → новые не брать (текущую довести); проверка перед каждым захватом. Прерывание ТЕКУЩЕЙ задачи — намеренно ручное действие оператора (прерывание замера = человеческое решение).
7. **Heartbeat**: `_running/<id>.heartbeat`, перезапись атомарно (tmp+rename), раз в 5 мин: монотонный счётчик, стадия, возраст mtime `dynamic_dialog_transcripts.jsonl` (stall-индикатор: жив, но не пишет). Тишина >30 мин или stall + max_hours — наружу видно/обрывается.

## Наблюдательский контракт (для главной машины)

Истина — терминальные папки и `_running/<id>.claimed` (host, pid'ы, ISO-8601 время). «Исчез из inbox, не появился в _running» = «в движении» (delete+create синка), не аномалия. `tasks/watcher_status.txt` перезаписывается каждый опрос: жив/версия/текущая задача/хвост лога. Логи — вне рабочих копий (`~/m1_watcher/logs/`, ротация по дате). Все таймстемпы ISO-8601 с офсетом.

## Очередь

FIFO лексикографически по имени файла (датный префикс = хронология), по одной задаче.

## Отчёт в _done

`<id>.report.md`: bundle/head + результат сверки манифеста, версия вотчера, версии codex/claude CLI, readiness, ПОЛНАЯ фактическая команда, host, старт/конец ISO, статус, путь к результатам, числа из dynamic_summary (PASS/PWN/FAIL, hard_gate, turns, llm_calls) — без интерпретации.

## Тест входа (Кодексу, read-only)

1. Формализовать текущие ручные шаги оператора (PROMT_M1-практика) — но развёртывание В ОТДЕЛЬНУЮ папку по манифесту, а не «чистое» с удалением (старый паттерн и вызвал инцидент).
2. Какие launchd/cron уже есть на М1 (не плодить дубли).
3. Дизайн на 1 страницу + финальная схема `.task.yaml` — мне до кода.

## NEG-контроли (11)

1. `parallel: 8` → отказ. 2. Имя env вне allowlist (`PATH`) → отказ; `set` с `..`/абсолютный наружу → отказ; конфликт-копия → игнор+предупреждение. 3. STOP → не берёт; после удаления — берёт. 4. Вторая задача при живом running → ждёт. 5. Бандл без манифеста / sha не сошёлся → ждёт → по таймауту `bundle_incomplete`. 6. Вотчер убит при мёртвом прогоне → `interrupted`, не перезапускает. 7. Вотчер убит при ЖИВОМ прогоне → `orphan_run_alive`, второй прогон не стартует. 8. Две задачи → FIFO по имени. 9. Обрезанный yaml без ready → не взят и НЕ в _failed; после докачки — взят. 10. `id`-дубль → `duplicate_id`. 11. max_hours истёк → terminate группы → `timeout`.

## Exit

Полный цикл без ручного вмешательства (задача в Яндекс → результаты+отчёт в Яндексе); все 11 NEG живьём на М1 (прогон-пустышка 1-2 диалога); Дмитрий за цикл — ноль действий.

## Политика — РЕШЕНО Дмитрием 2026-06-07

1. **Скользящее подтверждение «не более 3 без ревью»**: вотчер исполняет задачи свободно, но счётчик неподтверждённых задач ≤3. Подтверждение = файл `tasks/ACK_<дата>.md` от Дмитрия («всё хорошо» или правки процесса) — сбрасывает счётчик. 4-я задача без ACK ждёт в inbox со статусом `awaiting_ack` в watcher_status. Дмитрий остаётся в контексте, проверяет периодически. → строка в CLAUDE.md.
2. **Дневного лимита НЕТ** (решение Дмитрия: доверие + дисциплина «замер = 1 прогон»).
3. Регламент уборки старых копий бандлов (266 МБ каждая) и runs на М1 — периодический ручной ритуал с ведома Дмитрия (удаления только с ним).
