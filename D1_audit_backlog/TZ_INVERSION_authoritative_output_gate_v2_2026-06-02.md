## Принцип (без изменений)
Модель ПОНИМАЕТ (падает мягко переспросом); детерминированная охрана проверяет ГОТОВЫЙ ЧЕРНОВИК на ВЫХОДЕ,
только ПОНИЖАЕТ/блокирует, не выдумывает. Безопасность с ВХОДА → на ВЫХОД. Монолит держит 407 инлайн-решений
vs 41 вызов модулей — инвертируем по этому шву.

## ИСПРАВЛЕНИЕ 1 — гейт КОМПОНУЕТ существующие гарды, не 4 вызова (полнота безопасности)

`unsupported_entity`, `ai_disclosure`, `forbidden_scope`, `p0_promise`, `p0_semantic_risk`. НО монолит
ДОПОЛНИТЕЛЬНО ловит то, чего в verify_output НЕТ — гейт ОБЯЗАН их включить (переиспользуя существующие
функции, не переписывая):
- `guard_promocode_leak` (`PROMOCODE_DRAFT_RE` 459) — утечка рабочих кодов.
- `guard_draft_placeholder` (`DRAFT_PLACEHOLDER_RE` 455) — недоделанный черновик `[вставить]`.
- `guard_identity_disclosure` (1900) — «я ИИ/GPT».
- `find_unsupported_offline_visit_invitation_claims` (2169) — приглашение в офис без подтверждения.
- zero-collect (сбор ФИО/договора в P0) — из `apply_input_policy_guards`.
- операционная специфика: future_price / schedule_assumption / followup_deadline — из
  `apply_unconfirmed_operational_specificity_guard`.
- taxonomy/autonomy-matrix (тема валидна, тема×бренд разрешены для автономии).
Гейт = ЕДИНЫЙ конвейер, ВЫЗЫВАЮЩИЙ эти проверенные функции по черновику. ПЕРЕД фазой 4 (снос монолита) —
полный аудит: КАЖДАЯ инлайн-защита монолита либо в гейте, либо доказанно дублируется verify_output.

## ИСПРАВЛЕНИЕ 2 — верные сигнатуры (v1 ошибался)

## ИСПРАВЛЕНИЕ 3 — гейт во ВСЕХ точках финального return (v1 указал 1)
Финальные return: `SubscriptionLlmDraftProvider.build_draft` 1104 (v2-путь) И 1138 (legacy

плейсхолдер/identity/промокод/мета — они не требуют facts). Подтвердить чтением все 4 точки.

## ИСПРАВЛЕНИЕ 4 — конфликт «канон заморожен» vs калибровка (v1 тупик)
Заморожена ЛОГИКА модулей (что они детектят). НО ПОЛИТИКА гейта (какие коды findings → блок/понижение,
какие → только флаг) — НАСТРАИВАЕМАЯ (`GATE_BLOCKING_CODES`). Если замер покажет over-handoff от строгости
(напр. `fact_grounding` на валидном диапазоне) — калибруем BLOCKING-политику гейта, НЕ трогая логику
модуля. Это снимает тупик: канон неизменен, поведение гейта настраивается на его выходе.

## ИСПРАВЛЕНИЕ 5 — замер фазы 1 СОПОСТАВИМЫЙ (v1 сравнивал только routes)
Монолит судит по ВХОДУ, гейт по ВЫХОДУ — простое сравнение routes даст ложный сигнал. Правильный замер —
ТЕНЕВОЙ прогон «гейт БЕЗ монолита» в песочнике:
- временно отключить инлайн-власть монолита (флаг), оставить только гейт → прогон на наборе + P0/бренд/
  утечки кейсы;
- КРИТЕРИЙ: гейт-ОДИН ловит ВСЁ, что ловил монолит (HARD-P0→manager или безопасный хендофф; бренд/мета/
  промокод/плейсхолдер/identity/выдумки = 0 утечек), И over-handoff не вырос значимо, И тон не упал.
- Параллельно обычный прогон (монолит+гейт) — двойная защита, эталон безопасности.
Только если гейт-ОДИН держит — фаза 2. Это прямой тест «может ли гейт заменить монолит», а не косвенный
route-diff.

## ИСПРАВЛЕНИЕ 6 — фаза 4 НЕ сносит доменную бизнес-логику (v1 переоценил снос)
Адвокат дьявола прав: из 407 инлайн-точек значимая часть — НЕ костыли понимания, а доменные ХАРДКОД-тексты
(скидки/рассрочка по брендам, стэкинг, многодетные). Их НЕЛЬЗЯ удалить/выучить моделью. Уточнение фазы 4:
сносим лазанью ПОНИМАНИЯ+МАРШРУТИЗАЦИИ (regex-детекторы намерения, дубли route-override) + мёртвый код
(`_semantic_recover_or_handoff` и т.п.); доменные факты/тексты ПЕРЕНОСИМ в чистый слой фактов/шаблонов ЗА
гейтом (не удаляем). Фаза 4 = «убрать угадывание», а не «убрать бизнес-правила».

## Фазы (с учётом исправлений)
1. Единый гейт (Исправления 1-4) поверх всего, во всех точках return; монолит остаётся.
2. ★ ТЕНЕВОЙ замер «гейт без монолита» (Исправление 5) → опц. калибровка политики (Исправление 4).
3. Снять инлайн-власть монолита группами (route-override → hint), после каждой группы юнит-щит + теневой замер.
4. Заменить regex-понимание моделью волнами + реестр fact_id.
5. Снести лазанью понимания/маршрутизации + мёртвый код; доменные факты/тексты → чистый слой за гейтом.
Гейт не выкидывать никогда.

## СТРАТЕГИЧЕСКАЯ РАЗВИЛКА (решение Дмитрия — проверка усилила вопрос)
Адвокат дьявола + Клод 3 сходятся: НЕ запускать инверсию (фазы 3-5) параллельно пилоту. Вариант, который я
теперь считаю сильнее: **фаза 1 строится в песочнике (не в main) сейчас; пилот-черновики идут на текущей
базе (P0 8/8, безопасно) с тестом X2-теплоты; полная инверсия (фазы 2-5) — ПОСЛЕ пилота, на реальных
данных, а не предположениях.** Это и есть «заморозить, не переписывать до пилота» (Клод 3). Решение за
Дмитрием: (A) фаза 1 в песочнике + пилот сейчас, инверсия после; или (B) инверсия сразу, пилот потом.
