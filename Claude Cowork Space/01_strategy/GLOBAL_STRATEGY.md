# Глобальная стратегия Mango Analyse — диагноз, путь, улучшения

Пин: git `649c9397`. Основано на полном аудите кода (`PROJECT_AUDIT.md`), базы (`KB_QUESTIONS_AND_EDITS.md`),
бота (`FINDINGS_quality_strategy.md`) и собственных стратегических доках проекта.

## 1. Главный диагноз (неожиданный, но он и есть корень)

**Стратегия проекта НА БУМАГЕ правильная. Проблема — исполнение разошлось со стратегией.**

Собственные аудиты проекта (2026-05-19) уже сделали верный вывод: сначала доказать пользу внутреннего
ИИ-сотрудника (бот-черновик для менеджера), SaaS — потом, и №1 риск = распыление на SaaS/почту/runtime.
Но по факту построено мимо этого приоритета:
- **productization/SaaS — 43.5K строк**, бот импортит 0 раз, внешних клиентов 0;
- **customer_timeline — 14.9K строк** истории клиента, выключена и не подключена к боту;
- **agent_runtime + deal/AMO writeback — ~11K строк**, gated/dead;
- **mail_archive 8K + ASR-sandbox 7.2K + ~15 мёртвых quality-модулей** — под выключенные/чужие функции.

Это десятки тысяч строк без живого потребителя. А сам приоритет — измеримый полезный человечный бот —
не доведён и не измеряется честно (KB v6.3 без smoke, нет чистого holdout, качество через шаблоны).

**Тот же паттерн, что whack-a-mole в боте, виден на УРОВНЕ ПРОЕКТА: строится ширина и каркас вместо того,
чтобы довести до конца одну вещь, которая доказывает ценность.** Это не техническая, а дисциплинарно-
исполнительская проблема. И чинить её надо так же: радикальный фокус + измерение пользы, а не объём.

## 2. Северная звезда (одна, измеримая)

**Один внутренний ИИ-сотрудник продаж, которым менеджеры Фотона/УНПК пользуются КАЖДЫЙ ДЕНЬ, и чьи
черновики доказанно полезны на чистом holdout и по метрике «менеджер отправил бы без правок, %».**

Пока эта звезда не достигнута — никакой ширины (новые менеджеры, сайт, почта, Max, SaaS). Это ровно вывод
собственных аудитов проекта; нужно лишь привести исполнение в соответствие.

## 3. Единый roadmap (один трек, по приоритету)

1. **Довести качество бота по ТЗ v2** (`TZ_FOR_CODEX_quality_architecture_v2_FULL_BUILD`): качество — в
   генерации, правила — только safety; честный судья; контракт ответа; один rewrite. Это и есть доказательство ценности.
2. **Гигиена базы:** прогнать smoke v6.3; свести сборку KB к ОДНОМУ источнику (YAML, убрать хардкод-патчи цен
   в билдере); добить P0-recall (мошенн/незаконн/жаловаться) и в коде, и в KB-триггерах.
3. **Включить уже построенный актив истории клиента** (самый высокоокупаемый parked): ослабить слишком
   строгий soft-гейт customer_timeline (82% manual — самоналожено, не качество данных) и подключить
   `read_only_customer_context` к промпту (сейчас роняется белым списком ключей). Бот наконец начнёт
   ИСПОЛЬЗОВАТЬ историю — это и есть вся премиса проекта, а код уже написан (15K строк).
4. **Честный замер-петля** (BA-01/02/03 из их же register всё ещё ОТКРЫТЫ): единый журнал пилота, дневной
   отчёт, «отправил бы без правок %», чистый holdout. Главная метрика — сколько менеджеров пользуются ежедневно.
5. **Заморозить остальное:** SaaS (кроме ядра захвата), agent_runtime, deal/AMO writeback автоматизацию,
   mail_archive — снять с обязательного зелёного прогона; пометить как parked. Ретайрнуть `TZ_FULL_VISION`
   как исторический. Оставить ОДИН живой roadmap (этот), чтобы планы не конкурировали.
6. **Ворота к ширине:** только после ежедневного использования + holdout-доказательства — новые менеджеры,
   форма сайта, почтовые черновики; рабочее место менеджера (черновик → отправка по одобрению); Max и SaaS — сильно позже.

## 4. Стратегические улучшения (что менять в способе работы)

1. **Исполнительская дисциплина «доводить до пользы, не строить ширину».** Один блок за раз, finish-before-width.
   Метрика-предохранитель на уровне проекта: строки-без-потребителя НЕ должны расти (как branch-count в боте).
2. **Метрика успеха должна отличать «полезно, потому что верно и конкретно» от «полезно, потому что безопасно
   уклончиво».** Сейчас «% полезных черновиков / 0 P0» проходит и пустой «уточню у менеджера». Мерить на
   held-out, не использованном для сборки KB (их собственный слепой пятно — утечка train/eval).
3. **Единый источник правды по фактам** (YAML), без хардкод-патчей в коде сборки.
4. **Семантика там, где нужна** (intent/P0/PII), regex — только как страховочный пол. Сейчас regex-потолок
   везде: непокрытое имя/новая формулировка/искажённая цифра проходят.
5. **Развязка приоритета от балласта:** боту не нужен SaaS; нужно включить историю и заморозить остальное.
6. **Закрыть два открытых внешних сигнала**, которых нет ни в одном доке: возможная компрометация @unpk_mipt
   (входы из Таллина) и демпинг (Фотон 93100 vs УНПК 114000). Эскалировать ответственным.

## 5. Куда двигаться ПОСЛЕ доработок бота (стратегический путь)

Когда бот доказан (ежедневное использование + holdout):
- **Персонализация историей:** включённый timeline даёт ответы с учётом реальной истории клиента — следующий
  скачок полезности (актив уже построен).
- **Рабочее место менеджера:** черновик → отправка одной кнопкой с одобрением; мягкие напоминания после
  утверждения правил частоты.
- **Ширина каналов:** форма сайта, почтовые черновики, потом Max.
- **AMO/Tallanto-запись:** сначала через одобрение менеджера, потом точечная автоматизация безопасных полей
  (контур уже построен и gated — включать по готовности, не раньше).
- **SaaS для других организаций — последним**, только при неоспоримой внутренней пользе и повторяемой установке.
  Ядро захвата (`mango_office_client`/`product_db`) к тому моменту уже есть; остальной SaaS-каркас разморозить тогда.

## 6. Что я как ревьюер утверждаю честно
- Стратегическая самооценка проекта точна, но с лагом: реальность (v6.3, автономность вкл.) обогнала доки;
  образ «безопасный draft-only» больше не соответствует автономным ботам на непрогнанной smoke KB.
- Главный риск сейчас — не безопасность (она держится) и не ASR (он зрел), а **распыление усилий + честность
  измерения качества**. Лечится фокусом и чистым holdout, а не новой стройкой.
- Это read-only анализ на снимке `649c9397`; правки — за Кодексом после совместного решения. Числа/строки —
  на момент снимка, перед действиями пересверить.
