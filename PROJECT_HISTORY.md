# PROJECT_HISTORY.md — история проекта Mango Analyse

Этот файл объясняет, почему проект пришёл к текущему состоянию. Он не является источником текущей правды. Текущая правда для Claude CLI — в `CLAUDE.md`, затем в `docs/CURRENT_STATE.md`, `docs/DECISIONS_LOG.md` и свежих audit packs.

Последнее обновление: 2026-05-18.

## Этап 0. Исходная точка

Проект начинался как обработка звонков Mango: скачать записи, распознать, разделить говорящих, понять содержание разговора, собрать историю общения по телефонам и подготовить полезные тексты для CRM.

Со временем проект стал шире:

- звонки;
- Telegram-переписки;
- почта;
- AMO;
- Tallanto;
- история клиента;
- база знаний;
- будущий Telegram-бот и email-помощник.

Главное изменение подхода: проект больше не просто «анализ звонков», а основа внутреннего ИИ-сотрудника отдела продаж.

## Этап 1. Звонки и качество обработки

Был собран pipeline:

- загрузка звонков;
- ASR;
- Resolve+Analyze;
- фильтрация несодержательных звонков;
- нормализация анализа;
- построение цепочек по телефонам;
- экспорт в слои для CRM и deal-aware.

Позже были выполнены блоки X/Y/Z:

- X — улучшение анализа звонков и защита от выдуманных продажных полей;
- Y — чистка текстов, денег, процентов и брендовых терминов;
- Z — hygiene: подозрительные потери текста и pin-check конфигов.

Эти блоки довели обработку звонков до более безопасного состояния, но старые данные требуют выборочного или полного пересчёта только по отдельному решению.

## Этап 2. Deal-aware и AMO

Проект перешёл от «истории по контакту» к «пониманию конкретной сделки».

Собрано:

- 709 кандидатов AMO;
- Stage100-аудит;
- 6 батчей для РОП-проверки;
- dry-run и readback-контуры;
- quality gate для опасных AMO-текстов;
- rollback/snapshot контракт для будущих live-write.

Важное решение: live-запись в AMO возможна только после snapshot/rollback, dry-run, readback и отдельного подтверждения Дмитрия.

## Этап 3. Customer Timeline

Сначала customer timeline выглядел как большой слой без потребителя. После аудита решили не удалять его, а интегрировать как read-only слой истории клиента.

Контрольные прогоны показали:

- техническая склейка по телефонам работает хорошо;
- главная проблема не timeline, а смысл Tallanto-меток, семейные телефоны, дубли учеников и новые лиды без Tallanto;
- `timeline_preview_enabled` и `timeline_primary_read_enabled` пока не включать.

Следующий шаг по timeline: ручной разбор сложных Tallanto/семейных кейсов и уточнение правил.

## Этап 4. Каталог вопросов и РОП-опросник

Была собрана база клиентских вопросов из звонков, Telegram и почты. На её основе подготовлен опросник для РОПа:

- где бот может отвечать сам;
- где нужен менеджер;
- какие данные нужно спросить;
- какие обещания запрещены;
- где брать актуальные факты.

Позже стало понятно, что standalone-классификация «одна фраза → тема» хуже реального режима бота, потому что реальный бот будет видеть контекст: историю, сделку, правила РОПа и факты.

Решение:

- не делать standalone F1 0.85 блокером пилота;
- главный фокус — контекстная классификация и черновик в одном вызове;
- исторические 9 969 вопросов полезнее анализировать с контекстом, а не как вырванные фразы.

## Этап 5. Telegram-пилот

Приняты решения:

- первый запуск только как черновики для менеджера;
- long polling на MacBook;
- отдельный бот через BotFather;
- первый менеджер — Настя;
- клиенту напрямую не отправлять;
- AMO/Tallanto читать только read-only;
- high-risk темы вести к менеджеру;
- полезность измерять по ежедневному использованию, доле полезных черновиков и отсутствию опасных обещаний.

В коде появились:

- контекстный prompt builder;
- защита от prompt injection;
- debounce сообщений клиента;
- route `manager_only`;
- запрет раскрывать «я бот/ИИ/GPT/Claude/Codex»;
- защита от неподтверждённых чисел, процентов и сроков в черновике.

## Этап 6. База знаний

Первый ночной сбор дал инфраструктуру, но не дал готовой базы фактов: многие факты требовали подтверждения, а Stage 6 был частично на fake-provider.

После этого был сделан новый подход:

- факты разделены на Фотон и УНПК;
- введено жёсткое разделение брендов;
- промокоды убраны из клиентского слоя;
- скидки требуют условия применения;
- лицензии и юрлица вынесены во внутренний слой;
- добавлен semantic gate;
- добавлен smoke-прогон 50 вопросов через реальный Codex;
- собраны два пакета: для сотрудников и для бота.

Текущий статус v3.2:

- `formal_pass=true`;
- `semantic_pass=true`;
- блокеров 0;
- P2-риск: части фактов нужен явный срок действия;
- режим: внутренняя база черновиков и справочная база, не автономная клиентская отправка.

## Этап 7. Почему нужен Claude CLI как ревьюер

Проблема прошлых итераций: Codex успешно проверял форму, но иногда пропускал смысловые ошибки:

- «35 рублей за год»;
- смешение чисел занятий и цены;
- машинный текст в client_safe;
- слабые сроки актуальности;
- риск бренд-смешения.

Вывод: нужна постоянная независимая роль смыслового ревьюера.

Решение:

- `CLAUDE.md` хранит текущую правду для Claude CLI;
- `.claude/agents/kb-reviewer.md` задаёт роль независимого ревьюера;
- `.claude/commands/kb-review.md` запускает read-only аудит по явному пути;
- Codex после сборки может вызвать Claude CLI и получить внешний semantic review без ручного копирования папок.

## Этап 8. Первый тест Claude CLI review (2026-05-18)

Codex установил проектный `CLAUDE.md`, `PROJECT_HISTORY.md`, `.claude/agents/kb-reviewer.md` и `.claude/commands/kb-review.md`.

Проверка прошла в два шага:

1. Быстрый тест контекста: Claude CLI правильно назвал главное правило проекта — не смешивать Фотон и УНПК.
2. Полный read-only аудит bot pack через `/kb-review product_data/knowledge_base/kb_release_20260518_v3_2_bot_pack`.

Результат: `PASS_WITH_NOTES`.

Что подтвердилось:

- новый способ вызова работает;
- Claude CLI читает проектный контекст;
- отчёт сохраняется в `audits/_inbox/`;
- bot pack годится для внутреннего режима черновиков.

Что найдено:

- часть `client_safe_text` выглядит машинно и не должна уходить в ответ дословно;
- нескольким скидкам не хватает условия прямо в строке;
- `valid_until` нужно проставить для чувствительных к дате фактов до клиентского пилота;
- фильтры нужно проверить на реальные совпадения;
- runtime-бот должен читать `manifest.safety.client_send=false`.

## Ключевые решения

| Решение | Причина |
|---|---|
| Два отдельных Telegram-бота | AMO/Tallanto не разделены по брендам, а клиентский ответ должен быть brand-safe. |
| Бот сначала только пишет черновики | Нужно быстро дать пользу менеджеру и не рисковать автоответами. |
| `quality_passed=true` не равно готово | Формальные тесты не ловят смысловые ошибки. |
| Каждый смысловой баг становится тестом или правилом | Иначе один и тот же класс ошибок возвращается. |
| Промокоды убраны из клиентского слоя | Нет надёжных сроков действия, высокий риск ошибочного обещания. |
| Скидки только с условиями | Клиент не должен понять скидку шире, чем она реально действует. |
| Customer timeline не удалять | Это будущий слой истории клиента, но включать только после проверки. |

## Что осталось до первого сильного внутреннего запуска

1. Проверить bot pack через Claude CLI новым способом.
2. Прогнать Telegram-пилот на сотруднике в режиме черновиков.
3. Дать Насте 30-60 минут живой проверки черновиков.
4. Уточнить сроки действия для чувствительных фактов.
5. Разобрать семейные телефоны и дубли Tallanto.
6. Постепенно подключать реальные входящие вопросы.
