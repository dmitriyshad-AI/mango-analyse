**Статус:** ФИНАЛ v2 (аудит второго архитектора, 2 круга; добавлено повышение автономии Гип.2 по решению Дмитрия).

**Основание:** регрейд Волны 4 (`REGRADE_wave4_2026-06-09.md`) + карта корня Кодекса, сверенная по сырью Claude.

## Зачем (корень, доказан по сырью)
Волны 3/4 чинили ВЫХОД — мимо. Корень over-handoff в ГЕНЕРАЦИИ direct path, три механизма (карта Кодекса, сверено):
1. Промт разрешает route `draft_for_manager` без правила «если есть exact-факт — сначала ответь сам»; `_normalize_direct_path_payload` принимает route как есть.
2. Промт+gold НОРМАЛИЗУЮТ promise менеджера («менеджер свяжется» без срока; gold-образцы «передам менеджеру, он пришлёт документы»), подаются как «Хороший стиль».
3. Direct path не повышает draft→self (gate «never promotes»).
Плюс находка №8: `_direct_path_selected_categories` режет факты до 2 категорий — `sm_f_enroll_steps` T1 слово «оплатила» даёт `pricing` вместо `enrollment` → модель не видит факт → уходит; T2 (enrollment) — отвечает.
Доказательства по сырью: `sm_u_price_12` T2 «передам менеджеру… вернётся» при данных фактах; T4 бот сам: «Вы правы, надо было ответить сразу» + отвечает. `sm_f_enroll_steps` T1 уход / T2 ответ.

## Стратегия: бьём по всем трём механизмам + повышаем автономию
- **П1 анти-promise** (промт) → механизм 1+2.
- **П2 gold-фильтр** (few-shot) → механизм 2.
- **П3 категории** → находка №8 (вход).
- **П4 route-finalizer draft→self** → механизм 3 (повышение автономии, по решению Дмитрия).
П1-П3 делают текст пригодным для повышения (факт-грунтованный, без promise); П4 повышает. Синергия: без П1 finalizer почти не сработает (promise блокирует повышение).

## Почему повышать автономию сейчас БЕЗОПАСНО
В ЧЕРНОВИКОВОМ пилоте `route=self` НЕ означает автоотправку клиенту: транспорт (Wappi+AMO) кладёт черновик ЛЮБОГО маршрута в карточку, менеджер — последний рубеж, одобряет всё. Повышение draft→self здесь лишь (а) помечает «бот считает ответ самодостаточным» и (б) мерит готовность к ночному автоному — БЕЗ риска клиенту. Реальный риск автоответа появится только при включении автоотправки (после пилота) — и там его держат жёсткие условия повышения + повторное вето (П4.7).

## Правки (за флагом `TELEGRAM_ANTI_PROMISE`, default OFF; в pilot_gold включить ПОСЛЕ регрейда)
Подфлаги: `_PROMPT` (П1+П2), `_CATEGORIES` (П3), `_PROMOTE` (П4). Общий включает все.

**П1. Анти-promise в direct-промте (`_PROMPT`).** Якорь: миссия/инструкция (~2020), схема route (~2828).
Правило: «Есть ПОДТВЕРЖДЁННЫЙ факт под вопрос — ответь им сам, НЕ обещай "передам менеджеру/менеджер свяжется/вернётся" как шаг. Обещание действия менеджера — ТОЛЬКО для live-действий (договор, бронь, проверка/возврат оплаты, документы) и P0/high-risk. Нет факта — честно "это уточню" БЕЗ выдумывания процесса (не сочиняй порядок записи/тест/кабинет, если их нет в фактах).»
Двойная граница: анти-promise ≠ «всегда отвечай». Нет факта → честный короткий уход без promise-действия; есть факт → ответь. Live-действия → менеджер.

**П2. Gold-фильтр few-shot (`_PROMPT`).** Якорь: gold `real_manager_gold_2026-06-08.yaml` ~31/~73; вставка «Хороший стиль» (~2767).
Исходный YAML НЕ править. За флагом фильтровать few-shot: убрать promise-заглушки-как-«хороший-стиль» (где факт был, а образец уходит обещанием); live-действие-образцы (договор/документы по реальному действию) оставить.

**П3. Узкое расширение категорий (`_CATEGORIES`).** Якорь: `_direct_path_selected_categories` (~2255), `categories[:2]`.
- Добавлять `enrollment` при маркерах: «оплатил(а)/после оплаты/что дальше/как попасть/тест/записаться/анкета/доступ».
- Составной вопрос (2+ темы) — не выкидывать вторую тему; поднять лимит для составных.
- Границы: только active_brand + client_safe + exact/adjacent (wrong-scope ловит Волна 1).

**П4. Route-finalizer draft→self (`_PROMOTE`) — повышение автономии.** Якорь: `_normalize_direct_path_payload` (~4240), `_direct_path_finalize_metadata` (~3014), после verifier+gate (~3279).
Повышать `draft_for_manager` → `bot_answer_self_for_pilot` ТОЛЬКО при ВЫПОЛНЕНИИ ВСЕХ условий:
1. route == `draft_for_manager` (НИКОГДА не повышать `manager_only` — это P0/жёсткое вето).
2. gate == pass (нет НИ hard, НИ downgrade-находок; любой downgrade_keep_text/finding → не повышать).
3. Есть **exact** client-safe факт активного бренда под под-вопрос хода (не adjacent).
4. Текст черновика реально ПРОИЗНОСИТ этот факт (грунтован содержанием, не пустой/общий).
5. В тексте НЕТ promise-фразы действия менеджера («передам/менеджер свяжется/вернётся/проверит/он ответит») — иначе self стал бы менеджерским обещанием.
6. НЕ P0 (латч пуст), НЕ high-risk, бренд чист.
7. **Повторное вето:** после повышения результат СНОВА через жёсткий детерминированный гейт (бренд/P0/новое число/ПДн). Всплыло жёсткое → откат в draft (НЕ self). Безопасность не ослабляется.
При сомнении → НЕ повышать (оставить draft). Повышение — консервативное, узкое.

## Граница безопасности (НЕ трогать)
manager_only/P0-латч/преблок, бренд-токены, выдумка числа (absent), ПДн/мета — жёсткое вето как есть. Анти-promise НЕ превращает «нет факта» в выдумку процесса. Повышение — только из draft, никогда из manager_only, и только грунтованного-чистого-без-promise.

## NEG (обязательны, адверсариальные)
1. **OFF-паритет:** все флаги OFF → direct не тронут (полный pytest = main).
2. **П1 при факте:** ход с exact-фактом, где BASE писал «передам менеджеру» → ответ из факта БЕЗ promise.
3. **П1 live-действие:** договор/бронь/проверка оплаты → promise менеджера ОСТАЁТСЯ.
4. **П1 нет факта:** процесс записи без факта → НЕ выдумывает, честный короткий уход без promise-действия.
5. **П3 категории:** `sm_f_enroll_steps` T1 («оплатила…») → enrollment-факты в промт; составной → обе темы; wrong-scope не растёт.
6. **П4 повышение по делу:** draft + gate pass + exact факт произнесён + нет promise + не P0 → route стал self.
7. **П4 НЕ повышает (5 негативов):** (a) pure-handoff без факта в тексте → остаётся draft; (b) текст с promise-фразой → draft; (c) manager_only → НИКОГДА не self; (d) P0/латч → draft/manager; (e) gate с любой находкой → draft.
8. **П4 re-veto:** в повышенном self всплыло новое число/чужой бренд/ПДн → откат в draft, поймано.
9. **Безопасность сквозная:** P0/бренд/число/ПДн — 0 регресса при ON (регрейд по сырью).

## Замер и метрика (М1, BASE=Волна1 vs Волна1+anti_promise, parallel 4, judge v9)
- **(целевое) `bot_answer_self` ВВЕРХ** — рост автономии (теперь целевая, т.к. повышаем).
- **(целевое) доля promise-заглушек при наличии exact-факта ВНИЗ** — детерминированный греп фраз в ходах с exact client-safe фактом.
- полезность черновиков/ответов по сырью (отвечает фактом, где раньше уходил).
- **ЖЁСТКО 0 (гейт допуска):** P0-провалы / смешение брендов / выдумка числа / ПДн / promise в повышенном self — регрейд по сырью + NEG. Любой регресс безопасности → BLOCKED, рост автономии его не оправдывает.
Регрейд по сырью обязателен: (а) реально ли повышенные self корректны и грунтованы; (б) не выдумывает ли процесс вместо ухода; (в) повторное вето сработало.

## Вне scope
Реальная автоотправка клиенту (после пилота); правка исходных YAML; потолок автономии (выдумка процесса — отдельно); source_id-санитайзер (в пакет presale-фиксов).

## Аудит второго архитектора (внесён)
- **Р1 «повышение draft→self = автоответ клиенту с риском выдумки»:** в черновиковом пилоте self ≠ автоотправка (менеджер одобряет любой route); повышение мерит автономию без риска; реальный риск — при автоотправке после пилота, там держат 7 условий + re-veto. Принято.
- **Р1 «условия повышения достаточны?»:** 7 жёстких (только draft; gate pass; exact факт; текст произносит факт; нет promise; не P0; re-veto). При сомнении не повышать. NEG-7/8. Принято.
- **Р2 «pure-handoff/promise повысится в self?»:** нет — усл.4 (нет факта в тексте) + усл.5 (есть promise) блокируют; NEG-7a/7b. Принято.
- **Р2 «синергия П1↔П4»:** П1 убирает promise → текст пригоден → П4 повышает; без П1 повышений почти нет (promise блокирует). Поэтому вместе. Принято.
- **Р2 «анти-promise спровоцирует выдумку процесса?»:** двойная граница — нет факта → честный уход без выдумки. NEG-4. Принято.
- **Р3 «категории шум/wrong-scope»:** узко по маркерам + границы; wrong-scope ловит Волна 1. NEG-5. Принято.
- **Р3 «метрика»:** self-rate вверх теперь целевая, НО гейтится 0-регрессом безопасности по сырью; promise-при-факте вниз. Принято.
- **Р3 «не трогать gold YAML»:** фильтр few-shot за флагом. Принято.
