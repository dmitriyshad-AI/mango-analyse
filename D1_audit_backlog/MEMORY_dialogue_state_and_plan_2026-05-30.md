# Память диалога: состояние и пошаговый план «не терять нить». 2026-05-30.

Автор: Клод 1 (read-only разбор кода ревизии 20956e8e, выводы проверены лично, субагент скорректирован).

## Короткий вывод (TL;DR)

Дело НЕ в том, что «памяти нет». Память есть, и она богатая. Бот теряет нить по двум другим причинам:

1. **Память наполняется эвристиками (regex), а не пониманием.** Слоты, тема, открытый вопрос и
   «сводка» извлекаются регулярками из текста. Явное («10 класс», «онлайн») ловится, неявное /
   перефразированное / отсылки к прошлой реплике — теряются.
2. **Детерминированный слой память игнорирует.** Розыск фактов, маршрут и проверка верности работают
   по ТЕКСТУ текущего хода. Память видит только модель, пишущая черновик; но её черновик потом режет/
   перетирает слой, который темы предыдущих ходов не знает.

Итог: на уточняющих ходах с эллипсисом («а для 10?», «а это онлайн?», «а сколько стоит?») слой теряет
тему предыдущего хода → ищет факты не по той теме → либо уходит к менеджеру, либо подставляет факт из
соседней темы (= выдумка). Это та же механика, что даёт выдумки «информатика 10 → олимпиада 9/11».

То есть это НЕ «добавить память как у ChatGPT», а «(а) наполнять память пониманием и (б) заставить
логический слой её читать».

## Где именно теряется нить

1. **Эллипсис не разрешается перед розыском фактов.** «а для 10?» без «информатика/онлайн» из прошлого
   хода → факты ищутся по голому «10 класс» → не та тема. ГЛАВНАЯ причина.
2. **Слой решает по текущему ходу.** Маршрут и критик не знают, что тема уже установлена ранее → критик
   может счесть «10 класс» сменой scope, маршрут — уйти к менеджеру по уже отвеченному.
3. **Наполнение эвристическое.** Перефразировки и неявные отсылки в слоты/тему не попадают → память
   неполная именно там, где клиент говорит «по-человечески».
4. **Сводка — склейка слотов, не смысл.** Длинную нить (5+ ходов) держать нечем, кроме сырых
   recent_turns, которые модель должна склеивать сама без опоры.

## Пошаговый план (от дешёвого к архитектурному)

Каждый шаг — отдельная правка с тестом входа/выхода; не катить пачкой (правило «не менять две вещи»).

**Шаг 1. Разрешение эллипсиса перед розыском фактов (ядро, наибольший эффект, средняя цена).**
Перед `retrieve_facts` дополнять текущий вопрос темой из памяти (topic_focus + known_slots: предмет,
класс, формат, продукт), если текущий ход — уточнение/эллипсис. «а для 10?» → «информатика онлайн
10 класс». Чинит и потерю нити, и значимую долю выдумок (факт ищется по верной теме).
Критерий: многоходовой тест «предмет на ходе1 → класс на ходе2» находит факт верной темы; NEG —
явная смена темы клиентом не склеивается со старой.

**Шаг 2. Дать слою читать память (маршрут + критик).**
Маршрут учитывает route_history/answered_questions (не уходить повторно к менеджеру по уже безопасно
отвеченному); критик знает установленную тему (не метит «10 класс» как wrong_scope, если тема та же).
Критерий: повторный уточняющий ход по той же теме остаётся в bot_answer_self; NEG — реальная смена на
P0 по-прежнему уходит к менеджеру.

**Шаг 3. Модельное наполнение памяти (переход к «пониманию», главный архитектурный шаг).**
Заменить/дополнить regex-извлечение слотов/темы/открытого вопроса одним лёгким модельным шагом
«обнови рабочую память по последнему ходу» (вход: recent_turns + прошлая память; выход: слоты, тема,
открытый вопрос, что обещано). Это и есть формула «модель ПОНИМАЕТ, детерминированный слой
ВЕРИФИЦИРУЕТ» из [[bot_architecture_root_flaw]]. Критерий: на наборе с перефразировками слоты/тема
извлекаются там, где regex промахивался.

ТРЕБОВАНИЯ К СТОИМОСТИ ВЫЗОВА (решение Дмитрия 30.05) — обязательны, иначе шаг дорог в бою:
- low reasoning effort: наполнение памяти = извлечение/сжатие, не рассуждение; глубокий reasoning жжёт
  токены/время впустую;
- по возможности ОТДЕЛЬНАЯ мелкая/быстрая модель (haiku-класс) на этот вспомогательный шаг, основная
  модель — только на сам ответ клиенту (чистое разделение: дешёвая ведёт конспект, дорогая отвечает);
- наполнять память ПОСЛЕ отправки ответа (в точке `update_dialogue_memory_after_answer`, она уже
  вызывается после ответа), готовя память для СЛЕДУЮЩЕГО хода в фоне — тогда латентность текущего
  ответа клиенту НЕ растёт, остаётся только фоновый токен-расход, который и срезают low effort + мелкая
  модель. Это убирает главную цену шага 3 с критического пути.

**Шаг 4. Настоящая короткая сводка диалога.**
`conversation_summary_short` — заменить склейку слотов на сжатую смысловую сводку (1-2 фразы: кто
клиент, что обсуждали, на чём остановились). Можно тем же модельным шагом, что в шаге 3. Держит длинную
нить без раздувания промпта. Критерий: на диалоге 6+ ходов сводка отражает суть без потери ключевого
факта/обещания.

**Шаг 5. Лимиты истории (низкий приоритет).**
MAX_TURNS=20 / MAX_PROMPT_TURNS поднять только если на длинных диалогах подтвердится обрезание. Наши
сценарии короткие — не корень, делать в последнюю очередь.

**Шаг 6. Измерение «удержания нити».**
Отдельный набор многоходовых сценариев: эллипсис, возврат к теме через ход, уточнение класса/формата,
«а ещё». Метрика: доля ходов, где бот ответил в верной теме, не переспросил уже известное, не ушёл
зря к менеджеру. Без этого набора шаги 1-4 не измеримы.

## Связь с текущими правками

Правка 5.1 (4 типа выдумок в критике) латает СИМПТОМ в промпте судьи — она не даёт выдумке про
формат/расписание пройти, но не восстанавливает потерянную тему. Шаги 1-2 чинят КОРЕНЬ (тема не
теряется → факт верный → критику нечего резать). После шагов 1-2 часть нагрузки с 5.1 снимется
естественно. Поэтому память — логичное продолжение, а не параллельная ветка.

## Что НЕ делать

- НЕ выбрасывать существующую `DialogueMemory` — структура богатая и правильная, проблема в наполнении
  и в том, что слой её не читает. Достраиваем, не переписываем.
- НЕ начинать с лимитов/рефакторинга — это не корень.

## Подход к реализации (пересмотрено 30.05 по аргументу Дмитрия)

Прежняя идея «волна 1 → замер на M1 → волна 2» НЕ работает: между шагами памяти НЕ будет отдельного
M1-прогона (ритм — один прогон на батч, прогоны дорогие). Вся цепочка памяти пойдёт в ОДИН следующий
M1-прогон. Плюс шаги взаимозависимы (особенно 1b↔3: достройка вопроса должна опираться на тему, а её
качество задаёт модельное наполнение шага 3). Поэтому правильнее ОДНО целостное ТЗ на шаги 1-4, а не
куски, которые могут не состыковаться.

Чтобы единое ТЗ не превратилось в неразбираемый куст, реализация структурируется так:
- ТЗ описывает шаги 1a → 1b → 2 → 3 → 4 как ОТДЕЛЬНЫЕ коммиты в одной ветке, по порядку;
- после КАЖДОГО коммита Клод 1 гоняет юнит-тесты в песочнике (бесплатно) — ловит регрессию сразу,
  локализуя её до конкретного шага, без M1;
- ОДИН M1-прогон в конце меряет всю цепочку + набор удержания нити (шаг 6);
- если суммарный результат плох — регрейд по сырью + откат конкретного коммита-шага (коммиты по шагам
  это позволяют), а не всей цепочки.

То есть целостный дизайн (единое ТЗ) + пошаговая проверяемость (коммиты + юнит-щит) + один замер (M1).
Шаг 5 (лимиты) — вне этой цепочки, по необходимости. Шаг 6 (набор) — готов.

Очерёдность внутри ТЗ: 1a (память в промпт) → 1b (эллипсис перед розыском фактов) → 2 (слой читает
память) → 3 (модельное наполнение, low effort/мелкая модель/после ответа) → 4 (смысловая сводка тем же
вызовом). Не требует трогать 8000-строчный модуль структурно.
