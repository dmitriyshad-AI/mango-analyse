# ТЗ: ночной блок базы знаний, фактов и лучших ответов менеджеров для Telegram-пилота

Дата: 2026-05-17
Проект: Mango Analyse
Цель: сделать ответы Telegram-пилота заметно содержательнее, не снижая безопасность.

## 1. Контекст

Сейчас Telegram-пилот уже умеет:

- строить контекстный prompt для LLM;
- выбирать тему только из закрытой таксономии 32 тем + 5 служебных категорий;
- переводить опасные темы в `manager_only`;
- блокировать выдуманные темы;
- блокировать неподтвержденные числовые обещания в `draft_text`;
- работать через `codex exec` без OpenAI API-ключей;
- сохранять черновики для менеджера, а не отправлять клиентам.

Главная текущая проблема: ответы часто слишком нейтральные. Модель пишет в стиле “уточним”, “проверим”, “вернемся позже”, потому что у нее мало проверенных фактов и хороших примеров ответов.

В проекте уже есть начальный каркас T5:

- `src/mango_mvp/knowledge_base/fact_registry.py`;
- `src/mango_mvp/knowledge_base/kc_context.py`;
- `tests/test_kc_knowledge_snapshot.py`;
- `tests/test_kc_context.py`;
- audit pack `audits/_inbox/kc_knowledge_facts_pilot_20260516_161003/`.

Этот каркас умеет:

- регистрировать локальный `База знаний КЦ.docx`;
- регистрировать два Google Drive документа по ценам как metadata-only;
- извлекать разделы из DOCX;
- выбирать короткие фрагменты для prompt;
- блокировать точные ответы, если факт не свежий.

Но пока не сделано:

- полноценная обработка всех доступных документов Google Drive;
- извлечение проверяемых фактов из документов;
- нормальный snapshot базы знаний в `product_data/knowledge_base/...`;
- отдельная база хороших приемов из реальных ответов менеджеров;
- подключение этой базы к Stage 6 Telegram-прогону;
- сравнение “до/после” по содержательности ответов.

Дополнительная сверка текущего кода показывает:

- старые извлечения сайтов и документов в `.codex_local/kc_source_extract_20260513/` можно использовать как read-only вспомогательный материал, но не как утвержденный источник истины.

## 2. Главная идея

Нужно разделить два слоя:

1. **База фактов**
   Это источник истины: цены, правила, документы, адреса, форматы, условия оплаты, маткапитал, налоговый вычет, возвраты, ограничения. Эти данные можно использовать в ответах только если у факта есть источник, дата и статус свежести.

2. **База хороших ответов менеджеров**
   Это не источник истины, а источник стиля и подходов: как менеджеры объясняли цену, просили данные, снимали сомнения, переводили клиента к следующему шагу. Эти ответы нельзя автоматически считать правильными фактами, потому что в них могут быть старые цены, ошибки, персональные данные и устаревшие обещания.

Итоговая задача ночного блока: построить первый рабочий слой, где бот получает 3-8 коротких релевантных фрагментов: проверенные факты + правила РОПа + безопасные примеры хороших формулировок.

Ключевое правило: “факт найден в документе” не равно “бот может сказать это клиенту”. Право на точный ответ появляется только если факт прочитан, свежий, не конфликтует с другими источниками и разрешен для такого использования.

## 3. Жесткие границы

Нельзя:

- отправлять сообщения клиентам;
- писать в AMO/CRM;
- писать в Tallanto;
- менять `stable_runtime` DB/audio/transcripts;
- запускать ASR;
- запускать Resolve+Analyze;
- запускать тяжелые batch/start/run-ui скрипты;
- удалять файлы;
- делать `git reset`, `git checkout`, `git clean`;
- считать факт свежим только потому, что он найден в старом документе или ответе менеджера;
- использовать реальные ответы менеджеров как клиентский ответ без очистки и проверки;
- отправлять в prompt весь DOCX, весь Google Drive файл или всю историю клиента целиком.

Можно:

- читать локальные документы;
- читать `product_data/question_catalog`;
- читать существующие отчеты и audit packs;
- читать Google Drive, если доступен безопасный read-only способ;
- писать новые производные артефакты в `product_data/knowledge_base/...`;
- писать документы и audit pack;
- запускать точечные безопасные тесты;
- запускать локальные LLM-прогоны через `codex exec` по подписке, без API-ключей.

- `stale_or_conflicting` - найдено, но устарело или конфликтует;
- `internal_only` - только внутренняя инструкция;
- `do_not_use` - нельзя использовать.

Правило допуска к точному ответу:

- `usable_for_precise_answer=true` допустим только если одновременно выполнено:
  - источник прочитан;
  - есть дата или понятный период действия;
  - факт актуален для 2026/2027 учебного года или другого явно указанного текущего периода;
  - нет конфликта с другим источником;
  - факт не является внутренней инструкцией;
  - факт разрешен РОПом или ответственным сотрудником;
  - в `bot_permission` стоит не ниже `allowed_after_fact_check`.

Если хотя бы одно условие не выполнено, точный ответ запрещен. В prompt можно передавать только мягкую формулировку для менеджера: “нужно проверить актуальные условия”.

## 6. Критерии качества

### 6.1. Источники

Минимум:

- все локальные источники из ТЗ зарегистрированы;
- `База знаний КЦ.docx` прочитан;
- Google Drive документы либо прочитаны, либо честно отмечены как недоступные/metadata-only;
- два документа по ценам зарегистрированы отдельно;
- у каждого источника есть статус.

Хорошо:

- прочитаны все доступные документы Google Drive;
- по каждому документу есть краткий конспект;
- цены и правила оплаты выделены отдельно;
- противоречия и устаревшие места вынесены в отдельный отчет.

### 6.2. Факты

Минимум:

- есть факты по 5 главным слоям:
  - РОП-правила;
  - цены;
  - FAQ/типовые вопросы;
  - запреты и безопасные формулировки;
  - источники и свежесть;
- точные цены не используются без свежего подтверждения;
- расписание по-прежнему не считается готовым точным фактом.

Хорошо:

- есть факты по 10+ категориям;
- есть минимум 100-200 полезных фактов/фрагментов;
- каждый факт связан с темами таксономии;
- есть список пробелов, которые нужно спросить у РОПа/менеджеров.

### 6.3. Ответы менеджеров

Минимум:

- выбрано 300 строк;
- для каждой строки есть тема, канал, вопрос, исторический ответ или причина отсутствия ответа;
- выделены хорошие приемы;
- небезопасные/устаревшие ответы не попали в client-safe слой.

Хорошо:

- выбрано 500 строк;
- есть 50-100 хороших паттернов ответов;
- есть 20-50 примеров “как не надо отвечать”;
- каждый хороший паттерн связан с темой и риском.

### 6.4. Prompt и черновики

Минимум:

- prompt получает 3-8 коротких фрагментов;
- весь DOCX/Google Drive документ не отправляется в prompt;
- неподтвержденные цифры блокируются;
- high-risk темы остаются под менеджером;
- Stage 6 проходит без ошибок.

Хорошо:

- в Stage 6 невалидных тем: 0;
- неподтвержденных числовых обещаний: 0;
- минимум 50% черновиков стали содержательнее по сравнению с baseline;
- доля “только уточним и вернемся” заметно снизилась;
- опасные темы не стали менее осторожными.

Дополнительный порог перед расширением после Stage 6:

- `context_found_rate` для тестовой выборки не ниже 75%;
- не менее 80% выбранных фрагментов имеют источник и статус свежести;
- не менее 90% фрагментов укладываются в лимит длины без обрезки смысла;
- 0 случаев, где менеджерский исторический ответ попал в клиентский слой как факт.

## 7. Критерии остановки

Остановиться и не продолжать автоматическую обработку, если:

- нужен live-write в AMO/CRM/Tallanto;
- нужен ASR или Resolve+Analyze;
- нужно менять `stable_runtime`;
- Google Drive требует действия, которое невозможно выполнить read-only;
- источник содержит слишком много персональных данных и нет безопасного способа очистки;
- найдено противоречие в ценах, которое невозможно автоматически разрешить;
- LLM-прогон начинает возвращать невалидный JSON массово;
- Stage 6 показывает новые опасные обещания;
- тесты по safety падают.

Не останавливать весь блок, если:

- часть Google Drive файлов не читается - отметить статус и идти дальше;
- часть фактов не свежая - пометить как `needs_manager_confirmation`;
- часть ответов менеджеров плохая - отправить в `unsafe_or_outdated_manager_answers.csv`;
- выборка 500 не успевает - допустимый минимум 300.

## 8. Использование субагентов

Для реализации ночного блока использовать до 6 субагентов с уровнем рассуждения `xhigh`.

Распределение:

1. **Субагент 1: код и текущий T5**
   Проверяет существующие `fact_registry.py`, `kc_context.py`, тесты и предлагает минимальные правки без слома.

2. **Субагент 2: Google Drive и документы**
   Делает read-only инвентаризацию источников, статусы чтения, список документов, проблемы доступа.

3. **Субагент 3: извлечение фактов**
   Строит факты, статусы свежести, категории, конфликты, пробелы.

4. **Субагент 4: реальные ответы менеджеров**
   Делает выборку 300-500 вопросов, анализирует ответы, выделяет хорошие приемы и опасные ответы.

5. **Субагент 5: подключение к Telegram-prompt и Stage 6**
   Проверяет, что `facts_context` и `knowledge_snippets` реально попадают в prompt, запускает безопасный eval.

6. **Субагент 6: QA и audit pack**
   Проверяет тесты, safety, артефакты, готовит список рисков и утренний чеклист.

Писать в одни и те же файлы параллельно нельзя. Если нужны кодовые изменения, разделить write-set заранее.

## 12. Метрики, которые нужно посчитать

Источники:

- сколько источников найдено;
- сколько прочитано;
- сколько не прочитано;
- сколько metadata-only;
- сколько источников со свежими фактами.

Факты:

- всего фактов;
- фактов по категориям;
- fresh verified;
- needs manager confirmation;
- stale/conflicting;
- internal only;
- do not use.

Фрагменты:

- всего chunks;
- средняя длина;
- сколько пригодны для prompt;
- сколько содержат price/schedule/documents/restriction.

Ответы менеджеров:

- размер выборки;
- покрытие по каналам;
- покрытие по темам;
- сколько хороших приемов;
- сколько опасных/устаревших ответов;
- сколько примеров можно показать РОПу.

Stage 6:

- ошибок LLM;
- невалидных тем;
- `manager_only`;
- `draft_for_manager`;
- неподтвержденных числовых обещаний;
- сколько черновиков использовали базу знаний;
- сколько черновиков стали содержательнее;
- сколько требуют ручной проверки Дмитрия.

## 13. Утренняя проверка Дмитрия

Утром Дмитрию нужно смотреть не весь массив, а 4 файла:

1. `quality_summary.md`
   Общее состояние: что прочитано, что не прочитано, какие факты можно использовать.

2. `conflicts_and_gaps.md`
   Что нужно уточнить у РОПа/Насти/сотрудников.

3. `manager_answer_playbook.md`
   Какие хорошие приемы менеджеров выделены.

4. `stage6_kb_enriched_drafts.xlsx`
   Стали ли ответы лучше на реальных Telegram-диалогах.

Решение после проверки:

- если ответы стали содержательнее и safety не ухудшился - подключать слой к первому рабочему пилоту Насти;
- если фактов мало - донаполнить Google Drive/FAQ;
- если много противоречий - сначала вынести список РОПу;
- если ответы стали опаснее - откатить подключение базы знаний к prompt, но оставить snapshot для ручной работы.

## 14. Что считать успехом ночного блока

Минимальный успех:

- есть рабочий snapshot;
- есть источник-файл и свежесть у каждого факта;
- есть 300 строк анализа реальных ответов;
- есть 30+ хороших приемов менеджеров;
- Stage 6 повторен;
- safety не ухудшился;
- есть audit pack.

Хороший успех:

- обработаны все доступные Google Drive документы;
- есть 100+ полезных фактов;
- есть 50-100 хороших приемов менеджеров;
- Stage 6 показывает заметно более содержательные черновики;
- нет новых опасных обещаний;
- Дмитрий может утром открыть один Excel и быстро принять решение.

Недопустимый результат:

- факты без источников;
- точные цены без свежести;
- копирование сырых ответов менеджеров как истины;
- отправка всего документа в prompt;
- live-записи во внешние системы;
- изменения в `stable_runtime`;
- отсутствие audit pack.
