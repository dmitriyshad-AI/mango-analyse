# ТЗ: единое обновление базы знаний бота v6.1 по ответам команды

Дата: 2026-05-20

Статус: рабочее ТЗ для будущей реализации. Прогоны MEGA, `codex exec`, малый живой smoke и пересборку релиза не запускать до отдельного подтверждения Дмитрия, потому что ожидаются новые правки базы знаний.

## 1. Назначение

Нужно объединить две пачки правок от Claude по ответам команды в одно чистое ТЗ:

- `TZ_FOR_CODEX_kb_v5_team_answers_2026-05-20.md`
- `TZ_FOR_CODEX_kb_v6_team_part2_2026-05-20.md`

Это ТЗ заменяет оба документа как рабочий план для Codex. Из исходных документов берём факты и маршруты, но исправляем смысловые противоречия:

- не возвращаем фразу `Это автоматический ответ`;
- уточняем бренд-разделение для общих физических площадок;
- разделяем пробное занятие на онлайн и очное;
- разделяем сбор данных при записи и запрет сбора данных при возврате/жалобе/юридической угрозе;
- не используем сомнительные маркетинговые факты вроде `100 000 учеников`, если они не подтверждены отдельно;
- не запускаем тестовые прогоны до финального закрытия базы.

Цель v6.1: получить более точную, бизнес-безопасную и пригодную для Telegram-пилота базу знаний, где бот отвечает по подтверждённым фактам, не смешивает Фотон и УНПК, не даёт устаревшие условия и корректно передаёт рискованные темы менеджеру.

## 5. Исправления фактов Фотона

### 5.1. VK Фотона

Заменить старый VK:

- было: `vk.com/kmipt_edu`
- стало: `vk.ru/foton_edu`

Правила:

- бот Фотона может использовать `vk.ru/foton_edu`, если клиент спрашивает VK Фотона;
- бот УНПК не должен упоминать `vk.ru/foton_edu` и `foton_edu`;
- старый `vk.com/kmipt_edu` не должен попадать в клиентские ответы Фотона;
- если старый VK нужен для истории, держать только как internal/archived, не client-safe.

### 5.2. ЛВШ Менделеево Фотон

Добавить подтверждённые факты:

- питание: `Шведский стол, ресторанный уровень`;
- охрана: `24/7`;
- преподаватели: `Преподаватели из МГУ и МФТИ, вожатые из топовых вузов Москвы`.

Клиентский ответ не должен называть конкретные ФИО преподавателей, если клиент не спрашивает про конкретную группу/ситуацию, где это разрешено.

### 5.3. ЛШ городская Фотон: Предлёнка

Добавить:

- название: `Утренний клуб Предлёнка`;
- время: `09:45-11:45`;
- старт: со второго дня занятий;
- стоимость: полностью бесплатно.

Клиентская формулировка: `Во время городской летней школы есть бесплатный утренний клуб Предлёнка: со второго дня занятий, с 09:45 до 11:45.`

### 5.4. Старт учебного года Фотон 2026/27

Для Фотона использовать старт по формату:

- Москва: `12-13 сентября 2026`;
- онлайн: `19-20 сентября 2026`.

Старая общая фраза `13-14 сентября` не должна попадать в клиентский ответ.

### 5.5. Будущие цены после повышения

Бот не должен клиенту называть будущую цену после повышения, если клиент спрашивает “сколько будет после 1 июля / после 1 августа / с сентября”.

Правильное поведение:

- дать текущую актуальную цену, если вопрос про сейчас и факт подтверждён;
- если вопрос именно про будущую цену, передать менеджеру;
- не писать `цена вырастет до ...`, `после повышения будет ...`, `будущая цена ...`.

## 7. Общие правила для обоих брендов

### 7.1. Маткапитал для ребёнка старше 18

Для обоих брендов добавить фразу:

`По возрастным условиям маткапитала есть ограничения. Уточню у менеджера, подходит ли ваш случай — он свяжется в течение рабочего дня.`

Важно: если вопрос про маткапитал для 19-летнего, это не общий сценарий `adult_18_plus`. Приоритет у правила маткапитала, а не у фразы “курсов 18+ нет”.

### 7.2. Диагностика

Диагностика идёт после оплаты и включена в стоимость обучения. Она помогает распределить ученика в подходящую группу.

Запрещено:

- предлагать диагностику до оплаты как способ избежать возврата;
- использовать диагностику как маркетинговый бесплатный тест перед покупкой, если это не подтверждено отдельным фактом.

### 7.3. Запись на обучение

При записи можно собирать:

- класс ребёнка;
- предмет;
- формат: онлайн или очно;
- email родителя;
- ФИО ребёнка;
- телефон родителя.

Но формулировка “всегда просить всё” слишком грубая. Нужно просить только недостающие данные. Если телефон уже есть в Telegram/AMO/Tallanto и он надёжен, не заставлять клиента повторять его.

Правило:

- enrollment: можно собирать телефон и ФИО ребёнка;
- refund / complaint / legal_threat: ZERO collect остаётся, не собирать ФИО, договор, сумму, причину, телефон, email, кроме отдельно разрешённого случая, когда телефона вообще нет и нужен только телефон для обратного звонка по возврату.

### 7.4. Жалобы, возвраты, юридические угрозы

Не возвращать фразу `Это автоматический ответ`.

Для P0-жалоб и юридических угроз использовать нейтральную формулу:

`Ваше обращение зафиксировано, в ближайшее рабочее время с вами свяжется ответственный сотрудник.`

Запрещено:

- `я бот`;
- `это автоматический ответ`;
- `я ИИ`;
- `GPT`;
- `Claude`;
- `Codex`;
- извинения от лица компании: `приносим извинения`, `понимаю, что неприятно`, `нам очень жаль`;
- сбор ФИО, договора, телефона, суммы, причины, если тема refund/complaint/legal_threat.

### 7.5. Болезнь

Онлайн:

- можно сказать, что запись урока сохранится, если это подтверждено для онлайн-формата;
- можно указать, что запись доступна в системе/по ссылке, если есть свежий факт.

Очно:

- перенос, компенсация, отработка только через менеджера;
- справка от ребёнка не нужна для обычных занятий;
- справка 079у относится к лагерю, не к обычному пропуску урока.

### 7.6. B2B

Любые запросы от школ, организаций, классов, групповых закупок и партнёрств:

- route: `manager_only`;
- собираемые данные: название школы/организации, контактное лицо, должность, удобный контакт;
- срок: `в течение рабочего дня`;
- одинаково для Фотона и УНПК.

### 7.7. Комбинированные вопросы

Если в одном сообщении есть безопасная тема и рискованная тема, например:

- `сколько стоит + хочу вернуть деньги`;
- `есть ли скидка + буду жаловаться`;
- `можно оплатить + подам в суд`;

то общий маршрут должен быть `manager_only`. Бот может подготовить менеджеру черновик по безопасной части, но клиенту не должен автоматически отдавать точные цены или условия в этом же ответе.

Клиентская формула:

`По вашему вопросу подключу менеджера, чтобы он проверил детали и ответил корректно.`

### 7.8. Социальные доказательства и маркетинговые числа

Не использовать в клиентских ответах сомнительные или неподтверждённые маркетинговые числа:

- `100 000 учеников`;
- `Лидер 2025`;
- любые “лучшие”, “самые сильные”, “гарантируем результат”, если нет отдельного подтверждённого факта и разрешения.

Если такие факты остаются в базе, они должны быть internal или manager-only до отдельного подтверждения.

## 8. Бренд-разделение

Главное правило остаётся: один ответ говорит только от лица одного активного бренда.

Фотон не объясняет условия УНПК. УНПК не объясняет условия Фотона.

Нельзя сравнивать:

- цены;
- скидки;
- рассрочки;
- документы;
- лицензии;
- адреса;
- лагеря;
- условия оплаты.

Разрешённые общие физические площадки:

```yaml
shared_brand_entities_allowed:
  - entity: "Менделеево"
    contexts: ["lvsh_mendeleevo"]
  - entity: "Верхняя Красносельская ул., 30"
    contexts: ["ls_city_2026_foton", "ls_city_2026_unpk"]
```

Важно: `Верхняя Красносельская` для УНПК разрешена только в контексте городской летней школы, не как общий адрес регулярных курсов.

Для active brand `unpk` blocked terms должны включать Фотон, `vk.ru/foton_edu`, `foton_edu`, Т-Банк, Долями, контакты Фотона.

Для active brand `foton` blocked terms должны включать УНПК, `edu@kmipt.ru`, `kmipt.ru`, контакты УНПК и юрлица УНПК.
