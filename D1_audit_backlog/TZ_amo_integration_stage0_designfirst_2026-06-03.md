## Железный принцип (не нарушать)
- **Бот-мозг = НАШ сервис.** Чистая архитектура и выходной гейт (бренд/P0/выдумка/leak) ОСТАЮТСЯ в нашем коде. В
  amoCRM Salesbot/виджет логику и безопасность НЕ выносим — AMO это канал и интерфейс, не мозг.
- **CRM read-only** на старте (CLAUDE.md): бот НЕ пишет в AMO без отдельного OK Дмитрия. Единственная допускаемая
  запись на Этапе 0 — ВНУТРЕННИЙ черновик-нота в карточке (виден менеджеру, клиенту НЕ уходит). Её отдельно
  подтверждает Дмитрий.
- **Draft-only.** Бот НЕ отправляет клиенту ничего на Этапе 0. Только предлагает; шлёт менеджер.

## МОЙ ПЛАН (для сверки) — лучший способ встроить бота в AMO (Этап 0)
1. **Вход:** вебхук amoCRM на входящее сообщение клиента в сделке → наш сервис (текст + lead_id + pipeline_id +
   contact_id).
2. **active_brand = pipeline_id** (Фотон-сделки в воронке Фотона, УНПК — в своей). Бренд из КАНАЛА, не угадываем —
   это и удобно, и закрывает правило №1 (бренд-разделение) на входе. [штурм: Тир-1 #2]
3. **Контекст read-only:** сервис читает историю чата + поля сделки (класс/предмет/источник/этап) через REST API
   для персонализации — без записи в CRM. [штурм: #5]
4. **Черновик** генерит наш пайплайн (планнер→правила→композер→гейт). Draft-only.
5. **Сёрфинг в карточке:** черновик появляется менеджеру. v1 простой и надёжный — **нота/примечание** «🤖 Черновик:
   …», менеджер правит и шлёт сам. v2 — **виджет** с кнопками Отправить/Править/Менеджеру (Widgets SDK). Начать с
   ноты, апгрейд до виджета. [штурм: #4 + нота-лог #9] — НОТА = внутренняя запись, нужен OK Дмитрия.
6. **Менеджер вычитывает → шлёт сам.** Бот не отправляет.
7. **Мониторинг:** логировать каждый черновик + действие менеджера (отправил как есть / правил / проигнорировал) →
   метрика **unedited_rate** (северная звезда доверия). [штурм: #11]
Позже (Этапы 3-4): исходящие через Chats/amojo API с номера компании + слой автоотправки — НЕ сейчас.
