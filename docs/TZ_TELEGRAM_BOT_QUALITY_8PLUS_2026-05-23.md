# ТЗ: качество Telegram-ботов 8+ по памяти, пользе и человечности

Дата: 2026-05-23

Статус: `draft_ready_for_review`

Назначение: следующий этап доработок Telegram-ботов Фотона и УНПК МФТИ после внедрения journal, DialogueMemory, answer quality rewriter, skill для классов ошибок и первых targeted/focused прогонов.

## 1. Короткий ответ на главный вопрос

Нет, мы ещё не сделали всё, чтобы иметь память "на уровне веб-версии ChatGPT" даже в метафорическом смысле.

Что уже есть:

- структурная память текущей Telegram-сессии;
- известные слоты клиента: класс, предмет, формат, цель, бренд, часть контекста;
- запрет повторно спрашивать уже известное;
- журнал пилота;
- daily report;
- feedback import;
- очереди смысловой проверки;
- skill `bot-failure-class-review`;
- `business-semantic-review` как обязательная дисциплина;
- answer quality rewriter;
- LLM-rewriter под feature flag;
- первые исправления классов ошибок: wrong fact scope, self-pollution, live availability, multi-topic safe-part-first.

Что ещё не дотягивает до уровня устойчивого веб-ассистента:

- нет полноценной долгой памяти клиента между днями и каналами;
- нет устойчивой сжатой истории "что уже обсуждали" между сессиями;
- не все ответы сначала закрывают последний прямой вопрос;
- бот иногда звучит как база знаний или шаблон;
- бот иногда отдаёт менеджеру там, где мог дать больше полезной безопасной конкретики;
- нет чистого holdout-замера после всех правок;
- полный v8 ещё не пройден как финальный контроль;
- в некоторых сценариях ответ всё ещё нужно улучшать вторым проходом.

Итог: фундамент сделан, но следующий этап нужен именно для скачка качества до 8+/10, а не для добавления ещё фактов.

## 2. Цель этапа

Довести Telegram-ботов до минимального целевого качества:

- каждый критерий качества - минимум 8/10 на чистом holdout;
- P0 и brand hard gates - 0 нарушений;
- бот отвечает как внимательный консультант, а не как справочник;
- бот помнит текущий диалог и не теряет контекст на втором-третьем ходе;
- бот прямо отвечает на вопрос, затем делает один следующий шаг;
- бот мягко ведёт нового клиента к подбору, заявке или менеджеру;
- все FAIL/PASS_WITH_NOTES превращаются в класс проблемы, regression или ручной контроль.

## 4. Критерии 8/10

Оценка считается честной только на чистом holdout, который не использовался для настройки.

| Критерий | Цель |
|---|---|
| P0 safety | 10/10, 0 нарушений |
| Brand isolation | 10/10, 0 смешений Фотон/УНПК |
| Fact accuracy | >= 9/10, без неподтверждённых чисел/дат/мест/обещаний |
| Direct answer | >= 8/10, ответ закрывает последний прямой вопрос |
| Dialogue memory | >= 8/10, класс/предмет/формат/контекст не теряются |
| No known-data reask | >= 9/10, бот не спрашивает уже известное |
| Multi-topic handling | >= 8/10, закрыты все безопасные части составного вопроса |
| Human tone | >= 8/10, тепло, естественно, без канцелярита |
| Sales proactivity | >= 8/10, есть мягкий следующий шаг к заявке/подбору |
| Over-handoff control | >= 8/10, менеджер только там, где реально нужно действие/проверка |
| No unstated assumptions | >= 8/10, бот не приписывает клиенту цель или предмет |
| Next step clarity | >= 8/10, клиент понимает, что делать дальше |

## 5. Не делать в этом этапе

Нельзя:

- писать в AMO/CRM/Tallanto;
- менять `stable_runtime`;
- запускать ASR;
- запускать Resolve+Analyze;
- отправлять сообщения клиентам вне текущего уже запущенного пилота;
- запускать полный v8 первым шагом;
- считать targeted16 финальной проверкой;
- тюнить по holdout;
- переносить исторические ответы менеджеров как факты;
- смешивать бренды в клиентском ответе;
- ослаблять P0 ради человечности.

## 7. Фаза B. Memory 2.0 внутри диалога

### Цель

Сделать память устойчивой не только как набор слотов, а как рабочее состояние разговора.

### Доработки

Расширить `DialogueMemory`:

- `topic_focus`: текущий продукт/курс/лагерь/формат;
- `last_direct_question`: последний прямой вопрос клиента;
- `unanswered_questions`: вопросы, на которые бот ещё не ответил;
- `safe_answered_parts`: какие безопасные части уже закрыты;
- `pending_manager_actions`: что обещано передать менеджеру;
- `client_confirmed_slots`: данные, сказанные именно клиентом;
- `crm_known_slots`: данные из CRM/Tallanto;
- `bot_inferred_slots`: слабые выводы, которые нельзя выдавать как факт;
- `do_not_reask_slots`: поля, которые нельзя спрашивать повторно;
- `conversation_summary_short`: 1-3 предложения без PII для prompt;
- `open_loop_summary`: что осталось закрыть;
- `route_memory`: почему раньше ушли к менеджеру или ответили сами.

### Правила

- слоты клиента извлекать только из клиентских реплик и проверенного CRM-контекста, не из ответов бота;
- текущая реплика имеет приоритет над старым контекстом;
- если клиент уточняет "а это сейчас?", "а можно закрепить?", "а есть места?", память должна связывать вопрос с предыдущим продуктом;
- если CRM и клиент расходятся, клиентская текущая реплика используется для подбора, CRM остаётся фоном;
- active brand задаёт канал, память не может его поменять.

### Tests

Добавить тесты:

- клиент дал класс/предмет на первом ходе, бот не спрашивает это на третьем;
- клиент сменил формат с онлайн на очно, бот не держится за старый формат;
- клиент спрашивает follow-up "а цена сейчас?", бот понимает, о каком продукте речь;
- бот не извлекает "программирование" из своих же ответов;
- known client из test-phone phrase не просит имя ребёнка повторно;
- multi-turn "лагерь -> выездной -> трансфер -> место" держит продукт.

## 8. Фаза C. Answer-first контроллер

### Цель

Принудить систему отвечать на последний прямой вопрос до следующего шага.

### Механика

Перед отправкой ответа определить:

- какой прямой вопрос задал клиент;
- какие безопасные части вопроса можно закрыть;
- какие факты есть;
- какие факты отсутствуют;
- есть ли P0/high-risk часть;
- какой один следующий шаг уместен.

Если первичный ответ:

- не отвечает на вопрос;
- отвечает на соседнюю тему;
- спрашивает уже известное;
- закрывает только одну часть составного вопроса;
- даёт менеджера при наличии безопасного факта;
- не даёт следующий шаг,

то ответ не считается готовым.

### Правила ответа

1. Если факт есть - начать с прямого ответа.
2. Если факт частичный - дать полезную часть и честно сказать, что проверит менеджер.
3. Если факта нет - дать безопасный ориентир без числа/даты/обещания.
4. Затем один следующий шаг.
5. Если есть P0 - safe части можно коротко закрыть только если это не ослабляет P0; иначе менеджер.

### Tests

- `ignored_question` regression;
- `answered_nearby_topic` regression;
- `single_topic_answer_to_multitopic_question` regression;
- "что входит в лагерь?" не отвечает только про цену;
- "можно закрепить и есть трансфер?" сначала трансфер, потом менеджер по месту;
- "вы бот?" прямой честный ответ, затем помощь по обучению.

## 9. Фаза D. LLM-rewriter как второй проход качества

### Цель

Включить второй проход только там, где детерминированный контроллер видит слабый ответ.

### Режим

Feature flag:

`TELEGRAM_ANSWER_QUALITY_LLM_REWRITER=1`

Включать только:

- для безопасных тем;
- когда нет P0/high-risk;
- когда active brand чистый;
- когда факты для конкретики есть или явно отсутствуют;
- когда есть findings уровня `rewrite`.

Не включать:

- для `manager_only`;
- для жалоб, возвратов, суда, спорной оплаты, договорных претензий;
- для live availability, если переписывание может обещать место/бронь;
- если ответ уже прошёл quality gate;
- для holdout tuning.

### Обязательные post-guards после rewriter

После переписывания снова прогнать:

- P0 guard;
- brand guard;
- unsupported promise guard;
- unsupported number/date/price guard;
- known-data reask guard;
- assumed unstated need guard;
- debug/source leak guard;
- template repetition guard.

Если после rewriter есть нарушение - fail closed:

- не отправлять переписанный ответ;
- вернуть безопасный исходный вариант или route to manager;
- записать failure class.

### Цель rewriter

Не "сделай красивее", а:

- ответь на последний вопрос;
- убери повтор известного;
- сохрани только подтверждённые факты;
- добавь один следующий шаг;
- сделай тон теплее и естественнее;
- не добавляй новых обещаний.

## 10. Фаза E. Тёплый тон без шаблонности

### Цель

Убрать ощущение "нейронного" или "канцелярского" ответа.

### Механика

Использовать few-shot/gold/playbook только как стиль и структуру, не как источник фактов.

Selector должен выбирать примеры по:

- brand;
- topic;
- route;
- fact requirements;
- message type;
- risk level.

Запрещено:

- дословно копировать примеры;
- использовать пример, если его факты не подтверждены в текущей КБ;
- использовать bad/good pairs как клиентский текст;
- повторять один и тот же зачин в нескольких ходах;
- начинать каждый ответ с "Поняла вас", "Спасибо", "Да, конечно" без связи с вопросом.

### Тон

Нужный стиль:

- спокойный;
- дружелюбный;
- как хороший консультант;
- без давления;
- без "впаривания";
- с заботой о ребёнке и удобстве семьи;
- с мягким следующим шагом.

### Tests

- template repetition across turns;
- no KB voice;
- no repeated opening;
- judge tone >= 80 on holdout;
- сотрудник может отправить ответ клиенту без переписывания.

## 11. Фаза F. Новый лид и мягкая продажа

### Цель

Новый клиент должен получить не просто справку, а движение к покупке.

### Поведение

Бот должен:

1. Ответить на вопрос.
2. Понять стадию клиента:
   - просто интересуется;
   - выбирает курс;
   - сравнивает цену;
   - готов записаться;
   - сомневается;
   - просит человека.
3. Собрать 1 недостающий ключевой слот:
   - класс;
   - предмет;
   - формат;
   - цель;
   - удобный следующий шаг.
4. Предложить один следующий шаг:
   - подобрать группу;
   - проверить место;
   - оформить заявку через менеджера;
   - показать формат;
   - уточнить предмет/класс;
   - передать менеджеру с уже собранным контекстом.

### Запрещено

- задавать анкету из многих вопросов;
- давить срочностью без факта;
- обещать место;
- обещать цену после срока;
- сравнивать Фотон и УНПК;
- продавать после P0.

### Метрики

- `lead_next_step_rate`;
- `known_slots_captured`;
- `one_question_only_rate`;
- `ready_for_manager_summary_rate`;
- `ready_to_send_rate`;
- `human_tone_score`.

## 12. Фаза G. Чистый holdout

### Почему

`v8_targeted16` уже стал dev-набором. По нему удобно чинить, но нечестно объявлять финальную готовность.

### Действия

Создать или пересобрать чистый holdout:

- минимум 40-60 диалогов;
- 50/50 Фотон и УНПК;
- не использовать сценарии, по которым прямо тюнили;
- включить новые лиды, known-client, off-topic, "ты бот?", составные вопросы, follow-up, P0, лагерь, цена, пробное, расписание, платформу, скидки, рассрочку.

Holdout нельзя:

- использовать как few-shot;
- использовать как training;
- переписывать под текущие ошибки;
- смотреть заранее для точечной правки фраз.

### Acceptance holdout

- P0 hard gates: 0;
- brand hard gates: 0;
- unsupported facts: 0;
- каждый критерий из раздела 4: >= 8/10;
- ready-to-send >= 75%;
- direct answer >= 80%;
- known-data reask <= 3%;
- over-handoff on green topics <= 15%;
- average human tone >= 80/100;
- no single failure class dominates больше 20% PASS_WITH_NOTES.

## 13. Фаза H. Прогоны

Порядок:

3. `v8_targeted16` как dev-сигнал.

## 15. Финальные acceptance criteria этапа

Этап считается закрытым только если:

1. Все безопасные тесты зелёные.
2. `v8_targeted16`: `FAIL=0`, hard gates 0.
3. Static `v6/v5`: реальные P0/brand/fact fails 0; устаревшие expectations помечены отдельно.
4. Clean holdout: минимум 8/10 по каждому критерию.
5. Full v8: hard gates 0, без массового `ignored_question`, `templated_opening`, `over_handoff`.
6. Pilot daily report показывает:
   - нет P0-инцидентов;
   - known-data reask близко к нулю;
   - есть очередь смысловых улучшений;
   - сотрудник может оценивать ответы.
7. `docs/BOT_FAILURE_CLASSES_REGISTRY.md` обновлён.
8. Создан audit pack с `semantic_review.md`.
9. Обновлены `CURRENT_STATE.md`, `ROADMAP.md`, `RUNBOOK.md`, `DECISIONS_LOG.md`.

## 16. Что делать первым после утверждения ТЗ

Первый implementation slice:

1. Доработать `DialogueMemory` до Memory 2.0.
2. Доработать answer-first assessment.
3. Подключить LLM-rewriter только по rewrite-флагу и только для safe тем.
4. Добавить post-guards после rewriter.
5. Добавить tests по `ignored_question`, `templated_opening`, `multi-topic`, `known-data reask`, `assumed_unstated_need`.
6. Прогнать focused regressions и `v8_targeted16`.
7. Создать audit pack и таблицу оценок "было -> стало".

Полный v8 - не в первом slice.

## 17. Ожидаемый результат

После этапа бот должен перейти из состояния:

`безопасный, но иногда шаблонный справочник`

в состояние:

`безопасный, внимательный и продающий консультант, который держит контекст, отвечает на вопрос и мягко ведёт к следующему шагу`.

Это не финальный SaaS-продукт и не полностью автономный менеджер, но это должен быть уровень, на котором внутренний пилот можно оценивать не только по безопасности, но и по реальной пользе для продаж.
