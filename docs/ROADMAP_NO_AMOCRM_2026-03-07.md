# Mango Analyse: План без amoCRM (2026-03-07)

## 1) Цель (без интеграции CRM)

Система должна:
- массово брать записи звонков из экспорта Mango;
- превращать каждый звонок в удобный диалоговый текст с ролями и таймингами;
- формировать структурный конспект звонка (`history_short` + блоки параметров);
- сохранять результаты локально (файлы + база), чтобы потом можно было:
  - вручную проверять качество,
  - делать аналитику,
  - позже подключить любой CRM-канал.

## 2) Как работает программа (простыми словами)

1. Вы указываете папку со звонками.
2. Программа считывает имя каждого файла и достает оттуда полезные данные:
   - дату/время,
   - номер телефона,
   - ФИО менеджера.
3. Программа проверяет аудио (длина, каналы, частота).
4. Запускается распознавание речи:
   - основной движок (MLX Whisper),
   - при включении второй движок (GigaAM),
   - потом собирается итоговый вариант текста.
5. Если звонок стерео (часто: менеджер/клиент по каналам), программа разделяет роли.
6. Реплики сортируются по времени и записываются как диалог:
   - `Менеджер (...)`
   - `Клиент`
7. Программа чистит шумы и повторы (эхо, дубли, служебные фразы).
8. Из текста строится конспект разговора:
   - короткая история звонка;
   - структура по блокам: интересы, класс, предметы, возражения, следующий шаг и т.д.
9. Всё сохраняется локально:
   - текст диалога,
   - служебный JSON с вариантами распознавания,
   - JSON анализа.
10. Отчеты качества показывают, где все хорошо, а где нужен ручной контроль.

## 3) Что проект уже делает сейчас

- Надежный ingest и повторные запуски без дублирования.
- Полный цикл транскрибации с fallback-логикой.
- Диалоговый формат с таймингами и ролями.
- Dual-ASR режим (MLX + GigaAM) и merge.
- QC-отчеты по качеству на больших прогонах.
- Анализ в версии схемы `v2`:
  - `history_short`,
  - `crm_blocks`,
  - `evidence`,
  - `quality_flags`,
  - backward-compatible legacy поля.
- Миграция старых анализов в новую схему (`migrate-analysis-schema`).
- Стабильный runtime (изоляция от текущих правок в `src/`).

## 4) Синхронизация с исходным планом (без amoCRM)

Исходный план: `ingest -> transcribe -> analyze -> sync`.

Состояние:
1. `ingest` — выполнено.
2. `transcribe` — выполнено (включая диалоговый экспорт и quality-патчи).
3. `analyze` — выполнено в `v2`, есть миграция схемы.
4. `sync` — сознательно отключаем из плана (по вашей задаче “без amoCRM”).

Итог: в non-CRM контуре закрыты все этапы до получения и хранения качественного структурного результата.

## 5) Что остается доделать в non-CRM контуре

1. Экспорт анализа в отдельные файлы рядом с транскриптами:
   - `<audio_stem>_analysis.json`,
   - `<audio_stem>_history.txt`.
2. Автоматический quality-gate:
   - помечать звонки “готово” / “нужна ручная проверка”.
3. Отдельный экспорт “очереди ручной проверки”:
   - список файлов с причинами риска.
4. Регулярный сводный отчет по партии:
   - темы, возражения, интересы, качество.
