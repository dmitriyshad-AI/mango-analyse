## 2. Дизайн источника первой очереди (read-only Tallanto)

**Фундамент уже в коде** (не проектирование с нуля, а достройка): `build_live_tallanto_context(phone, tallanto_id)` (tallanto_context.py:60) уже достаёт по телефону: контакты (до 5 кандидатов), сделки-opportunities (sales_stage, filial=бренд), requests, **finances (payment_summa, payment_status, date_entered)**, course_relations, class_relations. API универсальный по модулям (module="Contact"/"Opportunity"/…), есть `list_possible_modules/fields` — схема открывается пробой. Включение контура: `crm_tallanto_mode=http` (сейчас выключен — включается выдачей доступа).

**Три класса данных — где лежат и как достаём:**

| Класс | Физика (модуль Tallanto) | Ключ | Свежесть/кэш |
|---|---|---|---|
| Платежи + баланс клиента (882) | finances: payment_summa, payment_status, date_entered — ЕСТЬ уже; баланс занятий — поле контакта (в снимке это tallanto_balance) ИЛИ вычисление по визитам — подтвердить пробой list_possible_fields (вопрос В2) | телефон → профиль → tallanto_contact_id (exact); multi/нет → карточки нет | запрос на лету при вопросе, кэш 5–15 мин |
| Календарь занятий группы: дата/время/кабинет (846) | class_relations → class_id → модуль классов/занятий (поля расписания и кабинета — подтвердить пробой; источник правды сверить с группами Tallanto из выгрузки D4 — вопрос В6) | class_id групп ученика | кэш 6–24 ч + сброс утром |
| Места / статус набора группы (452) | класс: лимит мест vs счёт активных class_relations ИЛИ статусное поле набора — подтвердить пробой (В2) | class_id | кэш 1–6 ч |

**Контракт «карточки живых данных» (передача в бот; потребление — не моя часть):**

```json
live_card v1 {
  client_key: { phone_norm, tallanto_contact_id, match: "exact" | "multi" | "none" },
  brand: "foton" | "unpk",            // из filial; расхождение с active_brand канала -> карточка не выдаётся
  fetched_at, request_id,
  payments:   { ttl_sec: 900,   items: [ {date, sum, status} ] (последние 5), has_pending: bool },
  balance:    { ttl_sec: 900,   value, unit: "занятия"|"руб", as_of },
  schedule:   { ttl_sec: 21600, groups: [ {class_id, name, next_lessons: [{date, time, room}]} ] },
  enrollment: { ttl_sec: 3600,  groups: [ {class_id, набор: "открыт"|"закрыт"|"лист_ожидания", мест_осталось | null} ] },
  _provenance: на каждый блок source_ref + fetched_at,
  _filtered_at_source: ["ФИО преподавателей (политика: регалии без ФИО)", "данные других клиентов", "служебные id в клиентский текст"]
}
```

Правила источника: matched=multi (общий телефон) или бренд-расхождение → карточка НЕ строится; API недоступен/таймаут → карточки нет (бот живёт по текущей политике «рамка+менеджер») — протухшие данные без пометки не выдаются никогда; пустой блок = блока нет, не «0»; ФИО преподавателя (283 вопроса) отфильтровывается на источнике — это политика, не пробел данных.

## 3. «Перезвоните мне» (441) → черновик-задача в AMO

Механика: тот же сканирующий контур, что мой шаг 1 (без вебхуков). Бот НЕ звонит и НЕ пишет в AMO напрямую — создаёт ЧЕРНОВИК задачи-звонка (одобрение — по правилам пилота, механика одобрения за главным архитектором).

Поля задачи (api/v4/tasks): `task_type=звонок`; привязка: активная сделка клиента, без сделки → контакт; `text`: «[БОТ] Клиент просит перезвонить. Бренд: X. Суть: <1 строка>. Фраза: "<цитата>". Канал: Telegram»; телефон НЕ дублируется в текст (он в карточке); `responsible_user_id`: менеджер активной сделки, иначе дефолт бренда (вопрос В4); `complete_till` из фразы клиента:

| Фраза | Дедлайн |
|---|---|
| «сейчас / поскорее» | +1 рабочий час |
| «завтра» / «завтра утром» | завтра 10:00 |
| «после N часов» | сегодня N+1:00 (или след. день) |
| «вечером» | сегодня 18:00 |
| ночь (вне рабочего окна) | следующий рабочий день 09:30 |
| не сказано | +2 рабочих часа |

Идемпотентность: dedupe-ключ (chat_id, дата, intent=callback) — повторная просьба в том же диалоге не плодит вторую задачу, а сдвигает дедлайн существующей.
