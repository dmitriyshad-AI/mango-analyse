# П1. Прототип customer_timeline по звонкам + покрытие. 2026-05-27

Это **аналитический прототип** (не production-модуль): собрал из корпуса звонков ленту по каждому клиенту, ключ = телефон, факты разнёс по детям. Read-only, в CRM/stable_runtime ничего не писал. ПД замаскированы (телефон → хеш, имена детей → ярлык `[ребёнок-N]`). Цель — снять на цифрах главный риск, из-за которого timeline отложен (смешение историй).

## Что показало покрытие (июн-окт 2025)
| Метрика | Значение |
|---|---|
| Уникальных клиентов (телефонов) | **9 508** |
| Телефонов с ≥2 звонками (есть глубина истории для бота) | **5 078 (53%)** |
| Телефонов с 1 звонком | 4 430 |
| Телефонов с определённым ребёнком | 3 968 (42%) |
| Распределение звонков на телефон | 1:4430 · 2:1634 · 3:907 · 4:632 · 5:496 · 6+:~1400 |

## Главный вопрос — смешение детей под одним телефоном (риск утечки)
- Сырьём: **27,8%** телефонов с ребёнком имели ≥2 разных строки-имени.
- **После нормализации имён** (склейка ASR-вариантов «Егор» / «Иванов Егор» / «Иванов Е.» в одного): **9,9%**.
- **Вывод:** телефон ≈ уникальный клиент работает (≈90% — один ребёнок). Мультидети — ~1 из 10 семей: не правило, но и не редкость. Значит **разнесение фактов по ребёнку нужно закладывать сразу** (для тех самых 10%), а не как пост-фикс. Прототип это уже делает (кластеризация имён по общему токену).
- Оговорка честности: 9,9% — оценка моей нормализации; точная цифра требует канонизации ФИО (лучше — по ФИО из CRM, а не из ASR).

## Как устроен прототип (логика для Codex)
1. Группировка событий по `phone`.
2. По каждому событию берём из `analysis_json`: дату, продукт, интересы, класс, следующий шаг, краткий конспект, `child_fio` из `crm_blocks.people`.
3. Кластеризация детей по нормализованным токенам ФИО → ярлык ребёнка.
4. Лента сортируется по дате; факты привязаны к ярлыку ребёнка.
Файлы прототипа (в рабочей папке, ПД замаскированы): `timeline_coverage.json`, `timeline_examples_masked.json` (5 масштабных примеров).

## Что это даёт для разблокировки timeline
- Подтверждает: телефон — валидный ключ; мультидетность измерима и невелика (~10%).
- Готовый механизм разнесения по ребёнку → закрывает заявленный риск «бот выдаст чужую историю» (для одной семьи — раздельные карточки по детям).
- Покрытие 53% с ≥2 контактами — боту реально будет что показать по половине клиентов.

## Границы (честно)
- Источник только **звонки**. Почта/AMO/Tallanto/Telegram в этой сессии не подключены → полная единая лента — за Codex.
- Это прототип на чтении, не наполнение боевой `customer_timeline.sqlite`.
- Канонизация ФИО — по-хорошему из CRM, не из шумного ASR.
- Не трогал гейты включения (`timeline_preview_enabled` и т.д.) — их статус решает Codex/Дмитрий.

## Связь с Блоком B (запись в CRM)
Понимание подтверждено: бот **готовит** данные к записи (карточка по клиенту/ребёнку), а пуш в AMO/Tallanto делает человек одной командой. Прототип даёт ровно тот payload-черновик, который потом можно валидировать гейтами и запушить — но сам ничего не пишет.
