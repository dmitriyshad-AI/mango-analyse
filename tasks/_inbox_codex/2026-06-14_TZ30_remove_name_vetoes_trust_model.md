# ТЗ-30 (D8): снять детерминированные вето по ИМЕНАМ в резолвере детей — доверить модели, гейт = ручной регрейд

Дата: 2026-06-14. Постановка: Claude (архитектор), решение Дмитрия. Исполнитель: D8 (worktree `codex/tz24-dubli-deti`). Поверх реализации v2/v3 (коммиты 09a0487, 013090e).

## Доказанное основание (ручной регрейд по ПОЛНЫМ именам, 14.06)

Прочитал полные настоящие имена в 18 name-отказах расширенной микропробы. Практически ВСЕ — это ОДИН ребёнок, верно собранный моделью: ФИО + имя + уменьшительное + ASR-вариант (напр. «Дубинина Елизавета Андреевна / Елизавета / Лиза»; «Колосов Даниил / Данила / Данил / Даня»; «Евгений Борисов / Женя»; «Майчислав Леонидович» = ASR-каша «Вячеслав Леонидович»). Реальных братьев («Кулаков Никита / Кулакова Дарья») модель верно РАЗДЕЛИЛА. Реального переслияния разных детей в выборке НЕ найдено.

Вето `different_child_names_merged` (`child_names_are_compatible`: словарик уменьшительных + Левенштейн ≤1) и `mention_name_misattributed` (членство нормализованного имени) — РЕШАЮТ ИДЕНТИЧНОСТЬ ИМЕНИ детерминированно и ложно режут почти всё: не знают Лиза=Елизавета, не сводят ФИО с именем/уменьшительным, не чинят ASR. Это рецидив корневого изъяна «понимание в детерминированном слое» — на уровне НАШЕЙ проверки. Идентичность имени решает только модель.

ВАЖНО про метод оценки: идентичность имён НЕЛЬЗЯ мерить сравнением строк (так провалились и вето, и мой авто-классификатор, и счёт «5 разных имён» в прежнем аудите). Гейт качества — ручной разбор имён ПОНИМАНИЕМ.

## Изменения

### 1. Убрать детерминированные вето по именам
- УДАЛИТЬ проверку `different_child_names_merged` и функцию `child_names_are_compatible` + словарь `_NAME_ALIASES` + использование Левенштейна для идентичности (если больше нигде не нужны).
- УДАЛИТЬ `mention_name_misattributed`.
Идентичность имени строкой не верифицируем вообще.

### 2. Оставить МЕХАНИКУ (не трогает идентичность)
- `invented_child_name`: каждое имя в ответе должно встречаться среди ВХОДНЫХ написаний (чистое членство, анти-галлюцинация). Промпт: `canonical_name` — ровно одно из ВХОДНЫХ написаний.
- `child_with_empty_mention_ids`; полнота мэппинга (каждый mention ровно у одного, без unknown/duplicate); `children_count_exceeds_mentions` и верхняя граница по числу написаний (это лишь верхний КОЛПАК — слияние его не нарушает).
- `incompatible_grades_same_period` — ОСТАВИТЬ как ФИЗИЧЕСКИЙ инвариант (один ребёнок не может быть в 5 и 11 классе в один период; класс — число, не имя). Это НЕ оспаривание понимания, а независимый числовой факт. Если на прогоне начнёт ложно резать взросление (9→11) — расширить окно/смягчить, но не убирать.
- Бренд-провенанс на факт; стоп-лист общих телефонов; fail-soft на невалидный JSON.

### 3. merge_confidence — оставить ТОЛЬКО логирование
Не использовать как жёсткое вето. (Прошлый go/no-go «high на known-bad» переинтерпретирован: те семьи были ВЕРНЫМИ слияниями ФИО/уменьшительных, high был оправдан.) Поле логируем для моего регрейда, на поведение не влияет.

### 4. Промпт модели
- идентичность «один ребёнок или разные» решаешь ты; сливай ФИО+имя+уменьшительное+опечатки/ASR ОДНОГО ребёнка; РАЗНЫХ детей (в т.ч. братьев с общей фамилией, но разными именами) — раздельно;
- `canonical_name` — ровно одно из ВХОДНЫХ написаний (не выдумывать «правильную» форму); перечисли ВСЕ сгруппированные написания в `name_variants`;
- класс растёт со временем (9→11 — один ребёнок).

## Гейт качества = РУЧНОЙ регрейд (не скрипт)
Приёмка по именам — только ручной разбор архитектором (Claude) выборки слияний по ПОЛНЫМ именам, понимая их. Никаких авто-классификаторов идентичности (они проваливаются так же, как вето). Это явный пункт приёмки.

## NEG-тесты
- OFF (default) → байт-в-байт как сейчас (существующий тест не трогать).
- Ранее ложно-отклонённые кейсы теперь ПРОХОДЯТ как один ребёнок: «ФИО + имя + уменьшительное (Лиза=Елизавета, Женя=Евгений, Даша=Дарья, Дима=Дмитрий, Вова=Владимир, Даня=Даниил)» и ASR-варианты (Майчислав→Вячеслав, Святой Слава→Святослав).
- Братья с общей фамилией, разные имена (Никита / Дарья) → остаются ДВА ребёнка.
- invented (canonical не из входа) → отклонено.
- Пустые mention_ids / неизвестный / дубль mention → отклонено.
- Несовместимые классы в один период → не слиты (физический страж).
- Идемпотентность: кэш по входу + промпт-версия; PROMPT_VERSION ПОВЫСИТЬ (старый кэш не реюзать); новый out-root/чистый кэш для перепрогона.

## Замер
Повторить расширенную микропробу (≈120 семей) в НОВЫЙ out-root. Ожидание: name-отказов почти не остаётся; слияния ФИО/уменьшительных проходят. Результат — мне на РУЧНОЙ регрейд по полным именам (ложных отказов ~0, реальных переслияний разных детей ~0). Доля low по confidence — для справки.

## После
Один ПОЛНЫЙ прогон 7 512 по «да» Дмитрия. Регрейд — мой ручной по выборке имён. Безопасность держится на: понимании модели + механических проверках + ФИЗИЧЕСКОМ классовом страже + моём ручном регрейде (НЕ на строковом правиле имён). Включение в прод = одноразовая полная пересборка.

## Правки по аудиту второго архитектора (раунд ТЗ-30, обязательны)

Аудит подтвердил посылку по полным именам (20/21 ложных name-отказа, реальных переслияний через name-вето нет) и дал PASS_WITH_NOTES. Внести:

1. **Идемпотентность `child_key` (важно).** Сейчас `child_key` = hash(`canonical_name`.lower()) (`target_child_key_for_child`). При правиле «canonical из входа» модель на чистой пересборке может выбрать РАЗНОЕ входное написание как canonical → разный child_key прогон-к-прогону (поведение карточки то же, но ключ дрейфует). Фикс: строить `child_key` из СТАБИЛЬНОГО признака группы — hash от ОТСОРТИРОВАННОГО множества `mention_id` группы (или норм-имён), а НЕ от ярлыка-canonical. Тогда ключ устойчив к выбору написания.
2. **Тесты под ЗАМЕНУ (не молча удалять).** Завязаны на снимаемое поведение: `test_..._rejects_name_misattribution` (tests:581), `test_..._rejects_merging_different_named_children` (617), `test_..._high_confidence_does_not_override_name_veto...` (653). Переписать в «теперь ПРОХОДИТ как один ребёнок» (на реальных кейсах ФИО+уменьшительное). Инфраструктуру `write_name_veto_diagnostic` СОХРАНИТЬ — она полезна как сырьё моего регрейда, даже без кодов-вето.
3. **`merge_confidence` — не топить семью.** Сейчас `invalid_merge_confidence` = жёсткий отказ. Поле объявлено необязательным для поведения → при невалидном/отсутствующем confidence НЕ отклонять семью, а дефолтить в `low` + лог. Иначе сбой модели по необязательному полю рушит верное слияние.
4. **Гейт регрейда + дешёвый НЕ-строковый авто-инвариант.** В мою выборку обязательно включить семьи с `merge_confidence=low` И ≥2 названными корнями — там концентрируется риск модельного переслияния, которое классовый страж пропустит при СОВМЕСТИМЫХ классах. Плюс добавить дешёвую регрессию (чистая механика, не понимание): «после резолва число детей у семьи НЕ РАСТЁТ относительно числа входных норм-имён».
5. **Записать остаточный риск и ограничение.** (а) После снятия name-вето, если модель сольёт двух РАЗНЫХ детей с СОВМЕСТИМЫМИ классами и без выдумок — ничто авто не перехватит (классовый страж молчит); теряется слот в карточке менеджера (не P0, не бренд-утечка) — ловит мой ручной регрейд. (б) Известное ограничение ASR/экстракции: одно упоминание «Артём, Марк» (два брата в одной строке) уходит ровно одному ребёнку — второй теряет это упоминание (не самого ребёнка); снятие вето это НЕ лечит, отдельный дефект.

Класс-страж проверен аудитом: режет только `|diff|≥3` при `≤200 дней`; взросление 9→11 (diff=2) НЕ блокируется по построению; диапазоны «5–6» → None → игнор. Оставить как есть; порог трогать только если на полном 7512 пойдут ложные на ASR-ошибке одной цифры.
