## Зачем (и почему это безопасно)
- **Свобода:** бот отвечает живо на дорогу/логистику/общий совет/быт с честной оценкой и маркером — без верхней
  классификации, которая сейчас душит травел.
- **Безопасность та же:** выдумок ПРОДУКТОВЫХ чисел клиенту = 0. Маркер НЕ спасает продуктовое число («скорее всего
  ~50 000 ₽» → блок): родитель анкорится на цену, а ошибка цены = сломанное доверие/претензия/P0.
- Это наша же философия «модель пишет — детерминированный гейт проверяет ВЫХОД», доведённая до уровня числа. Снимает
  верхний pricing-классификатор, с которым воюем (Лобня).

## Железный принцип (гейт по числу, в готовом тексте)
Для КАЖДОГО числа в выходном тексте:
1. **Грунтовано** (есть в фактах / названо клиентом / структурное) → ОК.
2. **Негрунтованное в ПРОДУКТОВОМ контексте** (рядом ₽/руб/%/скидк/рассрочк/цена/тариф/семестр/за год/HH:MM/дата/
   месяц/смена/лагерь/места/баллы/группа/ак.ч/недель/«длится»-длительность; НЕ просто слово «занятие») → **БЛОК ВСЕГДА**
   (маркер НЕ спасает).
3. **Негрунтованное ОБЩЕЕ** (минуты/км/часы дороги, «за 1.5-2 года» — рядом НЕТ продуктовых маркеров) → ОК **только
   при маркере неуверенности рядом**; без маркера → блок (даунгрейд).
Композиция СВОБОДНА. `answer_mode`/`estimate_domain` ПОНИЖАЮТСЯ до подсказки/метаданных (НЕ удаляются — на них завязан
маршрут/estimate_compose; снимаем лишь их АВТОРИТЕТНОСТЬ запрещать travel/general числа). Бренд/P0/давление/гарантии/
leak/fake-enrollment — СВЕРХУ, не ослаблять.
**КРИТИЧНО (правка Кодекса №2): semantic faithfulness-критик ОБЯЗАН работать и на free/estimate-ответах.** Сейчас
`_hard_check` ПРОПУСКАЕТ faithfulness при `answer_mode=="estimate_allowed"` — этот пропуск УБРАТЬ. Числовой гейт не видит
НЕчисловых выдумок («места есть», «можно записаться», «рассрочка доступна») — их ловит только faithfulness.
Фолбэк при блоке: даунгрейд на честный хендофф / на грунтованную часть (НЕ авто-правка числа). Флаг
`TELEGRAM_A_FREE_NUMBER_GATE` (default OFF), своя приёмка.

## ФАЗА 1 — ЗАКРЫТА (ревью Кодекса принято; 7 правок встроены)
Кодекс подтвердил по коду: `answer_mode/estimate_domain` реально управляют маршрутом (`_resolve_answer_mode`,
`estimate_compose`, `estimate_gate`); `verify_output` зависит от `answer_mode`; P0 на `answer_mode` НЕ держится
(`p0_pre_gate`/ранний P0/`classify_answer_safety`/финальный P0 — отдельны, не трогать). Формулировка: не «снять
классификацию», а «снять её АВТОРИТЕТНОСТЬ для числового estimate; финал решает output number gate».

### 7 правок Кодекса (ПРИНЯТЫ, обязательны):
1. **answer_mode ПОНИЗИТЬ до подсказки/метаданных, НЕ удалять** (на нём маршрут/estimate_compose).
2. **semantic faithfulness ВКЛ на free/estimate** — убрать пропуск в `_hard_check` при `estimate_allowed` (иначе
   НЕчисловые выдумки «места есть»/«рассрочка доступна» пройдут). КРИТИЧНО.
3. **Клиентское число НЕ грунтует продуктовый claim:** «курс 50000?»→«да,50000» = блок. Echo/контекст ≠ подтверждение.
4. **Шире продукт-контекст:** +ак.ч, занятий, недель, ЛВШ/ЛШ, интенсив, заезд, места, баллы, группа, СФР/ФНС, справка,
   короткие дни вт/чт, время «в 18», даты «3-14.08». (Встроено в Полуфабрикат 2.)
5. **«занятие» само ≠ продукт-контекст** (травел «15 мин до занятия» проходит); «занятие ДЛИТСЯ 90 мин» и «3 раза в
   неделю» — блок без факта.
6. **Сильнее нормализация чисел:** диапазоны/формы `15-20`, `1.5-2`, `29 750`, `50к`, `50 тыс.`, `18:00`, `2026/27`,
   `01.08`, `20 процентов`.
7. **Порядок:** факт → продукт-контекст → структурное/клиентское → маркер (структурное ПОСЛЕ продукт-контекста).

## ФАЗА 2 — полуфабрикаты

### Полуфабрикат 1 — что считается ГРУНТОВАННЫМ (чтобы не блокать безобидное)
```python
_STRUCTURAL_OK = {str(n) for n in range(1, 12)}          # классы 1-11
_YEARS_OK = {"2024","2025","2026","2027","2026/27","2025/26","2024/25"}
def _build_grounding(facts: dict, conversation_text: str):
    fact_norm = _normalize_number_surface(" ".join(facts.values()))
    client_nums = {_normalize_number_surface(t) for t in _SELLING_NUMBER_RE.findall(conversation_text or "")}
    return fact_norm, {n for n in client_nums if n}
```
`conversation_text` = реплики КЛИЕНТА в диалоге (он сам назвал класс/кол-во → это не выдумка бота).
**Нормализация (правка Кодекса №6):** `_normalize_number_surface` усилить, чтобы грунтовка/сравнение ловили формы и
диапазоны: `15-20`, `1.5-2`, `29 750`/`29750`, `50к`/`50 тыс.`→`50000`, `18:00`, `2026/27`, `01.08`, `20 процентов`→`20%`.
Иначе грунтованное «29 750 ₽» из факта не совпадёт с «29750» в тексте и даст ложный блок.

### Полуфабрикат 2 — ПРОДУКТОВЫЙ контекст числа (окно вокруг числа)
```python
import re
_PRODUCT_CTX_RE = re.compile(
    r"₽|руб|р\.|%|процент|скидк|рассрочк|долями|цена|стоит|стоимост|тариф|семестр|за год|оплат"
    r"|\b\d{1,2}:\d{2}\b|расписан|по (понедельник|вторник|сред|четверг|пятниц|суббот|воскресень)"
    r"|\b(пн|вт|ср|чт|пт|сб|вс)\b|\d{1,2}[-–]\d{1,2}\.\d{2}|\b\d{1,2}\.\d{2}\b"            # короткие дни + даты
    r"|январ|феврал|март|апрел|ма[яй]|июн|июл|август|сентяб|октяб|ноябр|декабр|смен[аеуы]|заезд|лагер"
    r"|лвш|\bлш\b|интенсив|мест[ао]\b|балл|групп|сфр|фнс|справк|ак\.?\s?ч|занятий|недел"          # правка Кодекса №4
    r"|длится|длительност|академ|раз в недел|час[аов]* в недел", re.I)   # длительность = ЯВНО «длится», НЕ слово «занятие»
def _is_product_number_context(text: str, token: str) -> bool:
    i = text.find(token)
    if i < 0:
        return bool(_PRODUCT_CTX_RE.search(text))
    win = text[max(0, i-28): i+len(token)+28]
    return bool(_PRODUCT_CTX_RE.search(win))
```
ВАЖНО (само-аудит): НЕ ставить слово «урок/занят» в продукт-контекст — иначе травел «15 минут до занятия» ложно
заблокируется как длительность (ровно то, что чиним). Длительность ловим только по «длится/длительност/академ.час».
Кодекс: добирать осторожно по NEG.

### Полуфабрикат 3 — маркер неуверенности РЯДОМ с числом
```python
_UNC_MARKERS = ("ориентировоч","примерно","навскидк","скорее всего","не уверен","точно подскажет менеджер",
                "точную информацию уточнит менеджер","около","порядка","в районе","приблизительно","где-то",
                "обычно","как правило","в среднем","чаще всего")  # «волшебные слова» — честные маркеры общей оценки
def _has_marker_near(text: str, token: str) -> bool:
    i = text.find(token)
    seg = text if i < 0 else text[max(0, i-60): i+len(token)+60]
    low = seg.casefold()
    return any(m in low for m in _UNC_MARKERS)
```
Маркер должен быть РЯДОМ (окно ~60), чтобы один далёкий маркер не «прикрывал» другое число.

### Полуфабрикат 4 — ядро: проверка ПО ЧИСЛУ (порядок важен)
```python
def verify_output_numbers(text, *, facts, conversation_text) -> tuple[bool, str]:
    fact_norm, client_nums = _build_grounding(facts, conversation_text)
    for token in _SELLING_NUMBER_RE.findall(text or ""):
        num = _normalize_number_surface(token)
        if not num:                       continue
        if num in fact_norm:              continue               # 1) грунтовано ФАКТОМ → ок (для ПРОДУКТОВЫХ — единственный путь)
        if _is_product_number_context(text, token):
            return False, "unsupported_product_number"           # 2) продуктовое НЕ из факта → блок (правка Кодекса №3:
                                                                 #    клиентский echo НЕ грунтует продукт! «курс 50000?»→«да,50000» = блок)
        if num in client_nums or token in _STRUCTURAL_OK or token in _YEARS_OK:
            continue                                             # 3) безобидное НЕ-продуктовое: клиент назвал / структурное (класс/год)
        if not _has_marker_near(text, token):
            return False, "general_number_without_marker"        # 4) общее без маркера → блок (даунгрейд)
        # 4') общее негрунтованное + маркер рядом → ПРОПУСК (свобода)
    return True, ""
```
ПОРЯДОК КРИТИЧЕН (правки Кодекса №3,№7): факт → продукт-контекст → клиент/структурное → маркер. Клиентское/структурное
грунтование ПОСЛЕ продукт-контекста: иначе «2 раза в неделю» (2∈1-11) или «да, 50 000» (клиент назвал) проскочат как
безобидные. Продуктовое число грунтуется ТОЛЬКО фактом (шаг 1), не клиентским вопросом.

### Полуфабрикат 5 — композер: свобода + честные маркеры (промпт)
Убрать из гейта верхнюю классификацию; композеру дать свободу + правило маркеров:
«Отвечай естественно и помогай. Если чего-то нет в подтверждённых фактах, но это бытовое/дорога/логистика/общий
совет — дай ПОЛЕЗНУЮ оценку, ОБЯЗАТЕЛЬНО пометив неуверенность ("ориентировочно", "скорее всего", "точно подскажет
менеджер"). НИКОГДА не называй продуктовую конкретику (цену, %, расписание, даты, длительность занятия), которой нет
в фактах — по ней честно к менеджеру. Не выдумывай продуктовые числа даже с оговоркой.»

### Полуфабрикат 6 — ПОНИЗИТЬ авторитетность верхней классификации (НЕ удалять)
`answer_mode`/`estimate_domain` остаются в контракте/логах/prompt-hint (на них завязан маршрут/estimate_compose — Кодекс
подтвердил), но числовой гейт им НЕ доверяет и НЕ форсит confirmed_only для travel/general чисел. `_is_product_question`
как АВТОРИТЕТНЫЙ запрет — снять. Маршрут больше не лепит pricing на «сколько ехать». И УБРАТЬ пропуск faithfulness при
`estimate_allowed` в `_hard_check` (правка №2). P0-слои (p0_pre_gate/ранний P0/classify_answer_safety/финальный P0) НЕ трогать.

### Полуфабрикат 7 — фолбэк при блоке
`unsupported_product_number` / `general_number_without_marker` → НЕ публиковать число; даунгрейд на честный хендофф
ИЛИ на ответ без негрунтованного числа (грунтованную часть оставить). НЕ авто-вставлять маркер и НЕ авто-править число.

### Полуфабрикат 8 — NEG-матрица (СТРОГО; это правка безопасности)
- **продуктовое с маркером (ГЛАВНЫЙ NEG):** «скорее всего курс ~50 000 ₽» → БЛОК (₽ рядом). «ориентировочно скидка
  20%» → блок (%). «вроде по вторникам в 18:00» → блок (HH:MM/день). «занятие ДЛИТСЯ минут 90» → блок (длится). Маркер НЕ спасает.
- **длительность без слова «длится»:** «занятие минут 90» без факта/маркера → блок (общее-без-маркера); с маркером —
  пропуск (низкие ставки; Кодекс может ужесточить в Фазе 1).
- **травел рядом со словом «занятие» (НЕ ложно блокать):** «ориентировочно 15 минут до занятия» → ПРОПУСК (это травел,
  не длительность; «занятие» само НЕ продукт-контекст).
- **травел общее с маркером:** «ориентировочно 15-20 минут на электричке» (нет продукт-маркеров рядом) → ПРОПУСК.
- **травел без маркера:** «ехать 15 минут» → блок (даунгрейд) → композер обязан добавлять маркер.
- **грунтованное продуктовое:** «онлайн 29 750 ₽ [из факта]» → пропуск (грунтовано).
- **безобидное структурное:** «для 9 класса», «2026/27 год», «двое детей» (клиент назвал) → НЕ блок.
- **частота/нагрузка выдуманная:** «занятия 3 раза в неделю» без факта → блок (раз в недел = продукт-контекст).
- **общий совет:** «обычно за 1.5-2 года» (маркер «обычно» рядом, нет продукт-контекста) → пропуск.
- **смешанный (Лобня):** «сколько дорога + класс/физика рядом» → бот даёт травел-оценку с маркером (пропуск); если
  выдумал цену — блок. Цена больше НЕ лезет (нет pricing-классификации).
- **нечисловая выдумка (правка №2):** «места есть» / «можно записаться» / «рассрочка доступна» БЕЗ факта → ловит
  semantic faithfulness-критик (числовой гейт их НЕ видит). faithfulness ОБЯЗАН быть ВКЛ на free/estimate-ответах.
- **бренд/P0/давление/гарантия/fake-enrollment:** без изменений, держатся.
- **флаг OFF:** TELEGRAM_A_FREE_NUMBER_GATE OFF → поведение как сейчас.

## Приёмка
pytest + NEG зелёные (флаг ON и OFF). Перепрогон ТОГО ЖЕ набора Level A на M1 (travel + general_advice + продукт-
адверсариал + длительность) + я регрейжу ПО СЫРЬЮ: травел даёт оценку с маркером; продуктовых выдумок 0 (даже с
маркером); безобидные числа не блокируются; бренд/P0 целы. Только после — катить (флаг default OFF).

проверять ДО структурного allowlist; не ослаблять бренд/P0/давление/гарантии/leak; не авто-править число; не кодить
до согласования Фазы 1; полный pytest green; не git reset.
