## C. Неэффективность

1. **Дублирующееся поле «Последняя AI-сводка»** — `_compose_last_summary()` (строка 176) сначала ищет «Краткое резюме последнего свежего звонка», если нет — берёт «Краткая история общения». При этом в `_compose_auto_history()` то же поле «Краткая история общения» уже используется для блока «Сводка клиента». Итог: одинаковый текст попадает в два поля AMO, что `crm_text_quality_detector.py` ловит как `cross_field_duplicate_information` (P3-предупреждение, не блокирует live).

2. **Захардкоженная дата анализа** — `analysis_date: str = "2026-05-13"` в `DealTextPaths` (deal_text_builder.py строка 126) и в `DealAwareStage6Paths` (deal_writeback.py строка 37) и в `DealQualityGatePaths` (deal_quality_gate.py строка 84). Влияет на `build_followup_hint()` — все даты «следующего касания» считаются относительно 13.05.2026, то есть уже полностью устарели. **ПРОВЕРЕНО.**

3. **Множественные точки нормализации** — `normalize_manager_text()` вызывается на каждом поле-входе `_compose_auto_history()` (строки 189–198), а потом ещё раз внутри `fit_text(normalize_manager_text(value), ...)` в `build_deal_payload()` (строка 519) — двойная нормализация. Не ломает текст, но избыточна.

4. **stage1_snapshot.py строка 222** — срез `[:1000]` на поле `auto_history_preview` при снятии снимка. Это только для preview-снимка, не для итоговой записи в AMO. **ПРОВЕРЕНО — не влияет на live-запись.**

## D. Все места обрезки/порчи текста

### D1. `_compact_without_ellipsis()` — обрезка без контекста конца предложения с суффиксом `[сжато]`
- Файл: `scripts/write_amo_ready_contacts.py`, строки 229–238
- Константы: `MAX_AMO_TEXT_FIELD_CHARS = 240`, `MAX_NEXT_STEP_CHARS = 800`, `MAX_LAST_SUMMARY_CHARS = 1200`, `MAX_AUTO_HISTORY_CHARS = 1600` (строки 67–70)
- Алгоритм: срез до `budget = limit - len(" [сжато]")`, затем ищет слово/знак препинания для красивого отрыва (rfind пробела, запятой, точки с запятой, точки); если не нашёл — жёсткий срез
- **Пример обрезки (обезличенно):** «Клиент интересовался курсом по математике для 9 класса, спрашивал о рассрочке. Запись [сжато]» — слово «Запись» оборвано после запятой, следующий шаг потерян
- Используется: для поля «AI-рекомендованный следующий шаг» (лимит 800 символов) и для «Последняя AI-сводка» (лимит 1 200 символов)
- **ПРОВЕРЕНО:** суффикс `[сжато]` гейт `crm_text_quality_detector` специально отслеживает как `lossy_ellipsis_truncation` (строка 72 deal_quality_gate.py: `ELLIPSIS_RE = re.compile(r"\.\.\.|…")`) — НО суффикс `[сжато]` другой, он не попадает под этот паттерн. **ГИПОТЕЗА (не подтверждено):** `[сжато]` может проходить гейт незамеченным.

### D2. `fit_text()` — обрезка текста AI-полей с суффиксом «Текст сокращен до лимита поля.»
- Файл: `src/mango_mvp/deal_aware/deal_text_builder.py`, строки 1257–1270
- Применяется в `build_deal_payload()` строка 519 для КАЖДОГО из 12 AI-полей сделки
- Лимиты (строки 1273–1280): «AI-история по сделке» → 4 500 симв.; «AI-сводка по сделке», «AI-Tallanto статус по сделке» → 1 600 симв.; «AI-рекомендованный следующий шаг», «AI-дата следующего касания», «AI-приоритет сделки» → 350 симв.; остальные → 1 300 симв.
- Алгоритм: ищет `last_stop` — последний `. `, `; `, `! `, `? ` в срезе; если позиция >= 160 — обрезает по нему, иначе жёсткий срез со strip. Добавляет «Текст сокращен до лимита поля.»
- **Пример обрезки (обезличенно):** поле «AI-рекомендованный следующий шаг» лимит 350 символов. Если шаг содержит «Связаться с клиентом и уточнить дату оплаты; если оплата не поступит — зафиксировать в AMO и передать менеджеру на...» — обрезается на «...передать менеджеру на. Текст сокращен до лимита поля.»
- **ПРОВЕРЕНО** по коду

### D3. `history_call_summary()` — обрезка строки истории звонка с суффиксом «Детали в полном звонке.»
- Файл: `src/mango_mvp/deal_aware/deal_text_builder.py`, строки 1236–1254
- Вызывается из `build_deal_history()` строка 566: `history_call_summary(..., max_sentences=2, max_chars=320)`
- Алгоритм: берёт первые 2 предложения; если результат > 320 симв. — разбивает по `;:,` и добирает по клаузам до `max_chars - 26`; если chunk < 80 симв. — жёсткий срез `[:max_chars-26]`; добавляет «. Детали в полном звонке.»
- **Пример (обезличенно):** «Звонили родители по поводу лагеря. Ребёнок 7 класс, интересует смена в июле, цена вопрос. Детали в полном звонке.» — второе предложение может быть обрезано на полуслове перед «. Детали в полном звонке.»
- **ПРОВЕРЕНО** по коду

### D4. `short_evidence()` — срез доказательного текста возражения до 151 символа с «[сжато]»
- Файл: `src/mango_mvp/deal_aware/deal_text_builder.py`, строки 780–784
- Используется при заполнении поля `evidence` в структурированных возражениях
- Алгоритм: `text[:151].rstrip() + " [сжато]"` — жёсткий срез без поиска границы слова
- **Пример (обезличенно):** «Клиент сказал что дорого, будет обсуждать с мужем на следующей недел [сжато]» — слово «неделе» оборвано на «недел»
- **ПРОВЕРЕНО** по коду; суффикс `[сжато]` тот же что в D1 — вероятно не ловится `lossy_ellipsis_truncation` гейтом (гипотеза)

### D5. `_compact_without_ellipsis()` на поле «Авто история общения»: нет финального лимит-гейта на весь блок
- Файл: `scripts/write_amo_ready_contacts.py`, строки 186–226 (`_compose_auto_history`)
- `_compose_auto_history()` собирает блоки через `"\n\n".join()` и `.strip()`, но **не вызывает `_compact_without_ellipsis()` на итоговом тексте** (константа `MAX_AUTO_HISTORY_CHARS = 1600` объявлена на строке 70, но не применяется внутри функции)
- Итоговый текст контролируется только внешним гейтом `detect_crm_text_quality_risks()` и ограничением AMO-поля (textarea)
- **ПРОВЕРЕНО:** итоговый результат `_compose_auto_history()` идёт в payload напрямую без обрезки. Если все подполя заполнены (история + хронология + tallanto), суммарный размер может превысить 1 600 симв. — гейт поймает как предупреждение, но не заблокирует

### D6. `normalize_manager_text()` — удаление маркеров `[сжато]` и `[truncated]` перед финальной обработкой
- Файл: `src/mango_mvp/deal_aware/deal_text_builder.py`, строки 1091–1099
- Строки 1093–1094: `text = text.replace("[сжато]", "").replace("[truncated]", "")`
- Это ПОЗИТИВНАЯ очистка — убирает маркеры из исходных данных менеджеров. Но если маркер стоял в середине предложения, текст после него исчезает, а удаление маркера может оставить двойной пробел или обрубленное предложение
- **Пример (обезличенно):** входной текст «Клиент хочет курс [сжато] договорились перезвонить» → после normalize → «Клиент хочет курс  договорились перезвонить» (двойной пробел схлопывается через `re.sub(r"\s+", " ", text)` — ОК). Но семантически текст между «курс» и «договорились» потерян без признаков потери.
- **ПРОВЕРЕНО** по коду; семантический контекст может теряться незаметно

### D7. `build_deal_history()` — обрезка истории до 12 последних звонков
- Файл: `src/mango_mvp/deal_aware/deal_text_builder.py`, строки 552–571
- `selected = calls[-max_calls:]` где `max_calls=12`; если всего звонков > 12, добавляется prefix-уведомление менеджеру «Показаны последние 12 из N релевантных звонков»
- **Это намеренное ограничение** с явным сообщением. Нарушение текста нет, есть информационная потеря. **ПРОВЕРЕНО** — норм.

### D8. `normalize_objection()` — тихое удаление возражений длиннее 90 символов
- Файл: `src/mango_mvp/deal_aware/deal_text_builder.py`, строка 1305: `if len(text) > 90: return ""`
- Длинные возражения (например «клиент сказал что слишком дорого и не понимает чем отличается от обычного онлайн-курса») тихо выбрасываются без сообщения об этом в итоговом поле
- **Пример (обезличенно):** возражение «не понимает в чём разница между очным и онлайн форматом, хочет поговорить с преподавателем» (94 символа) → в поле «AI-актуальные возражения» не попадает, нет никакого индикатора. **ПРОВЕРЕНО** по коду; это потенциальный источник неполноты поля.

### D9. `serialize_structured_objections()` — обрезка JSON возражений по байтам до 12 000 символов
- Файл: `src/mango_mvp/deal_aware/deal_text_builder.py`, строки 787–800
- max_items=50, max_chars=12000; если JSON превышает — цикл срезает последние элементы до fit; `truncated=True`
- Данные сохраняются во внутреннее поле `structured_objections_json`, не в прямое CRM-поле. **ПРОВЕРЕНО** — на видимый текст карточки не влияет.

## E. Проверено vs гипотеза

**ПРОВЕРЕНО по коду:**
- D1 `_compact_without_ellipsis()` существует и применяется с суффиксом `[сжато]`
- D2 `fit_text()` существует, лимиты полей верны, суффикс «Текст сокращен до лимита поля.»
- D3 `history_call_summary()` обрезает по 320 симв. с суффиксом «Детали в полном звонке.»
- D4 `short_evidence()` срез `[:151]` жёсткий, без границы слова
- D5 `_compose_auto_history()` не применяет `MAX_AUTO_HISTORY_CHARS` к итоговому тексту
- D6 `normalize_manager_text()` удаляет `[сжато]` и `[truncated]` бесследно
- D7 max_calls=12 с явным prefix-уведомлением — норма
- D8 возражения >90 симв. тихо выбрасываются
- Захардкоженная дата `2026-05-13` в трёх местах
- Контур 3 (agent_runtime) пишет `ctx.history_summary` напрямую без `fit_text()` или `_compact_without_ellipsis()` — **ГИПОТЕЗА: если history_summary длиннее лимита AMO-поля, API может вернуть ошибку без явной обработки длины**

**ГИПОТЕЗА (не подтверждено по тестовым данным, только по коду):**
- Суффикс `[сжато]` в D1 и D4 не попадает под паттерн `ELLIPSIS_RE = re.compile(r"\.\.\.|…")` в deal_quality_gate.py; значит `lossy_ellipsis_truncation` не срабатывает на этот маркер — обрезка может проходить незамеченной
- `_compose_auto_history()` без финального лимита может генерировать текст > 1 600 симв. при заполненном Tallanto-блоке; гейт поймает только как P3-warning, не блокирует

9. **[C1]** `write_amo_ready_contacts.py:176–183` — `_compose_last_summary()` и `_compose_auto_history()` оба используют «Краткая история общения»; один текст попадает в два поля AMO → риск `cross_field_duplicate_information`.
