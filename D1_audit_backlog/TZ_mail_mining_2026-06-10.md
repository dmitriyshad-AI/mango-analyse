## Этап 1. Каталог исходящих ШАБЛОНОВ менеджеров (самое ценное по словам владельца)

Кластеризовать исходящие по повторяемости → канонический текст каждого шаблона (после записи на курс, маткапитал, квитанция/оплата, летние школы, справки, программы...). Для каждого: бренд, частота, канонический текст (обезличенный), вариации, период использования (не устарел ли). Выход: `outgoing_templates_catalog.md` + ОТДЕЛЬНО таблица «шаблон ↔ что говорит база знаний по этой теме» — расхождения шаблонов с базой r3 (шаблоны менеджеров = источник правды о процессах не хуже документа: где письмо говорит иное, чем база — флаг владельцу).

## Этап 2. Эталонные пары «вопрос клиента → письменный ответ менеджера» (email-голд)

Из цепочек переписки: пары вопрос→ответ, где ответ менеджера содержательный (не отписка). Отобрать ~40-60 лучших по темам карты спроса. Формат как голд-пак (client / manager_response_masked / topic / brand / source), ВСЕ числа и даты → плейсхолдеры [из фактов]. Это сырьё двух применений: email-голд для будущего почтового канала и доукомплектация мессенджер-голда (письменная речь менеджеров в почте обычно полнее).

## Этап 3. Вопросы клиентов из почты → дельта для покрытия базы

Каталог тем входящих вопросов с частотами (метод как в DEMAND_map). Выход в формате, совместимом с работой Клода-2 (`kb_intake_20260610/questions_catalog.md`), отдельным файлом `mail_questions_delta.md` — Клод-2 или архитектор сольют с общей сверкой покрытия.

## Этап 4. Заметка «почта как канал бота» (1 страница, дизайн НЕ делать)

По фактуре архива: какова структура типичного диалога в почте (длина, время ответа, доля шаблонных ответов), что из механики мессенджер-бота переносится (черновик → менеджер), что нет (подписи, вложения, треды). Только наблюдения для будущего ТЗ канала.

## Дисциплина

Этапы сдавать по одному (0 → 1 → 2 → 3 → 4), после каждого — SUMMARY-абзац; работать скриптами+выборочным чтением, не «прочитать всё подряд»; объёмы обработанного указывать честно. Конвейер приёмки: смысловое ревью архитектора → решения владельца → внесение в базу/голд только через Кодекса.
