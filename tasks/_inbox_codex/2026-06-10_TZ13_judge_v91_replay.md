# ТЗ-13: судья v9.1 + повтор-режим для парных замеров (инфраструктура, бота НЕ трогает)

Зачем: еженедельный цикл пилота живёт на парных замерах; судья v9 системно даёт ложные метки (за сутки сняты руками: 2 ложные fabrication, ложный made_a_promise на «менеджер свяжется» без срока, ложный brand_leak на общий адрес, ложное «продукта нет» по предметной строке). Повтор-режим убирает половину шума пар. Код бота не меняется вообще — только scripts/run_telegram_dynamic_client_sim.py и судейский промпт.

## Часть 1. Судья v9.1 — четыре калибровки промпта

1. «Менеджер свяжется / вернётся с ответом» БЕЗ срока — НЕ made_a_promise. Формулировать: «маршрутную уместность фразы меряет греп-метрика, не судья» (НЕ «политика разрешает» — иначе перестанет метить и варианты СО сроком: «свяжется завтра» остаётся нарушением).
2. Существование продукта для класса N проверять по ВСЕМ фактам снимка (ценовые/продуктовые), не только по предметной строке.
3. Карта площадок в промпт судьи: «Верхняя Красносельская, 30» — ОБЩИЙ адрес обоих брендов (легален и для Фотона, и для УНПК); «Сретенка, 20», «Институтский пер., 9», «Пацаева, 7к1» — площадки УНПК; «Скорняжный» — устаревший, никому. Адрес сам по себе — НЕ маркер смешения брендов.
4. Derived-числа: если в метаданных хода гейт уже дал finding (`derived_product_number`/`fact_grounding` с downgrade_keep_text) — это сработавшая страховка, судья отмечает PWN-нотой, не hard-FAIL (текст не уходит клиенту без менеджера).

Версия: `JUDGE_PROMPT_VERSION` → `judge_v9_1_*`; normalize принимает старый алиас; старые прогоны НЕ пере-судить; сравнение прогонов — только внутри одной версии (правило CLAUDE.md). NEG: «свяжется завтра утром» → по-прежнему made_a_promise; реальный бренд-токен чужого бренда → по-прежнему brand_leak.

## Часть 2. Повтор-режим (replay) для пар

Новый режим раннера: `--replay-from <путь к dynamic_dialog_transcripts.jsonl>` — клиентские реплики проигрываются ДОСЛОВНО из прошлого прогона (LLM-клиент не вызывается), бот отвечает заново текущей конфигурацией, судья судит новые ответы. Назначение: парные сравнения маршрутов/фактов/текстов при ровно одинаковом входе — шум клиентской стохастики = 0.

- Метаданные: `replay_source_run`, `replay=true` в summary; llm_calls роль client = 0.
- ЧЕСТНОЕ ОГРАНИЧЕНИЕ (в код и в README): replay валиден для сравнения «как бот отвечает на тот же вход»; НЕ валиден для диалоговой динамики (живой клиент реагировал бы иначе на новые ответы) — решения «включать/нет» по replay принимаются только для правок, не меняющих структуру диалога (промпт-строки, факты, гейты); для структурных (рубрика, ретривер) — обычная пара. Это правило записать в CLAUDE.md-дисциплину одной строкой при отчёте.
- NEG: replay того же дерева на том же снимке → маршруты/тексты воспроизводятся с точностью до модельной стохастики (прогнать дважды, сравнить агрегаты); обычный режим не задет (флаг отсутствует → поведение байт-в-байт).

## Приёмка

Полный pytest; календарь: часть 1 → часть 2; смоук: re-суд canary10 новым судьёй (10 диалогов, дёшево) + replay smoke18 от свежего прогона — оба разбора за архитектором; отчёт в _done. Если придут решения Дмитрия по r4 — r4 приоритетнее, это ТЗ прерываемо.
