# Идеи: данные, персонализация и контекст клиента
Агент 5 — аналитик данных и персонализация  
Дата: 2026-06-03

---

## 1. Бесшовное запоминание профиля из AMO

**Суть.** При входящем сообщении бот делает read-only запрос к AMO и вытаскивает поля сделки: класс ребёнка, предмет, формат (очно/онлайн), город. Эти данные кладутся в контекст планнера ещё до первого ответа. Бот не переспрашивает «а какой класс?» — он уже знает.

**Почему растит качество.** Каждый лишний уточняющий вопрос = потеря темпа диалога. Клиент, которому не задали очевидный вопрос, чувствует себя узнанным и отвечает теплее. Реальная разница в конверсии первого диалога по отрасли — 10–20%.

**Сложность:** низкая (AMO уже подключён, поля есть).  
**Нужны данные:** поля сделки: class_grade, subject, format_preference, city.

---

## 2. Лид-скоринг по горячести на входе

**Суть.** Перед тем как бот начинает отвечать, он вычисляет скор горячести лида из AMO: этап воронки, время с последнего контакта, количество диалогов, наличие оплаты. Скор определяет стратегию: горячий лид (этап «принятие решения», диалог 3+) — бот сразу к конкретному офферу; холодный — информационный режим с мягкой вовлечённостью.

**Почему растит конверсию.** Горячий лид, которому снова объясняют базовые вещи, уходит. Холодному лиду слишком агрессивный оффер — тоже стоп. Разный тон = меньше потерь на обоих концах.

**Сложность:** средняя (нужен scoring-модуль, но данные уже в AMO).  
**Нужны данные:** этап воронки, дата последнего контакта, счётчик диалогов, статус оплаты.

## 3. Сегментация: новый / действующий / бывший ученик

**Суть.** Бот определяет сегмент клиента по AMO: первый контакт — новый, активный договор — действующий, завершённые программы в прошлом — бывший. Каждому сегменту — разный тон и разные акценты. Новому — снятие риска и объяснение формата. Действующему — апсейл второго предмета или лагеря. Бывшему — возвращение с упором на динамику результатов.

**Почему растит конверсию.** Действующему клиенту не нужно объяснять, кто такой Фотон. Бывшему нужен другой крючок, чем новому. Одинаковый скрипт для всех — упущенные продажи.

**Сложность:** низкая (сегментация по одному-двум полям AMO).  
**Нужны данные:** история сделок, статус текущего договора, дата последней оплаты.

## 4. Детектор «невысказанной потребности» из истории переписки

**Суть.** Планнер анализирует последние 5–10 сообщений истории из AMO (до текущего диалога). Если клиент ранее спрашивал про второй предмет, но не оформил — бот сам упоминает это как релевантный вариант при первой подходящей реплике. Работает как «коллега, который читал предыдущую переписку».

**Почему растит качество.** Менеджеры физически не успевают перечитывать всю историю перед каждым диалогом. Бот делает это мгновенно и не даёт клиенту снова чувствовать себя незамеченным.

**Сложность:** средняя (нужен LLM-проход по истории AMO перед ответом).  
**Нужны данные:** текст переписки из AMO за последние 30 дней.

## 5. Автоматический выбор «следующего шага» по этапу воронки

**Суть.** Композер формирует черновик с явным следующим шагом, привязанным к этапу воронки из AMO. Этап «первичный контакт» → черновик заканчивается приглашением на пробное занятие. Этап «знакомство с продуктом» → черновик ведёт к расчёту рассрочки. Этап «принятие решения» → черновик предлагает зафиксировать место.

**Почему растит конверсию.** Менеджер видит готовый черновик уже с правильным CTA. Скорость одобрения и отправки растёт. Не нужно каждый раз думать, что писать дальше.

**Сложность:** низкая (правило маппинга этапа на CTA, данные уже есть).  
**Нужны данные:** этап воронки AMO (pipeline stage).

## 6. Персональный оффер: второй предмет на основе профиля

**Суть.** Если в AMO у клиента один предмет в сделке, бот при подходящем вопросе предлагает второй предмет со скидкой (Фотон 30% онлайн / УНПК 20%), опираясь на класс ребёнка и текущий предмет. Связка «у вас математика 8 класс — вот физика онлайн со скидкой 30%» готовится автоматически из фактов базы знаний.

**Почему растит конверсию.** Апсейл второго предмета — подтверждённый высокомаржинальный шаг. Предложить его в нужный момент с конкретной цифрой скидки = убрать барьер «надо уточнить».

**Сложность:** низкая (данные и факт скидки уже в базе).  
**Нужны данные:** текущий предмет и класс из AMO, факты скидок из KB по active_brand.

## 7. Мониторинг «где бот/менеджер теряет клиента»

**Суть.** После каждого диалога в лог пишется: на каком ходе клиент замолчал, какой тип реплики был последним (оффер / вопрос / ссылка / менеджер-хендофф). Еженедельный агрегат показывает топ точек обрыва. Если 40% клиентов замолкают после вопроса о рассрочке — значит ответ про рассрочку не устраивает.

**Почему растит качество.** Без этой аналитики улучшения делаются вслепую. С ней понятно, что именно чинить в следующей волне — конкретный тип реплики, а не «тон вообще».

**Сложность:** средняя (нужен лог диалогов + агрегирующий скрипт).  
**Нужны данные:** лог ходов бота, исход сделки из AMO (оплата / отказ / тишина).

---

## 8. Дозаполнение базы знаний из реальных вопросов (gap-детектор)

**Суть.** Планнер фиксирует каждый вопрос клиента, по которому retrieve_facts вернул пустой результат или low-confidence. Раз в неделю скрипт собирает топ-10 таких вопросов и отдаёт Дмитрию как кандидатов на новые факты в KB. Бот сам не добавляет — только сигнализирует.

**Почему растит качество.** Сейчас белые пятна базы знаний видны только случайно. Автоматический список пробелов = управляемый процесс обновления KB, а не «что вспомним».

**Сложность:** низкая (счётчик промахов retrieve уже можно добавить в лог планнера).  
**Нужны данные:** лог вызовов retrieve_facts с флагом found/not_found.

---

## 9. Персональное предложение лагеря на основе класса и истории

**Суть.** Если в AMO класс ребёнка 5–8 и история без ЛВШ — бот при разговоре о лете предлагает лагерь с конкретными датами смен из KB (с учётом active_brand). Если лагерь уже был — тон другой: «в прошлом году были — вот что изменилось в этом». Черновик готовится из фактов, не из воздуха.

**Почему растит конверсию.** Лагерь — отдельная статья дохода с высоким сезонным пиком. Персональное упоминание в нужный момент (лето приближается, класс подходящий) — прямой рост записей.

**Сложность:** средняя (нужна связка «история ЛВШ в AMO» + факты KB по active_brand).  
**Нужны данные:** история сделок по типу «лагерь», класс ребёнка, active_brand.

---

## 10. ПДн-барьер: персонализация без утечки данных клиента

**Суть.** Все данные из AMO (имя, класс, история) используются только внутри контекста планнера — они не попадают в текст черновика и не видны менеджеру в явном виде как JSON. Черновик использует данные опосредованно: «у вас 9 класс — значит, физика онлайн подходит» без цитирования полей AMO. Правило: персонализация = поведение бота, не цитата из карточки.

**Почему важно.** Нарушение этого барьера — юридический риск (ПДн) и потеря доверия клиента, который не давал согласия на демонстрацию его данных. Правило встроенное, не зависит от внимания менеджера.

**Сложность:** низкая (архитектурное правило, не новый код — один запрет в шаблоне промпта).  
**Нужны данные:** не нужны дополнительные — нужен запрет на прямое цитирование полей AMO в черновике.
