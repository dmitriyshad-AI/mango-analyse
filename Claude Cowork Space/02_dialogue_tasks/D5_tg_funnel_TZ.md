# D5 — Телеграм-воронка: задача (путь до продажи как ПРИНЦИПЫ + примеры, не код-ветки)

Зона файлов: `product_data/` (воронка/набор приёмов), документы воронки, набор живых примеров (как ДАННЫЕ).
Не трогать `channels/` (поведение проектируем, код пишет главный диалог). Не заводить свой список критичных тем.

## Цель
Спроектировать путь нового лида до заявки/продажи так, чтобы он рождался В ОТВЕТЕ модели (контракт + факты +
живые примеры), а НЕ через новые код-ветки и шаблоны на каждую тему. Иначе вернётся «лечение по одному случаю».

## Что собрать (как принципы + примеры, которые потом подключит главный диалог)
1. **Карта пути лида**: этапы (первый контакт → понять задачу → польза и снятие тревоги → один шаг → запись/
   передача) как принципы и признаки перехода, не жёсткий скрипт.
2. **Матрица уточняющих вопросов** (1–2 по делу, не анкета): какой вопрос в какой ситуации и зачем (вопрос про
   реальную ситуацию, не наводящий «вам же интересно»).
3. **Набор приёмов на возражения** (дорого / не уверены, подойдёт / нет времени / хотим попробовать / страх
   онлайна / сравнение с конкурентом / хочу человека / подумаю / гарантии): причина → что сказать → чего нельзя
   → один шаг → ограничения по бренду. Формулировки — из тёплого набора и звонков, не выдумывать.
4. **Этичное влияние** (по Чалдини, только честное): польза → доверие → маленький шаг → менеджер на действие.
   Жёсткие запреты: искусственный дефицит, давление виной, недоказанное «все выбирают», недоказанный авторитет,
   просьба контакта до пользы. Срочность — только честная и не в каждом ответе.

## Машинно-читаемый вид (замечание Кодекса №6 — обязательно)
Отдавать НЕ только текстовую карту, а файл, который главный диалог сможет подключить к ответу модели.
Каждый элемент — по схеме:
```yaml
- brand: foton|unpk|any      # для какого бренда
  topic: <тема/класс ситуации>
  situation: <когда применять>
  example: <живой пример ответа, из тёплого набора/звонков, не выдумка>
  forbidden: [<что нельзя на этой теме>]
  safe_next_step: <один безопасный следующий шаг>
  requires_fact_ids: [<факты, без которых пример не подставлять>]
  source: <звонки/тёплый набор/воронка>
  dedup_ref: <ссылка на уже существующий элемент, если это повтор>
```
Текстовая карта и принципы — отдельно (для людей); бот потребляет машинный файл.

## Жёсткие рамки
- Книги/звонки — это каркас и примеры, НЕ источник фактов; цены/даты/скидки — только из базы знаний.
- Один бренд в ответе; политика «ты бот?» — честно по запросу (вариант C).
- Итог диалога — файлы `product_data/bot_sales_playbook_*` (по схеме выше) + живые примеры, которые главный
  диалог подключает по одной теме, под флагом, с замером на закрытой проверке. В `channels/` ничего не писать.
- У каждого элемента — `source`/`dedup_ref`, чтобы не плодить третий источник правды (есть тёплый набор Claude
  `03_corpus/` + звонки + воронка).

## Связка
Файлы и примеры вливаются в главный диалог ПОСЛЕ базового замера качества (чтобы не смешать причины). Повторы
с уже готовым набором Claude (`03_corpus/`) и приёмами из звонков — переиспользовать, не переписывать.

## Приёмка
Карта пути + матрица вопросов + набор приёмов на возражения + правила этичного влияния — как принципы и как
машинный файл по схеме; жёсткие запреты на манипуляции заданы; формулировки из набора/звонков, не выдуманы;
в `channels/` ничего не закоммичено; готово к поэтапному подключению главным диалогом с замером.
