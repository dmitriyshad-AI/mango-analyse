# ТЗ MAIN (Кодекс) — формат-гард цены: не подставлять цену чужого формата (очно↔онлайн). 2026-05-31.

Автор: Клод 1. Точки подтверждены чтением HEAD aef24d36 (`dialogue_contract_pipeline.py`). Узкая правка
слоя безопасности фактов, отдельный коммит. Закрывает 3 из 4 выдумок стабильности-212.

## Проблема (из регрейда stability-212, по сырью)

3 реальные выдумки одного корня — **путаница формата цены очно/онлайн**:
- `foton_E1_price_04` — клиент прямо «математика **очно** 6 класс сколько?», бот выдал **онлайн**-цены
  (44 600 / 74 500).
- `unpk_C2_multi_subj_02` — формат не указан, бот по умолчанию подставил онлайн-цену УНПК (41 800).
- `foton_C2_multi_subj_04` — онлайн-цена (44 600) + очная скидка 20% в одном ответе, несогласованно.

Числовой verifier их НЕ ловит: 44 600 / 41 800 — настоящие числа из фактов (`fact_grounding` зелёный),
просто факт ДРУГОГО формата. Формат не сверяется.

## Корень (доказано чтением)

`_price_for_composition` (строка ~1894) и `_second_subject_discount_pct` (~1926) скорят факты:
`+2`, если формат факта совпал с `preferred_format` из вопроса. НО при отсутствии факта нужного формата
функция всё равно берёт кандидат с наибольшим score — то есть **факт противоположного формата (score 0)**.
Возврата None при «нет факта нужного формата» нет. Итог — цена чужого формата уходит в ответ.

Инфраструктура формата уже есть: anchors `format:online`/`format:offline` (702-703), `preferred_format`
из `_contract_intent_text` (1896-1898), `_compose_price_plus_format` (1844), `_augment_with_format_guidance`
(956). Достраиваем поведение, новых сущностей не вводим.

## Решение — два места (композер + verifier), оба нужны

Цена чужого формата может прийти ДВУМЯ путями: детерминированный композер ИЛИ модельный draft.
Поэтому правка в обоих местах.

### Правка 1 — композеры не падают на чужой формат

В `_price_for_composition` (~1894) и `_second_subject_discount_pct` (~1926): если `preferred_format`
задан (клиент указал очно/онлайн), но среди кандидатов НЕТ ни одного, чей формат совпадает с
`preferred_format` → вернуть `None` (а не лучший из чужого формата). Тогда композер не соберёт
цену → ход уйдёт по обычному пути «нет факта» (уточнение/менеджер), не выдаст чужой формат.

Ориентир (адаптируй под код, подтверди чтением):
```python
    # после сбора candidates, ПЕРЕД выбором лучшего:
    if preferred_format and not any(
        _candidate_matches_format(value, preferred_format) for _, value in raw_candidates
    ):
        return None
```
(`_candidate_matches_format` — та же regex-логика, что уже в цикле: online → `онлайн|online`,
offline → `очно|очная|очный|offline`. Можно вынести в маленький хелпер или оставить инлайн —
правило «простота вперёд», не плодить абстракции.)

ВАЖНО: если `preferred_format` пуст (клиент НЕ указал формат) — поведение НЕ менять (как сейчас).
Это негативный контроль №2 ниже.

### Правка 2 — verifier ловит чужой формат в готовом тексте (модельный draft)

В `_wrong_intent_fact_findings` (2476) добавить проверку формата цены — по образцу существующих
(`_draft_uses_contact_hours_as_schedule`, `_draft_uses_address_fact`):
```python
    fmt = _requested_price_format(contract)   # "online" | "offline" | "" (из _contract_intent_text)
    if fmt and _draft_price_uses_opposite_format(draft, facts=facts, requested=fmt):
        findings.append(VerificationFinding(
            "wrong_intent_fact",
            "Цена другого формата (очно/онлайн), чем спросил клиент."))
```
`_draft_price_uses_opposite_format`: взять денежные числа из draft (`_numbers`/`_first_money_amount`);
для каждого найти факт(ы)-источник в `facts`, содержащие это число; если ВСЕ такие факты несут
формат, противоположный `requested` (и ни один — нужный) → True. Если число есть в факте нужного
формата — False (не флагать). Формат факта — той же regex (`format:online`/`format:offline` anchors).

Финдинг `wrong_intent_fact` уже заводит repair-петлю → при неуспехе хендофф. После Правки 1 композер
сам не подставит, а Правка 2 страхует модельный путь.

## Тесты + НЕГАТИВНЫЙ контроль (обязательно — без него защитная правка не закрыта)

- ПОЗИТИВ: вопрос «**очно** 6 класс цена», в фактах только онлайн-цена → композер None, draft с онлайн-
  ценой ловится verifier'ом → repair/менеджер, НЕ выдаёт онлайн как ответ. (воспроизводит E1_price_04)
- ПОЗИТИВ: 2 предмета, клиент сказал «очно», в фактах только онлайн-скидка → не подставлять онлайн-скидку.
  (C2_multi_subj_04)
- НЕГАТИВ №1 (критично — не ослабить): клиент указал «онлайн», в фактах ЕСТЬ онлайн-цена → отвечает
  как сейчас (формат совпал, флага нет, ответ идёт).
- НЕГАТИВ №2 (критично): клиент формат НЕ указал (`preferred_format`=="") → поведение прежнее, флага нет,
  цена отдаётся (иначе сломаем массовые обычные ценовые вопросы без указания формата). C2_multi_subj_02
  закрываем НЕ через флаг (формат не указан) — см. ниже.
- НЕГАТИВ №3: число цены совпало случайно (есть в факте нужного формата) → не флагать.

### По `unpk_C2_multi_subj_02` (формат не указан) — отдельно, мягче
Там клиент формат не назвал, бот выбрал онлайн по умолчанию. Жёстко флагать нельзя (сломает обычные
вопросы). Решение мягче: когда формат не указан И есть факты ОБОИХ форматов — draft-промпт должен либо
назвать оба («очно X, онлайн Y»), либо коротко уточнить формат, а не молча выбирать один. Это правка
ПРОМПТА, не гард. Если в наборе фактов формат один — отвечать им можно. Вынести в Правку 3 (промпт),
делать ПОСЛЕ 1+2, мерить отдельно. В этот коммит НЕ включать.
