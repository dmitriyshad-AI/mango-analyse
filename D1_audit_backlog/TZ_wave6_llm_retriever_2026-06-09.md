# ТЗ — Волна 6: LLM-ретривер фактов (вариант A) — замена keyword-понимания

**Статус:** ФИНАЛ (аудит второго архитектора, 2 круга).
**Дата:** 2026-06-09. **Дерево:** новая ветка `codex/wave6-llm-retriever` от main. В main не мержить до замера+регрейда.
**Основание:** находка №8 аудита + регрейд Волны 4/5 — keyword-отбор фактов режет вход (модель не видит нужный факт → уходит). Подтверждено по коду: `_direct_path_wide_fact_pack` (2425) сам вызывает `_direct_path_selected_categories` (keyword) → даже широкий пак фильтруется keyword.

## Зачем (корень)
Релевантность факта вопросу — задача ПОНИМАНИЯ (рассуждение: «оплатил → нужен факт о следующем шаге», хотя слова нет). Сейчас её решает keyword (категории + keyword-relevance-score) → лазанья на входе, нарушение ADR-001 («модель понимает, детерминизм проверяет выход»). Заменяем keyword-отбор на LLM-выбор фактов (вариант A — отдельный лёгкий вызов). Это первый шаг; D (интеграция в основной вызов + эмбеддинг-сужение) — апгрейд позже, вне scope.

## Несущий принцип (граница, нарушать нельзя)
- **LLM выбирает РЕЛЕВАНТНОСТЬ** (понимание) — какие факты относятся к вопросу.
- **Детерминизм держит БЕЗОПАСНОСТЬ:** фильтр по `active_brand` + `client_safe` происходит ДО LLM-ретрива (модель-ретривер видит ТОЛЬКО факты своего бренда — она физически не может выбрать чужой). Бренд из канала, не угадывается. Выходной гейт (бренд-токены/P0/число/ПДн) — без изменений.
- **Fail-soft:** ретрив-вызов упал/таймаут/пустой → откат на текущий keyword-отбор (не падать, не уходить).
- Модель-ретривер НЕ решает «отвечать/уйти» и не генерит клиентский текст — только выбирает id релевантных фактов.

## Правки (за флагом `TELEGRAM_LLM_RETRIEVE`, default OFF)

**П1. LLM-ретривер фактов.** Якоря: `_direct_path_selected_categories` (~2276), `_direct_path_wide_fact_pack` (~2425), сборка exact/adjacent блоков (~2783).
- Кандидаты: ВСЕ `client_safe` факты `active_brand` (как сейчас отбираются ДО категорийной фильтрации — фильтр бренда/client_safe остаётся детерминированным).
- Лёгкий вызов модели (low reasoning): подать список кандидатов (fact_id + краткая строка-суть, БЕЗ полных текстов для экономии) + вопрос клиента + краткий контекст нити → вернуть id релевантных фактов (exact — прямо отвечают; adjacent — смежные).
- Полные тексты выбранных id → в `exact_block`/`adjacent_block` основного промта (как сейчас, но отбор не keyword, а LLM).
- Промт ретривера: «выбери ВСЕ факты, которые могут помочь ответить (в т.ч. по смыслу/следующему шагу), не только дословные; не выдумывай id вне списка».

**П2. Fail-soft на keyword.** Якорь: тот же отбор. Если ретрив-вызов: rc≠0 / таймаут / пустой / вернул невалидные id → откат на текущий `_direct_path_selected_categories` + keyword-score (поведение main). Метка в metadata `llm_retrieve_fallback=<причина>`. Не уходить к менеджеру из-за сбоя ретрива.

**П3. Бренд-изоляция (безопасность, детерминизм).** Кандидаты для ретривера формируются ТОЛЬКО из `active_brand` + `client_safe` ДО вызова. Модель-ретривер не видит и не может выбрать чужой бренд. NEG обязателен. Выходной бренд-гейт без изменений (вторая страховка).

**П4. Счётчик и наблюдаемость.** Якорь: summary раннера (llm_calls_total). Ретрив добавляет +1 вызов на ход — отразить в `llm_calls_total` + разбивке по ролям (роль `retriever`). В metadata хода: выбранные id, fallback-флаг.

## Соотношение с Волной 5
- A (Волна 6) ЗАМЕЩАЕТ keyword-категории (П3 Волны 5) — в проде вместе их не держим (A — настоящее решение, П3 — костыль).
- A СОВМЕСТИМ с П1 анти-promise и П4 повышение Волны 5 (другие слои: генерация/маршрут, не отбор фактов). При замере Волны 6 базовый стек = Волна 5-принятое (если принято) + LLM_RETRIEVE.

## Граница безопасности (НЕ трогать)
Бренд-фильтр входа (детерминизм), P0-латч/преблок, выдумка числа (absent), ПДн/мета, выходной гейт — без изменений. LLM-ретривер не имеет доступа к маршруту и к чужому бренду.

## NEG (обязательны)
1. **OFF-паритет:** флаг OFF → отбор = keyword (main), байт-в-байт.
2. **Релевантность:** вопрос «оплатила, что дальше» → ретривер выбирает enrollment/process-факты (которые keyword пропускал из-за категории pricing). Составной вопрос → факты по обеим темам.
3. **Бренд-изоляция:** active_brand=foton → кандидаты и выбор только foton; чужой бренд не выбран (даже если ретривер «захочет»). И наоборот для unpk.
4. **Fail-soft:** ретрив rc≠0/таймаут/пустой/мусорные id → откат на keyword, ответ строится, маршрут не меняется, metadata fallback.
5. **Анти-выдумка id:** ретривер вернул id вне списка кандидатов → отброшены (не подставляются).
6. **Безопасность сквозная:** P0/бренд/число/ПДн — 0 регресса (выходной гейт ловит как раньше).

## Замер (М1) — ПОСЛЕ замера Волны 5
Базлайн = принятый стек Волны 5 (или wave4_base, если Волна 5 не прошла). Прогон + LLM_RETRIEVE.
**Метрика:**
- `wrong_intent_fact` ВНИЗ (модель лучше выбирает факт под интент) — главное.
- факт-под-вопрос найден чаще → меньше уходов «не увидел факт».
- ЖЁСТКО 0: бренд-смешение / P0 / выдумка числа / ПДн (регрейд по сырью + NEG).
- llm_calls_total вырос (норма, +1 на ход) — проверить, что в пределах ожидаемого.
**Стохастика:** LLM-ретрив добавляет вариативность. На этот шаг допустимо >1 прогона для уверенности (исключение из правила «1 прогон» — согласовать с Дмитрием), либо читать результат как направленческий.
Регрейд по сырью: реально ли ретривер выбирает релевантнее keyword (примеры, где keyword мазал, а LLM попал), и нет ли новых wrong-scope.

## Вне scope
Вариант D (интеграция в основной вызов + эмбеддинг-сужение) — апгрейд после A; снос keyword-кода (пока за флагом, оставляем как fallback); реальная автоотправка.

## Аудит второго архитектора (внесён)
- **Р1 «бренд при LLM-ретриве»:** кандидаты только active_brand+client_safe ДО вызова; ретривер не видит чужой бренд; выходной гейт — вторая страховка. NEG-3. Принято.
- **Р1 «ретривер пропустит/выдумает факт»:** fail-soft на keyword (П2); анти-выдумка id (только из списка, NEG-5); промт «выбери все потенциально релевантные»; verifier выхода ловит выдумку. Принято.
- **Р2 «стоимость/латентность +1 вызов»:** low-reasoning, дёшево; для черновика/ночного автонома ок; счётчик в summary (П4). Принято.
- **Р2 «стохастика бьёт по воспроизводимости замеров»:** закладываем; флаг для A/B; на этот шаг допустимо >1 прогона (согласовать). Принято.
- **Р2 «конфликт с Волной 5 П3»:** A замещает keyword-категории (П3); совместим с П1/П4; в проде не держим обе. Принято.
- **Р3 «когда замерять»:** строго после замера Волны 5 (базлайн + не смешивать эффекты). ТЗ пишем/реализуем сейчас, замер позже. Принято.
- **Р3 «D апгрейд»:** вне scope; A — первый шаг. Принято.
