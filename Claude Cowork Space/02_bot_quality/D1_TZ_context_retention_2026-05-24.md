# ТЗ для D1 — удержание контекста через заземлённую генерацию + 2 корневых класса

Утверждено Дмитрием 2026-05-24. Это следующий корневой слой поверх B→A→C. Цель — чинить механизмами, не по
кейсам/словам. Закрывает «потерю контекста после 2-го сообщения» И текущую серию dev-провалов (подстрочные
«кроты» + перезапись правильного ответа шаблоном).

## Рамка

Качество — из ГЕНЕРАЦИИ; правила конвейера — ИНСТРУКЦИЕЙ в промпте; гейты — тонкая страховка, которая ВЫЧИТАЕТ
(блок/увод), а не переписывает. Анти-«крот»: число подстрочных проверок и точечных шаблонов должно ПАДАТЬ —
общие правила заменяют частные.

## Решение 1 — Удержание контекста: структура промпта генерации

На каждом ходу генерация получает (в этом порядке):
1. **Инструкция (стабильный системный слой):** глобальный план диалога (понять цель → ответить по сути → один
   следующий шаг) + жёсткие правила как guidance: не смешивать бренд, не выдумывать цену/число, отвечать только
   подтверждёнными фактами, P0 → менеджер, политика «ты бот?».
2. **Подтверждённые факты ПО ТЕМЕ (scope-filtered)** — чтобы было чем отвечать; числа берутся только отсюда.
3. **Шапка состояния (append-only):** зафиксированные класс/предмет/формат, открытый вопрос, что уже отвечено,
   поправки клиента, что обещано/текущий маршрут.
4. **Полная история сообщений** — дословно, вся доступная (кап ~20 последних).
5. **Задача:** «ответь на ПОСЛЕДНЕЕ сообщение, держа весь диалог и состояние перед глазами; трактуй последнюю
   реплику в свете состояния, а не в отрыве».

Состояние только ДОПОЛНЯЕТСЯ, не пересобирается из последней реплики. Слот, раз названный, не сбрасывается до
явной смены клиентом. Это и есть «как у больших чат-моделей» (видят весь диалог), но с заземлением на факты.

## Решение 2 — Гейты: детерминированный + LLM сверху, ОБА вычитают

- **Детерминированный тонкий гейт (100%, мгновенно, тестируемо):** литералы чужого бренда по active_brand
  (имена/телефоны/домены/Долями/Т-Банк), число не привязанное к подтверждённому факту, служебные маркеры
  (fact_id/JSON/вендор), соблюдение P0-маршрута. Срабатывает → блок + увод к менеджеру.
- **LLM-gate СВЕРХУ (на смысловое, чего регулярка не видит):** скрытое обещание/гарантия, смешение брендов по
  смыслу, «ответил не на то», выдуманно звучащий факт, тон. Срабатывает → увод к менеджеру.
- **Оба ВЫЧИТАЮТ/блокируют, НЕ переписывают в заглушку.** Для экономии — LLM-gate гонять, когда детерминированный
  чист, или на рискованных ходах.

## Решение 3 — Бренд: знаем обе компании, в клиентский контекст — только активную

- Система хранит/связывает полный кросс-бренд профиль клиента (его видит МЕНЕДЖЕР).
- В КЛИЕНТСКУЮ генерацию активного бренда подаём ТОЛЬКО историю активного бренда + нейтральную пометку «клиент
  есть и в другом бренде → при вопросе о связи → менеджер». Без личных данных/ID в клиентском тексте. На первом
  шаге история клиента — контекст для ЧЕРНОВИКА менеджеру (timeline уровень 1).
- Причина: мягкая инструкция «не упоминай другой бренд» работает ~99%, не 100%; если чужая история в контексте —
  бот её однажды проговорит, а брендовая утечка — класс, где даже редкое недопустимо.

## Корневой класс A — распознавание по СЛОВАМ/токенам, НЕ по подстроке

Наблюдаемые dev-баги (это ОДИН класс, не три кейса): «очно» внутри «тОЧНОй даты» → формат «очно»; «дн» внутри
«многоДетНой» → шаблон расписания; процент скидки принят за вопрос о цене; «вторым предметом» не распознан.
**Чинить раз:** распознавание формата/темы/намерения по границам слов/токенам (и по fact_key/title/product, как
уже начали для scope), а НЕ по вхождению подстроки (`in`). Регрессия: набор слов-ловушек (точной, многодетной,
информатика, рассрочка и т.п.) не должны триггерить чужой интент/формат.

## Корневой класс B — НЕприкосновенность более точного ответа

Наблюдаемые dev-баги (тоже ОДИН класс): финальный/пробный шаблон затирает уже корректный handoff; rewriter
переписывает правильный прямой ответ в длинный генерик; короткие реплики («спасибо», «подумаю») получают
дублирующий шаблон. **Правило:** если уже сформирован прямой ответ по сути / явный handoff / следующий шаг —
шаблон и rewriter могут только УТОЧНИТЬ стиль, но НЕ заменять генериком и НЕ дублировать на короткие реплики.
Шаблон — fallback ТОЛЬКО когда содержательного ответа нет. (Это прямое следствие принципа «гейты/слои вычитают,
не переписывают».)

## Корневой класс C — приоритет ЯВНОЙ поправки клиента над прежним контекстом

Наблюдаемый dev-баг (лагерь): клиент говорит «я как раз про выездной», а бот держится прежней городской ветки.
Это ОДИН класс, не только лагерь — тот же эффект для онлайн/очно, предмета, продукта. **Правило приоритета:**
текущая ЯВНАЯ реплика клиента > удержанный контекст > предположение бота. Held-состояние нужно, чтобы НЕ
переспрашивать и не терять нить, НО явная поправка/уточнение («я как раз про выездной», «я же про онлайн», «не про
многодетную, а про второй предмет») ОБНОВЛЯЕТ слот, и последнее явное утверждение клиента побеждает и прежний
контекст, и прежнее предположение. Клиент — источник правды о своей потребности. Чинить общим приоритетом, а не
патчем под лагерь. (Это уточнение к Решению 1: «append-only» означает «не забывать и не переспрашивать», а НЕ
«игнорировать поправку клиента».)

## Процесс и приёмка

- Классы A и B чинить как механизмы (не по словам/кейсам). Анти-«крот»: подстрочных проверок и точечных шаблонов
  должно стать МЕНЬШЕ.
- **«ЗАМЕНЯТЬ, а не ДОБАВЛЯТЬ» — главный риск реализации (назван Codex, подтверждаю).** Смысловой распознаватель и
  шапка состояния должны ЗАМЕНИТЬ старую частную логику, а не лечь поверх неё. Старые наивные подстрочные `if` по
  словам-ловушкам (очно/дн/процент/запись и т.п.) и точечные шаблоны-перезаписи — УДАЛИТЬ, не оставлять «спящими».
  Измеримая приёмка: число (а) подстрочных проверок, (б) частных `if` распознавания, (в) точечных шаблонов/перезаписей
  в диффе должно УПАСТЬ, а старые ветки по этим словам — отсутствовать. **Claude проверит это по диффу на раунде-3:**
  если общий слой добавлен, но старые ветки на месте — это не «заменил», а «нарастил», и сложность снова выросла.
- **Строить как ОДИН переработанный путь ответа,** а не 3 новых прохода поверх старых: генерация видит
  правила + факты по теме + состояние + историю + последнее сообщение; распознавание по смыслу — часть интент-плана;
  пост-слои (гейты/шаблоны) только вычитают. Это и есть техническое определение «заменить, а не добавить».
- Dev-набор + targeted16 — отладка; чистый holdout не трогать.
