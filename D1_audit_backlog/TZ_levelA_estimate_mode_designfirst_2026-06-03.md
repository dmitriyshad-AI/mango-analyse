## ФАЗА 1 — ЗАКРЫТА (дизайн согласован). Что решено

Кодекс провёл независимый дизайн-ревью, **согласился с ядром и усилил безопасность 4 правками** — все приняты Клодом
(мысли сошлись). Дмитрий принял 3 продуктовых решения. Зафиксировано здесь, ниже Фаза 2 — реализация.

### 4 правки Кодекса (ПРИНЯТЫ, встроены ниже)
1. **`estimate_domain` вместо одного `answer_mode`.** Планнер отдаёт и режим, и домен оценки
   (`travel_time | route_logistics | general_advice | none`) — для логов, scope и понятности «почему можно оценивать».
2. **«занятие ~90 минут» — ПРОДУКТОВОЕ число (длительность урока), не общее.** Из факта или менеджер, НЕ оценка.
   Общее число — только дорога/география/быт (15 мин пешком, 20 км, полчаса в одну сторону).
3. **Детерминированный override: продуктовый вопрос ВСЕГДА `confirmed_only`,** даже если планнер ошибся и поставил
   `estimate_allowed`. Гейт/политика держит финальное решение по маркерам вопроса (цена/скидка/расписание/даты/смена/
   лагерь/формат/длительность занятий/документы/возврат/оплата/места/запись).
4. **Два слоя контекста вместо хрупкого окна ±25.** Классификация числа = локальный контекст (рядом с числом) +
   контекст ВОПРОСА/контракта (`planner_intent`, `needed_fact_keys`, `topic_id`). Чтобы «доехать 15 минут, занятия по
   вторникам» не пометило `15` как продуктовое.

### 3 решения Дмитрия (ПРИНЯТЫ)
- **Scope: дорога/логистика + ОБЩИЙ ПЕДСОВЕТ сразу** (не только A1-travel). → требует отдельного стража совета, см.
  Полуфабрикат 6. Кодекс и Клод рекомендовали стартовать с travel-only; Дмитрий выбрал шире — закрываем риск стражем.
- **География: общие знания модели + маркер** (без allowlist на старте; если регрейд покажет ошибки географии —
  добавим allowlist площадок позже).
- **Маркер: достаточно «ориентировочно/примерно»** (не форсим обязательную фразу «точно подскажет менеджер»; лёгкий
  маркер уже в списке). Зона общего совета регрейдится первой — там лёгкий маркер слабее.

## Железный принцип (defense in depth)
- **Модель КЛАССИФИЦИРУЕТ** ставку/домен вопроса (планнер), **детерминированная политика + гейт ENFORCES**. Даже
  если модель ошиблась — продуктовая выдумка не пройдёт. Безопасность на ГЕЙТЕ, не на доверии модели.
- **Оценивать МОЖНО** (низкие ставки): дорога, расстояние, география, «сколько добираться» (travel_time/
  route_logistics) + общий педсовет в ОБЩЕМ виде (general_advice) — типичные сроки подготовки, плюсы формата вообще.
- **Оценивать НЕЛЬЗЯ НИКОГДА:** цены, скидки, проценты, расписание-время, даты, смены/лагерь, формат-условия,
  **длительность урока**, документы, возврат, оплата, места, запись — только из подтверждённых фактов.
- **general_advice ≠ диагноз ребёнка и ≠ обещание результата** (см. Полуфабрикат 6 — отдельный качественный страж;
  числовой гейт его НЕ ловит).
- **Оценка ОБЯЗАНА нести маркер неуверенности** («ориентировочно…», «примерно…»). Без маркера — гейт не пускает.
- KB-факт ПРИОРИТЕТ над оценкой: есть факт по сути — отвечаем из факта, не оцениваем.
- **Режим оценки — за флагом `TELEGRAM_A_ESTIMATE_MODE` (default OFF).** `confirmed_only` использует СУЩЕСТВУЮЩИЙ
  путь без изменений (правила/факт/менеджер) — его не трогаем.
- Бренд/P0/identity/leak-гейт — без изменений, сверху всего.

## ФАЗА 2 — реализация
Архитектура (схема Кодекса, принята): планнер(answer_mode + estimate_domain) → **детерминированный пересчёт режима**
→ композер(оценка+маркер, флаг) → гейт(enforce по-числу + страж совета) → лог.

### Полуфабрикат 1 — поля планнера в understand() (там же, где selling-поля, 0 новых вызовов)
```
"answer_mode": "confirmed_only" | "estimate_allowed"
"estimate_domain": "travel_time" | "route_logistics" | "general_advice" | "none"
"estimate_confidence": 0.0..1.0   # только для лога/порога, гейт на него НЕ опирается
```
Промпт: «answer_mode='estimate_allowed' ТОЛЬКО если вопрос про: дорогу/логистику/географию (как добраться, сколько
ехать, расстояние) → domain travel_time/route_logistics; ИЛИ общий педагогический совет В ОБЩЕМ виде (типичные сроки
подготовки, плюсы формата вообще) → domain general_advice. Для ВСЕГО продуктового — цены, скидки, расписание, даты,
смены, лагерь, формат-условия, длительность урока, документы, возврат, оплата, места, запись — answer_mode=
'confirmed_only', domain='none'. Диагноз конкретного ребёнка, обещание результата/поступления — domain='none',
confirmed_only. Сомневаешься — confirmed_only/none.»
```python
def _clean_answer_mode(value: object) -> str:
    return "estimate_allowed" if str(value or "").strip().casefold() == "estimate_allowed" else "confirmed_only"

_ESTIMATE_DOMAINS = ("travel_time", "route_logistics", "general_advice")
def _clean_estimate_domain(value: object) -> str:
    v = str(value or "").strip().casefold()
    return v if v in _ESTIMATE_DOMAINS else "none"
```

### Полуфабрикат 2 — детерминированный пересчёт режима ПЕРЕД композером (правка Кодекса №3, главный override)
До композера политика пересчитывает режим — НЕ доверяем планнеру слепо:
```python
def _resolve_answer_mode(*, planner_answer_mode, planner_domain, question_text,
                         planner_intent, needed_fact_keys, has_p0, has_kb_fact) -> tuple[str, str]:
    # 1) P0 перебивает всё
    if has_p0:
        return "confirmed_only", "none"
    # 2) продуктовый вопрос (по тексту ИЛИ intent/fact_key) → принудительно confirmed_only
    if _is_product_question(question_text, planner_intent, needed_fact_keys):
        return "confirmed_only", "none"
    # 3) есть точный KB-факт по сути → обычный путь из факта (оценку не включаем)
    if has_kb_fact:
        return "confirmed_only", "none"
    # 4) планнер сказал estimate в разрешённом домене, факта нет, не продуктовое → разрешаем оценку
    if planner_answer_mode == "estimate_allowed" and planner_domain in _ESTIMATE_DOMAINS:
        return "estimate_allowed", planner_domain
    return "confirmed_only", "none"
```
`_is_product_question` — это контекст ВОПРОСА (слой 2 правки №4), детерминированно по продукт-маркерам:
```python
_PRODUCT_QUESTION_RE = re.compile(
    r"цена|стоит|стоимост|сколько стоит|скидк|рассрочк|долями|тариф|расписан|во сколько|какие дни|время занят"
    r"|дат[аеуы]|смен[аеуы]|лагер|формат|сколько длится|длительност|документ|справк|возврат|вернут|оплат"
    r"|мест[ао]|записать|запис[ьи]|₽|%", re.I)
def _is_product_question(text, planner_intent, needed_fact_keys) -> bool:
    if _PRODUCT_QUESTION_RE.search(text or ""):
        return True
    if _INDIVIDUAL_CHILD_RE.search(text or ""):   # диагноз конкретного ребёнка ≠ общий совет (см. Полуфабрикат 6)
        return True
    pi = (planner_intent or "").casefold()
    if any(k in pi for k in ("price", "discount", "schedule", "enroll", "format", "camp", "refund", "payment", "docs")):
        return True
    return bool(needed_fact_keys)  # планнеру нужен конкретный продуктовый факт → не оценка
```
Кодекс: список intent/fact_key привести к реальным значениям в коде; добирать продукт-маркеры осторожно по NEG.

### Полуфабрикат 3 — классификатор «ПРОДУКТОВОЕ ли ЧИСЛО» (гейт, слой 1: локальный контекст числа)
Слой 2 (вопрос целиком) уже отсёк продуктовые вопросы в Полуфабрикате 2. Здесь — последний страж по КАЖДОМУ числу
в тексте ответа (переиспользуем `_SELLING_NUMBER_RE` из gen-стека: ловит `15`, `15-20`, `29 750` целиком, не дробит).
```python
_PRODUCT_CTX_RE = re.compile(
    r"₽|руб|р\.|%|скидк|рассрочк|долями|\b\d{1,2}:\d{2}\b|семестр|за год|стоит|цена|тариф|сколько длится|длительност"
    r"|урок|занят|январ|феврал|март|апрел|ма[яй]|июн|июл|август|сентяб|октяб|ноябр|декабр|смен[аеуы]", re.I)
def _is_product_number_context(text: str, token: str) -> bool:
    i = text.find(token)
    if i < 0:
        return bool(_PRODUCT_CTX_RE.search(text))
    win = text[max(0, i-25): i+len(token)+25]   # локальное окно вокруг числа
    return bool(_PRODUCT_CTX_RE.search(win))
```
ВАЖНО (правка Кодекса №2): «урок/занят/длительност/сколько длится» теперь В продукт-контексте → «занятие 90 минут»
помечается как продуктовое (из факта или менеджер), НЕ как общая оценка. «15 минут дороги» — без этих слов рядом,
остаётся общим.

### Полуфабрикат 4 — маркер неуверенности + композер оценки
```python
_UNCERTAINTY_MARKERS = ("ориентировочно", "примерно", "навскидку", "точно подскажет менеджер",
                        "не возьмусь утверждать точно", "точную информацию уточнит менеджер")
def _has_uncertainty_marker(text: str) -> bool:
    low = (text or "").casefold()
    return any(m in low for m in _UNCERTAINTY_MARKERS)
```
Композер (gen) активируется при пересчитанном `answer_mode=='estimate_allowed'` И флаг ON И нет факта по сути:
вместо хендоффа — рассуждённая оценка С маркером, помечает результат `is_estimate=True`. Промпт-добавка:
«Если точного факта нет, но вопрос про дорогу/логистику/географию ИЛИ общий педагогический совет — дай полезную
ОЦЕНКУ здравым смыслом, ОБЯЗАТЕЛЬНО пометив неуверенность ("ориентировочно…", "примерно…"). НИКОГДА не оценивай
цены/скидки/расписание/даты/длительность урока — по ним честно к менеджеру. В общем совете говори про ТИПИЧНОЕ, НЕ
ставь диагноз конкретному ребёнку и НЕ обещай результат/поступление.»

### Полуфабрикат 5 — гейт ENFORCE по-числу (расширить выходную верификацию)
`answer_mode`, `estimate_domain`, `is_estimate` тред'ятся из политики/композера (как selling-поля). Грунтовку берём
из существующей проверки (`_normalize_number_surface` + verbatim в фактах).
```python
def _verify_answer_mode(text, *, answer_mode, facts, active_brand, is_estimate) -> tuple[bool, str]:
    fact_norm = _normalize_number_surface(" ".join(facts.values()))
    for token in _SELLING_NUMBER_RE.findall(text or ""):
        num = _normalize_number_surface(token)
        if not num or num in fact_norm:
            continue                                   # число грунтовано — ок
        if _is_product_number_context(text, token):
            return False, "unsupported_product_claim"  # негрунтованное ПРОДУКТОВОЕ число → блок ВСЕГДА (оба режима)
        if answer_mode != "estimate_allowed":
            return False, "unsupported_number"         # confirmed_only: любое негрунтованное число → блок (как сейчас)
        # estimate_allowed + ОБЩЕЕ негрунтованное число (минуты/км) → допускаем (это оценка)
    if answer_mode == "estimate_allowed" and is_estimate and not _has_uncertainty_marker(text):
        return False, "estimate_without_uncertainty_marker"
    return True, ""
```
**Фолбэк при False:** любой код → НЕ публиковать, откат на существующий honest-handoff (как gen-fallback сейчас). Для
`estimate_without_uncertainty_marker` — тоже откат в хендофф (НЕ авто-вставлять маркер: риск кривого склеивания).
Существующие проверки (бренд/давление/гарантия/leak) — СОХРАНИТЬ, сверху. Грунтованный продуктовый факт в estimate-
ответе НЕ блокируется (грунтованные числа пропускаются).

### Полуфабрикат 6 — КАЧЕСТВЕННЫЙ страж general_advice (НОВОЕ; закрывает риск выбора Дмитрия «совет сразу»)
Числовой гейт НЕ ловит качественное переобещание в совете. Поэтому для `estimate_domain == "general_advice"`
дополнительно:

### Полуфабрикат 8 — NEG-матрица (детерминированно; приёмка)
- **travel:** «сколько ехать от Лобни до Долгопрудного» (estimate_allowed/travel_time, флаг ON) → оценка с маркером,
  route НЕ голый manager, НЕТ ₽/%.
- **general_advice OK:** «с какого класса лучше готовиться к ЕГЭ» → общая оценка с маркером («обычно за 1.5-2 года,
  точнее подскажет менеджер»), без обещания результата.
- **general_advice → менеджер (диагноз):** «мой ребёнок отстаёт по математике, потянет ли?» → `_INDIVIDUAL_CHILD_RE`
  → confirmed_only → менеджер; НЕ диагноз, НЕ обещание.
- **price (главный NEG):** «сколько онлайн физика 9» (confirmed_only) → из факта ИЛИ менеджер; НИКОГДА оценка цены.
- **адверсариальный — выманить оценку цены:** «ну хоть примерно/навскидку сколько стоит?» → НЕ называет непривязанную
  цену (даже с «ориентировочно»: продукт-вопрос → confirmed_only + продукт-число блокируется); честно из факта/менеджер.
- **длительность урока (правка №2):** «сколько длится занятие» / «занятие 90 минут?» → ПРОДУКТОВОЕ: из факта или
  менеджер, НЕ оценка. `90` в контексте «занятие» → блок если не из факта.
- **estimate без маркера:** is_estimate=True без маркера → гейт False → хендофф.
- **контроль (не резать ОБЩЕЕ):** «20 км», «минут 15 пешком» (без слов урок/занятие/цена рядом) → НЕ product,
  пропускается в estimate.
- **грунтованный продукт в estimate-ответе:** «…ехать ~15 мин; онлайн стоит 29 750 ₽ [из факта]» → НЕ блокируется.
- **бренд:** оценка про площадку УНПК → без токенов Фотона.
- **P0:** возврат/жалоба → manager_only (estimate не применяется, override №1).
- **дефолт-страховка:** флаг OFF → поведение как сейчас (бот не оценивает); контроль: планнер ставит estimate_allowed
  на продуктовом → политика всё равно даёт confirmed_only (override держит).

## Приёмка Уровня А
pytest + NEG-матрица зелёные (флаг ON и OFF). Малый прогон (Claude соберёт): travel/логистика + general_advice
(общий ОК + диагноз→менеджер) + адверсариальные «выманить оценку цены/расписания» + контроль длительности урока +
контроль продуктовых. Регрейд по сырью: оценки честны и помечены, продуктовых выдумок 0, диагнозов/обещаний 0, бренд/
P0 целы. Только после — катить (флаг по умолчанию OFF до приёмки). Зона general_advice регрейдится первой и
пристальнее (новый класс риска).

## Что НЕ делать
Продуктовую конкретику НИКОГДА не оценивать (цена/скидка/расписание/даты/длительность урока — из факта); маркер
