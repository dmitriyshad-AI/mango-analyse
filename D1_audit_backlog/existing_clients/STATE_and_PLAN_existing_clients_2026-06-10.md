# Трек «действующие клиенты»: состояние и глобальный план (2026-06-10)

Статус: v2 ФИНАЛ — аудит вторым архитектором пройден (PASS_WITH_NOTES), 7 обязательных правок внесены (§5).
Опора: INVENTORY_timeline_dealaware_2026-06-10.md (задача 1, аудит пройден) + разведка конвейера звонков (этот документ, §2).

## 0. Решение Дмитрия по брендам (10.06, фиксирую для трека)

Раньше бренды сотрудничали, история клиента известна ПОЛНОСТЬЮ для каждого бренда. Поэтому: хранилище профиля/истории — ОБЩЕЕ, бот по умолчанию знает всё; разделение брендов — это фильтр на ВЫХОДЕ (что можно сказать клиенту в текущем боте), не на хранении. В CRM — полная история клиента по обоим брендам.

Следствия: «блокер №1 — бренд» из инвентаризации ПЕРЕКВАЛИФИЦИРОВАН: 84% «неизвестного бренда» в базе v5 не блокируют профиль (хранилище общее, tenant_id один — теперь это соответствует политике). Бренд остаётся обязательным атрибутом на СОБЫТИИ (откуда пришло) и правилом выдачи. Каноничная фраза разделения и запрет сравнения условий в одном ответе — без изменений (это выходной фильтр, он уже детерминирован в боте). Внесение в CLAUDE.md — главным диалогом.

7. **Событийное обновление**: НЕТ — всё батчами-снимками. Журнал цикла черновиков — кандидат-источник для триггера «диалог затих», механизм проектировать.

## 2. Качество Resolve+Analyze и вердикт по перепрогону

**Что проверено** (моя выборка 300 случайных строк canonical-базы read-only, 10.06): выжимка есть у 100%; текст плотный, фактический, с датой/менеджером/предметами (пример в инвентаризации); явных выдумок в просмотренных примерах нет. Слабые места — СТРУКТУРНЫЕ поля: target_product пуст у 64% (включая несодержательные звонки), next_step пуст у 44%, budget пуст у 98%. Исторический аудит 1000 звонков (22.03): на содержательных звонках хвост маленький (product 51, next_step 13), summary_missing=0 — «Analyze пригоден для массовой витрины». Стратегический вердикт 13.05 (super_resolve_strategy_review): полный пересчёт отклонён, DO_SELECTIVE — блокер не качество расшифровок, а протухшие next_step.

**Вердикт по перепрогону на более сильной модели:**
- **Расшифровку и resolve НЕ перепрогонять**: score 97.7, дорого, и стратегический аудит уже отклонил; ASR-запуски запрещены без явного разрешения.
- **Analyze перепрогонять ВЫБОРОЧНО и только после замера**: (а) сначала A/B на 30-50 содержательных звонках — текущая gpt-5.4-mini против сильной модели (готовый скрипт run_analyze_ab_test.py), судья — независимый разбор по сырью; критерии ДО прогона (правка аудита): полнота target_product и next_step на содержательных, 0 выдумок в именах/числах, выжимка не хуже (плотность, без пересказа-дампа); budget — только вторичный сигнал, НЕ критерий (98% пустых — чаще реальность разговора, чем слабость модели). Заодно проверить вариант «улучшить промпт, а не только модель»: пустые структурные поля могут лечиться промптом v7. Выход A/B — не только «да/нет», а матрица покрытия полей по обеим моделям (обоснование зоны перепрогона); (б) если выигрыш подтверждён — перепрогон НЕ всех 65к, а зоны использования: звонки клиентов, чьи профили реально пойдут в пилот/карточки CRM (действующие клиенты с активной сделкой/учеником, ориентир — последние 12-18 мес) + 20% запаса. Перепрогон одной командой reset_analysis + worker --stages analyze: расшифровки не трогаются, кэш не мешает (ключ включает модель). Обязательная правка ДО перепрогона: писать модель и версию промпта в analysis_json (сейчас не пишутся — иначе потом не разобрать, какой звонок чем считан).
- Любой запуск analyze — только после явного «да» Дмитрия (правило проекта).

**Почему не «перегнать всё на самой умной»**: 90% ценности выжимок уже снято (summary на месте у всех), пустой budget чаще отражает реальность разговора, чем слабость модели; жечь лимиты на 65к вызовов впрок — копить долг внимания, а не активы. Сильная модель нужна там, где данные реально пойдут в профиль и карточку.

## 3. Глобальный план (этапы; код — только после старта пилота и через главного Кодекса)

**Этап 1. Проект профиля клиента (задача 2 брифа) — сейчас, документы.**
Схема полей с происхождением (механика слотов ТЗ-8: значение + источник + дата + цитата; дети раздельно), хранилище рядом с timeline (фундамент готов), правила конфликтов «позднее побеждает, история хранится», ключи сшивки (телефон — главный, 12 типов готовы). Структура бренда (правка аудита, по §0): ОДИН профиль на клиента с полной историей; бренд — атрибут происхождения на уровне события/поля + правило выдачи на выходе; «бренд неизвестен» у старых событий допустим для хранения. Что показываем в карточке CRM (краткая выжимка, полная история обоих брендов) и что — боту. История Telegram/WhatsApp в timeline не импортирована (v5) — в схеме предусмотреть их место, сам импорт — этап 6. Аудит вторым архитектором. Выход: проектный документ + проект ТЗ Кодексу (в стол до пилота).

**Этап 2. Замер моделей на выжимках (A/B 30-50 звонков) — по «да» Дмитрия.**
Цель: решить, нужен ли выборочный перепрогон analyze. Критерии до прогона (см. §2). Выход: короткий отчёт с решением и оценкой объёма зоны перепрогона.

**Этап 3. Ответ про многоканальность (задача 3 брифа) — документы.**
Конкретный поток «сообщение в любом канале → когда и как обновился профиль»: событийные триггеры («диалог затих» по зазору времени в журнале цикла черновиков; «звонок обработан» по факту появления выжимки), задержки, отказоустойчивость (повторная сборка профиля из событий — профиль пересчитываем, не правим на месте). Выход: документ-ответ Дмитрию.

**Этап 4. Выборочный перепрогон analyze (если этап 2 покажет выигрыш) — по «да» Дмитрия.**
Зона использования + 20%; модель и промпт фиксируются в данных; сверка до/после на пересечении; контрольная ручная выборка 10% перепрогнанного — «новая выжимка не хуже и не теряет детали». Порядок (правка аудита): перепрогон ЗАВЕРШАЕТСЯ ДО пересборки timeline-базы v6, обе на одной дате снимка — иначе профили получат смесь «старые звонки миником, новые сильной» без возможности разобрать.

**Этап 5. План интеграции с пилотом (задача 4 брифа) — документы, включение ПОСЛЕ старта пилота.**
Известный клиент пишет → резолвер чат→контакт (реализован, в v1 кандидат идёт только в журнал) → выжимка профиля в контекст черновика (crm_context перестаёт быть пустым {}). Две явные развилки-решения Дмитрия (правка аудита): (а) включение crm_context = изменение политики безопасности v1 — отдельное письменное «да»; (б) граница между тестовым режимом сотрудника («Представь, что я пишу с номера») и предзагрузкой для реального клиента. Гарантии не ослабляются: P0/бренд-фильтр выхода/числовая защита как были. Запись выжимки профиля в карточку КОНТАКТА: до включения расширить защиту полей (добавить Телефон/ФИО/Email — сейчас не защищены), сделать гейт качества для контакта и снимок-откат по паттерну deal-aware.

**Этап 6. Реализация Кодексом по согласованным ТЗ — после старта пилота.**
Хранилище профилей + событийное обновление + импорт истории Telegram/WhatsApp в timeline + запись в карточку. Каждый шаг за флагом, с контрольными отрицательными тестами (примеры: crm_context пуст → поведение байт-в-байт как сейчас; CRM недоступна → черновик без профиля, не ошибка; бренд-фильтр выхода не ослаблен профилем), замеры по дисциплине проекта.

## 4. Границы (без изменений)

Боевой код не трогаем до старта пилота; ASR/Resolve+Analyze не запускаем без явного разрешения; профили — ПДн, в git/облако/выходные документы только обезличенное; решения уровня политики фиксирует Дмитрий.
