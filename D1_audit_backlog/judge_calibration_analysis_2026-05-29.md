## Задача 4 — предложение по калибровке

Принцип (CLAUDE.md §4): к каждой правке защитного слоя — контрольный ОТРИЦАТЕЛЬНЫЙ тест, доказывающий, что защита от настоящей выдумки/смешения брендов/P0 не ослабла. Ниже у каждой правки есть NEG (защита цела) и POS (артефакт снят).

### Правка №1 — fabrication при grade внутри подтверждённого диапазона (приоритет 1)

**Файл:** `scripts/run_telegram_dynamic_client_sim.py`, `build_judge_prompt`, блок 963-970.
**Что:** добавить в инструкцию пункт: *«Если клиент спрашивает про класс/возраст, а в подтверждённых фактах есть диапазон классов (например „5-11 класс“), и спрошенный класс попадает в диапазон — подтверждение существования курса для этого класса НЕ является fabrication. Если конкретный предмет внутри курса не выделен отдельным фактом — это максимум soft-flag (`assumed_unstated_need`/scope), не hard gate. Hard gate fabrication ставится только когда класс ВНЕ диапазона, либо ни диапазона, ни продукта в фактах бренда нет.»*
**Почему это калибровка, а не переписывание:** меняется одна формулировка существующего правила fabrication; новых веток в коде нет.

- NEG (защита цела): класс вне диапазона (например «3 класс» при факте «5-11») или предмет/продукт без единого факта бренда → fabrication по-прежнему ставится.
- POS (артефакт снят): S3 t1 «6 класс» при факте «5-11, очно, 49000/82000» → не fabrication; max soft-flag.

**Регресс-тест (по диалогам):** прогнать S3 t1 → ожидание `verdict ∈ {PASS, PASS_WITH_NOTES}`, `fabrication ∉ violated_gates`. Контроль-негатив: синтетический ход «есть курс для 3 класса?» при факте «5-11» → `fabrication ∈ violated_gates`.

### Правка №1-бис (durable, опционально) — детерминированный матчер диапазона классов

**Файл:** `src/mango_mvp/channels/fact_claim_audit.py`, `extract_semantic_fact_claims` (+ маленький helper).
**Что:** извлекать claim вида «курс есть для N класса» (число + «класс» + утверждение наличия) и матчить на факт с диапазоном «A-B класс», где A ≤ N ≤ B → выдавать `retrieved_match` (тип `course_grade_in_range`). Тогда LLM-судья видит grounded-метку и не фантазирует.
**Статус:** это узкое расширение существующей логики извлечения, но всё же НОВЫЙ тип claim — оставляю как кандидат после правки №1 (промпт), не вместо неё. **Открыто Claude #1.**

- NEG: «3 класс» при «5-11» → диапазон НЕ покрывает → `no_match` (риск fabrication сохраняется).
- POS: «6 класс» при «5-11» → `retrieved_match`.

### Правка №2 — `wrong_scope` адреса не должен срабатывать на топоним выездной школы

**Файл:** `src/mango_mvp/channels/fact_claim_audit.py`, `_mentions_address`, строка 244.
**Что:** убрать `менделеево` из regex стража уличного адреса: `(сретенк|лялин|адрес)`. «Менделеево» — название локации выездной школы (его корректно ловит `extract_semantic_fact_claims.address_mendeleevo` → `retrieved_match`), а не уличный адрес площадки. В `_asks_address` (248) «менделеево» можно оставить (определение «клиент спрашивает локацию» этим не ломается).
**Альтернатива (если не хотим трогать regex):** в условии 201 добавить исключение контекста лагеря — `and not re.search(r"(лвш|выездн|смена|лагер)", bot)`.

- NEG (защита цела): бот выдаёт «наш адрес: Сретенка, 20» на не-адресный вопрос → `wrong_scope` по-прежнему ставится.
- POS (артефакт снят): «выездная ЛВШ в Менделеево» на вопрос про выездные форматы → `wrong_scope` не ставится; `retrieved_match` на `lvsh_mendeleevo_2026.location.name` сохраняется.

**Регресс-тест:** S3 t4 → `has_wrong_scope=false`. Контроль-негатив: ход с «Сретенка, 20» на вопрос про цену → `has_wrong_scope=true`.

### Правка №3 — `semester_discount`/`annual_discount` требуют близости числа и слова

**Файл:** `src/mango_mvp/channels/fact_claim_audit.py`, `extract_semantic_fact_claims`, паттерны 111-112 и проверка триггеров 124.
**Проблема:** `all(trigger in normalized)` ловит токены «10»/«14», «семестр»/«год», «скид» в РАЗНЫХ предложениях перечня скидок и собирает ложный claim.
**Что:** для этих двух типов заменить bag-of-tokens на проверку близости в одном фрагменте, например regex `(10\s*%?[^.!?\n]{0,20}семестр|семестр[^.!?\n]{0,20}10\s*%)` (и аналогично «14…год»). То есть число процента и слово «семестр/год» должны стоять рядом, а не просто где-то в тексте.

- NEG (защита цела): реальный claim «за семестр действует скидка 10%» в неверном бренде → по-прежнему `other_brand_match`.
- POS (артефакт снят): перечень «10% многодетным … кэшбэк после семестра … скидки не суммируются» → claim `semester_discount` НЕ собирается.

**Регресс-тест:** E6_discount_01 t2, E6_discount_02 t1, E6_discount_05 t2 → в `judge_fact_audit` нет `other_brand_match` "semester_discount". Контроль-негатив: синтетический ход, где бренд-бот говорит «за семестр скидка 10%» по чужому факту → `other_brand_match` остаётся.

### Правка №4 — `contact_hours_as_class_schedule` учитывает дисклеймер
