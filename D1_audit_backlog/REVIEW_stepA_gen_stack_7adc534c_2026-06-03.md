## Детекция (understand модель + keyword floor, монотонно)
- `understand()` (dialogue_contract_pipeline) теперь возвращает `selling:{objection, exit_signal}` — модельно,
  БЕЗ нового вызова. Промпт: objection='price' по смыслу (в т.ч. подтекст), exit_signal на «подумаю/посоветуюсь с
  мужем/посмотрю другие»; нейтральное «сколько стоит» → none/false; **реальный возврат/жалоба/спор → is_p0=true,
  selling не меняет маршрут**. `_clean_selling` санитизирует (objection только 'price' иначе none).
- `_merged_selling_signals` (subscription_llm): objection='price' если **модель ИЛИ keyword**; exit = модель ИЛИ
  keyword. Монотонный союз → ловит и подтекст (модель), и явное (regex-пол d57bff06).

## Gen-композер
Через `context.selling_compose_fn` (+ счётчик роль `bot_selling_compose`). Промпт: признать→ценность/оплата→мягкий
шаг, тон playbook, «без давления, без обещаний результата/поступления, 1-3 предложения», в промпт вложен det-
fallback как референс. compose_fn None → fallback.

## ЖЕЛЕЗНЫЙ fallback — `_verify_selling_generated_text` (ядро безопасности)
Gen-текст валится на **det-текст того же правила** при любом из:
1. `_mentions_other_brand` → "other_brand" (бренд-разделение);
2. `_SELLING_PRESSURE_RE` → "pressure" (искусственная срочность);
3. `_SELLING_GUARANTEE_RE` → "guarantee" (гарантир/100%/обязательно поступит/точно сдаст);
4. **каждое число** из gen-текста (`_SELLING_NUMBER_RE`) обязано встречаться (нормализованно) в фактах
   active_brand, иначе "unsupported_number" (анти-выдумка цифр).
Это и делает «сразу gen» не-рискованным: модель пишет, но не может протащить чужой бренд, давление, обещание или
негрунтованное число — иначе откат на проверенный det.
